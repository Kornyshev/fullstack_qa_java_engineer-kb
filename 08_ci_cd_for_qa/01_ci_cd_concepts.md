# Концепции CI/CD для QA

## Обзор

CI/CD (Continuous Integration / Continuous Delivery / Continuous Deployment) — это набор практик и принципов,
которые позволяют командам разработки часто и надёжно доставлять изменения в production. Для QA-инженера понимание
CI/CD критически важно: именно в pipeline встраиваются автоматизированные тесты, формируются отчёты, принимаются
решения о качестве сборки. Современный QA не просто пишет тесты — он проектирует стратегию их запуска внутри
pipeline и обеспечивает быструю обратную связь для команды.

---

## Что такое CI/CD

### Continuous Integration (CI)

Continuous Integration — практика, при которой разработчики регулярно (несколько раз в день) интегрируют свой код
в общую ветку. Каждая интеграция автоматически проверяется сборкой и тестами.

**Ключевые принципы CI:**
- Единый репозиторий (single source of truth)
- Автоматическая сборка при каждом commit / push
- Автоматический запуск тестов (unit, integration)
- Быстрая обратная связь (feedback loop < 10 минут для быстрых тестов)
- Фиксация сломанной сборки — приоритет №1 для всей команды

### Continuous Delivery (CD)

Continuous Delivery — расширение CI, при котором код после прохождения всех этапов pipeline **готов** к
развёртыванию в production в любой момент. Финальное развёртывание требует ручного подтверждения.

**Отличия от CI:**
- Включает этапы deployment на staging-окружения
- Включает приёмочные и E2E-тесты
- Артефакт сборки версионируется и хранится в registry
- Ручной approval перед production

### Continuous Deployment

Continuous Deployment — полная автоматизация: каждое успешно прошедшее все этапы pipeline изменение
**автоматически** попадает в production без ручного вмешательства.

**Сравнительная таблица:**

| Аспект | CI | Continuous Delivery | Continuous Deployment |
|--------|-----|--------------------|-----------------------|
| Автосборка | Да | Да | Да |
| Автотесты | Unit, Integration | + E2E, Acceptance | + E2E, Acceptance |
| Deploy на staging | Нет | Да | Да |
| Deploy на production | Нет | Ручной | Автоматический |
| Требования к тестам | Базовые | Высокие | Максимальные |

## Этапы Pipeline

Типичный CI/CD pipeline состоит из последовательных этапов (stages), каждый из которых выполняет
определённую задачу и служит "воротами качества" (quality gate).

### Визуализация Pipeline (ASCII-диаграмма)

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                           CI/CD PIPELINE                                        │
│                                                                                 │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────────┐ │
│  │  COMMIT   │──▶│  BUILD   │──▶│  TEST    │──▶│  DEPLOY  │──▶│  RELEASE     │ │
│  │          │   │          │   │          │   │ (Staging) │   │ (Production) │ │
│  └──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────────┘ │
│       │              │              │               │               │           │
│   git push      compile        unit tests      deploy to        manual /       │
│   webhook       package        integration     staging env      auto deploy    │
│   lint check    artifacts      API tests       smoke tests      canary release │
│   SAST scan     Docker image   E2E tests       acceptance       health checks │
│                                code coverage   performance                     │
│                                                                                 │
│  ◄── Быстрая обратная связь ──►  ◄── Глубокая проверка ──►  ◄── Delivery ──►  │
│       (минуты)                        (десятки минут)          (минуты)         │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### Подробное описание этапов

**1. Commit / Source Stage**
- Триггер: push в репозиторий или создание Pull Request / Merge Request
- Действия: статический анализ кода (linting), SAST-сканирование, проверка форматирования
- Время: секунды — 1-2 минуты

**2. Build Stage**
- Компиляция проекта
- Сборка артефактов (JAR, WAR, Docker image)
- Разрешение зависимостей
- Время: 1-5 минут

**3. Test Stage**
- Запуск unit-тестов
- Запуск integration-тестов
- Генерация отчётов о покрытии (code coverage)
- Время: 2-15 минут для быстрых тестов

**4. Deploy to Staging**
- Развёртывание артефакта на тестовое окружение
- Запуск smoke-тестов для проверки работоспособности
- Запуск E2E-тестов и приёмочных тестов
- Время: 5-30 минут

**5. Release / Deploy to Production**
- Развёртывание на production (вручную или автоматически)
- Canary release / Blue-Green deployment
- Post-deployment smoke-тесты
- Мониторинг и health checks

## Где тесты встраиваются в Pipeline

Это один из ключевых вопросов для QA-инженера. Стратегия запуска тестов зависит от типа события
и уровня тестов.

### Тестовая пирамида в контексте CI/CD

```
         ▲ Время выполнения растёт
         │
         │      ┌───────────┐
         │      │  E2E /    │  ◄── Nightly / Release pipeline
         │      │  UI тесты │      (полный набор, 1-4 часа)
         │      ├───────────┤
         │      │  API /    │  ◄── Merge to main / Staging deploy
         │      │Integration│      (регрессия, 15-60 минут)
         │      ├───────────┤
         │      │  Unit     │  ◄── Каждый PR / Commit
         │      │  тесты    │      (быстрые, 2-10 минут)
         │      └───────────┘
         │
         └──────────────────▶ Количество тестов растёт
```

### Стратегия запуска тестов по событиям

**При создании Pull Request (PR) / Merge Request (MR):**
- Unit-тесты (обязательно, блокируют merge)
- Smoke-тесты API (быстрая проверка критичных эндпоинтов)
- Статический анализ и линтинг тестового кода
- Общее время: до 10-15 минут

**При merge в main / develop:**
- Полный набор unit-тестов
- Integration-тесты
- API-регрессия
- Smoke UI-тесты
- Deploy на staging + acceptance-тесты
- Общее время: до 30-60 минут

**Nightly (ночной запуск по расписанию):**
- Полная регрессия (все E2E-тесты)
- Кросс-браузерное тестирование
- Performance-тесты (нагрузочные)
- Security-сканирование (DAST)
- Общее время: 1-4 часа

**Release pipeline:**
- Полная регрессия
- Acceptance-тесты на pre-production окружении
- Smoke-тесты на production после deployment

## Артефакты в CI/CD

Артефакты — это результаты работы pipeline, которые сохраняются для дальнейшего использования.

### Типы артефактов для QA

| Артефакт | Описание | Пример |
|----------|----------|--------|
| Тестовые отчёты | Результаты выполнения тестов | Allure Report, JUnit XML |
| Code Coverage | Отчёт о покрытии кода тестами | JaCoCo HTML report |
| Скриншоты | Снимки экрана при падении UI-тестов | PNG-файлы |
| Видео | Запись выполнения UI-тестов | MP4-файлы от Selenoid |
| Логи | Логи приложения и тестов | test-execution.log |
| Сборка | Собранное приложение | JAR, Docker image |

### Хранение и использование артефактов

```yaml
# Пример: сохранение артефактов в GitLab CI
artifacts:
  when: always                    # Сохранять даже при падении тестов
  paths:
    - build/reports/tests/        # Стандартный отчёт JUnit
    - build/allure-results/       # Результаты для Allure
    - build/screenshots/          # Скриншоты при падении
  expire_in: 7 days               # Срок хранения
```

## Тестовые окружения

### Цепочка окружений

```
┌─────────┐     ┌─────────────┐     ┌─────────────┐     ┌────────────┐
│   DEV   │────▶│   STAGING   │────▶│ PRE-PROD    │────▶│ PRODUCTION │
│         │     │   (QA)      │     │ (UAT)       │     │            │
├─────────┤     ├─────────────┤     ├─────────────┤     ├────────────┤
│ Нестаб. │     │ Стабильная  │     │ Зеркало     │     │ Боевая     │
│ данные  │     │ тестовые    │     │ production  │     │ среда      │
│ Feature │     │ данные      │     │ Реальные    │     │ Реальные   │
│ branches│     │ Integration │     │ объёмы      │     │ пользовали │
│         │     │ + E2E тесты │     │ данных      │     │            │
└─────────┘     └─────────────┘     └─────────────┘     └────────────┘
```

**DEV (Development)**
- Окружение для разработчиков
- Часто нестабильно, данные могут быть неконсистентны
- Используется для первичной отладки

**STAGING (QA)**
- Основное окружение для тестирования
- Стабильные тестовые данные
- Максимально приближено к production по конфигурации
- Здесь запускаются integration, E2E, acceptance тесты

**PRE-PRODUCTION (UAT)**
- Зеркало production с реальными (обезличенными) данными
- Используется для User Acceptance Testing
- Performance-тестирование
- Финальная проверка перед релизом

**PRODUCTION**
- Боевая среда
- Только smoke-тесты и мониторинг
- Canary-тесты при постепенном развёртывании

## Роль QA в CI/CD

### Обязанности QA-инженера в контексте CI/CD

1. **Проектирование тестовой стратегии для pipeline** — определение, какие тесты на каком этапе запускать
2. **Написание и поддержка автотестов** — тесты должны быть стабильными и быстрыми
3. **Настройка pipeline** — конфигурация запуска тестов, генерации отчётов
4. **Мониторинг качества** — анализ тестовых отчётов, отслеживание flaky-тестов
5. **Quality Gates** — определение критериев прохождения каждого этапа
6. **Тестовые данные** — управление данными для разных окружений
7. **Инфраструктура** — настройка тестовых окружений (Selenoid, Docker и т.д.)

### Quality Gates — ворота качества

Quality Gate — набор критериев, которые должны быть выполнены для перехода к следующему этапу pipeline.

```
Пример Quality Gates:

✅ Все unit-тесты проходят            → можно собирать артефакт
✅ Code coverage >= 80%               → можно деплоить на staging
✅ Все smoke-тесты проходят           → staging работоспособен
✅ Регрессия: pass rate >= 98%        → можно деплоить на pre-prod
✅ Performance: p95 latency < 500ms   → можно деплоить на production
✅ Zero critical/blocker дефектов     → релиз одобрен
```

## Связь с тестированием

CI/CD напрямую определяет, как и когда автотесты приносят пользу команде.

- **Быстрая обратная связь:** разработчик узнаёт о сломанном тесте через 5-10 минут после push, а не через
  несколько дней при ручной проверке
- **Стабильность тестов:** flaky-тесты в CI/CD — серьёзная проблема, так как блокируют весь pipeline;
  QA обязан следить за стабильностью
- **Тестовые отчёты:** автоматическая генерация Allure Report после каждого прогона даёт прозрачность
  по состоянию качества
- **Shift Left:** чем раньше в pipeline обнаруживается дефект, тем дешевле его исправление
- **Параллелизация:** CI/CD позволяет запускать тесты параллельно (разные браузеры, разные модули),
  сокращая общее время

## Типичные ошибки

1. **Слишком долгий pipeline.** Если pipeline занимает больше часа, разработчики перестают ждать результатов
   и мержат код без проверки. Решение: разделять тесты на быстрые (PR) и полные (nightly).

2. **Игнорирование flaky-тестов.** Нестабильные тесты подрывают доверие к pipeline. Команда начинает
   перезапускать pipeline "на удачу". Решение: немедленно карантинить и чинить flaky-тесты.

3. **Отсутствие артефактов.** Тест упал, но нет скриншота, лога или видео — расследование занимает часы.
   Решение: всегда сохранять артефакты при падении.

4. **Тесты зависят от порядка выполнения.** Тесты проходят локально, но падают в CI, потому что
   в pipeline порядок запуска может отличаться. Решение: каждый тест должен быть изолирован.

5. **Нет Quality Gates.** Код попадает в production, хотя тесты падают, потому что никто не настроил
   блокировку merge при неуспешных тестах. Решение: обязательные checks в PR.

6. **Одинаковые тесты на всех этапах.** Запуск полного регрессионного набора на каждый PR приводит к
   огромному времени ожидания. Решение: тестовая пирамида и стратегия запуска по событиям.

7. **Hardcoded-конфигурация окружений.** URL-ы, учётные данные, порты прописаны в коде тестов.
   Решение: использовать переменные окружения и секреты CI/CD-системы.
