# Allure TestOps

## Обзор

Allure TestOps (ранее известный как Allure EE — Enterprise Edition) — это платформа
для управления тестированием от компании Qameta Software, которая объединяет ручное
и автоматизированное тестирование в единой системе. В отличие от бесплатного Allure Report,
TestOps предоставляет полноценную TMS (Test Management System) с аналитикой, интеграциями
и возможностью управления тестовыми запусками.

Ключевые характеристики:
- **Единая платформа** — ручные и автоматизированные тест-кейсы в одном месте
- **Живая документация** — тест-кейсы обновляются автоматически из кода тестов
- **Real-time отчётность** — результаты отображаются по мере выполнения тестов
- **Аналитика** — дашборды, метрики, тренды, flaky-анализ
- **Интеграции** — Jira, CI/CD (Jenkins, GitLab CI, GitHub Actions), мессенджеры

Для QA-инженера на интервью важно понимать, что такое Allure TestOps, чем он отличается
от бесплатного Allure Report, и в каких случаях имеет смысл его внедрять.

## Allure TestOps vs Allure Report

### Сравнительная таблица

| Характеристика | Allure Report (Open Source) | Allure TestOps (Платный) |
|---|---|---|
| **Лицензия** | Бесплатный, Apache 2.0 | Коммерческая, подписка |
| **Тип** | Генератор HTML-отчётов | Полноценная TMS-платформа |
| **Развёртывание** | Статические HTML-файлы | Серверное приложение (Docker / Kubernetes) |
| **Хранение результатов** | Файловая система | База данных (PostgreSQL) |
| **Ручные тест-кейсы** | Нет | Да, с markdown-редактором |
| **Управление запусками** | Нет | Да, запуск из UI |
| **Real-time результаты** | Нет (только после генерации) | Да, результаты по мере выполнения |
| **История тестов** | Ограничена (папка `history/`) | Полная история в БД |
| **Flaky-анализ** | Базовый (retries) | Продвинутый ML-анализ |
| **Аналитические дашборды** | Нет | Да, настраиваемые виджеты |
| **Интеграция с Jira** | Нет (только ссылки) | Двусторонняя интеграция |
| **API** | Нет | REST API для автоматизации |
| **Командная работа** | Нет (статический отчёт) | Роли, права доступа, комментарии |
| **Живая документация** | Нет | Да (auto-sync из кода) |
| **Уведомления** | Нет | Slack, Telegram, email |

### Основные компоненты

- **Web UI** — интерфейс для работы с тест-кейсами, запусками, аналитикой
- **API Server** — Spring Boot приложение, REST API
- **Workers** — фоновые процессы для парсинга результатов тестов
- **PostgreSQL** — хранение всех данных
- **S3-совместимое хранилище** — вложения (скриншоты, видео, логи)

## Возможности TMS

### Управление тест-кейсами

Allure TestOps позволяет работать с тест-кейсами двумя способами:

**1. Ручные тест-кейсы** — создаются вручную в Web UI:
- Markdown-редактор для описания шагов
- Прикрепление файлов и изображений
- Параметризация
- Иерархия: Epic → Feature → Story

**2. Автоматизированные тест-кейсы** — импортируются из кода:
- Автоматический синхрон с аннотациями `@Step`, `@Description`, `@Epic` и другими
- При изменении кода теста тест-кейс обновляется автоматически
- «Живая документация» — тест-кейсы всегда актуальны

### Концепция живой документации (Live Documentation)

Живая документация — ключевое преимущество Allure TestOps:

```
Код теста с аннотациями Allure
       ↓ (автоматический импорт)
Тест-кейс в Allure TestOps
       ↓ (при изменении кода)
Тест-кейс обновляется автоматически
```

Пример: если QA-инженер добавляет новый `@Step` в Page Object, этот шаг автоматически
появляется в тест-кейсе в TestOps после следующего прогона.

## Тестовые запуски (Test Launches)

### Типы запусков

- **CI-запуски** — автоматические запуски из CI/CD pipeline
- **Ручные запуски** — инициированные через Web UI
- **Селективные запуски** — запуск только выбранных тест-кейсов
- **Re-run** — повторный запуск упавших тестов

### Интеграция с CI/CD

Allure TestOps предоставляет плагины для популярных CI/CD систем:

**Jenkins:**
- Плагин Allure TestOps для Jenkins
- Автоматическая отправка результатов
- Запуск тестов из TestOps через Jenkins job

**GitLab CI:**
```yaml
# Пример .gitlab-ci.yml с интеграцией Allure TestOps
test:
  stage: test
  script:
    - mvn clean test
  after_script:
    # Отправка результатов в Allure TestOps
    - allurectl upload --launch-name "$CI_PIPELINE_ID"
      --results-dir target/allure-results
  variables:
    ALLURE_ENDPOINT: https://testops.company.com
    ALLURE_PROJECT_ID: "1"
    ALLURE_TOKEN: $ALLURE_TOKEN
```

**GitHub Actions:**
```yaml
- name: Upload results to Allure TestOps
  if: always()
  uses: allure-framework/setup-allurectl@v1
  with:
    allure-endpoint: https://testops.company.com
    allure-token: ${{ secrets.ALLURE_TOKEN }}
    allure-project-id: 1

- name: Upload results
  if: always()
  run: allurectl upload target/allure-results
```

### Утилита allurectl

`allurectl` — CLI-утилита для взаимодействия с Allure TestOps из командной строки:

```bash
# Установка
curl -o allurectl https://github.com/allure-framework/allurectl/releases/latest/download/allurectl_linux_amd64
chmod +x allurectl

# Загрузка результатов
allurectl upload \
    --endpoint https://testops.company.com \
    --token $ALLURE_TOKEN \
    --project-id 1 \
    --launch-name "Regression Run" \
    target/allure-results

# Запуск тестов через TestOps
allurectl job-run plan \
    --endpoint https://testops.company.com \
    --token $ALLURE_TOKEN \
    --project-id 1 \
    --job-run-id $JOB_RUN_ID
```

## Аналитические дашборды

Allure TestOps предоставляет настраиваемые дашборды с виджетами:

### Стандартные виджеты

- **Test Execution Trend** — тренд результатов тестов по запускам
- **Test Case Growth** — рост количества тест-кейсов во времени
- **Automation Coverage** — процент автоматизированных тест-кейсов
- **Flaky Tests** — список нестабильных тестов с частотой падений
- **Longest Tests** — самые долгие тесты
- **Most Failed Tests** — наиболее часто падающие тесты
- **Defect Distribution** — распределение дефектов по категориям
- **Team Activity** — активность членов команды

### Метрики для менеджмента

Allure TestOps предоставляет метрики, которые особенно ценны для менеджмента:

| Метрика | Описание |
|---------|----------|
| **Pass Rate** | Процент прошедших тестов |
| **Automation Rate** | Процент автоматизированных тест-кейсов |
| **Flaky Rate** | Процент нестабильных тестов |
| **Average Duration** | Средняя длительность тестового запуска |
| **Test Case Coverage** | Покрытие функциональности тест-кейсами |
| **Defect Density** | Плотность дефектов по модулям |

## Интеграция с Jira

### Двусторонняя связь

В отличие от бесплатного Allure Report, где `@Issue` — просто ссылка,
Allure TestOps обеспечивает полноценную двустороннюю интеграцию с Jira:

**Из Allure TestOps в Jira:**
- Создание задачи в Jira прямо из упавшего теста
- Привязка тест-кейсов к требованиям (User Stories, Epics)
- Отображение результатов тестов на панели Jira-задачи

**Из Jira в Allure TestOps:**
- Просмотр связанных тест-кейсов прямо в Jira
- Статус тестов отображается в Jira-задаче
- Панель Allure TestOps в Jira (плагин)

### Пример рабочего процесса

```
1. Разработчик создаёт User Story в Jira (PROJ-100)
2. QA создаёт тест-кейс в Allure TestOps, привязывая его к PROJ-100
3. QA автоматизирует тест с аннотацией @Issue("PROJ-100")
4. При запуске тестов результат автоматически обновляется в Jira
5. Менеджер видит в Jira: "5 из 7 тестов прошли для PROJ-100"
```

## Когда использовать TestOps vs бесплатный Allure Report

### Бесплатный Allure Report подходит, если:
- Небольшая команда (1-5 QA)
- Только автоматизированные тесты
- Достаточно статических HTML-отчётов
- Нет необходимости в TMS
- Бюджет ограничен
- CI/CD pipeline уже настроен для генерации отчётов

### Allure TestOps нужен, если:
- Средняя или большая команда (5+ QA)
- Есть и ручные, и автоматизированные тесты
- Нужна полноценная TMS с управлением тест-кейсами
- Требуется интеграция с Jira
- Нужна аналитика и метрики для менеджмента
- Нужна живая документация
- Требуется управление тестовыми запусками из UI
- Команда хочет единую точку входа для всей тестовой деятельности

### Альтернативы Allure TestOps

| Инструмент | Тип | Преимущества |
|------------|-----|-------------|
| **TestRail** | Коммерческая TMS | Зрелая платформа, широкие интеграции |
| **Zephyr** | Плагин для Jira | Глубокая интеграция с Jira |
| **qTest** | Коммерческая TMS | AI-аналитика, масштабируемость |
| **Xray** | Плагин для Jira | BDD-поддержка, интеграция с Cucumber |
| **TestLink** | Open Source TMS | Бесплатная, базовая функциональность |

## Развёртывание Allure TestOps

### Docker Compose (обзорный уровень)

Allure TestOps разворачивается как набор Docker-контейнеров:

```yaml
# Упрощённая схема docker-compose.yml
services:
  gateway:
    image: allure-gateway
    ports:
      - "8080:8080"
  uaa:
    image: allure-uaa           # Сервис аутентификации
  report:
    image: allure-report         # Сервис отчётов
  postgres:
    image: postgres:15           # База данных
  rabbitmq:
    image: rabbitmq:3            # Очередь сообщений
  minio:
    image: minio                 # Хранилище вложений (S3-совместимое)
```

### Требования к ресурсам

- **Минимальные**: 4 CPU, 8 GB RAM, 50 GB SSD
- **Рекомендуемые**: 8 CPU, 16 GB RAM, 200 GB SSD
- **PostgreSQL**: отдельный инстанс для продуктивного использования

## Связь с тестированием

Allure TestOps закрывает ряд задач, которые критичны для зрелого QA-процесса:

- **Единый источник правды** — все тест-кейсы (ручные и автоматизированные) в одном месте
- **Трассируемость** — связь тестов с требованиями через Jira
- **Отчётность для стейкхолдеров** — дашборды с метриками для менеджмента
- **Управление качеством** — тренды, flaky-анализ, coverage-метрики
- **Эффективность команды** — запуск тестов, re-run, селективный запуск
- **Живая документация** — тест-кейсы всегда актуальны и отражают реальный код

## Типичные ошибки

1. **Внедрение TestOps без реальной необходимости** — для маленькой команды достаточно бесплатного Allure Report
2. **Не настроена интеграция с CI/CD** — результаты загружаются вручную, теряется автоматизация
3. **Дублирование тест-кейсов** — ручные тест-кейсы в TestOps не синхронизированы с автоматизированными
4. **Игнорирование живой документации** — тест-кейсы создаются вручную вместо импорта из кода
5. **Не настроена интеграция с Jira** — теряется главное преимущество — трассируемость
6. **Не используются дашборды** — данные есть, но не визуализируются для менеджмента
7. **Не настроен `allurectl`** — результаты тестов не попадают в TestOps автоматически
8. **Неправильное разделение прав** — все пользователи имеют admin-доступ
