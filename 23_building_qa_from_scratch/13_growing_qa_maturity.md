# Рост зрелости тестирования: от хаоса к системе

## Обзор

Большинство проектов не начинаются с идеального процесса тестирования. Чаще всего QA-инженер приходит на проект и видит одно из двух: либо тестирования нет вообще, либо оно существует формально, но не приносит реальной пользы.

Рост зрелости тестирования — это последовательный, управляемый процесс. Нельзя перепрыгнуть с уровня "хаос" на уровень "качество как культура" за один спринт. Каждый уровень требует определённых инвестиций, инструментов и — самое сложное — изменения мышления команды.

В этом разделе разберём модель зрелости из 6 уровней (0-5), конкретные шаги для перехода между уровнями, реалистичные сроки и самый частый вопрос на собеседовании senior QA: "Вы приходите на проект без тестирования. Что делать?"

---

## Модель зрелости тестирования

### Общая картина

```
Level 5 ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  Quality Engineering Culture
Level 4 ▓▓▓▓▓▓▓▓▓▓▓▓▓▓     Comprehensive: метрики, shift-left, prevention
Level 3 ▓▓▓▓▓▓▓▓▓▓▓▓       Automation: CI/CD, автотесты, регресс
Level 2 ▓▓▓▓▓▓▓▓▓▓         Structured: планы, процессы, документация
Level 1 ▓▓▓▓▓▓▓▓           Basic Manual: ad-hoc тестирование, реакция на баги
Level 0 ▓▓▓▓▓▓             No Testing: "разработчик проверил — значит работает"
```

---

### Level 0: No Testing (отсутствие тестирования)

**Признаки:**
- Выделенных QA-специалистов нет
- Разработчики "проверяют сами" перед мержем
- Unit-тестов нет или они написаны формально и не поддерживаются
- Баги обнаруживают пользователи в production
- Нет понятия "тест-кейс", "тест-план", "regression"
- Релиз = "задеплоили и молимся"

**Типичная ситуация:** ранний стартап, прототип, внутренний инструмент, проект "на коленке".

**Последствия:**
- Высокая частота production-инцидентов
- Разработчики тратят 30-50% времени на исправление багов вместо фич
- Нет уверенности в качестве при каждом релизе
- Technical debt растёт экспоненциально

**Метрики (если их вообще можно измерить):**

```
Defect Leakage:       90-100% (все баги находят пользователи)
Automation Coverage:   0%
Regression Time:       0 (regression не делается)
Release Confidence:    Минимальная
```

---

### Level 1: Basic Manual (базовое ручное тестирование)

**Признаки:**
- Есть хотя бы один QA (часто совмещающий с другими ролями)
- Тестирование — ad-hoc: "поклиkали — вроде работает"
- Нет формализованных тест-кейсов
- Тестируются только новые фичи, regression не делается
- Баг-трекер используется, но хаотично (нет severity, приоритетов, процесса)
- Тестирование начинается после разработки

**Типичная ситуация:** стартап, который нанял первого QA, или проект, где тестирование "возложено" на junior-специалиста.

**Метрики:**

```
Defect Leakage:       50-70%
Automation Coverage:   0-5%
Regression Time:       Не определено
Release Confidence:    Низкая
```

---

### Level 2: Structured Testing (структурированное тестирование)

**Признаки:**
- Тест-кейсы написаны и хранятся в TMS (TestRail, Zephyr, Qase)
- Есть тест-план для каждого релиза
- Regression suite определён и выполняется перед релизом
- Баг-трекинг формализован: severity, priority, workflow
- QA участвует в планировании спринта
- Есть definition of done, включающий тестирование
- Smoke-тесты определены

**Типичная ситуация:** компания с устоявшейся командой разработки, 1-3 QA, регулярные релизы.

**Метрики:**

```
Defect Leakage:       20-40%
Automation Coverage:   5-20%
Regression Time:       2-5 дней (ручной)
Release Confidence:    Средняя
```

---

### Level 3: Automation & CI/CD (автоматизация и непрерывная интеграция)

**Признаки:**
- Автоматизированные unit-тесты с coverage > 60%
- API-автотесты для ключевых эндпоинтов
- CI/CD pipeline с автоматическим прогоном тестов
- UI-автотесты для smoke и критических потоков
- Regression частично или полностью автоматизирован
- Тестовые окружения стабильны и воспроизводимы
- QA пишет код (автотесты, утилиты, генераторы тестовых данных)

**Типичная ситуация:** зрелый продукт, команда 5-15 разработчиков, 2-4 QA (включая автоматизаторов).

**Метрики:**

```
Defect Leakage:       10-20%
Automation Coverage:   40-70%
Regression Time:       2-4 часа (автоматизированный)
Release Confidence:    Выше средней
```

---

### Level 4: Comprehensive (комплексное тестирование)

**Признаки:**
- Shift-left: QA участвует в design и code review
- Метрики качества отслеживаются и анализируются
- Performance и security testing — регулярные
- Трассируемость: требования → тесты → результаты → дефекты
- Contract testing для микросервисов
- Chaos engineering (эксперименты на устойчивость)
- Test data management автоматизирован
- Flaky-тесты отслеживаются и устраняются системно
- QA влияет на архитектурные решения (testability)

**Типичная ситуация:** enterprise-продукт, высоконагруженная система, строгие SLA.

**Метрики:**

```
Defect Leakage:       < 10%
Automation Coverage:   70-90%
Regression Time:       30-60 минут
Release Confidence:    Высокая
```

---

### Level 5: Quality Engineering Culture (культура качества)

**Признаки:**
- Качество — ответственность всей команды, не только QA
- Разработчики пишут качественные тесты как часть Definition of Done
- QA фокусируется на предотвращении, а не обнаружении дефектов
- Continuous deployment с автоматическими quality gates
- Observability: production мониторинг как часть стратегии тестирования
- A/B testing, canary releases, feature flags — инструменты QA
- Постоянное улучшение процесса на основе метрик
- QA contributes в код продукта (testability improvements)
- Нулевая толерантность к flaky-тестам
- Документация тестирования — живая и актуальная

**Типичная ситуация:** top-tier tech-компании (Google, Netflix, Spotify), зрелые enterprise-продукты с сильной инженерной культурой.

**Метрики:**

```
Defect Leakage:       < 5%
Automation Coverage:   85-95%
Regression Time:       < 30 минут
Release Confidence:    Очень высокая
```

## Переходы между уровнями

### Переход 0 → 1: от "нет тестирования" к базовому

**Срок: 1-3 месяца**

**Первая неделя:**

```
День 1-2: Аудит текущего состояния
├── Какой продукт? (web, mobile, API, всё вместе)
├── Какие технологии? (стек, инфраструктура)
├── Кто и как тестирует сейчас? (даже если "никак")
├── Какие баги были в production за последние 3 месяца?
├── Какой процесс разработки? (Scrum, Kanban, хаос)
└── Какие окружения есть? (dev, staging, production)

День 3-5: Quick Wins
├── Настроить баг-трекер (Jira, Linear, GitHub Issues)
├── Определить severity и priority для дефектов
├── Написать smoke-чеклист для основного бизнес-потока (10-15 проверок)
├── Начать тестировать текущие задачи в спринте
└── Провести exploratory session на продукте (найти самые "больные" места)
```

**Первый месяц:**

```
Неделя 2-4:
├── Составить список критических бизнес-потоков (top-10)
├── Написать чеклисты для каждого потока
├── Начать тестировать каждую задачу перед релизом
├── Ввести правило: задача не закрывается без тестирования QA
├── Начать smoke-тестирование перед каждым деплоем
└── Создать простой отчёт: что протестировано, что найдено
```

**Ключевые deliverables:**
1. Настроенный баг-трекер с workflow
2. Smoke-чеклист
3. Процесс "задача → разработка → QA → деплой"
4. Первый простой отчёт о тестировании

**Типичные препятствия:**
- "У нас нет времени на тестирование" → показать стоимость production-багов
- "Разработчики сами тестируют" → показать пропущенные баги
- "QA замедляет процесс" → показать, что hotfix-ы замедляют больше

### Переход 1 → 2: от ad-hoc к структуре

**Срок: 2-4 месяца**

**Действия:**

```
Месяц 1: Формализация
├── Выбрать и настроить TMS (TestRail / Qase / Zephyr)
├── Написать тест-кейсы для smoke-сценариев
├── Создать regression suite для основного функционала
├── Формализовать баг-репорт: шаблон, обязательные поля
├── Ввести Definition of Done, включающий QA-критерии
└── Начать участвовать в Sprint Planning

Месяц 2: Процессы
├── Написать тест-кейсы для 50% существующего функционала
├── Провести первый полный regression перед релизом
├── Ввести тест-план для каждого спринта
├── Начать вести release notes
├── Определить Entry/Exit Criteria для тестирования
└── Начать exploratory testing как регулярную практику

Месяц 3-4: Стабилизация
├── Покрыть тест-кейсами 80% функционала
├── Regression стал обязательным перед каждым релизом
├── Метрики: начать отслеживать defect leakage и pass rate
├── Провести первую ретроспективу QA-процесса
└── Составить roadmap автоматизации
```

**Ключевой результат:** тестирование из "кто-то что-то проверяет" превращается в управляемый процесс с документацией, планами и предсказуемым результатом.

### Переход 2 → 3: от ручного к автоматизированному

**Срок: 3-6 месяцев**

Самый технически сложный переход. Требует новых навыков (программирование, CI/CD) и инфраструктуры.

**Действия:**

```
Месяц 1: Фундамент автоматизации
├── Выбрать технологический стек:
│   ├── Unit: JUnit 5 / pytest / Jest (зависит от проекта)
│   ├── API: RestAssured / requests / Supertest
│   ├── UI: Playwright / Selenium / Cypress
│   └── Reporting: Allure
├── Настроить проект автотестов (структура, CI/CD интеграция)
├── Автоматизировать smoke suite (10-20 тестов)
├── Настроить запуск тестов в CI (GitHub Actions / GitLab CI / Jenkins)
└── Первый "зелёный" прогон в pipeline

Месяц 2-3: Наращивание покрытия
├── Автоматизировать API-тесты для ключевых эндпоинтов
├── Автоматизировать critical path UI-тестов
├── Настроить параллельный запуск тестов
├── Внедрить тестовые данные (fixtures, factories, database seeding)
├── Настроить Allure-отчёты
└── Интегрировать тесты в pull request workflow

Месяц 4-6: Зрелая автоматизация
├── Автоматизировать regression suite (80%+)
├── Настроить ночные прогоны full regression
├── Внедрить retries для нестабильных тестов (и план их починки)
├── Настроить тестовые окружения (Docker / TestContainers)
├── Code review для автотестов как обязательная практика
└── Метрики: automation coverage, execution time, flaky rate
```

**Критические решения на этом этапе:**

```
Вопрос                      │ Рекомендация
────────────────────────────┼──────────────────────────────────────
Кто пишет автотесты?        │ QA-автоматизаторы + разработчики (unit)
Какой процент автоматизировать? │ 80% regression, 100% smoke
Что НЕ автоматизировать?    │ Exploratory, usability, визуальную проверку
Как бороться с flaky?        │ Quarantine + отдельная задача на починку
Unit vs E2E пропорция?      │ 70% unit, 20% API, 10% UI
```

### Переход 3 → 4: от автоматизации к комплексному качеству

**Срок: 4-6 месяцев**

**Действия:**

```
Месяц 1-2: Расширение горизонта
├── Внедрить performance testing (JMeter / Gatling / k6)
│   ├── Baseline performance metrics
│   ├── Load тесты перед каждым релизом
│   └── Performance gate в CI/CD
├── Внедрить security testing
│   ├── SAST (SonarQube, Semgrep)
│   ├── DAST (OWASP ZAP)
│   └── Dependency scanning (Snyk, Dependabot)
└── Начать contract testing (Pact) для микросервисов

Месяц 3-4: Shift-Left
├── QA участвует в design review (находим проблемы до кода)
├── QA участвует в code review (testability, edge cases)
├── BDD: acceptance criteria в формате Given/When/Then
├── Three Amigos сессии (PO + Dev + QA) для каждой user story
├── Testability как архитектурное требование
└── Разработчики пишут integration тесты (не только unit)

Месяц 5-6: Метрики и оптимизация
├── Dashboard с ключевыми метриками качества
├── Traceability Matrix: требования → тесты → дефекты
├── Регулярный анализ production-инцидентов
├── Root Cause Analysis для каждого critical/blocker бага
├── Test data management автоматизирован
└── Chaos engineering (эксперименты: "что если БД упадёт?")
```

### Переход 4 → 5: от процесса к культуре

**Срок: 6-12 месяцев**

Самый сложный переход, потому что он не технический — он культурный. Нельзя "внедрить культуру" по приказу.

**Действия:**

```
Постоянно:
├── Обучение разработчиков практикам тестирования
│   ├── Воркшопы по написанию тестов
│   ├── Pair programming QA + Dev
│   └── Sharing sessions: "как QA нашёл этот баг"
├── Quality Champions в каждой команде
│   └── Разработчик, ответственный за качество в своей команде
├── "Quality Is Everyone's Responsibility" — не лозунг, а практика
│   ├── Разработчик не мержит без тестов
│   ├── PO понимает trade-off между скоростью и качеством
│   └── DevOps обеспечивает observability
├── QA как Quality Coach
│   ├── QA не только тестирует, но и учит команду
│   ├── QA помогает разработчикам писать лучшие тесты
│   └── QA анализирует процесс и предлагает улучшения
└── Continuous Improvement
    ├── Регулярные quality retrospectives
    ├── Эксперименты: "а что если попробовать X?"
    ├── Benchmarking: сравнение с лучшими практиками
    └── Внутренние tech talks по качеству
```

**Признаки достижения Level 5:**
- Разработчик сам говорит: "подождите, я напишу тесты"
- PO спрашивает: "а мы это протестировали?" до релиза
- DevOps автоматически добавляет мониторинг для новых фич
- Ретроспектива включает обсуждение качества как отдельную тему
- QA-инженер тратит больше времени на предотвращение, чем на обнаружение дефектов

## Реалистичные сроки

```
Переход        │ Минимум  │ Реалистично │ Если есть сопротивление
───────────────┼──────────┼─────────────┼────────────────────────
0 → 1          │ 1 мес    │ 2-3 мес     │ 3-4 мес
1 → 2          │ 2 мес    │ 3-4 мес     │ 4-6 мес
2 → 3          │ 3 мес    │ 4-6 мес     │ 6-9 мес
3 → 4          │ 4 мес    │ 5-6 мес     │ 6-12 мес
4 → 5          │ 6 мес    │ 9-12 мес    │ 12-24 мес
───────────────┼──────────┼─────────────┼────────────────────────
ИТОГО 0 → 5   │ 16 мес   │ 24-30 мес   │ 30-48 мес
```

> **Опытное мнение:** не торопитесь. Лучше прочно стоять на Level 2, чем шатко балансировать на Level 3. Каждый уровень должен быть стабильным, прежде чем переходить дальше. Команда на Level 2 с прочным фундаментом перейдёт на Level 3 быстрее, чем команда, которая "перепрыгнула" на Level 3 без базы.

## Как определить текущий уровень зрелости

### Диагностический чеклист

Ответьте "да" или "нет" на каждый вопрос:

```
LEVEL 1 — Базовое тестирование:
□ Есть хотя бы один человек, ответственный за тестирование?
□ Используется баг-трекер?
□ Новые фичи тестируются перед релизом?
□ Есть хотя бы informal smoke-чеклист?

LEVEL 2 — Структурированное тестирование:
□ Тест-кейсы документированы и хранятся в TMS?
□ Перед релизом выполняется regression?
□ Есть тест-план для каждого спринта/релиза?
□ QA участвует в планировании?
□ Есть Definition of Done, включающий тестирование?

LEVEL 3 — Автоматизация:
□ Автоматизированные тесты запускаются в CI/CD?
□ Automation coverage > 40%?
□ Unit-тесты пишутся разработчиками?
□ Есть API-автотесты?
□ Regression выполняется автоматически?

LEVEL 4 — Комплексное тестирование:
□ QA участвует в design review?
□ Есть performance testing?
□ Есть security testing?
□ Метрики качества отслеживаются и анализируются?
□ Есть traceability (требования → тесты → баги)?

LEVEL 5 — Культура качества:
□ Разработчики самостоятельно пишут качественные тесты?
□ Quality gates в CI/CD блокируют деплой при проблемах?
□ Production мониторинг интегрирован с процессом тестирования?
□ QA тратит больше времени на prevention, чем на detection?
□ Качество обсуждается на ретроспективах всей командой?
```

**Правило оценки:** ваш уровень — самый высокий, где вы ответили "да" на 80%+ вопросов.

## Как убедить руководство инвестировать в качество

### Говорить на языке бизнеса

QA часто допускают ошибку: говорят о "покрытии", "автоматизации", "CI/CD". Менеджмент слышит: "мы хотим заниматься интересными техническими штуками".

**Вместо этого — переводите на язык бизнеса:**

| QA говорит                           | Бизнес слышит                                      |
|--------------------------------------|-----------------------------------------------------|
| "Нам нужна автоматизация"            | "Мы хотим дорогой инструмент"                      |
| "Снизим regression time с 3 дней до 2 часов" | "Будем релизить еженедельно вместо ежемесячно" |
| "Увеличим test coverage"             | "Непонятная техническая метрика"                    |
| "Снизим defect leakage на 50%"       | "Вдвое меньше жалоб клиентов"                      |
| "Внедрим shift-left"                 | "Будем находить баги дешевле и раньше"              |

### Стоимость бага на разных этапах

Это самый сильный аргумент для бизнеса:

```
Этап обнаружения      │ Относительная стоимость исправления
──────────────────────┼────────────────────────────────────
Design / Requirements │ $1           (обсудили — исправили)
Development           │ $10          (переписали код)
QA / Testing          │ $100         (fix + retest + regression)
Staging               │ $500         (+ деплой + verification)
Production            │ $1,000-10,000 (+ support + reputation + hotfix)
```

**Пример расчёта ROI автоматизации:**

```
Текущая ситуация:
- Ручной regression: 3 дня × 2 QA = 6 человеко-дней на релиз
- 12 релизов в год = 72 человеко-дня = ~$36,000 (при зарплате QA)
- + 2 hotfix в месяц × 1 день = 24 человеко-дня = ~$12,000

Инвестиция в автоматизацию:
- QA-автоматизатор на 6 месяцев = ~$45,000

Результат через 6 месяцев:
- Автоматический regression: 2 часа (вместо 3 дней)
- Hotfix-ы сократились на 60% (благодаря автотестам в CI)
- Экономия: ~$30,000/год + ускорение time-to-market

ROI = (Экономия - Инвестиция) / Инвестиция = ($30,000 - $45,000) / $45,000
Первый год: -33% (инвестиция ещё не окупилась)
Второй год: +67% (автотесты работают, поддержка < $15,000/год)
```

### Реальные кейсы для презентации

**Кейс 1: "Баг в production, которого можно было избежать"**

```
Что случилось: баг в расчёте скидки → 10,000 заказов с неправильной ценой
Стоимость: $50,000 (возвраты) + $20,000 (support) + репутационные потери
Что бы помогло: unit-тест на граничное значение скидки
Стоимость предотвращения: 30 минут работы разработчика
```

**Кейс 2: "Hotfix, который сломал больше, чем починил"**

```
Что случилось: hotfix без тестирования → новый critical баг
Стоимость: 8 часов downtime → $80,000 потерянной выручки
Что бы помогло: автоматизированный smoke (15 минут)
Стоимость предотвращения: 1 неделя настройки smoke suite
```

## Интервью: "Вы приходите на проект без тестирования. Что делать?"

Это один из самых популярных вопросов для Senior QA. Вот структурированный ответ.

### Первая неделя: Discovery

```
1. Понять продукт
   ├── Что делает продукт? Кто пользователи?
   ├── Какие критические бизнес-потоки?
   ├── Какой стек технологий?
   └── Какие интеграции с внешними системами?

2. Понять команду и процессы
   ├── Сколько разработчиков? Какой процесс (Scrum/Kanban/хаос)?
   ├── Как происходит деплой? Как часто?
   ├── Кто и как тестирует сейчас (даже если "никак")?
   └── Какое отношение команды к тестированию?

3. Понять боль
   ├── Какие production-инциденты были за последние 3 месяца?
   ├── Сколько времени уходит на hotfix-ы?
   ├── Что больше всего "болит" у разработчиков и PO?
   └── Какие области кода самые проблемные?
```

### Первый месяц: Quick Wins

```
Неделя 2:
├── Настроить баг-трекер с минимальным процессом
├── Написать smoke-чеклист (10-15 пунктов для основного потока)
├── Начать тестировать задачи текущего спринта
└── Провести exploratory session, задокументировать находки

Неделя 3-4:
├── Определить top-10 критических бизнес-потоков
├── Написать чеклисты для каждого
├── Ввести правило: задача не закрывается без QA-проверки
├── Первый smoke перед деплоем
└── Показать команде первый "отчёт": что нашли, что рискованно
```

### Второй месяц: Установка процесса

```
├── Выбрать и настроить TMS
├── Начать писать тест-кейсы (начать с критических потоков)
├── Провести первый полный regression перед релизом
├── Участвовать в Sprint Planning: оценивать тестируемость задач
├── Начать вести метрики: количество багов, severity, где найдены
└── Предложить Definition of Done, включающий QA
```

### Третий месяц: Планирование развития

```
├── Составить Roadmap автоматизации (что, когда, зачем)
├── Начать автоматизацию smoke suite
├── Предложить улучшения процесса на ретроспективе
├── Презентовать результаты руководству:
│   ├── Сколько багов найдено до production (раньше все шли в production)
│   ├── Какие зоны риска выявлены
│   └── План на следующий квартал
└── Обосновать необходимость дополнительных ресурсов (если нужны)
```

> **Ключевой принцип:** не пытаться изменить всё сразу. Начать с того, что приносит максимальную пользу при минимальных усилиях (quick wins). Заручиться поддержкой команды через видимые результаты, а потом постепенно расширять.

## Связь с тестированием

Зрелость тестирования определяет, КАК вы тестируете:

| Уровень | Основные виды тестирования                          |
|---------|-----------------------------------------------------|
| Level 0 | Нет формального тестирования                        |
| Level 1 | Ad-hoc manual testing, exploratory                  |
| Level 2 | Structured manual: smoke, regression, checklist     |
| Level 3 | + Automated: unit, API, UI smoke, CI/CD             |
| Level 4 | + Performance, security, contract, chaos            |
| Level 5 | + Production testing, A/B, canary, observability    |

Каждый уровень не заменяет, а дополняет предыдущий. На Level 5 всё ещё выполняется exploratory testing — просто фокус сместился с обнаружения багов на предотвращение.

## Типичные ошибки

1. **Попытка перепрыгнуть уровни.** "Давайте сразу внедрим CI/CD с автотестами!" — без базовых тест-кейсов и процессов автоматизировать нечего. Сначала нужно знать ЧТО тестировать, потом КАК автоматизировать
2. **Фокус на инструментах вместо процессов.** "Мы купили TestRail, теперь у нас Level 2!" — TMS без наполненных тест-кейсов и процесса их выполнения — пустая трата денег
3. **Автоматизация ради автоматизации.** Автоматизировать 100% тестов — не цель. Цель — быстрая и надёжная обратная связь. 50% правильно автоматизированных тестов лучше 100% нестабильных
4. **Игнорирование культурных аспектов.** Можно внедрить лучшие инструменты и процессы, но если команда не верит в ценность качества — всё развалится. Level 5 невозможен без культурной трансформации
5. **Отсутствие поддержки руководства.** QA не может расти в вакууме. Если менеджмент не понимает ценности — нужно инвестировать время в убеждение (метрики, ROI, бизнес-кейсы)
6. **Одинаковый подход для всех проектов.** Стартап на ранней стадии не нуждается в Level 4. Банковское ПО не может оставаться на Level 1. Целевой уровень зависит от контекста
7. **Забыть про regression при быстром росте.** Команда активно добавляет фичи, но regression suite не обновляется. Через полгода regression неактуален и бесполезен
8. **Не измерять прогресс.** Без метрик невозможно понять, продвигаетесь ли вы между уровнями. Определите baseline и отслеживайте динамику

## Вопросы на интервью

### 🟢 Базовый уровень

**В: Что такое QA maturity model? Зачем она нужна?**
О: QA maturity model — фреймворк, описывающий уровни зрелости процесса тестирования: от полного отсутствия (Level 0) до культуры качества (Level 5). Нужна для: диагностики текущего состояния тестирования на проекте, определения конкретных шагов для улучшения, установки реалистичных ожиданий (нельзя "перепрыгнуть" уровни), обоснования инвестиций перед руководством.

**В: Какие основные уровни зрелости тестирования?**
О: Level 0 — нет тестирования, Level 1 — базовое ручное (ad-hoc), Level 2 — структурированное тестирование (планы, тест-кейсы, regression), Level 3 — автоматизация и CI/CD, Level 4 — комплексное (shift-left, performance, security, метрики), Level 5 — культура качества (вся команда ответственна, предотвращение вместо обнаружения). Каждый уровень дополняет, а не заменяет предыдущий.

**В: Сколько времени занимает переход между уровнями?**
О: Реалистичные сроки: 0→1 — 2-3 месяца, 1→2 — 3-4 месяца, 2→3 — 4-6 месяцев, 3→4 — 5-6 месяцев, 4→5 — 9-12 месяцев. Полный путь от 0 до 5 занимает 2-3 года. Сроки увеличиваются, если есть сопротивление команды или руководства. Ключевой фактор — не торопиться и стабилизировать каждый уровень перед переходом к следующему.

### 🟡 Средний уровень

**В: Вы пришли на проект, где тестирование есть, но хаотичное (Level 1). Какие первые три действия?**
О: (1) Аудит: понять, что тестируется, как, какие боли (production-инциденты, hotfix-ы). Это занимает 2-3 дня. (2) Quick win: написать smoke-чеклист для критического бизнес-потока и начать его выполнять перед каждым деплоем. Это даёт видимый результат за неделю. (3) Формализация: настроить TMS, создать шаблон баг-репорта, ввести правило "задача не закрывается без QA". Это закладывает фундамент для Level 2. Параллельно — собирать данные (сколько багов ловим, сколько пропускаем) для обоснования дальнейших шагов.

**В: Как определить целевой уровень зрелости для проекта?**
О: Зависит от контекста: (1) Критичность продукта — банковское ПО или медицинское должно стремиться к Level 4-5, internal tool может остаться на Level 2-3. (2) Размер команды — для команды из 3 разработчиков Level 4 может быть overkill. (3) Частота релизов — ежедневные деплои требуют минимум Level 3. (4) Бизнес-требования — compliance (PCI DSS, GDPR, HIPAA) требует определённого уровня. (5) Бюджет — переход на каждый уровень требует инвестиций. Правило: целевой уровень = минимальный уровень, обеспечивающий приемлемое качество для данного контекста.

**В: Как убедить руководство инвестировать в тестирование?**
О: Говорить на языке бизнеса: (1) Показать стоимость production-багов за последние 3-6 месяцев (hotfix-ы, потерянные клиенты, support). (2) Сравнить стоимость предотвращения vs. стоимость исправления (баг в design — $1, в production — $1000-10000). (3) Посчитать ROI автоматизации: сколько стоит ручной regression × количество релизов vs. стоимость настройки автотестов. (4) Показать конкретный план с milestones и ожидаемыми результатами. Менеджмент понимает деньги, риски и скорость — говорите на этом языке.

### 🔴 Продвинутый уровень

**В: Вы приходите на проект без тестирования. Что вы делаете в первые 3 месяца?**
О: Месяц 1 — Discovery и Quick Wins: первая неделя — аудит продукта, команды, процессов, production-инцидентов. Вторая неделя — настройка баг-трекера, smoke-чеклист для основного бизнес-потока, начинаю тестировать задачи спринта. Третья-четвёртая неделя — определяю top-10 критических потоков, чеклисты, ввожу правило "нет деплоя без smoke". Месяц 2 — Структура: выбираю TMS, пишу тест-кейсы для критичного функционала, провожу первый regression, участвую в Planning, ввожу DoD. Месяц 3 — Планирование роста: составляю roadmap автоматизации, начинаю автоматизацию smoke, презентую результаты руководству (сколько багов поймали, сколько пропускали раньше), обосновываю дополнительные ресурсы. К концу третьего месяца мы стабильно на Level 2 с чётким планом перехода на Level 3.

**В: На проекте Level 3 (есть автоматизация), но defect leakage остаётся высоким (20%+). Что делать?**
О: Высокий defect leakage при наличии автоматизации указывает на проблемы не в количестве, а в качестве тестирования. Мой анализ: (1) Классифицировать leaked дефекты: какого типа баги проскакивают? Если integration bugs — автотесты проверяют компоненты изолированно, нужны E2E. Если edge cases — тест-дизайн слабый, нужно улучшить coverage через boundary values, pairwise. Если environment-specific — staging не соответствует production. Если performance/security — эти виды тестирования отсутствуют. (2) Для каждого leaked бага — обратный анализ: "какой тест должен был это поймать?" и создать этот тест. (3) Системные меры: внедрить shift-left (QA на design review), улучшить staging, добавить недостающие виды тестирования. (4) Метрика прогресса: отслеживать leaked defects по категориям помесячно. Это переход с Level 3 на Level 4 — от "тесты есть" к "тесты ловят правильные баги".

**В: Как вы измеряете прогресс при росте зрелости тестирования?**
О: Для каждого перехода — свой набор leading и lagging indicators. Leading (опережающие): количество автоматизированных тестов, coverage, процент тестируемых задач. Lagging (отстающие, но более значимые): defect leakage, production incidents, hotfix frequency, mean time to detect. Конкретно: при переходе 0→1 слежу за тем, сколько задач тестируется до релиза (от 0% к 100%). При 1→2 — наличие и актуальность тест-кейсов, выполнение regression. При 2→3 — automation coverage, CI/CD execution time, flaky rate. При 3→4 — defect leakage trend, shift-left participation (% design reviews с QA), performance/security test coverage. При 4→5 — процент дефектов, найденных разработчиками самостоятельно, quality discussions на ретроспективах, prevention vs. detection ratio. Всё визуализируется на dashboard с трендами за 3-6 месяцев.
