# Стратегия автоматизации: что автоматизировать и когда

## Обзор

Автоматизация тестирования — это не цель, а **инструмент**. Цель — быстрый и надёжный
feedback о качестве продукта. Автоматизация — один из способов достичь этой цели.

Типичная ловушка: "У нас есть 500 автотестов → мы хорошо тестируем."
Реальность: 200 из них flaky, 100 проверяют неактуальные сценарии,
а критический бизнес-flow покрыт только ручными тестами.

Автоматизация без стратегии — это **расход бюджета**, а не **инвестиция**.

В этом разделе мы разберём: **что** автоматизировать, **когда** начинать,
как считать **ROI**, как **расти** в зрелости и как **продать** идею менеджменту.

---

## Решение об автоматизации: что автоматизировать

### Критерии отбора кандидатов

Не всё нужно и не всё можно автоматизировать. Вот критерии оценки:

| Критерий | Вопрос | Влияние |
|---------|--------|---------|
| **Частота выполнения** | Как часто запускается этот тест? | Ежедневно → автоматизируй. Раз в квартал → сомнительно. |
| **Стабильность** | Меняется ли часто тестируемая фича? | Стабильная → автоматизируй. Перебирается каждый спринт → подожди. |
| **Сложность ручного выполнения** | Сколько шагов, сколько данных? | 50 шагов → автоматизируй. 3 клика → возможно, не стоит. |
| **Критичность** | Насколько важен этот сценарий для бизнеса? | Оплата → автоматизируй. Tooltip → сомнительно. |
| **Время ручного выполнения** | Сколько минут/часов занимает? | 2 часа → автоматизируй. 30 секунд → не стоит. |
| **Предсказуемость результата** | Можно ли однозначно определить pass/fail? | Чёткий ответ → автоматизируй. "Выглядит нормально" → manual. |
| **Данные** | Нужно много комбинаций данных? | 100 комбинаций → автоматизируй (data-driven). 2 комбинации → manual. |

### Матрица решений

```
                    Высокая частота выполнения
                              │
         АВТОМАТИЗИРУЙ        │        АВТОМАТИЗИРУЙ
          ОБЯЗАТЕЛЬНО         │          ЖЕЛАТЕЛЬНО
         (regression,         │       (нестабильные, но
          smoke, data-driven) │        важные сценарии)
    ──────────────────────────┼──────────────────────────
         АВТОМАТИЗИРУЙ        │        ОСТАВЬ РУЧНЫМ
           ЕСЛИ ЕСТЬ          │       (exploratory,
           РЕСУРСЫ            │        usability, one-time)
                              │
                    Низкая частота выполнения

    Высокая стабильность ←────┼────→ Низкая стабильность
```

### Что автоматизировать: конкретные категории

**Обязательные кандидаты (автоматизируй первыми):**

1. **Regression-тесты** — выполняются каждый спринт/релиз, одинаковые шаги.
2. **Smoke-тесты** — быстрая проверка "живости" системы после деплоя.
3. **Data-driven тесты** — один сценарий, 50+ комбинаций данных.
4. **API-тесты** — быстрые, стабильные, дают высокий coverage.
5. **Cross-browser/cross-device** — одинаковые шаги, разные окружения.

**Хорошие кандидаты (автоматизируй во вторую очередь):**

6. **Performance-тесты** — нагрузку невозможно создать вручную.
7. **Интеграционные проверки** — сервис + БД, сервис + API.
8. **Проверки доступности** — WCAG-compliance (axe-core, Lighthouse).
9. **Security-сканирование** — OWASP ZAP, Dependency Check.

**Оставьте ручным:**

10. **Exploratory testing** — требует творческого мышления.
11. **Usability testing** — "удобно ли?" не определяется скриптом.
12. **Visual testing** (частично) — можно автоматизировать pixel comparison,
    но визуальную оценку делает человек.
13. **One-time testing** — миграция данных, одноразовый импорт.
14. **Тесты для нестабильных фич** — фича меняется каждую неделю →
    автотест будет переписываться чаще, чем запускаться.

### Правило 80/20

**80% ценности автоматизации приносят 20% тестов.** Определите эти 20%:

- Критические бизнес-flow (регистрация, авторизация, оплата, основной use case).
- Самые частые regression-баги (то, что ломается снова и снова).
- Самые трудоёмкие ручные проверки (data-driven, cross-browser).

Начните с них. Остальное — потом, если останутся ресурсы.

## Когда начинать автоматизацию

### Почему НЕ с первого дня

Типичная ошибка: "У нас новый проект, давайте сразу пишем автотесты!"

**Проблемы раннего старта:**
1. **Требования не стабилизировались** — фичи меняются ежедневно, автотесты устаревают.
2. **UI не стабилизировался** — дизайн переделывается, локаторы ломаются.
3. **Нет понимания приоритетов** — непонятно, что автоматизировать в первую очередь.
4. **Нет ручных тест-кейсов** — нечего автоматизировать (автотест ≠ тест-дизайн).
5. **Нет test infrastructure** — нет CI/CD, нет тестовых сред, нет тестовых данных.

### Типичный timeline

```
Неделя 1-2: РУЧНОЕ ТЕСТИРОВАНИЕ
├── Анализ требований
├── Написание тест-кейсов для критических flow
├── Exploratory testing
├── Smoke-чеклист
└── Понимание архитектуры и tech stack

Неделя 3-4: ПОДГОТОВКА К АВТОМАТИЗАЦИИ
├── Выбор инструментов и фреймворка
├── Настройка проекта (Maven/Gradle, зависимости)
├── Первые API-тесты (НЕ UI!)
├── Smoke-suite через API
└── Интеграция с CI (хотя бы ручной запуск)

Месяц 2: ПЕРВЫЕ UI-АВТОТЕСТЫ
├── Page Object Model для 2-3 основных страниц
├── 5-10 критических E2E-сценариев
├── Запуск в CI на каждый merge request
└── Отчёты (Allure)

Месяц 3-4: РАСШИРЕНИЕ
├── Полный regression suite (API)
├── 20-30 UI-сценариев для критических flow
├── Data-driven тесты
├── Запуск по расписанию (nightly)
└── Dashboard с метриками

Месяц 5+: ЗРЕЛОСТЬ
├── Интеграция с deployment pipeline
├── Параллельный запуск
├── Cross-browser testing
├── Performance testing
└── Мониторинг flaky-тестов
```

### Начинайте с API, а не с UI

**Почему API-тесты первые:**

| Критерий | API-тесты | UI-тесты |
|---------|-----------|----------|
| Скорость выполнения | Миллисекунды | Секунды |
| Стабильность | Высокая (контракт меняется редко) | Низкая (UI меняется часто) |
| Покрытие бизнес-логики | Высокое (вся логика на backend) | Опосредованное |
| Сложность написания | Низкая | Средняя-высокая |
| Стоимость поддержки | Низкая | Высокая |
| Debug при падении | Легко (JSON request/response) | Сложно (скриншоты, логи) |

**Практический пример:**

```java
// API-тест: написан за 30 минут, выполняется 200 мс, покрывает бизнес-логику
@Test
void shouldCreateOrderSuccessfully() {
    // Подготовка — создание тестовых данных через API
    var product = apiClient.createProduct(new ProductRequest("Laptop", 999.99));
    var user = apiClient.createUser(new UserRequest("test@example.com"));

    // Действие — создание заказа
    var order = apiClient.createOrder(new OrderRequest(user.getId(), product.getId(), 2));

    // Проверка — полный набор assertion-ов
    assertThat(order.getStatus()).isEqualTo("CREATED");
    assertThat(order.getTotalPrice()).isEqualTo(1999.98);
    assertThat(order.getItems()).hasSize(1);
}

// UI-тест для того же сценария: написан за 2 часа, выполняется 15 секунд,
// ломается при смене дизайна кнопки
```

## ROI автоматизации

### Формула расчёта ROI

```
ROI = (Экономия - Затраты) / Затраты × 100%
```

**Затраты на автоматизацию:**

| Категория | Единоразовые | Постоянные (в месяц) |
|----------|-------------|---------------------|
| Настройка фреймворка | 40-80 часов | — |
| Написание тестов | 2-8 часов / тест | — |
| Поддержка тестов | — | 20-30% от написания |
| Инфраструктура (CI/CD, Selenoid) | 16-40 часов | 5-10 часов |
| Обучение команды | 20-40 часов | 5 часов |

**Экономия от автоматизации:**

| Категория | Экономия (в месяц) |
|----------|-------------------|
| Ручной regression (вместо 2 дней → 1 час) | 14 часов / спринт |
| Раннее обнаружение багов (CI) | 5-20 часов / спринт |
| Cross-browser (ручной → параллельный) | 8-16 часов / спринт |
| Ночной прогон (тесты идут без людей) | Бесценно |

### Break-Even Point (точка безубыточности)

```
Время написания автотеста: 4 часа
Время ручного выполнения: 15 минут
Экономия за запуск: 15 минут

Break-even = 4 часа / 15 минут = 16 запусков

Если тест запускается 2 раза в неделю:
Break-even = 16 / 2 = 8 недель (2 месяца)
```

**Визуализация:**

```
Затраты (часы)
     │
  20 │  ╱ Ручное тестирование (линейный рост)
     │ ╱
  16 │╱─────────╳ Break-even point
     │         ╱│╲
  12 │        ╱ │  ╲ Автоматизация (стабилизируется)
     │       ╱  │
   8 │      ╱   │
     │     ╱    │
   4 │────╱─────│─────────────────── Стоимость поддержки автотестов
     │   ╱      │
   0 ├───┴──────┴────────────────→ Количество запусков
     0    8    16   24   32
```

### Нематериальные выгоды

Не всё можно измерить в часах:

1. **Уверенность при рефакторинге** — разработчики не боятся менять код,
   зная, что regression-тесты поймают проблему.
2. **Быстрый feedback** — баг найден через 10 минут после коммита, а не через 3 дня.
3. **Документация** — автотесты описывают ожидаемое поведение системы.
4. **Onboarding** — новый член команды запускает тесты и видит, как система работает.
5. **Качество релизов** — меньше багов на продакшене → меньше hotfix-ов → меньше стресса.
6. **Масштабируемость** — при росте продукта ручное тестирование не масштабируется
   линейно, а автоматизация — да.

## Пирамида автоматизации на практике

### Рекомендуемое распределение

```
                    ┌─────────┐
                    │ Manual  │ ~10%
                    │Exploratory│
                   ┌┴─────────┴┐
                   │  UI / E2E  │ ~20%
                  ┌┴────────────┴┐
                  │    API Tests   │ ~70%
                  └────────────────┘
```

**API 70%: основа автоматизации**

Почему именно 70%? Потому что:
- Бизнес-логика живёт на backend.
- API-тесты быстрые (100-500ms vs 5-30s для UI).
- API-тесты стабильные (контракт меняется реже, чем UI).
- Легко делать data-driven (один метод, 100 наборов данных).
- Покрывают интеграцию сервисов.

```java
// Пример: один API-тест заменяет 5 UI-тестов
// UI-тест: открыть форму → заполнить поля → нажать Submit → проверить результат
// Нужно 5 отдельных UI-тестов для 5 комбинаций данных

// API: один параметризованный тест
@ParameterizedTest
@CsvSource({
    "valid@email.com,   SecurePass1,  25,  201",  // Валидные данные
    "'',                SecurePass1,  25,  400",  // Пустой email
    "invalid-email,     SecurePass1,  25,  400",  // Невалидный email
    "valid@email.com,   short,        25,  400",  // Короткий пароль
    "valid@email.com,   SecurePass1,  15,  400",  // Возраст < 18
    "valid@email.com,   SecurePass1,  121, 400"   // Возраст > 120
})
void shouldValidateRegistration(String email, String password, int age, int expectedStatus) {
    given()
        .body(new RegistrationRequest(email, password, age))
        .contentType(ContentType.JSON)
    .when()
        .post("/api/register")
    .then()
        .statusCode(expectedStatus);
}
```

**UI/E2E 20%: критические пользовательские сценарии**

Автоматизируйте на UI **только то, что нельзя проверить через API:**
- Полный пользовательский flow (от landing page до оплаты).
- Визуальное отображение (элементы на месте, responsive layout).
- JavaScript-логика на frontend (SPA-routing, dynamic forms).
- Интеграция frontend + backend (data binding, websocket).

```java
// Пример: UI-тест только для full E2E flow
@Test
void shouldCompleteFullPurchaseFlow() {
    // Этот тест проверяет ПОЛНЫЙ пользовательский путь,
    // который невозможно покрыть одним API-вызовом
    loginPage.loginAs("user@example.com", "password");
    catalogPage.searchProduct("Laptop");
    catalogPage.addToCart("Laptop Pro 15");
    cartPage.proceedToCheckout();
    checkoutPage.fillShippingAddress(testAddress);
    checkoutPage.selectPaymentMethod("Credit Card");
    checkoutPage.fillCardDetails(testCard);
    checkoutPage.placeOrder();

    assertThat(confirmationPage.getOrderStatus()).isEqualTo("Confirmed");
    assertThat(confirmationPage.isOrderNumberDisplayed()).isTrue();
}
```

**Manual + Exploratory 10%: то, что нельзя автоматизировать**

- Exploratory sessions (поиск неизвестных багов).
- Usability review (удобно ли?).
- Первичное тестирование новых фич.
- Visual QA (общее впечатление от дизайна).

## Рост зрелости автоматизации

### Stage 1: Базовые скрипты (месяц 1-2)

**Характеристики:**
- Отдельные test-скрипты без общей архитектуры.
- Запуск вручную, локально.
- Нет отчётности (pass/fail в консоли).
- 10-20 тестов (smoke + несколько regression).

**Инструменты:** JUnit + RestAssured (API) или Selenium WebDriver (UI).

**Цель этапа:** доказать ценность автоматизации, получить quick wins.

```java
// Stage 1: простой скрипт без Page Object, без абстракций
@Test
void loginTest() {
    WebDriver driver = new ChromeDriver();
    driver.get("https://app.example.com/login");
    driver.findElement(By.id("email")).sendKeys("user@test.com");
    driver.findElement(By.id("password")).sendKeys("password");
    driver.findElement(By.id("loginBtn")).click();
    assertTrue(driver.findElement(By.id("dashboard")).isDisplayed());
    driver.quit();
}
```

### Stage 2: Фреймворк (месяц 2-4)

**Характеристики:**
- Page Object Model (для UI-тестов).
- Базовые абстракции (API-клиент, test data builder).
- Конфигурация через properties/YAML.
- Отчёты (Allure Report).
- 30-80 тестов.

**Инструменты:** + Allure, TestNG/JUnit 5, Maven/Gradle.

**Цель этапа:** создать поддерживаемую архитектуру.

```java
// Stage 2: Page Object + базовая архитектура
public class LoginPage extends BasePage {
    @FindBy(id = "email")
    private WebElement emailField;

    @FindBy(id = "password")
    private WebElement passwordField;

    @FindBy(id = "loginBtn")
    private WebElement loginButton;

    public DashboardPage loginAs(String email, String password) {
        emailField.sendKeys(email);
        passwordField.sendKeys(password);
        loginButton.click();
        return new DashboardPage(driver);
    }
}
```

### Stage 3: CI/CD-интеграция (месяц 3-6)

**Характеристики:**
- Тесты запускаются автоматически (на каждый merge request, nightly).
- Параллельный запуск (Selenoid, GridRouter).
- Тестовые данные управляются (fixtures, factories).
- Flaky-тесты отслеживаются и исправляются.
- 80-200 тестов.

**Инструменты:** + Jenkins/GitLab CI/GitHub Actions, Selenoid, Docker.

**Цель этапа:** автоматический feedback loop.

```yaml
# Stage 3: GitLab CI pipeline
stages:
  - test

api-tests:
  stage: test
  script:
    - mvn test -pl api-tests -Dgroups=smoke
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"

ui-tests:
  stage: test
  script:
    - mvn test -pl ui-tests -Dgroups=critical
  rules:
    - if: $CI_PIPELINE_SOURCE == "schedule"  # Ночной прогон
  artifacts:
    paths:
      - ui-tests/target/allure-results/
```

### Stage 4: Полный pipeline (месяц 6-12)

**Характеристики:**
- Автотесты — gate для деплоя (не прошли → нет деплоя).
- Разные suite-ы для разных целей (smoke → full regression → performance).
- Тестовые среды создаются и уничтожаются автоматически (infrastructure as code).
- Метрики: coverage, trend, flaky rate, execution time.
- 200-500 тестов.

**Инструменты:** + Kubernetes, Terraform, Grafana.

**Цель этапа:** автоматизация — часть delivery pipeline.

```
Commit → Build → Unit Tests → API Smoke → Deploy to Staging
                                              ↓
                              Full API Regression + UI Critical
                                              ↓
                                    Performance Tests
                                              ↓
                              ✅ All Green → Deploy to Production
                              ❌ Failure → Block + Notify
```

### Stage 5: Продвинутый уровень (12+ месяцев)

**Характеристики:**
- Contract testing (Pact) для микросервисов.
- Visual regression testing (Percy, Applitools).
- Chaos engineering (наблюдаемость под нагрузкой).
- AI-assisted test generation (экспериментально).
- Self-healing тесты (автоматическое обновление локаторов).
- 500+ тестов.

**Цель этапа:** проактивное обеспечение качества.

## Как "продать" автоматизацию менеджменту

### Говорите на языке бизнеса

Менеджмент не интересует "мы напишем Page Object Model с Allure-отчётами".
Менеджмент интересует:

| Ваш язык (технический) | Язык менеджмента (бизнес) |
|------------------------|--------------------------|
| "Напишем автотесты" | "Сократим regression с 3 дней до 2 часов" |
| "Внедрим CI/CD" | "Будем ловить баги до того, как они дойдут до пользователей" |
| "Используем Selenium" | "Сможем тестировать в 5 браузерах параллельно" |
| "Покроем API-тестами" | "Уменьшим количество hotfix-ов на 40%" |
| "Нужен Selenoid-кластер" | "Инвестиция $200/мес, экономия $2000/мес за счёт скорости" |

### Структура предложения

1. **Проблема:** "Сейчас regression занимает 3 дня. Это задерживает каждый релиз."
2. **Решение:** "Автоматизация regression suite с запуском за 2 часа."
3. **Затраты:** "2 месяца работы QA-инженера (160 часов). Инфраструктура: $200/мес."
4. **Ожидаемый результат:** "Через 3 месяца: regression за 2 часа, релизы 2 раза
   в неделю вместо 1 раза в 2 недели."
5. **Метрики успеха:** "Время regression (дни → часы), количество багов на production,
   время до hotfix."
6. **Risk mitigation:** "Начинаем с малого (smoke-тесты за 2 недели),
   показываем результат, потом масштабируем."

### Quick Wins для демонстрации ценности

Не начинайте с "нам нужен 6-месячный проект автоматизации".
Начните с **quick wins**, которые показывают результат через 1-2 недели:

1. **Автоматизируйте smoke-тесты (API)** — 10 тестов за 3 дня.
   Результат: "Теперь после каждого деплоя мы знаем за 30 секунд, что система жива."

2. **Настройте запуск в CI** — 1 день.
   Результат: "Разработчики видят, если их изменение сломало что-то, ещё до merge."

3. **Добавьте Allure-отчёт** — 2 часа.
   Результат: "Вот dashboard с историей прогонов. Вот тренд по стабильности."

После quick wins менеджмент обычно сам спрашивает: "Что ещё можно автоматизировать?"

## Связь с тестированием

Стратегия автоматизации — это **мост** между ручным тестированием и CI/CD pipeline.

- Без ручного тестирования (предыдущий раздел) нечего автоматизировать:
  сначала тест-дизайн, потом автоматизация.
- Без CI/CD (следующие разделы) автоматизация теряет 50% ценности:
  тесты, которые запускаются вручную — это лишь полдела.
- Автоматизация меняет роль QA: от "человека, который кликает" к
  "инженеру, который проектирует и поддерживает систему качества".

## Типичные ошибки

### Ошибка 1: "Автоматизируем всё!"

100% автоматизация — не цель. Exploratory testing, usability, visual QA —
это ручные активности, и они должны ими оставаться. Стремление автоматизировать
ВСЁ приводит к хрупким, медленным и дорогим тестам.

### Ошибка 2: Начинать с UI-тестов

UI-тесты — самые дорогие (по написанию, поддержке, времени выполнения) и самые
хрупкие. Начинайте с API. Переходите к UI только для критических full-flow сценариев.

### Ошибка 3: Не учитывать стоимость поддержки

Написать тест — это 30% затрат. 70% — это поддержка: обновление при изменении фичи,
исправление flaky-тестов, обновление тестовых данных. Бюджетируйте поддержку.

### Ошибка 4: Игнорировать flaky-тесты

Flaky-тест — это тест, который проходит и падает случайным образом.
**5% flaky-тестов = 50% потерянного доверия к автоматизации.** Команда начинает
игнорировать падения ("а, это опять flaky"), и настоящие баги пропускаются.

**Правило:** flaky-тест должен быть починен или удалён в течение 1 спринта.

### Ошибка 5: Нет отчётности

Автотесты запускаются, но результат знает только автор. Нужен dashboard (Allure,
Grafana), видимый всей команде. Без видимости автоматизация "невидима" для
менеджмента, и при сокращении бюджета её уберут первой.

### Ошибка 6: Автоматизация как замена ручного тестирования

"Мы автоматизировали regression, теперь QA не нужен." Автоматизация проверяет
**известные** сценарии. Новые баги находит человек — через exploratory testing,
анализ требований, code review, мониторинг production.

### Ошибка 7: Писать тесты без тест-дизайна

Автотест без продуманного тест-дизайна — это плохой тест, написанный быстро.
Сначала техники тест-дизайна (EP, BVA, decision tables), потом автоматизация.
Автоматизация ускоряет **выполнение**, а не заменяет **проектирование**.

## Вопросы на интервью

### 🟢 Junior

1. **Что бы вы автоматизировали в первую очередь на новом проекте?**
   Smoke-тесты через API — быстрая проверка основных endpoint-ов.
   Они дают максимальную ценность при минимальных затратах.

2. **Чем API-тесты лучше UI-тестов для автоматизации?**
   Быстрее (100ms vs 10-30s), стабильнее (контракт меняется реже),
   проще в написании и поддержке, покрывают бизнес-логику напрямую.

3. **Что такое flaky-тест?**
   Тест, который проходит и падает непредсказуемо при одинаковых условиях.
   Причины: race conditions, зависимость от порядка выполнения,
   нестабильная тестовая среда, hardcoded timeouts.

4. **Что такое пирамида автоматизации?**
   Модель, рекомендующая много API-тестов (основание), меньше UI-тестов (середина),
   минимум manual/exploratory (вершина). Оптимизирует баланс скорости и покрытия.

### 🟡 Middle

5. **Как вы определяете, что автоматизировать, а что оставить ручным?**
   Оцениваю по критериям: частота выполнения, стабильность фичи, сложность
   ручного выполнения, критичность, предсказуемость результата. Если тест
   выполняется часто, фича стабильна, результат детерминирован — автоматизирую.
   Exploratory, usability, one-time тесты — оставляю ручными.

6. **Как рассчитать ROI автоматизации?**
   (Экономия - Затраты) / Затраты × 100%. Учитываю: время написания,
   стоимость поддержки (20-30% от написания ежемесячно), инфраструктуру.
   Экономия: разница времени ручного и автоматического прогона × количество запусков.
   Break-even = время написания / экономия за запуск.

7. **Как бороться с flaky-тестами?**
   Отслеживание (dashboard с flaky rate), приоритизация (чинить в текущем спринте),
   карантин (отдельный suite, не блокирует pipeline), root cause analysis
   (чаще всего: async waits, тестовые данные, порядок выполнения).

8. **Расскажите о стадиях зрелости автоматизации.**
   Stage 1: отдельные скрипты. Stage 2: фреймворк (POM, отчёты). Stage 3: CI/CD
   интеграция. Stage 4: полный pipeline (тесты — gate для деплоя). Stage 5:
   contract testing, visual regression, chaos engineering.

### 🔴 Senior

9. **Как бы вы решили, что автоматизировать на новом проекте? Опишите весь процесс.**
   Неделя 1-2: ручное тестирование для понимания системы. Определяю критические
   бизнес-flow (с PO), текущий цикл regression (сколько времени, кто выполняет),
   tech stack (для выбора инструментов). Составляю матрицу: сценарий × частота ×
   стабильность × критичность → приоритизированный backlog автоматизации.
   Начинаю с API smoke, показываю quick win за 2 недели. Масштабирую
   на основе feedback и метрик.

10. **Менеджмент хочет "100% автоматизацию за 3 месяца". Как вы отвечаете?**
    Объясняю: 100% автоматизация — ложная цель. Правильная цель — "автоматизировать
    критические regression-тесты, чтобы regression занимал 2 часа вместо 3 дней".
    Предлагаю поэтапный план: месяц 1 — API smoke (30 тестов), месяц 2 —
    API regression (100 тестов), месяц 3 — UI critical (20 тестов) + CI pipeline.
    Показываю метрики после каждого этапа. Отдельно бюджетирую поддержку (20% времени).

11. **На проекте 500 автотестов, из них 30% flaky. Ваш план действий?**
    (1) Мораторий на новые тесты до стабилизации. (2) Dashboard: отслеживаю
    flaky rate по тестам. (3) Приоритизация: начинаю с тестов в критическом suite
    (smoke, blocker для pipeline). (4) Root cause analysis: группирую по причинам
    (async waits — 40%, test data — 30%, infrastructure — 20%, actual bugs — 10%).
    (5) Системные исправления: улучшаю wait-стратегию (explicit waits вместо sleep),
    изолирую тестовые данные (каждый тест создаёт свои данные), стабилизирую
    инфраструктуру (dedicated test environment). (6) Карантин: неисправленные
    flaky — в отдельный suite, не блокирует pipeline. (7) KPI: flaky rate < 5%
    за 6 недель.

12. **Как "продать" автоматизацию скептически настроенному менеджменту?**
    Не продаю автоматизацию — продаю решение бизнес-проблемы. "Regression
    занимает 3 дня → релизы раз в 2 недели → конкуренты быстрее." Предлагаю
    pilot: 2 недели, 10 smoke-тестов, 0 дополнительных затрат (делаю в рамках
    текущих задач). Показываю результат: "Вот dashboard, вот 30-секундная проверка
    после деплоя, вот баг, который нашли автоматически." После pilot —
    предложение масштабирования с ROI-расчётом.
