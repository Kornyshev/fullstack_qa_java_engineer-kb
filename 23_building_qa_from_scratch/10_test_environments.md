# Тестовые окружения: настройка и управление

## Обзор

Тестовое окружение — это не просто «сервер, где можно потыкать». Это воспроизводимая конфигурация инфраструктуры,
данных и сервисов, которая позволяет команде QA проверять работу системы в условиях, приближённых к продакшену. Плохо
настроенное окружение — это часы на расследование «у меня локально работает», конфликты между тестировщиками,
невоспроизводимые баги и релизы, которые ломаются на проде, хотя «на staging всё было зелёное».

В этом разделе мы разберём типы окружений, как ими управлять, как решать типичные проблемы и как отвечать на
интервью-вопрос, который задают на каждом втором собеседовании.

## Типы окружений

### Local (локальное окружение)

**Что это:** приложение запущено на машине разработчика или QA-инженера.

**Для чего:** быстрая проверка изменений, отладка, запуск unit и integration тестов.

**Типичный стек:**
```
Вариант 1 (минималистичный):
- Приложение запущено через IDE (IntelliJ IDEA)
- In-memory БД (H2)
- Mock внешних сервисов (WireMock)

Вариант 2 (docker-compose):
- Приложение в Docker
- PostgreSQL в Docker
- Redis в Docker
- Kafka в Docker
- Всё поднимается одной командой: docker-compose up
```

**Пример docker-compose для локального окружения:**
```yaml
# docker-compose.yml — локальное окружение для разработки и тестирования
version: '3.8'

services:
  # Основная база данных
  postgres:
    image: postgres:15
    environment:
      POSTGRES_DB: myapp_dev
      POSTGRES_USER: dev_user
      POSTGRES_PASSWORD: dev_password
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
      - ./sql/init.sql:/docker-entrypoint-initdb.d/init.sql  # Начальная схема

  # Кэш
  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"

  # Брокер сообщений (если нужен)
  kafka:
    image: confluentinc/cp-kafka:7.4.0
    environment:
      KAFKA_BROKER_ID: 1
      KAFKA_ZOOKEEPER_CONNECT: zookeeper:2181
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://localhost:9092
    ports:
      - "9092:9092"
    depends_on:
      - zookeeper

  zookeeper:
    image: confluentinc/cp-zookeeper:7.4.0
    environment:
      ZOOKEEPER_CLIENT_PORT: 2181

volumes:
  postgres_data:
```

**Кто использует:** разработчики, QA-инженеры (для написания и отладки тестов).

**Ключевое правило:** каждый член команды должен уметь поднять локальное окружение за 5 минут. Если это занимает час —
проблема в инфраструктуре, а не в инженере.

### Dev (окружение разработки)

**Что это:** общий сервер, куда автоматически деплоится каждый коммит в `main` (или `develop`).

**Для чего:** ранняя интеграция, быстрая проверка «собирается ли всё вместе», демо для команды.

**Характеристики:**
| Параметр | Значение |
|---|---|
| Стабильность | Низкая (часто ломается из-за свежих коммитов) |
| Данные | Тестовые, не чистятся регулярно |
| Деплой | Автоматический (CI/CD) при каждом коммите |
| Доступ | Вся команда разработки |
| Внешние сервисы | Mock или sandbox |

**Кто использует:** разработчики (проверка интеграции), QA (раннее exploratory testing).

**Типичная проблема:** dev-окружение постоянно сломано, потому что каждый коммит меняет его состояние. Решение: деплоить
только из `main` после прохождения CI.

### QA / Staging (окружение тестирования)

**Что это:** стабильное окружение, максимально приближённое к продакшену, выделенное для тестирования.

**Для чего:** полноценное ручное и автоматизированное тестирование, регрессия, приёмка.

**Характеристики:**
| Параметр | Значение |
|---|---|
| Стабильность | Высокая (деплой только проверенных версий) |
| Данные | Контролируемые, чистятся перед регрессией |
| Деплой | По запросу или по расписанию |
| Доступ | QA-команда, PM, stakeholders |
| Внешние сервисы | Sandbox (тестовые ключи платёжных систем и т.д.) |

**Кто использует:** QA-инженеры (основная работа), PM (приёмка), аналитики (проверка требований).

**Ключевое правило:** QA-окружение — это территория QA. Никто не деплоит туда без согласования. Иначе вы будете
тестировать одну версию, а в середине прогона кто-то задеплоит другую.

### Pre-Prod (предпродакшен)

**Что это:** точная копия продакшена (или максимально близкая) с точки зрения инфраструктуры.

**Для чего:** финальная проверка перед релизом, проверка миграций, нагрузочное тестирование.

**Характеристики:**
| Параметр | Значение |
|---|---|
| Стабильность | Максимальная |
| Данные | Анонимизированная копия продакшен-данных или реалистичный seed |
| Деплой | Только release candidate |
| Доступ | Ограниченный (QA lead, DevOps, release manager) |
| Внешние сервисы | Sandbox (но конфигурация как на проде) |
| Инфраструктура | Идентична продакшену (тот же cloud, те же ресурсы) |

**Кто использует:** QA (smoke после деплоя), DevOps (проверка деплоя и миграций), SRE (нагрузочное тестирование).

**Типичная проблема:** pre-prod отличается от прода (меньше памяти, другая версия ОС, другой cloud provider). Это
обесценивает всё тестирование на этом окружении. Решение: Infrastructure as Code (Terraform, Ansible) для идентичной
конфигурации.

### Production (продакшен)

**Что это:** боевое окружение, где работают реальные пользователи.

**Для чего QA:** smoke-тесты после релиза, мониторинг, canary testing.

**Что QA делает на продакшене:**
```
1. Post-release smoke (5-10 тестов): после деплоя проверяем критичные пути
   - Логин работает
   - Главная страница загружается
   - API health check отвечает 200
   - Основной бизнес-сценарий проходит

2. Мониторинг (пассивный):
   - Alerting на рост 5xx-ошибок
   - Мониторинг response time
   - Отслеживание ошибок через Sentry/Datadog

3. Canary testing (при canary-деплое):
   - Новая версия получает 5% трафика
   - Сравниваем метрики с baseline
   - При деградации — автоматический rollback
```

**Чего QA НИКОГДА не делает на продакшене:**
- Не создаёт тестовых пользователей (засоряет данные)
- Не тестирует деструктивные сценарии (удаление, массовые операции)
- Не запускает полную регрессию

## Управление окружениями для QA

### Запрос и конфигурация окружения

Чеклист при запросе нового окружения:

```
□ Определить назначение (dev / QA / pre-prod)
□ Список сервисов и их версии
□ Базы данных: тип, версия, начальные данные
□ Внешние зависимости: какие, sandbox/mock
□ Сеть: доступы, VPN, firewall-правила
□ Секреты: API-ключи, токены (тестовые!)
□ CI/CD: как будет происходить деплой
□ Мониторинг: логи, метрики, алерты
□ Ответственный за поддержку
□ Процедура восстановления (если окружение упало)
```

### Управление тестовыми данными

Это самая больная тема в тестировании окружений. Вот три подхода:

**1. Seeding (начальное заполнение)**
```sql
-- seed.sql — начальные данные для QA-окружения
-- Пользователи для тестирования разных ролей
INSERT INTO users (id, email, role, name) VALUES
    (1, 'admin@test.com', 'ADMIN', 'Тестовый Админ'),
    (2, 'user@test.com', 'USER', 'Тестовый Пользователь'),
    (3, 'manager@test.com', 'MANAGER', 'Тестовый Менеджер');

-- Товары для тестирования каталога
INSERT INTO products (id, name, price, category, stock) VALUES
    (1, 'Ноутбук', 75000.00, 'electronics', 50),
    (2, 'Мышь', 1500.00, 'electronics', 200),
    (3, 'Книга', 500.00, 'books', 1000);

-- Тестовые заказы в разных статусах
INSERT INTO orders (id, user_id, status, total) VALUES
    (1, 2, 'CREATED', 76500.00),
    (2, 2, 'PAID', 1500.00),
    (3, 2, 'DELIVERED', 500.00),
    (4, 2, 'CANCELLED', 75000.00);
```

**2. Cleanup (очистка после тестов)**
```java
// Вариант 1: очистка в @AfterEach (для интеграционных тестов)
@AfterEach
void cleanup() {
    orderRepository.deleteAll();
    userRepository.deleteAll();
}

// Вариант 2: Transactional rollback (Spring)
@Transactional // Все изменения откатываются после теста
@Test
void createOrder_savesToDatabase() {
    // Данные создаются и удаляются автоматически
}

// Вариант 3: скрипт полной очистки (для QA-окружения)
// cleanup.sh — запускать перед регрессией
// #!/bin/bash
// psql -U qa_user -d qa_db -f truncate_all_tables.sql
// psql -U qa_user -d qa_db -f seed.sql
```

**3. Isolation (изоляция данных)**
```java
// Каждый тест создаёт уникальные данные и не зависит от seed
@Test
void userCanCreateOrder() {
    // Уникальный email для каждого запуска теста
    String uniqueEmail = "user-" + UUID.randomUUID() + "@test.com";

    // Создаём пользователя через API
    UserDto user = apiClient.createUser(uniqueEmail, "password123");

    // Создаём заказ
    OrderDto order = apiClient.createOrder(user.getId(), List.of(productId));

    // Проверяем
    assertThat(order.getStatus()).isEqualTo("CREATED");

    // Cleanup: удаляем созданные данные
    apiClient.deleteOrder(order.getId());
    apiClient.deleteUser(user.getId());
}
```

**Рекомендация:** комбинируйте все три подхода:
- **Seeding** — для справочных данных (роли, категории, статусы), которые нужны всем тестам
- **Isolation** — для данных, которые создаёт каждый тест (пользователи, заказы)
- **Cleanup** — перед полной регрессией (reset до известного состояния)

### Docker-Compose для локального окружения

```yaml

services:
```

### Environment-Specific конфигурация в тестах

```java
// application-test.yml — конфигурация для тестового окружения
// spring:
//   datasource:
//     url: jdbc:postgresql://${DB_HOST:localhost}:${DB_PORT:5432}/${DB_NAME:testdb}
//     username: ${DB_USER:test}
//     password: ${DB_PASSWORD:test}
//   redis:
//     host: ${REDIS_HOST:localhost}
//     port: ${REDIS_PORT:6379}

// Переключение окружения в тестах
@SpringBootTest
@ActiveProfiles("test") // Используем application-test.yml
class OrderServiceTest {
    // ...
}

// Для разных окружений — разные конфигурации
// application-local.yml  — для локальной разработки (H2, без Redis)
// application-test.yml   — для CI (TestContainers)
// application-staging.yml — для staging (реальные адреса сервисов)
```

**Паттерн для работы с несколькими окружениями:**
```java
// BaseTest.java — базовый класс для всех тестов
public abstract class BaseTest {

    protected static final String BASE_URL = System.getProperty(
        "test.base-url",
        "http://localhost:8080"  // По умолчанию — локальное окружение
    );

    protected RequestSpecification baseSpec() {
        return given()
            .baseUri(BASE_URL)
            .contentType(ContentType.JSON)
            .log().ifValidationFails();
    }
}

// Запуск на разных окружениях:
// Локально:    mvn test
// На staging:  mvn test -Dtest.base-url=https://staging.company.com
// На pre-prod: mvn test -Dtest.base-url=https://preprod.company.com
```

## Типичные проблемы и решения

### 1. Stale Data (устаревшие данные)

**Проблема:** тестовая БД содержит данные от предыдущих прогонов. Тест «создать первый заказ» падает, потому что в БД
уже 500 заказов от прошлых запусков.

**Решения:**

| Подход | Когда использовать | Плюсы | Минусы |
|---|---|---|---|
| `@Transactional` + rollback | Unit/integration тесты | Быстро, автоматически | Не работает для HTTP-тестов |
| `@BeforeEach` cleanup | Integration тесты | Явно, предсказуемо | Медленно при больших данных |
| Пересоздание контейнера | CI pipeline | Гарантированно чисто | Медленный старт |
| Уникальные данные (UUID) | Все уровни | Не нужна очистка | Данные накапливаются |

### 2. Environment Down (окружение недоступно)

**Проблема:** staging упал, QA-команда не может работать.

**Решения:**
```
1. Мониторинг окружения:
   - Health check каждые 5 минут (curl /actuator/health)
   - Алерт в Slack при падении
   - Dashboard со статусом всех окружений

2. Runbook для восстановления:
   - Кто ответственный (DevOps / SRE)
   - Как перезапустить: kubectl rollout restart deployment/app
   - Как проверить: curl https://staging.company.com/health
   - Escalation path: если не восстановилось за 30 минут

3. Fallback-план для QA:
   - Переключиться на локальное окружение (docker-compose up)
   - Продолжить тестирование того, что не зависит от staging
   - Обновить тест-план с учётом простоя
```

### 3. Version Mismatch (несоответствие версий)

**Проблема:** на staging задеплоена версия 2.3, а в тестах ожидается поведение версии 2.4. Или: фронтенд обновился,
а бэкенд — нет.

**Решения:**
```
1. Версионирование окружений:
   - Каждый деплой логирует версию: "Deployed: backend v2.3.1, frontend v2.4.0"
   - Dashboard показывает текущие версии всех компонентов

2. Синхронный деплой:
   - Фронтенд и бэкенд деплоятся вместе (mono-repo или coordinated release)
   - Или: backward-compatible API (v1 + v2 одновременно)

3. В тестах:
   - Проверять версию API перед запуском: GET /api/version
   - Пропускать тесты для функций, которые ещё не задеплоены
   - @EnabledIf("apiVersionAtLeast('2.4')")
```

### 4. Shared State (разделяемое состояние)

**Проблема:** два QA-инженера работают на одном staging: один создаёт пользователя `test@mail.com`, другой тоже. Второй
получает ошибку «email уже занят».

**Решения:**

```
1. Изоляция через namespace:
   - QA-1 работает с данными: qa1-user@test.com, qa1-order-001
   - QA-2 работает с данными: qa2-user@test.com, qa2-order-001
   - Каждый инженер использует свой префикс

2. Изоляция через tenant (если приложение мультитенантное):
   - QA-1 работает в tenant "qa-team-1"
   - QA-2 работает в tenant "qa-team-2"

3. Ephemeral environments (эфемерные окружения):
   - Каждый PR/каждый QA получает своё изолированное окружение
   - Поднимается автоматически, удаляется после завершения
   - Технологии: Kubernetes namespaces, Docker-compose per branch

4. Расписание:
   - 09:00-12:00 — QA-1 работает на staging
   - 13:00-17:00 — QA-2 работает на staging
   - Примитивно, но иногда единственный вариант
```

**Пример ephemeral environment:**
```yaml
# CI/CD pipeline: создать окружение для PR
# .github/workflows/pr-environment.yml
name: PR Environment
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  deploy-pr-env:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy PR environment
        run: |
          # Создаём namespace для PR
          kubectl create namespace pr-${{ github.event.number }}
          # Деплоим приложение в этот namespace
          helm install app ./chart \
            --namespace pr-${{ github.event.number }} \
            --set image.tag=${{ github.sha }}
          # Выводим URL окружения
          echo "Environment URL: https://pr-${{ github.event.number }}.dev.company.com"
```

### 5. Зависимость от внешних сервисов

**Проблема:** тесты зависят от внешнего Payment Gateway, который недоступен / rate-limited / нестабилен.

**Решения:**
```
По окружениям:

Local:     WireMock (полный mock, всё контролируемо)
Dev:       WireMock (те же mock-и, что и локально)
QA:        Sandbox Payment Gateway (тестовые ключи)
Pre-Prod:  Sandbox Payment Gateway (тестовые ключи, конфигурация как на проде)
Prod:      Реальный Payment Gateway
```

```java
// WireMock для локального и dev-окружения
@WireMockTest(httpPort = 8089)
class PaymentServiceLocalTest {

    @Test
    void processPayment_success() {
        // Мокаем ответ платёжной системы
        stubFor(post(urlEqualTo("/api/v1/charges"))
            .willReturn(aResponse()
                .withStatus(200)
                .withHeader("Content-Type", "application/json")
                .withBody("""
                    {
                        "chargeId": "ch_123",
                        "status": "succeeded",
                        "amount": 1500
                    }
                """)));

        PaymentResult result = paymentService.charge(1500, "tok_test");
        assertThat(result.getStatus()).isEqualTo("succeeded");
    }

    @Test
    void processPayment_gatewayTimeout() {
        // Мокаем таймаут от платёжной системы
        stubFor(post(urlEqualTo("/api/v1/charges"))
            .willReturn(aResponse()
                .withFixedDelay(30000))); // 30-секундная задержка

        assertThatThrownBy(() -> paymentService.charge(1500, "tok_test"))
            .isInstanceOf(PaymentTimeoutException.class);
    }
}
```

## Связь с тестированием

Тестовые окружения — это **фундамент**, на котором стоит всё тестирование. Без правильно настроенного окружения даже
идеальные тесты бесполезны.

Связь с уровнями тестирования:

| Уровень тестов | Окружение | Зачем именно это окружение |
|---|---|---|
| Unit-тесты | Local (IDE) | Никаких внешних зависимостей, максимальная скорость |
| Integration-тесты | Local (TestContainers) | Реальные зависимости в Docker, изоляция |
| API-тесты | QA / Staging | Полноценный стек, приближённый к проду |
| UI-тесты | QA / Staging | Реальный UI + реальный бэкенд |
| Performance-тесты | Pre-Prod | Инфраструктура идентична проду |
| Smoke-тесты | Production | Проверка после деплоя на реальном окружении |

Типичная ошибка: запускать integration-тесты на staging. Это медленно (сеть), нестабильно (staging может быть занят) и
не изолированно (другие тестировщики могут менять данные). Integration-тесты должны запускаться локально с TestContainers.

## Типичные ошибки

### 1. Одно окружение на всё

Команда использует один staging-сервер для ручного тестирования, автоматизированных тестов, демо и нагрузочного
тестирования. Результат: тесты падают, потому что кто-то запустил нагрузочный прогон. QA не может работать, потому
что разработчик задеплоил ломающие изменения.

**Решение:** минимум 2 окружения: dev (нестабильный, для разработки) и QA (стабильный, для тестирования).

### 2. Ручная настройка окружения

«Серёга настроил staging полгода назад, и никто не знает, как он это сделал. Серёга уволился.»

**Решение:** Infrastructure as Code. Всё в Git: Terraform для инфраструктуры, Ansible/Docker для конфигурации.
Окружение должно пересоздаваться одной командой.

### 3. Тестовые данные на проде

QA-инженер создал тестового пользователя `test@test.com` на продакшене для проверки. Через год маркетинг отправил email
всем пользователям — включая `test@test.com`. Клиент получил рассылку и пожаловался.

**Решение:** никогда не создавать тестовые данные на продакшене. Для post-release smoke использовать feature flags или
специальный тестовый аккаунт с меткой `is_test=true`, который исключается из всех рассылок и аналитики.

### 4. Отсутствие мониторинга окружения

Staging упал в пятницу вечером. Узнали в понедельник утром, когда QA пришёл на работу и обнаружил, что ничего не
работает. Потеряли полдня.

**Решение:** health check + алерт в Slack/Telegram. Простейший вариант: cron-задача `curl -f https://staging/health ||
notify-slack`.

### 5. Различия между окружениями

На staging — PostgreSQL 14, на проде — PostgreSQL 15. На staging — 2 GB RAM, на проде — 16 GB. Баг воспроизводится
только на проде из-за различий в конфигурации.

**Решение:** Docker-образы для всех окружений идентичны. Различия — только в конфигурации (через environment variables).
Инфраструктура — через Terraform с модулями для каждого окружения.

### 6. Отсутствие процедуры восстановления

Окружение сломалось — и никто не знает, как его починить. Ждём DevOps-инженера, который в отпуске.

**Решение:** runbook (инструкция по восстановлению) для каждого окружения. Минимум: как перезапустить, как проверить
статус, к кому обращаться.

## Вопросы на интервью

### 🟢 Junior

1. **Какие типы тестовых окружений вы знаете?**
   Local (на машине разработчика), Dev (общий для команды), QA/Staging (для тестирования), Pre-Prod (копия прода),
   Production. Каждое имеет свою цель: local для быстрой проверки, QA для полноценного тестирования, pre-prod для
   финальной проверки.

2. **Чем staging отличается от production?**
   Staging — тестовое окружение, приближённое к проду, но с тестовыми данными и sandbox-интеграциями. На staging можно
   ломать и экспериментировать. Production — боевое окружение с реальными пользователями, где нужна максимальная
   осторожность.

3. **Как вы управляете тестовыми данными?**
   Три подхода: seeding (начальные данные через SQL-скрипт), cleanup (очистка после тестов), isolation (уникальные
   данные для каждого теста через UUID). Комбинирую все три.

### 🟡 Middle

4. **Как вы управляете тестовыми окружениями на вашем проекте? (Классический вопрос!)**

   Пример ответа:
   «На моём проекте мы использовали 4 окружения: local, dev, staging и pre-prod.

   **Local:** docker-compose с PostgreSQL, Redis и WireMock для внешних сервисов. Поднималось за 2 минуты. Здесь
   разработчики и я запускали unit и integration тесты.

   **Dev:** автодеплой из ветки develop. Нестабильное, использовал для раннего exploratory testing. Мониторинг через
   Grafana.

   **Staging:** деплой по запросу, только проверенные версии. Основное окружение для ручного и автоматизированного
   тестирования. Перед каждой регрессией запускал скрипт очистки + seeding. Для автотестов — изоляция через UUID.
   Договорились с командой: деплой только после уведомления в Slack.

   **Pre-prod:** точная копия прода (тот же Terraform-модуль). Использовали для проверки миграций и нагрузочного
   тестирования. Данные — анонимизированная копия прода.

   Главная проблема — конфликты на staging (два QA работают одновременно). Решили внедрением ephemeral environments:
   каждый PR получал своё окружение через Kubernetes namespace.»

5. **Как решаете проблему тестовых данных при параллельной работе нескольких QA на одном окружении?**
   Изоляция через уникальные данные (UUID-префиксы). Каждый тестировщик и каждый прогон автотестов создаёт данные с
   уникальными идентификаторами. Не полагаемся на предзаполненные данные. Если позволяет инфраструктура — ephemeral
   environments.

6. **Что такое ephemeral environments и зачем они нужны?**
   Временные окружения, которые создаются автоматически (обычно для PR или фича-ветки) и удаляются после завершения.
   Решают проблему конфликтов на shared-окружениях. Реализуются через Kubernetes namespaces или Docker-compose per
   branch. Минус — стоимость инфраструктуры.

### 🔴 Senior

7. **Как бы вы спроектировали стратегию тестовых окружений для проекта с 10 микросервисами и командой из 20 человек?**
   Local: docker-compose с основными сервисами + WireMock для остальных. Dev: автодеплой всех сервисов, GitOps
   (ArgoCD). Staging: стабильное окружение, деплой по запросу. Для параллельной работы — ephemeral environments на
   Kubernetes (namespace per PR для критичных сервисов). Pre-prod: Terraform-модуль идентичный проду. Мониторинг всех
   окружений через Grafana + алерты. Тестовые данные: seed для справочников, UUID-изоляция для всего остального.
   TestContainers для CI.

8. **Как обеспечить, что баг, найденный на staging, воспроизведётся на проде?**
   Infrastructure as Code: все окружения создаются из одних Terraform-модулей, различия только в переменных (размер
   инстансов, количество реплик). Docker-образы идентичные во всех окружениях. Конфигурация через environment
   variables. Анонимизированные данные прода на pre-prod. Мониторинг различий: автоматическая проверка версий
   зависимостей и конфигураций между окружениями.

9. **Расскажите о самой сложной проблеме с тестовым окружением, которую вы решали.**
   (Здесь ожидается реальная история. Пример: «Staging падал каждые 2 дня из-за утечки памяти, которая проявлялась
   только при накоплении тестовых данных. Решение: настроил автоматическое пересоздание окружения каждую ночь через
   Kubernetes CronJob + Terraform. Плюс добавил мониторинг использования памяти с алертом при 80%.»)

## Практические задания

### Задание 1: Проектирование окружений
Дан проект: веб-приложение (React + Spring Boot + PostgreSQL + Redis), команда 8 человек (3 разработчика, 2 QA, PM,
DevOps, аналитик). Спроектируйте набор тестовых окружений: какие окружения нужны, для чего каждое, кто использует,
как деплоится, какие данные.

### Задание 2: Docker-Compose
Напишите `docker-compose.yml` для локального окружения: Spring Boot приложение, PostgreSQL, Redis. Добавьте: health
checks, начальные данные (SQL-скрипт), environment variables. Проверьте, что поднимается командой `docker-compose up`.

### Задание 3: Тестовые данные
Опишите стратегию управления тестовыми данными для QA-окружения с 5 микросервисами. Для каждого сервиса: какие seed-данные
нужны, как очищать, как изолировать при параллельной работе двух QA.

### Задание 4: Runbook
Напишите runbook для восстановления staging-окружения. Включите: проверку статуса, типичные проблемы и решения,
escalation path, контакты ответственных. Формат: markdown с чеклистами.

### Задание 5: Environment-Specific тесты
Напишите базовый класс для тестов, который поддерживает запуск на разных окружениях (local, staging, pre-prod) через
параметр командной строки. Включите: различную конфигурацию URL, авторизации и тестовых данных для каждого окружения.
