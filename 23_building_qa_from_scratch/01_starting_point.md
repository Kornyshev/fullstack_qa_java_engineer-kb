# С чего начать: первые шаги QA на новом проекте

## Обзор

Ты пришёл на проект. Ты — первый (или единственный) QA. Тестирования нет вообще, либо оно в зачаточном состоянии.
Никто толком не понимает, что ты будешь делать. Разработчики привыкли жить без тестировщика. Менеджер ожидает,
что ты «просто потестишь и найдёшь все баги». У тебя нет ни документации, ни test cases, ни автотестов, ни даже
внятного описания продукта.

Это — одна из самых сложных и одновременно самых интересных ситуаций в карьере QA-инженера. Именно здесь
проявляется разница между «тестировщиком, который выполняет чужие test cases» и «QA-инженером, который
выстраивает процесс обеспечения качества». Эта глава — про второе.

Мы разберём пошагово, что делать в первый день, первую неделю и первый месяц. Без абстракций — конкретные
действия, конкретные вопросы, конкретные артефакты.

---

## День первый: ориентация на местности

### Что НЕ надо делать в первый день

Прежде чем говорить, что делать — скажем, чего делать категорически не стоит:

- **Не пытайся автоматизировать.** У тебя нет понимания продукта, архитектуры, приоритетов. Автоматизация без контекста — это трата времени на автоматизацию не тех вещей.
- **Не критикуй существующие процессы.** Даже если тебе кажется, что всё ужасно — люди работали так до тебя, и у них были причины. Сначала пойми, потом предлагай.
- **Не обещай конкретных сроков.** «Через две недели у нас будет полное покрытие автотестами» — такого не будет. Не создавай ложных ожиданий.
- **Не замыкайся.** Если ты сидишь молча, читаешь документацию и ни с кем не общаешься — ты теряешь самый ценный ресурс: знания команды.

### Что делать в первый день

**1. Познакомься с командой лично**

Не через Slack, не через email — лично (или по видеосвязи). Каждый разработчик, PM, дизайнер, DevOps — все, с кем ты будешь работать. Спроси у каждого:

- Чем ты занимаешься на проекте?
- Что тебя больше всего бесит в текущем процессе?
- Где, по-твоему, самые большие проблемы с качеством?
- Что бы ты хотел от QA?

Записывай ответы. Они дадут тебе карту «боли» команды, которая важнее любой документации.

**2. Получи доступы**

Составь чек-лист и пройдись по нему в первый же день:

```
# Чек-лист доступов для нового QA
- [ ] Репозиторий(ии) проекта (GitHub / GitLab / Bitbucket)
- [ ] Task tracker (Jira / YouTrack / Linear / etc.)
- [ ] CI/CD система (Jenkins / GitLab CI / GitHub Actions)
- [ ] Тестовые окружения (dev, staging, QA)
- [ ] Базы данных (хотя бы read-only на staging)
- [ ] Логи приложения (ELK / Grafana / CloudWatch)
- [ ] Мессенджер команды (Slack / Teams / etc.)
- [ ] Confluence / Wiki / Notion (документация)
- [ ] Figma / дизайн-макеты
- [ ] Почтовый ящик для тестовых уведомлений
- [ ] VPN (если нужен)
- [ ] Учётные записи в сторонних сервисах (если есть интеграции)
```

**3. Изучи продукт как пользователь**

Открой приложение и пройди основные пользовательские сценарии. Не тестируй — просто пользуйся. Пойми:

- Что делает продукт?
- Кто им пользуется?
- Какие основные функции?
- Что выглядит стабильным, а что — шатким?

Записывай первые впечатления. «Взгляд новичка» — это ценный ресурс, который быстро исчезает.

## Discovery-фаза: понимание продукта и контекста

### Ключевые вопросы, которые нужно задать

Это не просто список для галочки — это фундамент для всех дальнейших решений по тестированию. Каждый вопрос
имеет конкретную цель.

#### Про продукт

| Вопрос | Зачем нужен ответ |
|--------|-------------------|
| Что делает продукт? Какую проблему решает? | Определяет критичные user flows |
| Кто пользователи? Сколько их? | Определяет нагрузку и приоритеты UX |
| Какие функции используются чаще всего? | Приоритизация тестирования |
| Какой SLA / uptime обещан клиентам? | Требования к reliability testing |
| Есть ли сезонность нагрузки? | Планирование performance testing |
| Какие были самые серьёзные инциденты? | Фокус на уязвимые места |

#### Про архитектуру

| Вопрос | Зачем нужен ответ |
|--------|-------------------|
| Monolith или microservices? | Определяет стратегию integration testing |
| Какой tech stack? (Backend, frontend, DB, cache) | Определяет инструменты тестирования |
| Где хостится? (Cloud provider, on-premise) | Влияет на test environments |
| Какие внешние интеграции? | Определяет необходимость mock/stub |
| Есть ли API? REST, GraphQL, gRPC? | Определяет подход к API testing |
| Как устроена аутентификация? | Критично для security testing |

#### Про процессы

| Вопрос | Зачем нужен ответ |
|--------|-------------------|
| Как часто релизы? | Определяет cadence тестирования |
| Есть ли CI/CD? | Определяет возможность автоматизации |
| Какой процесс разработки? (Scrum, Kanban, etc.) | Определяет как встроить QA в процесс |
| Как сейчас выявляются баги? | Понимание текущего уровня качества |
| Есть ли code review? | Индикатор culture of quality |
| Кто решает «релизить или нет»? | Определяет decision-making процесс |

#### Про существующее тестирование

| Вопрос | Зачем нужен ответ |
|--------|-------------------|
| Есть ли unit tests? Какой coverage? | Базовая линия |
| Есть ли автоматизированные тесты? | Что можно переиспользовать |
| Есть ли test cases / checklists? | Что можно взять за основу |
| Кто тестировал раньше? Разработчики? PM? | Понимание, к кому обращаться за знаниями |
| Есть ли баг-трекер с историей? | Анализ паттернов дефектов |

### Как задавать эти вопросы

Не надо приходить на встречу со списком из 30 вопросов — это допрос, а не разговор. Лучше:

1. **Поговори с PM/PO** — про продукт, пользователей, бизнес
2. **Поговори с Tech Lead** — про архитектуру, tech stack, инфраструктуру
3. **Поговори с DevOps** — про CI/CD, environments, деплой
4. **Поговори с разработчиками** — про код, тесты, проблемные места
5. **Поговори с Support** — про самые частые баги, жалобы пользователей

## Картирование системы

### Зачем рисовать архитектуру

Карта системы — это твой главный инструмент для принятия решений: что тестировать, на каком уровне, с какими
приоритетами. Без неё ты тестируешь вслепую.

### Что должно быть на карте

```
# Пример структуры карты системы (текстовое описание)

## Компоненты:
1. Frontend (React SPA)
2. API Gateway (nginx)
3. Auth Service (Spring Boot)
4. User Service (Spring Boot)
5. Payment Service (Spring Boot)
6. Notification Service (Node.js)
7. PostgreSQL (основная БД)
8. Redis (кеш + сессии)
9. RabbitMQ (очередь сообщений)
10. Stripe API (внешняя интеграция — платежи)
11. SendGrid (внешняя интеграция — email)

## Связи:
- Frontend → API Gateway → все сервисы
- User Service → PostgreSQL
- Payment Service → PostgreSQL + Stripe API
- Auth Service → Redis + PostgreSQL
- Notification Service → RabbitMQ + SendGrid
- Payment Service → RabbitMQ → Notification Service

## Критичные потоки данных:
1. Регистрация: Frontend → Auth → User Service → PostgreSQL
2. Оплата: Frontend → Payment → Stripe → RabbitMQ → Notification
3. Логин: Frontend → Auth → Redis (сессия)
```

### Как рисовать

Используй любой инструмент, но рисуй обязательно:

- **draw.io / diagrams.net** — бесплатно, в браузере, можно хранить в репозитории
- **Miro / FigJam** — для совместной работы
- **PlantUML** — если предпочитаешь описывать диаграммы кодом
- **Бумага + ручка** — если быстро набрасываешь в разговоре

Покажи результат Tech Lead и попроси проверить. Это одновременно валидация и демонстрация того, что ты вникаешь.

## Определение scope тестирования

### Что МОЖНО тестировать vs. что НУЖНО тестировать первым

Типичная ошибка новичка — пытаться покрыть тестированием всё сразу. На реальном проекте ресурсы ограничены,
и нужно выбирать.

**Матрица приоритизации:**

```
                    Высокая вероятность бага
                    │
         ┌──────────┼──────────┐
         │  Средний  │  ВЫСШИЙ  │
         │ приоритет │ приоритет │
         │          │          │
─────────┼──────────┼──────────┼─────────
         │  Низкий  │  Средний  │
         │ приоритет │ приоритет │
         │          │          │
         └──────────┼──────────┘
                    │
                    Низкая вероятность бага

    Низкий                    Высокий
    бизнес-impact             бизнес-impact
```

**Что тестировать первым (неделя 1-2):**

1. **Core business flows** — то, за что пользователи платят деньги
2. **Регистрация / логин** — если пользователь не может войти, остальное неважно
3. **Платежи** (если есть) — деньги = максимальный риск
4. **Самые частые баги** (спроси у Support)
5. **То, что релизится прямо сейчас** — текущий sprint/release

**Что можно отложить:**

- Admin-панель (меньше пользователей, ниже риск)
- Edge cases в некритичных функциях
- Performance testing (если нет явных проблем)
- Кросс-браузерное тестирование (начни с основного браузера)

## Quick wins: немедленная ценность

### Что ты можешь сделать СЕГОДНЯ

Быстрые победы критически важны для установления доверия. Вот конкретные действия, которые дают ценность сразу:

**1. Smoke test основного flow**

Пройди основной пользовательский сценарий и задокументируй результат. Даже если всё работает — это ценная
информация: «основной flow проверен, работает».

**2. Exploratory testing session (60-90 минут)**

Проведи структурированную exploratory session на самой «горячей» части продукта. Используй charter:

```
# Пример Exploratory Testing Charter
Цель: Исследовать процесс оформления заказа
Область: Корзина → Оформление → Подтверждение
Время: 90 минут
Фокус: Граничные случаи и обработка ошибок
Записывать: Баги, вопросы, идеи для тестов
```

**3. Найди и оформи 2-3 бага**

Ничто не доказывает ценность QA быстрее, чем найденные баги. Особенно если это реальные проблемы, которые
команда не замечала. Оформи их качественно — с шагами воспроизведения, ожидаемым/фактическим результатом,
скриншотами.

**4. Задокументируй знания**

То, что ты узнал за день — запиши. Создай страницу в Wiki: «QA: текущее состояние тестирования на проекте».
Это демонстрирует системный подход и создаёт основу для дальнейшей работы.

**5. Настрой базовый мониторинг качества**

Если есть доступ к bug tracker — посмотри статистику: сколько багов открыто, какой severity, какие
компоненты чаще ломаются. Подготовь краткий отчёт.

## Первая неделя: выстраивание фундамента

### День 2-3: Углубление в продукт

- Пройди ВСЕ основные user flows, не только happy path
- Начни вести список test ideas — всё, что приходит в голову при использовании продукта
- Изучи историю багов в tracker — ищи паттерны (какие компоненты чаще ломаются, какие типы багов повторяются)
- Прочитай документацию (если есть) — API docs, README, architecture decision records

### День 3-4: Процессы и инструменты

- Пойми текущий workflow разработки от задачи до production
- Разберись с CI/CD pipeline — что запускается, что проверяется автоматически
- Определи, какие test environments доступны и как они настроены
- Начни формировать мнение о том, какие инструменты тебе понадобятся

### День 4-5: Первые артефакты

- Напиши high-level test plan (черновик) — хотя бы на одну страницу
- Создай smoke test checklist для основных функций
- Подготовь proposal: что ты планируешь делать в ближайший месяц
- Презентуй команде свои наблюдения и план (даже если он предварительный)

## Коммуникация с командой

### Как объяснить роль QA

Многие разработчики воспринимают QA как «того, кто ищет баги в моём коде». Это ограниченное понимание.
Твоя задача — перенастроить ожидания:

**Что говорить:**
- «Моя цель — не ловить ваши ошибки, а помогать предотвращать их»
- «Я хочу понять продукт так же хорошо, как вы, чтобы задавать правильные вопросы на этапе разработки»
- «QA — это не этап в конце, а часть всего процесса»
- «Я буду участвовать в планировании, чтобы помочь выявить риски заранее»

**Что НЕ говорить:**
- «Я буду проверять ваш код» (звучит как контроль)
- «Без моего одобрения ничего не релизим» (звучит как блокировка)
- «Раньше вы делали всё неправильно» (звучит как критика)

### Установление процессов

Не пытайся внедрить всё сразу. Начни с минимума:

1. **Неделя 1-2:** Договорись, что ты смотришь задачи перед релизом (даже неформально)
2. **Неделя 2-4:** Внедри review требований (участие QA в grooming / refinement)
3. **Месяц 2:** Формализуй Definition of Done с учётом тестирования
4. **Месяц 2-3:** Внедри bug triage процесс

Постепенность — ключ. Каждое изменение должно показать ценность, прежде чем добавляешь следующее.

## Первый месяц: от хаоса к системе

### Неделя 1-2: Понимание и быстрые победы

- Завершить discovery-фазу
- Создать карту системы
- Провести 3-5 exploratory testing sessions
- Найти и задокументировать первые баги
- Создать smoke test checklist

### Неделя 2-3: Стратегия

- Написать драфт test strategy (подробнее — в следующем файле)
- Определить приоритеты тестирования
- Выбрать инструменты
- Начать участвовать в sprint ceremonies (planning, grooming, retro)

### Неделя 3-4: Систематизация

- Создать первый test plan для текущего спринта
- Начать вести test cases / checklists для критичных функций
- Настроить reporting (даже простой — «сколько тестов прошло, сколько багов нашли»)
- Подготовить и презентовать «Состояние качества» — первый QA-отчёт

### Метрики первого месяца

Что можно показать по итогам первого месяца:

```
# Пример отчёта «QA: итоги первого месяца»

## Что сделано:
- Проведено N exploratory sessions, покрывающих X% основных функций
- Найдено M багов (из них K критичных)
- Создан smoke test checklist (N проверок)
- Составлен драфт test strategy
- Настроено участие QA в sprint ceremonies

## Текущее состояние качества:
- Основные risk areas: [список]
- Стабильные области: [список]
- Открытых багов: N (critical: X, major: Y, minor: Z)

## План на следующий месяц:
- Расширить test coverage на [области]
- Начать автоматизацию smoke tests
- Внедрить regression testing процесс
```

## Связь с тестированием

Всё, что описано выше, — это и есть тестирование в его полном смысле. Не «проверка кнопок», а системная
работа по обеспечению качества.

Ключевые связи:

- **Exploratory testing** — основной инструмент на начальном этапе (когда нет документации и автотестов)
- **Risk-based testing** — приоритизация того, что тестировать первым
- **Smoke testing** — первый уровень автоматизации, который даёт быструю обратную связь
- **Shift-left testing** — участие QA в requirements review, design review, code review
- **Test strategy / test plan** — формализация подхода (подробно — в следующих файлах)

Первые шаги QA на проекте определяют, каким будет тестирование на месяцы вперёд. Если ты не потратишь время
на discovery — все твои дальнейшие тесты будут построены на неправильных предположениях.

## Типичные ошибки

### 1. «Автоматизация с первого дня»
**Проблема:** QA-инженер приходит на проект и сразу начинает писать Selenium-тесты, не понимая, что именно
тестировать и какие сценарии приоритетны.
**Результат:** Через месяц — 50 автотестов, из которых 40 тестируют неважные сценарии, а критичные flows
не покрыты.
**Решение:** Сначала discovery, потом strategy, потом ручное тестирование, потом автоматизация.

### 2. «Работа в изоляции»
**Проблема:** QA сидит тихо, пишет test cases, ни с кем не общается.
**Результат:** Test cases не покрывают реальные сценарии, команда не понимает, чем QA занимается, доверие
не формируется.
**Решение:** Общайся каждый день. Задавай вопросы. Показывай промежуточные результаты.

### 3. «Перфекционизм в документации»
**Проблема:** QA тратит две недели на написание идеального test plan на 50 страниц.
**Результат:** К моменту завершения документа продукт уже изменился, а QA так и не начал тестировать.
**Решение:** Начни с одностраничного черновика. Итерируй.

### 4. «Непонимание бизнес-контекста»
**Проблема:** QA тестирует технические аспекты, игнорируя бизнес-логику.
**Результат:** Технически всё работает, но пользователи не могут выполнить основной сценарий, потому что
бизнес-логика реализована некорректно.
**Решение:** Всегда начинай с вопроса «зачем эта функция пользователю?».

### 5. «Сразу менять процессы»
**Проблема:** QA приходит и говорит: «У вас всё неправильно, надо делать так».
**Результат:** Сопротивление команды, конфликты, потеря доверия.
**Решение:** Сначала пойми, почему процессы такие, какие они есть. Потом предлагай улучшения постепенно,
с доказательствами ценности.

### 6. «Игнорирование существующих знаний»
**Проблема:** QA не спрашивает команду о проблемных местах и начинает искать баги с нуля.
**Результат:** Тратит время на то, что команда и так знает, упускает неочевидные проблемы.
**Решение:** Первым делом — опроси всех: «Где, по-твоему, самые большие проблемы?».

## Вопросы на интервью

### 🟢 Junior

1. **Ты пришёл на проект как первый QA. Что ты сделаешь в первый день?**
   *Ожидаемый ответ:* Познакомлюсь с командой, получу доступы, попробую продукт как пользователь, начну задавать вопросы о продукте и процессах.

2. **Какие вопросы ты задашь команде в первую очередь?**
   *Ожидаемый ответ:* Что делает продукт? Кто пользователи? Какой tech stack? Как часто релизы? Какие самые частые баги? Есть ли существующие тесты?

3. **Что такое smoke testing и почему оно важно на старте?**
   *Ожидаемый ответ:* Быстрая проверка основных функций — работает ли система в принципе. На старте важно, потому что даёт моментальную ценность и baseline.

4. **Что такое exploratory testing и когда оно особенно полезно?**
   *Ожидаемый ответ:* Тестирование без заранее написанных test cases, основанное на знании продукта и интуиции. Особенно полезно, когда нет документации и ты только изучаешь систему.

### 🟡 Middle

5. **Как ты определишь, что тестировать в первую очередь на новом проекте?**
   *Ожидаемый ответ:* Через risk-based подход — приоритизирую по бизнес-критичности и вероятности дефектов. Сначала core flows, потом расширяю coverage. Учитываю историю багов и мнение команды.

6. **Как ты выстроишь коммуникацию с командой разработчиков, которая раньше не работала с QA?**
   *Ожидаемый ответ:* Объясню, что QA — это не контроль, а помощь. Начну с неформального участия, покажу быструю ценность (найду реальные баги), постепенно интегрируюсь в процесс. Не буду навязывать изменения — предложу и покажу выгоду.

7. **Опиши, как бы ты составил карту системы для нового проекта.**
   *Ожидаемый ответ:* Поговорю с Tech Lead, изучу репозитории и конфигурацию CI/CD, нарисую диаграмму с компонентами и связями, обозначу внешние интеграции, определю критичные потоки данных, провалидирую с командой.

8. **Как измерить успешность своей работы по итогам первого месяца?**
   *Ожидаемый ответ:* Количество найденных и исправленных багов, покрытие основных flows тестами/чеклистами, наличие test strategy, интеграция в процесс разработки, обратная связь от команды.

### 🔴 Senior / Lead

9. **Тебя наняли как QA Lead на проект с 5 разработчиками. Тестирования нет вообще. Как ты построишь QA-процесс за 3 месяца?**
   *Ожидаемый ответ:* Детальный план по месяцам: месяц 1 — discovery, exploratory testing, smoke checklist, первый draft стратегии; месяц 2 — формализация процессов, начало автоматизации критичных smoke tests, внедрение QA в SDLC; месяц 3 — расширение автоматизации, regression suite, метрики, возможно найм второго QA. Плюс конкретные метрики для каждого этапа.

10. **Как ты определишь, нужна ли тебе помощь (дополнительные QA) и как обоснуешь это перед менеджментом?**
    *Ожидаемый ответ:* Проанализирую объём тестирования vs доступное время. Покажу метрики: время на регресс, coverage gap, количество багов в production. Подготовлю ROI-расчёт: стоимость QA vs стоимость багов на production. Предложу варианты: найм, аутсорс, shift-left (разработчики пишут больше unit tests).

11. **Как ты балансируешь между «делать правильно» и «делать быстро» при построении QA с нуля?**
    *Ожидаемый ответ:* Прагматичный подход: идеальный процесс — враг работающего. Начинаю с минимального viable QA процесса и итерирую. Принцип: сначала value, потом polish. Если надо выбирать между 10 идеальными test cases и 50 базовых проверок — выбираю 50 базовых, потому что coverage важнее глубины на старте.

12. **Расскажи о ситуации, когда ты приходил на проект без тестирования и что ты делал. Какие ошибки совершил?**
    *Ожидаемый ответ:* Реальный опыт с конкретными примерами, честное признание ошибок (например, «пытался автоматизировать слишком рано» или «не уделил достаточно времени пониманию бизнеса»), выводы и что бы сделал иначе.

## Практические задания

### Задание 1: Discovery-интервью (Junior)
Представь, что ты пришёл на проект — интернет-магазин одежды. Напиши список из 20+ вопросов, которые ты
задашь команде в первые 2 дня. Раздели по категориям (продукт, архитектура, процессы, существующее тестирование).
Для каждого вопроса объясни, зачем тебе этот ответ.

### Задание 2: Карта системы (Middle)
Тебе дали доступ к GitHub-репозиторию микросервисного приложения. Изучи `docker-compose.yml`, `README.md`,
структуру сервисов. Нарисуй:
- Диаграмму компонентов с указанием связей
- Список критичных потоков данных
- Топ-5 рисковых точек (где наиболее вероятны проблемы)

Используй для практики любой open-source проект (например, microservices-demo от Google).

### Задание 3: План первого месяца (Middle)
Составь детальный план работы QA на первый месяц для одного из сценариев:
- a) Стартап, 3 разработчика, MVP web-приложения
- b) Банковское приложение, 15 разработчиков, legacy монолит
- c) Mobile приложение, 5 разработчиков, еженедельные релизы

Для каждого укажи: приоритеты, инструменты, процессы, ожидаемые результаты.

### Задание 4: Quick wins (Junior)
Выбери любое публичное web-приложение (например, Wikipedia, GitHub, Trello). Проведи 60-минутную
exploratory testing session. Задокументируй:
- Charter (цель, область, фокус)
- Найденные баги или потенциальные проблемы (минимум 5)
- Вопросы, которые возникли
- Идеи для test cases

### Задание 5: Коммуникация (Senior)
Напиши текст сообщения (максимум 500 слов) для всей команды, в котором ты:
- Представляешь себя как нового QA
- Объясняешь свою роль и подход
- Описываешь план на первые 2 недели
- Приглашаешь к сотрудничеству

Тон должен быть профессиональным, дружелюбным и не навязчивым.

## Дополнительные ресурсы

### Книги
- **"Lessons Learned in Software Testing"** — Cem Kaner, James Bach, Bret Pettichord. Бесценная книга с практическими уроками. Особенно главы про начало работы на проекте.
- **"Explore It!"** — Elisabeth Hendrickson. Лучшая книга про exploratory testing, критически важный навык для первых дней на проекте.
- **"Agile Testing"** — Lisa Crispin, Janet Gregory. Как встроить тестирование в agile-процесс.
- **"The Art of Software Testing"** — Glenford Myers. Классика, полезна для формирования фундаментального мышления.

### Статьи и блоги
- **Ministry of Testing** (ministryoftesting.com) — крупнейшее QA-сообщество, много материалов про onboarding QA
- **"The First 90 Days as a QA Lead"** — серия статей на Medium
- **Google Testing Blog** (testing.googleblog.com) — практики тестирования в Google

### Шаблоны
- **Test Strategy Template** — подробно разбираем в следующем файле (02_test_strategy.md)
- **Test Plan Template** — подробно разбираем в файле 03_test_plan.md
- **Exploratory Testing Charter Template** — пример приведён выше в этом файле

### Инструменты для начала
- **Jira / YouTrack** — task tracking и bug tracking
- **Confluence / Notion** — документация
- **draw.io / Miro** — диаграммы и mind maps
- **Charles Proxy / Fiddler** — анализ сетевого трафика
- **Postman** — API testing (быстрый старт)
- **Browser DevTools** — незаменимый инструмент для web-тестирования

> **Ключевой вывод:** Первые шаги QA на проекте — это не про тестирование. Это про понимание. Пойми продукт,
> пойми команду, пойми риски — и только тогда начинай тестировать. Тот, кто бросается тестировать не разобравшись,
> тестирует не то, не так и не в том порядке.
