# Тест-план: детальное планирование тестирования

## Обзор

Test plan — это конкретный документ, который отвечает на вопрос: **«Что именно мы тестируем прямо
сейчас, как, когда и кем?»**. Если test strategy — это конституция, то test plan — это закон для
конкретного случая.

На практике test plan — самый используемый QA-документ. Его читают разработчики (чтобы понять, что
будет проверяться), менеджеры (чтобы понять сроки), и другие QA (чтобы знать, кто за что отвечает).

Проблема в том, что большинство test plans либо слишком формальные (50 страниц по IEEE 829, которые
никто не читает), либо слишком поверхностные (чеклист из 10 пунктов без контекста). В этом файле мы
найдём баланс: практичный, полезный, живой документ.

Мы разберём: структуру test plan, разницу между планами для фичи/релиза/продукта, дадим полный
практический пример, и объясним, как оценивать трудозатраты на тестирование.

## Структура тест-плана

### Вдохновлённая IEEE 829, но адаптированная для реального мира

IEEE 829 — это стандарт, который описывает формальную структуру тестовой документации. Он полезен
как reference, но в чистом виде слишком громоздкий. Мы возьмём из него лучшее и сделаем практичным.

### Обязательные секции

```
1.  Идентификация (ID, версия, автор, дата)
2.  Scope и цели
3.  Что тестируем / что НЕ тестируем
4.  Подход к тестированию
5.  Тестовые сценарии (высокоуровневые)
6.  Требования к окружению
7.  Тестовые данные
8.  Расписание
9.  Роли и ответственности
10. Риски и mitigation
11. Критерии входа / выхода
12. Deliverables (что получаем на выходе)
```

### Опциональные секции (добавлять по необходимости)

```
- Зависимости от других команд / систем
- Требования к инструментам
- Метрики
- Приёмочные критерии
- Процедура эскалации
- Approvals (подписи)
```

### 2. Scope и цели

Что тестируем и зачем. Это самая важная секция — она определяет всё остальное.

```markdown
## Scope
Данный test plan покрывает тестирование нового модуля оплаты (Payment v2.0),
включающего:
- Интеграцию со Stripe для карточных платежей
- Интеграцию с PayPal
- Сохранение платёжных данных (токенизация)
- Refund flow (возвраты)
- Webhook обработку от Stripe/PayPal

## Цели тестирования
1. Убедиться, что все платёжные сценарии работают корректно
2. Проверить обработку ошибок (declined cards, timeout, etc.)
3. Убедиться в безопасности обработки платёжных данных
4. Проверить совместимость с существующим order flow
5. Валидировать performance под ожидаемой нагрузкой
```

### 3. Что тестируем / что НЕ тестируем

Это критически важно. «Что не тестируем» — не менее важно, чем «что тестируем».

```markdown
## Features to Test (с обоснованием приоритета)

| Feature | Приоритет | Обоснование |
|---------|----------|-------------|
| Оплата картой (Stripe) | P0 — Critical | Основной flow, 80% транзакций |
| Оплата через PayPal | P1 — High | Вторичный flow, 15% транзакций |
| Refund (полный) | P1 — High | Юридическое требование |
| Refund (частичный) | P2 — Medium | Бизнес-требование, но редкий сценарий |
| Сохранение карты | P2 — Medium | Удобство, но не блокирует оплату |
| Webhook обработка | P0 — Critical | Без неё статусы не обновляются |
| Повторная оплата | P2 — Medium | Улучшение UX |

## Features NOT to Test (с обоснованием)

| Feature | Причина |
|---------|---------|
| Внутренняя логика Stripe API | Покрыта их тестами и SLA |
| Email-уведомления о платеже | Отдельный test plan (TP-2026-Q1-NOTIF) |
| Администрирование платежей | Будет тестироваться в следующем спринте |
| Mobile-native payment (Apple/Google Pay) | Не входит в scope v2.0 |
```

### 4. Подход к тестированию

Как именно будем тестировать каждый аспект.

```markdown
## Подход

### Функциональное тестирование
- **API-level:** REST Assured тесты для всех endpoints модуля оплаты
- **UI-level:** Playwright E2E тесты для основных user flows
- **Exploratory:** 2 сессии по 90 минут, фокус на edge cases и error handling

### Integration Testing
- Stripe: тестирование через Stripe Test Mode (test API keys)
- PayPal: тестирование через PayPal Sandbox
- Webhook: тестирование с использованием Stripe CLI для локальной эмуляции

### Security Testing
- Проверка, что карточные данные не сохраняются в логах
- Проверка, что PAN не передаётся в URL
- Проверка HTTPS для всех платёжных endpoints
- Проверка токенизации (карточные данные не хранятся на нашей стороне)

### Performance Testing
- Load test: 100 concurrent payments, response time < 3 sec
- Stress test: увеличение нагрузки до отказа, проверка graceful degradation
```

## Виды тест-планов: фича vs. релиз vs. master

### Feature Test Plan (для одной фичи)

**Когда:** Для каждой значимой фичи в спринте.
**Объём:** 1-3 страницы.
**Автор:** QA Engineer.

```markdown
# Feature Test Plan: Добавление фильтра по цене в каталог

## Scope
- Фильтр по цене (min/max) на странице каталога
- API endpoint GET /products?price_min=X&price_max=Y

## Тестовые сценарии
1. Фильтрация по min цене (только min задан)
2. Фильтрация по max цене (только max задан)
3. Фильтрация по диапазону (min + max)
4. Пустой результат (нет товаров в диапазоне)
5. Невалидные значения (отрицательные, буквы, min > max)
6. Граничные значения (0, максимальная цена в системе)
7. Взаимодействие с другими фильтрами (категория + цена)
8. Performance: фильтрация на 100K товаров

## Сроки: 2 дня
## Окружение: QA
## Зависимости: Backend ready by Jan 15
```

### Release Test Plan (для релиза)

**Когда:** Перед каждым релизом.
**Объём:** 3-7 страниц.
**Автор:** QA Lead / Senior QA.

Включает:
- Все фичи в релизе + их test plans
- Regression scope (что проверяем из старого функционала)
- Release-specific risks
- Go/no-go criteria

### Master Test Plan (для продукта)

**Когда:** Один раз, обновляется редко.
**Объём:** 5-15 страниц.
**Автор:** QA Lead / QA Manager.

Включает:
- Общий подход к тестированию продукта (по сути, test strategy + высокоуровневый план)
- Все компоненты продукта и их testing approach
- Долгосрочные цели по качеству
- Ресурсное планирование

### Связь между видами

```
Master Test Plan (продукт)
    │
    ├── Release Test Plan v2.0 (январь)
    │     ├── Feature Test Plan: Payment v2.0
    │     ├── Feature Test Plan: Новый фильтр
    │     └── Regression Test Plan
    │
    ├── Release Test Plan v2.1 (февраль)
    │     ├── Feature Test Plan: Mobile notifications
    │     ├── Feature Test Plan: Admin dashboard
    │     └── Regression Test Plan
    │
    └── ...
```

───────────────────────────────────────────────────────────────
1. ВВЕДЕНИЕ
───────────────────────────────────────────────────────────────

ShopApp переходит с прямой интеграции с банком на Stripe + PayPal.
Это критичное изменение, затрагивающее основной revenue flow.
Любой дефект в платёжном модуле = потеря дохода + потеря доверия.

Цели тестирования:
- Обеспечить 100% работоспособность основных платёжных сценариев
- Проверить все error scenarios
- Убедиться в PCI DSS compliance (базовый уровень)
- Валидировать performance при пиковой нагрузке

───────────────────────────────────────────────────────────────
2. SCOPE
───────────────────────────────────────────────────────────────

IN SCOPE:
✅ Оплата банковской картой через Stripe
✅ Оплата через PayPal
✅ Сохранение карты для повторных покупок
✅ Полный и частичный refund
✅ Webhook обработка (payment events)
✅ UI checkout flow
✅ API endpoints
✅ Безопасность платёжных данных
✅ Performance (до 100 одновременных транзакций)

OUT OF SCOPE:
❌ Внутренняя логика Stripe/PayPal (их зона ответственности)
❌ Apple Pay / Google Pay (v3.0)
❌ Криптовалютные платежи (не планируется)
❌ Admin-панель для управления платежами (отдельный plan)
❌ Email/SMS уведомления о платежах (отдельный plan)

───────────────────────────────────────────────────────────────
3. ПОДХОД
───────────────────────────────────────────────────────────────

Уровни тестирования:
- Unit tests (разработчики): покрытие бизнес-логики платежей
- Integration tests (QA + Dev): Stripe/PayPal sandbox
- E2E tests (QA): полный checkout flow
- Exploratory (QA): edge cases, error handling, UX

Типы тестирования:
- Functional: основной фокус
- Security: PCI DSS checklist
- Performance: load testing при peak load
- Regression: существующий order flow

## Test Plan как living document

### Почему план меняется

Test plan — это не высеченный в камне документ. Он эволюционирует потому что:

1. **Requirements меняются** — бизнес уточняет или добавляет требования
2. **Архитектура меняется** — разработчики находят лучшее решение
3. **Риски проявляются** — в процессе тестирования обнаруживаются новые risk areas
4. **Сроки сдвигаются** — реальность корректирует планы

### Как управлять изменениями

```markdown
## Change Log

| Версия | Дата | Автор | Изменение | Причина |
|--------|------|-------|-----------|---------|
| 1.0 | 10.01 | Иванов | Initial version | — |
| 1.1 | 15.01 | Иванов | Добавлен PayPal refund | Новое требование от PM |
| 1.2 | 20.01 | Иванов | Убраны тесты Apple Pay | Перенесено на v3.0 |
| 1.3 | 25.01 | Козлова | Добавлены performance тесты | Результаты smoke показали проблемы |
```

**Правила:**
- Каждое значимое изменение — новая версия
- Change log ведётся в самом документе
- Рецензенты уведомляются об изменениях
- Старые версии не удаляются (история сохраняется)

## Оценка трудозатрат на тестирование

### Почему оценка сложна

Оценка тестирования — одна из самых сложных задач. Причины:
- Зависимость от качества кода (больше багов → больше времени)
- Зависимость от стабильности окружения
- Непредсказуемость exploratory testing
- Необходимость retesting после bug fixes

### Методы оценки

#### 1. Expert Judgment (экспертная оценка)

Самый распространённый метод. Основан на опыте QA-инженера.

```
# Формула грубой оценки
Время на тестирование ≈ Время на разработку × Коэффициент

Коэффициенты (из практики):
- Простая фича (CRUD, UI-изменение): 0.3-0.5
- Средняя фича (бизнес-логика, интеграция): 0.5-0.8
- Сложная фича (платежи, security, data migration): 0.8-1.2
- Критичная фича (финансы, медицина, safety): 1.0-1.5
```

**Пример:** Разработка платёжного модуля оценена в 20 человеко-дней. Это сложная фича →
коэффициент 1.0 → тестирование: ~20 человеко-дней (с подготовкой, execution, bug verification).

#### 2. Разбивка по активностям

Более точный метод — разбить тестирование на конкретные активности.

```markdown
## Оценка для Payment v2.0

| Активность | Время (часы) | Комментарий |
|-----------|-------------|-------------|
| Изучение requirements | 4 | Чтение, вопросы, уточнения |
| Написание test plan | 8 | Этот документ |
| Подготовка test data | 4 | Stripe credentials, тестовые карты |
| Написание test cases | 16 | 47 test cases |
| Test execution (ручное) | 24 | P0 + P1 сценарии |
| Exploratory testing | 6 | 2 сессии по 3 часа |
| Автоматизация E2E | 16 | Top-10 critical scenarios |
| Security testing | 8 | PCI DSS checklist |
| Performance testing | 8 | Подготовка + execution |
| Bug reporting | 8 | Оформление ~15 багов (оценка) |
| Bug verification | 8 | Проверка исправлений |
| Regression | 8 | Существующий функционал |
| Reporting | 4 | Ежедневные + итоговый отчёт |
| **ИТОГО** | **122 часа** | **≈ 15 рабочих дней** |
| Buffer (20%) | 24 часа | Непредвиденные ситуации |
| **ИТОГО С БУФЕРОМ** | **146 часов** | **≈ 18 рабочих дней** |
```

#### 3. Test Points (тестовые поинты)

Аналог story points, но для тестирования. Полезно для относительной оценки.

```markdown
## Шкала Test Points

| Поинты | Описание | Пример |
|--------|----------|--------|
| 1 | Тривиальное — один сценарий, без подготовки | Проверка текста на странице |
| 2 | Простое — 2-3 сценария, минимальная подготовка | Проверка валидации формы |
| 3 | Среднее — 5-10 сценариев, нужны тестовые данные | Тестирование фильтра |
| 5 | Сложное — много сценариев, интеграции, данные | Тестирование checkout |
| 8 | Очень сложное — cross-system, security, performance | Тестирование платёжного модуля |
| 13 | Огромное — нужно декомпозировать | Полный regression release |

## Исторические данные
- 1 Test Point ≈ 2 часа работы QA (на нашем проекте)
- Payment v2.0 оценён в 55 Test Points ≈ 110 часов ≈ 14 дней
```

#### 4. Historical Data (исторические данные)

Самый точный метод — если у тебя есть данные с прошлых проектов/спринтов.

```markdown
## Исторические данные по нашему проекту

| Фича | Story Points (dev) | Время тестирования | Коэффициент |
|------|-------------------|-------------------|-------------|
| Регистрация v2 | 13 SP | 5 дней | 0.77 |
| Каталог фильтры | 8 SP | 3 дня | 0.75 |
| Email уведомления | 5 SP | 2 дня | 0.80 |
| Поиск по каталогу | 8 SP | 4 дня | 1.00 |
| Корзина v2 | 13 SP | 6 дней | 0.92 |

Средний коэффициент: 0.85
Payment v2.0 = 21 SP (dev estimate) × 0.85 ≈ 18 дней

Примечание: коэффициент > среднего, т.к. Payment более критичен
(применяем повышающий коэффициент 1.2) → 18 × 1.2 ≈ 22 дня
```

### Советы по оценке

1. **Всегда добавляй буфер 15-25%.** Вещи всегда занимают больше времени, чем кажется.
2. **Не забудь про retesting.** Каждый исправленный баг нужно проверить — это ~10-15% общего времени.
3. **Учитывай зависимости.** Если окружение нестабильно — добавь время на ожидание.
4. **Декомпозируй.** Оценка «тестирование всего» — плохая. Оценка по активностям — точнее.
5. **Трекай актуальные данные.** Сравнивай оценку с фактом, корректируй коэффициенты.

## Связь с тестированием

Test plan — это мост между стратегией (абстрактной) и тестами (конкретными). Его роль:

- **Для QA:** Чёткое понимание, что делать, в каком порядке, к какому сроку
- **Для разработчиков:** Понимание, что будет проверяться → мотивация писать качественный код
- **Для PM:** Прозрачность процесса, возможность планировать релиз
- **Для бизнеса:** Управление рисками, обоснование сроков

Test plan связывает тестирование с бизнес-целями. Вместо абстрактного «мы тестируем» —
конкретное «мы проверяем эти 47 сценариев к этой дате с таким-то ожидаемым результатом».

Без test plan тестирование — это хаотичная проверка «что попадётся». С test plan — это
управляемый процесс с прогнозируемым результатом.

## Типичные ошибки

### 1. «Plan написан, но не используется»
**Проблема:** Документ создан для галочки (или для аудита), реальное тестирование идёт по-другому.
**Результат:** Двойная работа, несоответствие между планом и реальностью.
**Решение:** Plan должен быть рабочим инструментом. Обновляй его, ссылайся на него, используй для отчётности.

### 2. «Слишком детальный план»
**Проблема:** 50 страниц с описанием каждого клика. Устаревает через неделю.
**Результат:** Невозможно поддерживать, никто не читает.
**Решение:** High-level plan + детальные test cases отдельно (в test management системе). Plan — обзор и навигация.

### 3. «Нет секции "What not to test"»
**Проблема:** Не определено, что вне scope.
**Результат:** Неопределённые ожидания. «А вы это проверяли?» — «А должны были?»
**Решение:** Явно указать, что не тестируем, с обоснованием.

### 4. «Нереалистичные сроки»
**Проблема:** PM просит: «Можно за 2 дня?» — QA отвечает: «Да».
**Результат:** Тестирование неполное, баги утекают в production.
**Решение:** Оценивай честно. Если сроки жмут — сокращай scope (P2/P3 переноси), а не качество.

### 5. «Забыли про тестовые данные»
**Проблема:** Plan готов, окружение настроено, а данных для тестирования нет.
**Результат:** Потеря 1-2 дней на подготовку данных посреди тестирования.
**Решение:** Тестовые данные — обязательная секция. Подготовка данных — часть расписания.

### 6. «Нет буфера в расписании»
**Проблема:** Расписание «впритык», без запаса.
**Результат:** Любая задержка (нестабильное окружение, больше багов, чем ожидалось) ломает весь график.
**Решение:** Закладывай 15-25% буфера. Это не лень, это реализм.

### 7. «План не согласован с командой»
**Проблема:** QA написал план, никому не показал, начал тестировать.
**Результат:** Разработчики не знают, что будет проверяться. PM не знает сроков. Конфликты на этапе sign-off.
**Решение:** Plan review — обязательный этап. Tech Lead и PM должны подтвердить scope и сроки.

## Вопросы на интервью

### 🟢 Junior

1. **Что такое test plan и зачем он нужен?**
   *Ожидаемый ответ:* Документ, описывающий что, как, когда и кем будет тестироваться. Нужен для планирования, координации, управления ожиданиями и отчётности. Без него тестирование хаотичное и непрозрачное.

2. **Какие основные секции должны быть в test plan?**
   *Ожидаемый ответ:* Scope, цели, что тестируем / не тестируем, подход, тестовые сценарии, окружение, данные, расписание, роли, риски, entry/exit criteria. Не все секции обязательны для каждого плана — зависит от контекста.

3. **Что такое entry и exit criteria? Приведи примеры.**
   *Ожидаемый ответ:* Entry: условия для начала тестирования (фича задеплоена, unit tests проходят, данные готовы). Exit: условия для завершения (все P0 PASS, нет Critical багов, regression пройден). Примеры из реальных проектов.

4. **Как ты оценишь время на тестирование фичи?**
   *Ожидаемый ответ:* Разбивка по активностям: изучение требований, написание test cases, подготовка данных, execution, bug reporting, retesting. Суммирование + буфер 15-25%. Для простых фич — экспертная оценка на основе опыта.

### 🟡 Middle

5. **Чем отличается feature test plan от release test plan?**
   *Ожидаемый ответ:* Feature plan — для одной конкретной фичи (1-3 страницы, конкретные сценарии). Release plan — для всего релиза (включает все фичи + regression, 3-7 страниц, общая картина). Release plan ссылается на feature plans. Разная аудитория и уровень детализации.

6. **Как ты управляешь изменениями в test plan во время проекта?**
   *Ожидаемый ответ:* Версионирование, change log, уведомление рецензентов. Если изменение значительное (>30% scope) — пересогласование с PM. Если scope увеличился — пересмотр сроков и ресурсов. Test plan — living document, но с контролируемыми изменениями.

7. **PM говорит: «Нам нужно релизить через 3 дня, тестировать некогда». Что делаешь?**
   *Ожидаемый ответ:* Не говорю «нельзя». Предлагаю risk-based подход: тестируем только P0 за 3 дня, остальное — сразу после релиза. Явно коммуницирую риски: «При таком scope мы покрываем X%, вот что может пойти не так». Решение — за PM/бизнесом, но с полной информацией.

8. **Как ты декомпозируешь тестирование большой фичи?**
   *Ожидаемый ответ:* По компонентам (UI, API, DB), по типам тестирования (functional, security, performance), по приоритетам (P0 → P1 → P2), по пользовательским сценариям (happy path → error handling → edge cases). Каждый блок можно оценить и планировать отдельно.

### 🔴 Senior / Lead

9. **Как ты организуешь тестовую документацию на проекте с 5 QA и 20 разработчиками?**
   *Ожидаемый ответ:* Master test plan на уровне продукта, release test plans на каждый релиз, feature test plans — назначаются конкретным QA. Единый template для consistency. Test management tool (Zephyr, TestRail) для test cases. Confluence для планов и отчётов. Code review для автотестов. Регулярные sync-и QA-команды.

10. **Как ты обосновываешь менеджменту, что тестирование требует больше времени, чем выделено?**
    *Ожидаемый ответ:* Данные, а не эмоции. Показываю: (1) детальную разбивку по активностям; (2) исторические данные (на аналогичных фичах тестирование занимало X дней); (3) альтернативы (сократить scope, но вот риски); (4) стоимость багов в production vs стоимость тестирования. Предлагаю варианты, а не просто говорю «мало времени».

11. **Ты унаследовал проект, где test plan — это Excel с 2000 строками, который не обновлялся год. Что делаешь?**
    *Ожидаемый ответ:* Не удаляю и не переписываю с нуля. Анализирую: что из этих 2000 строк актуально? Приоритизирую (P0/P1 сценарии). Мигрирую актуальную часть в нормальный формат (test management tool). Создаю новый lightweight test plan, ссылающийся на актуальные сценарии. Старый файл — в архив с пометкой «legacy, не использовать».

12. **Как связать test plan с requirements traceability?**
    *Ожидаемый ответ:* Requirement Traceability Matrix (RTM): каждый requirement → test case(s), каждый test case → requirement(s). В инструментах типа Jira+Zephyr это делается через linking. Позволяет ответить на вопросы: «Все ли requirements покрыты тестами?» и «Какие тесты нужно обновить при изменении requirement?». Критично для compliance (FDA, HIPAA, PCI DSS).
