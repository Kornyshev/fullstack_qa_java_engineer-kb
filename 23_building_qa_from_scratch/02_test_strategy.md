# Тестовая стратегия: что тестируем, как, зачем

## Обзор

Test strategy — это документ верхнего уровня, который отвечает на вопрос: **«Как мы обеспечиваем качество
на этом проекте?»**. Не «какие test cases написать» (это test plan), не «какие автотесты запустить» (это
test suite) — а именно общий подход к тестированию.

Если test plan — это GPS-маршрут («поверни направо, через 500 метров налево»), то test strategy — это решение
«едем через город или по объездной». Стратегия определяет направление, план — конкретные шаги.

Многие QA-инженеры путают эти понятия или считают test strategy чем-то бюрократическим. Это ошибка. Без
стратегии ты тестируешь реактивно — просто проверяешь то, что прилетает. Со стратегией ты тестируешь
проактивно — знаешь, что самое важное, и фокусируешь усилия.

В этом файле мы разберём: что входит в test strategy, как её составить, как обосновать перед командой и
руководством, и дадим готовый шаблон с примером.

---

## Test Strategy vs. Test Plan

Это два разных документа, и оба важны. Разберём различия детально.

| Аспект | Test Strategy | Test Plan |
|--------|--------------|-----------|
| **Уровень** | Проект / продукт | Релиз / фича / спринт |
| **Горизонт** | Долгосрочный (месяцы, годы) | Краткосрочный (недели) |
| **Автор** | QA Lead / QA Manager | QA Engineer |
| **Аудитория** | Management, вся команда | QA-команда, разработчики |
| **Содержание** | Подход, принципы, уровни тестирования | Конкретные test cases, расписание, ресурсы |
| **Изменяемость** | Редко меняется | Меняется каждый спринт/релиз |
| **Вопрос** | «Как мы тестируем?» | «Что мы тестируем сейчас?» |

### Почему нужны оба

**Без стратегии:** Каждый test plan создаётся с нуля, нет единого подхода, разные QA тестируют по-разному,
нет приоритизации на уровне проекта.

**Без плана:** Стратегия остаётся теорией, нет конкретики для текущей работы, невозможно отслеживать
прогресс.

**С обоими:** Стратегия задаёт рамки, план наполняет их конкретикой. Каждый новый test plan создаётся
быстрее, потому что основные решения уже приняты в стратегии.

## Структура тестовой стратегии

### 1. Scope и цели (Scope & Objectives)

Определяем границы: что входит в scope тестирования, а что нет. И почему.

```markdown
## Scope

### В scope:
- Web-приложение (frontend + backend API)
- Mobile-адаптивная версия (через браузер, не нативное приложение)
- Интеграции с платёжными системами (Stripe, PayPal)
- Email-уведомления

### Вне scope:
- Тестирование сторонних сервисов (Stripe, SendGrid) — у них свои SLA
- Legacy admin-панель (будет полностью переписана в Q3)
- Нативные мобильные приложения (будут разрабатываться позже)

### Objectives:
1. Обеспечить отсутствие critical/blocker багов в production
2. Сократить время обнаружения дефектов (shift-left)
3. Достичь 80% coverage unit tests на backend к Q2
4. Автоматизировать smoke regression для CI/CD pipeline
```

**Важно:** Чётко указать, что вне scope, и обосновать. Это защищает тебя от вопросов: «А вы это
проверяли?» — «Нет, это за пределами scope, потому что [причина]».

### 2. Подход к тестированию (Testing Approach)

Определяем, какие виды и уровни тестирования применяем.

#### Уровни тестирования

```
┌─────────────────────────────────────────────┐
│              E2E Tests                      │  ← Мало, дорого, медленно
│          (критичные user flows)             │     Фокус: core business scenarios
├─────────────────────────────────────────────┤
│           Integration Tests                 │  ← Средне
│     (API, межсервисное взаимодействие)      │     Фокус: контракты и потоки данных
├─────────────────────────────────────────────┤
│           Component Tests                   │  ← Больше
│      (отдельные сервисы / модули)           │     Фокус: бизнес-логика модуля
├─────────────────────────────────────────────┤
│             Unit Tests                      │  ← Много, дёшево, быстро
│        (отдельные классы / функции)         │     Фокус: логика, edge cases
└─────────────────────────────────────────────┘
          Тестовая пирамида (Test Pyramid)
```

Для каждого уровня указываем:
- **Кто пишет** (разработчики, QA, совместно)
- **Инструменты** (JUnit, Testcontainers, Selenium, Playwright, etc.)
- **Когда запускается** (на каждый commit, перед merge, по расписанию)
- **Целевой coverage** (не всегда числовой — иногда «все критичные сценарии»)

#### Виды тестирования

Не все виды нужны на каждом проекте. Определяем, какие применяем и почему:

```markdown
## Виды тестирования

### Функциональное тестирование
- **Применяем:** Да, основной вид
- **Подход:** Автоматизация smoke + regression, ручной exploratory testing
- **Приоритет:** Высокий

### Performance Testing
- **Применяем:** Да, начиная с Q2
- **Обоснование:** Ожидается рост нагрузки в 3 раза после маркетинговой кампании
- **Подход:** Load testing основных API, stress testing платёжного потока
- **Инструменты:** k6 / Gatling
- **Приоритет:** Средний

### Security Testing
- **Применяем:** Да, базовый уровень
- **Подход:** OWASP Top 10 проверки, SAST в CI/CD (SonarQube)
- **Полный pentest:** Аутсорс раз в квартал
- **Приоритет:** Высокий (есть платежи)

### Accessibility Testing
- **Применяем:** Да, базовый уровень
- **Обоснование:** Законодательные требования + этика
- **Подход:** Axe автоматические проверки + ручная проверка с screen reader
- **Приоритет:** Средний

### Compatibility Testing
- **Применяем:** Ограниченно
- **Браузеры:** Chrome (основной), Firefox, Safari (на основных flows)
- **Mobile:** iOS Safari, Android Chrome
- **Приоритет:** Низкий (90% пользователей на Chrome)
```

### 3. Risk-Based Testing

Это сердце тестовой стратегии. Мы не можем протестировать всё — поэтому тестируем то, что несёт
наибольший риск.

#### Как оценить риск

**Risk = Вероятность дефекта × Влияние дефекта**

**Факторы вероятности:**
- Сложность кода (больше сложность → больше багов)
- Частота изменений (часто меняется → больше регрессий)
- Новый код vs зрелый код (новый → больше багов)
- Опыт разработчика (джуниор → больше багов — не обижайте, но это статистика)
- Наличие unit tests (нет тестов → больше багов)
- Технический долг (больше долг → больше багов)

**Факторы влияния:**
- Финансовый ущерб (потеря денег клиентов)
- Репутационный ущерб (публичный скандал)
- Количество затронутых пользователей
- Юридические последствия (GDPR, PCI DSS)
- Невозможность использовать продукт (блокирующий баг)

#### Матрица рисков

```
         │ Низкое      │ Среднее     │ Высокое
         │ влияние     │ влияние     │ влияние
─────────┼─────────────┼─────────────┼──────────────
Высокая  │ СРЕДНИЙ     │ ВЫСОКИЙ     │ КРИТИЧЕСКИЙ
вероят.  │ Авто-тесты  │ Глубокое    │ Максимальное
         │             │ тестирование│ покрытие
─────────┼─────────────┼─────────────┼──────────────
Средняя  │ НИЗКИЙ      │ СРЕДНИЙ     │ ВЫСОКИЙ
вероят.  │ Базовые     │ Авто-тесты  │ Глубокое
         │ проверки    │ + ручные    │ тестирование
─────────┼─────────────┼─────────────┼──────────────
Низкая   │ МИНИМАЛЬНЫЙ │ НИЗКИЙ      │ СРЕДНИЙ
вероят.  │ Smoke       │ Базовые     │ Авто-тесты
         │ только      │ проверки    │
```

#### Пример risk assessment

```markdown
| Компонент        | Вероятность | Влияние | Риск        | Подход              |
|------------------|------------|---------|-------------|---------------------|
| Оплата           | Средняя    | Высокое | ВЫСОКИЙ     | E2E + API + ручные  |
| Регистрация      | Низкая     | Высокое | СРЕДНИЙ     | E2E автотесты       |
| Каталог товаров  | Высокая    | Среднее | ВЫСОКИЙ     | API + UI автотесты  |
| Поиск            | Высокая    | Среднее | ВЫСОКИЙ     | API тесты + ручные  |
| Профиль          | Низкая     | Низкое  | МИНИМАЛЬНЫЙ | Smoke только        |
| Отчёты (admin)   | Средняя    | Низкое  | НИЗКИЙ      | Базовые ручные      |
| Уведомления      | Средняя    | Среднее | СРЕДНИЙ     | API тесты           |
```

### 4. Testing Quadrants (модель Brian Marick)

Полезная модель для того, чтобы убедиться, что мы не забыли ни одного аспекта тестирования.

```
         Technology-Facing              Business-Facing
         (для разработчиков)            (для бизнеса)
    ┌────────────────────────┬────────────────────────┐
    │                        │                        │
    │  Q1: Unit Tests        │  Q2: Functional Tests  │
S   │      Component Tests   │      User Stories      │
u   │      Integration Tests │      Prototypes        │
p   │                        │      Simulations       │
p   │  Автоматизированные    │  Авто + ручные         │
o   │                        │                        │
r   ├────────────────────────┼────────────────────────┤
t   │                        │                        │
i   │  Q3: Performance       │  Q4: Exploratory       │
n   │      Security          │      Usability         │
g   │      Load              │      A/B Testing       │
    │      Reliability       │      UAT               │
C   │                        │                        │
r   │  Инструменты + ручные  │  Преимущественно       │
i   │                        │  ручные                │
t   │                        │                        │
i   └────────────────────────┴────────────────────────┘
q
u
e
```

**Q1 (Supporting, Technology):** Unit и integration tests. Пишутся разработчиками (часто с помощью QA).
Запускаются в CI. Обеспечивают быструю обратную связь.

**Q2 (Supporting, Business):** Функциональные тесты, проверяющие бизнес-сценарии. Пишутся QA совместно
с PO/BA. Автоматизируются для regression.

**Q3 (Critique, Technology):** Performance, security, stress testing. Обычно планируются заранее,
запускаются периодически или перед крупными релизами.

**Q4 (Critique, Business):** Exploratory testing, usability, UAT. Требуют человеческого суждения,
сложно автоматизировать. Критически важны для обнаружения неожиданных проблем.

**Как использовать:** Убедись, что в твоей стратегии покрыты все 4 квадранта. Пропорции зависят от
проекта, но полное отсутствие какого-то квадранта — красный флаг.

### 5. Entry и Exit Criteria

Определяем, когда тестирование начинается и когда заканчивается.

```markdown
## Entry Criteria (когда начинаем тестировать)
- Фича задеплоена на тестовое окружение
- Unit tests проходят (coverage не ниже 70%)
- Code review пройден
- Тестовые данные подготовлены
- Acceptance criteria определены в задаче

## Exit Criteria (когда считаем тестирование завершённым)
- Все test cases выполнены
- Нет открытых Critical/Blocker багов
- Major баги — либо исправлены, либо осознанно приняты (с документацией)
- Regression suite — 100% pass rate
- Smoke test на staging — pass
- Performance тесты (если применимо) — в пределах SLA

## Suspension Criteria (когда останавливаем тестирование)
- Тестовое окружение недоступно
- Найден blocker, блокирующий основной flow
- Критическое изменение requirements
```

### 6. Тестовые окружения (Test Environments)

```markdown
## Environments

| Окружение | Назначение | Данные | Обновление |
|-----------|-----------|--------|------------|
| Dev       | Разработка, unit tests | Минимальные, фейковые | Каждый commit |
| QA/Test   | Функциональное тестирование | Тестовые, приближенные к реальным | Каждый PR merge |
| Staging   | Pre-production проверка | Анонимизированные копии production | Перед релизом |
| Production | Live | Реальные | Релиз |

## Правила:
- QA НЕ тестирует на dev-окружении (нестабильно)
- Production-like тестирование только на staging
- Performance тестирование — на отдельном изолированном окружении
- Тестовые данные не содержат реальные персональные данные
```

### 7. Инструменты (Tools)

```markdown
## Инструменты тестирования

### Test Management
- **Jira + Zephyr** — test cases, test execution, traceability

### Автоматизация
- **Backend:** JUnit 5 + Testcontainers (unit + integration)
- **API:** REST Assured (Java) / Playwright API testing
- **UI:** Playwright (основной) / Selenium (legacy)
- **Mobile:** нет (пока нет мобильного приложения)

### Performance
- **k6** — load testing (скрипты на JavaScript, интеграция с CI)

### CI/CD
- **GitHub Actions** — запуск тестов на каждый PR и перед релизом

### Мониторинг
- **Grafana + Prometheus** — метрики приложения
- **Sentry** — error tracking

### Вспомогательные
- **Postman** — ad-hoc API testing, exploratory
- **Charles Proxy** — перехват и анализ трафика
- **DBeaver** — работа с БД (проверка данных)
```

### 8. Метрики (Metrics)

```markdown
## Метрики качества

### Метрики продукта
- Количество багов в production (за спринт/месяц)
- Mean Time To Detect (MTTD) — среднее время обнаружения бага
- Mean Time To Resolve (MTTR) — среднее время исправления бага
- Escape rate — % багов, дошедших до production

### Метрики процесса
- Test coverage (unit, integration, E2E)
- Test pass rate
- Время на тестирование фичи (от «готово к тесту» до «протестировано»)
- Количество багов, найденных на каждом этапе (shift-left метрика)

### Метрики автоматизации
- Процент автоматизированных regression tests
- Время выполнения test suite
- Flaky test rate (% нестабильных тестов)
```

## Как выбрать уровни тестирования для ВАШЕГО проекта

Тестовая пирамида — это ориентир, а не догма. Реальная пропорция зависит от проекта.

### Сценарий 1: API-centric backend (микросервисы)

```
Фокус: Integration tests (контракты между сервисами)
Много:   Unit tests + Contract tests
Средне:  Integration tests (API)
Мало:    E2E tests (только critical paths)

Обоснование: UI минимальный, основная логика в backend.
Риск в интеграции между сервисами — туда и направляем усилия.
```

### Сценарий 2: UI-heavy web application (SPA)

```
Фокус: Component tests + E2E tests
Много:   Unit tests (логика) + Component tests (UI компоненты)
Средне:  Integration tests (API)
Средне:  E2E tests (пользовательские сценарии)

Обоснование: Много UI-логики, визуальные регрессии критичны.
Component tests дешевле E2E, но ловят большинство UI-багов.
```

### Сценарий 3: Data-intensive application (ETL, analytics)

```
Фокус: Data validation + Integration tests
Много:   Unit tests (трансформации данных)
Много:   Integration tests (data pipelines)
Средне:  Data quality tests (проверка данных)
Мало:    E2E / UI tests

Обоснование: Основной риск — некорректные данные.
UI простой, бизнес-логика в обработке данных.
```

### Сценарий 4: Mobile application

```
Фокус: UI tests + Integration tests
Средне:  Unit tests
Средне:  Integration tests (API)
Много:   UI tests (на реальных устройствах)
Средне:  E2E tests

Обоснование: Фрагментация устройств, UI — основной
интерфейс, сетевые условия непредсказуемы.
```

## Пример тестовой стратегии для web-приложения

Ниже — полноценный пример стратегии для типичного e-commerce проекта. Используй его как шаблон,
адаптируя под свой контекст.

```markdown

# Версия: 1.0 | Дата: 2026-01-15 | Автор: QA Lead

## 1. Введение
ShopApp — e-commerce платформа для продажи одежды. 50K MAU,
3 разработчика, 1 QA. Stack: React + Spring Boot + PostgreSQL.
Релизы каждые 2 недели.

## 2. Scope
### В scope:
- Web-приложение (desktop + mobile browser)
- REST API
- Интеграция со Stripe (платежи)
- Email-уведомления (SendGrid)

### Вне scope:
- Сторонние сервисы (Stripe, SendGrid — покрыты их SLA)
- SEO (ответственность маркетинга)
- Инфраструктура (ответственность DevOps)

## 3. Подход
### Уровни тестирования:
- Unit: разработчики, JUnit 5, target 80% coverage

| Корзина | ВЫСОКИЙ | E2E + API автотесты |
| Профиль | НИЗКИЙ | Smoke |
```

## Как презентовать стратегию и получить buy-in

### Типичные возражения и как на них отвечать

**«Нам некогда тестировать, надо фичи делать»**
→ «Мы не предлагаем остановить разработку. Мы предлагаем параллельно инвестировать в качество.
Каждый баг на production стоит в 10-30 раз дороже, чем баг, найденный до релиза. Вот статистика
наших багов за последний квартал: [цифры].»

**«Unit tests достаточно»**
→ «Unit tests прекрасны для проверки отдельных функций, но не ловят проблемы интеграции.
Вот пример: [конкретный баг, который unit tests не поймали]. Integration и E2E тесты
закрывают эту брешь.»

**«Автоматизация — пустая трата времени, всё и так работает»**
→ «Сейчас — да. Но с каждым релизом regression scope растёт. Через полгода ручной regression
будет занимать 2 дня. Автоматизация — это инвестиция, которая окупается с третьего прогона.»

**«Это слишком сложно / бюрократично»**
→ «Стратегия — это не 50-страничный документ. Это одна страница, которая описывает наш подход.
Она экономит время, потому что не нужно каждый раз решать заново, что и как тестировать.»

### Формат презентации

1. **Проблема** — покажи реальные баги, инциденты, стоимость
2. **Предложение** — стратегия (кратко, 5-7 слайдов максимум)
3. **Ожидаемый результат** — конкретные метрики, которые улучшатся
4. **Ресурсы** — что нужно (время, инструменты, доступы)
5. **Таймлайн** — когда будут первые результаты

---

## Эволюция стратегии

Test strategy — это living document, но меняется он редко. Триггеры для пересмотра:

- **Изменение архитектуры** (monolith → microservices)
- **Значительный рост команды** (2 → 10 разработчиков)
- **Новые требования** (выход на новый рынок, compliance)
- **Серьёзный инцидент** (провал, которого стратегия не предвидела)
- **Смена технологий** (новый framework, новый cloud provider)

Рекомендация: пересматривай стратегию раз в квартал. Даже если ничего не меняешь — убедись, что
она всё ещё актуальна.

---

## Связь с тестированием

Test strategy — это мета-уровень тестирования. Она не заменяет test cases, а определяет:

- **Какие виды тестирования** мы применяем (и какие осознанно пропускаем)
- **На каком уровне** фокусируем усилия (пирамида тестирования)
- **Что приоритетно** (risk-based approach)
- **Кто за что отвечает** (разработчики → unit, QA → E2E, совместно → integration)
- **Когда тестирование начинается и заканчивается** (entry/exit criteria)

Без стратегии test plan — это набор задач без контекста. Со стратегией test plan — это конкретная
реализация общего видения.

Стратегия также определяет баланс между ручным и автоматизированным тестированием, между глубиной
и шириной, между скоростью и тщательностью.

## Типичные ошибки

### 1. «Стратегия = список инструментов»
**Проблема:** Документ описывает, какие инструменты используем, но не объясняет, почему именно эти,
и не покрывает approach.
**Решение:** Инструменты — это одна секция. Стратегия — это про подход, приоритеты, риски.

### 2. «Копирование чужой стратегии»
**Проблема:** QA берёт шаблон из интернета и заполняет его, не адаптируя под свой проект.
**Результат:** Стратегия не отражает реальность, команда её игнорирует.
**Решение:** Шаблон — это отправная точка. Каждый пункт должен быть осмыслен в контексте вашего проекта.

### 3. «Стратегия написана и забыта»
**Проблема:** Документ создан, лежит в Confluence, никто его не открывает.
**Решение:** Ссылайся на стратегию при принятии решений. «Мы не тестируем это, потому что по стратегии
риск минимальный — вот пункт 4.3.»

### 4. «Всё тестируем одинаково»
**Проблема:** Нет приоритизации — каждая фича получает одинаковый объём тестирования.
**Результат:** Критичные функции недотестированы, некритичные — перетестированы.
**Решение:** Risk-based approach. Распределяй усилия пропорционально риску.

### 5. «Игнорирование non-functional testing»
**Проблема:** Стратегия покрывает только функциональное тестирование.
**Результат:** Приложение работает правильно, но медленно, небезопасно или неудобно.
**Решение:** Определи, какие NFR критичны для вашего проекта, и включи их в стратегию.

### 6. «Нереалистичные цели»
**Проблема:** Стратегия обещает 100% coverage, zero defects, полную автоматизацию.
**Результат:** Цели недостижимы → стратегия теряет доверие → её игнорируют.
**Решение:** Ставь реалистичные, измеримые цели с конкретными сроками.

### 7. «Стратегия без buy-in команды»
**Проблема:** QA написал стратегию в одиночку, не обсудив с разработчиками и менеджментом.
**Результат:** Команда не чувствует ownership, не следует стратегии.
**Решение:** Пиши драфт → обсуди с командой → учти feedback → финализируй совместно.

## Вопросы на интервью

### 🟢 Junior

1. **Что такое test strategy и чем она отличается от test plan?**
   *Ожидаемый ответ:* Test strategy — документ верхнего уровня, описывающий общий подход к тестированию на проекте (какие виды, какие уровни, какие приоритеты). Test plan — конкретный план тестирования для конкретного релиза/фичи (какие test cases, какие сроки, какие ресурсы). Стратегия — долгосрочная, план — краткосрочный.

2. **Что такое тестовая пирамида?**
   *Ожидаемый ответ:* Модель, показывающая оптимальное распределение тестов по уровням: много unit tests (быстрые, дешёвые), средне integration tests, мало E2E tests (медленные, дорогие). Идея в том, что большинство проблем ловятся на нижних уровнях быстрее и дешевле.

3. **Какие виды тестирования ты знаешь?**
   *Ожидаемый ответ:* Функциональное, нагрузочное (performance), security, usability, accessibility, compatibility, regression, smoke, sanity, exploratory, acceptance (UAT). Важно понимать, когда и зачем применяется каждый вид.

4. **Что такое entry и exit criteria?**
   *Ожидаемый ответ:* Entry criteria — условия, при которых тестирование может начаться (например, фича задеплоена, unit tests пройдены). Exit criteria — условия, при которых тестирование считается завершённым (например, все critical баги исправлены, regression прошёл).

### 🟡 Middle

5. **Как бы ты определил приоритеты тестирования для нового проекта?**
   *Ожидаемый ответ:* Risk-based approach: оцениваю вероятность дефекта и его влияние на бизнес. Начинаю с core business flows (то, за что платят деньги), потом расширяю на менее критичные области. Учитываю: частоту использования, сложность кода, историю багов, бизнес-impact.

6. **Опиши testing quadrants Brian Marick. Как ты их используешь?**
   *Ожидаемый ответ:* 4 квадранта: Q1 (technology-facing, supporting — unit/integration tests), Q2 (business-facing, supporting — functional tests), Q3 (technology-facing, critiquing — performance/security), Q4 (business-facing, critiquing — exploratory/usability). Использую для проверки: не пропустил ли я какой-то аспект тестирования? Если один из квадрантов пустой — это повод задуматься.

7. **Когда тестовая пирамида НЕ работает? Приведи пример.**
   *Ожидаемый ответ:* Пирамида не универсальна. Пример: UI-heavy приложение с минимальной бизнес-логикой на backend — там E2E/component тесты важнее unit tests. Или legacy-система без тестов — начинать с E2E smoke tests проще и ценнее, чем с unit tests. Важно адаптировать модель под контекст.

8. **Как ты обоснуешь необходимость performance testing перед менеджментом?**
   *Ожидаемый ответ:* Покажу цифры: сколько пользователей сейчас, какой ожидаемый рост, что будет при x3 нагрузке. Приведу примеры инцидентов (своих или индустриальных) из-за отсутствия performance testing. Посчитаю стоимость downtime vs стоимость testing.

### 🔴 Senior / Lead

9. **Как бы ты построил тестовую стратегию для проекта с микросервисной архитектурой (10+ сервисов)?**
   *Ожидаемый ответ:* Contract testing между сервисами (Pact), integration testing через Testcontainers, E2E только для top-N critical flows. Service-level testing > system-level testing. Отдельная стратегия для каждого сервиса + общая стратегия для интеграции. CDC (Consumer-Driven Contracts). Feature flags для тестирования в production. Observability как часть стратегии (мониторинг + alerting).

10. **Ты пришёл на проект и обнаружил, что предыдущий QA написал 500 E2E тестов и 0 unit tests. Что будешь делать?**
    *Ожидаемый ответ:* Не удалять существующие тесты. Проанализировать, что они покрывают и сколько стоят (время выполнения, flakiness, maintenance). Начать инвестировать в unit tests для нового кода. Постепенно заменять E2E тесты, которые можно покрыть на более низком уровне. Оставить E2E только для real user scenarios. Мигрировать поэтапно, не ломая текущий процесс.

11. **Как интервьюер: «Как бы ты построил тестовую стратегию для нового проекта?» — ответь развёрнуто.**
    *Ожидаемый ответ:* Системный подход: (1) понять продукт, пользователей, архитектуру, риски; (2) определить scope и приоритеты через risk assessment; (3) выбрать уровни и виды тестирования, адаптировав пирамиду; (4) определить инструменты и процессы; (5) установить entry/exit criteria и метрики; (6) согласовать с командой и итерировать. Подчеркнуть, что стратегия — живой документ, привязанный к контексту конкретного проекта, а не шаблон из интернета.

12. **Как ты определяешь, что «достаточно протестировано» для релиза?**
    *Ожидаемый ответ:* Комбинация факторов: (1) exit criteria выполнены (формальные); (2) risk coverage — все high-risk области проверены; (3) уверенность команды (субъективно, но важно); (4) метрики (test pass rate, defect trend, escape rate); (5) бизнес-контекст (deadline, market pressure). Идеальных релизов не бывает — это всегда осознанное управление рисками, а не достижение нуля дефектов.

здоровой пирамиде. Какие конкретные шаги ты предпримешь?
