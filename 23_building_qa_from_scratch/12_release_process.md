# Релизный процесс: роль QA от Feature Freeze до Production

## Обзор

Релиз — это момент истины для QA. Всё, что команда делала в течение спринта или итерации, проверяется одним простым фактом: работает ли продукт в production или нет.

Роль QA в релизном процессе выходит далеко за рамки "прогнать тесты перед деплоем". QA-инженер — это хранитель качества на каждом этапе: от момента, когда код замораживается, до мониторинга production после выкатки. QA участвует в принятии решения Go/No-Go, определяет критерии отката, формирует release notes и следит за тем, чтобы hotfix-процесс не превращался в хаос.

В этом разделе разберём весь lifecycle релиза глазами QA-инженера, включая современные подходы: canary releases, feature flags, continuous deployment.

---

## Этапы релизного процесса

### Общая схема классического релиза

```
Feature Freeze
     │
     ▼
Regression Testing (на стабильной ветке)
     │
     ▼
Bug Fixing (только критические)
     │
     ▼
Smoke Testing на Staging
     │
     ▼
Go/No-Go Meeting
     │
     ├── Go → Production Deployment
     │         │
     │         ▼
     │    Smoke Testing на Production
     │         │
     │         ▼
     │    Post-Release Monitoring (24-72 часа)
     │
     └── No-Go → Откат в Sprint Backlog, перепланирование
```

### Этап 1: Feature Freeze

**Что это:** момент, после которого в релизную ветку не добавляются новые фичи. Только bug fixes.

**Роль QA:**
- Убедиться, что все запланированные фичи действительно смержены и задеплоены на тестовое окружение
- Сверить scope релиза с планом: что вошло, что отложено
- Начать подготовку regression test suite: актуализировать тест-кейсы, убрать неактуальные, добавить для новых фичей
- Проверить тестовые данные и окружение

**Чеклист Feature Freeze для QA:**

```
□ Все фичи релиза смержены в release branch
□ Release branch задеплоена на staging/QA-окружение
□ Тестовые данные подготовлены
□ Regression suite актуализирован
□ Автотесты "зелёные" на текущей сборке
□ Известные проблемы задокументированы
□ Команда уведомлена: "Feature Freeze, только bug fixes"
```

### Этап 2: Regression Testing

**Что это:** полный прогон тестов для проверки, что новые изменения не сломали существующую функциональность.

**Стратегия QA:**

```
Приоритет 1: Smoke-тесты (автоматизированные)
             Время: 15-30 минут
             Цель: убедиться, что приложение запускается и основные потоки работают

Приоритет 2: Тесты новых фичей (ручные + автоматизированные)
             Время: 1-2 дня
             Цель: проверить, что новый функционал работает по спецификации

Приоритет 3: Full regression (автоматизированные + ручные критические)
             Время: 1-3 дня
             Цель: убедиться, что ничего не сломано

Приоритет 4: Exploratory testing на стыках новой и старой функциональности
             Время: 0.5-1 день
             Цель: найти неочевидные проблемы интеграции
```

**Критерии завершения regression:**

```
✅ Все Blocker и Critical баги исправлены и перепроверены
✅ Автоматизированные тесты: pass rate >= 95% (с объяснением оставшихся 5%)
✅ Ручной regression: все High-priority тест-кейсы пройдены
⚠️ Known Issues задокументированы с workarounds
❌ Ни одного открытого Blocker-бага
```

### Этап 3: Bug Fixing в период Regression

Важный нюанс: баги, найденные во время regression, нужно классифицировать.

**Матрица решений:**

| Severity    | Новый баг (в новой фиче) | Регрессия (сломано старое) |
|-------------|--------------------------|----------------------------|
| Blocker     | Чинить обязательно       | Чинить обязательно         |
| Critical    | Чинить обязательно       | Чинить обязательно         |
| Major       | Чинить, если есть время  | Чинить обязательно         |
| Minor       | Отложить на следующий релиз | Оценить риск            |
| Trivial     | Отложить                 | Отложить                   |

> **Опытное мнение:** каждый bug fix во время regression — это риск новой регрессии. Правило: после каждого исправления — re-test бага + smoke-тест затронутой области. Не нужно перезапускать full regression на каждый fix, но и не нужно чинить "втихаря".

### Этап 4: Smoke Testing на Staging

Staging-окружение должно максимально соответствовать production.

**Что проверять (Staging Smoke Checklist):**

```
□ Приложение запускается без ошибок
□ Логин/логаут работает
□ Основной бизнес-поток проходит от начала до конца
  (например: поиск → добавление в корзину → оформление → оплата)
□ Интеграции с внешними системами работают (платёжный шлюз, email, SMS)
□ Миграции БД применены корректно
□ Конфигурация окружения корректна (URLs, API keys для staging)
□ Логи не содержат критических ошибок
□ Performance не деградировал радикально (базовая проверка response time)
```

**Типичная проблема:** staging не соответствует production. Данные другие, конфигурация отличается, нагрузки нет. QA должен постоянно поднимать этот вопрос и добиваться максимального соответствия.

### Этап 5: Go/No-Go Meeting

Критический момент: команда решает, выкатывать релиз или нет.

**Участники:** Product Owner, Tech Lead, QA Lead, DevOps/SRE, иногда — менеджмент.

**Что готовит QA:**

```
RELEASE READINESS REPORT
========================

Release: v2.5.0
Date: 2026-03-20
QA Lead: [Имя]

TESTING SUMMARY:
- Total test cases executed: 847
- Passed: 821 (96.9%)
- Failed: 12 (1.4%) — all Minor severity
- Blocked: 8 (0.9%) — зависимость от внешней системы
- Not Run: 6 (0.7%) — out of scope для данного релиза

OPEN DEFECTS:
- Blocker: 0
- Critical: 0
- Major: 2 (описание + workaround + план исправления)
- Minor: 8

KNOWN ISSUES:
1. [BUG-456] Медленная загрузка отчётов > 10K строк
   Workaround: экспорт в CSV и открытие локально
   Fix planned: v2.6.0

2. [BUG-461] Неверная локализация даты для zh-CN
   Workaround: нет (затрагивает < 0.1% пользователей)
   Fix planned: v2.5.1

RECOMMENDATION: GO ✅
Обоснование: все критические сценарии проверены, blocker/critical дефектов нет,
известные проблемы имеют workarounds и не затрагивают основной бизнес-поток.
```

**Когда рекомендовать No-Go:**
- Есть открытые Blocker или Critical дефекты
- Критический бизнес-поток не работает
- Staging-тестирование выявило проблемы, которые не были замечены ранее
- Недостаточно времени на тестирование (scope был расширен без увеличения QA-ресурсов)
- Инфраструктурные проблемы (staging нестабилен, и нет уверенности в результатах)

### Этап 6: Production Deployment

**Роль QA во время деплоя:**

1. **Быть на связи.** QA должен быть доступен во время деплоя, даже если непосредственно его не выполняет
2. **Smoke-тест на production** сразу после выкатки
3. **Мониторинг** ошибок в реальном времени

**Production Smoke Checklist (выполнять в первые 15-30 минут после деплоя):**

```
□ Главная страница загружается
□ Регистрация нового пользователя работает
□ Логин существующего пользователя работает
□ Основной бизнес-поток проходит (end-to-end)
□ API health check endpoints возвращают 200
□ Нет всплеска ошибок в логах / мониторинге
□ Email/SMS-уведомления отправляются
□ Интеграции с внешними сервисами работают
```

> **Важно:** smoke на production нужно выполнять на реальных данных, но аккуратно. Используйте тестовые аккаунты, не создавайте реальные заказы (или используйте специальные тестовые платёжные карты).

### Этап 7: Post-Release Monitoring

Первые 24-72 часа после релиза — критический период.

**Что мониторить:**

```
Метрика                          │ Где смотреть          │ Порог тревоги
─────────────────────────────────┼───────────────────────┼──────────────────
Error rate (5xx)                 │ Grafana / DataDog     │ > 1% от запросов
Response time (p95)              │ APM (New Relic и др.) │ > 2x от baseline
Количество support-тикетов       │ Jira / Zendesk        │ > 150% от нормы
Crash rate (mobile)              │ Firebase / Sentry     │ > 0.5%
Бизнес-метрики (конверсия и пр.) │ Analytics             │ Падение > 10%
```

**Таймлайн мониторинга:**

```
0-1 час:   Smoke-тест на production, наблюдение за error rate
1-4 часа:  Мониторинг support-тикетов, проверка логов
4-24 часа: Анализ бизнес-метрик (конверсия, активность пользователей)
24-72 часа: Сравнение ключевых бизнес-показателей с предыдущей неделей
```

## Hotfix-процесс

### Когда нужен hotfix

Hotfix — экстренное исправление критической проблемы в production. Не каждый баг требует hotfix.

**Критерии для hotfix:**

```
✅ Hotfix нужен:
- Потеря данных пользователей
- Невозможность оплаты / основного бизнес-потока
- Уязвимость безопасности
- Массовый сбой (> 10% пользователей затронуты)
- Нарушение compliance (GDPR, PCI DSS)

❌ Hotfix НЕ нужен (ждём следующий релиз):
- Косметический баг
- Баг, затрагивающий < 1% пользователей с workaround
- Проблема производительности, не влияющая на функциональность
- Неудобство UX без потери функциональности
```

### Процесс hotfix

```
1. Инцидент обнаружен
   │
   ▼
2. Оценка severity (QA + Dev + PO) — 15-30 минут
   │
   ├── Не критично → Заводим баг, планируем в следующий релиз
   │
   └── Критично → Начинаем hotfix
         │
         ▼
3. Разработка fix (из release/main branch)
   │
   ▼
4. QA: Targeted Testing
   │   ├── Проверка исправления бага
   │   ├── Smoke-тест затронутой области
   │   └── Regression smoke (автоматизированный, 15-30 мин)
   │
   ▼
5. Деплой на staging → Smoke → Деплой на production
   │
   ▼
6. Post-deploy verification (10-15 минут)
   │
   ▼
7. Merge hotfix обратно в develop/main
   │
   ▼
8. Post-mortem (в течение 1-3 дней)
```

**Что тестировать при hotfix:**
- Только целевое исправление
- Smoke-тест ближайших затронутых функций
- Автоматизированный smoke всего приложения
- НЕ нужен full regression — это заблокирует hotfix

> **Опытное мнение:** после каждого hotfix нужен post-mortem. Не для поиска виновных, а для улучшения процесса. Вопросы: "Почему баг не поймали до релиза?" и "Как предотвратить подобное в будущем?" Обычно ответ — дополнительный тест-кейс или улучшение мониторинга.

### Структура Release Notes

```markdown
# Release Notes — v2.5.0 (2026-03-20)

## Новые возможности
- [FEAT-123] Добавлена двухфакторная аутентификация
- [FEAT-130] Экспорт отчётов в формате PDF

## Исправления
- [BUG-401] Исправлена ошибка расчёта налога для заказов > $1000
- [BUG-415] Исправлено отображение кириллицы в PDF-отчётах

## Известные проблемы
- [BUG-456] Медленная загрузка отчётов > 10K строк
  Workaround: используйте CSV-экспорт для больших отчётов
- [BUG-461] Неверный формат даты для zh-CN локали
  Статус: исправление запланировано на v2.5.1

## Области риска (внутренний документ)

- Миграция БД: добавлен новый индекс на таблицу orders.
  Риск: возможно временное увеличение response time при высокой нагрузке.
```

**Что QA добавляет:**
- Known Issues с workarounds — честность перед пользователями и support-командой
- Области риска — внутренний документ для DevOps/SRE, чтобы знали, куда смотреть
- Тестовое покрытие новых фич — для уверенности команды

## Критерии отката (Rollback)

### Когда откатывать релиз

Откат — болезненное решение, но иногда необходимое. Критерии должны быть определены ДО релиза, а не в момент паники.

**Автоматический откат (если настроен):**

```
- Error rate > 5% в течение 5 минут после деплоя
- Health check endpoint не отвечает 3 раза подряд
- P95 response time > 5x от baseline
```

**Ручной откат (решение принимает команда):**

```
ROLLBACK, если:
├── Критический бизнес-поток сломан (оплата, регистрация, core feature)
├── Потеря данных пользователей
├── Уязвимость безопасности
├── > 20% пользователей затронуты
└── Fix займёт > 2 часов (быстрее откатить и починить спокойно)

НЕ ОТКАТЫВАТЬ, если:
├── Баг затрагивает < 1% пользователей
├── Есть workaround
├── Fix готов за < 30 минут
├── Откат сам по себе рискован (необратимая миграция БД)
└── Проблема косметическая
```

**Важно:** план отката нужно тестировать! QA должен убедиться, что:
1. Процедура отката задокументирована
2. Откат выполняется за предсказуемое время
3. Данные, созданные после деплоя, не теряются при откате (или потеря приемлема)
4. Миграции БД обратимы (или есть план, если необратимы)

## Canary Releases и Feature Flags

### Canary Releases

Вместо выкатки на 100% пользователей сразу — сначала на 1-5%, затем постепенно увеличиваем.

```
Этап 1: 1% трафика     → мониторинг 1-2 часа
Этап 2: 5% трафика     → мониторинг 2-4 часа
Этап 3: 25% трафика    → мониторинг 4-8 часов
Этап 4: 50% трафика    → мониторинг 8-12 часов
Этап 5: 100% трафика   → полный мониторинг 24-72 часа
```

**Как это меняет роль QA:**
- Smoke-тест на каждом этапе canary
- Мониторинг метрик (error rate, latency) для canary-группы vs. остальных
- Определение критериев для продвижения на следующий этап или отката
- Тестирование совместимости: canary-пользователи и основные пользователи работают с одной БД

### Feature Flags

Feature flags позволяют включать/выключать функциональность без деплоя.

```java
// Пример использования feature flag
if (featureFlags.isEnabled("new-payment-flow", user)) {
    // Новый платёжный поток — включён для тестовой группы
    return newPaymentService.processPayment(order);
} else {
    // Старый платёжный поток — для остальных пользователей
    return legacyPaymentService.processPayment(order);
}
```

**Дополнительная нагрузка на QA:**

```
Без feature flags:                С feature flags:
─────────────────                 ──────────────────────
Тестируем 1 версию                Тестируем 2+ версии:
                                  ├── Flag ON (новый код)
                                  ├── Flag OFF (старый код)
                                  ├── Переключение ON → OFF
                                  ├── Переключение OFF → ON
                                  └── Частичное включение
                                      (у одних ON, у других OFF)
```

**Рекомендации для QA:**
1. Вести реестр всех feature flags и их статуса
2. Тестировать оба состояния каждого flag
3. Тестировать переключение flag в runtime (без рестарта)
4. Следить за "техническим долгом": старые flags нужно удалять после полного раскатывания
5. Учитывать комбинации flags (если flag A и flag B влияют на один модуль)

## Continuous Deployment: QA без "дня релиза"

### Как меняется роль QA

В Continuous Deployment каждый коммит, прошедший все проверки, автоматически попадает в production. "Дня релиза" нет — релиз происходит десятки раз в день.

**Традиционный процесс vs. Continuous Deployment:**

```
ТРАДИЦИОННЫЙ:
  Код → Manual QA → Staging → Go/No-Go → Production
  Цикл: 2-4 недели

CONTINUOUS DEPLOYMENT:
  Код → Автотесты → Staging (автоматически) → Canary → Production
  Цикл: минуты-часы
```

**Что QA делает в мире Continuous Deployment:**

| Было (традиционно)              | Стало (CD)                                  |
|----------------------------------|---------------------------------------------|
| Ручной regression перед релизом  | Автоматизированный regression в CI/CD       |
| Go/No-Go meeting                 | Автоматические quality gates                |
| Smoke на staging вручную         | Автоматический smoke после каждого деплоя   |
| Post-release monitoring вручную  | Автоматические алерты + dashboards          |
| Release notes раз в спринт      | Changelog генерируется автоматически        |
| QA тестирует перед релизом       | QA создаёт тесты ДО/ВО ВРЕМЯ разработки    |

### Quality Gates в CI/CD Pipeline

```
Commit → Build → Unit Tests → Integration Tests → Deploy Staging
                                                        │
                                    ┌───────────────────┘
                                    ▼
                              E2E Tests → Security Scan → Performance Check
                                                               │
                                              ┌────────────────┘
                                              ▼
                                     Canary Deploy (1%)
                                              │
                                              ▼
                                     Health Check (авто)
                                              │
                                    ┌─────────┴─────────┐
                                    ▼                   ▼
                               OK → Rollout         FAIL → Auto Rollback
                              (5% → 25% → 100%)
```

**Каждый quality gate — ответственность QA:**

```
Gate 1: Unit Tests           — QA review тестов + контроль coverage
Gate 2: Integration Tests    — QA пишет/ревьюит интеграционные сценарии
Gate 3: E2E Tests            — QA владеет E2E framework
Gate 4: Security Scan        — QA + Security совместно определяют правила
Gate 5: Performance Check    — QA определяет baseline и пороги
Gate 6: Health Check         — QA определяет, что проверять
Gate 7: Canary Metrics       — QA определяет критерии успеха/отката
```

### Shift-Left в контексте CD

В Continuous Deployment QA не может тестировать "после разработки" — времени нет. Поэтому тестирование сдвигается влево:

```
Традиционно:       Design → Dev → QA → Deploy
                                   ^^^ тестирование здесь

Shift-Left (CD):   Design+QA → Dev+QA → Auto-Test → Auto-Deploy
                    ^^^^^^^^    ^^^^^^
                    Тестирование начинается здесь
```

**Конкретные практики:**
- QA участвует в design review: находит edge cases до написания кода
- QA пишет acceptance criteria в формате BDD до начала разработки
- Разработчик и QA pair на сложных фичах
- Тест-кейсы создаются параллельно с кодом, а не после

## Связь с тестированием

Релизный процесс — это контекст, в котором все виды тестирования обретают практический смысл:

1. **Regression testing** — основа уверенности, что старое не сломано
2. **Smoke testing** — быстрая проверка работоспособности на каждом окружении
3. **Exploratory testing** — поиск неочевидных проблем на стыке нового и старого кода
4. **Performance testing** — baseline для post-release мониторинга
5. **Security testing** — обязательный quality gate перед production
6. **Автоматизация** — единственный способ масштабировать проверки в Continuous Deployment

## Типичные ошибки

1. **Нет чёткого Feature Freeze.** Разработчики продолжают мержить фичи "ещё вот это одно" → regression нестабилен, тестирование превращается в стрельбу по движущейся мишени
2. **Staging не соответствует production.** Тесты на staging проходят, а на production — сюрприз. QA должен постоянно отмечать расхождения и требовать их устранения
3. **Нет критериев отката до деплоя.** Решение об откате принимается в панике, без критериев. Это приводит к затягиванию решения и увеличению ущерба
4. **Hotfix без тестирования.** "Это же одна строчка, что тут ломаться!" — знаменитые последние слова перед production outage
5. **Отсутствие post-mortem.** Hotfix выпущен, проблема решена, все забыли. Через месяц — аналогичная ситуация
6. **QA не участвует в Go/No-Go.** Решение о релизе принимает менеджмент без input от QA → выкатывается сырой продукт
7. **Feature flags без учёта QA.** Разработчики добавляют flags, QA не знает о них, тестирование проходит с одним состоянием, а в production включают другое
8. **Переход на CD без автоматизации.** "Мы теперь Continuous Deployment!" — но автоматизированных тестов нет, и каждый деплой — это русская рулетка

## Вопросы на интервью

### 🟢 Базовый уровень

**В: Опишите типичный релизный процесс и роль QA на каждом этапе.**
О: Типичный процесс: Feature Freeze → Regression Testing → Smoke на Staging → Go/No-Go → Production Deploy → Post-Release Monitoring. QA участвует на каждом этапе: при freeze — проверяет scope и готовит regression suite; во время regression — выполняет тесты и классифицирует найденные баги; на staging — smoke-тестирование; на Go/No-Go — представляет release readiness report с рекомендацией; после деплоя — smoke на production и мониторинг первые 24-72 часа.

**В: Что такое Smoke Testing и когда его нужно выполнять?**
О: Smoke Testing — минимальный набор тестов, проверяющих, что приложение запускается и основные потоки работают. Выполняется: после каждого деплоя на любое окружение, как первый шаг regression testing, сразу после production-деплоя, после hotfix. Smoke должен быть автоматизирован и выполняться за 15-30 минут.

**В: Когда нужен hotfix, а когда баг может подождать до следующего релиза?**
О: Hotfix нужен при: потере данных, невозможности выполнения основного бизнес-потока (оплата, регистрация), уязвимости безопасности, сбое, затрагивающем > 10% пользователей. Может подождать: косметические баги, проблемы с workaround, баги затрагивающие < 1% пользователей, не влияющие на функциональность UX-неудобства.

### 🟡 Средний уровень

**В: Как вы принимаете решение Go/No-Go? Что должен содержать ваш отчёт?**
О: Отчёт включает: общую статистику тестирования (executed, passed, failed, blocked), список открытых дефектов по severity, известные проблемы с workarounds, области повышенного риска, рекомендацию (Go/No-Go) с обоснованием. Рекомендую No-Go при: открытых Blocker/Critical дефектах, неработающем критическом бизнес-потоке, недостаточном объёме тестирования. Финальное решение — за Product Owner, но QA обязан дать честную экспертную оценку.

**В: Как Feature Flags влияют на стратегию тестирования?**
О: Feature Flags увеличивают объём тестирования: нужно проверять оба состояния (ON/OFF), переключение в runtime, совместимость данных между состояниями, комбинации flags. QA должен вести реестр flags, включать проверку flags в regression, следить за удалением неактуальных flags. Преимущество: возможность быстрого "отката" конкретной фичи без деплоя.

**В: Какие критерии отката вы бы определили для релиза e-commerce приложения?**
О: Автоматический откат: error rate > 5% в течение 5 минут, health check не проходит 3 раза подряд. Ручной откат: сломана оплата или оформление заказа, потеря данных пользователей, уязвимость. Не откатывать: косметические баги, проблемы < 1% пользователей, если fix готов за 30 минут. Критерии определяются и согласовываются ДО релиза, задокументированы в runbook, доступны всей команде.

### 🔴 Продвинутый уровень

**В: Как QA адаптируется при переходе от ручного релиза к Continuous Deployment?**
О: Фундаментальный сдвиг: от gate-keeper к quality enabler. Конкретные изменения: (1) Автоматизация становится обязательной — ручной regression невозможен при десятках деплоев в день. (2) Quality gates в CI/CD заменяют ручные проверки — QA определяет критерии gates, а не выполняет проверки сам. (3) Shift-left — QA участвует в design и пишет тесты параллельно с кодом. (4) Мониторинг production становится частью тестирования — observability важнее pre-release проверок. (5) Feature flags заменяют release branches — QA тестирует flags, а не релизы. Переход занимает 6-12 месяцев и требует инвестиций в инфраструктуру, обучение и культурные изменения.

**В: На проекте выходит 3-4 hotfix в неделю. Это нормально? Что бы вы предложили?**
О: Это сигнал системной проблемы. 3-4 hotfix в неделю означает, что стандартный процесс тестирования не ловит критические баги. Мой план: (1) Провести анализ всех hotfix за последний месяц — классифицировать по типу (regression, не покрыто тестами, environment-specific, edge case). (2) По каждой категории — корневая причина. Обычно: недостаточный regression, staging не соответствует production, нет E2E для критических потоков. (3) Конкретные actions: добавить тест-кейсы для каждого hotfix (чтобы не повторялись), улучшить staging-окружение, внедрить canary-деплой, усилить code review для критических модулей. (4) Метрика успеха: снижение hotfix до 1 или менее в неделю за 2-3 месяца. Если hotfix-ы разных типов — возможно, нужны разные меры для каждого.

### Задание 1: Составление Release Readiness Report

Ситуация: вы QA Lead на проекте e-commerce. Через 2 дня — релиз v3.0 с тремя новыми фичами: программа лояльности, новый чекаут, интеграция с новым провайдером доставки. Regression выявил следующее:
- 2 Critical бага в программе лояльности
- 1 Major баг в новом чекаут (исправлен, ждёт перепроверку)
- Smoke на staging пройден успешно
- Интеграция с доставкой работает нестабильно (таймауты 10% запросов)

Задача: составить Release Readiness Report и дать рекомендацию Go/No-Go.

### Задание 2: Проектирование Hotfix-процесса

Задача:
1. Спроектировать hotfix-процесс, адекватный размеру команды
2. Определить минимальный набор проверок для hotfix
3. Описать, как предотвратить повторение проблемы "hotfix вносит новый баг"

### Задание 3: Стратегия тестирования с Feature Flags

У вас приложение с 5 активными feature flags:
- `new-search-algorithm` (влияет на поиск)
- `premium-subscription` (влияет на оплату и профиль)
- `dark-mode` (влияет на UI)
- `new-notification-engine` (влияет на email и push)
- `ai-recommendations` (влияет на каталог и поиск)

Задача:
1. Определить, какие flags могут конфликтовать друг с другом
2. Сколько комбинаций нужно тестировать? Как сократить без потери качества?
3. Составить минимальный тест-план для релиза с включением `premium-subscription`

### Задание 4: Миграция на Continuous Deployment

Ситуация: компания переходит с ежемесячного релиза на Continuous Deployment. Текущее состояние: 200 manual regression тест-кейсов, 50 автотестов (unit), 0 E2E-автотестов, нет staging-окружения с автодеплоем.

Задача:
1. Составить план миграции на 6 месяцев (что делать в каждом месяце)
2. Определить quality gates для CI/CD pipeline
3. Определить, какие из 200 manual тест-кейсов автоматизировать в первую очередь и почему
4. Описать, от чего QA НЕ должен отказываться даже при полной автоматизации
