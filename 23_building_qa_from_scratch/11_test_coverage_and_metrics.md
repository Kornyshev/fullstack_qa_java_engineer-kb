# Тестовое покрытие и метрики качества

## Обзор

Метрики — это язык, на котором QA разговаривает с бизнесом. Без измеримых показателей тестирование превращается в "мы вроде проверили". С правильными метриками — в управляемый инженерный процесс, где решения принимаются на основе данных, а не интуиции.

Однако метрики — это обоюдоострый инструмент. Неправильно выбранные или неверно интерпретированные показатели наносят больше вреда, чем их полное отсутствие. Классический пример: команда гонится за 100% code coverage и пишет бессмысленные тесты, которые проверяют геттеры и сеттеры, вместо того чтобы покрыть критические бизнес-сценарии.

В этом разделе разберём, что измерять, как измерять, и — самое важное — когда перестать измерять.

---

## Виды тестового покрытия

### Requirements Coverage (покрытие требований)

Самый важный вид покрытия для QA. Отвечает на вопрос: "Все ли требования проверены хотя бы одним тест-кейсом?"

**Формула:**

```
Requirements Coverage = (Количество требований с тест-кейсами / Общее количество требований) × 100%
```

**Практический пример — Traceability Matrix:**

| Requirement ID | Описание                     | Test Cases     | Статус    |
|----------------|------------------------------|----------------|-----------|
| REQ-001        | Регистрация пользователя     | TC-001, TC-002 | Покрыто   |
| REQ-002        | Авторизация через OAuth      | TC-003         | Покрыто   |
| REQ-003        | Восстановление пароля        | —              | Не покрыто |
| REQ-004        | Двухфакторная аутентификация | TC-004, TC-005 | Покрыто   |

**Целевой показатель:** 100% для critical и high priority требований, 80%+ для medium.

**Проблема:** требования часто неформализованы. Если у вас нет чёткого списка требований — начните с его создания, иначе покрытие не измерить.

### Code Coverage (покрытие кода)

Измеряется автоматически инструментами (JaCoCo, Istanbul, Coverage.py). Показывает, какой процент кода выполняется при запуске тестов.

**Уровни code coverage (от простого к сложному):**

```
Line Coverage (покрытие строк)
─────────────────────────────
Самый базовый. Какой процент строк кода был выполнен.

Branch Coverage (покрытие ветвей)
─────────────────────────────────
Все ли ветви условных операторов (if/else, switch) были пройдены.

Condition Coverage (покрытие условий)
─────────────────────────────────────
Каждое булево подвыражение принимало значения true и false.

Path Coverage (покрытие путей)
──────────────────────────────
Все возможные пути выполнения кода пройдены. На практике недостижимо
для сложного кода из-за комбинаторного взрыва.
```

**Пример разницы между Line и Branch Coverage:**

```java
// Метод для расчёта скидки
public double calculateDiscount(User user, Order order) {
    double discount = 0;                          // Строка 1
    if (user.isPremium() && order.getTotal() > 100) {  // Строка 2
        discount = 0.15;                          // Строка 3
    }
    return discount;                              // Строка 4
}
```

```
Тест: calculateDiscount(premiumUser, orderWith150)
  Line Coverage: 100% (все 4 строки выполнены)
  Branch Coverage: 50% (ветвь else не проверена)

Нужен второй тест: calculateDiscount(regularUser, orderWith50)
  Теперь Branch Coverage: 100%

Но Condition Coverage всё ещё не 100%!
  Не проверен случай: premium user + order <= 100
  И: non-premium user + order > 100
```

**Что считается "good enough" code coverage:**

| Тип кода                    | Целевой Line Coverage | Целевой Branch Coverage |
|-----------------------------|----------------------|------------------------|
| Бизнес-логика (core)        | 80-90%               | 70-80%                 |
| Утилитарный код             | 70-80%               | 60-70%                 |
| Контроллеры / API endpoints | 70-80%               | 60-70%                 |
| Конфигурация, DTO, модели   | Не гнаться           | Не гнаться             |
| Сгенерированный код         | Исключить из отчёта  | Исключить из отчёта    |

> **Опытное мнение:** 80% line coverage для бизнес-логики — отличный баланс. Переход от 80% к 95% обычно требует непропорционально больших усилий и даёт минимальную отдачу. Оставшиеся 20% часто содержат обработку редких ошибок, которую эффективнее проверять другими способами.

### Risk Coverage (покрытие рисков)

Привязывает тестирование к бизнес-рискам. Наиболее зрелый подход.

**Матрица рисков:**

```
              Высокая вероятность    Низкая вероятность
              ┌─────────────────────┬─────────────────────┐
Высокий       │   КРИТИЧНО          │   ВАЖНО             │
ущерб         │   Максимум тестов   │   Ключевые сценарии │
              │   + автоматизация   │   + мониторинг      │
              ├─────────────────────┼─────────────────────┤
Низкий        │   УМЕРЕННО          │   МИНИМАЛЬНО        │
ущерб         │   Основные сценарии │   Smoke-тесты       │
              │                     │   или пропустить    │
              └─────────────────────┴─────────────────────┘
```

**Пример для e-commerce:**
- Высокий ущерб + высокая вероятность: оплата заказа (баг = потерянные деньги, частый сценарий)
- Высокий ущерб + низкая вероятность: сбой при одновременных 10K заказах (катастрофа, но редко)
- Низкий ущерб + высокая вероятность: опечатка в footer (часто меняется, не критично)
- Низкий ущерб + низкая вероятность: некорректное отображение на устаревшем браузере

### API Coverage (покрытие API)

Специфический вид покрытия для backend-тестирования. Строится на пересечении трёх измерений.

**Матрица API Coverage:**

```
Endpoint          │ GET  │ POST │ PUT  │ DELETE │
──────────────────┼──────┼──────┼──────┼────────┤
/api/users        │ ✅   │ ✅   │ N/A  │ N/A    │
/api/users/{id}   │ ✅   │ N/A  │ ✅   │ ❌     │
/api/orders       │ ✅   │ ✅   │ N/A  │ N/A    │
/api/orders/{id}  │ ✅   │ N/A  │ ❌   │ ❌     │
```

Но этого недостаточно! Для каждой комбинации Endpoint × Method нужны сценарии:

```
POST /api/users:
  ✅ Успешная регистрация (201)
  ✅ Дублирующий email (409)
  ✅ Невалидные данные (400)
  ❌ Превышение rate limit (429)        ← не покрыто
  ❌ Сервер авторизации недоступен (503) ← не покрыто
```

## Traceability Matrix: от требований до багов

Traceability Matrix (матрица трассируемости) связывает всю цепочку: требование → тест-кейс → результат → дефект.

**Полная структура:**

```
Requirement → Test Case → Test Run Result → Defect (если найден)
    │              │              │               │
  REQ-042      TC-117, TC-118   PASS/FAIL      BUG-205
```

**Зачем это нужно:**
1. **Анализ воздействия:** изменилось требование — видно, какие тесты затронуты
2. **Поиск пробелов:** видно непокрытые требования
3. **Привязка багов:** видно, какие требования проблемные
4. **Отчётность:** можно показать руководству процент готовности фичи

**Практический пример в таблице:**

| Req ID  | Requirement             | Test Cases         | Last Run | Defects  | Status       |
|---------|-------------------------|--------------------|----------|----------|--------------|
| REQ-101 | Создание заказа         | TC-201, TC-202     | PASS     | —        | Готово       |
| REQ-102 | Оплата картой           | TC-203, TC-204, TC-205 | FAIL | BUG-301  | Блокировано  |
| REQ-103 | Отмена заказа           | TC-206             | PASS     | —        | Готово       |
| REQ-104 | Возврат средств         | —                  | —        | —        | Не покрыто   |

> **Инструменты:** в реальных проектах matрица живёт в TestRail, Zephyr, Allure TestOps или даже в хорошо организованных Google Sheets. Главное — она должна обновляться, иначе бесполезна.

## Ключевые метрики качества

### Defect Density (плотность дефектов)

```
Defect Density = Количество дефектов / Размер модуля (KLOC или Story Points)
```

**Зачем:** выявить "проблемные" модули. Если модуль А имеет 15 багов на 1000 строк, а модуль Б — 2, то модуль А требует дополнительного внимания (ревью кода, рефакторинг, больше тестов).

**Нормы:** типичный показатель для production-кода — 1-5 дефектов на KLOC. Выше 10 — серьёзная проблема с качеством кода.

### Test Pass Rate (процент прохождения тестов)

```
Test Pass Rate = (Пройденные тесты / Общее количество запущенных тестов) × 100%
```

**Важно:** отслеживать тренд, а не абсолютное значение.

```
Sprint 1:  ████████████████░░░░ 82%   ← начало проекта, нормально
Sprint 5:  ██████████████████░░ 91%   ← улучшение, тренд правильный
Sprint 8:  ███████████████░░░░░ 78%   ← деградация! Сигнал к разбору
Sprint 9:  ████████████░░░░░░░░ 63%   ← критическая ситуация
```

### Automation Coverage (покрытие автоматизацией)

```
Automation Coverage = (Автоматизированные тесты / Общее количество тестов) × 100%
```

**Целевые показатели по уровням пирамиды:**

| Уровень         | Целевая автоматизация |
|-----------------|----------------------|
| Unit-тесты      | 90-100%              |
| API/Integration | 70-90%               |
| UI/E2E          | 30-50%               |
| Exploratory     | 0% (по определению)  |

### Defect Leakage (утечка дефектов)

Самая болезненная метрика для QA — показывает, сколько багов "пробили" тестирование и попали в production.

```
Defect Leakage = (Баги, найденные в production / Все найденные баги) × 100%
```

**Нормы:**
- < 5% — отличный результат
- 5-15% — приемлемо для большинства проектов
- \> 15% — процесс тестирования нужно серьёзно пересмотреть

### Test Execution Time (время выполнения тестов)

Критично для CI/CD pipeline. Если тесты выполняются 3 часа — разработчики не будут ждать и будут мержить без прогона.

**Ориентиры:**

```
Unit-тесты:        < 5 минут   (блокирующая проверка на каждый коммит)
API/Integration:   < 15 минут  (блокирующая проверка на каждый PR)
E2E:               < 30 минут  (запуск по расписанию или перед релизом)
Полный регресс:    < 2 часов   (ночной прогон)
```

### Flaky Test Rate (процент нестабильных тестов)

```
Flaky Test Rate = (Тесты с нестабильным результатом / Общее количество тестов) × 100%
```

**Допустимый уровень:** < 2%. Если flaky-тестов больше 5% — доверие к автоматизации разрушается, и команда начинает игнорировать падения.

**Как отслеживать:** пометить тест как flaky, если он показал разный результат на одном и том же коммите за последние N прогонов. Allure, ReportPortal, встроенные механизмы CI (retry) помогают это визуализировать.

## Подход "Good Enough" к покрытию

### Принцип Парето в тестировании

80% багов находятся в 20% кода. Поэтому стратегия "покрыть всё равномерно" — неэффективна.

**Прагматичный подход:**

```
1. Определить критические бизнес-процессы         → покрытие 90%+
2. Определить часто изменяемые модули              → покрытие 80%+
3. Стабильный, редко меняющийся код                → покрытие 60-70%
4. Конфигурация, DTO, сгенерированный код          → не тратить время
```

### Когда НЕ нужно увеличивать покрытие

- Код планируется к удалению или рефакторингу
- Сгенерированный код (protobuf, OpenAPI models)
- Тривиальные геттеры/сеттеры без логики
- Legacy-код без документации (сначала нужны спецификации, потом тесты)

### Когда покрытие ОБЯЗАТЕЛЬНО высокое

- Финансовые расчёты (деньги, налоги, скидки)
- Аутентификация и авторизация
- Обработка персональных данных
- Интеграции с платёжными системами
- Код, связанный с безопасностью

## Как презентовать метрики руководству

### Правило: тренды, а не числа

Менеджеры не понимают "branch coverage 73.2%". Они понимают "за последний квартал качество выросло на 20%, количество production-инцидентов снизилось вдвое".

**Эффективный Dashboard:**

```
┌──────────────────────────────────────────────────────┐
│  QUALITY DASHBOARD — Sprint 15                       │
├──────────────────────────────────────────────────────┤
│                                                      │
│  Production Incidents:  3 → 1  (↓ 67%)  ✅          │
│  Defect Leakage:       12% → 7% (↓ тренд)  ✅      │
│  Automation Coverage:  45% → 62% (↑ тренд)  ✅      │
│  Avg Bug Fix Time:     4.2d → 3.1d (↓ лучше)  ✅   │
│  Flaky Tests:          8% → 3% (↓ тренд)  ⚠️       │
│  Regression Time:      3.5h → 1.2h (↓ лучше)  ✅   │
│                                                      │
│  TOP RISK: Модуль платежей — 5 open Critical bugs   │
│  ACTION: Дополнительное тестирование перед релизом   │
└──────────────────────────────────────────────────────┘
```

### Привязка метрик к бизнес-ценности

| QA-метрика            | Перевод для бизнеса                                       |
|-----------------------|-----------------------------------------------------------|
| Defect Leakage ↓      | Меньше жалоб клиентов, меньше возвратов                  |
| Test Execution Time ↓ | Быстрее выпускаем фичи (time to market)                  |
| Automation Coverage ↑ | Меньше ручной работы = меньше затрат на регресс           |
| Flaky Tests ↓         | Разработчики не тратят время на ложные срабатывания       |
| Defect Density ↓      | Меньше затрат на исправление багов после релиза           |

## Чего НЕ нужно измерять (vanity metrics)

### Количество написанных тест-кейсов

"Мы написали 500 тестов!" — это ничего не говорит о качестве. 50 хорошо спроектированных тестов могут быть эффективнее 500 бездумных.

### Code coverage ради code coverage

Тесты вроде этого приносят coverage, но не приносят пользы:

```java
// Бесполезный тест ради покрытия
@Test
void testGettersAndSetters() {
    User user = new User();
    user.setName("Test");
    assertEquals("Test", user.getName()); // Что мы здесь тестируем? Язык Java?
}
```

### Количество найденных багов как KPI тестировщика

Это провоцирует заведение мелких или дублирующих дефектов. QA должен находить **важные** баги, а не максимум багов.

### Процент автоматизации без учёта качества тестов

100% автоматизированных, но нестабильных, медленных и плохо поддерживаемых тестов хуже, чем 60% надёжных и быстрых.

## Связь с тестированием

Метрики — это инструмент управления тестированием, а не самоцель. Они помогают:

1. **Планировать тестирование:** метрика defect density показывает, где сфокусировать усилия
2. **Оценивать готовность к релизу:** набор Exit Criteria на основе метрик
3. **Улучшать процесс:** ретроспективный анализ метрик спринт за спринтом
4. **Обосновывать ресурсы:** "для снижения defect leakage с 15% до 5% нужен ещё один QA-инженер на 3 месяца"
5. **Выявлять системные проблемы:** резкий рост flaky-тестов может указывать на проблемы с инфраструктурой

---

## Типичные ошибки

1. **Измерять всё подряд.** Выберите 3-5 ключевых метрик. Больше — информационный шум
2. **Не учитывать контекст.** 70% coverage для стартапа — отлично. Для банковского ПО — недостаточно
3. **Сравнивать метрики между командами.** У каждой команды свой контекст, технологии, сложность
4. **Не пересматривать метрики.** То, что было важно полгода назад, может быть неактуально сейчас
5. **Делать метрики KPI отдельных людей.** Это приводит к манипуляции показателями
6. **Игнорировать flaky-тесты.** "У нас 95% pass rate" — но 10% тестов flaky и просто перезапускаются
7. **Не визуализировать тренды.** Абсолютные числа без динамики бесполезны
8. **Путать coverage и quality.** Высокое покрытие не гарантирует отсутствие багов — оно гарантирует, что код выполнялся, но не что он проверен корректно

## Вопросы на интервью

### 🟢 Базовый уровень

**В: Что такое test coverage? Какие виды покрытия вы знаете?**
О: Test coverage — метрика, показывающая, какая часть тестируемого объекта проверена тестами. Виды: requirements coverage (процент покрытых требований), code coverage (line, branch, condition — процент выполненного кода), risk-based coverage (покрытие относительно бизнес-рисков). Наиболее важно requirements coverage, потому что оно привязано к тому, что пользователь реально ожидает от системы.

**В: Чем отличается line coverage от branch coverage?**
О: Line coverage показывает процент выполненных строк кода. Branch coverage — процент пройденных ветвей условных операторов. Branch coverage строже: можно иметь 100% line coverage, но менее 100% branch coverage, если не все ветви if/else были пройдены.

**В: Какой процент code coverage считается достаточным?**
О: Универсального ответа нет, но прагматичный ориентир — 80% line coverage для бизнес-логики. Критический код (финансы, безопасность) — 90%+. Для DTO и конфигураций специально добиваться высокого покрытия не нужно. Важнее качество тестов, чем процент покрытия.

### 🟡 Средний уровень

**В: Что такое Traceability Matrix и зачем она нужна?**
О: Traceability Matrix — документ, связывающий требования с тест-кейсами, результатами тестирования и дефектами. Нужна для: выявления непокрытых требований, анализа влияния изменений (какие тесты обновить при изменении требования), отчётности о готовности (какой процент требований протестирован и работает корректно).

**В: Какие метрики качества вы бы отслеживали на проекте?**
О: Основные: defect leakage (% багов, дошедших до production), test pass rate (тренд прохождения тестов), automation coverage (доля автоматизированных тестов), flaky test rate (процент нестабильных тестов), regression execution time (скорость обратной связи). Для каждой метрики определяются пороговые значения и отслеживается динамика.

**В: Как бы вы объяснили руководству, зачем инвестировать в автоматизацию тестирования?**
О: Через бизнес-язык: "Сейчас регрессионное тестирование занимает 3 дня ручной работы двух QA перед каждым релизом. С автоматизацией — 1 час машинного времени. Это позволит релизить еженедельно вместо ежемесячно и освободит QA для exploratory-тестирования, которое находит баги, не покрываемые стандартными проверками." Далее показать ROI: стоимость автоматизации vs. стоимость ручного регресса × количество релизов.

### 🔴 Продвинутый уровень

**В: Вы пришли на проект, где code coverage 95%, но production-баги всё равно регулярны. В чём может быть проблема?**
О: Высокий coverage не равен качеству тестов. Возможные причины: (1) тесты проверяют что код выполняется, но не проверяют корректность результата (assertions отсутствуют или слабые); (2) тесты не покрывают интеграционные сценарии и граничные случаи; (3) тесты написаны на основе текущей реализации, а не спецификации — они подтверждают баги, а не ловят их; (4) не тестируются нефункциональные аспекты: нагрузка, безопасность, конкурентный доступ; (5) environment-specific проблемы: staging не соответствует production. Нужно провести аудит: проанализировать production-баги за 3 месяца, сопоставить с существующими тестами и выявить паттерны пробелов.

**В: Как вы определите оптимальный набор метрик для нового проекта?**
О: Начну с вопроса: "Какие решения мы будем принимать на основе этих метрик?" Для стартапа на ранней стадии — минимум: defect leakage и test execution time (быстрая обратная связь важнее покрытия). Для зрелого enterprise-продукта — полный набор: defect density, coverage by risk, automation ROI, mean time to detect/fix. Каждую метрику ввожу поэтапно: начинаю с baseline, устанавливаю цель, отслеживаю тренд. Через 2-3 спринта — ретроспектива: какие метрики реально влияют на решения, а какие стали "шумом". Последние убираю.

## Практические задания

### Задание 1: Анализ покрытия требований

Дан набор из 10 пользовательских историй для интернет-магазина (регистрация, каталог, корзина, оплата, доставка). Для каждой истории написаны от 0 до 5 тест-кейсов. Задача:
1. Построить Traceability Matrix
2. Определить непокрытые требования
3. Приоритизировать покрытие: что покрывать в первую очередь и почему
4. Рассчитать Requirements Coverage до и после ваших дополнений

### Задание 2: Интерпретация code coverage отчёта

Дан JaCoCo-отчёт, показывающий:
- Модуль `payment`: line 92%, branch 85%
- Модуль `catalog`: line 45%, branch 30%
- Модуль `notification`: line 78%, branch 65%
- Модуль `admin-panel`: line 20%, branch 10%

Задача:
1. Какой модуль требует внимания в первую очередь и почему?
2. Для какого модуля низкое покрытие может быть приемлемо?
3. Составьте план повышения покрытия с приоритетами и обоснованием

### Задание 3: Dashboard для руководства

Команда за последние 5 спринтов имеет следующие данные:
- Defect Leakage: 18%, 15%, 12%, 14%, 9%
- Automation Coverage: 30%, 35%, 42%, 48%, 55%
- Flaky Tests: 12%, 10%, 8%, 9%, 4%
- Regression Time: 4h, 3.5h, 3h, 2.5h, 1.5h

Задача:
1. Визуализировать тренды (можно текстом / ASCII-графиком / любым инструментом)
2. Написать краткий отчёт для менеджера (3-5 предложений на бизнес-языке)
3. Определить, какая метрика вызывает беспокойство (подсказка: спринт 4 vs спринт 3)
4. Предложить Action Items на следующий спринт

### Задание 4: Аудит метрик

На проекте используются следующие KPI для QA-команды:
- Количество написанных тест-кейсов в спринт
- Количество найденных багов в спринт
- Code coverage > 90%

Задача:
1. Объяснить, почему каждая из этих метрик может быть вредной как KPI
2. Предложить альтернативный набор метрик с обоснованием
3. Описать, как переход к новым метрикам представить руководству
