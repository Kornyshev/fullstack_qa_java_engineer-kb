# Уровни тестирования: что, когда и почему

## Обзор

Уровни тестирования — это не абстрактная теория из учебника ISTQB.
Это **практический инструмент**, который определяет, какие дефекты вы ловите, на каком этапе и за какую цену.

Типичная ошибка новичка: "У нас есть Selenium-тесты, значит, мы тестируем".
Типичная ошибка компании: "Давайте напишем 2000 E2E-тестов и будем спать спокойно".

Реальность: каждый уровень тестирования решает свою задачу.
Пропуск уровня не компенсируется удвоением усилий на другом.

В этом разделе мы разберём каждый уровень глубоко, с позиции QA-инженера,
который должен **принимать решения**, а не просто знать определения.

## Unit Testing: что должен знать QA

### Зачем QA разбираться в unit-тестах?

Многие QA считают: "Unit-тесты — это дело разработчиков". Формально — да.
Но на практике:

1. **Вы ревьюите покрытие** — если разработчик говорит "я покрыл unit-тестами",
   вы должны уметь оценить, что именно покрыто.
2. **Вы пишете unit-тесты** — в некоторых командах QA-автоматизатор пишет тесты
   на уровне сервисов, утилитных классов, валидаторов.
3. **Вы принимаете решения** — "стоит ли автоматизировать этот сценарий на UI
   или достаточно unit-теста?"

### Когда QA пишет unit-тесты

| Ситуация | Пример |
|----------|--------|
| Тестирование утилитных методов | Валидация email, форматирование даты |
| Тестирование бизнес-логики в сервисах | Расчёт скидки, проверка прав доступа |
| Тестирование custom-маппинга | DTO → Entity преобразования |
| Тестирование валидаторов | Custom constraint validators |

### Метрики покрытия кода

#### Line Coverage (покрытие строк)

Самая простая и часто обманчивая метрика.

```java
// Пример: line coverage 100%, но баг не найден
public double calculateDiscount(double price, boolean isVip) {
    if (isVip) {
        return price * 0.8; // Скидка 20%
    }
    return price; // Без скидки
}

// Тест покрывает 100% строк, но не проверяет граничные случаи:
// price = 0, price < 0, price = Double.MAX_VALUE
```

**Line coverage 80%** — разумный минимум для большинства проектов.
Но сама по себе цифра ничего не гарантирует.

#### Branch Coverage (покрытие ветвей)

Учитывает все возможные пути выполнения: if/else, switch, тернарные операторы.

```java
// Branch coverage требует проверки ОБОИХ путей
public String getStatus(int score) {
    if (score >= 50 && score <= 100) {  // Две ветки: true и false
        return "PASS";                   // Но false включает score < 50 И score > 100
    }
    return "FAIL";
}

// Для полного branch coverage нужны тесты:
// score = 75 (true path)
// score = 30 (false path, первое условие false)
// score = 150 (false path, второе условие false)
```

#### Mutation Testing (мутационное тестирование)

**Самая честная метрика качества тестов.** Инструменты: PIT (PITest) для Java.

Идея: инструмент вносит мелкие изменения (мутации) в код и проверяет,
обнаружит ли тест эту мутацию.

```java
// Исходный код
if (age >= 18) { return "adult"; }

// Мутация 1: заменить >= на >
if (age > 18) { return "adult"; }

// Мутация 2: заменить >= на ==
if (age == 18) { return "adult"; }

// Мутация 3: заменить 18 на 19
if (age >= 19) { return "adult"; }

// Если тест проходит после мутации — мутант "выжил".
// Это значит, что тест недостаточно точный.
```

**Mutation score 60%+** — хороший показатель. 80%+ — отличный.

### Инструменты для unit-тестирования (Java-экосистема)

| Инструмент | Назначение |
|-----------|-----------|
| JUnit 5 | Фреймворк для запуска тестов |
| Mockito | Мокирование зависимостей |
| AssertJ | Читаемые assertion-ы |
| JaCoCo | Измерение line/branch coverage |
| PITest | Mutation testing |

## Integration Testing: стратегии

### Почему integration-тесты — самый недооценённый уровень

Unit-тесты проверяют компоненты изолированно. E2E-тесты проверяют всё вместе.
Но **большинство багов живёт на стыках**: между сервисами, между сервисом и БД,
между API и внешней системой.

Integration-тесты ловят именно эти баги.

### Стратегии интеграционного тестирования

#### Top-Down (сверху вниз)

**Суть:** начинаем с верхнего уровня (UI или API), нижние слои мокируем.
Постепенно заменяем моки реальными компонентами.

```
[API Controller] ← тестируем
       ↓
[Service Layer]  ← тестируем
       ↓
[Repository]     ← мок
       ↓
[Database]       ← мок
```

**Когда использовать:**
- Верхние слои готовы, нижние — нет
- Нужно быстро проверить пользовательские сценарии
- Команда работает feature-first

**Плюсы:** рано видим рабочий flow, удобно для демонстрации stakeholder-ам.
**Минусы:** много stub-ов/mock-ов, баги в нижних слоях находим поздно.

#### Bottom-Up (снизу вверх)

**Суть:** начинаем с нижнего уровня (БД, внешние сервисы), поднимаемся вверх.

```
[API Controller] ← мок/не тестируем пока
       ↓
[Service Layer]  ← тестируем
       ↓
[Repository]     ← тестируем
       ↓
[Database]       ← реальная (TestContainers)
```

**Когда использовать:**
- Критична корректность работы с данными
- Сложная бизнес-логика в сервисном слое
- Микросервисная архитектура (тестируем каждый сервис + его БД)

**Плюсы:** ранняя проверка data layer, меньше моков.
**Минусы:** нет полной картины пользовательского flow до поздних этапов.

#### Sandwich / Hybrid (гибридный)

**Суть:** тестируем и сверху, и снизу одновременно, встречаемся посередине.

```
[API Controller] ← тестируем (top-down)
       ↓
[Service Layer]  ← точка встречи
       ↓
[Repository]     ← тестируем (bottom-up)
       ↓
[Database]       ← реальная
```

**Когда использовать:**
- Большая команда (можно параллелить)
- Средний и большой проект
- Есть ресурсы на оба подхода

**Плюсы:** баланс между скоростью и глубиной.
**Минусы:** требует координации, возможно дублирование тестов.

#### Big Bang (всё сразу)

**Суть:** собираем все компоненты вместе и тестируем целиком.

**Когда использовать:**
- Маленький проект (до 5 сервисов)
- Все компоненты готовы одновременно
- Одноразовая интеграция (миграция)

**Плюсы:** просто — не нужна стратегия, тестируем реальную систему.
**Минусы:** сложно локализовать баг, поздний feedback, хаос при ошибках.

### Матрица выбора стратегии

| Критерий | Top-Down | Bottom-Up | Sandwich | Big Bang |
|----------|----------|-----------|----------|----------|
| Размер проекта | Средний | Любой | Большой | Маленький |
| Готовность компонентов | Сверху | Снизу | Параллельно | Все сразу |
| Критичность data layer | Низкая | Высокая | Средняя | Любая |
| Размер команды | 1-3 QA | 1-3 QA | 3+ QA | 1 QA |
| Скорость feedback | Быстрый | Средний | Быстрый | Медленный |

### Integration Testing в микросервисах

В микросервисной архитектуре integration-тесты делятся на два вида:

**1. Внутренняя интеграция (intra-service)**
Тестируем сервис + его БД + его очередь сообщений.

```java
// Пример: тест сервиса с реальной PostgreSQL через TestContainers
@SpringBootTest
@Testcontainers
class OrderServiceIntegrationTest {

    @Container
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:15")
            .withDatabaseName("testdb");

    @Test
    void shouldCreateOrderAndSaveToDatabase() {
        // Тестируем полный flow: Controller → Service → Repository → DB
        // Без моков — реальная БД в Docker-контейнере
    }
}
```

**2. Межсервисная интеграция (inter-service)**
Тестируем взаимодействие между сервисами.

Подходы:
- **Contract Testing** (Pact, Spring Cloud Contract) — каждый сервис проверяет
  соблюдение контракта, не поднимая другие сервисы.
- **TestContainers + docker-compose** — поднимаем зависимые сервисы в контейнерах.
- **WireMock / MockServer** — мокируем внешние HTTP-зависимости.

### TestContainers — must-have инструмент

TestContainers решает главную проблему integration-тестов: "а где взять реальную БД / Kafka / Redis?"

```java
// Основные контейнеры для типичного Java-проекта
@Container
static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:15");

@Container
static KafkaContainer kafka = new KafkaContainer(DockerImageName.parse("confluentinc/cp-kafka:7.4.0"));

@Container
static GenericContainer<?> redis = new GenericContainer<>("redis:7")
        .withExposedPorts(6379);

// Преимущества:
// - Изолированная среда для каждого теста
// - Одинаковое поведение на CI и локально
// - Не нужен отдельный test-environment
```

## System Testing: полная система как чёрный ящик

### Отличие от E2E

| Критерий | System Testing | E2E Testing |
|----------|---------------|-------------|
| Фокус | Вся система, все функции | Пользовательские сценарии |
| Подход | Систематический (feature by feature) | Scenario-based |
| Кто выполняет | QA-команда | QA + бизнес |
| Когда | После integration testing | Перед релизом |
| Тип | Функциональное + нефункциональное | Преимущественно функциональное |

### Что включает system testing

1. **Функциональное тестирование** — каждая фича работает по требованиям.
2. **Тестирование производительности** — система выдерживает нагрузку.
3. **Тестирование безопасности** — нет критических уязвимостей.
4. **Тестирование совместимости** — работает в разных браузерах/ОС.
5. **Тестирование установки/обновления** — можно развернуть и обновить.
6. **Тестирование восстановления** — система восстанавливается после сбоя.

### Практический подход

Не пытайтесь покрыть всё. Используйте **risk-based подход**:

1. Определите критические бизнес-процессы (обычно 20% функций = 80% ценности).
2. Тестируйте их глубоко (позитивные, негативные, граничные случаи).
3. Остальное — smoke + exploratory.

## Acceptance Testing: UAT, Alpha, Beta

### User Acceptance Testing (UAT)

**Кто:** конечные пользователи или их представители (product owner, бизнес-аналитик).
**Цель:** убедиться, что система решает бизнес-задачу.

QA-инженер в UAT:
- **Не выполняет** UAT (это делают бизнес-пользователи).
- **Готовит** тестовые сценарии на языке бизнеса.
- **Обеспечивает** тестовую среду и данные.
- **Собирает** и приоритизирует найденные проблемы.
- **Координирует** процесс и помогает пользователям.

### Alpha-тестирование

Внутреннее тестирование реальными сценариями, но внутри компании.
Часто проводится другими командами (не теми, кто разрабатывал).

### Beta-тестирование

Тестирование ограниченной группой внешних пользователей.

**Формы:**
- **Closed beta** — приглашённые пользователи, NDA.
- **Open beta** — все желающие, обычно с пометкой "beta" в интерфейсе.
- **A/B testing** — часть пользователей видит новую фичу, часть — старую.

### Критерии приёмки (Acceptance Criteria)

Хорошие acceptance criteria следуют формату **Given-When-Then**:

```gherkin
# Пример: регистрация пользователя
Сценарий: Успешная регистрация с валидными данными
  Допустим пользователь на странице регистрации
  Когда он вводит email "test@example.com"
  И вводит пароль "SecurePass123!"
  И нажимает "Зарегистрироваться"
  Тогда создаётся новый аккаунт
  И пользователь перенаправляется на страницу приветствия
  И на email отправляется письмо подтверждения
```

## Пирамида тестирования: пересмотр

### Классическая пирамида (Mike Cohn)

```
        /  E2E  \           ← Мало, медленные, дорогие
       /----------\
      / Integration \       ← Средне
     /----------------\
    /    Unit Tests     \   ← Много, быстрые, дешёвые
   /--------------------\
```

**Идея:** чем ниже уровень, тем больше тестов.
Unit-тесты быстрые и дешёвые → их должно быть больше всего.

### Антипаттерн: Ice Cream Cone

```
   /--------------------\
  \    Manual Testing    /  ← Много ручного
   \--------------------/
    \   E2E / UI Tests  /  ← Много UI-автотестов
     \-----------------/
      \  Integration  /    ← Мало
       \-------------/
        \   Unit    /      ← Почти нет
         \---------/
```

**Почему это плохо:**
- Медленный feedback (ручное тестирование — дни, unit-тесты — секунды).
- Хрупкие тесты (UI-тесты ломаются при каждом redesign).
- Дорого (ручной регресс на каждый релиз).
- Баги находятся поздно.

**Как это происходит:** команда начинает с ручного тестирования, потом "автоматизирует"
через Selenium, но unit-тесты никто не пишет, потому что "это дело разработчиков,
а они заняты фичами".

### Testing Trophy (Kent C. Dodds)

```
        /  E2E  \
       /----------\
      /             \
     /  Integration  \     ← БОЛЬШЕ ВСЕГО
    /                  \
   /--------------------\
  |    Static Analysis   |  ← TypeScript, ESLint, etc.
   \--------------------/
```

**Идея:** integration-тесты дают лучший баланс между confidence и скоростью.
Static analysis (типизация, линтеры) ловит целый класс ошибок бесплатно.

**Применимость:** больше актуально для frontend (React), но идея переносится
на backend: Spring Boot integration tests с TestContainers дают высокий confidence
при разумной скорости.

### Практическая пирамида для вашего проекта

Универсальной пирамиды не существует. Вот ориентиры:

**Для микросервисного backend (Java/Spring Boot):**
```
Unit:        40% — бизнес-логика, валидаторы, маппинг
Integration: 40% — сервис + БД (TestContainers), API-тесты
E2E:         15% — критические бизнес-сценарии
Manual:       5% — exploratory, UX
```

**Для монолитного web-приложения:**
```
Unit:        50% — бизнес-логика
Integration: 25% — модули + БД
E2E/UI:      15% — пользовательские сценарии
Manual:      10% — exploratory, cross-browser
```

**Для мобильного приложения:**
```
Unit:        30% — бизнес-логика, ViewModel
Integration: 20% — API-клиент + mock-сервер
UI/E2E:      30% — пользовательские flow (Appium/Espresso/XCTest)
Manual:      20% — устройства, UX, жесты
```

## Связь с тестированием

Уровни тестирования — это **каркас** всей тестовой стратегии. Каждый последующий
раздел курса (ручное тестирование, автоматизация, CI/CD) опирается на понимание:

- На каком уровне мы находимся?
- Какие дефекты этот уровень ловит?
- Какие инструменты на этом уровне используются?

Без чёткого понимания уровней вы будете писать E2E-тесты для проверки валидации email
(дорого и бессмысленно) вместо unit-теста (дёшево и быстро).

## Типичные ошибки

### Ошибка 1: "Нам нужно 100% code coverage"

Code coverage — это **метрика процесса**, а не **метрика качества**.
100% line coverage не гарантирует отсутствие багов. Гоняться за цифрой —
значит писать бесполезные тесты ради метрики.

**Правильный подход:** 80% line coverage как baseline + mutation testing
для критичных модулей.

### Ошибка 2: "Integration-тесты слишком сложные, давайте просто E2E"

E2E-тесты не заменяют integration. E2E проверяет happy path через весь стек.
Integration проверяет **конкретный стык** с разными сценариями (позитивные,
негативные, граничные). У них разные задачи.

### Ошибка 3: "Unit-тесты — ответственность разработчиков"

Формально — да. Практически — QA должен:
- Знать, что покрыто unit-тестами (чтобы не дублировать на уровне E2E).
- Уметь прочитать unit-тест и оценить его качество.
- Иногда писать unit-тесты на utility-код и сервисы.

### Ошибка 4: "У нас Selenium — значит, у нас автоматизация"

Selenium — это один инструмент на одном уровне (UI E2E).
Автоматизация включает unit, integration, API, performance, security тесты.
Selenium — это 10-20% от полной картины.

### Ошибка 5: "Мы тестируем Big Bang, потому что так быстрее"

Big Bang кажется быстрее, потому что не нужна стратегия. Но при первом баге
вы потратите часы на локализацию, потому что непонятно, какой компонент виноват.

## Вопросы на интервью

### 🟢 Junior

1. **Назовите четыре уровня тестирования по ISTQB.**
   Unit → Integration → System → Acceptance.

2. **Чем unit-тест отличается от integration-теста?**
   Unit тестирует один компонент изолированно (с моками). Integration — взаимодействие
   нескольких компонентов (с реальными или частично реальными зависимостями).

3. **Что такое code coverage? Какие виды бывают?**
   Метрика, показывающая, какая доля кода исполняется тестами. Виды: line, branch,
   condition, path coverage.

4. **Что такое пирамида тестирования?**
   Модель, предлагающая много unit-тестов (основание), меньше integration (середина),
   ещё меньше E2E (вершина).

### 🟡 Middle

5. **Как вы принимаете решение о балансе unit, integration и E2E тестов?**
   Анализирую: архитектуру (монолит vs микросервисы), критичность бизнес-процессов,
   зрелость команды, скорость feedback loop. Для микросервисов — больше integration.
   Для UI-heavy приложения — больше E2E. Всегда начинаю с risk assessment.

6. **Что такое TestContainers и зачем они нужны?**
   Библиотека для запуска Docker-контейнеров в тестах (БД, брокеры, кэши).
   Позволяет тестировать с реальными зависимостями без отдельного test-environment.

7. **Объясните разницу между top-down и bottom-up integration testing.**
   Top-down: начинаем с верхних модулей, нижние мокируем; ранний пользовательский feedback.
   Bottom-up: начинаем с нижних модулей, верхние добавляем позже; ранняя проверка данных.

8. **Что такое mutation testing?**
   Метод оценки качества тестов: инструмент вносит мелкие изменения (мутации) в код
   и проверяет, обнаружат ли тесты эти изменения. Чем больше мутантов "убито", тем
   лучше тесты.

### 🔴 Senior

9. **На вашем проекте 90% тестов — это E2E через Selenium. Как бы вы изменили стратегию?**
   Провёл бы аудит: какие сценарии покрыты E2E, что можно перенести вниз.
   Составил бы план миграции: выделяю критические бизнес-сценарии (оставляю E2E),
   переношу data validation в unit, API-логику — в integration. Внедряю TestContainers.
   Показываю метрики: время прогона до/после, стабильность, количество flaky-тестов.

10. **Как организовать integration-тестирование в микросервисной архитектуре с 20+ сервисами?**
    Два уровня: intra-service (сервис + его БД, TestContainers, ответственность команды сервиса)
    и inter-service (contract testing через Pact/Spring Cloud Contract).
    Полноценные E2E-тесты — только для 3-5 критических бизнес-flow в staging-окружении.
    CI/CD pipeline: unit → integration → contract → deploy → E2E.

11. **Расскажите о Testing Trophy. Когда этот подход лучше классической пирамиды?**
    Testing Trophy делает акцент на integration-тестах как основном слое.
    Лучше пирамиды, когда: (a) много межкомпонентных взаимодействий,
    (b) unit-тесты с моками дают ложное чувство безопасности,
    (c) есть хорошая инфраструктура для быстрых integration-тестов (TestContainers).
    Хуже пирамиды, когда: (d) сложная изолированная бизнес-логика (финансы, расчёты),
    (e) медленные интеграционные тесты без возможности ускорить.

## Практические задания

### Задание 1: Анализ пирамиды
Возьмите ваш текущий проект (или open-source проект). Посчитайте:
- Количество unit, integration, E2E тестов.
- Время прогона каждого уровня.
- Нарисуйте реальную "пирамиду" вашего проекта.
- Определите: это пирамида, ромб, или ice cream cone?
- Предложите план улучшения.

### Задание 2: Проектирование integration-тестов
Дано: REST API для интернет-магазина (Spring Boot + PostgreSQL + Kafka).
Основные сервисы: OrderService, PaymentService, NotificationService.
Задача:
1. Выберите стратегию integration testing (обоснуйте выбор).
2. Напишите список integration-тестов для OrderService (минимум 10).
3. Определите, что мокировать, а что поднимать через TestContainers.

### Задание 3: Mutation testing
1. Возьмите Java-проект с unit-тестами.
2. Подключите PITest (pitest-maven или pitest-gradle).
3. Запустите mutation testing.
4. Проанализируйте выживших мутантов (survived mutants).
5. Допишите тесты, чтобы "убить" хотя бы 5 мутантов.

### Задание 4: Выбор уровня тестирования
Для каждого сценария определите оптимальный уровень тестирования и обоснуйте:
1. Проверка формулы расчёта НДС.
2. Проверка, что после оплаты заказ появляется в личном кабинете.
3. Проверка, что сервис корректно обрабатывает timeout от внешнего API.
4. Проверка, что email-уведомление приходит после регистрации.
5. Проверка, что кнопка "Купить" доступна только авторизованным пользователям.

1. **"Test Driven Development" — Kent Beck** — классика о unit-тестировании и TDD.

   практический подход к integration testing через TDD.

   детальное объяснение пирамиды с примерами.

   официальная документация и примеры.
5. **PITest documentation** — https://pitest.org/ —
   инструмент для mutation testing в Java.
6. **"Testing Trophy" — Kent C. Dodds (статья)** — альтернативный взгляд
   на распределение тестов.
7. **"Microservices Testing" — Toby Clemson (статья на martinfowler.com)** —
   стратегии тестирования микросервисов.

   с использованием @SpringBootTest, @DataJpaTest, @WebMvcTest.
