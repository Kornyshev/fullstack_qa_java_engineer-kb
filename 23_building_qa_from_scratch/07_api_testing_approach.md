# Подход к API-тестированию: покрытие и кейсы

## Обзор

API — это контракт между компонентами системы. Если UI можно перерисовать за неделю, то сломанный API — это упавший
продакшен, потерянные заказы и разбуженный дежурный инженер в три часа ночи. Именно поэтому API-тестирование — самый
выгодный слой автоматизации: тесты быстрые, стабильные, покрывают бизнес-логику напрямую и не зависят от капризов
браузера.

В этом разделе мы разберём системный подход к тестированию API: от чеклиста на каждый endpoint до стратегии покрытия
всего сервиса. Без «проверил happy path и пошёл пить кофе» — с реальными кейсами, которые ловят баги на проде.

## Чеклист тестирования одного endpoint

Каждый endpoint заслуживает полного прохода по чеклисту. Ниже — структурированный список категорий тестов, который
стоит применять к каждому REST-endpoint.

### 1. Positive cases (позитивные сценарии)

Цель — убедиться, что endpoint работает как задумано при корректных входных данных.

| Что проверяем | Пример |
|---|---|
| Запрос с полным набором обязательных полей | `POST /users` с `name`, `email`, `password` |
| Запрос с обязательными + опциональными полями | `POST /users` + поле `phone` |
| Минимальный валидный запрос | Только обязательные поля |
| Корректный формат ответа | JSON, правильная структура |
| Корректный статус-код | `201 Created` для создания ресурса |
| Идемпотентность (для PUT, DELETE) | Повторный вызов не создаёт дубликат |

### 2. Negative cases (негативные сценарии)

Это основной источник багов. Здесь нужно быть параноиком.

**Отсутствующие поля:**
```
# Отправляем запрос без обязательного поля email
POST /users
{
  "name": "Ivan"
}
# Ожидаем: 400 Bad Request с понятным сообщением об ошибке
```

**Невалидные типы данных:**
```
# Отправляем строку вместо числа
POST /orders
{
  "quantity": "abc",
  "product_id": 123
}
# Ожидаем: 400 Bad Request, а НЕ 500 Internal Server Error
```

**Пустое тело запроса:**
```
# Пустой POST-запрос
POST /users
{}
# Ожидаем: 400 с перечислением всех отсутствующих полей
```

**Неверный HTTP-метод:**
```
# PATCH на endpoint, который поддерживает только GET и POST
PATCH /users
# Ожидаем: 405 Method Not Allowed
```

**Невалидный Content-Type:**
```
# XML вместо JSON
POST /users
Content-Type: application/xml
<user><name>Ivan</name></user>
# Ожидаем: 415 Unsupported Media Type
```

**Несуществующий ресурс:**
```
GET /users/999999999
# Ожидаем: 404 Not Found
```

### 3. Boundary cases (граничные значения)

| Параметр | Граничные значения для проверки |
|---|---|
| Строка | Пустая строка `""`, 1 символ, максимальная длина, максимум + 1 |
| Число | `0`, `-1`, `MAX_INT`, `MIN_INT`, дробное число |
| Массив | Пустой `[]`, 1 элемент, максимум элементов |
| Дата | Прошлая дата, текущая, будущая, `29.02` в високосный/невисокосный год |
| Файл | 0 байт, 1 байт, максимальный размер, максимум + 1 байт |

### 4. Authentication (аутентификация)

```
# Валидный токен — доступ разрешён
GET /orders
Authorization: Bearer <valid_token>
# Ожидаем: 200 OK

# Без токена — доступ запрещён
GET /orders
# Ожидаем: 401 Unauthorized

# Истёкший токен
GET /orders
Authorization: Bearer <expired_token>
# Ожидаем: 401 Unauthorized

# Невалидный формат токена
GET /orders
Authorization: Bearer malformed.token.here
# Ожидаем: 401 Unauthorized
```

### 5. Authorization и IDOR

IDOR (Insecure Direct Object Reference) — одна из самых распространённых уязвимостей в API.

```
# Пользователь A пытается получить заказ пользователя B
GET /users/USER_B_ID/orders/123
Authorization: Bearer <user_a_token>
# Ожидаем: 403 Forbidden (или 404, чтобы не раскрывать существование ресурса)
```

Что проверять:
- Пользователь видит только свои данные
- Роль `USER` не может выполнить действия роли `ADMIN`
- Нельзя изменить чужой ресурс, подставив чужой `id` в URL
- Нельзя через query-параметр `?userId=другой_id` получить чужие данные

### 6. Pagination (пагинация)

```
# Стандартный запрос с пагинацией
GET /products?page=1&size=10
# Ожидаем: ровно 10 элементов (или меньше на последней странице)

# Граничные случаи
GET /products?page=0          # Ожидаем: 400 или первая страница (зависит от спецификации)
GET /products?page=-1         # Ожидаем: 400 Bad Request
GET /products?page=999999     # Ожидаем: пустой массив, НЕ ошибку
GET /products?size=0          # Ожидаем: 400 Bad Request
GET /products?size=1000000    # Ожидаем: ограничение максимальным размером страницы
```

### 7. Filtering и Sorting

```
# Фильтрация по существующему полю
GET /products?category=electronics
# Ожидаем: все элементы имеют category=electronics

# Фильтрация по несуществующему значению
GET /products?category=nonexistent
# Ожидаем: пустой массив, НЕ ошибку

# Сортировка
GET /products?sort=price&order=asc
# Ожидаем: элементы отсортированы по возрастанию цены

# Невалидное поле сортировки
GET /products?sort=hacker_field
# Ожидаем: 400 Bad Request (а не SQL-инъекция!)
```

### 8. Error handling (обработка ошибок)

Каждый endpoint должен возвращать ошибки в **единообразном формате**:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Поле 'email' обязательно для заполнения",
    "details": [
      {
        "field": "email",
        "constraint": "required"
      }
    ]
  }
}
```

Что проверять:
- Формат ошибки единообразен по всему API
- Код ошибки машинно-читаемый (не просто строка "что-то пошло не так")
- Сообщение понятно пользователю
- Нет утечки внутренней информации (stack trace, имена таблиц, SQL-запросы)

## Стратегия покрытия API

### Матрица CRUD

Для каждого ресурса (entity) строим матрицу:

| Ресурс | Create | Read (list) | Read (single) | Update | Delete |
|---|---|---|---|---|---|
| User | `POST /users` | `GET /users` | `GET /users/{id}` | `PUT /users/{id}` | `DELETE /users/{id}` |
| Order | `POST /orders` | `GET /orders` | `GET /orders/{id}` | `PUT /orders/{id}` | `DELETE /orders/{id}` |

**Правило:** каждая ячейка — это минимум 1 positive + 3-5 negative тестов.

### Формула покрытия

```
Минимальное покрытие = Кол-во endpoints × (1 positive + 3-5 negative) + интеграционные сценарии
```

Пример для сервиса с 15 endpoints:
- 15 × 1 positive = 15 тестов
- 15 × 4 negative (среднее) = 60 тестов
- 10 интеграционных сценариев = 10 тестов
- **Итого: ~85 тестов** — это минимум для нормального покрытия

### Интеграционные сценарии (цепочки вызовов)

Отдельные endpoint-тесты не гарантируют, что система работает целиком. Нужны end-to-end цепочки:

```
# Сценарий: полный цикл заказа
1. POST /auth/register      → создать пользователя
2. POST /auth/login          → получить токен
3. GET /products             → получить список товаров
4. POST /cart/items          → добавить товар в корзину
5. POST /orders              → оформить заказ
6. GET /orders/{id}          → проверить статус заказа
7. Проверить запись в базе данных (опционально)
```

### Приоритизация тестов

| Приоритет | Что покрываем | Когда запускаем |
|---|---|---|
| P0 — Critical | Авторизация, основные CRUD, оплата | Каждый коммит (CI) |
| P1 — High | Валидация полей, пагинация, фильтрация | Каждый PR |
| P2 — Medium | Граничные значения, edge cases | Ночной прогон |
| P3 — Low | Редкие комбинации, производительность | Еженедельно |

## Уровни валидации ответа

Одна из самых частых ошибок начинающих — проверять только статус-код. Это как проверять, что магазин открыт, но не
смотреть, продают ли там то, что нужно.

### Level 1: Статус-код

```java
// Самый базовый уровень — просто проверяем код ответа
given()
    .when()
    .get("/users/1")
    .then()
    .statusCode(200);
```

Ловит: упавшие endpoint-ы, ошибки маршрутизации, 500-ки.

### Level 2: Статус-код + структура тела (JSON Schema)

```java
// Проверяем, что структура ответа соответствует контракту
given()
    .when()
    .get("/users/1")
    .then()
    .statusCode(200)
    .body(matchesJsonSchemaInClasspath("schemas/user-response.json"));
```

Ловит: пропавшие поля, изменение типов, сломанный контракт.

### Level 3: Статус-код + структура + значения полей

```java
// Проверяем конкретные значения в ответе
given()
    .when()
    .get("/users/1")
    .then()
    .statusCode(200)
    .body("id", equalTo(1))
    .body("name", equalTo("Ivan"))
    .body("email", matchesPattern(".*@.*\\..*"))
    .body("roles", hasItem("USER"));
```

Ловит: неправильную бизнес-логику, перепутанные поля, кривые маппинги.

### Level 4: Всё вышеперечисленное + заголовки + время ответа

```java
// Полная проверка: тело, заголовки, производительность
given()
    .when()
    .get("/users/1")
    .then()
    .statusCode(200)
    .header("Content-Type", "application/json")
    .header("X-Request-Id", notNullValue())
    .time(lessThan(2000L)) // ответ за менее чем 2 секунды
    .body("id", equalTo(1))
    .body("name", equalTo("Ivan"));
```

Ловит: проблемы с заголовками безопасности, деградацию производительности, отсутствие correlation ID.

**Рекомендация:** Level 2 — обязательный минимум для всех тестов. Level 3 — для критичных endpoint-ов. Level 4 — для
ключевых бизнес-сценариев.

## Contract Testing

### Когда и зачем

Contract testing решает конкретную проблему: **сервис A ожидает от сервиса B ответ определённой структуры, но B может
измениться без ведома A**.

Классическая ситуация:
1. Команда B добавляет поле `middleName` в ответ — ок, обратная совместимость
2. Команда B переименовывает `firstName` в `first_name` — сервис A ломается
3. Узнаём об этом в четверг вечером на staging

### Consumer-Driven Contracts

Идея: **потребитель (consumer) определяет, что ему нужно от поставщика (provider)**.

```
Поток:
1. Consumer (фронтенд) описывает контракт: «Я ожидаю от GET /users/{id} поля: id, name, email»
2. Контракт сохраняется в общем месте (Pact Broker, репозиторий)
3. Provider (бэкенд) прогоняет тесты против контракта
4. Если Provider ломает контракт — тесты падают ДО мержа в main
```

### Provider Verification

```java
// На стороне провайдера: проверяем, что наш API удовлетворяет все контракты
@Provider("user-service")
@PactBroker(url = "https://pact-broker.company.com")
class UserServiceContractTest {

    @TestTemplate
    @ExtendWith(PactVerificationInvocationContextProvider.class)
    void verifyPact(PactVerificationContext context) {
        // Pact автоматически проверяет все контракты от всех потребителей
        context.verifyInteraction();
    }
}
```

### Когда использовать Contract Testing

| Ситуация | Нужен contract testing? |
|---|---|
| Монолит | Нет, достаточно интеграционных тестов |
| 2-3 микросервиса, одна команда | Опционально |
| Много микросервисов, разные команды | Обязательно |
| Публичный API с внешними потребителями | Крайне желательно |

## Связь с тестированием

API-тестирование — это **центральный слой** тестовой пирамиды для большинства современных приложений.

- **Unit-тесты** проверяют отдельные методы — они не гарантируют, что API отвечает корректно.
- **API-тесты** проверяют бизнес-логику через HTTP — быстро, стабильно, покрывают интеграцию слоёв.
- **UI-тесты** проверяют визуальное отображение — они медленные и хрупкие, поэтому бизнес-логику лучше покрывать на уровне API.

Практическое правило: **если можно проверить через API — проверяй через API, а не через UI**.

Пример: проверку того, что пользователь с дублирующимся email не может зарегистрироваться, не нужно делать через
Selenium. Один POST-запрос с дублирующимся email — и проверяем код ответа. Быстрее в 100 раз.

## Типичные ошибки

### 1. Неправильные статус-коды

```
# Баг: сервер возвращает 200 OK при ошибке валидации
POST /users { "name": "" }
Response: 200 OK { "error": "Name is required" }

# Правильно: 400 Bad Request
POST /users { "name": "" }
Response: 400 Bad Request { "error": { "code": "VALIDATION_ERROR", ... } }
```

Почему это плохо: клиенты полагаются на статус-коды для обработки ошибок. Если 200 — значит успех.

### 2. Отсутствие валидации входных данных

```
# Баг: сервер принимает всё подряд и падает с 500
POST /users { "email": 12345 }
Response: 500 Internal Server Error { "message": "ClassCastException: Integer cannot be cast to String" }
```

### 3. Непоследовательный формат ошибок

```
# Endpoint A:
{ "error": "Not found" }

# Endpoint B:
{ "message": "Resource not found", "code": 404 }

# Endpoint C:
{ "errors": [{ "detail": "not found" }] }
```

Клиенты не могут написать единую обработку ошибок. Нужен **единый формат** для всего API.

### 4. Утечка чувствительных данных

```json
{
  "id": 1,
  "name": "Ivan",
  "email": "ivan@mail.com",
  "password_hash": "$2b$10$...",
  "internal_role_id": 3,
  "created_by_admin": "admin@company.internal"
}
```

Поля `password_hash`, `internal_role_id`, `created_by_admin` не должны попадать в ответ API.

### 5. Отсутствие пагинации

```
GET /logs
# Возвращает 2 миллиона записей в одном ответе
# Результат: OutOfMemoryError, убитый pod, пейджер у SRE
```

### 6. Race conditions

```
# Два одновременных запроса на списание одного и того же баланса
POST /wallet/withdraw { "amount": 100 }  # Баланс: 100
POST /wallet/withdraw { "amount": 100 }  # Баланс: 100 (ещё не обновился)
# Результат: баланс = -100, компания потеряла деньги
```

Проверяется через параллельную отправку запросов (например, `ExecutorService` в Java или `Promise.all` в JavaScript).

## Вопросы на интервью

### 🟢 Junior

1. **Какие HTTP-методы вы знаете и для чего каждый используется?**
   GET — получить ресурс, POST — создать, PUT — заменить, PATCH — частично обновить, DELETE — удалить. GET и DELETE не
   имеют тела запроса. PUT — идемпотентный, POST — нет.

2. **Что такое статус-код 401 и чем он отличается от 403?**
   401 Unauthorized — клиент не аутентифицирован (нет или невалидный токен). 403 Forbidden — клиент аутентифицирован,
   но не имеет прав на этот ресурс.

3. **Как вы тестируете API без UI?**
   С помощью инструментов Postman, curl или автоматизированных тестов (RestAssured, requests). Формирую HTTP-запрос
   вручную: URL, метод, заголовки, тело — и проверяю ответ.

4. **Что проверяете в ответе API помимо статус-кода?**
   Тело ответа (структуру и значения полей), заголовки (Content-Type, Cache-Control), время ответа.

### 🟡 Middle

5. **Как бы вы построили покрытие API-тестами для нового сервиса из 20 endpoint-ов?**
   Составил бы CRUD-матрицу всех ресурсов. На каждый endpoint — 1 positive + 3-5 negative тестов. Добавил бы 5-10
   end-to-end цепочек для критичных бизнес-сценариев. Приоритизировал бы по бизнес-критичности. Итого ~100-120 тестов.

6. **Что такое IDOR и как его тестировать?**
   Insecure Direct Object Reference — уязвимость, когда пользователь может получить доступ к чужому ресурсу, подставив
   чужой ID в URL. Тестирую: создаю двух пользователей, авторизуюсь за первого, пытаюсь получить/изменить ресурс
   второго. Ожидаю 403 или 404.

7. **Зачем нужна JSON Schema валидация и как её применяете?**
   Чтобы автоматически проверять структуру ответа: наличие полей, их типы, обязательность. Храню schema-файлы в
   проекте, валидирую каждый ответ. Это ловит поломку контракта до того, как она попадёт на прод.

8. **Как тестировать пагинацию?**
   Проверяю: первая страница, последняя, за пределами диапазона, размер страницы 0 и отрицательный, общее количество
   элементов совпадает с суммой по страницам, элементы не повторяются между страницами.

### 🔴 Senior

9. **Как бы вы организовали contract testing между 5 микросервисами?**
   Использовал бы Pact с Pact Broker. Каждый consumer публикует свои ожидания (контракты) в Broker. Provider при
   каждом PR верифицирует все контракты от всех потребителей. Can-I-Deploy проверяет совместимость перед деплоем. Webhook
   в Broker уведомляет команды о сломанных контрактах.

10. **Как тестировать race conditions в API?**
    Запускаю N параллельных одинаковых запросов (через `ExecutorService` / `CompletableFuture` в Java). Проверяю
    инвариант: итоговое состояние в БД корректно. Пример: 10 одновременных запросов на списание — баланс не должен стать
    отрицательным. Так я нашёл баг с двойным списанием на реальном проекте.

11. **Как вы решаете проблему тестовых данных в API-тестах?**
    Тесты сами создают данные через API (setup) и удаляют после себя (teardown). Не завишу от предзаполненной БД.
    Для изоляции — уникальные идентификаторы (UUID/timestamp в данных). В CI — каждый прогон поднимает чистую БД через
    TestContainers.
