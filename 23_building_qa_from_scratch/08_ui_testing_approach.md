# Подход к UI-тестированию: что автоматизировать в интерфейсе

## Обзор

UI-тесты — самая дорогая и хрупкая часть тестовой пирамиды. Один Selenium-тест выполняется 10-30 секунд, а один
API-тест — 100-300 миллисекунд. При этом UI-тест может упасть из-за медленной загрузки изображения, изменённого текста
кнопки или обновления библиотеки компонентов. Именно поэтому ключевой навык QA-инженера — понимать, **что стоит
автоматизировать в UI, а что — нет**.

Этот раздел — практическое руководство по UI-автоматизации: от выбора сценариев до борьбы с flaky-тестами. Без иллюзий
«автоматизируем всё» — с прагматичным подходом, который работает в реальных проектах.

---

## Что автоматизировать в UI

### Критерии отбора сценариев для автоматизации

Не каждый UI-тест стоит автоматизировать. Перед тем как написать тест, задайте три вопроса:

1. **Этот сценарий критичен для бизнеса?** (Если пользователь не сможет это сделать — бизнес теряет деньги?)
2. **Этот сценарий стабилен?** (UI этой фичи не будет переделан в ближайший месяц?)
3. **Этот сценарий нельзя проверить на уровне API?** (Есть визуальный компонент, который важен?)

Если на все три ответ «да» — автоматизируем. Если хотя бы на один «нет» — скорее всего, не стоит.

### Что СТОИТ автоматизировать

| Категория | Примеры | Почему |
|---|---|---|
| Критичные пользовательские пути | Логин, регистрация, checkout, оплата | Поломка = прямая потеря денег |
| Smoke-тесты | Главная страница открывается, меню работает, поиск выдаёт результаты | Быстрая проверка «жив ли сайт» |
| Формы с валидацией | Регистрация, создание заказа, профиль | Валидация на фронтенде часто ломается |
| Навигация | Переходы между ключевыми страницами | Роутинг ломается при рефакторинге |
| Кроссбраузерные проверки | Те же critical paths в Chrome, Firefox, Safari | Браузерные различия реальны |

### Что НЕ стоит автоматизировать

| Категория | Почему не стоит | Что делать вместо |
|---|---|---|
| Pixel-perfect верстка | Слишком хрупко, ломается от любого CSS-изменения | Visual regression testing (отдельный инструмент) |
| Быстро меняющийся UI | Тесты устаревают быстрее, чем пишутся | Ручное тестирование, пока UI не стабилизируется |
| Анимации и переходы | Нестабильно, зависит от производительности машины | Ручная проверка + отключение анимаций в тестах |
| Тексты и переводы | Тексты часто меняются, тесты постоянно падают | Отдельные тесты на локализацию или snapshot-тесты |
| Бизнес-логика без визуального компонента | Расчёт скидки, валидация email | Тестировать через API или unit-тесты |
| Интеграция с третьими сторонами | Платёжные системы, OAuth-провайдеры | Mock на уровне API, manual тест на staging |

### Типичная ошибка: «Давайте автоматизируем всё через UI»

Команда пишет 500 Selenium-тестов. Через полгода:
- Прогон занимает 4 часа
- 15% тестов flaky
- Каждый рефакторинг UI ломает 50 тестов
- Команда тратит больше времени на поддержку тестов, чем на тестирование

**Правильный подход:** 30-50 UI-тестов на критичные пути + 300-500 API-тестов на бизнес-логику.

## Стратегия покрытия UI-тестами

### Уровень 1: Critical User Journeys (5-10 сценариев)

Это самые важные пути пользователя через приложение. Если они сломаны — релиз невозможен.

```
Пример для интернет-магазина:

1. Регистрация нового пользователя
   → Открыть /register → Заполнить форму → Нажать «Зарегистрироваться» → Проверить редирект на /dashboard

2. Логин существующего пользователя
   → Открыть /login → Ввести email/пароль → Нажать «Войти» → Проверить отображение имени в header

3. Поиск товара
   → Открыть главную → Ввести запрос в поиск → Проверить отображение результатов

4. Добавление товара в корзину
   → Открыть карточку товара → Нажать «В корзину» → Проверить счётчик корзины

5. Оформление заказа (checkout)
   → Открыть корзину → Заполнить адрес доставки → Выбрать способ оплаты → Подтвердить заказ → Проверить страницу подтверждения

6. Просмотр истории заказов
   → Открыть /orders → Проверить список заказов → Открыть детали заказа

7. Выход из системы
   → Нажать «Выйти» → Проверить редирект на /login → Проверить, что защищённая страница недоступна
```

### Уровень 2: Page-Level Smoke (по 1-2 проверки на страницу)

Для каждой ключевой страницы — минимальная проверка, что она загружается и содержит основные элементы.

```java
// Пример: smoke-тест страницы каталога
@Test
void catalogPageLoadsSuccessfully() {
    // Открываем страницу каталога
    page.navigate("/catalog");

    // Проверяем основные элементы
    assertThat(page.locator("[data-testid='catalog-header']")).isVisible();
    assertThat(page.locator("[data-testid='product-list']")).isVisible();
    assertThat(page.locator("[data-testid='product-card']")).hasCountGreaterThan(0);
    assertThat(page.locator("[data-testid='search-input']")).isVisible();
}
```

### Уровень 3: Form Validation (каждая форма)

Формы — частый источник багов на фронтенде. Для каждой формы проверяем:

```
Checklist для формы:
✓ Отправка с пустыми обязательными полями → сообщения об ошибках
✓ Отправка с невалидными данными (email без @) → корректная ошибка
✓ Отправка с валидными данными → успешный результат
✓ Проверка обязательных полей маркированы звёздочкой (*)
✓ Tab-навигация между полями работает
✓ Enter отправляет форму (если это ожидаемое поведение)
```

### Уровень 4: Cross-Browser Testing

Не все тесты нужно запускать во всех браузерах. Стратегия:

| Набор тестов | Chrome | Firefox | Safari | Edge |
|---|---|---|---|---|
| Critical Journeys | ✓ | ✓ | ✓ | ✓ |
| Page Smoke | ✓ | ✓ | — | — |
| Form Validation | ✓ | — | — | — |

**Почему так:** большинство rendering-багов проявляются именно в critical paths. Если форма корректно работает в Chrome,
вероятность проблемы в Firefox крайне мала (если это не CSS-специфичная проблема).

## Принципы проектирования UI-тестов

### 1. Тестируй через API, верифицируй через UI

Золотое правило: **Setup данных через API, проверка результата через UI**.

```java
// Плохо: создаём пользователя через UI (медленно, хрупко)
@Test
void userCanViewOrder() {
    page.navigate("/register");
    page.fill("[data-testid='name']", "Ivan");
    page.fill("[data-testid='email']", "ivan@test.com");
    page.fill("[data-testid='password']", "Password123");
    page.click("[data-testid='register-btn']");
    // ... ещё 15 шагов через UI для создания заказа
    // И только потом — проверка того, что мы хотели проверить
}

// Хорошо: создаём данные через API, проверяем UI
@Test
void userCanViewOrder() {
    // Setup через API — быстро и надёжно
    var user = apiClient.createUser("Ivan", "ivan@test.com");
    var order = apiClient.createOrder(user.getId(), List.of(productId));

    // Логинимся (можно тоже через API — установить cookie/token)
    apiClient.loginAndSetCookies(page, user);

    // Проверяем только то, что хотим проверить
    page.navigate("/orders/" + order.getId());
    assertThat(page.locator("[data-testid='order-status']")).hasText("Создан");
    assertThat(page.locator("[data-testid='order-total']")).hasText("1 500 ₽");
}
```

### 2. Независимые тесты

Каждый тест должен работать изолированно: не зависеть от порядка запуска и от результатов других тестов.

```java
// Плохо: тесты зависят друг от друга
@Test
@Order(1)
void createUser() { /* создаёт пользователя */ }

@Test
@Order(2)
void loginAsCreatedUser() { /* использует пользователя из теста 1 */ }

@Test
@Order(3)
void deleteUser() { /* удаляет пользователя из теста 1 */ }

// Хорошо: каждый тест — самодостаточный
@Test
void userCanLogin() {
    var user = apiClient.createUser(uniqueEmail()); // свои данные
    page.navigate("/login");
    page.fill("[data-testid='email']", user.getEmail());
    page.fill("[data-testid='password']", user.getPassword());
    page.click("[data-testid='login-btn']");
    assertThat(page.locator("[data-testid='user-name']")).hasText(user.getName());
}
```

### 3. Стабильные локаторы

Иерархия надёжности локаторов (от лучшего к худшему):

| Приоритет | Тип локатора | Пример | Надёжность |
|---|---|---|---|
| 1 | `data-testid` | `[data-testid='login-btn']` | Максимальная — создан специально для тестов |
| 2 | `id` (если стабильный) | `#login-button` | Высокая — но может конфликтовать с JS |
| 3 | Текст (для кнопок, ссылок) | `text=Войти` | Средняя — меняется при смене текста |
| 4 | CSS-класс | `.btn-primary` | Низкая — меняется при рефакторинге стилей |
| 5 | XPath по структуре | `//div[3]/form/button` | Минимальная — ломается при любом изменении DOM |

```java
// Плохо: хрупкий XPath, который сломается при изменении структуры
page.locator("//div[@class='container']/div[2]/form/div[3]/button").click();

// Хорошо: стабильный data-testid
page.locator("[data-testid='submit-order-btn']").click();
```

**Договаривайтесь с разработчиками:** добавление `data-testid` — это 5 минут работы фронтендера, но экономит часы
на поддержку тестов.

### 4. Умные ожидания (Smart Waits)

```java
// Плохо: Thread.sleep — ненадёжно и замедляет тесты
page.click("[data-testid='submit']");
Thread.sleep(3000); // может не хватить, а может быть лишним
assertThat(page.locator("[data-testid='success-message']")).isVisible();

// Хорошо: явное ожидание конкретного условия
page.click("[data-testid='submit']");
page.locator("[data-testid='success-message']").waitFor(
    new Locator.WaitForOptions().setState(WaitForSelectorState.VISIBLE).setTimeout(10000)
);
assertThat(page.locator("[data-testid='success-message']")).hasText("Заказ создан");
```

Стратегия ожиданий:
- **Никогда** не используйте `Thread.sleep()` в тестах
- Используйте **явные ожидания** конкретного элемента или состояния
- Устанавливайте **разумные таймауты** (10-30 секунд для обычных операций)
- Ожидайте **конкретное состояние**: видимость элемента, наличие текста, изменение URL

## Борьба с Flaky-тестами

### Что такое flaky-тест

Flaky-тест — тест, который проходит и падает на одном и том же коде без изменений. Сегодня зелёный, завтра красный,
послезавтра снова зелёный. Это самый демотивирующий вид технического долга: команда перестаёт доверять тестам и начинает
игнорировать падения.

### Основные причины flaky-тестов

| Причина | Частота | Пример |
|---|---|---|
| Race conditions в UI | 40% | Клик по кнопке до её полной загрузки |
| Зависимость от внешних сервисов | 20% | Тест падает, потому что staging API медленный |
| Зависимость от порядка тестов | 15% | Тест B работает только после теста A |
| Нестабильные тестовые данные | 10% | Данные удалены другим тестом или предыдущим прогоном |
| Таймауты | 10% | `Thread.sleep(2000)` не хватает на CI |
| Проблемы с инфраструктурой | 5% | Нехватка памяти, нестабильный Docker |

### Как предотвращать

```java
// 1. Ждать конкретные условия, а не время
// Плохо:
Thread.sleep(5000);
// Хорошо:
page.waitForSelector("[data-testid='order-list']");

// 2. Изолировать тестовые данные
// Плохо:
var user = getUser("ivan@test.com"); // общий пользователь для всех тестов
// Хорошо:
var user = createUser("ivan-" + UUID.randomUUID() + "@test.com"); // уникальный

// 3. Не зависеть от внешних сервисов
// Плохо:
var paymentResult = realPaymentGateway.charge(card); // может быть недоступен
// Хорошо:
var paymentResult = mockPaymentGateway.charge(card); // всегда стабилен

// 4. Retry с умом (для инфраструктурных проблем, НЕ для багов)
// В конфигурации тестового фреймворка:
// retryCount = 1 (один retry, не больше)
```

### Quarantine-подход

Когда flaky-тест обнаружен, но нет времени чинить сразу:

```
Процесс:
1. Пометить тест тегом @Flaky или @Quarantine
2. Перенести в отдельный test suite
3. Не блокировать CI/CD основным тестом
4. Завести задачу в Jira с дедлайном на починку
5. Каждую неделю разбирать карантин: чинить или удалять

Правило: в карантине не более 10% тестов.
Если больше — проблема системная, нужен рефакторинг тестовой инфраструктуры.
```

## Visual Regression Testing

### Что это и зачем

Visual regression testing — автоматическое сравнение скриншотов страниц между версиями. Ловит визуальные баги, которые
не видны в DOM: съехавшую верстку, перекрытые элементы, неправильные цвета.

### Как работает

```
Процесс:
1. Делаем скриншот страницы (baseline)
2. После изменений делаем новый скриншот
3. Сравниваем попиксельно (или с алгоритмами нечёткого сравнения)
4. Если разница > порог — тест падает
5. QA-инженер решает: это баг или ожидаемое изменение (обновляет baseline)
```

### Инструменты

| Инструмент | Тип | Особенности |
|---|---|---|
| Playwright (встроенный) | Open-source | `expect(page).toHaveScreenshot()`, бесплатно, в составе фреймворка |
| Percy (BrowserStack) | SaaS | Облачное сравнение, поддержка CI/CD, платный |
| Applitools | SaaS | AI-сравнение (игнорирует незначительные изменения), платный |
| BackstopJS | Open-source | Конфигурация через JSON, хорош для статических страниц |
| Loki | Open-source | Интеграция со Storybook, для компонентных тестов |

### Пример с Playwright

```java
// Снимок полной страницы
@Test
void catalogPageLooksCorrect() {
    page.navigate("/catalog");
    page.waitForLoadState(LoadState.NETWORKIDLE); // ждём загрузки всех ресурсов

    // Сравнение со сохранённым снимком
    assertThat(page).hasScreenshot("catalog-page.png",
        new PageAssertions.HasScreenshotOptions()
            .setMaxDiffPixelRatio(0.01) // допускаем 1% различий
    );
}

// Снимок конкретного компонента
@Test
void productCardLooksCorrect() {
    page.navigate("/catalog");
    var productCard = page.locator("[data-testid='product-card']").first();

    assertThat(productCard).hasScreenshot("product-card.png");
}
```

### Когда использовать

| Ситуация | Visual regression нужен? |
|---|---|
| Библиотека UI-компонентов | Да — ловит визуальные регрессии в компонентах |
| Лендинг / маркетинговые страницы | Да — верстка критична |
| Админ-панель | Скорее нет — функциональность важнее визуала |
| Часто меняющийся дизайн | Нет — baseline устаревает быстрее, чем обновляется |

## Связь с тестированием

UI-тесты — это **верхушка тестовой пирамиды**. Они проверяют, что пользователь видит и может сделать то, что ожидается.
Но они НЕ заменяют другие уровни тестирования:

- **Unit-тесты** покрывают логику компонентов фронтенда (форматирование дат, валидация, расчёты)
- **API-тесты** покрывают бизнес-логику бэкенда (создание заказа, расчёт скидки)
- **UI-тесты** покрывают **интеграцию фронтенда с бэкендом** и визуальную корректность

Типичная пропорция для web-приложения:
```
Unit-тесты:  60-70% (быстрые, дешёвые, стабильные)
API-тесты:   20-30% (средняя скорость, покрывают бизнес-логику)
UI-тесты:     5-10% (медленные, дорогие, но незаменимые для critical paths)
```

Если ваш UI-тест проверяет бизнес-логику (например, правильный расчёт скидки) — это сигнал, что тест написан на
неправильном уровне. Перенесите проверку на уровень API или unit.

## Типичные ошибки

### 1. Автоматизация всего через UI

Команда пишет 500 Selenium-тестов на все возможные сценарии. Через 3 месяца тестовый прогон занимает 5 часов, 20% тестов
flaky, и на поддержку тестов уходит больше времени, чем на разработку новых фич.

**Правило:** если сценарий можно проверить через API — проверяйте через API.

### 2. Thread.sleep() вместо явных ожиданий

```java
// Это ГАРАНТИЯ flaky-тестов
page.click("[data-testid='submit']");
Thread.sleep(3000);
page.click("[data-testid='confirm']");
```

На локальной машине 3 секунды хватает. На CI — нет. Увеличиваете до 5 секунд. Тесты замедляются.
Где-то снова не хватает. Бесконечная гонка.

### 3. XPath-локаторы по структуре DOM

```java
// Сломается при ЛЮБОМ изменении структуры страницы
page.locator("//div[@class='main']/div[2]/div[1]/form/div[3]/button[1]");
```

Это ломается, когда дизайнер добавит обёртку `<div>`, когда разработчик поменяет порядок элементов,
когда CSS-класс переименуют.

### 4. Зависимые тесты

Тест «удалить заказ» зависит от теста «создать заказ». Если первый упал — весь набор красный. Если порядок
запуска изменится — тоже красный.

### 5. Тестирование стороннего кода

Не тестируйте работу DatePicker-библиотеки или Google Maps. Тестируйте только интеграцию: что ваш код правильно
передаёт данные в компонент и правильно обрабатывает результат.

### 6. Игнорирование flaky-тестов

«Это просто flaky, перезапусти» — фраза, с которой начинается деградация тестовой культуры. Каждый flaky-тест
требует расследования: это проблема теста, инфраструктуры или продукта?

## Вопросы на интервью

### 🟢 Junior

1. **Что бы вы автоматизировали в UI в первую очередь?**
   Критичные пользовательские пути: логин, регистрация, основной бизнес-сценарий (оформление заказа, создание
   документа и т.д.). Это 5-10 тестов, которые дают максимальную ценность.

2. **Какие локаторы вы используете и почему?**
   Приоритет: `data-testid` (специально для тестов, не меняется), `id` (если стабильный), текст (для кнопок). Избегаю
   XPath по структуре DOM и CSS-классов — они хрупкие.

3. **Что такое flaky-тест?**
   Тест, который проходит и падает на одном и том же коде без изменений. Основные причины: race conditions, зависимость
   от данных или порядка тестов, `Thread.sleep()`.

4. **Почему нельзя использовать Thread.sleep() в тестах?**
   Потому что время ожидания непредсказуемо: на быстрой машине 2 секунды достаточно, на CI — нет. Нужны явные ожидания
   конкретного состояния (элемент виден, текст появился, URL изменился).

### 🟡 Middle

5. **Как бы вы организовали UI-тестирование для нового проекта?**
   Начал бы с 5-10 critical user journeys. Договорился бы с разработчиками о `data-testid`. Настроил бы setup данных
   через API. Запускал бы тесты в CI на каждый PR. Добавлял бы по 2-3 теста в спринт. Через 3 месяца — стабильный
   набор из 30-50 тестов.

6. **Как вы боретесь с flaky-тестами?**
   Карантин: помечаю flaky-тест, переношу в отдельный suite, не блокирую CI. Расследую причину: обычно это
   race condition (нужен wait), нестабильные данные (нужна изоляция) или зависимость тестов (нужен рефакторинг).
   Правило: в карантине не более 10% тестов.

7. **Когда использовать visual regression testing?**
   Когда визуальная корректность критична: библиотека компонентов, маркетинговые лендинги, публичный интерфейс. Не
   использую для админок и часто меняющегося UI — baseline устаревает быстрее, чем обновляется.

8. **Как вы решаете проблему тестовых данных в UI-тестах?**
   Данные создаю через API в `@BeforeEach`, удаляю в `@AfterEach`. Каждый тест работает со своими уникальными данными
   (UUID в имени/email). Не зависию от предзаполненной базы. На CI — каждый прогон поднимает чистое окружение.

### 🔴 Senior

9. **Как бы вы определили оптимальное количество UI-тестов для проекта с 50 страницами?**
   Начал бы с анализа user journeys: определил бы 10-15 критичных путей (обычно покрывают 80% бизнес-ценности). К ним
   добавил бы page-level smoke (по 1 тесту на страницу = 50 тестов) и form validation для ключевых форм (10-15 тестов).
   Итого: 75-80 UI-тестов. Остальное — через API. Мониторил бы метрики: если прогон > 30 минут или flaky rate > 5% —
   нужна оптимизация.

10. **Как масштабировать UI-тестирование при переходе от монолита к микрофронтендам?**
    Каждый микрофронтенд получает свои компонентные тесты (Storybook + visual regression). E2E-тесты остаются на уровне
    собранного приложения, но только для critical journeys. Contract testing между микрофронтендами и BFF. Параллелизация
    запуска на CI (sharding). Ключевое: не дублировать тесты между уровнями.

11. **Как интегрировать UI-тесты в CI/CD pipeline так, чтобы они не блокировали деплой на 40 минут?**
    Разделяю на два набора: smoke (5 минут, блокирует мерж) и full regression (30 минут, запускается параллельно с
    деплоем на staging). Sharding: разделяю тесты на 4 части, запускаю параллельно на 4 машинах. Результат: 30 минут /
    4 = 7-8 минут. Плюс smart test selection: на основе изменённых файлов запускаю только релевантные тесты.
