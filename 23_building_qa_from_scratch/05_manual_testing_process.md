# Ручное тестирование: процесс от требований до Sign-off

## Обзор

Ручное тестирование — это не "кликать по кнопкам и надеяться на лучшее".
Это **систематический инженерный процесс** с чёткими входами, выходами и критериями качества.

Многие начинающие QA думают, что ручное тестирование — это "пока не научился автоматизировать".
Это фундаментальная ошибка. Даже в командах с 95% автоматизацией ручное тестирование
остаётся незаменимым для:

- **Exploratory testing** — поиск багов, которые не предусмотрены сценариями.
- **Usability testing** — оценка удобства, которую не измерить автотестами.
- **Первичная проверка** — быстрый smoke до запуска автоматизации.
- **Визуальная проверка** — вёрстка, дизайн, анимации.
- **Новая функциональность** — первый цикл тестирования до написания автотестов.

В этом разделе мы пройдём **полный цикл**: от получения требований до финального sign-off.

## Анализ требований

### Почему анализ требований — это ПЕРВАЯ задача QA

Статистика: **56% всех дефектов** в программном обеспечении имеют корни в неточных,
неполных или противоречивых требованиях (по данным IBM Systems Sciences Institute).
Стоимость исправления бага, найденного на этапе требований, в **100 раз ниже**,
чем на продакшене.

Shift-left начинается здесь.

### Что искать в требованиях

#### 1. Неоднозначность (Ambiguity)

```
❌ "Система должна быстро загружать страницу"
   → Что такое "быстро"? 1 секунда? 5 секунд? При какой нагрузке?

✅ "Страница каталога должна загружаться за ≤ 2 секунды при 100 одновременных пользователях,
    измеренная как Time to First Meaningful Paint"
```

```
❌ "Пользователь может загрузить файл"
   → Какие форматы? Какой максимальный размер? Сколько файлов одновременно?

✅ "Пользователь может загрузить до 5 файлов форматов JPG, PNG, PDF.
    Максимальный размер каждого файла — 10 МБ. Суммарный размер — 30 МБ."
```

#### 2. Пропущенные сценарии (Missing Scenarios)

Чек-лист для каждого требования:

| Категория | Вопросы |
|----------|---------|
| Негативные сценарии | Что если пользователь вводит невалидные данные? |
| Граничные значения | Что на границах диапазонов? Пустая строка? Максимальная длина? |
| Параллельный доступ | Что если два пользователя делают это одновременно? |
| Прерывание | Что если пользователь прерывает операцию? Закрывает браузер? |
| Права доступа | Кто может выполнить это действие? Что видит пользователь без прав? |
| Состояние данных | Что если данных нет? Что если их миллион? |
| Интеграция | Что если внешний сервис недоступен? Что если он отвечает медленно? |
| Локализация | Работает ли с кириллицей? Иероглифами? RTL-языками? |

#### 3. Тестируемость (Testability)

Требование должно быть **проверяемым**. Если вы не можете написать тест-кейс,
значит, требование сформулировано неправильно.

```
❌ "Система должна быть надёжной"
   → Невозможно проверить. Что такое "надёжная"?

✅ "Система должна иметь uptime ≥ 99.9% в месяц.
    При падении основного сервера переключение на резервный — за ≤ 30 секунд."
```

### Shift-Left: участие в grooming / refinement

QA должен присутствовать на **Backlog Refinement** и **Sprint Planning**.
Не как слушатель, а как активный участник.

**Ваши задачи на grooming:**
1. Задавать вопросы об edge cases ("А что если корзина пуста?").
2. Уточнять acceptance criteria ("Как мы поймём, что это сделано?").
3. Оценивать тестируемость ("Мне понадобится доступ к логам сервиса").
4. Предлагать разбиение на testable increments.
5. Выявлять зависимости ("Для тестирования оплаты нужен sandbox от платёжной системы").

### Техника "Три амиго" (Three Amigos)

Перед началом работы над user story проводится короткая встреча (15-30 мин):

- **Developer** — "Как я это реализую?"
- **QA** — "Как я это проверю? Какие edge cases?"
- **Product Owner** — "Правильно ли вы поняли бизнес-цель?"

Результат: общее понимание, уточнённые acceptance criteria, раннее выявление рисков.

## Тест-дизайн по требованиям

### От требований к тест-кейсам: практический workflow

```
Требование → Анализ → Техники тест-дизайна → Тест-кейсы → Ревью
```

### Применение техник тест-дизайна к реальным требованиям

#### Пример: форма регистрации

**Требование:** "Пользователь может зарегистрироваться, указав email, пароль
(от 8 до 30 символов, минимум 1 заглавная, 1 строчная, 1 цифра) и возраст (18-120)."

**Equivalence Partitioning (EP) для пароля:**

| Класс | Диапазон | Ожидание |
|-------|---------|----------|
| Слишком короткий | 1-7 символов | Ошибка |
| Валидная длина | 8-30 символов | Успех (при выполнении остальных условий) |
| Слишком длинный | 31+ символов | Ошибка |
| Пустой | 0 символов | Ошибка |

**Boundary Value Analysis (BVA) для пароля:**

| Граница | Значение | Ожидание |
|---------|---------|----------|
| Нижняя - 1 | 7 символов | Ошибка |
| Нижняя | 8 символов | Успех |
| Нижняя + 1 | 9 символов | Успех |
| Верхняя - 1 | 29 символов | Успех |
| Верхняя | 30 символов | Успех |
| Верхняя + 1 | 31 символ | Ошибка |

**Decision Table для пароля (упрощённая):**

| Длина OK | Заглавная | Строчная | Цифра | Результат |
|----------|-----------|----------|-------|-----------|
| Да | Да | Да | Да | Успех |
| Нет | Да | Да | Да | Ошибка: длина |
| Да | Нет | Да | Да | Ошибка: заглавная |
| Да | Да | Нет | Да | Ошибка: строчная |
| Да | Да | Да | Нет | Ошибка: цифра |
| Нет | Нет | Нет | Нет | Ошибка: все |

### Типы тест-кейсов

**Позитивные** — система работает как задумано при корректных входных данных.
Это минимум, который ВСЕГДА должен быть протестирован.

**Негативные** — система корректно обрабатывает невалидные данные.
Здесь живёт большинство багов.

**Edge cases (граничные)** — поведение на границах допустимых значений.
Классические источники: пустая строка, null, максимальная длина, 0, отрицательные числа.

**Деструктивные** — что будет, если сломать flow?
Закрыть браузер во время оплаты, отключить интернет при загрузке файла,
нажать "Назад" после submit формы.

### Cross-browser-соображения при тест-дизайне

При ручном тестировании web-приложений учитывайте:

| Аспект | На что влияет |
|--------|--------------|
| Браузер (Chrome, Firefox, Safari, Edge) | Рендеринг, JS-движок, CSS-поведение |
| ОС (Windows, macOS, Linux) | Шрифты, скроллинг, нативные элементы |
| Размер экрана | Адаптивная вёрстка, mobile vs desktop |
| Сеть | Быстрый Wi-Fi vs медленный 3G |

**Практический подход:** не тестируйте ВСЕМ по ВСЕМУ. Определите основную матрицу:
- Основной браузер + ОС — полное тестирование (например, Chrome + Windows).
- Вторичные комбинации — smoke testing (Safari + macOS, Firefox + Linux).
- Мобильные — критические сценарии (Chrome Mobile, Safari Mobile).

## Выполнение тестов

### Structured Testing vs Exploratory Testing

Это **не** конкуренты. Это **дополняющие** подходы.

#### Structured (Scripted) Testing

**Суть:** выполнение заранее написанных тест-кейсов, шаг за шагом.

**Когда использовать:**
- Регрессионное тестирование (повторяемые сценарии).
- Compliance-тестирование (нужно доказать, что проверено).
- Критические бизнес-сценарии (оплата, регистрация).
- Передача тестирования другому человеку.

**Формат тест-кейса:**

```
ID: TC-REG-001
Название: Успешная регистрация с валидными данными
Предусловия: Пользователь не авторизован, на странице /register
Шаги:
  1. Ввести email: test_unique@example.com
  2. Ввести пароль: SecurePass1
  3. Ввести возраст: 25
  4. Нажать кнопку "Зарегистрироваться"
Ожидаемый результат:
  - Пользователь перенаправлен на /welcome
  - Отображается сообщение "Добро пожаловать!"
  - В БД создана запись пользователя
Приоритет: High
```

#### Exploratory Testing

**Суть:** одновременное изучение системы, проектирование и выполнение тестов.
Тестировщик использует своё мышление, интуицию и опыт для поиска багов.

**Когда использовать:**
- Новая функциональность (ещё нет полных тест-кейсов).
- Поиск "неочевидных" багов.
- Оценка общего качества перед релизом.
- Когда scripted testing "не находит" баги, а пользователи находят.

#### Session-Based Exploratory Testing (SBET)

Структурированный подход к exploratory testing:

**Структура сессии:**
- **Charter (цель):** "Исследовать процесс оформления заказа с точки зрения
  нового пользователя, фокус на error handling."
- **Время:** 60-90 минут (таймбокс!).
- **Заметки:** записывать всё — баги, вопросы, идеи, области для исследования.
- **Дебрифинг:** 15-минутное обсуждение результатов с командой.

**Шаблон отчёта SBET:**

```
Сессия: #12
Дата: 2026-03-23
Тестировщик: Иван
Charter: Исследовать корзину покупок при различных состояниях
Продолжительность: 75 мин (60 тестирование + 15 оформление)

Покрытие:
- Пустая корзина: ✅ проверено
- 1 товар: ✅ проверено
- 100 товаров: ✅ проверено
- Удаление товаров: ✅ проверено
- Изменение количества: ✅ проверено

Найденные баги: 3
- BUG-1: При добавлении 100-го товара корзина зависает на 8 секунд (критично)
- BUG-2: Нет подтверждения при очистке корзины (medium)
- BUG-3: Кнопка "Оформить" доступна при пустой корзине (low)

Области для дальнейшего исследования:
- Корзина + промокоды (не успел)
- Поведение при параллельном доступе с двух устройств
```

### Типы тестовых прогонов

| Тип | Цель | Объём | Когда |
|-----|------|-------|-------|
| **Smoke** | Система "жива" | 10-20 ключевых сценариев | После каждого деплоя |
| **Sanity** | Конкретная фича работает | Тесты для изменённой области | После багфикса |
| **Regression** | Ничего не сломалось | Все критические тест-кейсы | Перед релизом |
| **Re-test** | Баг исправлен | Конкретный баг + связанные | После исправления бага |
| **Exploratory** | Поиск неизвестных багов | По charter-у | Регулярно, перед релизом |

### Приоритизация при нехватке времени

Времени всегда не хватает. Вот framework для принятия решений:

1. **Smoke-тесты** — всегда (10 минут, показывают, жива ли система).
2. **Изменённые области** — всегда (если починили баг — проверяем именно его).
3. **Критические бизнес-flow** — высокий приоритет (оплата, регистрация, авторизация).
4. **Regression смежных областей** — средний приоритет.
5. **Полный regression** — если есть время.
6. **Exploratory** — если есть время (но НЕ пропускайте полностью!).

## Баг-репортинг

### Анатомия отличного баг-репорта

Разработчик должен понять проблему за 30 секунд. Если он спрашивает уточнения —
ваш отчёт можно улучшить.

**Структура:**

```
Заголовок: [Модуль] Краткое описание проблемы
           Плохо: "Кнопка не работает"
           Хорошо: "[Корзина] Кнопка 'Оформить заказ' не реагирует на клик при 50+ товарах"

Окружение: Chrome 120 / Windows 11 / Staging (v2.3.1)

Предусловия: Авторизован как обычный пользователь, в корзине 50 товаров

Шаги воспроизведения:
1. Открыть /cart
2. Нажать кнопку "Оформить заказ"

Фактический результат: Кнопка не реагирует, в консоли ошибка TypeError (скриншот)

Ожидаемый результат: Переход на страницу /checkout

Частота воспроизведения: Всегда при 50+ товарах. При 49 — работает.

Вложения: screenshot_error.png, console_log.txt

Severity: High
Priority: High
```

### Severity vs Priority: framework решений

**Severity** (техническая серьёзность) — определяет QA.

| Severity | Описание | Примеры |
|----------|---------|---------|
| Critical / Blocker | Полная потеря функциональности, нет workaround | Падение приложения, потеря данных, невозможность авторизации |
| Major / High | Основная функция нарушена, есть workaround | Ошибка оплаты картой (можно PayPal), сбой поиска |
| Medium | Второстепенная функция нарушена | Неправильная сортировка, битая ссылка в footer |
| Minor / Low | Косметические проблемы | Опечатка, кривой pixel, неточный цвет |

**Priority** (бизнес-приоритет) — определяет Product Owner / менеджер.

| Priority | Описание | Примеры |
|----------|---------|---------|
| Urgent | Исправить немедленно | Critical баг на продакшене |
| High | Исправить в текущем спринте | Баг в ключевой фиче нового релиза |
| Medium | Исправить в следующем спринте | Улучшение пользовательского опыта |
| Low | Когда будет время | Косметические правки |

**Ключевой инсайт:** Severity и Priority не всегда совпадают.

| Ситуация | Severity | Priority |
|----------|----------|----------|
| Падение сайта | Critical | Urgent |
| Опечатка на главной странице | Low | High (бренд!) |
| Падение в редкоиспользуемом отчёте | Critical | Medium |
| Неправильный цвет footer-а | Low | Low |

### Жизненный цикл бага

```
New → Open → In Progress → Ready for QA → Verified → Closed
                  ↓                            ↓
              Won't Fix                    Reopened → In Progress → ...
                  ↓
              Deferred (отложен на потом)
```

**Статусы, которые использует QA:**

| Статус | Кто ставит | Что значит |
|--------|-----------|-----------|
| New | QA | Баг зарегистрирован |
| Verified | QA | Баг исправлен и проверен |
| Reopened | QA | Баг НЕ исправлен или вернулся |
| Closed | QA / PM | Баг закрыт (после верификации) |

### Когда Reopen vs новый баг

| Reopen | Новый баг |
|--------|----------|
| Тот же сценарий воспроизведения | Другой сценарий |
| Те же шаги, тот же результат | Схожая проблема, но в другом месте |
| Исправление не работает | Исправление сломало что-то другое |
| Прошло < 2 спринтов после закрытия | Прошло много времени, другой контекст |

**Правило:** если разработчику нужно открыть тот же участок кода — Reopen.
Если нужно смотреть другой код — новый баг.

## Sign-off: принятие решения о готовности

### Exit Criteria (критерии выхода)

Exit criteria определяются **до начала тестирования** (в тест-плане).

**Типичные exit criteria:**

| Критерий | Пример порога |
|----------|-------------|
| Все critical/blocker баги исправлены и верифицированы | 0 open critical bugs |
| Процент выполнения тест-кейсов | ≥ 95% executed |
| Процент прохождения тест-кейсов | ≥ 90% passed |
| Open defect density | ≤ 5 medium bugs, ≤ 20 low bugs |
| Regression testing завершён | 100% regression suite |
| Smoke-тесты проходят на production-like среде | 100% pass |
| Performance тесты (если applicable) | Response time ≤ SLA |

### Risk-Based Sign-off

В реальности exit criteria редко выполняются на 100%.
QA должен уметь принимать решение на основе **оценки рисков**.

**Framework:**

1. **Какие баги остаются открытыми?**
   - Сколько? Какой severity?
   - Есть ли workaround?
   - Какие пользователи затронуты?

2. **Какие области НЕ протестированы?**
   - Почему? (не хватило времени, нет тестовых данных, среда не готова)
   - Какой риск? (насколько вероятен баг и каковы последствия?)

3. **Какова "цена" задержки релиза?**
   - Бизнес-давление (маркетинговая кампания, контракт, конкуренты).
   - Можно ли выпустить hotfix быстро?

**Результат — один из трёх:**
- **Go** — релиз. Остаточные риски приемлемы.
- **Conditional Go** — релиз с оговорками (список known issues, план hotfix).
- **No-Go** — релиз невозможен. Чёткий список блокеров.

### Go/No-Go Meeting: роль QA

На Go/No-Go meeting QA представляет:

1. **Статус тестирования** — % выполнения, % прохождения.
2. **Открытые дефекты** — список с severity/priority и рекомендацией.
3. **Непокрытые области** — что не тестировалось и почему.
4. **Рекомендация** — Go / Conditional Go / No-Go **с обоснованием**.

**Важно:** QA **рекомендует**, но **не принимает** решение о релизе.
Финальное решение — за Product Owner / менеджером. Но ваша рекомендация
должна быть профессиональной, объективной и подкреплённой данными.

**Пример рекомендации:**

```
Рекомендация: Conditional Go

Обоснование:
- 98% тест-кейсов выполнено, 94% прошли.
- 0 critical / 0 high / 3 medium / 12 low открытых багов.
- 3 medium бага — в модуле отчётов, workaround доступен.
  План исправления — hotfix через 2 дня.
- Не протестировано: интеграция с новым провайдером SMS (среда не готова).
  Риск: низкий (менее 1% пользователей используют SMS-уведомления).

Условия:
- Hotfix для 3 medium багов в течение 48 часов.
- SMS-интеграция тестируется в рамках следующего спринта.
```

## Связь с тестированием

Ручное тестирование — это **фундамент**, на который ложится всё остальное:

- **Автоматизация** строится на тест-кейсах, созданных при ручном тестировании.
  Нельзя автоматизировать то, что не понимаешь.
- **Exploratory testing** остаётся незаменимым даже при 90% автоматизации.
- **Bug-репортинг** — навык, который используется ежедневно вне зависимости
  от уровня автоматизации.
- **Risk assessment** и **sign-off** — ответственность QA на каждом релизе.

## Типичные ошибки

### Ошибка 1: "Я протестировал — всё работает" без доказательств

Нет задокументированных тест-кейсов, нет отчёта, нет метрик.
Когда баг всплывёт на продакшене, невозможно будет ответить: "А это тестировали?"

**Правильно:** даже минимальная документация (чек-лист) лучше, чем ничего.

### Ошибка 2: Тестирование только "happy path"

QA, который проверяет только успешные сценарии, находит 20% багов.
**80% багов — в негативных, граничных и деструктивных сценариях.**

### Ошибка 3: Баг-репорт без шагов воспроизведения

"Не работает корзина" — это не баг-репорт. Это жалоба.
Разработчик не будет тратить время на "угадывание" проблемы. Это и его право.

### Ошибка 4: Exploratory testing без структуры

"Я похожу по системе и посмотрю" — это не exploratory testing, а прогулка.
Без charter-а, таймбокса и заметок результат будет слабым и невоспроизводимым.

### Ошибка 5: Не участвовать в обсуждении требований

"Мне дадут задачу — я протестирую." Если вы не задаёте вопросы на refinement,
вы тестируете требования, а не продукт. Баги в требованиях самые дорогие.

### Ошибка 6: Путать Severity и Priority

QA определяет Severity (техническая серьёзность). Priority определяет бизнес.
"Опечатка на главной" — Low severity, но может быть High priority для бренда.

### Ошибка 7: Подписать sign-off под давлением

"Нас просят выпустить, давайте подпишем." Если вы знаете о critical-баге
и подписываете Go — это непрофессионально. Ваша задача — дать честную оценку.
Решение принимает менеджмент, но ваша рекомендация должна быть объективной.

## Вопросы на интервью

### 🟢 Junior

1. **Что такое тест-кейс? Какова его структура?**
   Пошаговая инструкция проверки функциональности. Структура: ID, название,
   предусловия, шаги, ожидаемый результат, приоритет.

2. **В чём разница между smoke и regression тестированием?**
   Smoke — быстрая проверка основных функций (система "жива"). Regression — проверка,
   что новые изменения не сломали существующую функциональность (полная или частичная).

3. **Что такое Severity и Priority?**
   Severity — техническая серьёзность бага (Critical → Low). Priority — бизнес-приоритет
   исправления (Urgent → Low). Не всегда совпадают.

4. **Какие данные должен содержать баг-репорт?**
   Заголовок, окружение, предусловия, шаги воспроизведения, фактический результат,
   ожидаемый результат, severity, priority, скриншоты/логи.

5. **Что такое exploratory testing?**
   Одновременное изучение, проектирование и выполнение тестов. Тестировщик использует
   опыт и интуицию для поиска багов, не ограничиваясь заранее написанными сценариями.

### 🟡 Middle

6. **Расскажите о вашем процессе анализа требований.**
   Читаю требования с чек-листом: неоднозначность, пропущенные сценарии,
   тестируемость, edge cases. Участвую в grooming/refinement, задаю вопросы.
   Использую "Три амиго" для уточнения. Документирую выявленные проблемы.

7. **Как вы решаете, когда использовать scripted testing, а когда — exploratory?**
   Scripted — для regression, compliance, критических flow. Exploratory — для
   новой функциональности, поиска "неочевидных" багов, оценки качества.
   Обычно 70% scripted + 30% exploratory.

8. **Как организовать regression testing при нехватке времени?**
   Risk-based подход: (1) smoke-тесты всегда, (2) изменённые области,
   (3) критические бизнес-flow, (4) смежные области, (5) полный regression
   если есть время. Автоматизировать то, что выполняется каждый раз.

9. **Что такое exit criteria? Приведите примеры.**
   Условия, при которых тестирование считается завершённым. Примеры:
   0 critical/high open bugs, 95%+ тест-кейсов выполнено, 90%+ passed,
   regression suite пройден, smoke на staging пройден.

### 🔴 Senior

10. **Продакшен падает после релиза, который вы подписали. Ваши действия?**
    Немедленно: помогаю диагностировать (логи, шаги воспроизведения).
    Краткосрочно: анализирую, почему баг не был найден при тестировании.
    Долгосрочно: root cause analysis — улучшаю процесс (добавляю тесты,
    меняю exit criteria, добавляю окружение, усиливаю regression).
    Ключевое: не ищу виноватых, а ищу системные улучшения.

11. **Как выстроить процесс ручного тестирования с нуля на новом проекте?**
    Неделя 1: анализ требований, smoke-чеклист, exploratory сессии.
    Неделя 2: тест-кейсы для критических flow, bug-tracking процесс.
    Неделя 3-4: полный regression suite, exit criteria, sign-off процесс.
    Месяц 2+: оптимизация — risk-based приоритизация, начало автоматизации.

12. **Product Owner просит подписать Go, но у вас есть 2 open high-priority бага. Что вы делаете?**
    Показываю данные: какие баги, кого затрагивают, есть ли workaround,
    каков реальный риск. Даю рекомендацию Conditional Go с планом hotfix
    или No-Go с timeline исправления. Решение — за PO, но я даю
    профессиональную оценку и фиксирую её письменно.

## Практические задания

### Задание 1: Анализ требований
Возьмите следующее требование и найдите все проблемы:
"Система должна позволять пользователям создавать заказы. Заказ может содержать
товары. После создания заказа пользователь получает уведомление."

Цель: найти минимум 10 вопросов / пропущенных требований.

### Задание 2: Тест-дизайн
Напишите тест-кейсы для следующего требования:
"Поле 'Возраст' принимает целые числа от 18 до 120. При невалидном вводе
показывается сообщение об ошибке."

Применяйте EP, BVA, decision table. Минимум 15 тест-кейсов.

### Задание 3: Exploratory testing session
Выберите любое веб-приложение (можно тестовое: http://the-internet.herokuapp.com/).
Проведите 60-минутную exploratory session:
1. Напишите charter.
2. Засеките время.
3. Ведите заметки по ходу тестирования.
4. Оформите отчёт по шаблону SBET.
5. Заведите найденные баги по формату из этого раздела.

### Задание 4: Баг-репортинг
Найдите 5 багов в любом публичном веб-приложении. Оформите каждый по формату:
заголовок, окружение, предусловия, шаги, фактический результат, ожидаемый,
severity, priority. Попросите коллегу воспроизвести баги только по вашим отчётам.

### Задание 5: Sign-off решение
Дано: перед релизом осталось 2 critical, 5 high, 15 medium, 30 low открытых багов.
Выполнено 88% тест-кейсов, из них 85% passed. Deadline — завтра, маркетинговая
кампания уже запущена.
Задача: подготовьте рекомендацию (Go / Conditional Go / No-Go) с обоснованием.
