# Вопросы к интервьюеру

## Обзор

"У вас есть вопросы к нам?" — этот момент наступает в конце каждого собеседования,
и многие кандидаты его недооценивают. Хорошие вопросы показывают вашу заинтересованность,
глубину понимания профессии и зрелость. Плохие (или их отсутствие) — сигнализируют
о безразличии или неопытности. Для QA инженера вопросы к интервьюеру — это, по сути,
**requirements gathering** для принятия решения о работе. Как и при тестировании:
чем лучше вы понимаете систему, тем лучше решение вы примете.

---

## Почему важно задавать вопросы

### Что видит интервьюер

| Кандидат | Впечатление |
|----------|------------|
| Не задаёт вопросов | "Ему всё равно, куда идти" |
| Спрашивает только про зарплату и отпуск | "Думает только о себе" |
| Задаёт шаблонные вопросы | "Подготовился формально" |
| Задаёт умные, глубокие вопросы | "Разбирается в профессии, думает о деле" |

### Что получаете вы

- **Реальное понимание** позиции и компании (job description часто приукрашен)
- **Red flags** — если интервьюер уклоняется от ответов, это сигнал
- **Дополнительные очки** — хороший вопрос может выделить вас среди кандидатов
- **Основу для решения** — стоит ли принимать оффер

### Сколько вопросов задавать

- **Минимум:** 3-4 вопроса
- **Оптимально:** 5-7 вопросов
- **Максимум:** не более 10 (не превращайте в допрос)
- **Время:** 10-15 минут на вопросы и ответы

## Категория 1: QA процессы

Эти вопросы показывают, что вы думаете о процессах, а не только о технологиях.

### Вопросы

1. **"Какое соотношение manual и automation тестирования на проекте?"**
   - Зачем: понять зрелость QA и свою будущую роль
   - Red flag: "У нас всё мануально, но мы хотим автоматизировать" (часто означает, что вы будете строить всё с нуля без поддержки)
   - Green flag: конкретные цифры, чёткое понимание стратегии

2. **"Есть ли тестовая стратегия? Кто её формирует?"**
   - Зачем: понять, насколько зрелый QA-процесс
   - Red flag: "Что такое тестовая стратегия?"
   - Green flag: "Да, мы пересматриваем её каждый квартал, QA Lead отвечает"

3. **"Что входит в Definition of Done для задачи?"**
   - Зачем: понять, насколько качество встроено в процесс
   - Red flag: "Когда разработчик говорит, что готово"
   - Green flag: "Код прошёл code review, unit-тесты написаны, автотесты зелёные, QA провёл тестирование"

4. **"Как устроен релизный процесс? Как часто деплоите?"**
   - Зачем: понять скорость и процессы
   - Red flag: "Релизим, когда готово, обычно раз в 2-3 месяца" (возможно, waterfall)
   - Green flag: "CI/CD, деплой несколько раз в день с feature flags"

5. **"На каком этапе подключается QA — после разработки или раньше?"**
   - Зачем: понять, есть ли shift-left
   - Red flag: "QA получает задачу, когда она уже в тестовом окружении"
   - Green flag: "QA участвует в refinement, пишет тест-сценарии до начала разработки"

6. **"Как происходит управление тестовыми данными и окружениями?"**
   - Зачем: понять инфраструктурную зрелость
   - Red flag: "Все тестируют на одном окружении, данные общие"
   - Green flag: "У каждого QA своё окружение, тестовые данные генерируются автоматически"

## Категория 2: Технический стек

Эти вопросы показывают вашу техническую глубину.

### Вопросы

7. **"Какой стек используется для автоматизации тестирования?"**
   - Зачем: понять, совпадает ли с вашим опытом
   - Уточнение: "Есть ли возможность влиять на выбор инструментов?"

8. **"Как устроен CI/CD pipeline для тестов?"**
   - Зачем: понять зрелость инфраструктуры
   - Уточнение: "Тесты запускаются на каждый PR или только перед релизом?"
   - Red flag: "Тесты запускают руками, когда не забудут"
   - Green flag: "Smoke на каждый PR, полный регресс ночью, отчёты в Slack"

9. **"Какая архитектура у продукта — монолит, микросервисы?"**
   - Зачем: понять сложность тестирования
   - Уточнение: "Как тестируете интеграции между сервисами?"

10. **"Используете ли вы contract testing, performance testing, security testing?"**
    - Зачем: понять ширину QA-покрытия
    - Green flag: если есть хотя бы 2 из 3

11. **"Как устроена отчётность по тестам? Какие метрики отслеживаете?"**
    - Зачем: понять data-driven подход к QA
    - Red flag: "Никаких метрик не собираем"
    - Green flag: "Allure отчёты, dashboard в Grafana, отслеживаем стабильность и время прогона"

12. **"Есть ли технический долг в тестах? Как с ним работаете?"**
    - Зачем: понять реалистичность ожиданий
    - Red flag: уклонение от ответа
    - Green flag: "Да, есть, выделяем 20% спринта на tech debt"

## Категория 3: Команда и взаимодействие

Эти вопросы показывают, что вы умеете работать с людьми.

### Вопросы

13. **"Сколько человек в QA команде? Какая структура?"**
    - Зачем: понять масштаб и свою роль
    - Уточнение: "QA embedded в Scrum-команды или отдельный отдел?"

14. **"Как устроено взаимодействие QA с разработчиками?"**
    - Зачем: понять культуру коллаборации
    - Red flag: "QA отдельно, разработчики отдельно, общаемся через Jira"
    - Green flag: "QA и разработчики в одной Scrum-команде, парное тестирование, обсуждения"

15. **"Есть ли code review для тестового кода? Кто ревьюит?"**
    - Зачем: понять отношение к качеству тестов
    - Red flag: "Нет, тестовый код не ревьюим"
    - Green flag: "Да, другие QA и иногда разработчики"

16. **"Как происходит онбординг нового QA?"**
    - Зачем: понять, будет ли поддержка на старте
    - Red flag: "Вот тебе доступы, разбирайся"
    - Green flag: "Менторство первые 2 месяца, план онбординга, buddy"

17. **"Кому подчиняется QA — QA Lead, Engineering Manager, CTO?"**
    - Зачем: понять структуру принятия решений
    - Уточнение: "Есть ли у QA голос в технических решениях?"

18. **"Как разрешаются конфликты по severity / приоритету багов?"**
    - Зачем: понять зрелость процесса принятия решений
    - Red flag: "Разработчик решает, что чинить"
    - Green flag: "Есть процесс triage с участием PO, QA Lead и Tech Lead"

---

## Категория 4: Культура и развитие

Эти вопросы показывают вашу ориентацию на рост.

### Вопросы

19. **"Какое отношение к качеству в компании? Качество — приоритет или 'если успеем'?"**
    - Зачем: понять реальное отношение к QA
    - Red flag: "Бизнес всегда давит на скорость" (без уточнения, как с этим справляются)
    - Green flag: "Качество — часть DoD, мы не выпускаем с known critical bugs"

20. **"Есть ли бюджет на обучение? Конференции, курсы, книги?"**
    - Зачем: понять инвестиции в развитие сотрудников
    - Уточнение: "Какие последние конференции / курсы посещала команда?"

21. **"Как выглядит карьерный рост для QA в компании?"**
    - Зачем: понять перспективы
    - Red flag: "У нас плоская структура" (может означать отсутствие роста)
    - Green flag: "Junior → Middle → Senior → Lead → QA Manager, есть грейдовая система"

22. **"Есть ли внутренние knowledge sharing сессии, tech talks?"**
    - Зачем: понять культуру обмена знаниями
    - Green flag: "Да, каждые 2 недели, и QA активно участвует"

23. **"Какие главные QA-вызовы сейчас стоят перед командой?"**
    - Зачем: понять, что вас ждёт, и показать готовность к решению проблем
    - Отличный вопрос, потому что ответ раскрывает реальное состояние дел

24. **"Что вам лично нравится / не нравится в работе здесь?"**
    - Зачем: получить честный ответ (личный вопрос вызывает более искренний ответ)
    - Задавайте, если чувствуете доброжелательную атмосферу

---

## Категория 5: Продукт и бизнес

Эти вопросы показывают, что вы думаете шире, чем "просто тестирование".

### Вопросы

25. **"Кто основные пользователи продукта? Какой масштаб?"**
    - Зачем: понять бизнес-контекст и нагрузку

26. **"Какие ближайшие планы по продукту на 3-6 месяцев?"**
    - Зачем: понять стабильность и направление развития

27. **"Как вы собираете обратную связь от пользователей о качестве?"**
    - Зачем: понять, есть ли feedback loop
    - Green flag: "Мониторинг ошибок в Sentry, NPS, анализ support-тикетов"

28. **"Какой SLA по времени реакции на production-баги?"**
    - Зачем: понять серьёзность отношения к продакшену
    - Green flag: "Critical — 1 час, Major — 4 часа, есть on-call ротация"

## Стратегия задавания вопросов

### Подготовка

1. **Изучите компанию** до собеседования — не задавайте вопросы, ответы на которые есть на сайте
2. **Подготовьте 10-12 вопросов** — часть ответов вы получите в ходе интервью
3. **Приоритизируйте** — самые важные вопросы задавайте первыми
4. **Запишите** — блокнот или заметки на экране (для онлайн)

### Во время интервью

1. **Слушайте внимательно** — не задавайте вопросы, на которые уже ответили
2. **Задавайте follow-up** — "Вы упомянули CI/CD — как именно это устроено?"
3. **Не спорьте** с ответами — вы собираете информацию, а не доказываете правоту
4. **Делайте заметки** — запишите ответы для анализа после интервью

### После интервью

1. **Проанализируйте** ответы: сколько green flags vs red flags?
2. **Сравните** с другими компаниями, если есть несколько предложений
3. **Уточните** оставшиеся вопросы на следующем этапе собеседования

## Связь с тестированием

Задавание вопросов интервьюеру — это **requirements gathering и exploratory testing** одновременно:

- **Requirements gathering:** вы собираете "требования" к будущей работе, как QA собирает требования к системе
- **Exploratory testing:** вы исследуете компанию, ищете "баги" в процессах и культуре
- **Risk analysis:** каждый red flag — это потенциальный "баг", который может повлиять на вашу работу
- **Test plan:** ваш список вопросов — это test plan для оценки компании

QA инженер, который задаёт хорошие вопросы на интервью, демонстрирует те же навыки,
что и при тестировании: любопытство, структурность, внимание к деталям, умение
докапываться до сути.

## Типичные ошибки

### 1. "У меня нет вопросов"
**Почему плохо:** сигнализирует безразличие или неподготовленность
**Что делать:** всегда иметь хотя бы 3-4 вопроса

### 2. Вопросы, ответы на которые есть на сайте
**Плохо:** "Чем занимается ваша компания?"
**Хорошо:** "Я видел, что вы работаете в сфере X. Какие основные QA-вызовы в этом домене?"

### 3. Только вопросы про зарплату и бенефиты
**Плохо:** "Сколько дней отпуска? Есть ли ДМС? Какой оклад?"
**Хорошо:** Эти вопросы уместны, но не единственные. Спросите после профессиональных вопросов

### 4. Слишком агрессивные вопросы
**Плохо:** "Почему от вас уволился предыдущий QA?"
**Хорошо:** "Как давно открыта эта позиция? Это новая роль или замена?"

### 5. Вопросы, показывающие незаинтересованность
**Плохо:** "Можно ли работать 100% удалённо и не включать камеру?"
**Хорошо:** "Какой формат работы у команды? Есть ли гибкость?"

### 6. Слишком много вопросов
**Плохо:** 20 вопросов в режиме допроса
**Хорошо:** 5-7 вопросов с follow-up по ответам

### 7. Не слушать ответы
**Плохо:** Задавать следующий вопрос, не отреагировав на ответ
**Хорошо:** "Интересно, что вы используете Selenoid. А как решаете проблему параллельного запуска?"

## Вопросы на интервью

Вопросы, которые могут задать вам о ваших вопросах:

### 🟢 Базовый уровень
1. **"Какие вопросы вы бы задали, чтобы понять зрелость QA-процесса?"** — покажите знание QA maturity model
2. **"Что для вас важно при выборе работы?"** — честность + профессиональные приоритеты
3. **"Как вы оцениваете, подходит ли вам компания?"** — критерии и процесс принятия решений

### 🟡 Средний уровень
4. **"На что бы вы обратили внимание в описании вакансии QA?"** — red flags и green flags
5. **"Что бы вас оттолкнуло от оффера?"** — покажите зрелые критерии оценки
6. **"Как бы вы оценили текущее состояние QA в компании за 30 минут разговора?"** — структурный подход

### 🔴 Продвинутый уровень
7. **"Если бы вы нанимали QA, какие вопросы бы задали?"** — понимание профессии с обеих сторон
8. **"Как вы оцениваете культуру качества в организации?"** — зрелое понимание Quality Engineering

## Практические задания

### Задание 1: Составьте свой список вопросов
Выберите 10-12 вопросов из этого файла, которые наиболее релевантны для вас.
Добавьте 2-3 своих уникальных вопроса, основанных на вашем опыте.

### Задание 2: Приоритизация для конкретной компании
Возьмите реальную вакансию и составьте список из 7 вопросов, адаптированных
именно под эту компанию. Используйте информацию из описания вакансии,
сайта компании, отзывов на Glassdoor / Habr Career.

### Задание 3: Анализ red flags
Вспомните свои предыдущие собеседования. Были ли red flags, которые вы
пропустили? Какие вопросы помогли бы выявить проблемы раньше?

### Задание 4: Mock-интервью — задаём вопросы
Попросите коллегу или друга сыграть роль интервьюера. Задайте ему
ваши вопросы. Попросите ответить максимально реалистично (в том числе
уклончиво). Тренируйте follow-up вопросы.

### Задание 5: Сравнительная таблица
Если вы проходите интервью в нескольких компаниях, создайте таблицу:

| Критерий | Компания A | Компания B | Компания C |
|----------|-----------|-----------|-----------|
| Соотношение manual/auto | | | |
| CI/CD зрелость | | | |
| Размер QA команды | | | |
| Code review тестов | | | |
| Бюджет на обучение | | | |
| Карьерный рост | | | |
| Отношение к качеству | | | |
| Общее впечатление | | | |

Заполняйте после каждого интервью на основе ответов на ваши вопросы.

### Задание 6: Вопросы на английском
Подготовьте 5-7 ключевых вопросов на английском. Обратите внимание:
- "Could you tell me about..." (мягче, чем "Tell me about...")
- "I'm curious about..." (показывает интерес)
- "How does the team handle..." (про процессы)
- "What does a typical day look like for a QA here?" (конкретика)

## Дополнительные ресурсы

1. **"Reverse Interview"** — репозиторий на GitHub: github.com/viraptor/reverse-interview (список вопросов к компаниям, в том числе для инженеров)

5. **Telegram-каналы QA** — обсуждения реальных собеседований и вопросов
6. **Подкасты**: "QA Guild", "Software Testing Podcast" — интервью с QA-лидерами о процессах и культуре
