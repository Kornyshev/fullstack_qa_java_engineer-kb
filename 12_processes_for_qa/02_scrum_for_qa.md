# QA в Scrum

## Обзор

Scrum — наиболее популярный Agile-фреймворк, используемый в разработке ПО. Для QA-инженера важно не просто знать
теорию Scrum, а понимать, **как именно тестирование встраивается в каждую церемонию и артефакт Scrum**. QA в Scrum —
это не отдельная фаза «после разработки», а непрерывный процесс обеспечения качества, интегрированный в каждый
аспект спринта.

В этом разделе разберём каждую Scrum-церемонию с точки зрения QA, роль Definition of Done, особенности планирования
тестирования в спринтах и типичные проблемы, с которыми сталкивается QA в Scrum-команде.

---

## Scrum: ключевые элементы

### Роли в Scrum

| Роль | Описание | Взаимодействие с QA |
|---|---|---|
| **Product Owner** | Определяет что делать, приоритизирует backlog | QA уточняет acceptance criteria, обсуждает edge cases |
| **Scrum Master** | Фасилитирует процесс, устраняет impediments | QA сообщает о блокерах, SM помогает их устранить |
| **Development Team** | Реализует user stories | QA — часть Dev Team; тесно работает с разработчиками |

> **Важно:** В Scrum QA-инженер — полноценный член Development Team, а не внешний наблюдатель.

### Артефакты Scrum

- **Product Backlog** — приоритизированный список всех user stories и задач
- **Sprint Backlog** — user stories, взятые в текущий спринт
- **Increment** — работоспособный прирост продукта к концу спринта

## Церемонии Scrum с позиции QA

### 1. Sprint Planning (Планирование спринта)

**Цель:** определить, какие user stories команда возьмёт в спринт и как будет их реализовывать.

**Действия QA на Sprint Planning:**

- **Оценка трудозатрат на тестирование** — каждая story должна учитывать время на QA
- **Обсуждение acceptance criteria** — проверка, что критерии достаточно конкретны и тестируемы
- **Выявление зависимостей** — нужны ли тестовые данные, mock-сервисы, доступ к окружениям
- **Планирование типов тестирования** — для каждой story определить: ручное, автоматизированное, нагрузочное
- **Оценка capacity** — сколько QA-часов доступно, есть ли отпуска, другие обязательства

**Пример участия QA в оценке story:**

```
User Story: "Как пользователь, я хочу сбросить пароль по email"

QA-вклад в оценку:
- Написание test cases: 2ч
- Ручное тестирование (happy path + negative): 3ч
- Написание API-автотестов: 4ч
- Тестирование email-уведомлений: 1ч
- Итого QA effort: ~10ч (добавляется к общей оценке story)
```

**Красные флаги на Planning:**
- Story без acceptance criteria → вернуть в backlog для уточнения
- Story слишком большая для одного спринта → предложить разбить
- Все stories зависят друг от друга → QA не сможет тестировать параллельно

---

### 2. Grooming / Refinement (Уточнение Backlog)

**Цель:** подготовить user stories из backlog к будущим спринтам. Обычно проводится за 1-2 спринта до взятия story в работу.

**Действия QA на Grooming:**

- **Анализ тестируемости (testability review):**
  - Можно ли проверить каждое acceptance criteria?
  - Есть ли чёткие входные и выходные данные?
  - Определены ли граничные условия?

- **Уточнение acceptance criteria:**
  - Добавление недостающих критериев
  - Формулирование негативных сценариев
  - Определение non-functional требований (performance, security)

- **Формулирование вопросов:**
  - Что происходит при ошибке?
  - Как система ведёт себя при concurrent-запросах?
  - Есть ли backward compatibility?

**Пример уточнения acceptance criteria:**

```
Исходные AC:
✓ Пользователь может добавить товар в корзину

Уточнённые AC после вопросов QA:
✓ Пользователь может добавить товар в корзину (максимум 99 единиц одного товара)
✓ При попытке добавить товар с количеством 0 — показать ошибку валидации
✓ При добавлении товара, которого нет в наличии, — показать сообщение "Нет в наличии"
✓ Корзина сохраняется после logout/login
✓ Цена в корзине обновляется при изменении цены товара
✓ Время добавления товара в корзину < 500ms (при 1000 concurrent пользователях)
```

### 3. Daily Standup (Ежедневная встреча)

**Цель:** синхронизация команды. Каждый участник отвечает на три вопроса: что сделал вчера, что планирует сегодня,
есть ли блокеры.

**Типичные ответы QA на Daily:**

```
Вчера:
- Завершил тестирование story QA-123 "Сброс пароля" — нашёл 2 бага
- Написал автотесты для API endpoint POST /users — 5 test cases
- Проверил fix бага BUG-456 — подтверждён

Сегодня:
- Начну тестирование story QA-124 "Профиль пользователя"
- Исследую flaky test в модуле авторизации
- Обновлю тестовые данные на staging

Блокеры:
- Story QA-125 не могу тестировать — нет доступа к payment gateway на staging
- Жду от Ивана API-документацию по новому endpoint
```

**Что QA должен сообщать на Daily:**
- Статус тестирования текущих stories
- Количество и критичность найденных багов
- Блокеры: окружение не работает, нет тестовых данных, зависимость от другой story
- Риски: не успеваю протестировать до конца спринта

**Антипаттерны на Daily для QA:**
- Подробное описание каждого найденного бага (для этого есть bug tracker)
- Обсуждение технических деталей (выносить в отдельный разговор после daily)
- Молчание о блокерах («как-нибудь разберусь сам»)

### 4. Sprint Review (Обзор спринта / Демо)

**Цель:** продемонстрировать стейкхолдерам работоспособный прирост продукта.

**Действия QA:**

- **Подготовка к демо:**
  - Убедиться, что демо-окружение стабильно
  - Подготовить тестовые данные для демонстрации
  - Проверить, что все stories, идущие в демо, полностью протестированы

- **Участие в демо:**
  - QA может демонстрировать фичи, которые он тестировал
  - Предоставить информацию о качестве: какие areas были протестированы, какие риски остались
  - Ответить на вопросы стейкхолдеров о тестовом покрытии

- **Предоставление метрик:**
  - Количество найденных/исправленных дефектов
  - Процент автоматизации
  - Результаты регрессионного тестирования

**Пример QA-слайда для Sprint Review:**

```
Sprint 14 — QA Summary:
- Stories протестированы: 8/8 (100%)
- Дефекты найдены: 12 (3 Critical, 5 Major, 4 Minor)
- Дефекты исправлены: 11/12 (1 Minor перенесён в следующий спринт)
- Автотесты написаны: 23 новых (API: 15, UI: 8)
- Регрессия: 245 автотестов пройдены, 2 flaky исправлены
- Тестовое покрытие: 87% (requirements coverage)
```

### 5. Retrospective (Ретроспектива)

**Цель:** обсудить, что было хорошо, что можно улучшить, и определить конкретные action items.

**QA-темы для ретроспективы:**

**Что пошло хорошо (went well):**
- Раннее подключение QA к grooming сэкономило время на перетестирование
- Новый подход к тестовым данным ускорил тестирование на 30%
- Автоматизация smoke-тестов сократила время проверки билда

**Что можно улучшить (to improve):**
- Stories приходят на тестирование в последний день спринта → предложить лимит WIP
- Acceptance criteria недостаточно детальны → ввести QA sign-off на grooming
- Flaky tests замедляют CI → выделить время на стабилизацию

**Пример action items от QA:**
- Ввести правило: story нельзя взять в спринт без acceptance criteria, одобренных QA
- Создать шаблон для bug-репорта, чтобы улучшить коммуникацию с разработчиками
- Настроить автоматическую нотификацию QA при смене статуса story на «Ready for QA»

## Definition of Done (DoD) с позиции QA

### Что такое DoD

**Definition of Done** — это чеклист критериев, которым должна соответствовать каждая user story,
чтобы считаться завершённой. DoD — соглашение всей команды.

### Типичные QA-пункты в DoD

```
Definition of Done:

Разработка:
☐ Код написан и прошёл code review
☐ Unit-тесты написаны (покрытие ≥ 80%)
☐ Код смержен в develop-ветку
☐ CI pipeline проходит успешно (green build)

Тестирование:
☐ Все acceptance criteria проверены и пройдены
☐ Негативные сценарии протестированы
☐ API-автотесты написаны и проходят
☐ Регрессионные тесты не сломаны
☐ Нет открытых Critical/Blocker дефектов
☐ Тестовая документация обновлена

Деплой:
☐ Feature задеплоена на staging
☐ Smoke-тестирование на staging пройдено

Документация:
☐ API-документация обновлена (Swagger/OpenAPI)
☐ Release notes подготовлены (при необходимости)
```

### Зачем QA нужен DoD

- **Предотвращает «done, but not tested»** — story не может быть закрыта без QA sign-off
- **Стандартизирует качество** — каждая story проходит одинаковый набор проверок
- **Защищает QA от давления** — «мы не можем сократить тестирование, это в DoD»
- **Делает качество измеримым** — можно отслеживать, как часто DoD нарушается

## Планирование тестирования в спринте

### Распределение времени QA в спринте (пример для 2-недельного спринта)

```
День 1-2:   Sprint Planning + начало анализа stories
День 3-4:   Написание test cases, подготовка тестовых данных
День 5-7:   Тестирование первых готовых stories, написание автотестов
День 8-9:   Тестирование оставшихся stories, retesting багов
День 10:    Регрессионное тестирование, подготовка к Sprint Review
```

### Стратегия: не копить testing debt

Один из главных антипаттернов — когда все stories приходят на тестирование в последние 2 дня спринта.

**Решения:**
- **WIP Limit** — ограничить количество stories «In Development» (например, максимум 3)
- **Swarming** — вся команда фокусируется на завершении одной story, включая тестирование
- **Continuous testing** — QA начинает тестировать частично готовую функциональность
- **Test-first approach** — автотесты пишутся до или одновременно с кодом

## Связь с тестированием

- Sprint Planning определяет **scope тестирования** на ближайший спринт
- Grooming обеспечивает **качество требований** — основу для хороших тестов
- Daily даёт **быструю обратную связь** о блокерах и статусе тестирования
- Sprint Review демонстрирует **результат тестирования** стейкхолдерам
- Retrospective позволяет **улучшать процесс тестирования** спринт за спринтом
- DoD гарантирует, что **тестирование не будет пропущено** под давлением сроков

---

## Типичные ошибки

1. **QA как bottleneck в конце спринта** — все stories приходят одновременно, QA не успевает
2. **Отсутствие QA на Grooming** — test cases пишутся по неполным требованиям
3. **«QA — не часть Dev Team»** — QA сидит отдельно, получает задачи «через забор»
4. **DoD без QA-пунктов** — story закрывается без тестирования
5. **Нет estimation на тестирование** — при планировании учитывается только время разработки
6. **QA не говорит о блокерах на Daily** — проблемы накапливаются
7. **Пропуск ретроспективы** — QA-процесс не улучшается от спринта к спринту
8. **Mini-waterfall внутри спринта** — сначала все разрабатывают, потом все тестируют

## Практические задания

### Задание 1: DoD для вашего проекта
Создайте Definition of Done для вашего текущего (или гипотетического) проекта.
Включите минимум 5 QA-специфичных пунктов. Обоснуйте каждый.

### Задание 2: Sprint Planning симуляция
Возьмите 3 user stories. Для каждой:
- Оцените QA effort в story points или часах
- Определите виды тестирования
- Выявите зависимости и риски

### Задание 3: Retrospective analysis
Вспомните последние 3 спринта. Для каждого определите:
- Одну вещь, которая в QA-процессе работала хорошо
- Одну вещь, которую нужно улучшить
- Один конкретный action item

### Задание 4: Grooming checklist
Создайте чеклист вопросов, которые QA должен задать на Grooming для любой user story.
Минимум 10 вопросов, сгруппированных по категориям (функциональность, производительность, безопасность, UX).
