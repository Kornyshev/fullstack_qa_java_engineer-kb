# SDLC и STLC

## Обзор

**SDLC** (Software Development Life Cycle) и **STLC** (Software Testing Life Cycle) — два фундаментальных процесса,
которые определяют, как создаётся и тестируется программное обеспечение. Понимание этих циклов критически важно для
QA-инженера: оно позволяет встроиться в процесс разработки на правильных этапах, выбрать подходящие виды тестирования
и обеспечить качество продукта от момента появления идеи до выпуска в production.

QA-инженер — не просто «тестировщик, который проверяет готовый код». Это участник каждой фазы разработки, который
привносит перспективу качества на всех этапах.

---

## SDLC — Software Development Life Cycle

### Определение

SDLC — это структурированный процесс создания программного обеспечения, который описывает последовательность этапов
от идеи до вывода продукта из эксплуатации.

### Фазы SDLC

#### 1. Requirements Gathering & Analysis (Сбор и анализ требований)

На этом этапе бизнес-аналитики, product owner и стейкхолдеры формулируют, **что** должна делать система.

**Роль QA:**
- Участие в review требований (requirements review)
- Выявление неоднозначностей, противоречий и пробелов в требованиях
- Формулирование вопросов к BA/PO: «А что произойдёт, если...?»
- Оценка тестируемости (testability) требований
- Раннее создание high-level test scenarios

**Пример вклада QA:**
```
Требование: "Пользователь может зарегистрироваться в системе."

Вопросы QA:
- Какие поля обязательны для заполнения?
- Есть ли ограничения на пароль (длина, сложность)?
- Что происходит при попытке регистрации с уже существующим email?
- Нужна ли верификация email?
- Есть ли возрастные ограничения?
- Поддерживается ли регистрация через OAuth (Google, GitHub)?
```

#### 2. System Design (Проектирование системы)

Архитекторы и senior-разработчики определяют **как** будет реализована система: архитектура, технологический стек,
схема базы данных, API-контракты.

**Роль QA:**
- Участие в design review
- Анализ API-контрактов для будущего API-тестирования
- Понимание архитектуры для планирования интеграционного тестирования
- Выявление потенциальных точек отказа (failure points)
- Подготовка тестовой инфраструктуры (тестовые окружения, данные)

#### 3. Implementation / Coding (Разработка)

Разработчики пишут код. Параллельно QA активно работает.

**Роль QA:**
- Написание детальных test cases на основе требований и дизайна
- Подготовка тестовых данных
- Разработка автоматизированных тестов (API-тесты можно писать по контракту до готовности кода)
- Настройка CI/CD-пайплайнов для тестирования
- Code review с точки зрения тестируемости

#### 4. Testing (Тестирование)

Основная фаза верификации и валидации.

**Роль QA:**
- Выполнение всех уровней тестирования (unit, integration, system, acceptance)
- Выполнение различных видов тестирования (functional, non-functional, regression)
- Репортинг дефектов
- Ретестирование исправленных багов
- Отчётность о статусе тестирования

#### 5. Deployment (Развёртывание)

Выпуск продукта в production.

**Роль QA:**
- Smoke-тестирование на staging-окружении
- Участие в go/no-go meeting с рекомендацией о готовности к релизу
- Smoke-тестирование после deployment в production
- Мониторинг ошибок в первые часы после релиза

#### 6. Maintenance (Сопровождение)

Поддержка продукта после выпуска: исправление багов, доработки, обновления.

**Роль QA:**
- Регрессионное тестирование при каждом изменении
- Верификация hotfix-ов
- Обновление тестовой документации и автотестов
- Анализ production-инцидентов

## STLC — Software Testing Life Cycle

### Определение

STLC — это последовательность действий, выполняемых в процессе тестирования. STLC является подмножеством SDLC,
но имеет собственную структуру и артефакты.

### Фазы STLC

| Фаза | Входные данные | Выходные данные |
|---|---|---|
| **Requirements Analysis** | Requirements, acceptance criteria | RTM (Requirements Traceability Matrix), список вопросов |
| **Test Planning** | Requirements, RTM | Test Plan, оценка трудозатрат |
| **Test Case Design** | Test Plan, requirements | Test Cases, Test Data |
| **Environment Setup** | Test Plan, архитектура | Готовое тестовое окружение |
| **Test Execution** | Test Cases, тестовое окружение | Execution Report, Bug Reports |
| **Test Closure** | Execution Report | Test Summary Report, Lessons Learned |

### Детали каждой фазы STLC

**Requirements Analysis** — QA изучает требования, определяет scope тестирования, создаёт RTM для отслеживания
покрытия каждого требования тестами.

**Test Planning** — создание стратегии тестирования: какие виды тестирования применять, какие инструменты использовать,
сколько времени нужно, какие риски существуют.

**Test Case Design** — написание конкретных шагов для проверки каждого требования. Включает позитивные, негативные
сценарии, граничные значения, edge cases.

**Environment Setup** — подготовка серверов, баз данных, тестовых учётных записей, mock-сервисов.

**Test Execution** — запуск тестов, документирование результатов, создание bug-репортов.

**Test Closure** — подведение итогов: какие тесты пройдены, какие дефекты найдены, какие области недостаточно покрыты.

## V-Model (Верификация и Валидация)

V-Model — модель разработки, в которой каждому этапу разработки (левая сторона «V») соответствует
этап тестирования (правая сторона «V»).

```
Requirements Analysis  ←————————→  Acceptance Testing
    ↓                                    ↑
  System Design        ←————————→  System Testing
      ↓                                ↑
    Architecture Design ←————————→  Integration Testing
        ↓                            ↑
      Coding           ←————————→  Unit Testing
```

**Принцип V-Model:** тестирование планируется параллельно с разработкой, а не после неё. Это одна из первых моделей,
продвигающих идею раннего вовлечения QA.

**Связь уровней:**
- **Unit Testing** верифицирует код отдельных модулей → соответствует этапу Coding
- **Integration Testing** проверяет взаимодействие модулей → соответствует Architecture Design
- **System Testing** проверяет систему целиком → соответствует System Design
- **Acceptance Testing** валидирует соответствие бизнес-требованиям → соответствует Requirements Analysis

## Agile SDLC

### Итеративный подход

В Agile SDLC все фазы классического цикла (требования, дизайн, разработка, тестирование, деплой) происходят
**в каждом спринте** (обычно 1-4 недели).

```
Sprint 1:  [Requirements → Design → Code → Test → Deploy]
Sprint 2:  [Requirements → Design → Code → Test → Deploy]
Sprint 3:  [Requirements → Design → Code → Test → Deploy]
...
```

### QA в Agile

В Agile-командах QA-инженер — полноправный член команды, а не внешний «проверяющий».

**Ключевые принципы:**
- **Whole team approach** — качество является ответственностью всей команды
- **Continuous testing** — тестирование происходит непрерывно, а не в конце спринта
- **Test-first mindset** — тесты пишутся до или одновременно с кодом (TDD, BDD, ATDD)
- **Fast feedback** — автоматизация обеспечивает быструю обратную связь

### Сравнение Waterfall vs Agile для QA

| Аспект | Waterfall | Agile |
|---|---|---|
| Когда тестирование | После разработки | Параллельно с разработкой |
| Тестовая документация | Подробный Test Plan | Lightweight test charters |
| Требования | Фиксированные | Эволюционирующие |
| Обратная связь | В конце цикла | Каждый спринт |
| Автоматизация | Опционально | Критически необходима |
| Роль QA | Gatekeeper | Team player |

## Shift-Left Testing

### Концепция

Shift-left — стратегия, при которой тестирование сдвигается «влево» по временной шкале, то есть начинается
как можно раньше в SDLC.

```
Традиционный подход:
Requirements → Design → Code → ███ Testing ███ → Deploy

Shift-left:
███ Testing ███ → ███ Testing ███ → ███ Testing ███ → Deploy
Requirements      Design & Code      Integration
```

### Практики Shift-Left

1. **Static Testing** — review требований, дизайна, кода без запуска программы
2. **TDD (Test-Driven Development)** — сначала пишется тест, потом код
3. **BDD (Behavior-Driven Development)** — тесты описываются на языке бизнеса (Gherkin)
4. **Pair Testing** — QA и разработчик тестируют вместе на этапе разработки
5. **Shift-left security** — SAST/DAST-анализ на ранних этапах
6. **Contract Testing** — проверка API-контрактов до интеграции

### Экономический эффект

Стоимость исправления дефекта растёт экспоненциально в зависимости от этапа обнаружения:

| Этап обнаружения | Относительная стоимость |
|---|---|
| Requirements | 1x |
| Design | 5x |
| Coding | 10x |
| Testing | 20x |
| Production | 100x |

Shift-left помогает находить дефекты на ранних этапах, когда их исправление дёшево.

## Связь с тестированием

- Понимание SDLC помогает QA планировать свою работу: **когда** и **что** тестировать
- STLC даёт чёткую структуру: от анализа требований до итогового отчёта
- V-Model объясняет, **почему** нужны разные уровни тестирования
- Agile SDLC определяет ритм работы QA в современных командах
- Shift-left обосновывает раннее вовлечение QA и инвестиции в автоматизацию

---

## Типичные ошибки

1. **«QA подключается только после кода»** — классическая ошибка; QA должен участвовать с этапа требований
2. **Путаница между SDLC и STLC** — STLC это часть SDLC, а не его замена
3. **Игнорирование test planning** — «просто начнём тестировать» без плана приводит к пропуску критических сценариев
4. **Shift-left = только автоматизация** — shift-left включает review требований, pair testing, static analysis
5. **«В Agile не нужна документация»** — документация нужна, но в облегчённом формате
6. **Отсутствие Test Closure** — забывают подвести итоги и зафиксировать lessons learned
7. **Копирование процесса без адаптации** — слепое следование «книжному» SDLC без учёта специфики проекта

## Практические задания

### Задание 1: Маппинг SDLC → QA-активности
Возьмите любой знакомый вам проект. Для каждой фазы SDLC опишите:
- Какие QA-активности вы выполняете
- Какие артефакты создаёте
- С кем взаимодействуете

### Задание 2: Создание RTM
Для feature «Авторизация пользователя» создайте Requirements Traceability Matrix:
- Минимум 5 требований
- Для каждого требования — минимум 3 test case

### Задание 3: Test Plan
Напишите lightweight Test Plan для спринта, включающего 3 user stories. Укажите:
- Scope тестирования
- Виды тестирования
- Критерии входа и выхода
- Риски

### Задание 4: Shift-Left анализ
Возьмите 5 последних багов, найденных в production. Определите, на каком этапе SDLC каждый из них мог быть обнаружен
раньше и какая практика shift-left помогла бы.
