# Релизный процесс

## Обзор

Релизный процесс — это набор шагов, необходимых для безопасного выпуска новой версии ПО в production. Для QA-инженера
релиз — один из самых ответственных моментов: именно от качества тестирования зависит, будет ли релиз успешным или
приведёт к инцидентам и откату.

QA играет ключевую роль на каждом этапе: от предрелизного регрессионного тестирования до принятия решения go/no-go
и верификации после деплоя. В этом разделе рассмотрим полный цикл релизного процесса с позиции QA.

---

## Этапы релизного процесса

### Общая схема

```
Code Freeze → Regression Testing → Staging Deploy → Smoke on Staging →
→ Go/No-Go Meeting → Production Deploy → Smoke on Production →
→ Monitoring → Release Notes → Post-Release Review
```

---

### 1. Code Freeze (Замораживание кода)

**Что это:** момент, после которого в release branch не добавляется новый функционал. Разрешены только bug fixes.

**Роль QA:**
- Убедиться, что все stories, запланированные в релиз, имеют статус «QA Passed»
- Подтвердить, что code freeze объявлен и все участники команды об этом знают
- Зафиксировать список изменений, вошедших в релиз (changelog)

**QA-чеклист для Code Freeze:**

```
☐ Все planned stories протестированы и имеют статус "Done"
☐ Все Critical/Blocker баги, найденные в спринте, исправлены и верифицированы
☐ Release branch создан и стабилен
☐ Автотесты проходят на release branch (green build)
☐ Список known issues задокументирован
☐ Зафиксировано, какие changes входят в релиз (tag или changelog)
```

---

### 2. Регрессионное тестирование

**Что это:** полная проверка того, что существующая функциональность не сломана новыми изменениями.

**Стратегия регрессионного тестирования:**

```
Уровень 1 — Автоматические тесты (CI/CD):
├── Unit-тесты (Dev)
├── API integration tests
├── UI smoke tests
└── Performance baseline tests

Уровень 2 — Ручная проверка критических путей:
├── Регистрация и авторизация
├── Основной бизнес-процесс (checkout, оплата)
├── Работа с данными (CRUD-операции)
└── Интеграции с внешними системами

Уровень 3 — Risk-based testing:
├── Области, затронутые изменениями в этом релизе
├── Области с историей частых дефектов
└── Области, критичные для бизнеса
```

**Пример Regression Test Suite:**

```
Regression Suite v3.2 — E-Commerce Platform

Модуль "Авторизация" (15 тестов):
  ✅ Автоматизировано: 13
  🖐 Ручные: 2 (OAuth через Google, OAuth через Apple)

Модуль "Каталог" (22 теста):
  ✅ Автоматизировано: 20
  🖐 Ручные: 2 (визуальная проверка карточек товара)

Модуль "Корзина и Checkout" (30 тестов):
  ✅ Автоматизировано: 25
  🖐 Ручные: 5 (платёжные сценарии с реальным sandbox)

Модуль "Уведомления" (10 тестов):
  ✅ Автоматизировано: 7
  🖐 Ручные: 3 (push-уведомления на реальные устройства)

Итого: 77 тестов (65 авто + 12 ручных)
Ожидаемое время: авто — 45 мин, ручные — 4 часа
```

---

### 3. Деплой на Staging

**Что это:** развёртывание release candidate на окружении, максимально приближенном к production.

**Чеклист QA для staging:**

```
☐ Билд задеплоен успешно (проверить deployment logs)
☐ Все сервисы запущены (health check endpoints)
☐ База данных мигрирована (если были миграции)
☐ Конфигурация соответствует ожидаемой (feature flags, env variables)
☐ Интеграции с внешними системами работают (mock или sandbox)
☐ Тестовые данные подготовлены
```

---

### 4. Smoke-тестирование на Staging

**Что это:** быстрая проверка основных функций на staging перед полным регрессионным тестированием.

**Типичный Smoke Test Suite (15-30 минут):**

```
Smoke Tests — Release 3.2:

1. ✅ Главная страница загружается (< 3 сек)
2. ✅ Регистрация нового пользователя
3. ✅ Логин существующего пользователя
4. ✅ Поиск товара
5. ✅ Просмотр карточки товара
6. ✅ Добавление товара в корзину
7. ✅ Оформление заказа (до этапа оплаты)
8. ✅ Получение email-уведомления
9. ✅ API health endpoints отвечают 200
10. ✅ Основные API-операции (GET, POST, PUT, DELETE)
```

**Правило:** если smoke-тесты не проходят — дальнейшее тестирование бессмысленно. Сообщить DevOps/Dev, ждать fix.

---

### 5. Go/No-Go Meeting

**Что это:** встреча, на которой команда принимает решение: выпускаем релиз или нет.

**Участники:**
- Product Owner / Product Manager
- QA Lead / QA Engineer
- Tech Lead / Senior Developer
- DevOps Engineer
- Scrum Master (фасилитатор)

**QA предоставляет на Go/No-Go:**

```
Release 3.2 — QA Status Report

Регрессионное тестирование:
- Автотесты: 487/487 passed (100%) ✅
- Ручные тесты: 12/12 passed (100%) ✅

Дефекты:
- Blocker: 0 ✅
- Critical: 0 ✅
- Major: 1 (BUG-567 — workaround есть, fix в следующем релизе)
- Minor: 3 (косметические, не влияют на функциональность)

Тестовое покрытие:
- Requirements coverage: 95%
- Все новые features из release scope протестированы

Known Issues:
- BUG-567: При экспорте отчёта >10K записей — задержка 30 сек
  (workaround: экспортировать частями)

Рекомендация QA: GO ✅
(При условии, что BUG-567 будет исправлен в release 3.3)
```

**Критерии Go/No-Go от QA:**

| Критерий | Go | No-Go |
|---|---|---|
| Blocker/Critical баги | 0 | ≥ 1 |
| Регрессионные тесты | ≥ 98% passed | < 95% passed |
| Smoke на staging | Все passed | Есть failures |
| New features tested | 100% | < 100% |
| Known issues | Задокументированы, есть workaround | Есть без workaround |

---

### 6. Production Deploy

**Что это:** развёртывание на production.

**QA во время деплоя:**
- Быть на связи (или присутствовать на «деплой-встрече»)
- Быть готовым к немедленному smoke-тестированию
- Иметь под рукой чеклист для smoke на production
- Знать процедуру rollback

---

### 7. Smoke-тестирование на Production

**Что это:** быстрая проверка основных функций на production сразу после деплоя.

**Особенности smoke на production:**
- Используются специальные тестовые аккаунты (не клиентские данные!)
- Тестируются только read-операции или операции с тестовыми данными
- Время ограничено (обычно 15-30 минут)
- Результат сообщается немедленно

```
Post-Deploy Smoke — Production:

1. ✅ Health check endpoints: 200 OK
2. ✅ Главная страница: загружается за 1.2 сек
3. ✅ Логин тестового пользователя: успешно
4. ✅ API GET /api/products: 200, ответ корректный
5. ✅ API POST /api/orders (тестовый): 201, заказ создан
6. ✅ Email-уведомление получено (тестовый email)
7. ✅ Версия API соответствует ожидаемой: v3.2.0

Результат: PASS ✅
Время: 12 минут
Дата: 2025-03-15 14:32 UTC
```

---

### 8. Мониторинг после релиза

**Что это:** наблюдение за production в первые часы/дни после релиза.

**Что QA мониторит:**

```
Метрики (Grafana/Datadog):
- Error rate (% ошибок) — не должен расти
- Response time (p50, p95, p99) — не должен деградировать
- Throughput (requests/sec) — не должен падать
- CPU/Memory usage — не должен аномально расти

Логи (ELK/Kibana):
- Новые типы ошибок (exceptions, которых не было до релиза)
- Увеличение частоты существующих ошибок
- 5xx ошибки

Бизнес-метрики:
- Количество успешных регистраций
- Количество успешных заказов
- Conversion rate
```

**Правило мониторинга:** если метрики ухудшились после деплоя — немедленно сигнализировать команде.

## Hotfix-процесс

### Когда нужен Hotfix

Hotfix — это срочное исправление critical/blocker бага, обнаруженного на production.

**Триггеры для Hotfix:**
- Пользователи не могут выполнить основное действие (оплата, регистрация)
- Потеря данных
- Уязвимость безопасности
- Значительная деградация производительности

### Процесс Hotfix

```
1. Incident detected → Алерт или жалоба пользователя
2. Triage → Определить severity и impact
3. Hotfix branch → Создать от production (main/master)
4. Fix → Разработчик пишет минимальный fix
5. QA verification → Тестирование fix на staging
6. Regression (сокращённая) → Проверить, что fix не сломал другое
7. Deploy → Деплой на production
8. Verify → Smoke на production
9. Post-mortem → Анализ причин и предотвращение
```

**QA в hotfix-процессе:**

```
Получение информации:
- Что сломалось? (симптомы)
- Кого затрагивает? (scope)
- Когда началось? (после деплоя или спонтанно?)

Тестирование fix:
- Воспроизвести баг на staging (до fix)
- Применить fix
- Убедиться, что баг исправлен
- Провести mini-regression (затронутые области)
- Дать GO на деплой

Время: обычно 1-4 часа от обнаружения до деплоя fix
```

## Rollback (Откат)

### Когда нужен Rollback

Rollback — возврат к предыдущей версии, если новая версия создаёт серьёзные проблемы.

**Критерии для rollback:**
- Массовые ошибки, затрагивающие большинство пользователей
- Потеря данных или их повреждение
- Невозможность быстро найти и исправить проблему (hotfix слишком сложен)
- Деградация производительности до неприемлемого уровня

**QA после rollback:**
```
1. Smoke-тестирование откаченной версии
2. Подтверждение, что проблема устранена
3. Документирование инцидента
4. Участие в post-mortem
5. Создание тестов, которые бы поймали эту проблему до релиза
```

## Release Notes с позиции QA

### QA вклад в Release Notes

QA помогает составить полные и точные release notes:

```
Release 3.2.0 — 2025-03-15

Новые возможности:
- Экспорт отчётов в PDF [QA: протестировано ✅]
- Фильтрация каталога по категориям [QA: протестировано ✅]
- Push-уведомления о статусе заказа [QA: протестировано ✅]

Исправленные дефекты:
- BUG-456: Исправлена ошибка при оплате картой Mir
- BUG-478: Исправлено отображение длинных названий товаров
- BUG-491: Исправлена сортировка по дате

Известные ограничения:
- BUG-567: Экспорт отчёта >10K записей может занять до 30 сек
  Рекомендация: экспортировать частями
```

## Связь с тестированием

- Регрессионное тестирование — основной QA-вклад в релизный процесс
- Smoke-тестирование обеспечивает быструю обратную связь после деплоя
- Go/No-Go meeting — формальное подтверждение качества от QA
- Hotfix-процесс требует от QA скорости без потери качества
- Мониторинг после релиза — расширение ответственности QA за пределы традиционного тестирования

---

## Типичные ошибки

1. **Нет code freeze** — новый код добавляется до самого деплоя, regression невозможен
2. **Smoke тестирование пропускается** — «всё же работало на staging»
3. **QA не участвует в Go/No-Go** — решение принимается без оценки качества
4. **Нет rollback plan** — при проблемах паника вместо чёткого плана
5. **Hotfix без тестирования** — «это маленький fix, тестить нечего» → сломали другое
6. **Нет мониторинга после деплоя** — проблемы обнаруживают пользователи, а не команда
7. **Smoke на production = полная регрессия** — 3 часа smoke на production — это перебор
8. **Known issues не задокументированы** — поддержка не знает об ограничениях новой версии
9. **Нет post-mortem** — одни и те же проблемы повторяются из релиза в релиз

## Практические задания

### Задание 1: Release Checklist
Создайте release checklist для вашего текущего проекта. Включите все этапы с конкретными действиями, ответственными и критериями прохождения.

### Задание 2: Go/No-Go Report
Подготовьте пример QA-отчёта для Go/No-Go meeting. Включите:
- Статус регрессионного тестирования
- Список открытых дефектов с severity
- Known issues с workarounds
- Рекомендацию (Go или No-Go) с обоснованием

### Задание 3: Hotfix Playbook
Создайте пошаговый playbook для hotfix-процесса:
- Кто и как принимает решение о hotfix
- Роль QA на каждом этапе
- Критерии для «выпускаем hotfix» vs «откатываем релиз»
- Шаблон post-mortem

### Задание 4: Мониторинг после релиза
Определите 10 метрик, которые QA должен мониторить в первые 2 часа после релиза. Для каждой метрики укажите: нормальное значение, порог для алерта, действие при отклонении.
