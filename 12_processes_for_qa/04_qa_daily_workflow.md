# Рабочий день QA

## Обзор

Рабочий день QA-инженера — это сочетание плановых активностей и реактивных задач. В отличие от разработчика, который
может несколько дней работать над одной feature, QA часто переключается между задачами: тестирование новой
функциональности, анализ упавших автотестов, воспроизведение бага от поддержки, обновление тестовой документации.

Умение структурировать свой день, расставлять приоритеты и эффективно управлять временем — критически важный навык для
QA. В этом разделе разберём типичный рабочий день с конкретными примерами и рекомендациями.

---

## Структура рабочего дня

### Утро (9:00 - 12:00)

#### 9:00 - 9:30 — Начало дня: проверка состояния систем

Первое, что делает QA утром, — оценивает текущее состояние проекта.

**Чеклист утренней проверки:**

```
☐ Проверить CI/CD pipeline:
  - Ночной регрессионный прогон — есть ли новые failures?
  - Статус последнего билда в develop/main — green или red?

☐ Проверить тестовые окружения:
  - Staging доступен и работает?
  - Тестовая БД не переполнена?
  - Mock-сервисы работают?

☐ Проверить Jira/таск-трекер:
  - Появились ли новые задачи или баги за ночь?
  - Изменились ли приоритеты?
  - Есть ли задачи со статусом "Ready for QA"?

☐ Проверить уведомления:
  - Сообщения в рабочем чате (Slack, Teams)
  - Email-уведомления о багах, PR, комментариях
  - Алерты мониторинга (если QA следит за production)
```

**Пример утреннего анализа CI:**

```
Allure Report — Ночной прогон (02:00):
Всего тестов: 487
✅ Passed: 461
❌ Failed: 18
⚠️ Broken: 8

Новые failures (вчера не было):
- POST /api/orders — 500 Internal Server Error (3 теста)
  → Вероятно, баг в новом коммите. Проверить git log

- GET /api/users/{id} — timeout (2 теста)
  → Вероятно, проблема с тестовым окружением

Известные flaky (были вчера):
- Login UI test — StaleElementException (повторяется 3-й день)
  → Нужно исправить локатор. Создать задачу
```

#### 9:30 - 9:45 — Daily Standup

**Подготовка к Daily (за 5 минут до встречи):**
- Освежить в памяти, что делал вчера
- Сформулировать план на сегодня
- Определить блокеры

**Структура ответа QA:**

```
Вчера:
- Протестировал story QA-567 "Фильтрация каталога" — found 2 bugs (Major)
- Написал 7 API-автотестов для endpoint /api/products
- Верифицировал fix BUG-890 — confirmed fixed

Сегодня:
- Анализирую 18 новых failures в ночном прогоне
- Начну тестирование story QA-568 "Корзина"
- Исправлю flaky test LoginUITest

Блокеры:
- Staging не стабилен — timeout при запросах к Payment Service
```

#### 9:45 - 12:00 — Основная работа (утренний блок)

Утренний блок — наиболее продуктивное время. Его лучше использовать для задач, требующих концентрации.

**Приоритеты утреннего блока:**

1. **Анализ ночного регрессионного прогона** — нельзя откладывать, т.к. новые failures могут сигнализировать о серьёзном баге
2. **Ретестирование critical/blocker багов** — разработчики ждут подтверждения fix
3. **Тестирование stories с близким дедлайном** — приближается конец спринта

**Пример утренней работы — анализ failures:**

```
Шаг 1: Открыть Allure Report
Шаг 2: Отсортировать failures по категориям:
  - Product defects (реальные баги): 3 теста → создать bug report
  - Test defects (проблема в тесте): 2 теста → исправить тест
  - Environment issues: 5 тестов → сообщить DevOps
  - Known issues (flaky): 8 тестов → обновить статус

Шаг 3: Для product defects:
  - Воспроизвести вручную
  - Написать bug report с шагами, скриншотами, логами
  - Уведомить Dev-команду

Шаг 4: Для test defects:
  - Создать задачу на исправление автотеста
  - Если быстрый fix — исправить сразу
```

Перерыв важен для продуктивности. После обеда QA часто переключается на другой тип задач.

### День (13:00 - 17:00)

#### 13:00 - 15:00 — Тестирование новой функциональности

Это основная активность QA. Может быть ручным или автоматизированным тестированием.

**Workflow тестирования user story:**

```
1. Прочитать story и acceptance criteria
2. Проверить, что feature задеплоена на staging
3. Выполнить smoke-проверку (happy path работает)
4. Пройтись по подготовленным test cases:
   - Позитивные сценарии
   - Негативные сценарии
   - Граничные значения
   - Edge cases
5. Провести exploratory testing (исследовательское)
6. Зафиксировать результаты:
   - Пройденные тесты → отметить в TMS
   - Найденные баги → создать в Jira
   - Вопросы к BA/PO → написать в чате или комментарии
```

#### 15:00 - 16:30 — Автоматизация тестирования

После ручного тестирования QA пишет или обновляет автотесты.

**Типичные задачи автоматизации:**

```
- Написать новые API-тесты для только что протестированного endpoint
- Обновить существующие тесты после изменения бизнес-логики
- Исправить flaky-тесты
- Провести code review автотестов коллеги
- Обновить тестовые данные (fixtures, factories)
- Улучшить test framework (новые утилиты, helper-методы)
```

**Пример задачи автоматизации:**

```java
// Задача: написать автотест для нового endpoint POST /api/orders

@Test
@Story("QA-568 Корзина")
@Severity(SeverityLevel.CRITICAL)
@Description("Проверка создания заказа с валидными данными")
public void testCreateOrderSuccess() {
    // Подготовка тестовых данных
    OrderRequest order = OrderRequest.builder()
        .productId(1001)
        .quantity(2)
        .deliveryAddress("ул. Тестовая, д. 1")
        .build();

    // Отправка запроса
    OrderResponse response = given()
        .spec(authorizedSpec)
        .body(order)
        .post("/api/orders")
        .then()
        .statusCode(201)
        .extract().as(OrderResponse.class);

    // Проверки
    assertThat(response.getStatus()).isEqualTo("CREATED");
    assertThat(response.getTotalPrice()).isEqualTo(2 * PRODUCT_1001_PRICE);
}
```

#### 16:30 - 17:00 — Завершение дня

**Чеклист завершения дня:**

```
☐ Обновить статусы задач в Jira:
  - Протестированные stories → "Done" или "QA Passed"
  - Найденные баги → созданы с правильным приоритетом
  - Незавершённые задачи → обновить комментарий с текущим статусом

☐ Закоммитить и запушить код автотестов:
  - Все локальные изменения закоммичены
  - PR создан (если готов к review)
  - CI pipeline запущен

☐ Подготовить план на завтра:
  - Какие stories нужно протестировать
  - Какие баги нужно ретестировать
  - Какие автотесты написать/исправить

☐ Сообщить команде о важном:
  - Критические баги — убедиться, что Dev знает
  - Блокеры — убедиться, что SM знает
  - Изменения в покрытии — обновить dashboard
```

## Распределение времени QA

### Типичное соотношение активностей

```
Тестирование (ручное + автоматизация):     50-60%
Анализ и планирование:                      15-20%
Коммуникации (митинги, обсуждения, review):  15-20%
Поддержка инфраструктуры (CI, окружения):    5-10%
Обучение и развитие:                         5%
```

### Распределение по дням спринта (2 недели)

```
Неделя 1:
Пн: Planning + анализ stories + написание test cases
Вт: Тестирование первых ready stories + автоматизация
Ср: Тестирование + bug reporting + ретестирование
Чт: Автоматизация + стабилизация flaky tests
Пт: Тестирование + review автотестов

Неделя 2:
Пн: Тестирование оставшихся stories + ретестирование
Вт: Автоматизация + тестирование
Ср: Регрессионное тестирование (ручное + авто)
Чт: Финальное тестирование + подготовка к demo
Пт: Sprint Review + Retrospective + Test Closure
```

## Управление приоритетами

### Матрица приоритизации для QA

| | Срочно | Не срочно |
|---|---|---|
| **Важно** | Critical баг на production; Blocker для команды | Написание автотестов; Улучшение test framework |
| **Не важно** | Minor баг, о котором спросил PO | Рефакторинг старых test cases; Обновление документации |

### Правила приоритизации

1. **Production issues** — всегда первый приоритет
2. **Blocker/Critical баги** — второй приоритет, блокируют разработку
3. **Testing stories in sprint** — основная работа спринта
4. **Automation** — инвестиция в будущее, но не за счёт текущего тестирования
5. **Process improvements** — важно, но в выделенное время (например, пятница)

## Связь с тестированием

- Структурированный рабочий день гарантирует, что ни одна QA-активность не будет пропущена
- Утренняя проверка CI помогает обнаружить regression в первые часы
- Правильная приоритизация обеспечивает фокус на критически важном тестировании
- Баланс ручного и автоматизированного тестирования поддерживает и текущее, и долгосрочное качество
- Чеклист завершения дня предотвращает потерю информации

---

## Типичные ошибки

1. **Нет утренней проверки CI** — failures обнаруживаются только к обеду, теряется полдня
2. **Реактивный режим** — QA только реагирует на запросы, не планирует свой день
3. **100% времени на ручное тестирование** — нет инвестиций в автоматизацию
4. **100% времени на автоматизацию** — текущие story не тестируются вовремя
5. **Отсутствие приоритизации** — QA занимается minor-багами, пока critical лежит в backlog
6. **Multitasking** — попытка тестировать 5 stories одновременно вместо последовательного фокуса
7. **Не обновляет статусы в Jira** — команда не знает, что протестировано, а что нет
8. **Не делает перерывы** — выгорание и снижение внимательности

## Советы по продуктивности

### Time Boxing

Используйте технику time boxing — выделяйте фиксированное время на задачу:

```
9:00 - 9:30    Утренняя проверка (30 мин, не больше)
9:30 - 9:45    Daily
10:00 - 11:30  Тестирование story (90 мин — Pomodoro x3)
11:30 - 12:00  Перерыв + ответы на сообщения
13:00 - 14:30  Автоматизация (90 мин)
14:30 - 15:00  Code review автотестов
15:00 - 16:30  Тестирование / ретестирование
16:30 - 17:00  Завершение дня
```

### Batch Processing

Группируйте однотипные задачи:
- Все bug-репорты пишите одним блоком, а не по одному между тестами
- Все code review делайте за один подход
- Все сообщения / вопросы отправляйте разом, а не по мере возникновения

### «Два списка»

В начале дня составьте два списка:
1. **Must do today** — 2-3 задачи, без которых день неуспешен
2. **Nice to have** — задачи, которые хорошо бы сделать, но могут подождать

## Практические задания

### Задание 1: Дневник QA
В течение одной рабочей недели записывайте, на что тратите время с точностью до 30 минут.
В конце недели проанализируйте: соответствует ли распределение вашим приоритетам?

### Задание 2: Утренний чеклист
Создайте персональный утренний чеклист для вашего проекта. Включите:
- Конкретные URL для проверки (CI, окружения, мониторинг)
- Конкретные JQL-запросы для Jira
- Конкретные действия при обнаружении проблем

### Задание 3: Приоритизация
Вам приходят одновременно:
- Critical баг от поддержки (клиент не может оплатить заказ)
- Разработчик просит ретестировать его fix
- PO спрашивает статус тестирования для Sprint Review (через 2 часа)
- CI pipeline красный с 15 failures
Опишите, в каком порядке и почему вы будете решать эти задачи.

### Задание 4: Оптимизация
Проанализируйте свой текущий рабочий день. Найдите:
- 3 активности, которые можно автоматизировать
- 3 активности, которые можно делегировать
- 3 активности, которые можно исключить
