# Роли в команде

## Обзор

QA-инженер не работает в вакууме. Эффективное тестирование невозможно без тесного взаимодействия с другими членами
команды: бизнес-аналитиками, разработчиками, архитекторами, DevOps-инженерами и менеджерами. Понимание зон
ответственности каждой роли, знание того, **когда и к кому обращаться**, а также умение выстраивать коммуникацию —
ключевые soft skills для QA.

В этом разделе разберём основные роли в команде разработки, их обязанности и паттерны взаимодействия с QA.

---

## Основные роли в команде

### Product Manager / Product Owner (PM / PO)

**Зона ответственности:**
- Определение бизнес-целей и приоритетов
- Управление Product Backlog
- Формулирование user stories и acceptance criteria
- Принятие решений о scope функциональности
- Коммуникация со стейкхолдерами

**Взаимодействие с QA:**

| Ситуация | Инициатор | Формат |
|---|---|---|
| Уточнение acceptance criteria | QA → PO | Grooming, сообщение в чате |
| Вопросы по бизнес-логике | QA → PO | Сообщение, встреча |
| Приоритизация багов | QA ↔ PO | Bug triage meeting |
| Демо готовой функциональности | QA → PO | Sprint Review |
| Изменение scope / новые требования | PO → QA | Planning, backlog update |

**Что QA получает от PO:**
- Чёткие acceptance criteria
- Приоритеты для тестирования
- Решения по спорным вопросам бизнес-логики
- Подтверждение, что функциональность соответствует ожиданиям (User Acceptance)

**Что PO получает от QA:**
- Информацию о качестве продукта
- Выявленные пробелы в требованиях
- Рекомендации go/no-go для релиза
- Метрики качества (дефекты, покрытие, стабильность)

**Пример коммуникации:**
```
QA → PO:
"В story QA-234 'Экспорт отчёта в PDF' не указано:
1. Максимальное количество записей в отчёте
2. Поведение при экспорте пустого отчёта
3. Формат дат и чисел (локализация)
Можешь уточнить эти моменты до начала тестирования?"
```

### Business Analyst (BA)

**Зона ответственности:**
- Детальный анализ и документирование требований
- Создание спецификаций, wireframes, user flows
- Проведение requirements elicitation sessions
- Маппинг бизнес-процессов
- Управление изменениями требований

**Взаимодействие с QA:**

QA и BA — естественные союзники. Оба фокусируются на детальном понимании того, **как система должна работать**.

**Что QA получает от BA:**
- Детальные спецификации для написания test cases
- User flow diagrams для определения тестовых сценариев
- Разъяснения по сложной бизнес-логике
- Информацию об интеграциях с внешними системами

**Совместная работа QA и BA:**
- **Requirements Review** — QA ревьюирует спецификации BA до передачи в разработку
- **Three Amigos** — встреча BA + Dev + QA для обсуждения user story с трёх перспектив
- **Acceptance Criteria co-creation** — совместное написание AC
- **Test case review** — BA проверяет, что test cases покрывают все требования

**Пример Three Amigos:**
```
User Story: "Система должна отправлять уведомление при истечении срока подписки"

BA: "Уведомление отправляется за 7 дней до истечения"
Dev: "Нужен scheduled job, который будет запускаться ежедневно"
QA: "А что если пользователь продлил подписку после получения уведомления?
      Нужно ли отменить последующие напоминания?
      Что если email пользователя невалидный — нужен ли retry?"
```

### Solution Architect (SA)

**Зона ответственности:**
- Проектирование архитектуры системы
- Выбор технологического стека
- Определение API-контрактов
- Проектирование схемы базы данных
- Принятие решений по интеграциям
- Обеспечение non-functional requirements (performance, scalability, security)

**Взаимодействие с QA:**

| Ситуация | Что QA получает |
|---|---|
| Design Review | Понимание архитектуры для планирования интеграционных тестов |
| API Contract Design | Спецификации для API-тестирования |
| Database Schema | Структуру для тестирования данных и миграций |
| Performance Requirements | NFR для нагрузочного тестирования |
| Security Design | Требования для security-тестирования |

**Вопросы QA к архитектору:**
- Какие точки интеграции существуют между сервисами?
- Какие API-контракты определены? Есть ли OpenAPI/Swagger-спецификации?
- Какие ограничения по производительности (latency, throughput)?
- Какая стратегия обработки ошибок (retry, circuit breaker, dead letter queue)?
- Как организовано логирование и мониторинг?

### Software Developer (Dev)

**Зона ответственности:**
- Написание кода
- Unit-тестирование
- Code review
- Исправление дефектов
- Техническая документация

**Взаимодействие с QA:**

Это самое частое и важное взаимодействие для QA. Разработчик — основной «поставщик» того, что QA тестирует.

**Паттерны взаимодействия:**

1. **Bug Communication (обсуждение багов)**
```
Хорошо:
QA: "При POST /api/users с пустым полем email сервер возвращает 500 вместо 400.
     Шаги: [ссылка на bug report]. Воспроизводится стабильно на staging."

Плохо:
QA: "Регистрация не работает"
```

2. **Уточнение реализации**
```
QA → Dev: "В какой момент отправляется email-уведомление —
синхронно в рамках запроса или через очередь сообщений?
Это влияет на то, как я буду тестировать timing."
```

3. **Pair Testing**
```
QA и Dev вместе исследуют новую функциональность.
Dev объясняет, как работает код, QA предлагает edge cases.
Результат: баги находятся быстрее, Dev лучше понимает перспективу QA.
```

4. **Code Review с QA-перспективой**
```
QA → Dev (в code review):
"Вижу, что обработка ошибок в PaymentService не покрывает
случай timeout от платёжного шлюза. Добавлен ли unit-тест для этого?"
```

**Здоровые и нездоровые отношения QA-Dev:**

| Здоровые | Нездоровые |
|---|---|
| «Мы вместе отвечаем за качество» | «QA ищет баги, Dev защищает код» |
| Обсуждение бага = решение проблемы | Обсуждение бага = обвинение |
| QA уважает работу Dev, Dev уважает находки QA | «Это не баг, это фича» как шаблон ответа |
| Совместный анализ сложных дефектов | «Закрой баг, это works as designed» без объяснения |

### DevOps / SRE

**Зона ответственности:**
- Настройка и поддержка CI/CD-пайплайнов
- Управление инфраструктурой (серверы, контейнеры, облака)
- Мониторинг и алертинг
- Управление окружениями (dev, staging, production)
- Автоматизация деплоя

**Взаимодействие с QA:**

| Ситуация | QA → DevOps |
|---|---|
| Настройка тестового окружения | «Нужен staging с последним билдом и seed-данными» |
| CI/CD для тестов | «Нужно добавить step для запуска автотестов в pipeline» |
| Проблемы с окружением | «Staging упал, не могу тестировать — блокер» |
| Доступ к логам | «Нужен доступ к Kibana для анализа ошибок» |
| Контейнеризация тестов | «Помоги настроить Docker для Selenium Grid» |

**Что QA должен уметь обсуждать с DevOps:**
- Docker, Docker Compose — для локального запуска сервисов
- CI/CD pipeline (Jenkins, GitLab CI, GitHub Actions) — для интеграции автотестов
- Мониторинг (Grafana, Prometheus, ELK) — для анализа production-инцидентов
- Infrastructure as Code (Terraform, Ansible) — для понимания окружений

---

### Scrum Master / Team Lead

**Зона ответственности:**
- Фасилитация Scrum-церемоний
- Устранение impediments
- Защита команды от внешнего давления
- Содействие улучшению процессов

**Когда QA обращается к Scrum Master:**
- Блокеры, которые QA не может решить самостоятельно (доступ, окружение, зависимости)
- Процессные проблемы (stories приходят на тестирование в последний день)
- Конфликты с другими ролями
- Предложения по улучшению процесса (ретроспектива)

## Коммуникационные паттерны

### Матрица коммуникаций QA

```
                    QA
                   / | \
                  /  |  \
                 /   |   \
               PO   Dev  DevOps
              /      |      \
            BA      SA    Scrum Master
```

### Формат коммуникации по ситуации

| Ситуация | Формат | Приоритет |
|---|---|---|
| Critical bug на production | Звонок / срочное сообщение | Немедленно |
| Вопрос по требованиям | Сообщение в чате / комментарий в Jira | В течение дня |
| Bug report | Jira-тикет | Зависит от severity |
| Статус тестирования | Daily Standup / Jira Board | Ежедневно |
| Предложение по процессу | Ретроспектива | Раз в спринт |
| Обсуждение архитектуры тестов | Встреча / design review | По необходимости |

### Правила эффективной коммуникации

1. **Будь конкретным** — «Баг в авторизации» → «POST /api/auth/login возвращает 500 при пустом пароле»
2. **Давай контекст** — указывай окружение, версию, шаги воспроизведения
3. **Предлагай решение** — не только проблему, но и возможный путь решения
4. **Используй правильный канал** — не надо писать о critical баге в общем чате; не надо звонить с вопросом о приоритете minor-бага
5. **Документируй решения** — договорённости из устных обсуждений фиксируй в Jira/Confluence

## Связь с тестированием

- Качество коммуникации напрямую влияет на качество тестирования
- Чем раньше QA получает информацию от BA/PO, тем лучше test cases
- Тесное сотрудничество с Dev сокращает время исправления багов
- Работа с DevOps обеспечивает стабильную тестовую инфраструктуру
- Three Amigos (BA + Dev + QA) — один из самых эффективных паттернов для предотвращения дефектов

---

## Типичные ошибки

1. **«QA — отдельная команда»** — изоляция QA от Dev и BA приводит к позднему обнаружению проблем
2. **Нет прямого канала QA ↔ PO** — QA узнаёт о требованиях через разработчиков (испорченный телефон)
3. **QA не участвует в design review** — архитектурные решения принимаются без учёта тестируемости
4. **Конфликтная модель QA-Dev** — «ловцы багов» vs «защитники кода» вместо совместной работы
5. **QA не общается с DevOps** — проблемы с окружением решаются днями вместо часов
6. **Избыточная формальность** — для каждого вопроса создаётся митинг вместо сообщения в чате
7. **Недостаточная формальность** — критические решения принимаются устно и нигде не фиксируются

## Вопросы на интервью

### 🟢 Базовый уровень
1. С какими ролями в команде взаимодействует QA? Назовите основные
2. Кто формулирует acceptance criteria?
3. Что такое «Three Amigos»?
4. В каких Scrum-церемониях участвует QA?
5. К кому QA обращается при обнаружении бага?

### 🟡 Средний уровень
6. Как выстроить продуктивные отношения с разработчиком, который не согласен, что найденный дефект — баг?
7. Как QA может помочь BA улучшить качество требований?
8. Какую информацию QA должен запросить у архитектора перед тестированием нового микросервиса?
9. Опишите идеальный процесс коммуникации при обнаружении critical бага на production
10. Как QA взаимодействует с DevOps при настройке автотестов в CI/CD?

### 🔴 Продвинутый уровень
11. Как организовать QA-процесс в распределённой команде (разные часовые пояса)?
12. Как QA может влиять на архитектурные решения в пользу тестируемости?
13. Как разрешить конфликт между PO (хочет быстрый релиз) и QA (считает, что качество недостаточно)?
14. Как масштабировать QA-коммуникации при переходе от одной Scrum-команды к нескольким?
15. Как внедрить культуру quality ownership во всей команде, а не только в QA?

## Практические задания

### Задание 1: RACI-матрица
Создайте RACI-матрицу (Responsible, Accountable, Consulted, Informed) для следующих QA-активностей:
- Написание test cases
- Bug triage
- Решение о релизе (go/no-go)
- Настройка тестового окружения
- Приёмочное тестирование

### Задание 2: Коммуникационный план
Для нового проекта создайте план коммуникации QA с каждой ролью:
- С кем, как часто, каким способом, по каким вопросам

### Задание 3: Three Amigos сессия
Возьмите user story из вашего backlog. Проведите Three Amigos:
- Запишите 5 вопросов, которые задал бы QA
- Запишите 5 вопросов, которые задал бы Dev
- Запишите 5 уточнений, которые сделал бы BA

### Задание 4: Конфликтная ситуация
Разработчик закрыл ваш баг как «Not a Bug». Вы уверены, что это баг. Напишите:
- Сообщение разработчику с аргументацией
- Эскалацию (если нужна) Scrum Master / PO
