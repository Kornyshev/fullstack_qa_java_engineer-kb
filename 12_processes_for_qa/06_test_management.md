# Управление тестированием

## Обзор

Управление тестированием (Test Management) — это процесс организации, планирования, отслеживания и контроля
всех активностей, связанных с тестированием. Для QA-инженера это означает не только выполнение тестов, но и
управление тестовыми артефактами: test cases, test suites, test runs, дефекты, отчёты и метрики.

В современных проектах управление тестированием осуществляется с помощью специализированных TMS (Test Management
Systems), которые интегрируются с таск-трекерами, CI/CD и reporting-инструментами. В этом разделе рассмотрим
основные TMS, практики управления тестами и метрики, необходимые для принятия решений о качестве.

---

## Test Management Systems (TMS)

### Зачем нужна TMS

Без TMS тестирование хаотично:
- Test cases хранятся в Google Docs, теряются и устаревают
- Нет связи между test cases и requirements
- Невозможно отследить, что было протестировано, а что нет
- Нет исторических данных о прогонах
- Невозможно сгенерировать отчёт для менеджмента

### Основные TMS на рынке

#### TestRail

**Описание:** одна из наиболее популярных коммерческих TMS.

**Ключевые возможности:**
- Организация test cases в sections и suites
- Test runs с назначением на конкретных тестировщиков
- Интеграция с Jira, Jenkins, Selenium
- Отчёты и dashboards
- API для автоматизации

**Структура в TestRail:**
```
Project
├── Test Suites
│   ├── Suite: Авторизация
│   │   ├── Section: Логин
│   │   │   ├── TC-001: Логин с валидными данными
│   │   │   ├── TC-002: Логин с невалидным паролем
│   │   │   └── TC-003: Логин с пустым email
│   │   └── Section: Регистрация
│   │       ├── TC-010: Регистрация нового пользователя
│   │       └── TC-011: Регистрация с существующим email
│   └── Suite: Каталог
│       └── ...
├── Test Runs
│   ├── Run: Sprint 14 Regression (2025-03-10)
│   └── Run: Release 3.2 Smoke (2025-03-15)
└── Reports
```

#### Allure TestOps

**Описание:** платформа для управления тестированием с глубокой интеграцией с Allure Report.

**Ключевые возможности:**
- Автоматический импорт результатов автотестов из Allure Report
- Объединение ручных и автоматизированных тестов
- Test layers: Allure TestOps позволяет связать тесты с features, stories, epics
- Live-отчёты во время прогона
- Интеграция с CI/CD (Jenkins, GitLab CI, GitHub Actions)
- Flaky-test detection

**Преимущества для автоматизации:**
```
Allure TestOps синхронизирует тесты из кода:

@Test
@AllureId("12345")    // ← связь с test case в Allure TestOps
@Story("QA-567")
@Description("Проверка создания пользователя через API")
public void testCreateUser() {
    // ...
}

Результат: при запуске автотеста его результат автоматически
появляется в Allure TestOps с привязкой к test case
```

#### Jira + Zephyr (Zephyr Scale / Zephyr Squad)

**Описание:** плагин для Jira, добавляющий функционал управления тестированием прямо в Jira.

**Ключевые возможности:**
- Test cases как специальный тип issue в Jira
- Прямая связь между test cases и user stories
- Test cycles (аналог test runs)
- Traceability reports
- Интеграция с CI/CD через API

**Преимущества:**
- Всё в одном месте (Jira)
- Не нужно переключаться между инструментами
- Нативная связь test case ↔ story ↔ bug

#### Сравнение TMS

| Критерий | TestRail | Allure TestOps | Jira + Zephyr |
|---|---|---|---|
| Тип | Standalone TMS | Standalone TMS | Plugin для Jira |
| Ручные тесты | Отлично | Хорошо | Хорошо |
| Автоматизация | Интеграция через API | Нативная (Allure) | Интеграция через API |
| Отчётность | Хорошо | Отлично | Средне |
| Стоимость | Платно | Платно | Платно (плагин) |
| Простота внедрения | Средняя | Средняя | Высокая (уже есть Jira) |
| Flaky detection | Нет | Да | Нет |

## Управление Test Cases

### Структура Test Case

Хороший test case должен содержать:

```
ID:            TC-001
Название:      Логин с валидным email и паролем
Приоритет:     High
Тип:           Functional
Модуль:        Авторизация
Связанная story: QA-123

Предусловия:
- Пользователь зарегистрирован (email: test@example.com, пароль: Test123!)
- Пользователь не заблокирован

Шаги:
1. Открыть страницу логина (/login)
2. Ввести email: test@example.com
3. Ввести пароль: Test123!
4. Нажать кнопку "Войти"

Ожидаемый результат:
- Пользователь перенаправлен на главную страницу (/dashboard)
- Отображается имя пользователя в header
- В cookies установлен auth_token

Постусловия:
- Выполнить logout для очистки сессии

Тестовые данные:
- Email: test@example.com
- Пароль: Test123!
```

### Организация Test Cases

**По модулям / функциональности:**
```
├── Авторизация
│   ├── Логин
│   ├── Регистрация
│   ├── Восстановление пароля
│   └── OAuth
├── Каталог
│   ├── Поиск
│   ├── Фильтрация
│   └── Сортировка
├── Корзина
│   ├── Добавление товара
│   ├── Изменение количества
│   └── Удаление товара
└── Оплата
    ├── Банковская карта
    ├── Электронный кошелёк
    └── Рассрочка
```

**По приоритетам:**
```
Critical (всегда тестируем):  20% test cases
High (тестируем в каждом релизе): 30% test cases
Medium (тестируем при изменениях): 30% test cases
Low (тестируем периодически): 20% test cases
```

### Жизненный цикл Test Case

```
Draft → Review → Approved → Active → Obsolete
  ↑                                      |
  └──────────── Update ←─────────────────┘
```

- **Draft** — test case написан, но не проверен
- **Review** — на проверке у коллеги или BA
- **Approved** — утверждён, можно использовать
- **Active** — используется в test runs
- **Obsolete** — устарел (функциональность изменилась или удалена)

## Управление Test Execution

### Test Run (Прогон тестов)

Test Run — это конкретное выполнение набора test cases в определённых условиях.

**Пример создания Test Run:**
```
Test Run: Sprint 14 Regression
Дата: 2025-03-10
Окружение: Staging (v3.1.5-rc1)
Тестировщик: Иван Петров
Тип: Full Regression

Test Cases включены:
- Критический набор (45 тестов)
- Модуль "Корзина" (30 тестов) — затронут изменениями
- Модуль "Оплата" (20 тестов) — затронут изменениями
Итого: 95 тестов
```

### Статусы выполнения тестов

| Статус | Описание | Действие |
|---|---|---|
| **Passed** | Тест прошёл успешно | Отметить, двигаться дальше |
| **Failed** | Тест не прошёл (баг) | Создать bug report, привязать к test case |
| **Blocked** | Невозможно выполнить | Указать причину блокировки, сообщить команде |
| **Skipped** | Пропущен осознанно | Указать причину (не в scope, нет окружения) |
| **Retest** | Нужно перетестировать | Баг исправлен, нужна повторная проверка |

## Метрики и отчёты

### Основные QA-метрики

#### 1. Test Execution Metrics

```
Test Pass Rate = (Passed Tests / Total Executed Tests) × 100%

Пример: 450 passed / 487 total = 92.4%

Интерпретация:
- > 98% — отличное состояние
- 95-98% — приемлемо, нужно разобраться с failures
- < 95% — серьёзные проблемы, рассмотреть блокировку релиза
```

#### 2. Defect Metrics

```
Defect Density = Количество дефектов / Размер модуля (LOC, story points)

Defect Leakage = Баги найденные в production / Все найденные баги × 100%

Пример Defect Leakage:
- Найдено в тестировании: 45 багов
- Найдено в production: 3 бага
- Leakage = 3/48 × 100% = 6.25%

Целевой показатель: < 5%
```

#### 3. Automation Metrics

```
Automation Coverage = Автоматизированные TC / Все TC × 100%

Automation ROI:
- Время ручного прогона: 8 часов
- Время автоматического прогона: 45 минут
- Экономия за 1 прогон: 7.25 часов
- Количество прогонов в месяц: 20
- Ежемесячная экономия: 145 часов
```

#### 4. Test Coverage Metrics

```
Requirements Coverage = Требования с тестами / Все требования × 100%

Пример:
- Всего требований: 50
- Покрыто тестами: 47
- Coverage = 94%
- Непокрытые: REQ-23 (low priority), REQ-41 (new), REQ-48 (deferred)
```

## Управление ручными и автоматизированными тестами

### Стратегия: что автоматизировать, а что оставить ручным

**Автоматизировать:**
- Regression test suite (повторяется часто)
- Smoke tests (нужны при каждом деплое)
- API-тесты (стабильны, быстры)
- Data-driven тесты (много комбинаций)
- Performance baseline tests

**Оставить ручным:**
- Exploratory testing (исследовательское)
- Usability testing (UX)
- Тесты с реальными внешними системами (платёжные шлюзы)
- Визуальное тестирование (до внедрения visual regression tools)
- Одноразовые проверки (ad hoc)

### Связь ручных и автоматизированных тестов в TMS

```
Test Case TC-001: "Логин с валидными данными"
├── Тип: Manual + Automated
├── Ручное выполнение: последний раз 2025-03-05 — Passed
├── Автотест: LoginTest.testSuccessfulLogin()
├── Последний авто-прогон: 2025-03-10 — Passed
├── Allure ID: 12345
└── Связанная story: QA-123

Правило: если есть автотест — ручной прогон не нужен
(за исключением регрессии перед релизом)
```

### Трассировка: Requirements → Test Cases → Defects

```
Requirement REQ-001: "Пользователь может войти по email и паролю"
├── Test Cases:
│   ├── TC-001: Логин с валидными данными [Auto] — Passed
│   ├── TC-002: Логин с невалидным паролем [Auto] — Passed
│   ├── TC-003: Логин заблокированного пользователя [Manual] — Passed
│   └── TC-004: Логин с SQL injection [Auto] — Passed
├── Defects:
│   ├── BUG-123: 500 ошибка при пустом пароле — Fixed ✅
│   └── BUG-156: Нет rate limiting на login endpoint — Open
└── Coverage: 4/4 test cases (100%)
```

## Связь с тестированием

- TMS обеспечивает **организованность** — каждый test case имеет место, статус и историю
- Метрики позволяют **принимать решения** на основе данных, а не интуиции
- Трассировка гарантирует, что **ни одно требование не пропущено**
- Интеграция с CI/CD обеспечивает **актуальность данных** без ручного обновления
- Отчёты помогают **коммуницировать качество** стейкхолдерам

---

## Типичные ошибки

1. **Нет TMS** — test cases в Google Docs, результаты в голове тестировщика
2. **TMS есть, но не обновляется** — данные устарели, никто не доверяет отчётам
3. **Слишком детальные test cases** — 20 шагов для простой проверки, невозможно поддерживать
4. **Нет связи test case ↔ requirement** — невозможно определить coverage
5. **Автотесты не интегрированы с TMS** — ручные и автоматизированные тесты живут отдельно
6. **Метрики ради метрик** — собираем 50 метрик, но не используем ни одну для решений
7. **Нет prioritization** — все тесты одинаково важны (что означает — ни один не важен)
8. **Test cases пишет один человек, выполняет другой** — без context transfer результат теряется

### Задание 1: Выбор TMS
Для вашего текущего проекта оцените 3 TMS по критериям:
- Стоимость, интеграции, удобство, подходящая для вашего стека
Обоснуйте свой выбор.

- Минимум 5 модулей
- В каждом модуле минимум 5 test cases
- Определите приоритет каждого test case

### Задание 3: QA Dashboard
Спроектируйте QA dashboard для вашего проекта:
- 5-7 ключевых метрик
- Для каждой метрики: формула, целевое значение, источник данных
