# Пример задачи: ручное тестирование

## Обзор

В этом разделе разберём **полный цикл ручного тестирования** на конкретном примере. Рассмотрим user story
«Регистрация пользователя с верификацией email» от момента получения задачи до подписания (sign-off). Каждый
шаг сопровождается примерами артефактов, которые создаёт QA.

Такой пошаговый пример поможет понять, как выглядит реальная работа QA-инженера при ручном тестировании, и подготовит
к ответу на интервью «Опишите, как вы тестировали конкретную фичу».

---

## User Story

```
Story: QA-234
Title: Регистрация пользователя с верификацией email
Priority: High
Sprint: Sprint 15

Как новый пользователь,
Я хочу зарегистрироваться в системе и подтвердить свой email,
Чтобы получить доступ ко всем функциям приложения.

Acceptance Criteria:
1. Пользователь может зарегистрироваться, указав имя, email и пароль
2. Пароль должен содержать минимум 8 символов, 1 заглавную букву и 1 цифру
3. После регистрации на указанный email отправляется письмо с ссылкой подтверждения
4. Ссылка подтверждения действительна 24 часа
5. После подтверждения email пользователь может войти в систему
6. До подтверждения email доступ ограничен (только просмотр профиля)
7. При регистрации с уже существующим email — показать ошибку
```

---

## Шаг 1: Получение и анализ задачи

### Действия QA

1. **Прочитать story и acceptance criteria** внимательно, дважды
2. **Отметить статус** в Jira: «In QA Analysis»
3. **Выявить неоднозначности и пробелы** в требованиях
4. **Составить список вопросов** к BA/PO

### Список вопросов QA

```
Вопросы к BA/PO по story QA-234:

Регистрация:
Q1: Максимальная длина пароля? (AC говорит "минимум 8", а максимум?)
Q2: Допустимы ли спецсимволы в пароле? Какие?
Q3: Есть ли валидация формата email? (какой стандарт — RFC 5322?)
Q4: Максимальная длина имени пользователя?
Q5: Можно ли зарегистрироваться с кириллическим именем?

Верификация email:
Q6: Что происходит при повторном клике по ссылке подтверждения (после первого)?
Q7: Можно ли запросить повторную отправку письма подтверждения?
Q8: Что происходит, если ссылка истекла (24 часа)? Есть ли способ получить новую?

Ошибки и edge cases:
Q9: Что происходит при concurrent-регистрации с одним email?
Q10: Есть ли CAPTCHA или rate limiting для защиты от бот-регистраций?
Q11: Как система обрабатывает email с "+" (например, user+test@gmail.com)?
Q12: Есть ли регистрация через OAuth (Google, GitHub) в этой story?
```

### Ответы BA/PO

```
A1: Максимум 128 символов
A2: Да, любые спецсимволы допустимы
A3: Стандартная валидация: должен содержать @ и домен с точкой
A4: 1-100 символов
A5: Да, поддерживаем Unicode
A6: Показать сообщение "Email уже подтверждён"
A7: Да, кнопка "Отправить повторно" на странице профиля
A8: Показать "Ссылка истекла" с кнопкой "Отправить новую"
A9: Первая успешная регистрация проходит, вторая получает ошибку
A10: Rate limiting: максимум 5 попыток регистрации с одного IP за 10 минут
A11: Обрабатываем как обычный email (user+test@gmail.com ≠ user@gmail.com)
A12: Нет, OAuth будет в отдельной story
```

## Шаг 2: Написание Test Cases

После получения ответов QA пишет test cases. Группируем по категориям.

### Позитивные сценарии

```
TC-001: Успешная регистрация с валидными данными
Предусловия: Email user_new@example.com не зарегистрирован
Шаги:
1. Открыть страницу регистрации (/register)
2. Ввести имя: "Иван Петров"
3. Ввести email: user_new@example.com
4. Ввести пароль: Test1234
5. Подтвердить пароль: Test1234
6. Нажать "Зарегистрироваться"
Ожидаемый результат:
- Показано сообщение "Проверьте ваш email для подтверждения"
- На email отправлено письмо с ссылкой подтверждения
- В БД создана запись пользователя с is_verified = false

TC-002: Подтверждение email по ссылке
Предусловия: Выполнен TC-001, получено письмо
Шаги:
1. Открыть email (user_new@example.com)
2. Найти письмо "Подтверждение регистрации"
3. Кликнуть по ссылке подтверждения
Ожидаемый результат:
- Открывается страница "Email подтверждён"
- В БД: is_verified = true
- Пользователь может войти в систему

TC-003: Вход после подтверждения email
Предусловия: Выполнен TC-002
Шаги:
1. Открыть страницу логина (/login)
2. Ввести email: user_new@example.com
3. Ввести пароль: Test1234
4. Нажать "Войти"
Ожидаемый результат:
- Пользователь перенаправлен на /dashboard
- Все функции доступны
```

### Негативные сценарии

```
TC-010: Регистрация с существующим email
Предусловия: Email existing@example.com уже зарегистрирован
Шаги:
1. Открыть /register
2. Ввести email: existing@example.com
3. Заполнить остальные поля валидными данными
4. Нажать "Зарегистрироваться"
Ожидаемый результат:
- Ошибка: "Пользователь с таким email уже зарегистрирован"
- Регистрация не происходит

TC-011: Регистрация с невалидным паролем (без цифры)
Шаги:
1. Открыть /register
2. Ввести пароль: Testtest (нет цифры)
3. Заполнить остальные поля валидными данными
4. Нажать "Зарегистрироваться"
Ожидаемый результат:
- Ошибка валидации: "Пароль должен содержать хотя бы 1 цифру"

TC-012: Регистрация с невалидным паролем (менее 8 символов)
Шаги:
1. Ввести пароль: Te1 (3 символа)
Ожидаемый результат:
- Ошибка: "Пароль должен содержать минимум 8 символов"

TC-013: Регистрация с невалидным email
Шаги:
1. Ввести email: "notanemail"
Ожидаемый результат:
- Ошибка: "Введите корректный email"

TC-014: Регистрация с пустыми полями
Шаги:
1. Открыть /register
2. Нажать "Зарегистрироваться" без заполнения полей
Ожидаемый результат:
- Ошибки валидации для всех обязательных полей

TC-015: Несовпадение пароля и подтверждения
Шаги:
1. Ввести пароль: Test1234
2. Ввести подтверждение: Test5678
Ожидаемый результат:
- Ошибка: "Пароли не совпадают"
```

### Граничные значения и Edge Cases

```
TC-020: Пароль ровно 8 символов (граница)
Пароль: "Test1234" (8 символов) → должен пройти

TC-021: Пароль 7 символов (ниже границы)
Пароль: "Test123" (7 символов) → ошибка валидации

TC-022: Пароль 128 символов (верхняя граница)
Пароль: 128-символьный пароль с буквами и цифрами → должен пройти

TC-023: Пароль 129 символов (выше верхней границы)
Пароль: 129 символов → ошибка валидации

TC-024: Имя с Unicode-символами
Имя: "東京太郎" → должен пройти

TC-025: Email с "+" символом
Email: "user+test@gmail.com" → должен пройти

TC-026: Повторный клик по ссылке подтверждения
Предусловия: Email уже подтверждён
→ Сообщение "Email уже подтверждён"

TC-027: Клик по просроченной ссылке (>24 часа)
→ "Ссылка истекла" + кнопка "Отправить новую"

TC-028: Попытка логина до подтверждения email
→ Вход разрешён, но доступ ограничен (только профиль)

TC-029: Rate limiting (6-я регистрация с одного IP за 10 мин)
→ "Слишком много попыток. Попробуйте позже"
```

## Шаг 4: Создание Bug Reports

### BUG-567: Пароль без цифры принимается

```
ID: BUG-567
Severity: Major
Priority: High
Компонент: Registration
Story: QA-234
Окружение: Staging (v3.2.0-rc1)
Тестировщик: Иван Петров

Заголовок: Пароль без цифры принимается при регистрации

Описание:
При регистрации система принимает пароль, не содержащий цифру,
хотя по AC пароль должен содержать минимум 1 цифру.

Шаги воспроизведения:
1. Открыть /register
2. Ввести имя: Test User
3. Ввести email: test_567@example.com
4. Ввести пароль: Testtest (8 символов, 1 заглавная, 0 цифр)
5. Подтвердить пароль: Testtest
6. Нажать "Зарегистрироваться"

Фактический результат:
Регистрация проходит успешно. Пользователь создан в БД.

Ожидаемый результат:
Ошибка валидации: "Пароль должен содержать хотя бы 1 цифру"

Дополнительно:
- Валидация "минимум 8 символов" работает корректно
- Валидация "минимум 1 заглавная" не проверялась (нужен отдельный тест)
- Скриншот: [ссылка]
- Логи: [ссылка на Kibana]
```

### BUG-568: Кнопка регистрации доступна при пустых полях

```
ID: BUG-568
Severity: Minor
Priority: Medium
Компонент: Registration
Story: QA-234

Заголовок: Отсутствует client-side валидация при пустых полях

Описание:
При нажатии "Зарегистрироваться" с пустыми полями, запрос
уходит на сервер. Ожидается client-side валидация.

Шаги:
1. Открыть /register
2. Не заполнять ни одного поля
3. Нажать "Зарегистрироваться"

Фактический результат:
Запрос отправляется на сервер (видно в DevTools → Network).
Сервер возвращает 400, ошибки отображаются.

Ожидаемый результат:
Client-side валидация блокирует отправку запроса.
Ошибки отображаются без обращения к серверу.

Примечание:
Серверная валидация работает корректно. Баг касается UX —
лишний запрос к серверу при очевидно невалидных данных.
```

## Связь с тестированием

- Данный пример демонстрирует полный цикл ручного тестирования от анализа до sign-off
- Качественный анализ требований (Шаг 1) предотвращает пропуск важных сценариев
- Структурированные test cases (Шаг 2) обеспечивают воспроизводимость и полноту
- Хорошие bug reports (Шаг 4) ускоряют исправление дефектов
- Ретестирование (Шаг 5) гарантирует, что fix не создал новых проблем

---

## Типичные ошибки

1. **Пропуск анализа требований** — сразу начинать тестировать без понимания что и зачем
2. **Нет вопросов к BA/PO** — «всё понятно» (на самом деле нет)
3. **Только happy path** — тестируются только позитивные сценарии
4. **Нет граничных значений** — не проверены границы (8 символов, 128 символов)
5. **Bug report без шагов воспроизведения** — «регистрация не работает»
6. **Нет ретестирования** — «разработчик сказал, что исправил, значит всё ок»
7. **Нет sign-off** — story закрывается без формального подтверждения QA
8. **Не проверяется API** — тестируется только UI, а API может иметь свои баги
