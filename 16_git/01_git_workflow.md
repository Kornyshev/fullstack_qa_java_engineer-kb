# Git для QA

## Обзор

Git — распределённая система контроля версий, без которой невозможно представить современную разработку и автоматизацию тестирования. Для QA-инженера Git — это не просто инструмент хранения кода: это способ организации работы над тестовым фреймворком, совместной разработки автотестов в команде, управления конфигурациями, а также интеграции с CI/CD-пайплайнами. Понимание Git на уровне уверенного пользователя — обязательное требование для позиций Middle и Senior QA Automation.

## Базовые команды

### Инициализация и клонирование

```bash
# Создать новый репозиторий в текущей директории
git init

# Клонировать существующий репозиторий (например, тестовый фреймворк команды)
git clone https://github.com/company/test-framework.git

# Клонировать конкретную ветку
git clone -b develop https://github.com/company/test-framework.git
```

### Основной цикл работы: add, commit, push, pull, fetch

```bash
# Посмотреть текущее состояние рабочей директории
git status

# Добавить конкретный файл в staging area (индекс)
git add src/test/java/LoginTest.java

# Добавить все изменённые файлы
git add .

# Зафиксировать изменения с сообщением
git commit -m "Добавить тест авторизации с невалидным паролем"

# Отправить коммиты в удалённый репозиторий
git push origin feature/tests/login

# Получить изменения из удалённого репозитория и объединить
git pull origin develop

# Получить изменения без автоматического объединения (только скачать)
git fetch origin
```

**Важное различие для QA:**
- `git pull` = `git fetch` + `git merge` — сразу пытается влить изменения в текущую ветку
- `git fetch` — безопасно скачивает изменения, не меняя рабочую директорию. Полезно, когда нужно сначала посмотреть, что изменилось, прежде чем вливать

## Ветвление (Branching)

### Основные команды для работы с ветками

```bash
# Создать новую ветку
git branch feature/tests/registration

# Переключиться на ветку (классический способ)
git checkout feature/tests/registration

# Создать ветку и сразу переключиться на неё
git checkout -b feature/tests/registration

# Современная альтернатива checkout для переключения (Git 2.23+)
git switch feature/tests/registration

# Создать и переключиться (через switch)
git switch -c feature/tests/registration

# Посмотреть все ветки (локальные)
git branch

# Посмотреть все ветки (включая удалённые)
git branch -a

# Удалить ветку (после мержа)
git branch -d feature/tests/registration

# Принудительное удаление ветки (даже если не слита)
git branch -D feature/tests/old-experiment
```

### checkout vs switch

| Аспект | `git checkout` | `git switch` |
|--------|---------------|--------------|
| Назначение | Универсальная (ветки + файлы) | Только переключение веток |
| Доступность | Все версии Git | Git 2.23+ |
| Безопасность | Может случайно затереть файлы | Безопаснее, только для веток |
| Рекомендация | Знать обязательно | Предпочтительнее для веток |

## Merge vs Rebase

### git merge

```bash
# Влить ветку develop в текущую ветку
git merge develop
```

Merge создаёт **merge commit** — специальный коммит, объединяющий две ветки. История сохраняется нелинейной.

```
      A---B---C  feature/tests/login
     /         \
D---E---F---G---H  develop  (H — merge commit)
```

### git rebase

```bash
# Перебазировать текущую ветку на develop
git rebase develop
```

Rebase «перемещает» коммиты текущей ветки поверх целевой. История становится линейной.

```
# До rebase:
      A---B---C  feature/tests/login
     /
D---E---F---G  develop

# После rebase:
              A'--B'--C'  feature/tests/login
             /
D---E---F---G  develop
```

### Сравнение

| Критерий | Merge | Rebase |
|----------|-------|--------|
| История | Нелинейная, сохраняет факт слияния | Линейная, «чистая» |
| Безопасность | Безопасен для shared-веток | Опасен для shared-веток (переписывает историю) |
| Конфликты | Решаются один раз | Могут возникать на каждом коммите |
| Когда использовать | Вливание feature-ветки в develop | Обновление feature-ветки перед merge |

**Золотое правило:** никогда не делайте rebase ветки, которая уже опубликована и используется другими людьми.

## Git Flow модель

Git Flow — популярная модель ветвления, широко используемая в командах с QA Automation.

### Основные ветки

| Ветка | Назначение | Время жизни |
|-------|-----------|-------------|
| `main` (master) | Стабильный production-код | Вечная |
| `develop` | Основная ветка разработки | Вечная |
| `feature/*` | Новый функционал / новые тесты | Временная |
| `release/*` | Подготовка к релизу | Временная |
| `hotfix/*` | Срочные исправления в production | Временная |

### Жизненный цикл в Git Flow

```
main:     ──●──────────────────●────────●──
             \                /        /
release:      \        ──●──●        /
               \      /      \      /
develop:  ──●───●────●────●───●────●──
               \    /        \
feature:        ●──●          ●──●
```

### Ветки для автоматизации тестирования

В команде QA Automation часто используют дополнительные соглашения об именовании:

```bash
# Новые автотесты для конкретной фичи
feature/tests/JIRA-123-login-validation

# Рефакторинг тестового фреймворка
feature/framework/update-base-test

# Исправление упавших тестов
fix/tests/flaky-payment-test

# Обновление тестовых данных или конфигурации
chore/tests/update-test-config
```

**Типичные соглашения (conventions):**
- Префикс `feature/tests/` — для новых автотестов
- Префикс `fix/tests/` — для починки сломанных тестов
- Префикс `feature/framework/` — для изменений инфраструктуры фреймворка
- Имя ветки содержит номер задачи из Jira / YouTrack

## Разрешение merge-конфликтов в тестовом коде

Merge-конфликты — частая ситуация, когда несколько QA-инженеров работают над одним фреймворком.

### Типичные причины конфликтов в тестах

1. Два человека добавили тесты в один и тот же тестовый класс
2. Изменения в базовом классе (BaseTest) затрагивают всех
3. Параллельное обновление конфигурационных файлов (`application.properties`, `testng.xml`)
4. Обновление зависимостей в `pom.xml` / `build.gradle`

### Как выглядит конфликт

```java
public class LoginTest extends BaseTest {

<<<<<<< HEAD
    @Test
    void testLoginWithExpiredToken() {
        // Ваш код
    }
=======
    @Test
    void testLoginWithRevokedToken() {
        // Код коллеги
    }
>>>>>>> feature/tests/auth-tokens
}
```

### Алгоритм разрешения

```bash
# 1. Обновить develop
git fetch origin

# 2. Попытаться смержить
git merge origin/develop

# 3. Увидеть файлы с конфликтами
git status

# 4. Открыть файл, разрешить конфликт (оставить оба теста!)

# 5. После разрешения — добавить и закоммитить
git add src/test/java/LoginTest.java
git commit -m "Разрешить конфликт: объединить тесты auth"
```

**Совет для QA:** в тестовых классах чаще всего нужно оставить **оба** варианта, а не выбирать один. Каждый тест — это самостоятельная единица, и удаление чужого теста недопустимо.

## .gitignore для тестовых проектов

Файл `.gitignore` определяет, какие файлы и директории Git должен игнорировать. Для тестового фреймворка на Java типичный `.gitignore` выглядит так:

```gitignore
# Скомпилированные файлы и артефакты сборки
target/
build/
out/

# Результаты отчётов (генерируются при каждом прогоне)
allure-results/
allure-report/
reports/
surefire-reports/

# Настройки IDE (у каждого свои)
.idea/
*.iml
.vscode/
.settings/
.project
.classpath

# Файлы ОС
.DS_Store
Thumbs.db

# Логи
*.log
logs/

# Переменные окружения и секреты
.env
*.properties.local
secrets/

# Скриншоты неудачных тестов (генерируются при прогоне)
screenshots/
test-output/
```

**Почему важно:**
- `allure-results/` — генерируется при каждом прогоне; хранить в Git не нужно
- `target/` — результат сборки Maven; пересоздаётся командой `mvn clean`
- `.idea/` — настройки IntelliJ IDEA; у каждого разработчика могут отличаться
- `.env` — может содержать пароли и токены; **никогда** не коммитить!

## git stash

`git stash` — временное сохранение незакоммиченных изменений. Незаменимо, когда нужно срочно переключиться на другую ветку.

```bash
# Сохранить текущие изменения в stash
git stash

# Сохранить с описанием (рекомендуется)
git stash save "Недописанный тест оплаты"

# Посмотреть список stash-записей
git stash list

# Применить последний stash и удалить его из списка
git stash pop

# Применить stash, не удаляя
git stash apply

# Применить конкретный stash
git stash apply stash@{2}

# Удалить все stash-записи
git stash clear
```

**Сценарий для QA:**
1. Вы пишете тест авторизации, он ещё не готов
2. Приходит срочная задача — починить flaky-тест на другой ветке
3. `git stash save "WIP: тест авторизации"` — сохранить текущую работу
4. Переключиться на другую ветку, починить тест
5. Вернуться и `git stash pop` — продолжить работу

## git log, blame, diff

### git log — просмотр истории

```bash
# Полная история
git log

# Краткий формат (одна строка на коммит)
git log --oneline

# История конкретного файла
git log --oneline src/test/java/LoginTest.java

# Красивый граф ветвления
git log --oneline --graph --all

# Последние 5 коммитов
git log -5

# Коммиты конкретного автора
git log --author="Иванов"
```

### git blame — кто изменил строку

```bash
# Показать, кто и когда изменил каждую строку файла
git blame src/test/java/LoginTest.java

# Blame конкретных строк (с 10 по 20)
git blame -L 10,20 src/test/java/LoginTest.java
```

**Для QA:** `git blame` помогает найти, кто и когда сломал тест, или кому задать вопрос по конкретному куску кода.

### git diff — просмотр различий

```bash
# Различия между рабочей директорией и staging area
git diff

# Различия в staging area (что войдёт в следующий коммит)
git diff --staged

# Различия между двумя ветками
git diff develop..feature/tests/login

# Различия в конкретном файле
git diff src/test/java/LoginTest.java

# Различия между двумя коммитами
git diff abc1234..def5678
```

## Практический ежедневный workflow QA-инженера

### Утро: начало рабочего дня

```bash
# 1. Переключиться на develop и обновить
git switch develop
git pull origin develop

# 2. Создать ветку для новой задачи
git switch -c feature/tests/JIRA-456-cart-validation

# 3. Писать тесты...
```

### В течение дня: регулярные коммиты

```bash
# Посмотреть, что изменилось
git status
git diff

# Добавить изменения порционно (атомарные коммиты)
git add src/test/java/cart/CartTest.java
git commit -m "JIRA-456: добавить тест проверки добавления товара в корзину"

git add src/test/java/cart/CartEmptyTest.java
git commit -m "JIRA-456: добавить тест пустой корзины"

# Отправить в удалённый репозиторий
git push origin feature/tests/JIRA-456-cart-validation
```

### Перед merge: обновить ветку

```bash
# Вариант 1: merge develop в свою ветку
git fetch origin
git merge origin/develop

# Вариант 2: rebase (если ветка ещё не расшарена)
git fetch origin
git rebase origin/develop
```

### Создание Pull Request / Merge Request

После push создаётся PR / MR в GitHub / GitLab. В команде QA Automation типичный процесс:
1. Создать PR с описанием: какие тесты добавлены, что покрывают
2. Запускается CI-пайплайн — прогон тестов
3. Code review от коллег-автоматизаторов
4. После одобрения — merge в develop

## Связь с тестированием

| Аспект | Как Git помогает QA |
|--------|-------------------|
| Версионирование тестов | Вся история изменений в тестовом фреймворке сохраняется |
| Совместная работа | Несколько QA-инженеров работают над одним фреймворком параллельно |
| CI/CD интеграция | Jenkins / GitLab CI запускают тесты при push / merge |
| Трассируемость | Через `git log` и `git blame` можно понять, когда и почему тест был изменён |
| Откат изменений | Можно вернуться к рабочей версии, если новые тесты сломали сборку |
| Ветки для экспериментов | Можно безопасно пробовать новые подходы в отдельной ветке |

## Типичные ошибки

1. **Коммит секретов** — пароли, токены, API-ключи попали в `.env` и были закоммичены. Даже после удаления из файла они остаются в истории Git. Решение: `.gitignore` + `git-secrets` / pre-commit hooks.

2. **Работа напрямую в develop/main** — без feature-ветки. Приводит к конфликтам и невозможности сделать code review.

3. **Один огромный коммит** — «добавить все тесты». Невозможно понять, что изменилось, и трудно откатить. Практика: атомарные коммиты, один тест — один коммит (или логическая группа).

4. **Rebase shared-ветки** — rebase ветки `develop` или любой ветки, которую используют другие люди. Переписывает историю и ломает работу коллег.

5. **Забыть обновить ветку перед PR** — создание PR без актуальной версии develop. Результат: конфликты и failed CI.

6. **Коммит allure-results / target** — коммит артефактов сборки засоряет репозиторий и увеличивает его размер.

7. **Непонятные сообщения коммитов** — «fix», «update», «test». Через месяц невозможно понять, что было изменено. Хорошее сообщение: `JIRA-123: добавить тест валидации email при регистрации`.

8. **Игнорирование git status перед коммитом** — можно случайно добавить лишние файлы.

## Практические задания

### Задание 1: Базовый workflow

1. Создайте новый Git-репозиторий
2. Создайте файл `LoginTest.java` с простым тестом
3. Добавьте `.gitignore` для Java-проекта
4. Сделайте коммит с осмысленным сообщением
5. Создайте ветку `feature/tests/registration`
6. Добавьте `RegistrationTest.java`
7. Закоммитьте и смержите ветку обратно в `main`

### Задание 2: Работа с конфликтами

1. Создайте ветку `feature/tests/branchA` от `main`
2. В `main` измените файл `LoginTest.java`
3. В `feature/tests/branchA` измените тот же файл в тех же строках
4. Попробуйте смержить — разрешите конфликт
5. Проверьте историю через `git log --oneline --graph`

### Задание 3: Stash и переключение контекста

1. Начните писать тест в ветке `feature/tests/payment`
2. Не коммитя, сделайте `git stash`
3. Переключитесь на другую ветку, внесите изменения, закоммитьте
4. Вернитесь на `feature/tests/payment` и восстановите stash

### Задание 4: Исследование истории

1. Клонируйте любой открытый тестовый проект на GitHub
2. Используя `git log`, `git blame` и `git diff`, ответьте:
   - Кто последним изменял конкретный тестовый файл?
   - Какие изменения были сделаны в последних 5 коммитах?
