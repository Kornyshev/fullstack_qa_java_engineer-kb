# Практика ручного тестирования на открытых веб-приложениях

## Введение

Ручное тестирование начинается с практики на реальных приложениях. Ниже — список открытых
веб-приложений, доступных бесплатно, с конкретными заданиями для каждого. Каждое задание
имеет чёткий результат (deliverable), который вы сохраняете в своё портфолио.

**Правило**: не просто «потыкайте приложение». У каждого задания есть конкретная цель,
конкретные шаги и конкретный ожидаемый результат.

## 1. SauceDemo

**URL**: https://www.saucedemo.com

**Описание**: учебный интернет-магазин от Sauce Labs. Идеален для первой практики —
простой, предсказуемый, с намеренно встроенными багами.

### Учётные записи для тестирования
| Логин | Пароль | Особенность |
|-------|--------|-------------|
| `standard_user` | `secret_sauce` | Стандартный пользователь |
| `locked_out_user` | `secret_sauce` | Заблокированный пользователь |
| `problem_user` | `secret_sauce` | Пользователь с ошибками в интерфейсе |
| `performance_glitch_user` | `secret_sauce` | Медленная загрузка |
| `error_user` | `secret_sauce` | Пользователь, вызывающий ошибки |
| `visual_user` | `secret_sauce` | Визуальные дефекты интерфейса |

### Задание 1.1: Тестирование авторизации
**Цель**: проверить все сценарии логина.

**Конкретные проверки**:
1. Успешный логин с `standard_user` → ожидаем переход на страницу каталога
2. Логин с `locked_out_user` → ожидаем сообщение об ошибке
3. Логин с пустым логином → ожидаем валидационное сообщение
4. Логин с пустым паролем → ожидаем валидационное сообщение
5. Логин с пустыми обоими полями → ожидаем валидационное сообщение
6. Логин с неверным паролем → ожидаем сообщение об ошибке
7. Логин с несуществующим пользователем → ожидаем сообщение об ошибке
8. Проверка, что пароль маскируется (тип поля `password`)
9. Проверка кнопки закрытия сообщения об ошибке
10. Логин с `performance_glitch_user` → замерить время загрузки

**Deliverable**: файл `saucedemo-login-tests.md` с 10 тест-кейсами в формате:
ID, название, предусловия, шаги, ожидаемый результат, фактический результат, статус (Pass/Fail).

### Задание 1.2: Тестирование каталога товаров
**Цель**: проверить отображение и сортировку товаров.

**Конкретные проверки**:
1. После логина отображаются 6 товаров — проверить количество
2. У каждого товара есть: название, описание, цена, картинка, кнопка «Add to cart»
3. Сортировка A-Z — товары в алфавитном порядке
4. Сортировка Z-A — товары в обратном алфавитном порядке
5. Сортировка Price (low to high) — цены по возрастанию
6. Сортировка Price (high to low) — цены по убыванию
7. Клик на название товара → переход на страницу товара
8. Клик на картинку товара → переход на страницу товара
9. Кнопка «Back to products» на странице товара → возврат в каталог
10. Проверка сортировки с `problem_user` — найти баги

**Deliverable**: чек-лист из 10 пунктов с результатами проверки + список найденных багов.

### Задание 1.3: Тестирование корзины и checkout
**Цель**: проверить полный flow покупки.

**Конкретные проверки**:
1. Добавить 1 товар → счётчик корзины показывает «1»
2. Добавить 3 товара → счётчик показывает «3»
3. Удалить товар из каталога (кнопка «Remove») → счётчик уменьшается
4. Перейти в корзину → отображаются добавленные товары с правильными ценами
5. Удалить товар из корзины → товар исчезает
6. «Continue Shopping» → возврат в каталог, корзина сохраняется
7. «Checkout» → форма ввода данных (First Name, Last Name, Zip Code)
8. Отправка формы с пустыми полями → валидационные сообщения
9. Заполнить форму → страница подтверждения с итоговой суммой
10. Проверить, что сумма = сумма товаров + налог (Tax)
11. «Finish» → страница благодарности «Thank you for your order!»
12. «Back Home» → возврат в каталог, корзина пуста

**Deliverable**: файл `saucedemo-e2e-scenario.md` — полный end-to-end тест-кейс
от логина до завершения заказа с ожидаемыми и фактическими результатами на каждом шаге.

### Задание 1.4: Тестирование с problem_user
**Цель**: найти все баги, специфичные для `problem_user`.

**Конкретные задачи**:
1. Залогиниться как `problem_user`
2. Проверить все картинки товаров — все ли корректны?
3. Попробовать сортировку — работает ли?
4. Добавить товар в корзину, затем удалить — работает ли кнопка «Remove»?
5. Пройти checkout — заполняются ли поля формы?
6. Попробовать оформить заказ до конца

**Deliverable**: файл `saucedemo-problem-user-bugs.md` — баг-репорты для каждого
найденного дефекта (минимум 3 бага).

## 2. The Internet (Heroku App)

**URL**: https://the-internet.herokuapp.com

**Описание**: коллекция отдельных страниц, каждая демонстрирует конкретный UI-паттерн.
Идеально для тренировки тестирования различных UI-элементов.

### Задание 2.1: Тестирование основных страниц
Протестируйте следующие страницы и запишите результаты:

| Страница | URL | Что проверять |
|----------|-----|---------------|
| A/B Testing | `/abtest` | Отображается ли заголовок, меняется ли вариант |
| Add/Remove Elements | `/add_remove_elements/` | Добавление/удаление кнопок, корректность счётчика |
| Basic Auth | `/basic_auth` | Логин admin/admin, отказ при неверных данных |
| Broken Images | `/broken_images` | Какие картинки не загружаются (проверить через DevTools) |
| Checkboxes | `/checkboxes` | Начальное состояние, переключение, независимость чекбоксов |
| Disappearing Elements | `/disappearing_elements` | Сколько элементов в навигации, появляется ли 5-й элемент при обновлении |
| Drag and Drop | `/drag_and_drop` | Перетаскивание блока A в B и обратно |

**Deliverable**: файл `the-internet-checklist.md` — чек-лист из 10 страниц,
для каждой: 3-5 проверок с результатами (Pass/Fail).

### Задание 2.2: Тестирование форм и аутентификации
**Конкретные проверки**:
1. **Form Authentication** (`/login`): логин `tomsmith`/`SuperSecretPassword!` →
   проверить flash-сообщение, logout, попытка доступа к `/secure` без логина
2. **Forgot Password** (`/forgot_password`): ввод email, отправка формы
3. **Inputs** (`/inputs`): ввод чисел, букв, спецсимволов, отрицательных чисел, стрелки вверх/вниз
4. **Key Presses** (`/key_presses`): нажатие различных клавиш, проверка отображения

**Deliverable**: тест-кейсы для каждой формы (минимум 5 на форму).

### Задание 2.3: Тестирование JavaScript-элементов
**Конкретные проверки**:
1. **JavaScript Alerts** (`/javascript_alerts`): Alert, Confirm (OK и Cancel), Prompt (текст и Cancel)
2. **JavaScript Onload Event Error** (`/javascript_error`): открыть DevTools Console, найти ошибку
3. **Infinite Scroll** (`/infinite_scroll`): прокрутить минимум 3 раза, проверить подгрузку
4. **Notification Messages** (`/notification_message_rendered`): обновить 10 раз,
   записать все варианты сообщений

**Deliverable**: файл `the-internet-js-tests.md` с результатами проверок.

## 6. Reqres.in

**URL**: https://reqres.in

**Описание**: открытый REST API для тренировки. Поддерживает GET, POST, PUT, PATCH, DELETE.
Идеален для первого знакомства с API-тестированием.

### Задание 6.1: Тестирование API через браузер и DevTools
**Конкретные проверки** (используйте DevTools → Network или Postman):
1. `GET /api/users?page=2` → проверить структуру ответа: `page`, `per_page`, `total`, `data[]`
2. `GET /api/users/2` → проверить поля пользователя: `id`, `email`, `first_name`, `last_name`, `avatar`
3. `GET /api/users/23` → проверить ответ 404 (пользователь не найден)
4. `POST /api/users` с телом `{"name": "morpheus", "job": "leader"}` → проверить 201 и поля `id`, `createdAt`
5. `PUT /api/users/2` → проверить 200 и поле `updatedAt`
6. `DELETE /api/users/2` → проверить 204 (No Content)
7. `POST /api/register` с `{"email": "eve.holt@reqres.in", "password": "pistol"}` → проверить `token`
8. `POST /api/register` без пароля → проверить 400 и сообщение об ошибке
9. `POST /api/login` с валидными данными → проверить `token`
10. `POST /api/login` без пароля → проверить 400

**Deliverable**: файл `reqres-api-checklist.md` — таблица: метод, URL, тело запроса,
ожидаемый статус, ожидаемое тело, фактический результат, статус (Pass/Fail).

### Задание 6.2: Тестирование пагинации
**Конкретные проверки**:
1. `GET /api/users?page=1` → `per_page` = 6, `total` = 12, `total_pages` = 2
2. `GET /api/users?page=2` → другой набор пользователей
3. `GET /api/users?page=3` → пустой массив `data`
4. `GET /api/users?page=0` → что возвращает?
5. `GET /api/users?page=-1` → что возвращает?
6. `GET /api/users?page=abc` → что возвращает?
7. `GET /api/users?delay=3` → ответ приходит через 3 секунды

**Deliverable**: 7 тест-кейсов для пагинации API.

## Советы по выполнению заданий

1. **Не торопитесь**: лучше 5 качественных тест-кейсов, чем 20 поверхностных
2. **Используйте DevTools**: F12 → Network tab показывает реальные запросы к серверу

6. **Записывайте всё**: даже если что-то кажется незначительным — запишите. Лучше удалить лишнее, чем забыть важное
