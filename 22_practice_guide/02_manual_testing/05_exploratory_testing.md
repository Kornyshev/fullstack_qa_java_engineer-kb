# Практика Exploratory Testing

## Введение

Exploratory testing — это одновременное проектирование, выполнение и изучение тестов.
В отличие от scripted testing (выполнение заранее написанных тест-кейсов), exploratory testing
полагается на знания, интуицию и наблюдательность тестировщика. Это не «случайное тыканье» —
это структурированное исследование с чётким фокусом и документированием результатов.

### Когда exploratory testing эффективнее scripted testing

| Ситуация | Scripted | Exploratory |
|----------|---------|-------------|
| Новый функционал без требований | Не подходит (нет базы для скриптов) | **Идеально** — исследование «что это и как работает» |
| Регрессия стабильного функционала | **Идеально** — повторяемые проверки | Менее эффективно — автотесты лучше |
| Поиск неочевидных багов | Ограниченно — проверяет только то, что задумано | **Идеально** — свобода исследования |
| Оценка usability | Не подходит — формальные шаги не покрывают UX | **Идеально** — тестировщик оценивает удобство |
| Ограниченное время (1 час на тестирование) | Неэффективно — написание тест-кейсов занимает время | **Идеально** — сразу к делу |
| Compliance / аудит | **Идеально** — документированные доказательства | Не подходит — нет формальной трассируемости |
| Сложная бизнес-логика | Нужны оба подхода | Нужны оба подхода |

### Мифы об exploratory testing

**Миф 1**: «Exploratory testing — это тестирование без плана»
**Реальность**: есть charter (задание), time box (ограничение по времени), структурированные заметки.

**Миф 2**: «Exploratory testing может заменить scripted testing»
**Реальность**: они дополняют друг друга. Scripted — для регрессии и критических путей. Exploratory — для поиска неожиданных проблем.

**Миф 3**: «Exploratory testing может делать только опытный тестировщик»
**Реальность**: начинающие тоже могут. Просто у опытных — больше эвристик и паттернов для поиска.

## Session-Based Test Management (SBTM)

SBTM — это методология управления exploratory testing, предложенная James Bach и Jonathan Bach.
Основная идея: разделить exploratory testing на управляемые сессии с чётким фокусом и отчётностью.

### Ключевые элементы SBTM

1. **Charter** (Задание) — что исследовать и зачем
2. **Time Box** (Временные рамки) — обычно 60-90 минут (для практики — 30 минут)
3. **Session Notes** (Заметки сессии) — что делал, что наблюдал, что нашёл
4. **Debrief** (Разбор) — анализ результатов после сессии

### Шаблон сессии

```
=====================================
EXPLORATORY TESTING SESSION REPORT
=====================================

Session ID:     [ET-001]
Date:           [дата]
Tester:         [имя]
Duration:       [фактическое время]

CHARTER
-------
Исследовать [область/функционал]
с целью [что хотим узнать/проверить]
используя [подход/эвристики]

TIME BOX
--------
Запланировано: [30 мин / 60 мин / 90 мин]
Фактически:    [сколько потрачено]

TIME BREAKDOWN
--------------
- Исследование (Testing):     [мин] ([%])
- Изучение (Learning):        [мин] ([%])
- Настройка (Setup):          [мин] ([%])
- Отвлечения (Distractions):  [мин] ([%])

AREAS COVERED
-------------
1. [Перечень проверенных областей/страниц/функций]
2. ...
3. ...

NOTES
-----
[Хронологические заметки: что делал, что наблюдал]

Время | Действие | Наблюдение
------|----------|----------
00:00 | ...      | ...
05:00 | ...      | ...

BUGS FOUND
----------
[Список найденных багов с краткими описаниями]

Bug 1: [Summary]
  Severity: [Severity]
  Steps: [краткие шаги]
  Screenshot: [ссылка]

Bug 2: ...

QUESTIONS / CONCERNS
--------------------
[Вопросы, которые возникли во время сессии]
[Области, требующие дополнительного исследования]

IDEAS FOR FUTURE SESSIONS
--------------------------
[Что стоит проверить в следующих сессиях]

SUMMARY
-------
[Краткий вывод: общее впечатление о качестве, рекомендации]
```

## Задание 1: 30-минутная exploratory-сессия на SauceDemo

### Charter
```
Исследовать интернет-магазин SauceDemo (https://www.saucedemo.com)
с целью найти функциональные и визуальные дефекты
используя все доступные типы пользователей (standard, problem, error, visual)
```

### Подготовка (5 минут ДО старта таймера)
1. Откройте https://www.saucedemo.com
2. Подготовьте шаблон сессии (скопируйте шаблон выше)
3. Откройте DevTools (F12) → Console (для отслеживания JS-ошибок)
4. Откройте DevTools → Network (для отслеживания HTTP-ошибок)
5. Установите таймер на 30 минут

### План исследования (рекомендация, не обязательство)
```
Минуты 00-10: Быстрый обзор всех страниц с standard_user
  - Логин → Каталог → Товар → Корзина → Checkout → Complete
  - Фиксировать всё необычное

Минуты 10-18: Исследование с problem_user
  - Те же страницы, но внимание на отличия от standard_user
  - Картинки, кнопки, формы

Минуты 18-25: Исследование с error_user
  - Фокус на ошибках JavaScript (Console)
  - Попытка пройти весь flow

Минуты 25-30: Исследование с visual_user
  - Визуальные дефекты
  - Сравнение с standard_user
```

### Эвристики для поиска багов

Используйте эти эвристики как «линзы» для исследования:

**CRUD** — Create, Read, Update, Delete:
- Можно ли создать? (добавить в корзину)
- Можно ли прочитать? (просмотреть товар)
- Можно ли обновить? (изменить количество)
- Можно ли удалить? (удалить из корзины)

**Навигация**:
- Все ли ссылки работают?
- Кнопка «Назад» браузера — ломает ли что-нибудь?
- Прямой доступ по URL (без логина) — что происходит?

**Данные**:
- Что если ввести пустые данные?
- Что если ввести очень длинные данные?
- Что если ввести спецсимволы?
- Совпадают ли цены на всех страницах?

**Состояние**:
- Что если обновить страницу посреди checkout?
- Что если открыть в двух вкладках?
- Что если выйти и войти снова — корзина сохранилась?

**Производительность**:
- Есть ли заметные задержки? (performance_glitch_user)
- Какие запросы летят на сервер? (Network tab)

## Задание 2: Exploratory-сессия на The Internet

### Charter
```
Исследовать https://the-internet.herokuapp.com
с целью найти функциональные дефекты и неочевидные edge cases
используя эвристику "Что если..." для каждой страницы
```

### Time Box
30 минут

### Рекомендуемый маршрут
Вместо последовательного тестирования каждой страницы, выберите 5-7 страниц
и исследуйте каждую глубоко.

**Рекомендуемые страницы для исследования**:
1. `/login` — Что если ввести SQL-инъекцию? Что если 100 неудачных попыток?
2. `/dynamic_loading/1` и `/dynamic_loading/2` — Что если нажать Start дважды?
3. `/drag_and_drop` — Работает ли без JavaScript? Работает ли на тачскрине?
4. `/add_remove_elements/` — Что если добавить 100 элементов? Что если быстро кликать?
5. `/upload` и `/download` — Что если загрузить файл 0 байт? Что если .exe?
6. `/floating_menu` — Меню следует за прокруткой? Перекрывает ли контент?
7. `/frames` — Работает ли навигация внутри iframe?

### Эвристики для The Internet
- **«Что если...»**: для каждого элемента задайте вопрос «что если сделать не так?»
- **«Никогда / Всегда»**: проверьте крайности — 0 элементов, 1000 элементов
- **«Быстро / Медленно»**: быстрые клики, медленное соединение
- **«Два раза»**: выполните любое действие дважды — что произойдёт?
- **«Назад»**: после каждого действия нажмите кнопку «Назад» в браузере

### Deliverable
Файл `the-internet-exploratory-session.md` — полный отчёт о 30-минутной сессии.

## Задание 4: Тематическая exploratory-сессия (Security)

### Charter
```
Исследовать https://juice-shop.herokuapp.com
с целью найти уязвимости безопасности базового уровня
используя OWASP Top 10 как руководство
```

### Time Box
45 минут

### Чек-лист проверок (используйте как отправные точки, не ограничивайтесь ими)

**Injection**:
- Попробовать `' OR 1=1--` в поле логина
- Попробовать `<script>alert(1)</script>` в поле поиска
- Попробовать `<img src=x onerror=alert(1)>` в поле отзыва

**Broken Authentication**:
- Можно ли получить доступ к `/administration` без логина?
- Что хранится в localStorage / sessionStorage после логина?
- Можно ли декодировать JWT-токен? (jwt.io)

**Sensitive Data Exposure**:
- Передаётся ли пароль в открытом виде? (Network tab)
- Есть ли чувствительные данные в HTML-коде страницы?
- Что возвращает API в ответах — нет ли лишних полей?

**Security Misconfiguration**:
- Открыть `/ftp` — есть ли доступные файлы?
- Проверить заголовки ответа (X-Frame-Options, Content-Security-Policy)
- Есть ли `robots.txt` с интересной информацией?

## Задание 5: Парная exploratory-сессия (если есть напарник)

### Формат
Два человека работают вместе:
- **Driver** (Водитель) — управляет мышью и клавиатурой
- **Navigator** (Навигатор) — наблюдает, задаёт вопросы, фиксирует находки

Меняются ролями каждые 15 минут.

### Charter
```
Исследовать https://demo.opencart.com
с целью проверить процесс покупки от выбора товара до оформления заказа
используя эвристики «Что если» и «Никогда/Всегда»
```

### Time Box
60 минут (30 + 30 со сменой ролей)

### Правила парной сессии
1. Navigator не трогает клавиатуру — только говорит
2. Driver не молчит — комментирует каждое действие
3. Оба записывают находки (каждый в своих заметках)
4. После сессии — 15 минут на debrief: сравнить заметки, обсудить находки

### Deliverable
Два файла: `opencart-session-driver.md` и `opencart-session-navigator.md` —
заметки каждого участника + общий debrief-отчёт.

## Как документировать находки

### Уровень 1: Минимальная документация (во время сессии)
Пишите быстро, разберёте потом. Главное — не забыть.

```
11:05 - problem_user, каталог - ВСЕ КАРТИНКИ ОДИНАКОВЫЕ!!!
11:07 - сортировка Z-A не работает (problem_user)
11:09 - Remove не работает для Onesie (problem_user)
11:15 - error_user: JS error в консоли при добавлении в корзину
```

### Уровень 2: Средняя документация (после сессии, в течение 15 минут)
Расширьте минимальные заметки до читаемого формата.

```
## Найденные дефекты

### Баг 1: Одинаковые картинки для problem_user
- Где: каталог товаров
- Пользователь: problem_user
- Суть: все 6 товаров отображают одну и ту же картинку (sl-404.jpg)
- Severity: Major

### Баг 2: Сортировка не работает для problem_user
- Где: каталог товаров
- Пользователь: problem_user
- Суть: при выборе Z-A порядок товаров не меняется
- Severity: Major
```

### Уровень 3: Полная документация (для критических багов)
Оформите полный баг-репорт по шаблону из раздела о баг-репортах.

### Когда какой уровень использовать
- **Уровень 1**: во время сессии — всегда
- **Уровень 2**: после каждой сессии — обязательно
- **Уровень 3**: только для багов, которые будут заведены в трекер (Jira, Bugzilla)

## Эвристики для exploratory testing

### Мнемоника SFDPOT (James Bach)

| Буква | Область | Что проверять |
|-------|---------|---------------|
| **S** — Structure | Из чего состоит продукт | Файлы, код, базы данных, интерфейсы |
| **F** — Function | Что делает продукт | Каждая функция, входы/выходы, ошибки |
| **D** — Data | Какие данные обрабатывает | Типы данных, объёмы, форматы |
| **P** — Platform | На чём работает | ОС, браузеры, устройства |
| **O** — Operations | Как будут использовать | Типичные сценарии, нагрузка, обслуживание |
| **T** — Time | Временные аспекты | Таймауты, сессии, кеширование |

### Мнемоника FEW HICCUPS

| Буква | Эвристика | Пример |
|-------|-----------|--------|
| **F** — Familiar | Знакомые проблемы | Этот баг я уже видел в другом проекте |
| **E** — Explainability | Объяснимость | Могу ли я объяснить, что произошло? |
| **W** — World | Реальный мир | Как реальный пользователь будет это использовать? |
| **H** — History | История | Были ли баги в этом модуле раньше? |
| **I** — Image | Имидж | Как это влияет на репутацию продукта? |
| **C** — Claims | Заявления | Продукт делает то, что обещано? |
| **C** — Comparable | Сравнимость | Как это работает у конкурентов? |
| **U** — User | Пользователь | Что подумает пользователь? Удобно ли? |
| **P** — Product | Продукт | Соответствует ли стандартам и спецификации? |
| **S** — Statutes | Законы и регуляции | Соответствует ли требованиям закона (GDPR, 152-ФЗ)? |

### Практическое применение

Перед сессией выберите 2-3 эвристики из списка. Используйте их как «линзы»:

```
Charter: Исследовать модуль оплаты
Линзы: Data (граничные значения сумм) + Time (таймаут сессии) + User (удобство)
```

## Метрики exploratory testing

### Что отслеживать после каждой сессии

| Метрика | Описание | Цель |

## Советы для эффективных exploratory-сессий

### Перед сессией
1. **Определите charter** — без фокуса сессия превратится в бесцельное блуждание
2. **Установите time box** — ограничение по времени создаёт продуктивное давление
3. **Подготовьте инструменты**: DevTools открыт, шаблон заметок готов, таймер запущен
4. **Изучите приложение** за 5 минут до старта (если новое) — не тратьте время сессии на ориентирование

### Во время сессии
1. **Записывайте ВСЁ** — даже если кажется несущественным. Заметка на 3 секунды может спасти 30 минут потом
2. **Не уходите в кроличью нору** — нашли баг? Запишите минимальную информацию и двигайтесь дальше. Полное оформление — после сессии
3. **Следуйте интуиции** — если что-то кажется подозрительным — исследуйте. Интуиция тестировщика — ценный инструмент
4. **Меняйте подход** — если 10 минут ничего не нашли, смените эвристику, смените область, смените пользователя
5. **Делайте скриншоты** — лучше 10 лишних скриншотов, чем ни одного для найденного бага

### После сессии
1. **Сразу оформите заметки** (в течение 15 минут) — через день вы забудете контекст
2. **Определите следующий шаг** — на основе находок и вопросов напишите charter для следующей сессии
3. **Оформите критические баги** — полные баг-репорты для найденных дефектов с высоким Severity
4. **Оцените продуктивность** — сколько полезного времени было? Что можно улучшить?
