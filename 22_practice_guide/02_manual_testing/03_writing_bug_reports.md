# Практика написания баг-репортов

## Введение

Баг-репорт — это документ, по которому разработчик должен воспроизвести и исправить дефект.
Хороший баг-репорт экономит часы. Плохой — создаёт раунды уточнений, переоткрытий и
фрустрации. В этом разделе вы научитесь писать баг-репорты, которые разработчики будут
благодарить, а не проклинать.

---

## Шаблон баг-репорта

```
ID:                 [BUG-001]
Summary:            [Краткое описание бага — ЧТО сломано и ГДЕ]
Severity:           [Blocker / Critical / Major / Minor / Trivial]
Priority:           [High / Medium / Low]
Environment:        [ОС, браузер, версия, разрешение экрана]
Preconditions:      [Что должно быть выполнено до воспроизведения]
Steps to Reproduce:
  1. [Шаг 1]
  2. [Шаг 2]
  3. [Шаг N]
Expected Result:    [Что должно произойти]
Actual Result:      [Что происходит на самом деле]
Attachments:        [Скриншоты, видео, логи]
Additional Info:    [Любая полезная информация: частота воспроизведения, workaround]
```

### Обязательные поля
Каждое из этих полей **необходимо**. Без них баг-репорт неполный:
- **Summary** — первое, что видит разработчик. Должно быть понятно без открытия деталей.
- **Steps to Reproduce** — конкретные шаги. Если разработчик не может воспроизвести — баг не починят.
- **Expected Result** — что должно быть по требованиям или здравому смыслу.
- **Actual Result** — что происходит на самом деле. Точно, конкретно.
- **Severity** — насколько критичен баг для системы.

## Пример хорошего баг-репорта vs плохого

### Плохой баг-репорт

```
Summary: Корзина не работает
Steps: Добавил товар, что-то пошло не так
Expected: Должно работать
Actual: Не работает
```

**Почему плохой:**
- Summary неинформативный — «не работает» КАК?
- Steps невоспроизводимы — какой товар? На какой странице?
- Expected / Actual — бессмысленные фразы
- Нет environment, severity, скриншотов
- Разработчик потратит 30 минут на уточнения

### Хороший баг-репорт

```
ID:                 BUG-SD-001
Summary:            [SauceDemo] Кнопка "Remove" не работает для товара "Sauce Labs Onesie"
                    при авторизации как problem_user
Severity:           Major
Priority:           High
Environment:        Windows 11, Chrome 120.0.6099.130, разрешение 1920x1080
Preconditions:
  1. Открыта страница https://www.saucedemo.com
  2. Пользователь авторизован как problem_user / secret_sauce

Steps to Reproduce:
  1. На странице каталога найти товар "Sauce Labs Onesie"
  2. Нажать кнопку "Add to cart" под товаром
  3. Убедиться, что кнопка изменилась на "Remove"
  4. Нажать кнопку "Remove"

Expected Result:
  Товар удалён из корзины. Кнопка изменилась обратно на "Add to cart".
  Badge на иконке корзины уменьшился на 1.

Actual Result:
  Кнопка "Remove" не реагирует на нажатие. Товар остаётся в корзине.
  Badge на иконке корзины не изменяется.
  В console DevTools нет ошибок.

Attachments:
  - screenshot_remove_button.png (скриншот состояния кнопки)
  - video_reproduction.mp4 (видео воспроизведения — 15 сек)

Additional Info:
  - Воспроизводится в 100% случаев
  - Баг специфичен для problem_user — у standard_user кнопка Remove работает корректно
  - Workaround: удалить товар из корзины (страница /cart.html), кнопка Remove там работает
```

**Почему хороший:**
- Summary: сразу понятно ЧТО, ГДЕ и ПРИ КАКИХ УСЛОВИЯХ
- Steps: можно воспроизвести за 30 секунд
- Expected vs Actual: чёткое сравнение
- Есть workaround — разработчик может снизить приоритет, зная обходной путь
- Есть дополнительная информация о scope бага

## Severity vs Priority

### Severity (серьёзность) — технический импакт

| Severity | Описание | Примеры |
|----------|----------|---------|
| **Blocker** | Система неработоспособна. Нельзя продолжить тестирование. | Приложение не открывается, crash при запуске, потеря данных |
| **Critical** | Основной функционал не работает. Нет workaround. | Невозможно оформить заказ, невозможно залогиниться |
| **Major** | Основной функционал работает с ошибками. Есть workaround. | Кнопка Remove не работает (но можно удалить из корзины), неверный расчёт суммы |
| **Minor** | Второстепенный функционал работает некорректно. | Сортировка не работает, тултип не отображается, неверный формат даты |
| **Trivial** | Косметический дефект. Не влияет на функциональность. | Опечатка в тексте, неровное выравнивание, лишний пиксель в отступе |

### Priority (приоритет) — бизнес-импакт

| Priority | Описание | Примеры |
|----------|----------|---------|
| **High** | Исправить немедленно. Блокирует релиз или бизнес. | Баг на странице оплаты, утечка персональных данных |
| **Medium** | Исправить в текущем спринте. Важно, но не критично. | Некорректная сортировка, баг в редко используемой функции |
| **Low** | Исправить когда будет время. Минимальный бизнес-импакт. | Опечатка на странице «О компании», неверный цвет иконки |

### Матрица Severity × Priority

| | High Priority | Medium Priority | Low Priority |
|---|---|---|---|
| **Blocker** | Crash на главной странице | Crash в админке (мало пользователей) | — (практически невозможно) |
| **Critical** | Оплата не проходит | Экспорт отчётов сломан | — |
| **Major** | Неверная цена на главной | Фильтр в каталоге не работает | Баг в устаревшем API |
| **Minor** | Опечатка в названии компании на главной | Тултип не появляется | Неверная иконка в футере |
| **Trivial** | — | — | Лишний пробел в тексте |

**Ключевой вывод**: Severity и Priority — разные оси. Опечатка в названии компании на главной
странице: Severity = Trivial (косметика), Priority = High (видят все пользователи, репутация).

## Советы по написанию эффективных баг-репортов

### 1. Summary — самое важное поле
Разработчик видит список из 50 багов. По Summary он решает, какой открыть первым.

**Формула хорошего Summary**:
```
[Где] [Что происходит] [При каких условиях]
```

Примеры:
- `[Checkout] Итоговая сумма не включает налог при заказе более 3 товаров`
- `[Каталог] Сортировка Z-A не работает для problem_user`
- `[Корзина] Кнопка Remove не удаляет товар "Sauce Labs Onesie"`

### 2. Steps to Reproduce — минимальные и достаточные
- Уберите лишние шаги (не нужно «открыть браузер, ввести URL, нажать Enter»)
- Но не пропускайте важные (указать конкретного пользователя, конкретный товар)

### 3. Один баг — один репорт
**Плохо**: «На странице три ошибки: кнопка не работает, картинка не та, текст с опечаткой»
**Хорошо**: Три отдельных баг-репорта. У каждого свой Severity, свой fix, свой статус.

### 4. Скриншоты решают
- Подчеркните проблемное место красной рамкой или стрелкой
- Для визуальных багов — скриншот обязателен
- Для сложных сценариев — видеозапись (используйте OBS или встроенную запись экрана)

### 5. Проверьте воспроизводимость
Перед отправкой баг-репорта — воспроизведите баг ещё раз. Если не воспроизводится —
укажите частоту: «воспроизводится в 3 из 5 попыток».
