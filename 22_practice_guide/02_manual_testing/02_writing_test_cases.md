# Практика написания тест-кейсов

## Введение

Тест-кейс — основной артефакт QA-инженера. Умение писать чёткие, воспроизводимые и полные
тест-кейсы отличает профессионала от любителя. В этом разделе — конкретные задания на написание
тест-кейсов с шаблонами, примерами и критериями качества.

---

## Шаблон тест-кейса

Каждый тест-кейс должен содержать следующие обязательные поля:

```
ID:              [уникальный идентификатор, например TC-REG-001]
Название:        [краткое, понятное описание того, ЧТО проверяется]
Приоритет:       [High / Medium / Low]
Предусловия:     [что должно быть выполнено ДО начала теста]
Тестовые данные: [конкретные значения, которые используются в тесте]
Шаги:            [пронумерованные шаги — что делать]
Ожидаемый результат: [что должно произойти после выполнения шагов]
```

### Опциональные поля
```
Модуль:          [функциональный модуль: авторизация, корзина, оплата]
Тип:             [Positive / Negative / Boundary]
Автор:           [кто написал тест-кейс]
Дата создания:   [когда написан]
Статус:          [Not Executed / Pass / Fail / Blocked / Skipped]
Связанные баги:  [ID багов, найденных при выполнении]
```

## Пример хорошего тест-кейса vs плохого

### Плохой тест-кейс

```
ID: TC-001
Название: Проверить логин
Шаги: Ввести данные и нажать кнопку
Ожидаемый результат: Логин работает
```

**Почему плохой:**
- Название неинформативное — «логин» какой? Успешный? С неверным паролем?
- Нет предусловий — откуда начинаем? Какая страница открыта?
- Нет тестовых данных — какие «данные» вводить?
- Шаги неконкретные — в какое поле? Какую кнопку?
- Ожидаемый результат размытый — «работает» это как?

### Хороший тест-кейс

```
ID:              TC-AUTH-001
Название:        Успешная авторизация с валидными учётными данными
Приоритет:       High
Модуль:          Авторизация
Тип:             Positive

Предусловия:
  1. Открыт браузер Chrome (последняя версия)
  2. Открыта страница https://www.saucedemo.com

Тестовые данные:
  - Логин: standard_user
  - Пароль: secret_sauce

Шаги:
  1. Ввести "standard_user" в поле "Username"
  2. Ввести "secret_sauce" в поле "Password"
  3. Нажать кнопку "Login"

Ожидаемый результат:
  1. Выполнен переход на страницу каталога (URL: /inventory.html)
  2. Отображается заголовок "Products"
  3. Отображается список из 6 товаров
  4. В шапке видна иконка корзины
  5. В шапке видно боковое меню (гамбургер)
```

**Почему хороший:**
- Название точно описывает сценарий
- Есть предусловия — понятно, откуда начинать
- Тестовые данные конкретны — можно скопировать и вставить
- Шаги пронумерованы, каждый — одно действие
- Ожидаемый результат детальный и проверяемый

## Уровни детализации тест-кейсов

### Уровень 1: Высокоуровневый (High-Level Test Case)
Подходит для опытных тестировщиков. Описывает ЧТО проверить, но не КАК.

```
ID: TC-CART-001
Название: Добавление товара в корзину
Шаги: Добавить любой товар в корзину с главной страницы каталога
Ожидаемый результат: Товар добавлен, счётчик корзины обновлён
```

### Уровень 2: Пошаговый (Step-by-Step Test Case)
Стандартный уровень. Описывает конкретные шаги.

```
ID: TC-CART-001
Название: Добавление товара в корзину из каталога
Предусловия: Пользователь авторизован как standard_user, открыта страница каталога
Шаги:
  1. Найти товар "Sauce Labs Backpack" в каталоге
  2. Нажать кнопку "Add to cart" под товаром
Ожидаемый результат:
  1. Кнопка изменилась на "Remove"
  2. На иконке корзины появился badge с числом "1"
```

### Уровень 3: Детальный (Detailed Test Case)
Максимальная детализация. Для новичков или критически важных модулей.

```
ID: TC-CART-001
Название: Добавление товара "Sauce Labs Backpack" в корзину из каталога
Приоритет: High
Модуль: Корзина
Тип: Positive

Предусловия:
  1. Открыт браузер Chrome v120+
  2. Перейти на https://www.saucedemo.com
  3. Ввести логин "standard_user", пароль "secret_sauce"
  4. Нажать "Login"
  5. Убедиться, что открыта страница каталога (URL содержит /inventory.html)
  6. Убедиться, что корзина пуста (нет badge на иконке корзины)

Тестовые данные:
  - Товар: Sauce Labs Backpack
  - Цена: $29.99

Шаги:
  1. Найти карточку товара с названием "Sauce Labs Backpack"
  2. Проверить, что под карточкой отображается кнопка "Add to cart"
  3. Нажать кнопку "Add to cart"

Ожидаемый результат:
  1. Кнопка "Add to cart" заменена на кнопку "Remove" (красного цвета)
  2. На иконке корзины (верхний правый угол) появился badge с числом "1"
  3. Товар "Sauce Labs Backpack" сохранён в корзине (проверка: клик на иконку корзины →
     товар присутствует в списке с ценой $29.99)
```

### Когда какой уровень использовать
| Уровень | Когда использовать |
|---------|-------------------|
| Высокоуровневый | Agile-команда с опытными тестировщиками, smoke-тесты |
| Пошаговый | Стандартная документация, регрессия, передача знаний |
| Детальный | Критические модули (оплата, безопасность), новички в команде |

## Задание 2: Чек-лист для корзины покупок

**Приложение**: https://www.saucedemo.com

**Задача**: составить чек-лист для модуля «Корзина». Чек-лист — это упрощённая форма
тест-кейса: только проверка и ожидаемый результат, без детальных шагов.

### Шаблон чек-листа

```
| # | Проверка | Ожидаемый результат | Статус |
|---|----------|---------------------|--------|
| 1 | Описание проверки | Что должно произойти | Pass/Fail |
```

### Пример выполнения (первые 5 пунктов — остальные допишите сами)

| # | Проверка | Ожидаемый результат | Статус |
|---|----------|---------------------|--------|
| 1 | Добавить 1 товар в корзину | Badge на иконке корзины показывает «1» | |
| 2 | Добавить все 6 товаров в корзину | Badge показывает «6», все кнопки стали «Remove» | |
| 3 | Удалить товар из каталога (кнопка Remove) | Badge уменьшается на 1, кнопка снова «Add to cart» | |
| 4 | Открыть корзину (клик на иконку) | Отображаются все добавленные товары с правильными ценами | |
| 5 | В корзине: название товара кликабельно | Клик на название → переход на страницу товара | |

### Требования к вашему чек-листу
- **Минимум 20 пунктов**
- Покрыть сценарии:
  - Добавление товаров (1, несколько, все)
  - Удаление товаров (из каталога, из корзины, из страницы товара)
  - Отображение данных в корзине (названия, цены, количество)
  - Навигация (Continue Shopping, Checkout)
  - Пустая корзина
  - Сохранение корзины после logout/login (если применимо)
  - Корзина при использовании `problem_user`
  - Корзина при использовании `error_user`

### Deliverable
Файл `shopping-cart-checklist.md` с таблицей из 20+ пунктов.

## Задание 4: Тест-кейсы для API

**Приложение**: https://reqres.in

**Задача**: написать 5 тест-кейсов для API в адаптированном формате.

### Шаблон тест-кейса для API

```
ID:              [TC-API-001]
Название:        [описание проверки]
Метод:           [GET / POST / PUT / PATCH / DELETE]
URL:             [полный URL эндпоинта]
Headers:         [заголовки запроса]
Request Body:    [тело запроса в JSON]
Ожидаемый код:   [HTTP status code]
Ожидаемое тело:  [ключевые поля ответа]
```

### Пример

```
ID:              TC-API-001
Название:        Получение списка пользователей (страница 1)
Метод:           GET
URL:             https://reqres.in/api/users?page=1
Headers:         Content-Type: application/json
Request Body:    — (нет тела для GET)
Ожидаемый код:   200 OK
Ожидаемое тело:
  {
    "page": 1,
    "per_page": 6,
    "total": 12,
    "total_pages": 2,
    "data": [массив из 6 объектов с полями id, email, first_name, last_name, avatar]
  }
```

### Deliverable
Файл `api-test-cases.md` с 5 тест-кейсами: GET (список), GET (один пользователь),
POST (создание), PUT (обновление), DELETE (удаление).

## Критерии качества тест-кейсов

Используйте этот чек-лист при самопроверке каждого тест-кейса:

| # | Критерий | Вопрос для проверки |
|---|----------|---------------------|
| 1 | Атомарность | Тест-кейс проверяет ОДНУ вещь? |
| 2 | Воспроизводимость | Другой человек сможет выполнить тест по этим шагам? |
| 3 | Конкретность | Тестовые данные указаны явно, а не «ввести валидные данные»? |
| 4 | Проверяемость | Ожидаемый результат можно однозначно проверить (не «работает», а «отображается текст X»)? |
| 5 | Независимость | Тест не зависит от результата другого теста? |
| 6 | Полнота | Есть предусловия, шаги и ожидаемый результат? |
| 7 | Краткость | Шаги не избыточны (не «кликнуть на поле, ввести текст» — а «ввести X в поле Y»)? |
| 8 | Трассируемость | Тест-кейс связан с требованием или user story? |

## Типичные ошибки при написании тест-кейсов

### Ошибка 1: Несколько проверок в одном тест-кейсе
**Плохо**: «Проверить логин, добавление в корзину и оформление заказа»
**Хорошо**: Три отдельных тест-кейса для каждого действия.

**Почему важно**: если тест падает на шаге логина, вы не узнаете, работает ли корзина.

### Ошибка 2: Абстрактные тестовые данные
**Плохо**: «Ввести валидный email»
**Хорошо**: «Ввести test@example.com»

**Почему важно**: «валидный» — понятие субъективное. Для одного это `a@b.com`,
для другого — `firstname.lastname@company.co.uk`.

### Ошибка 3: Отсутствие предусловий
**Плохо**: (предусловий нет) «Шаг 1: Нажать кнопку Checkout»
**Хорошо**: «Предусловия: пользователь авторизован, в корзине минимум 1 товар»

**Почему важно**: без предусловий непонятно, в каком состоянии должна быть система.

### Ошибка 4: Размытый ожидаемый результат
**Плохо**: «Страница загрузилась корректно»
**Хорошо**: «Отображается заголовок "Products", список содержит 6 товаров,
иконка корзины видна в правом верхнем углу»

**Почему важно**: «корректно» — не проверяемый критерий. Что конкретно проверять?

### Ошибка 5: Избыточная детализация
**Плохо**: «1. Открыть браузер. 2. В адресную строку ввести URL. 3. Нажать Enter.
4. Дождаться загрузки. 5. Найти поле. 6. Кликнуть на поле. 7. Ввести текст...»
**Хорошо**: «1. Открыть https://www.saucedemo.com. 2. Ввести "standard_user" в поле Username»

**Почему важно**: избыточность мешает читать и поддерживать тест-кейсы.

## Организация тест-кейсов

### Именование
```
TC-[МОДУЛЬ]-[НОМЕР]

Примеры:
TC-AUTH-001  — Авторизация, тест #1
TC-CART-003  — Корзина, тест #3
TC-CHK-007   — Checkout, тест #7
TC-SRCH-002  — Поиск, тест #2
TC-API-015   — API, тест #15
```

### Группировка по модулям
```
test-cases/
├── auth/           # TC-AUTH-001 ... TC-AUTH-015
├── catalog/        # TC-CAT-001 ... TC-CAT-010
├── cart/           # TC-CART-001 ... TC-CART-020
├── checkout/       # TC-CHK-001 ... TC-CHK-010
├── search/         # TC-SRCH-001 ... TC-SRCH-007
├── profile/        # TC-PROF-001 ... TC-PROF-005
└── api/            # TC-API-001 ... TC-API-020
```

### Приоритизация
| Приоритет | Описание | Примеры |
|-----------|----------|---------|
| High | Критический функционал, блокирует основной flow | Логин, добавление в корзину, оплата |
| Medium | Важный функционал, но есть workaround | Сортировка, фильтрация, редактирование профиля |
| Low | Косметика, удобство, edge cases | Тултипы, иконки, очень длинные строки |
