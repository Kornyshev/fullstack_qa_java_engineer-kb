# Упражнения по техникам тест-дизайна

## Введение

Техники тест-дизайна позволяют системно выбирать тестовые данные вместо случайного перебора.
Вместо «попробую 100 значений наугад» — «выберу 10 значений, которые покроют все ситуации».
В этом разделе — конкретные упражнения с полными решениями.

---

## 1. Equivalence Partitioning (EP) — Эквивалентное разбиение

### Принцип
Разделить все возможные входные данные на классы эквивалентности. Из каждого класса
достаточно взять одно значение — все значения внутри класса обрабатываются одинаково.

### Упражнение 1.1: Поле «Возраст» (1-120)

**Условие**: форма регистрации содержит поле «Возраст». Допустимые значения: целые числа от 1 до 120.

**Задача**: определить классы эквивалентности и выбрать тестовые значения.

#### Решение

| # | Класс эквивалентности | Тип | Пример значения | Ожидаемый результат |
|---|----------------------|-----|-----------------|---------------------|
| 1 | Число от 1 до 120 | Валидный | 25 | Принято |
| 2 | Число меньше 1 (отрицательные) | Невалидный | -5 | Ошибка валидации |
| 3 | Число равно 0 | Невалидный | 0 | Ошибка валидации |
| 4 | Число больше 120 | Невалидный | 150 | Ошибка валидации |
| 5 | Дробное число | Невалидный | 25.5 | Ошибка валидации |
| 6 | Буквы | Невалидный | abc | Ошибка валидации |
| 7 | Спецсимволы | Невалидный | @#$ | Ошибка валидации |
| 8 | Пустое значение | Невалидный | (пусто) | Ошибка валидации |
| 9 | Пробелы | Невалидный | "   " | Ошибка валидации |
| 10 | Очень большое число | Невалидный | 999999999 | Ошибка валидации |

**Минимальный набор тестов**: 10 тестов (по одному из каждого класса).

значение НА границе, значение ПЕРЕД границей (граница - 1) и ПОСЛЕ границы (граница + 1).

## 3. Decision Table — Таблица решений

### Принцип
Когда результат зависит от комбинации условий, используется таблица решений.
Каждый столбец — уникальная комбинация условий и соответствующий результат.

### Упражнение 3.1: Система скидок интернет-магазина

**Условие**: интернет-магазин имеет систему скидок:
- **VIP-клиент**: скидка 10%
- **Промокод**: скидка 5%
- **Праздничный день**: скидка 15%
- Скидки суммируются, но максимальная скидка — 25%
- Если сумма заказа > 10000, дополнительная скидка 3%

**Задача**: построить Decision Table.

#### Решение

**Условия**:
- C1: VIP-клиент (Да/Нет)
- C2: Промокод (Да/Нет)
- C3: Праздничный день (Да/Нет)
- C4: Сумма > 10000 (Да/Нет)

**Полная таблица решений** (2^4 = 16 комбинаций):

| Правило | C1 VIP | C2 Promo | C3 Holiday | C4 >10K | Расчёт | Итог |
|---------|--------|----------|------------|---------|--------|------|
| R1 | Нет | Нет | Нет | Нет | 0% | 0% |
| R2 | Нет | Нет | Нет | Да | 0% + 3% | 3% |
| R3 | Нет | Нет | Да | Нет | 15% | 15% |
| R4 | Нет | Нет | Да | Да | 15% + 3% | 18% |
| R5 | Нет | Да | Нет | Нет | 5% | 5% |
| R6 | Нет | Да | Нет | Да | 5% + 3% | 8% |
| R7 | Нет | Да | Да | Нет | 5% + 15% = 20% | 20% |
| R8 | Нет | Да | Да | Да | 20% + 3% = 23% | 23% |
| R9 | Да | Нет | Нет | Нет | 10% | 10% |
| R10 | Да | Нет | Нет | Да | 10% + 3% | 13% |
| R11 | Да | Нет | Да | Нет | 10% + 15% = 25% | 25% (макс) |
| R12 | Да | Нет | Да | Да | 25% + 3% = 28% → 25% | 25% (макс) |
| R13 | Да | Да | Нет | Нет | 10% + 5% = 15% | 15% |
| R14 | Да | Да | Нет | Да | 15% + 3% = 18% | 18% |
| R15 | Да | Да | Да | Нет | 10% + 5% + 15% = 30% → 25% | 25% (макс) |
| R16 | Да | Да | Да | Да | 30% + 3% = 33% → 25% | 25% (макс) |

**Тест-кейсы**: по одному тест-кейсу на каждое правило (16 тест-кейсов).

**Обратите внимание**: правила R11, R12, R15, R16 все дают 25% — это максимальная скидка.
Можно оптимизировать: объединить правила с одинаковым результатом. Но для полноты тестирования
лучше проверить все комбинации, чтобы убедиться, что ограничение в 25% работает во всех случаях.

## 4. State Transition — Диаграмма переходов состояний

### Принцип
Когда система имеет конечное количество состояний и переходы между ними зависят от событий,
используется State Transition диаграмма.

### Упражнение 4.1: Статусы заказа в интернет-магазине

**Условие**: заказ в интернет-магазине проходит через следующие состояния:
1. **Создан** (Created) — заказ оформлен
2. **Оплачен** (Paid) — оплата прошла успешно
3. **В обработке** (Processing) — собирается на складе
4. **Отправлен** (Shipped) — передан курьеру
5. **Доставлен** (Delivered) — получен покупателем
6. **Отменён** (Cancelled) — отменён
7. **Возврат** (Returned) — товар возвращён

**Переходы**:
```
Создан → Оплачен         (событие: успешная оплата)
Создан → Отменён          (событие: отмена покупателем или таймаут оплаты)
Оплачен → В обработке     (событие: начало сборки)
Оплачен → Отменён         (событие: отмена покупателем до начала сборки)
В обработке → Отправлен   (событие: передача курьеру)
В обработке → Отменён     (событие: товара нет на складе)
Отправлен → Доставлен     (событие: подтверждение получения)
Доставлен → Возврат       (событие: запрос на возврат в течение 14 дней)
```

**Задача**: построить State Transition Table и написать тест-кейсы.

#### Решение: State Transition Table

| Текущее состояние | Событие | Следующее состояние | Условия/Примечания |
|-------------------|---------|---------------------|-------------------|
| Создан | Успешная оплата | Оплачен | Деньги списаны с карты |
| Создан | Отмена покупателем | Отменён | — |
| Создан | Таймаут 30 минут | Отменён | Автоматическая отмена |
| Оплачен | Начало сборки | В обработке | Оператор взял в работу |
| Оплачен | Отмена покупателем | Отменён | Возврат средств |
| В обработке | Передача курьеру | Отправлен | Трек-номер присвоен |
| В обработке | Товара нет | Отменён | Возврат средств |
| Отправлен | Подтверждение получения | Доставлен | Подпись получателя |
| Доставлен | Запрос возврата | Возврат | В течение 14 дней |

#### Тест-кейсы для позитивных переходов (happy path)

```
TC-ST-001: Создан → Оплачен
  Предусловия: Заказ в статусе "Создан"
  Действие: Оплатить заказ
  Ожидаемый результат: Статус изменился на "Оплачен"

TC-ST-002: Оплачен → В обработке
  Предусловия: Заказ в статусе "Оплачен"
  Действие: Оператор берёт заказ в работу
  Ожидаемый результат: Статус изменился на "В обработке"

TC-ST-003: В обработке → Отправлен
  Предусловия: Заказ в статусе "В обработке"
  Действие: Передача курьеру
  Ожидаемый результат: Статус "Отправлен", присвоен трек-номер

TC-ST-004: Отправлен → Доставлен
  Предусловия: Заказ в статусе "Отправлен"
  Действие: Покупатель подтверждает получение
  Ожидаемый результат: Статус "Доставлен"

TC-ST-005: Полный happy path
  Создан → Оплачен → В обработке → Отправлен → Доставлен
```

#### Тест-кейсы для негативных переходов (невозможные)

```
TC-ST-006: Попытка перевести "Отменён" → "Оплачен"
  Предусловия: Заказ в статусе "Отменён"
  Действие: Попытка оплатить
  Ожидаемый результат: Ошибка — невозможный переход

TC-ST-007: Попытка перевести "Доставлен" → "В обработке"
  Ожидаемый результат: Ошибка — невозможный переход

TC-ST-008: Попытка возврата через 15 дней
  Предусловия: Заказ "Доставлен" 15 дней назад
  Ожидаемый результат: Ошибка — срок возврата истёк
```

**Визуализация** (текстовая диаграмма):
```
[Создан] --оплата--> [Оплачен] --сборка--> [В обработке] --курьер--> [Отправлен] --получен--> [Доставлен]
    |                     |                       |                                                 |
    |--отмена-->     |--отмена-->         |--нет товара-->                                   |--возврат-->
    v                     v                       v                                                 v
[Отменён]           [Отменён]             [Отменён]                                          [Возврат]
```

## 5. Pairwise Testing — Попарное тестирование

### Принцип
Когда нужно протестировать форму с множеством полей и каждое поле имеет несколько значений,
полный перебор всех комбинаций невозможен. Pairwise гарантирует, что каждая пара значений
любых двух параметров встречается хотя бы в одном тест-кейсе.

### Упражнение 5.1: Форма поиска авиабилетов

**Условие**: форма поиска имеет 4 параметра, каждый с 3 значениями:

| Параметр | Значение 1 | Значение 2 | Значение 3 |
|----------|-----------|-----------|-----------|
| Класс (Class) | Эконом | Бизнес | Первый |
| Пассажиры (Passengers) | 1 | 2 | 5 |
| Багаж (Luggage) | Без багажа | 1 место | 2 места |
| Маршрут (Route) | В одну сторону | Туда-обратно | Мультигород |

**Полный перебор**: 3 × 3 × 3 × 3 = 81 комбинация.
**Pairwise**: покроем все пары значительно меньшим числом тестов.

#### Ручное решение (без инструмента)

Для каждой пары параметров нужно покрыть все 9 комбинаций (3×3).
Количество пар: C(4,2) = 6 пар.

| # | Class | Passengers | Luggage | Route |
|---|-------|-----------|---------|-------|
| 1 | Эконом | 1 | Без багажа | В одну сторону |
| 2 | Эконом | 2 | 1 место | Туда-обратно |
| 3 | Эконом | 5 | 2 места | Мультигород |
| 4 | Бизнес | 1 | 1 место | Мультигород |
| 5 | Бизнес | 2 | 2 места | В одну сторону |
| 6 | Бизнес | 5 | Без багажа | Туда-обратно |
| 7 | Первый | 1 | 2 места | Туда-обратно |
| 8 | Первый | 2 | Без багажа | Мультигород |
| 9 | Первый | 5 | 1 место | В одну сторону |

**9 тестов вместо 81** — покрытие всех пар. Проверим:
- Class × Passengers: (Э,1), (Э,2), (Э,5), (Б,1), (Б,2), (Б,5), (П,1), (П,2), (П,5) = 9/9
- Class × Luggage: (Э,Без), (Э,1), (Э,2), (Б,1), (Б,2), (Б,Без), (П,2), (П,Без), (П,1) = 9/9
- Все пары покрыты.

### Упражнение 5.2: Использование PICT (Microsoft)

**PICT** — бесплатный инструмент от Microsoft для генерации pairwise-комбинаций.

**Установка**:
- Windows: скачать с https://github.com/microsoft/pict
- macOS: `brew install pict`
- Linux: собрать из исходников

**Задача**: создать входной файл для PICT и сгенерировать тест-кейсы.

## 6. Комбинированное упражнение

### Задание: Тестирование поля «Номер телефона»

**Условие**: поле ввода номера телефона. Требования:
- Формат: +7 (XXX) XXX-XX-XX
- Допустимые символы: цифры, +, (, ), -, пробел
- Минимум 11 цифр (с кодом страны)
- Максимум 15 цифр (международный стандарт E.164)
- Код страны: +7 (Россия) или +375 (Беларусь) или +380 (Украина)

**Задача**: применить EP, BVA, Decision Table.

#### EP (классы эквивалентности)

| # | Класс | Пример | Тип |
|---|-------|--------|-----|
| 1 | Валидный российский | +7 (999) 123-45-67 | Валидный |
| 2 | Валидный белорусский | +375 (29) 123-45-67 | Валидный |
| 3 | Валидный украинский | +380 (50) 123-45-67 | Валидный |
| 4 | Недопустимый код страны | +1 (999) 123-45-67 | Невалидный |
| 5 | Без кода страны | (999) 123-45-67 | Невалидный |
| 6 | Буквы в номере | +7 (ABC) 123-45-67 | Невалидный |
| 7 | Пустое значение | (пусто) | Невалидный |
| 8 | Только цифры без форматирования | 79991234567 | Зависит от требований |

#### BVA (граничные значения для количества цифр)

| # | Количество цифр | Пример | Ожидаемый результат |
|---|----------------|--------|---------------------|
| 1 | 10 цифр | +7 999 123-45-6 | Ошибка (мало) |
| 2 | 11 цифр | +7 999 123-45-67 | Принято (минимум) |
| 3 | 12 цифр | +375 29 123-45-67 | Принято |
| 4 | 15 цифр | +7 999 123-45-67-890 | Принято (максимум) |
| 5 | 16 цифр | +7 999 123-45-67-8901 | Ошибка (много) |

#### Decision Table (код страны + длина + формат)

| Правило | Код страны | Длина (цифры) | Формат | Результат |
|---------|-----------|---------------|--------|-----------|
| R1 | +7 | 11 | Верный | Принято |
| R2 | +7 | 11 | Неверный | Зависит от настроек (автоформат?) |
| R3 | +7 | <11 | Верный | Ошибка: мало цифр |
| R4 | +7 | >15 | Верный | Ошибка: много цифр |
| R5 | Другой | 11 | Верный | Ошибка: недопустимый код |
| R6 | Нет | 11 | — | Ошибка: нет кода страны |
