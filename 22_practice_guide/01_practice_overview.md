# Общий план практики — дорожная карта QA Engineer

## Введение

Этот документ — практическая дорожная карта для становления QA-инженером. Здесь нет абстрактных
советов в духе «изучите тестирование». Здесь — конкретные шаги, конкретные сроки, конкретные
результаты, которые можно показать на собеседовании.

Главный принцип: **каждый этап заканчивается артефактом в вашем GitHub-портфолио**. Не «я прочитал
книгу», а «вот репозиторий с моими тест-кейсами / автотестами / CI-пайплайном».

---

## Порядок освоения навыков

Порядок ниже — не случайный. Каждый следующий этап опирается на предыдущий. Перепрыгивание
через этапы — самая частая ошибка начинающих.

```
Manual Testing → API (Postman) → API Automation → UI Automation → Framework Design → CI/CD
     ↓               ↓                ↓                ↓               ↓              ↓
  4-6 недель      2-3 недели       3-4 недели       4-6 недель      3-4 недели     2-3 недели
```

### Общая продолжительность: 18-26 недель (4.5-6.5 месяцев)

Это при условии 2-3 часов ежедневной практики. При меньшей интенсивности — пропорционально больше.

---

## Этап 1: Ручное тестирование (4-6 недель)

### Цель
Научиться находить баги, описывать их и структурировать проверки.

### Что делать
1. **Неделя 1-2**: Тестирование открытых веб-приложений (SauceDemo, DemoQA, The Internet)
2. **Неделя 2-3**: Написание тест-кейсов и чек-листов
3. **Неделя 3-4**: Написание баг-репортов по шаблону
4. **Неделя 4-5**: Техники тест-дизайна (EP, BVA, Decision Table, State Transition, Pairwise)
5. **Неделя 5-6**: Exploratory testing — сессионный подход

### Артефакт для портфолио
- GitHub-репозиторий `manual-testing-portfolio` со структурой:
```
manual-testing-portfolio/
├── test-cases/
│   ├── saucedemo-login.md
│   ├── saucedemo-cart.md
│   └── demoqa-forms.md
├── bug-reports/
│   ├── saucedemo-bugs.md
│   └── the-internet-bugs.md
├── checklists/
│   ├── checkout-checklist.md
│   └── registration-checklist.md
├── test-design/
│   ├── equivalence-partitioning.md
│   ├── boundary-values.md
│   └── decision-table.md
└── exploratory/
    └── saucedemo-session-report.md
```

### Конкретное задание для завершения этапа
- Написать **минимум 30 тест-кейсов** для SauceDemo (логин, каталог, корзина, оформление заказа)
- Найти и описать **минимум 10 багов** в открытых приложениях
- Составить Decision Table для системы скидок
- Провести 30-минутную exploratory-сессию и оформить отчёт

## Этап 2: API-тестирование в Postman (2-3 недели)

### Цель
Понять, как работает API, научиться тестировать его вручную и писать простые автотесты в Postman.

### Что делать
1. **Неделя 1**: Основы HTTP, REST, работа с Postman (GET/POST/PUT/DELETE на Reqres.in)
2. **Неделя 2**: Переменные, окружения, Pre-request Script, Tests в Postman
3. **Неделя 3**: Коллекции, Newman, запуск из командной строки

### Артефакт для портфолио
- GitHub-репозиторий `api-testing-postman` с экспортированными коллекциями:
```
api-testing-postman/
├── collections/
│   ├── reqres-api-tests.json
│   ├── petstore-api-tests.json
│   └── conduit-api-tests.json
├── environments/
│   ├── dev.json
│   └── staging.json
├── newman-reports/
│   └── .gitkeep
└── README.md        # описание коллекций, как запустить
```

### Конкретное задание для завершения этапа
- Создать коллекцию из **минимум 20 запросов** к Reqres.in с assertions в Tests
- Создать коллекцию для Petstore API (https://petstore.swagger.io/) — CRUD для сущности Pet
- Настроить переменные окружения и цепочку запросов (создать → получить → обновить → удалить)
- Запустить коллекцию через Newman и сохранить HTML-отчёт

## Этап 3: API Automation на Java (3-4 недели)

### Цель
Перейти от ручного API-тестирования к автоматизации на Java с использованием REST Assured.

### Что делать
1. **Неделя 1**: Основы Java (если нет опыта), Maven, JUnit 5 — структура проекта
2. **Неделя 2**: REST Assured — базовые GET/POST/PUT/DELETE тесты
3. **Неделя 3**: Спецификации, сериализация/десериализация (Jackson/Gson), POJO
4. **Неделя 4**: Allure-отчёты, параметризованные тесты, тестовые данные

### Артефакт для портфолио
- GitHub-репозиторий `api-tests-java`:
```
api-tests-java/
├── src/
│   ├── main/java/models/       # POJO-классы
│   └── test/java/
│       ├── tests/              # тестовые классы
│       ├── specs/              # RequestSpecification, ResponseSpecification
│       └── utils/              # вспомогательные методы
├── pom.xml
├── allure-results/
└── README.md
```

### Конкретное задание для завершения этапа
- Написать **минимум 15 автотестов** для Reqres.in (позитивные + негативные)
- Реализовать POJO для всех моделей API
- Использовать RequestSpecification для базового URL и заголовков
- Настроить Allure-отчёты и сгенерировать отчёт

## Этап 4: UI Automation на Java (4-6 недель)

### Цель
Освоить Selenium WebDriver / Selenide для автоматизации UI-тестов.

### Что делать
1. **Неделя 1-2**: Selenium WebDriver — локаторы, действия, ожидания
2. **Неделя 2-3**: Page Object Pattern — структуризация кода
3. **Неделя 3-4**: Selenide — упрощение работы с WebDriver
4. **Неделя 4-5**: Сложные сценарии (iframe, alerts, file upload, drag-and-drop)
5. **Неделя 5-6**: Скриншоты при падении, параллельный запуск, Allure

### Артефакт для портфолио
- GitHub-репозиторий `ui-tests-java`:
```
ui-tests-java/
├── src/
│   ├── main/java/
│   │   ├── pages/              # Page Object классы
│   │   ├── components/         # переиспользуемые компоненты
│   │   └── utils/              # вспомогательные утилиты
│   └── test/java/
│       └── tests/              # тестовые классы
├── pom.xml
└── README.md
```

### Конкретное задание для завершения этапа
- Написать **минимум 20 UI-автотестов** для SauceDemo с Page Object
- Покрыть полный flow: логин → каталог → корзина → checkout → завершение
- Реализовать тесты для The Internet (alerts, frames, drag-and-drop, file upload)
- Настроить скриншоты при падении тестов и Allure-отчёты

## Этап 5: Проектирование фреймворка (3-4 недели)

### Цель
Объединить API и UI тесты в единый фреймворк промышленного уровня.

### Что делать
1. **Неделя 1**: Архитектура фреймворка, конфигурации, профили
2. **Неделя 2**: Управление тестовыми данными, фикстуры, Builder pattern
3. **Неделя 3**: Логирование, кастомные аннотации, retry-механизм
4. **Неделя 4**: Параллельный запуск, кроссбраузерность, Docker

### Артефакт для портфолио
- GitHub-репозиторий `test-automation-framework`:
```
test-automation-framework/
├── src/
│   ├── main/java/
│   │   ├── config/             # конфигурации
│   │   ├── api/                # API-слой
│   │   ├── ui/pages/           # Page Object
│   │   ├── ui/components/      # компоненты
│   │   ├── data/               # тестовые данные, Builder
│   │   └── utils/              # утилиты
│   └── test/java/
│       ├── api/                # API-тесты
│       └── ui/                 # UI-тесты
├── docker-compose.yml          # Selenoid / Selenium Grid
├── pom.xml
└── README.md
```

### Конкретное задание для завершения этапа
- Объединить API и UI тесты в одном проекте с общей конфигурацией
- Реализовать **минимум 3 сценария**, где API используется для подготовки данных, а UI — для проверки
- Настроить запуск в Docker через Selenoid или Selenium Grid
- Написать README с описанием архитектуры и инструкцией по запуску

## Этап 6: CI/CD (2-3 недели)

### Цель
Интегрировать тесты в CI/CD-пайплайн — финальный штрих портфолио.

### Что делать
1. **Неделя 1**: GitHub Actions — запуск тестов по push/PR
2. **Неделя 2**: Allure-отчёты в GitHub Pages, Telegram-уведомления
3. **Неделя 3**: Jenkins (опционально) — pipeline as code

### Артефакт для портфолио
- `.github/workflows/tests.yml` в репозитории фреймворка
- Allure-отчёт, опубликованный на GitHub Pages
- Badge в README с результатом последнего запуска

### Конкретное задание для завершения этапа
- Настроить GitHub Actions workflow, который запускает тесты при каждом push
- Опубликовать Allure-отчёт на GitHub Pages
- Добавить badge со статусом в README
- (Бонус) Настроить Telegram-уведомление о результатах прогона

## Использование GitHub как портфолио

### Структура GitHub-профиля
```
github.com/your-username/
├── manual-testing-portfolio      # этап 1
├── api-testing-postman           # этап 2
├── api-tests-java                # этап 3
├── ui-tests-java                 # этап 4
├── test-automation-framework     # этап 5 + 6
└── your-username                 # Profile README
```

### Правила оформления репозиториев
1. **Каждый репозиторий имеет README** с описанием: что тестируется, какие технологии, как запустить
2. **Осмысленные коммиты**: не «fix», а «добавить тест-кейсы для негативных сценариев логина»
3. **Ветки и PR**: показывайте, что умеете работать с Git — feature-ветки, pull requests
4. **Регулярные коммиты**: лучше по 1-2 коммита в день, чем 50 коммитов раз в месяц
5. **Нет секретов в коде**: пароли, ключи — только через environment variables

### Profile README (файл в репозитории your-username/your-username)
```markdown
# Привет! Я QA Engineer

## Мой стек
- Manual Testing: тест-кейсы, чек-листы, баг-репорты
- API Testing: Postman, REST Assured, Java
- UI Testing: Selenium, Selenide, Page Object
- CI/CD: GitHub Actions, Docker, Allure

## Мои проекты
- [Test Automation Framework](ссылка) — фреймворк для API + UI тестов
- [API Tests](ссылка) — автотесты REST API на Java
- [Manual Testing Portfolio](ссылка) — тест-кейсы и баг-репорты
```

## Что показывать на собеседованиях

### Junior QA (Manual)
- Портфолио с тест-кейсами и баг-репортами
- Понимание техник тест-дизайна (EP, BVA, Decision Table)
- Умение объяснить, почему выбрали именно эти проверки
- Живая демонстрация: «вот приложение, покажите, как бы вы его тестировали»

### Junior QA Automation
- Работающий фреймворк на GitHub с зелёными тестами
- Понимание Page Object Pattern — зачем, как, когда
- CI/CD pipeline — тесты запускаются автоматически
- Allure-отчёт с историей прогонов
- Умение объяснить архитектурные решения: «почему REST Assured, а не HttpClient?»

### Что НЕ нужно показывать
- Курсовые сертификаты без практических работ
- Репозиторий с единственным коммитом «initial commit» месячной давности
- Тесты, скопированные из туториалов без изменений
- Фреймворк без README и с непонятной структурой

## Типичные ошибки и как их избежать

### Ошибка 1: Прыжок в автоматизацию без понимания тестирования
**Симптом**: человек пишет автотесты, но не понимает, что именно проверяет и зачем.

**Пример**: написано 50 автотестов для формы логина, но все проверяют только happy path.
Ни одного теста на пустые поля, SQL-инъекции, XSS, граничные значения длины пароля.

**Решение**: пройдите этап ручного тестирования полностью. Автоматизация — это инструмент
исполнения тестов. Сначала нужно научиться проектировать тесты.

### Ошибка 2: Бесконечная теория без практики
**Симптом**: прочитано 5 книг по тестированию, пройдено 3 курса, но ни одного тест-кейса
не написано, ни одного бага не найдено.

**Пример**: человек знает определение Equivalence Partitioning, но не может применить
его к конкретному полю ввода.

**Решение**: правило 20/80 — 20% теории, 80% практики. Прочитали про технику — сразу
применяйте на реальном приложении.

### Ошибка 3: Идеальный фреймворк с нуля
**Симптом**: три месяца проектирования «идеальной архитектуры», ни одного работающего теста.

**Пример**: создан проект с 15 модулями Maven, кастомными аннотациями, абстрактными фабриками —
но тестов ноль.

**Решение**: начните с простого. Один тестовый класс, один Page Object. Рефакторинг — по мере роста.
Работающий тест важнее идеальной архитектуры.

### Ошибка 4: Игнорирование Git
**Симптом**: весь код на локальной машине, без версионного контроля.

**Пример**: «у меня был фреймворк, но я переустановил систему и всё потерял».

**Решение**: с первого дня — Git + GitHub. Каждый этап = отдельный репозиторий.
Коммиты каждый день. Это и бэкап, и портфолио, и привычка.

### Ошибка 5: Копирование без понимания
**Симптом**: код работает, но человек не может объяснить, что делает каждая строка.

**Пример**: скопировал WebDriverWait из StackOverflow, тест проходит, но на вопрос
«чем explicit wait отличается от implicit?» — молчание.

**Решение**: если скопировали код — разберите каждую строку. Измените параметры,
сломайте, почините. Только так приходит понимание.

### Ошибка 6: Тестирование только happy path
**Симптом**: все тесты — позитивные сценарии. Негативных — ноль.

**Пример**: тест логина проверяет только «правильный логин + правильный пароль = успех».
Нет тестов на пустые поля, неверный пароль, заблокированного пользователя, SQL-инъекции.

**Решение**: для каждого позитивного сценария — минимум 3 негативных. Используйте EP и BVA
для системного подхода к негативному тестированию.

## Итого: ваш план на 6 месяцев

```
Месяц 1:   Ручное тестирование — тест-кейсы, баг-репорты, тест-дизайн
Месяц 2:   API в Postman + начало Java
Месяц 3:   REST Assured — API-автотесты
Месяц 4:   Selenium/Selenide — UI-автотесты
Месяц 5:   Фреймворк — объединение API + UI
Месяц 6:   CI/CD + полировка портфолио + подготовка к собеседованиям
```

Каждый месяц заканчивается артефактом на GitHub. Через 6 месяцев у вас — портфолио
из 5 репозиториев, которое покрывает весь стек Junior/Middle QA Automation Engineer.
