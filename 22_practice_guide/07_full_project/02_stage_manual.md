# Этап 1: Ручное тестирование

## Введение

Перед автоматизацией необходимо провести ручное тестирование: составить тест-план,
написать тест-кейсы, создать чек-листы и зафиксировать найденные дефекты.
Без этого этапа автоматизация будет хаотичной — вы будете автоматизировать то,
что не нужно, и пропускать критические сценарии.

> **Цель:** Создать полный комплект тестовой документации для приложения Conduit,
> который станет основой для автоматизации на следующих этапах.

---

## Часть 1: Тест-план

### Шаблон тест-плана

```markdown
# Тест-план: Conduit Blog Platform

## 1. Введение
Данный тест-план описывает стратегию тестирования блог-платформы Conduit.
Приложение предоставляет функциональность публикации статей, комментирования,
подписок и управления профилем.

## 2. Объём тестирования

### В объёме (In Scope)
- Регистрация и авторизация пользователей
- CRUD-операции со статьями
- Управление комментариями
- Подписки на авторов
- Добавление статей в избранное
- Лента статей (глобальная и персональная)
- Управление тегами
- Редактирование профиля
- REST API (все эндпоинты)

### Вне объёма (Out of Scope)
- Нагрузочное тестирование
- Тестирование безопасности (penetration testing)
- Мобильная версия (responsive)
- Интернационализация (i18n)

## 3. Типы тестирования
- Функциональное тестирование (UI + API)
- Интеграционное тестирование (API + DB)
- Smoke-тестирование (после каждого деплоя)
- Регрессионное тестирование (еженедельно)

## 4. Окружения
| Окружение | URL | База данных |
|-----------|-----|------------|
| Local     | http://localhost:3000 | PostgreSQL (Docker) |
| Staging   | https://staging.conduit.io | PostgreSQL (RDS) |

## 5. Критерии входа и выхода

### Критерии входа
- Приложение развёрнуто и доступно
- Тестовые данные подготовлены
- API-документация актуальна

### Критерии выхода
- Все критические тест-кейсы выполнены
- Нет открытых blocker/critical дефектов
- Smoke-тесты проходят на 100%

## 6. Риски
- Нестабильность тестового окружения
- Изменение API без обновления документации
- Зависимость от внешних сервисов

## 7. Расписание
| Активность | Начало | Окончание |
|-----------|--------|-----------|
| Анализ требований | День 1 | День 2 |
| Написание тест-кейсов | День 2 | День 5 |
| Выполнение тестов | День 5 | День 7 |
| Фиксация дефектов | День 5 | День 7 |
```

## Часть 2: Тест-кейсы по модулям

### Модуль: Регистрация (Registration)

#### TC-REG-001: Успешная регистрация с валидными данными

```
ID:           TC-REG-001
Приоритет:    Critical
Модуль:       Регистрация
Предусловие:  Открыта страница регистрации (/register)

Шаги:
1. Ввести уникальное имя пользователя (например, "testuser123")
2. Ввести валидный email (например, "testuser123@test.com")
3. Ввести пароль (минимум 8 символов, например, "Password123")
4. Нажать кнопку "Sign up"

Ожидаемый результат:
- Пользователь перенаправлен на главную страницу
- В навигационном меню отображается имя пользователя
- В навигации доступны пункты: "New Article", "Settings", username
- Пункты "Sign in" и "Sign up" отсутствуют

Постусловие:
- Пользователь зарегистрирован и авторизован
```

#### TC-REG-002: Регистрация с уже существующим email

```
ID:           TC-REG-002
Приоритет:    High
Модуль:       Регистрация
Предусловие:  Пользователь с email "existing@test.com" уже зарегистрирован

Шаги:
1. Открыть страницу регистрации
2. Ввести уникальное имя пользователя
3. Ввести email "existing@test.com"
4. Ввести валидный пароль
5. Нажать кнопку "Sign up"

Ожидаемый результат:
- Отображается сообщение об ошибке "email has already been taken"
- Пользователь остаётся на странице регистрации
- Регистрация не выполнена
```

#### TC-REG-003: Регистрация с пустыми обязательными полями

```

  - "username can't be blank"
```

## Часть 4: Шаблон баг-репорта

### Шаблон

```
ID:            BUG-XXX
Заголовок:     [Модуль] Краткое описание бага
Серьёзность:   Blocker / Critical / Major / Minor / Trivial
Приоритет:     High / Medium / Low
Компонент:     UI / API / Database
Окружение:     Browser: Chrome 125, OS: macOS 14.5
Версия:        Conduit v1.0.0

Предусловие:
(Описание состояния системы перед выполнением шагов)

Шаги воспроизведения:
1. Шаг первый
2. Шаг второй
3. Шаг третий

Фактический результат:
(Что произошло)

Ожидаемый результат:
(Что должно было произойти)

Вложения:
- Скриншот
- Видео
- Логи (консоль браузера / сервер)
```

### Пример заполненного баг-репорта

```
ID:            BUG-001
Заголовок:     [Регистрация] Сервер возвращает 500 при регистрации с email без домена
Серьёзность:   Major
Приоритет:     High
Компонент:     API
Окружение:     API v1.0.0, PostgreSQL 15

Предусловие:
Приложение развёрнуто и доступно.

Шаги воспроизведения:
1. Отправить POST /api/users с телом:
   {
     "user": {
       "username": "testuser",
       "email": "invalid-email",
       "password": "Password123"
     }
   }

Фактический результат:
- HTTP 500 Internal Server Error
- Тело ответа: {"errors":{"message":"Internal Server Error"}}
- В логах сервера: "Unhandled exception: invalid email format"

Ожидаемый результат:
- HTTP 422 Unprocessable Entity
- Тело ответа: {"errors":{"email":["is invalid"]}}
- Валидация на стороне сервера должна возвращать понятное сообщение об ошибке,
  а не Internal Server Error

Вложения:
- screenshot_500_error.png
- server_log_excerpt.txt
```

## Практическое задание

### Задание 1: Тест-план

1. Составьте тест-план по шаблону, приведённому выше
2. Адаптируйте его под ваше конкретное окружение
3. Определите приоритеты модулей для тестирования
4. Согласуйте критерии входа и выхода

**Ожидаемый результат:** Документ тест-плана в Google Docs или Confluence.

### Задание 2: Тест-кейсы

1. Напишите минимум 50 тест-кейсов по шаблону выше
2. Распределите по модулям: Registration (8+), Login (5+), Articles (15+),
   Comments (5+), Follow/Favorite (5+), Profile (5+), API (7+)
3. Каждый тест-кейс должен иметь уникальный ID, приоритет и чёткие шаги

**Ожидаемый результат:** Тест-кейсы в TestRail, Zephyr, Google Sheets или Excel.

### Задание 3: Выполнение тестов и баг-репорты

1. Выполните все smoke-тесты из чек-листа
2. Выполните минимум 30 тест-кейсов вручную
3. Зафиксируйте все найденные дефекты по шаблону баг-репорта
4. Создайте матрицу трассируемости

**Критерии оценки:**
- Тест-план содержит все обязательные разделы
- Тест-кейсы покрывают позитивные и негативные сценарии
- Каждый тест-кейс имеет чёткие шаги и ожидаемый результат
- Баг-репорты содержат достаточно информации для воспроизведения
- Smoke чек-лист содержит минимум 10 проверок
