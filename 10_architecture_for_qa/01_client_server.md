# Клиент-серверная архитектура

## Обзор

Клиент-серверная архитектура — это фундаментальная модель взаимодействия в современных веб-приложениях.
Клиент (browser, mobile app, desktop app) отправляет запросы серверу, сервер обрабатывает их и возвращает ответ.
Для QA-инженера понимание этой архитектуры критически важно: оно определяет, **где** именно искать баги,
**какой** уровень тестировать и **как** правильно локализовать дефекты.

---

## Frontend vs Backend

### Frontend (клиентская часть)

Frontend — это всё, что пользователь видит и с чем взаимодействует в браузере или приложении.

**Основные технологии:**
- **HTML** — структура страницы
- **CSS** — стилизация и визуальное оформление
- **JavaScript** — логика на стороне клиента
- **Frameworks:** React, Angular, Vue.js

**Что происходит на frontend:**
- Рендеринг UI-компонентов
- Валидация форм на стороне клиента
- Управление состоянием приложения (state management)
- Отправка HTTP-запросов к backend (через `fetch`, `axios`)
- Обработка ответов и отображение данных

### Backend (серверная часть)

Backend — это серверная логика, невидимая пользователю, но выполняющая основную работу.

**Основные технологии:**
- **Java** (Spring Boot), **Python** (Django, Flask), **Node.js** (Express), **C#** (.NET)
- **Базы данных:** PostgreSQL, MySQL, MongoDB
- **Кэширование:** Redis, Memcached

**Что происходит на backend:**
- Обработка бизнес-логики
- Аутентификация и авторизация
- Валидация данных (серверная)
- Взаимодействие с базой данных (CRUD-операции)
- Интеграция со сторонними сервисами

### Ключевое различие для QA

| Аспект | Frontend | Backend |
|--------|----------|---------|
| Тип багов | Визуальные, UX, кросс-браузерные | Логические, безопасности, производительности |
| Инструменты тестирования | Selenium, Cypress, Playwright | RestAssured, Postman, JUnit |
| Валидация данных | Клиентская (можно обойти) | Серверная (надёжная) |
| Скорость тестов | Медленные (UI) | Быстрые (API) |

## SPA vs MPA

### SPA (Single Page Application)

SPA — приложение, которое загружается один раз, а затем динамически обновляет содержимое страницы
без полной перезагрузки.

**Как работает:**
```
1. Браузер загружает index.html + JavaScript bundle
2. JavaScript рендерит UI на стороне клиента
3. При навигации — AJAX-запросы к API, обновление DOM
4. URL меняется через History API (без перезагрузки)
```

**Примеры:** Gmail, Trello, Figma

**Особенности тестирования SPA:**
- Элементы появляются динамически — нужны **explicit waits**
- Навигация без перезагрузки — нельзя полагаться на `page load`
- Состояние хранится в памяти — важно тестировать **state management**
- Роутинг на клиенте — тестировать deep links и обновление страницы

### MPA (Multi Page Application)

MPA — классическое приложение, где каждая страница загружается с сервера целиком.

**Как работает:**
```
1. Браузер запрашивает URL
2. Сервер генерирует полный HTML (server-side rendering)
3. Браузер отображает страницу
4. При навигации — новый запрос, полная перезагрузка
```

**Примеры:** Amazon, Wikipedia, классические интернет-магазины

**Особенности тестирования MPA:**
- Каждая страница — отдельный запрос к серверу
- Проще определить момент полной загрузки
- Серверный рендеринг — SEO-тестирование актуально
- Формы отправляются синхронно — стандартный flow

### Сравнение с точки зрения QA

| Аспект | SPA | MPA |
|--------|-----|-----|
| Загрузка страницы | Одна загрузка, далее AJAX | Полная загрузка при каждом переходе |
| Ожидания в тестах | Сложные (динамический контент) | Простые (page load) |
| Сетевые запросы | Много мелких API-вызовов | Один запрос на страницу |
| Тестирование offline | Актуально (Service Workers) | Не актуально |
| Кэширование | На клиенте (localStorage, Redux) | На сервере |

## Request Flow: от клиента до базы данных и обратно

Полный путь запроса в типичном веб-приложении:

```
┌─────────┐     HTTP Request      ┌──────────────┐
│  Client  │ ──────────────────►   │  Web Server   │
│ (Browser)│                       │ (Nginx/Apache)│
└─────────┘                       └──────┬───────┘
                                         │
                                         ▼
                                  ┌──────────────┐
                                  │ App Server    │
                                  │ (Spring Boot) │
                                  └──────┬───────┘
                                         │
                                         ▼
                                  ┌──────────────┐
                                  │  Database     │
                                  │ (PostgreSQL)  │
                                  └──────────────┘
```

### Детальный путь запроса

**Шаг 1: Клиент формирует запрос**
```http
POST /api/users HTTP/1.1
Host: example.com
Content-Type: application/json
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...

{
  "name": "Иван Иванов",
  "email": "ivan@example.com"
}
```

**Шаг 2: DNS resolution** — браузер определяет IP-адрес сервера по доменному имени.

**Шаг 3: TCP/TLS handshake** — устанавливается защищённое соединение.

**Шаг 4: Web Server (reverse proxy)** — Nginx принимает запрос, проверяет rate limiting,
перенаправляет на application server.

**Шаг 5: Application Server** обрабатывает запрос:
```
Middleware/Filters → Controller → Service → Repository → Database
```

- **Filters/Interceptors** — проверка авторизации, логирование
- **Controller** — приём запроса, десериализация JSON
- **Service** — бизнес-логика (валидация, расчёты)
- **Repository** — формирование SQL-запроса

**Шаг 6: База данных** выполняет SQL-запрос:
```sql
INSERT INTO users (name, email) VALUES ('Иван Иванов', 'ivan@example.com');
```

**Шаг 7: Ответ идёт обратно** по тому же пути:
```
Database → Repository → Service → Controller → Web Server → Client
```

**Шаг 8: Клиент получает ответ:**
```http
HTTP/1.1 201 Created
Content-Type: application/json

{
  "id": 42,
  "name": "Иван Иванов",
  "email": "ivan@example.com"
}
```

## Как архитектура влияет на тестирование

### Уровни тестирования

```
┌─────────────────────────────────────────┐
│            UI Tests (E2E)               │  ← Медленные, хрупкие, дорогие
│         Selenium / Playwright           │
├─────────────────────────────────────────┤
│           API Tests                     │  ← Баланс скорости и покрытия
│       RestAssured / Postman             │
├─────────────────────────────────────────┤
│         Unit Tests                      │  ← Быстрые, дешёвые, много
│          JUnit / Mockito                │
└─────────────────────────────────────────┘
```

### Что тестировать на каком уровне

**UI Layer (Frontend):**
- Корректность отображения элементов
- Навигация между страницами
- Формы: валидация на клиенте, маски ввода
- Кросс-браузерная совместимость
- Адаптивность (responsive design)
- Accessibility (a11y)

**API Layer:**
- Контракт API (request/response schema)
- HTTP status codes для разных сценариев
- Валидация на сервере (обход клиентской валидации)
- Авторизация и аутентификация
- Обработка граничных значений
- Производительность эндпоинтов

**Database Layer:**
- Целостность данных после операций
- Constraints (unique, not null, foreign keys)
- Миграции базы данных
- Очистка тестовых данных

### API vs UI: где тестировать конкретные кейсы

```java
// Пример: тестирование создания пользователя

// API-уровень (быстро, надёжно) — проверяем бизнес-логику
@Test
void shouldCreateUserViaApi() {
    // Отправляем POST-запрос напрямую к API
    given()
        .contentType(ContentType.JSON)
        .body(new UserRequest("Иван", "ivan@example.com"))
    .when()
        .post("/api/users")
    .then()
        .statusCode(201)
        .body("name", equalTo("Иван"));
}

// UI-уровень (медленно, но проверяет интеграцию) — проверяем пользовательский сценарий
@Test
void shouldCreateUserViaForm() {
    // Заполняем форму в браузере
    open("/users/new");
    $("#name").setValue("Иван");
    $("#email").setValue("ivan@example.com");
    $("#submit").click();
    // Проверяем результат на UI
    $(".success-message").shouldHave(text("Пользователь создан"));
}
```

## Связь с тестированием

Понимание клиент-серверной архитектуры позволяет QA-инженеру:

1. **Локализовать баги быстрее** — определить, на каком уровне проблема (frontend, backend, DB)
2. **Выбирать правильный уровень тестов** — не дублировать проверки на UI, если достаточно API
3. **Писать эффективные баг-репорты** — с указанием конкретного запроса/ответа, а не только скриншота
4. **Анализировать DevTools** — Network tab показывает реальные запросы и ответы
5. **Тестировать безопасность** — клиентская валидация не заменяет серверную

---

## Типичные ошибки

1. **Тестирование только через UI** — медленно, хрупко, не покрывает edge cases на API
2. **Игнорирование серверной валидации** — клиентскую валидацию можно обойти через DevTools или curl
3. **Непонимание async-запросов** — в SPA данные загружаются асинхронно, тесты падают из-за race conditions
4. **Отсутствие тестов на error-ответы** — проверяют только happy path (200 OK), забывают про 400, 401, 403, 500
5. **Путаница frontend-бага и backend-бага** — без анализа Network tab невозможно правильно локализовать
6. **Тестирование в одном браузере** — frontend может вести себя по-разному в Chrome и Firefox
7. **Игнорирование кэширования** — баг может быть связан с кэшем, а не с логикой

## Практические задания

### Задание 1: Анализ запроса в DevTools
Откройте любое веб-приложение, перейдите в DevTools → Network. Выполните действие (например, логин).
Зафиксируйте:
- Метод и URL запроса
- Headers (Content-Type, Authorization)
- Request body
- Response status code и body
- Время выполнения запроса

### Задание 2: Обход клиентской валидации
Найдите форму с валидацией на frontend (например, обязательное поле email). Попробуйте отправить
некорректные данные напрямую через curl или Postman, минуя frontend. Проверьте, есть ли серверная
валидация.

```bash
# Пример: отправка запроса с некорректным email
curl -X POST http://localhost:8080/api/users \
  -H "Content-Type: application/json" \
  -d '{"name": "Test", "email": "не-email"}'
```

### Задание 3: Сравнение UI и API тестов
Для одного и того же функционала (например, создание заказа):
1. Напишите UI-тест (Selenium/Playwright)
2. Напишите API-тест (RestAssured/Postman)
3. Сравните: время выполнения, надёжность, покрытие

### Задание 4: Построение request flow
Для вашего рабочего проекта нарисуйте полную схему прохождения запроса от пользователя до БД.
Укажите все промежуточные компоненты (load balancer, reverse proxy, API gateway, кэш).
