# Основы сетей

## Обзор

Сетевые знания — один из ключевых фундаментальных навыков для QA-инженера. Когда тест падает
с ошибкой «Connection refused», когда API возвращает timeout, когда SSL-сертификат не валидный —
без понимания сетей невозможно правильно диагностировать проблему. В этом разделе разбираем
модели OSI и TCP/IP, протоколы TCP и UDP, DNS, TLS, WebSocket и другие концепции, с которыми
QA-инженер сталкивается ежедневно. Фокус — на практическом понимании, а не на академической теории.

---

## Модель OSI (Open Systems Interconnection)

Модель OSI — эталонная модель сетевого взаимодействия из 7 уровней. Каждый уровень отвечает
за свою область задач и взаимодействует с соседними уровнями.

### 7 уровней OSI

```
┌─────────────────────────────────────────────────────────────────┐
│ Уровень 7: Application (Прикладной)                             │
│ Протоколы: HTTP, HTTPS, FTP, SMTP, DNS, WebSocket               │
│ Что делает: Взаимодействие с приложениями пользователя           │
│ QA-контекст: API-тестирование, работа с REST/GraphQL             │
├─────────────────────────────────────────────────────────────────┤
│ Уровень 6: Presentation (Представления)                         │
│ Протоколы: SSL/TLS, JPEG, JSON, XML                              │
│ Что делает: Шифрование, сжатие, кодирование данных               │
│ QA-контекст: TLS-сертификаты, Content-Type, кодировки            │
├─────────────────────────────────────────────────────────────────┤
│ Уровень 5: Session (Сеансовый)                                  │
│ Протоколы: NetBIOS, RPC                                          │
│ Что делает: Управление сеансами связи                             │
│ QA-контекст: Сессии пользователей, cookies                       │
├─────────────────────────────────────────────────────────────────┤
│ Уровень 4: Transport (Транспортный)                             │
│ Протоколы: TCP, UDP                                              │
│ Что делает: Надёжная доставка данных, управление потоком          │
│ QA-контекст: Порты, TCP-соединения, таймауты                     │
├─────────────────────────────────────────────────────────────────┤
│ Уровень 3: Network (Сетевой)                                    │
│ Протоколы: IP, ICMP, ARP                                         │
│ Что делает: Маршрутизация пакетов между сетями                   │
│ QA-контекст: IP-адреса, ping, traceroute                         │
├─────────────────────────────────────────────────────────────────┤
│ Уровень 2: Data Link (Канальный)                                │
│ Протоколы: Ethernet, Wi-Fi (802.11)                              │
│ Что делает: Передача кадров в локальной сети                      │
│ QA-контекст: MAC-адреса (редко нужны)                            │
├─────────────────────────────────────────────────────────────────┤
│ Уровень 1: Physical (Физический)                                │
│ Что делает: Передача битов по физическому каналу                  │
│ QA-контекст: Кабели, Wi-Fi-сигнал (очень редко)                  │
└─────────────────────────────────────────────────────────────────┘
```

### Как данные проходят через уровни

При отправке HTTP-запроса данные «оборачиваются» заголовками каждого уровня (encapsulation):

```
Приложение создаёт HTTP-запрос:
  GET /api/users HTTP/1.1

Level 7 (Application):  HTTP-данные
Level 6 (Presentation): + TLS-шифрование
Level 5 (Session):      + управление сессией
Level 4 (Transport):    + TCP-заголовок (порты: src 54321, dst 443)
Level 3 (Network):      + IP-заголовок (src 192.168.1.10, dst 93.184.216.34)
Level 2 (Data Link):    + Ethernet-заголовок (MAC-адреса)
Level 1 (Physical):     → электрические сигналы / Wi-Fi
```

## TCP vs UDP

### TCP (Transmission Control Protocol)

TCP — надёжный, ориентированный на соединение протокол. Гарантирует доставку данных
в правильном порядке.

**Установка соединения — Three-Way Handshake:**

```
Клиент                          Сервер
  │                                │
  │ ── SYN (seq=100) ──────────►   │   1. Клиент: "Хочу подключиться"
  │                                │
  │ ◄── SYN-ACK (seq=200, ack=101) │   2. Сервер: "Принято, тоже хочу"
  │                                │
  │ ── ACK (seq=101, ack=201) ──►  │   3. Клиент: "Подтверждаю"
  │                                │
  │ ═══ Соединение установлено ═══ │
```

**Завершение соединения — Four-Way Handshake:**

```
Клиент                          Сервер
  │                                │
  │ ── FIN ─────────────────────►  │   1. Клиент: "Заканчиваю"
  │ ◄── ACK ────────────────────   │   2. Сервер: "Принял"
  │ ◄── FIN ────────────────────   │   3. Сервер: "Тоже заканчиваю"
  │ ── ACK ─────────────────────►  │   4. Клиент: "Подтверждаю"
  │                                │
  │ ═══ Соединение закрыто ═══════ │
```

**Механизмы надёжности:**
- **Acknowledgment (ACK)** — подтверждение получения каждого сегмента
- **Retransmission** — повторная отправка при потере пакета
- **Flow Control** — регулирование скорости отправки (sliding window)
- **Congestion Control** — адаптация к перегрузке сети

### UDP (User Datagram Protocol)

UDP — ненадёжный, без установления соединения протокол. Не гарантирует доставку и порядок.

```
Клиент                          Сервер
  │                                │
  │ ── Datagram 1 ─────────────►   │   Просто отправляем
  │ ── Datagram 2 ─────────────►   │   Без подтверждения
  │ ── Datagram 3 ─────────────►   │   Без соединения
  │                                │
```

### Сравнение

| Аспект | TCP | UDP |
|--------|-----|-----|
| Соединение | Устанавливается (handshake) | Без соединения |
| Надёжность | Гарантирует доставку | Не гарантирует |
| Порядок | Гарантирует порядок | Не гарантирует |
| Скорость | Медленнее (overhead) | Быстрее |
| Применение | HTTP, HTTPS, SSH, FTP | DNS (запросы), видео-стриминг, VoIP, игры |
| Overhead | Больше (заголовок 20+ байт) | Меньше (заголовок 8 байт) |

**Для QA:**
- Большинство тестируемых сервисов используют TCP (HTTP/HTTPS)
- UDP встречается в DNS, real-time приложениях (видеозвонки, игры)
- Ошибки TCP: `Connection refused`, `Connection timeout`, `Connection reset`

## DNS (Domain Name System)

### Что такое DNS

DNS — «телефонная книга интернета». Преобразует доменные имена (example.com) в IP-адреса
(93.184.216.34), потому что компьютеры работают с IP-адресами, а люди — с именами.

### Процесс DNS Resolution

```
Пользователь вводит: www.example.com

┌──────────┐    1. Есть в кэше?     ┌──────────────┐
│  Browser  │ ──────────────────►    │ Browser Cache │
│           │ ◄── Нет               │              │
└─────┬─────┘                        └──────────────┘
      │
      │ 2. Есть в /etc/hosts?
      ▼
┌──────────┐    3. Запрос к          ┌──────────────────┐
│    OS     │ ──────────────────►    │ Recursive Resolver│
│  Resolver │                        │ (ISP DNS / 8.8.8.8)│
└──────────┘                        └─────────┬────────┘
                                              │
                  4. Запрос к Root DNS        │
                  ┌──────────────┐ ◄──────────┤
                  │ Root Server  │            │
                  │ (.)          │ ──────────►│  "Спроси .com сервер"
                  └──────────────┘            │
                                              │
                  5. Запрос к TLD DNS         │
                  ┌──────────────┐ ◄──────────┤
                  │ TLD Server   │            │
                  │ (.com)       │ ──────────►│  "Спроси ns.example.com"
                  └──────────────┘            │
                                              │
                  6. Запрос к авторитетному DNS│
                  ┌──────────────┐ ◄──────────┤
                  │ Authoritative│            │
                  │ (example.com)│ ──────────►│  "IP = 93.184.216.34"
                  └──────────────┘            │
                                              │
                  7. Ответ клиенту            │
                  ◄───────────────────────────┘
                  IP: 93.184.216.34 (TTL: 3600)
```

### Типы DNS-записей

| Тип | Назначение | Пример |
|-----|-----------|--------|
| **A** | Домен → IPv4 | `example.com → 93.184.216.34` |
| **AAAA** | Домен → IPv6 | `example.com → 2606:2800:220:1:248:1893:...` |
| **CNAME** | Алиас (ссылка на другой домен) | `www.example.com → example.com` |
| **MX** | Mail сервер | `example.com → mail.example.com` |
| **TXT** | Текстовая информация | SPF, DKIM для email |
| **NS** | Nameserver | `example.com → ns1.example.com` |

### Диагностика DNS для QA

```bash
# Получить IP по домену
nslookup example.com
dig example.com

# Использовать конкретный DNS-сервер
nslookup example.com 8.8.8.8
dig @8.8.8.8 example.com

# Посмотреть все записи
dig example.com ANY

# Проверить TTL (время жизни кэша)
dig example.com | grep -E "^example"
# example.com.  3600  IN  A  93.184.216.34
#               ↑ TTL в секундах

# Проверить /etc/hosts (локальные DNS-переопределения)
cat /etc/hosts
# 127.0.0.1  localhost
# 192.168.1.100  test-server.local
```

**Типичные DNS-проблемы для QA:**
- Неправильная DNS-запись после деплоя на новый сервер
- DNS-кэш — старый IP после миграции (нужно ждать TTL)
- Локальные /etc/hosts переопределяют DNS (может мешать тестированию)

## HTTP/HTTPS и TLS Handshake

### HTTP vs HTTPS

| Аспект | HTTP | HTTPS |
|--------|------|-------|
| Порт | 80 | 443 |
| Шифрование | Нет | TLS/SSL |
| Сертификат | Не нужен | Нужен |
| Безопасность | Данные в открытом виде | Зашифрованы |
| Производительность | Чуть быстрее | Чуть медленнее (TLS handshake) |

### TLS Handshake (упрощённо)

```
Клиент                              Сервер
  │                                    │
  │ ── ClientHello ─────────────────►  │  1. "Привет! Поддерживаю TLS 1.3,
  │    (версия TLS, cipher suites)     │     шифры: AES-256, ChaCha20..."
  │                                    │
  │ ◄── ServerHello ────────────────   │  2. "Выбираю TLS 1.3, AES-256"
  │     + Certificate (публичный ключ) │     + Вот мой сертификат
  │     + ServerKeyExchange            │
  │                                    │
  │     Клиент проверяет сертификат:   │  3. Проверка сертификата:
  │     - Не истёк?                    │     - Доверенный CA?
  │     - Правильный домен?            │     - Цепочка доверия?
  │     - Доверенный CA?               │
  │                                    │
  │ ── ClientKeyExchange ───────────►  │  4. Обмен ключами (Diffie-Hellman)
  │    (Pre-Master Secret)             │
  │                                    │
  │ ══ Обе стороны вычисляют ═════════ │  5. Вычисляют общий Session Key
  │    Session Key                     │     для симметричного шифрования
  │                                    │
  │ ── Finished (зашифровано) ──────►  │  6. Готово!
  │ ◄── Finished (зашифровано) ─────   │
  │                                    │
  │ ═══ Защищённый канал ═════════════ │  Все дальнейшие данные зашифрованы
```

### Что проверяет QA в контексте TLS

```bash
# Проверить TLS-сертификат домена
openssl s_client -connect example.com:443 -servername example.com 2>/dev/null | \
  openssl x509 -noout -dates -subject
# Показывает: домен, даты начала/окончания действия сертификата

# Проверить, что HTTP перенаправляет на HTTPS
curl -v http://example.com 2>&1 | grep -i "location"
# Ожидаем: Location: https://example.com/

# Проверить поддерживаемые версии TLS
curl -v --tlsv1.2 https://example.com 2>&1 | grep "SSL connection"
```

**Типичные TLS-проблемы:**
- Истёкший сертификат → ошибка в браузере
- Self-signed сертификат → предупреждение в браузере (в тестовых средах — норма)
- Неправильный домен в сертификате → ERR_CERT_COMMON_NAME_INVALID
- Mixed content — HTTPS-страница загружает ресурсы по HTTP

## Порты

Порт — числовой идентификатор (0-65535), который определяет, какому приложению на сервере
предназначен пакет. IP-адрес определяет сервер, порт — приложение на сервере.

### Стандартные порты

| Порт | Протокол | Назначение |
|------|----------|------------|
| 20, 21 | FTP | Передача файлов |
| 22 | SSH | Безопасный удалённый доступ |
| 25 | SMTP | Отправка электронной почты |
| 53 | DNS | Разрешение доменных имён |
| 80 | HTTP | Веб-трафик (незашифрованный) |
| 443 | HTTPS | Веб-трафик (зашифрованный) |
| 3306 | MySQL | База данных MySQL |
| 5432 | PostgreSQL | База данных PostgreSQL |
| 6379 | Redis | Кэш Redis |
| 8080 | HTTP (alt) | Часто используется для dev-серверов |
| 9092 | Kafka | Apache Kafka |
| 27017 | MongoDB | База данных MongoDB |

### Диагностика портов

```bash
# Какие порты слушает сервер
ss -tlnp
netstat -tlnp

# Проверить доступность порта
nc -zv hostname 8080
telnet hostname 8080

# Какой процесс занимает порт
lsof -i :8080
ss -tlnp | grep 8080
```

## IP-адреса

### IPv4

Формат: `192.168.1.100` (4 октета по 8 бит, всего 32 бита)

**Типы адресов:**
- **Публичные** — уникальные в интернете (93.184.216.34)
- **Приватные** — используются в локальных сетях:
  - `10.0.0.0/8` (10.x.x.x)
  - `172.16.0.0/12` (172.16.x.x - 172.31.x.x)
  - `192.168.0.0/16` (192.168.x.x)
- **Loopback** — `127.0.0.1` (localhost) — сам компьютер
- **0.0.0.0** — все интерфейсы (при bind) или «любой адрес»

### IPv6

Формат: `2001:0db8:85a3:0000:0000:8a2e:0370:7334` (128 бит)
Сокращённо: `2001:db8:85a3::8a2e:370:7334`
Loopback: `::1`

### Подсети и CIDR

```
192.168.1.0/24
              ↑ Количество бит для сети

/24 = 255.255.255.0 = 256 адресов (192.168.1.0 - 192.168.1.255)
/16 = 255.255.0.0   = 65536 адресов
/8  = 255.0.0.0     = 16 777 216 адресов
```

**Для QA:** понимание IP нужно при работе с Docker networks, Kubernetes services,
тестовыми средами в разных подсетях.

## WebSocket

### Что такое WebSocket

WebSocket — протокол для двунаправленной коммуникации в реальном времени между клиентом
и сервером. В отличие от HTTP (запрос-ответ), WebSocket позволяет серверу отправлять данные
клиенту без запроса (push).

### Установка WebSocket-соединения (Upgrade)

```
Клиент                              Сервер
  │                                    │
  │ ── HTTP Upgrade Request ────────►  │
  │    GET /ws HTTP/1.1                │
  │    Upgrade: websocket              │
  │    Connection: Upgrade             │
  │    Sec-WebSocket-Key: dGhlIHNhbX...│
  │                                    │
  │ ◄── HTTP 101 Switching Protocols ─ │
  │     Upgrade: websocket             │
  │     Connection: Upgrade            │
  │     Sec-WebSocket-Accept: s3pPLM...│
  │                                    │
  │ ═══ WebSocket Connection ══════════│
  │                                    │
  │ ←→ Двунаправленные сообщения  ←→   │
  │     (текст или бинарные данные)    │
```

### HTTP vs WebSocket

| Аспект | HTTP | WebSocket |
|--------|------|-----------|
| Модель | Запрос-ответ | Двунаправленный канал |
| Инициатор | Всегда клиент | Обе стороны |
| Соединение | Новое для каждого запроса* | Постоянное |
| Overhead | Заголовки в каждом запросе | Минимальный после handshake |
| Применение | REST API, веб-страницы | Чаты, real-time уведомления, игры |

\* Keep-Alive переиспользует TCP-соединение, но протокол всё равно запрос-ответ.

### Тестирование WebSocket

```javascript
// Простой WebSocket-клиент для тестирования (JavaScript)
const ws = new WebSocket('ws://localhost:8080/ws/chat');

// Обработка открытия соединения
ws.onopen = () => {
    console.log('Соединение установлено');
    ws.send(JSON.stringify({type: 'message', text: 'Привет!'}));
};

// Обработка входящих сообщений
ws.onmessage = (event) => {
    console.log('Получено:', event.data);
};

// Обработка ошибок
ws.onerror = (error) => {
    console.error('Ошибка WebSocket:', error);
};

// Обработка закрытия
ws.onclose = (event) => {
    console.log('Соединение закрыто:', event.code, event.reason);
};
```

**Что тестировать в WebSocket:**
- Установка соединения (handshake)
- Отправка и получение сообщений
- Обработка разрыва соединения (reconnect)
- Формат сообщений (JSON schema)
- Аутентификация (через query params или первое сообщение)
- Нагрузка (множество одновременных соединений)

**Инструменты для тестирования WebSocket:**
- DevTools → Network → WS (фильтр по WebSocket)
- Postman (поддерживает WebSocket)
- wscat (CLI-утилита)
- Gatling, k6 (нагрузочное тестирование)

```bash
# wscat — CLI для WebSocket
npm install -g wscat
wscat -c ws://localhost:8080/ws/chat
> {"type": "message", "text": "Тест"}
```

## Зачем QA нужны знания сетей

### 1. Отладка connectivity issues

```bash
# Приложение не подключается к БД
# Шаг 1: Проверяем DNS
nslookup db-server.example.com
# Шаг 2: Проверяем доступность порта
nc -zv db-server.example.com 5432
# Шаг 3: Проверяем маршрут
traceroute db-server.example.com
# Шаг 4: Проверяем firewall
curl -v telnet://db-server.example.com:5432
```

### 2. Понимание latency

```
Общее время ответа = DNS + TCP Handshake + TLS Handshake + Request + Server Processing + Response

Типичные значения:
- DNS Lookup:       1-100 мс (с кэшем: 0 мс)
- TCP Handshake:    1-100 мс (зависит от расстояния)
- TLS Handshake:    50-200 мс
- Server Processing: 10-5000 мс (зависит от логики)
- Content Download:  1-1000 мс (зависит от размера)
```

### 3. API testing context

Понимание HTTP-заголовков, status codes, CORS, cookies — всё это сетевые знания,
которые напрямую используются в API-тестировании.

### 4. Диагностика в production-like средах

```bash
# Проверка здоровья сервиса
curl -s -o /dev/null -w "%{http_code}" http://service:8080/health

# Проверка времени ответа
curl -w "DNS: %{time_namelookup}s\nConnect: %{time_connect}s\nTLS: %{time_appconnect}s\nTTFB: %{time_starttransfer}s\nTotal: %{time_total}s\n" \
  -o /dev/null -s https://api.example.com/users

# Проверка SSL-сертификата
echo | openssl s_client -connect api.example.com:443 2>/dev/null | openssl x509 -noout -dates
```

## Связь с тестированием

| Сетевая концепция | Применение в тестировании |
|-------------------|-------------------------|
| TCP/UDP | Понимание ошибок соединения (refused, timeout, reset) |
| DNS | Диагностика проблем с доступностью сервисов |
| HTTP/HTTPS | API-тестирование, анализ Network tab |
| TLS | Тестирование безопасности, проверка сертификатов |
| Порты | Конфигурация тестовых сред (Docker, K8s port-forward) |
| IP-адреса | Docker networks, Kubernetes networking |
| WebSocket | Тестирование real-time функциональности |
| OSI-модель | Систематическая диагностика проблем по уровням |

## Типичные ошибки

1. **Не различают «Connection refused» и «Connection timeout»:**
   - `Refused` — сервер доступен, но порт не слушается (приложение не запущено)
   - `Timeout` — сервер недоступен (сеть, firewall) или отвечает слишком долго
2. **Игнорируют DNS-кэш** — после смены IP тесты идут на старый сервер
3. **Не учитывают TLS** — тесты в HTTP, production в HTTPS → разное поведение (cookies Secure, CORS)
4. **Путают порт приложения и маппинг** — в Docker `8080:3000` означает: хост:контейнер
5. **Не проверяют сертификаты** — self-signed в тестовой среде, но забывают для staging
6. **Не анализируют timing** — не знают, на что уходит время ответа (DNS? TLS? Сервер?)
7. **Не тестируют WebSocket disconnect/reconnect** — только happy path

### Задание 2: Анализ TCP-соединения
Используя `curl -v`, отправьте запрос к HTTPS-сайту и идентифицируйте:
1. DNS resolution
2. TCP connection
3. TLS handshake (версия TLS, cipher suite)

```bash
curl -v -w "\nDNS: %{time_namelookup}\nConnect: %{time_connect}\nTLS: %{time_appconnect}\nTTFB: %{time_starttransfer}\nTotal: %{time_total}\n" \
  -o /dev/null -s https://example.com
```
