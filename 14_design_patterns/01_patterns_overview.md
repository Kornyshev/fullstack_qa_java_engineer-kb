# Обзор паттернов проектирования

## Обзор

Паттерны проектирования (Design Patterns) — это проверенные временем типовые решения часто встречающихся
проблем в разработке программного обеспечения. Термин получил широкую известность после выхода книги
"Design Patterns: Elements of Reusable Object-Oriented Software" (1994), написанной так называемой
**"Бандой четырёх"** (Gang of Four, GoF): Erich Gamma, Richard Helm, Ralph Johnson, John Vlissides.

Для QA-инженера знание паттернов проектирования — не просто теоретический багаж. Это практический
инструмент, позволяющий:

- **Писать поддерживаемые автотесты** — фреймворк, построенный на паттернах, легче расширять и менять.
- **Читать код разработчиков** — понимание паттернов помогает быстрее разбираться в продуктовом коде.
- **Находить архитектурные баги** — неправильное применение паттернов часто приводит к дефектам.
- **Проходить техническое интервью** — вопросы о паттернах стандартны для позиций QA Automation.

---

## Зачем QA-инженеру знать паттерны

### Чтение и ревью продуктового кода

Когда QA-инженер участвует в code review или исследует баг в исходном коде приложения, знание паттернов
позволяет быстро ориентироваться в архитектуре. Если вы видите класс `UserFactory`, вы сразу понимаете,
что это фабрика, создающая объекты `User`, и не тратите время на расшифровку логики.

### Построение тестовой инфраструктуры

Автоматизированное тестирование — это полноценная разработка. Тестовые фреймворки используют те же
паттерны, что и продуктовый код:

| Паттерн              | Где используется в тестировании                        |
|----------------------|--------------------------------------------------------|
| **Page Object**      | Инкапсуляция элементов и действий на странице          |
| **Factory**          | Создание WebDriver для разных браузеров                |
| **Builder**          | Конструирование тестовых данных и запросов              |
| **Singleton**        | Единственный экземпляр конфигурации или драйвера       |
| **Strategy**         | Переключение стратегий запуска (local, remote, CI)     |
| **Template Method**  | Базовый класс теста с setup/teardown                   |
| **Observer**         | Слушатели событий в TestNG/JUnit                       |

### Общий язык с командой

Паттерны — это общепринятый словарь. Когда вы говорите "давайте применим Builder для тестовых данных",
команда мгновенно понимает идею, без длинных объяснений.

## Три категории GoF-паттернов

GoF выделили **23 паттерна**, разделив их на три категории.

### 1. Порождающие паттерны (Creational Patterns)

Отвечают за **создание объектов**, абстрагируя процесс инстанцирования.

| Паттерн                | Суть                                                             |
|------------------------|------------------------------------------------------------------|
| **Singleton**          | Гарантирует единственный экземпляр класса                        |
| **Factory Method**     | Делегирует создание объектов подклассам                           |
| **Abstract Factory**   | Создаёт семейства связанных объектов                             |
| **Builder**            | Пошаговое конструирование сложных объектов                       |
| **Prototype**          | Создание объекта путём клонирования существующего                 |

**Для QA наиболее важны:** Singleton, Factory Method, Builder.

### 2. Структурные паттерны (Structural Patterns)

Описывают, как **компоновать объекты и классы** в более крупные структуры.

| Паттерн          | Суть                                                                |
|------------------|---------------------------------------------------------------------|
| **Adapter**      | Приводит интерфейс класса к ожидаемому                              |
| **Bridge**       | Разделяет абстракцию и реализацию                                   |
| **Composite**    | Представляет древовидные структуры как единый объект                 |
| **Decorator**    | Динамически добавляет объекту новую функциональность                |
| **Facade**       | Предоставляет упрощённый интерфейс к сложной подсистеме             |
| **Flyweight**    | Разделяет общие данные между множеством мелких объектов              |
| **Proxy**        | Контролирует доступ к объекту через подставной объект                |

**Для QA наиболее важны:** Adapter, Decorator, Proxy, Facade.

### 3. Поведенческие паттерны (Behavioral Patterns)

Определяют **взаимодействие между объектами** и распределение обязанностей.

| Паттерн                      | Суть                                                        |
|------------------------------|-------------------------------------------------------------|
| **Strategy**                 | Инкапсулирует семейство алгоритмов, делает их взаимозаменяемыми |
| **Template Method**          | Определяет скелет алгоритма, делегируя шаги подклассам      |
| **Observer**                 | Уведомляет зависимые объекты об изменениях                  |
| **Chain of Responsibility**  | Передаёт запрос по цепочке обработчиков                     |
| **Command**                  | Инкапсулирует запрос как объект                             |
| **Iterator**                 | Последовательный доступ к элементам коллекции               |
| **Mediator**                 | Централизует взаимодействие между объектами                 |
| **Memento**                  | Сохраняет и восстанавливает состояние объекта               |
| **State**                    | Изменяет поведение объекта при изменении состояния          |
| **Visitor**                  | Добавляет операции к объектам без изменения их классов      |
| **Interpreter**              | Определяет грамматику и интерпретирует предложения          |

**Для QA наиболее важны:** Strategy, Template Method, Observer, Chain of Responsibility.

## Паттерн vs Антипаттерн

### Паттерн (Pattern)

Паттерн — это **рекомендуемое решение** повторяющейся проблемы в определённом контексте.
Паттерн описывает проблему, решение и последствия применения.

### Антипаттерн (Anti-pattern)

Антипаттерн — это **типичное плохое решение**, которое кажется привлекательным на первый взгляд,
но ведёт к проблемам в долгосрочной перспективе.

| Антипаттерн в тестировании    | Описание                                                        |
|-------------------------------|-----------------------------------------------------------------|
| **God Test**                  | Один тест проверяет всё подряд, невозможно понять причину падения |
| **Fragile Test**              | Тест ломается при любом незначительном изменении UI              |
| **Copy-Paste Test**           | Дублирование кода между тестами вместо выделения общего          |
| **Hardcoded Wait**            | `Thread.sleep()` вместо явных ожиданий (Explicit Waits)         |
| **Mystery Guest**             | Тест зависит от внешних данных, которые нигде не описаны        |
| **Spaghetti Test**            | Отсутствие структуры, все действия в одном методе               |

### Пример: паттерн vs антипаттерн

```java
// Антипаттерн: вся логика в тесте, хардкод, дублирование
@Test
void loginTest() {
    driver.get("http://localhost:8080/login");
    driver.findElement(By.id("username")).sendKeys("admin");
    driver.findElement(By.id("password")).sendKeys("password123");
    driver.findElement(By.id("loginBtn")).click();
    Thread.sleep(3000);
    String title = driver.findElement(By.id("dashboard-title")).getText();
    assertEquals("Dashboard", title);
}

// Паттерн Page Object: чистый, поддерживаемый код
@Test
void loginTest() {
    LoginPage loginPage = new LoginPage(driver);
    DashboardPage dashboard = loginPage.loginAs("admin", "password123");
    assertEquals("Dashboard", dashboard.getTitle());
}
```

## Ключевые паттерны в тестовых фреймворках

### Page Object Model (POM)

Хотя Page Object не входит в GoF, это **самый важный паттерн** в UI-автоматизации.
Каждая страница приложения представлена отдельным классом, инкапсулирующим элементы и действия.

```
┌─────────────────────┐       ┌─────────────────────┐
│     Test Class      │──────>│     LoginPage       │
│  testLogin()        │       │  - usernameField    │
│  testLogout()       │       │  - passwordField    │
│                     │       │  + login(user, pwd) │
└─────────────────────┘       └─────────────────────┘
                                       │
                                       v
                              ┌─────────────────────┐
                              │   DashboardPage     │
                              │  - title            │
                              │  + getTitle()       │
                              └─────────────────────┘
```

### Fluent Interface / Method Chaining

Позволяет выстраивать цепочки вызовов, что делает тесты читаемыми:

```java
// Свободный интерфейс для создания тестовых данных
User user = User.builder()
    .name("Иван")
    .email("ivan@test.com")
    .role(Role.ADMIN)
    .build();
```

### Data-Driven Testing

Шаблон, при котором тестовая логика отделена от тестовых данных. Реализуется через
`@ParameterizedTest` в JUnit 5 или `@DataProvider` в TestNG.

## Как выбрать подходящий паттерн

```
Проблема: нужен единственный экземпляр?          → Singleton
Проблема: нужно создавать разные объекты?         → Factory / Abstract Factory
Проблема: объект сложный, много параметров?       → Builder
Проблема: нужно адаптировать интерфейс?           → Adapter
Проблема: нужно добавить поведение динамически?   → Decorator
Проблема: нужно упростить сложный API?            → Facade
Проблема: нужно переключать алгоритмы?            → Strategy
Проблема: общий алгоритм с разными шагами?        → Template Method
Проблема: нужно уведомлять об изменениях?          → Observer
```

## Связь с тестированием

Паттерны проектирования — фундамент качественной автоматизации тестирования:

1. **Page Object** снижает стоимость поддержки UI-тестов при изменении интерфейса.
2. **Factory** позволяет легко переключаться между браузерами и окружениями.
3. **Builder** упрощает создание сложных тестовых данных.
4. **Strategy** даёт возможность запускать одни и те же тесты в разных конфигурациях.
5. **Template Method** стандартизирует жизненный цикл теста (setup, execute, verify, teardown).

Понимание паттернов помогает QA-инженеру не только писать хорошие тесты, но и находить
дефекты, связанные с нарушением архитектурных принципов в продуктовом коде.

## Типичные ошибки

1. **Overengineering** — применение паттернов там, где достаточно простого решения. Не нужно
   создавать Abstract Factory для двух классов.
2. **Неправильный выбор паттерна** — например, использование Singleton там, где нужен
   обычный объект с ограниченным временем жизни.
3. **Паттерн ради паттерна** — внедрение паттерна только потому, что "так положено", без
   реальной необходимости.
4. **Смешение паттернов без понимания** — попытка совместить несовместимые подходы.
5. **Игнорирование контекста** — паттерн, идеальный для enterprise-приложения, может быть
   избыточен для маленького тестового проекта.
6. **Нарушение принципов SOLID при реализации** — паттерн реализован, но с нарушением
   базовых принципов, что нивелирует его преимущества.

## Практические задания

1. **Анализ фреймворка.** Возьмите любой open-source тестовый фреймворк (Selenide, RestAssured)
   и определите, какие паттерны GoF в нём применяются. Составьте список с обоснованием.

2. **Рефакторинг с антипаттерна на паттерн.** Возьмите тест с `Thread.sleep()`, хардкодом
   локаторов и дублированием — перепишите его, применив Page Object и Explicit Waits.

3. **Выбор паттерна.** Для каждой из следующих задач определите подходящий паттерн и обоснуйте:
   - Создание тестовых пользователей с разными ролями.
   - Переключение между локальным и удалённым запуском тестов.
   - Единая точка доступа к конфигурации тестового окружения.
   - Добавление логирования ко всем HTTP-запросам в тестах.

4. **Диаграмма архитектуры.** Нарисуйте диаграмму вашего текущего (или желаемого) тестового
   фреймворка, отметив все используемые паттерны.

5. **Code Review.** Проведите ревью следующего кода и определите, какие антипаттерны в нём
   присутствуют:
   ```java
   @Test
   void test1() {
       WebDriver d = new ChromeDriver();
       d.get("http://prod.company.com/admin");
       d.findElement(By.xpath("/html/body/div[3]/form/input[1]")).sendKeys("admin");
       d.findElement(By.xpath("/html/body/div[3]/form/input[2]")).sendKeys("admin123");
       d.findElement(By.xpath("/html/body/div[3]/form/button")).click();
       Thread.sleep(5000);
       assertTrue(d.findElement(By.xpath("/html/body/div[1]/span")).getText().contains("Welcome"));
       d.findElement(By.xpath("/html/body/div[2]/a[3]")).click();
       Thread.sleep(3000);
       assertEquals(5, d.findElements(By.tagName("tr")).size());
   }
   ```

## Дополнительные ресурсы

- **"Design Patterns: Elements of Reusable Object-Oriented Software"** — Gamma, Helm, Johnson,
  Vlissides (GoF) — оригинальная книга, ставшая классикой.
- **"Head First Design Patterns"** — Freeman, Robson — доступное изложение паттернов с примерами.
- **Refactoring Guru** (refactoring.guru) — интерактивные объяснения паттернов с диаграммами.
- **"Clean Code"** — Robert C. Martin — принципы чистого кода, тесно связанные с паттернами.
- **"xUnit Test Patterns"** — Gerard Meszaros — паттерны, специфичные для тестирования.
- **Selenium Wiki — Page Object Model** — официальная документация по ключевому паттерну
  автоматизации.
- **Baeldung** (baeldung.com) — статьи о паттернах с примерами на Java.
