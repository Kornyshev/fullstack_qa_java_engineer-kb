# Основы REST API

## Обзор

REST (Representational State Transfer) — архитектурный стиль взаимодействия между клиентом и сервером, предложенный Роем Филдингом в 2000 году. Для QA-инженера глубокое понимание REST API является фундаментальным навыком: большинство современных приложений строятся на микросервисной архитектуре, где сервисы общаются именно через REST API. Тестирование API — это первый уровень проверки бизнес-логики без участия UI.

В этом разделе рассмотрены основы HTTP-протокола, методы запросов, коды ответов, заголовки, форматы тела запроса/ответа, ограничения REST и правила проектирования URI.

## HTTP-протокол

HTTP (HyperText Transfer Protocol) — протокол прикладного уровня, работающий поверх TCP/IP. Каждое взаимодействие состоит из **запроса** (request) и **ответа** (response).

### Структура HTTP-запроса

```
METHOD /path/to/resource?query=value HTTP/1.1
Host: api.example.com
Content-Type: application/json
Authorization: Bearer <token>

{
  "key": "value"
}
```

Основные компоненты:
- **Request line** — метод, URI, версия протокола
- **Headers** — метаданные запроса (пары ключ-значение)
- **Body** — тело запроса (не у всех методов)

### Структура HTTP-ответа

```
HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: no-cache

{
  "id": 1,
  "name": "Test User"
}
```

Основные компоненты:
- **Status line** — версия протокола, код статуса, описание
- **Headers** — метаданные ответа
- **Body** — тело ответа

### HTTP/1.1 vs HTTP/2

| Характеристика | HTTP/1.1 | HTTP/2 |
|----------------|----------|--------|
| Мультиплексирование | Нет (один запрос за раз на соединение) | Да (параллельные потоки) |
| Сжатие заголовков | Нет | Да (HPACK) |
| Server Push | Нет | Да |
| Формат | Текстовый | Бинарный |

## HTTP-методы

### Основные методы

| Метод | Назначение | Тело запроса | Идемпотентность | Safe |
|-------|-----------|-------------|-----------------|------|
| GET | Получение ресурса | Нет | Да | Да |
| POST | Создание ресурса | Да | Нет | Нет |
| PUT | Полная замена ресурса | Да | Да | Нет |
| PATCH | Частичное обновление ресурса | Да | Нет* | Нет |
| DELETE | Удаление ресурса | Редко | Да | Нет |
| HEAD | Аналогичен GET, но без тела ответа | Нет | Да | Да |
| OPTIONS | Получение допустимых методов | Нет | Да | Да |

> **Идемпотентность** — свойство операции, при котором многократное выполнение даёт тот же результат, что и однократное. Например, DELETE /users/5 — при первом вызове удалит пользователя, при повторном вернёт 404, но состояние системы не изменится.

> **Safe (безопасный)** — метод, который не изменяет состояние сервера. GET и HEAD — безопасные методы.

> *PATCH формально не идемпотентен по спецификации, хотя на практике часто реализуется как идемпотентный.

### PUT vs PATCH — разница для QA

```json
// Исходный ресурс
{
  "id": 1,
  "name": "Иван",
  "email": "ivan@test.com",
  "age": 25
}

// PUT /users/1 — полная замена, нужно передать ВСЕ поля
{
  "name": "Иван Обновлённый",
  "email": "ivan@test.com",
  "age": 25
}

// PATCH /users/1 — частичное обновление, только изменяемые поля
{
  "name": "Иван Обновлённый"
}
```

**Что проверять:** при PUT без обязательного поля сервер должен вернуть 400; при PATCH отсутствующие поля не должны обнуляться.

## Коды состояния HTTP (Status Codes)

### 1xx — Информационные

| Код | Описание | Когда встречается |
|-----|----------|-------------------|
| 100 | Continue | Сервер готов принять тело запроса |
| 101 | Switching Protocols | Переключение на WebSocket |

### 2xx — Успешные

| Код | Описание | Типичный контекст |
|-----|----------|-------------------|
| 200 | OK | Успешный GET, PUT, PATCH |
| 201 | Created | Успешный POST (ресурс создан) |
| 204 | No Content | Успешный DELETE (тело ответа пустое) |

### 3xx — Перенаправления

| Код | Описание | Типичный контекст |
|-----|----------|-------------------|
| 301 | Moved Permanently | Ресурс перемещён навсегда |
| 302 | Found | Временное перенаправление |
| 304 | Not Modified | Ресурс не изменился (кэширование) |

### 4xx — Ошибки клиента

| Код | Описание | Типичный контекст |
|-----|----------|-------------------|
| 400 | Bad Request | Невалидное тело запроса, отсутствие обязательных полей |
| 401 | Unauthorized | Отсутствует или невалидный токен авторизации |
| 403 | Forbidden | Токен валиден, но нет прав доступа |
| 404 | Not Found | Ресурс не найден |
| 405 | Method Not Allowed | Метод не поддерживается для данного endpoint |
| 409 | Conflict | Конфликт (например, дублирование уникального поля) |
| 415 | Unsupported Media Type | Неподдерживаемый Content-Type |
| 422 | Unprocessable Entity | Тело запроса валидно синтаксически, но не семантически |
| 429 | Too Many Requests | Превышен rate limit |

### 5xx — Ошибки сервера

| Код | Описание | Типичный контекст |
|-----|----------|-------------------|
| 500 | Internal Server Error | Необработанная ошибка на сервере |
| 502 | Bad Gateway | Проблема с upstream-сервером |
| 503 | Service Unavailable | Сервер временно недоступен |
| 504 | Gateway Timeout | Upstream-сервер не ответил вовремя |

> **Для QA:** разница между 401 и 403 — частый вопрос на интервью. 401 означает «мы не знаем кто ты» (нет токена или токен невалиден). 403 означает «мы знаем кто ты, но у тебя нет прав».

## HTTP-заголовки (Headers)

### Основные заголовки запроса

| Заголовок | Назначение | Пример |
|-----------|-----------|--------|
| Content-Type | Формат тела запроса | `application/json` |
| Accept | Ожидаемый формат ответа | `application/json` |
| Authorization | Данные аутентификации | `Bearer eyJhbGciOi...` |
| Cookie | Передача cookies | `session_id=abc123` |
| User-Agent | Информация о клиенте | `Mozilla/5.0...` |
| Cache-Control | Управление кэшированием | `no-cache` |
| X-Request-Id | Кастомный заголовок для трассировки | `uuid-value` |

### Основные заголовки ответа

| Заголовок | Назначение | Пример |
|-----------|-----------|--------|
| Content-Type | Формат тела ответа | `application/json; charset=utf-8` |
| Set-Cookie | Установка cookie на клиенте | `session=abc; HttpOnly; Secure` |
| Location | URI созданного ресурса (с 201) | `/api/users/42` |
| X-RateLimit-Remaining | Оставшееся кол-во запросов | `98` |
| Access-Control-Allow-Origin | CORS — разрешённые домены | `https://app.example.com` |

### Кастомные заголовки

Кастомные заголовки обычно начинаются с `X-` (хотя это соглашение устарело по RFC 6648). Примеры:
- `X-Api-Key` — API-ключ
- `X-Correlation-Id` — идентификатор для трассировки в микросервисах
- `X-Forwarded-For` — реальный IP клиента за proxy

## Тело запроса и ответа

### JSON (JavaScript Object Notation)

Наиболее распространённый формат для REST API.

```json
{
  "id": 1,
  "name": "Тестовый пользователь",
  "email": "test@example.com",
  "roles": ["ADMIN", "USER"],
  "address": {
    "city": "Москва",
    "zipCode": "101000"
  },
  "active": true,
  "createdAt": "2025-01-15T10:30:00Z"
}
```

Типы данных в JSON: `string`, `number`, `boolean`, `null`, `object`, `array`.

### XML (eXtensible Markup Language)

Встречается в legacy-системах и SOAP-сервисах.

```xml
<user>
  <id>1</id>
  <name>Тестовый пользователь</name>
  <email>test@example.com</email>
  <roles>
    <role>ADMIN</role>
    <role>USER</role>
  </roles>
</user>
```

### Что проверять в теле ответа

- Корректность структуры (наличие всех ожидаемых полей)
- Типы данных (число, а не строка; массив, а не объект)
- Формат дат (ISO 8601: `yyyy-MM-dd'T'HH:mm:ss'Z'`)
- Отсутствие чувствительных данных (пароли, внутренние ID)
- Корректность вложенных объектов и массивов

## REST Constraints (Ограничения REST)

Рой Филдинг определил 6 ограничений, которым должен соответствовать RESTful-сервис:

### 1. Client-Server (Клиент-сервер)
Клиент и сервер разделены. Клиент не знает о бизнес-логике, сервер не знает о UI.

### 2. Stateless (Без состояния)
Каждый запрос содержит **всю** информацию, необходимую серверу. Сервер не хранит состояние сессии между запросами. Токен авторизации передаётся в каждом запросе.

### 3. Cacheable (Кэшируемость)
Ответы должны указывать, можно ли их кэшировать. Заголовки `Cache-Control`, `ETag`, `Last-Modified`.

### 4. Uniform Interface (Единообразный интерфейс)
- Идентификация ресурсов через URI
- Манипуляция ресурсами через представления
- Самоописывающие сообщения (Content-Type указывает формат)
- HATEOAS (Hypermedia as the Engine of Application State) — ответ содержит ссылки на связанные ресурсы

### 5. Layered System (Слоёная система)
Клиент не знает, общается ли он с конечным сервером или с промежуточным (load balancer, API gateway).

### 6. Code on Demand (Код по запросу) — необязательное
Сервер может отправить исполняемый код клиенту (JavaScript).

## RESTful vs REST-like

| Критерий | RESTful | REST-like |
|----------|---------|-----------|
| Все 6 ограничений | Соблюдены | Частично |
| HATEOAS | Да | Обычно нет |
| Stateless | Строго | Может использовать сессии |
| Коды статуса | Семантически корректные | Часто всё через 200 |
| URI design | Ресурсо-ориентированный | Может содержать глаголы |

> **На практике** большинство API являются REST-like, а не строго RESTful. Это нормально и важно понимать для QA.

## Проектирование URI

### Правила хорошего URI

```
# Существительные, а не глаголы
GET /api/users           — получить список пользователей
GET /api/users/42        — получить пользователя с id=42
POST /api/users          — создать пользователя
PUT /api/users/42        — обновить пользователя
DELETE /api/users/42     — удалить пользователя

# Вложенные ресурсы
GET /api/users/42/orders         — заказы пользователя 42
GET /api/users/42/orders/7       — заказ 7 пользователя 42

# Плохие примеры (антипаттерны)
GET /api/getUsers                — глагол в URI
POST /api/deleteUser/42          — DELETE через POST
GET /api/Users                   — заглавная буква
```

### Path Parameters vs Query Parameters

| Аспект | Path Parameters | Query Parameters |
|--------|----------------|-----------------|
| Синтаксис | `/users/{id}` | `/users?role=admin` |
| Назначение | Идентификация конкретного ресурса | Фильтрация, сортировка, пагинация |
| Обязательность | Обычно обязательные | Обычно опциональные |
| Пример | `/api/users/42` | `/api/users?page=1&size=10&sort=name` |

### Типичные query parameters

```
# Пагинация
GET /api/users?page=0&size=20

# Сортировка
GET /api/users?sort=name,asc

# Фильтрация
GET /api/users?status=active&role=admin

# Поиск
GET /api/users?search=Иван

# Комбинирование
GET /api/users?status=active&page=0&size=10&sort=createdAt,desc
```

## Связь с тестированием

### Что проверяет QA-инженер при тестировании API

1. **Функциональное тестирование** — корректность бизнес-логики через API
2. **Валидация ответов** — статус-коды, заголовки, структура тела
3. **Негативное тестирование** — невалидные данные, отсутствующие поля, неправильные типы
4. **Граничные значения** — пустые строки, максимальные длины, специальные символы
5. **Авторизация** — доступ без токена, с невалидным токеном, с недостаточными правами
6. **Идемпотентность** — повторные запросы дают ожидаемый результат
7. **Совместимость версий** — API v1 vs API v2

### Практика: анализ API через Browser DevTools

1. Откройте любой сайт (например, GitHub)
2. Нажмите F12 → вкладка **Network**
3. Отфильтруйте по **Fetch/XHR**
4. Выполните действие на сайте (поиск, переход по странице)
5. Кликните на запрос и изучите:
   - **Headers** — метод, URL, заголовки запроса/ответа
   - **Payload** — тело запроса
   - **Preview/Response** — тело ответа
   - **Timing** — время выполнения запроса

## Типичные ошибки

1. **Путать 401 и 403** — 401 = не аутентифицирован, 403 = не авторизован (нет прав)
2. **Не проверять заголовки ответа** — Content-Type, Cache-Control и другие содержат важную информацию
3. **Игнорировать идемпотентность** — повторный POST не должен создавать дубликат (зависит от реализации)
4. **Считать, что 200 = всё хорошо** — тело ответа может содержать ошибки при коде 200 (плохой дизайн, но встречается)
5. **Тестировать только happy path** — забывать про невалидные данные, отсутствие авторизации, несуществующие ресурсы
6. **Не проверять формат дат** — разные часовые пояса, форматы (ISO vs timestamp)
7. **Путать PUT и PATCH** — PUT заменяет весь ресурс, PATCH — только переданные поля
8. **Не учитывать пагинацию** — API может возвращать только первые N записей

## Практические задания

### Задание 1: Анализ API через DevTools
Откройте сайт `https://jsonplaceholder.typicode.com` в браузере. Используя DevTools, выполните запросы к `/posts`, `/posts/1`, `/comments?postId=1`. Для каждого запроса зафиксируйте: метод, URL, заголовки, статус-код, тело ответа.

### Задание 2: Тест-кейсы для CRUD
Напишите список тест-кейсов (позитивных и негативных) для API управления пользователями:
- `GET /api/users` — список
- `GET /api/users/{id}` — конкретный пользователь
- `POST /api/users` — создание
- `PUT /api/users/{id}` — обновление
- `DELETE /api/users/{id}` — удаление

Учтите: валидацию полей, авторизацию, несуществующие ресурсы, дублирование, пагинацию.

### Задание 3: Проектирование URI
Спроектируйте URI для интернет-магазина. Ресурсы: товары (products), категории (categories), заказы (orders), отзывы (reviews). Определите, какие параметры будут path, а какие — query.
