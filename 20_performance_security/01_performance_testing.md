# Нагрузочное тестирование

## Обзор

Нагрузочное тестирование (Performance Testing) -- это процесс проверки поведения системы
под различными уровнями нагрузки. Цель -- убедиться, что приложение работает стабильно,
быстро и надёжно при реальных и экстремальных условиях эксплуатации.

Для QA-инженера понимание нагрузочного тестирования критически важно: даже если функционал
работает корректно, медленное или нестабильное приложение теряет пользователей. По данным
Google, увеличение времени загрузки страницы с 1 до 3 секунд повышает показатель отказов
на 32%.

Нагрузочное тестирование -- это не просто "запустить скрипт с множеством потоков".
Это системный процесс: планирование, разработка сценариев, выполнение, анализ и отчётность.

---

## Виды нагрузочного тестирования

### Load Testing -- нагрузочное тестирование

Проверка системы при **ожидаемой** (нормальной) нагрузке. Цель -- убедиться, что система
справляется с плановым количеством пользователей.

| Параметр | Пример |
|----------|--------|
| Сценарий | 500 одновременных пользователей совершают покупки |
| Длительность | 30-60 минут |
| Ожидание | Время отклика < 2 сек, ошибок < 1% |

### Stress Testing -- стресс-тестирование

Проверка системы при нагрузке, **превышающей** нормальную. Цель -- найти предел прочности
и понять, как система деградирует.

- Постепенно увеличиваем нагрузку до отказа
- Фиксируем точку перелома (breaking point)
- Проверяем, восстанавливается ли система после снятия нагрузки

### Spike Testing -- тестирование пиковых нагрузок

Проверка реакции системы на **резкий** скачок нагрузки. Моделирует ситуации вроде
распродажи Black Friday или вирусного контента.

```
Нагрузка
  ^
  |        ____
  |       |    |
  |       |    |
  |_______|    |_______
  +-----------------------> Время
```

### Soak / Endurance Testing -- тестирование на стабильность

Проверка системы при **длительной** нагрузке (часы, дни). Цель -- выявить утечки памяти
(memory leaks), деградацию производительности, проблемы с пулом соединений.

| Параметр | Пример |
|----------|--------|
| Длительность | 8-24 часа |
| Нагрузка | Умеренная (50-70% от максимальной) |
| Что ищем | Memory leaks, connection pool exhaustion, disk space |

### Volume Testing -- объёмное тестирование

Проверка работы системы с **большим объёмом данных**. Как поведёт себя база данных
с 10 миллионами записей? Отработает ли поиск за приемлемое время?

### Scalability Testing -- тестирование масштабируемости

Проверка, насколько эффективно система масштабируется при увеличении ресурсов:
- **Вертикальное масштабирование** -- увеличение мощности одного сервера (CPU, RAM)
- **Горизонтальное масштабирование** -- добавление новых серверов / инстансов

## Ключевые метрики производительности

### Response Time -- время отклика

Время от отправки запроса до получения ответа. Измеряется в перцентилях:

| Метрика | Значение | Почему важно |
|---------|----------|-------------|
| **p50 (медиана)** | 50% запросов быстрее этого значения | Типичный пользовательский опыт |
| **p90** | 90% запросов быстрее | Опыт большинства пользователей |
| **p95** | 95% запросов быстрее | Часто используется для SLA |
| **p99** | 99% запросов быстрее | Выявляет "хвост" медленных запросов |
| **max** | Максимальное время отклика | Наихудший случай |

> **Почему среднее (average) -- плохая метрика?** Потому что оно скрывает аномалии.
> При среднем времени 200 мс у 95% пользователей может быть 100 мс, а у 5% -- 2000 мс.

### Throughput -- пропускная способность

Количество обработанных запросов в единицу времени. Обычно измеряется в **requests per second (RPS)**
или **transactions per second (TPS)**.

### Error Rate -- частота ошибок

Процент запросов, завершившихся ошибкой (HTTP 5xx, таймауты, ошибки соединения).
Допустимый уровень обычно: **< 0.1-1%** в зависимости от SLA.

### Concurrent Users -- одновременные пользователи

Количество пользователей, одновременно работающих с системой. Важно различать:
- **Concurrent users** -- активно выполняют действия
- **Active sessions** -- имеют открытую сессию, но могут быть неактивны

### Ресурсные метрики (серверная сторона)

| Метрика | Что показывает | Критический порог |
|---------|---------------|-------------------|
| **CPU Usage** | Загрузка процессора | > 80% |
| **Memory Usage** | Использование памяти | > 85% |
| **Disk I/O** | Нагрузка на диск | Зависит от системы |
| **Network I/O** | Сетевой трафик | Близко к пропускной способности |
| **Thread Count** | Количество потоков | Рост без снижения |
| **Connection Pool** | Занятые соединения БД | > 90% пула |
| **GC Activity** | Частота и длительность сборки мусора (Java) | Длинные Full GC паузы |

## SLA, SLO и SLI

### SLA -- Service Level Agreement

Формальный договор между провайдером и клиентом о гарантируемом уровне сервиса.
Нарушение SLA влечёт финансовые и юридические последствия.

**Пример SLA:**
- Доступность: 99.95% (не более 4.38 часа простоя в год)
- Время отклика API: p95 < 500 мс
- Время восстановления после сбоя: < 15 минут

### SLO -- Service Level Objective

Внутренние целевые показатели, обычно строже, чем SLA. Служат "буфером" безопасности.

### SLI -- Service Level Indicator

Фактически измеренные метрики, на основании которых оцениваются SLO и SLA.

```
SLI (что измеряем) → SLO (к чему стремимся) → SLA (что гарантируем клиенту)
     p95 = 320ms        p95 < 400ms              p95 < 500ms
```

## Анализ узких мест (Bottleneck Analysis)

Узкое место -- это компонент системы, ограничивающий общую производительность.

### Типичные узкие места

1. **База данных**
   - Медленные запросы (отсутствие индексов, N+1 проблема)
   - Блокировки (deadlocks, lock contention)
   - Недостаточный пул соединений (connection pool exhaustion)

2. **Сеть**
   - Высокая латентность между сервисами
   - Недостаточная пропускная способность
   - DNS-резолвинг

3. **Приложение**
   - Утечки памяти (memory leaks)
   - Неэффективные алгоритмы
   - Блокирующие операции в main thread
   - Неоптимальная сериализация/десериализация

4. **Инфраструктура**
   - Нехватка CPU / RAM
   - Медленный диск (HDD вместо SSD)
   - Лимиты ОС (file descriptors, max connections)

### Методология анализа

```
1. Определить симптом (медленный отклик, ошибки)
         ↓
2. Собрать метрики (CPU, память, сеть, БД)
         ↓
3. Локализовать проблему (какой компонент тормозит?)
         ↓
4. Проанализировать причину (profiling, логи, traces)
         ↓
5. Предложить и проверить решение
```

## Процесс нагрузочного тестирования

### 1. Планирование (Plan)

- Определить цели тестирования (что проверяем и зачем)
- Согласовать критерии успеха (SLO)
- Определить профиль нагрузки (сколько пользователей, какие сценарии, в каких пропорциях)
- Выбрать среду тестирования (Production-like)
- Определить инструменты

### 2. Разработка сценариев (Script)

- Написать скрипты на основе реальных пользовательских сценариев
- Параметризовать данные (CSV Data Set, переменные)
- Добавить паузы между действиями (think time) -- реальные пользователи не кликают без остановки
- Настроить ramp-up (постепенное наращивание нагрузки)

### 3. Выполнение (Execute)

- Запустить тест на подготовленной среде
- Мониторить метрики в реальном времени
- Не менять конфигурацию во время теста
- Запустить несколько итераций для подтверждения результатов

### 4. Анализ (Analyze)

- Сравнить результаты с SLO
- Найти узкие места
- Коррелировать клиентские метрики с серверными
- Определить деградацию при росте нагрузки

### 5. Отчёт (Report)

- Описать конфигурацию теста (среда, данные, профиль нагрузки)
- Представить ключевые метрики (таблицы, графики)
- Зафиксировать найденные проблемы с приоритетами
- Дать рекомендации по оптимизации
- Сделать вывод: пройден ли тест согласно критериям

## Capacity Planning -- планирование ёмкости

Capacity Planning -- это процесс определения, сколько ресурсов нужно системе для обработки
текущей и будущей нагрузки.

### Подход

1. **Определить текущую нагрузку** -- сколько пользователей / запросов сейчас
2. **Спрогнозировать рост** -- на основе бизнес-планов (рост на 50% за год)
3. **Протестировать текущие лимиты** -- стресс-тест для определения максимальной ёмкости
4. **Рассчитать необходимые ресурсы** -- сколько серверов / инстансов нужно
5. **Заложить запас** -- обычно 30-50% сверх прогнозируемой нагрузки

## Обзор инструментов

### Apache JMeter

- Самый популярный open-source инструмент
- Java-based, GUI и CLI-режимы
- Поддерживает HTTP, JDBC, JMS, FTP, SOAP, REST
- Большая экосистема плагинов
- Подробнее -- в следующем файле

### Gatling

- Scala/Java-based, скрипты как код
- Высокая производительность (неблокирующая архитектура)
- Генерирует красивые HTML-отчёты
- Хорошо подходит для CI/CD

```scala
// Пример сценария Gatling
class BasicSimulation extends Simulation {
  val httpProtocol = http.baseUrl("https://api.example.com")

  val scn = scenario("Базовый сценарий")
    .exec(http("Главная страница").get("/"))
    .pause(2)
    .exec(http("Поиск товара").get("/search?q=phone"))

  setUp(
    scn.inject(rampUsers(100).during(60)) // 100 пользователей за 60 секунд
  ).protocols(httpProtocol)
}
```

### k6

- Современный инструмент от Grafana Labs
- Скрипты на JavaScript
- Лёгкий, быстрый (написан на Go)
- Отличная интеграция с Grafana/InfluxDB

```javascript
// Пример скрипта k6
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '1m', target: 50 },  // Нарастание до 50 пользователей
    { duration: '3m', target: 50 },  // Удержание 50 пользователей 3 минуты
    { duration: '1m', target: 0 },   // Снижение до 0
  ],
};

export default function () {
  const res = http.get('https://api.example.com/products');
  check(res, {
    'статус 200': (r) => r.status === 200,
    'время отклика < 500ms': (r) => r.timings.duration < 500,
  });
  sleep(1); // Think time -- пауза между запросами
}
```

### Locust

- Python-based, код как сценарий
- Распределённое тестирование из коробки
- Web UI для мониторинга в реальном времени
- Лёгкий старт для Python-разработчиков

```python
# Пример скрипта Locust
from locust import HttpUser, task, between

class WebsiteUser(HttpUser):
    wait_time = between(1, 5)  # Пауза 1-5 сек между запросами

    @task(3)  # Вес задачи -- выполняется в 3 раза чаще
    def view_products(self):
        self.client.get("/api/products")

    @task(1)
    def create_order(self):
        self.client.post("/api/orders", json={"product_id": 1, "quantity": 2})
```

## Связь с тестированием

Нагрузочное тестирование -- неотъемлемая часть нефункционального тестирования:

- **Функциональные тесты** отвечают на вопрос "работает ли?", а нагрузочные -- "работает ли быстро и надёжно?"
- Нагрузочные тесты часто выявляют баги, которые не проявляются при одном пользователе (race conditions, deadlocks)
- Результаты нагрузочного тестирования влияют на решение о релизе (Go / No-Go)
- QA-инженер должен уметь читать и интерпретировать результаты, даже если не пишет скрипты лично
- Нагрузочное тестирование должно быть частью CI/CD pipeline (как минимум smoke performance tests)

## Типичные ошибки

1. **Тестирование на среде, отличной от Production** -- результаты будут нерелевантны.
   Среда должна быть максимально приближена к боевой.

2. **Отсутствие think time** -- скрипт без пауз между запросами создаёт нереалистичную нагрузку,
   превышающую реальную в 10-100 раз.

3. **Использование одних и тех же тестовых данных** -- все пользователи логинятся под одним
   аккаунтом, что не выявляет проблемы с кэшированием и конкурентным доступом.

4. **Игнорирование ramp-up** -- мгновенный старт всех потоков не моделирует реальность
   и может вызвать ложные ошибки при старте.

5. **Анализ только средних значений** -- среднее скрывает проблемы. Всегда смотрите на перцентили.

6. **Запуск JMeter GUI в режиме нагрузки** -- GUI потребляет много ресурсов и искажает результаты.
   Нагрузку генерируем только в CLI-режиме.

7. **Одноразовое тестирование** -- нагрузочное тестирование должно быть регулярным, а не разовым.

8. **Отсутствие baseline** -- без базовой линии нельзя понять, стала ли система быстрее или медленнее.
