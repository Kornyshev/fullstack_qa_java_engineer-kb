# Сети для QA

## Обзор

Сетевые технологии — фундамент, на котором работают все современные приложения. Для QA-инженера понимание сетей критически важно: при тестировании API нужно понимать, как работает HTTP; при отладке проблем — уметь читать сетевые логи; при тестировании безопасности — разбираться в TLS/HTTPS; при анализе производительности — понимать, где возникают задержки. Знание моделей OSI и TCP/IP, протоколов TCP/UDP, принципов работы DNS и WebSocket — обязательная часть подготовки к техническому интервью на позицию QA.

---

## Модель OSI (Open Systems Interconnection)

Модель OSI — теоретическая семиуровневая модель, описывающая, как данные передаются по сети. Каждый уровень выполняет свою функцию и взаимодействует с соседними.

| Уровень | Название | Протоколы / Технологии | Что делает | Аналогия |
|---------|----------|----------------------|-----------|----------|
| 7 | Application | HTTP, HTTPS, FTP, SMTP, DNS, WebSocket | Взаимодействие с пользователем | Письмо (содержание) |
| 6 | Presentation | SSL/TLS, JPEG, JSON, XML | Шифрование, сжатие, формат данных | Перевод на язык получателя |
| 5 | Session | NetBIOS, RPC | Управление сессиями | Диалог (кто говорит, кто слушает) |
| 4 | Transport | TCP, UDP | Доставка данных между приложениями | Курьерская служба |
| 3 | Network | IP, ICMP, ARP | Маршрутизация между сетями | Навигатор (выбор маршрута) |
| 2 | Data Link | Ethernet, Wi-Fi, MAC | Передача кадров в локальной сети | Почтальон в районе |
| 1 | Physical | Кабели, радиоволны, оптоволокно | Физическая передача битов | Дорога, по которой едет почтальон |

### Как данные проходят через уровни

```
Отправитель:                          Получатель:
┌───────────────┐                    ┌───────────────┐
│ 7. Application│ ──── HTTP ────→    │ 7. Application│
│ 6. Presentation│ ── TLS/JSON ──→   │ 6. Presentation│
│ 5. Session    │ ── Сессия ────→    │ 5. Session    │
│ 4. Transport  │ ── TCP/UDP ───→    │ 4. Transport  │
│ 3. Network    │ ── IP-пакеты ──→   │ 3. Network    │
│ 2. Data Link  │ ── Ethernet ──→    │ 2. Data Link  │
│ 1. Physical   │ ── Биты ─────→    │ 1. Physical   │
└───────────────┘                    └───────────────┘
```

**Инкапсуляция:** при отправке каждый уровень оборачивает данные своим заголовком. При получении — каждый уровень «снимает» свой заголовок.

## Модель TCP/IP

TCP/IP — практическая модель, которая реально используется в интернете. Она проще OSI и состоит из 4 уровней.

| Уровень TCP/IP | Соответствие OSI | Протоколы | Назначение |
|----------------|-----------------|-----------|-----------|
| Application | 5, 6, 7 (Session + Presentation + Application) | HTTP, HTTPS, DNS, FTP, SMTP, WebSocket | Прикладные протоколы |
| Transport | 4 (Transport) | TCP, UDP | Доставка данных между приложениями |
| Internet | 3 (Network) | IP, ICMP, ARP | Маршрутизация пакетов |
| Network Access | 1, 2 (Physical + Data Link) | Ethernet, Wi-Fi | Физическая передача |

### OSI vs TCP/IP

| Критерий | OSI | TCP/IP |
|----------|-----|--------|
| Количество уровней | 7 | 4 |
| Назначение | Теоретическая модель | Практическая модель |
| Использование | Для обучения и стандартизации | В реальных сетях |
| Разработчик | ISO | DARPA (Министерство обороны США) |

**Для QA:** на интервью могут спросить про обе модели. OSI — для теоретического понимания, TCP/IP — для практического.

## TCP vs UDP

### TCP (Transmission Control Protocol)

TCP — протокол транспортного уровня с гарантированной доставкой. Перед передачей данных устанавливает соединение (трёхстороннее рукопожатие — three-way handshake).

**Three-way handshake:**
```
Клиент                  Сервер
   │                      │
   │── SYN ──────────────→│  1. Клиент: «Хочу соединиться»
   │                      │
   │←── SYN-ACK ──────────│  2. Сервер: «Согласен, готов»
   │                      │
   │── ACK ──────────────→│  3. Клиент: «Подтверждаю»
   │                      │
   │══ Соединение установлено ══│
```

### UDP (User Datagram Protocol)

UDP — протокол без установки соединения и без гарантии доставки. Просто отправляет данные.

### Сравнительная таблица TCP vs UDP

| Критерий | TCP | UDP |
|----------|-----|-----|
| Установка соединения | Да (three-way handshake) | Нет |
| Гарантия доставки | Да (повторная отправка при потере) | Нет |
| Порядок пакетов | Гарантирован | Не гарантирован |
| Контроль потока | Да | Нет |
| Скорость | Медленнее (из-за overhead) | Быстрее |
| Размер заголовка | 20-60 байт | 8 байт |
| Применение | HTTP/HTTPS, email, файлы | Видео-стриминг, DNS, VoIP, онлайн-игры |
| Надёжность | Высокая | Низкая (данные могут потеряться) |

**Для QA:**
- Тестирование API (HTTP) — всегда TCP
- Тестирование стриминга / видеозвонков — часто UDP
- Тестирование WebSocket — TCP (начальное соединение через HTTP Upgrade)

## HTTP и HTTPS

### HTTP (HyperText Transfer Protocol)

HTTP — протокол прикладного уровня для передачи данных в вебе. Работает по модели «запрос-ответ».

**Структура HTTP-запроса:**
```
GET /api/users/123 HTTP/1.1        ← Стартовая строка (метод, путь, версия)
Host: api.example.com              ← Заголовки
Content-Type: application/json
Authorization: Bearer eyJhbG...
                                   ← Пустая строка (разделитель)
                                   ← Тело запроса (для POST/PUT)
```

**Структура HTTP-ответа:**
```
HTTP/1.1 200 OK                    ← Стартовая строка (версия, статус-код, описание)
Content-Type: application/json     ← Заголовки
Content-Length: 85
                                   ← Пустая строка
{"id": 123, "name": "John"}       ← Тело ответа
```

### HTTP-методы

| Метод | Назначение | Идемпотентный | Тело запроса |
|-------|-----------|:-------------:|:------------:|
| GET | Получить ресурс | Да | Нет |
| POST | Создать ресурс | Нет | Да |
| PUT | Полностью обновить ресурс | Да | Да |
| PATCH | Частично обновить ресурс | Нет | Да |
| DELETE | Удалить ресурс | Да | Нет (обычно) |
| HEAD | Получить только заголовки | Да | Нет |
| OPTIONS | Получить допустимые методы | Да | Нет |

**Идемпотентность** — повторный вызов даёт тот же результат. Важно для QA: повторный `PUT` не должен создавать дубликат, а повторный `POST` может.

### Коды ответов HTTP

| Группа | Диапазон | Описание | Примеры |
|--------|---------|----------|---------|
| 1xx | 100-199 | Информационные | 100 Continue |
| 2xx | 200-299 | Успех | 200 OK, 201 Created, 204 No Content |
| 3xx | 300-399 | Перенаправления | 301 Moved Permanently, 302 Found, 304 Not Modified |
| 4xx | 400-499 | Ошибки клиента | 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 409 Conflict |
| 5xx | 500-599 | Ошибки сервера | 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable |

### HTTPS (HTTP Secure)

HTTPS = HTTP + TLS (Transport Layer Security). Данные шифруются, что защищает от перехвата и подмены.

**Что обеспечивает HTTPS:**
- **Шифрование** — данные не могут быть прочитаны третьей стороной
- **Целостность** — данные не могут быть изменены при передаче
- **Аутентификация** — подтверждение подлинности сервера (через сертификат)

## TLS Handshake

TLS Handshake — процесс установки защищённого соединения между клиентом и сервером.

```
Клиент                              Сервер
   │                                   │
   │── ClientHello ───────────────────→│  1. Версия TLS, поддерживаемые шифры
   │                                   │
   │←── ServerHello ──────────────────│  2. Выбранный шифр, сертификат сервера
   │←── Certificate ──────────────────│
   │←── ServerHelloDone ─────────────│
   │                                   │
   │   (Клиент проверяет сертификат)   │  3. Проверка подлинности сервера
   │                                   │
   │── ClientKeyExchange ────────────→│  4. Обмен ключами (асимметричное шифрование)
   │── ChangeCipherSpec ─────────────→│
   │── Finished ─────────────────────→│  5. Переход на симметричное шифрование
   │                                   │
   │←── ChangeCipherSpec ────────────│
   │←── Finished ────────────────────│
   │                                   │
   │══ Шифрованное соединение (симметричное) ══│
```

**Упрощённо:**
1. Клиент и сервер договариваются о методе шифрования
2. Сервер предъявляет сертификат (подтверждение подлинности)
3. Обмен ключами через асимметричное шифрование (медленное, но безопасное)
4. Дальнейшая передача данных — симметричное шифрование (быстрое)

**Для QA:**
- При тестировании API на HTTPS-endpoint нужно понимать, почему бывают ошибки сертификатов
- В тестах часто отключают проверку сертификатов (`trustAllCertificates`) — это допустимо только в тестовом окружении
- Ошибки TLS handshake могут быть причиной failed-тестов (просроченный сертификат, несовпадение домена)

## DNS (Domain Name System)

DNS — система, преобразующая доменное имя (например, `api.example.com`) в IP-адрес (например, `93.184.216.34`).

### Процесс DNS-разрешения (DNS Resolution)

```
1. Браузер: "Какой IP у api.example.com?"
2. Проверка локального кэша (браузер → ОС → файл hosts)
3. Если нет в кэше → запрос к DNS-серверу провайдера
4. Если провайдер не знает → запрос к корневому DNS-серверу
5. Корневой сервер → направляет к серверу зоны .com
6. Сервер .com → направляет к серверу example.com
7. Сервер example.com → возвращает IP: 93.184.216.34
8. Результат кэшируется (TTL — время жизни записи)
```

```
Клиент → DNS-кэш (локальный) → DNS-рекурсор → Root DNS → .com DNS → example.com DNS
                                                                         │
                                                               IP: 93.184.216.34
```

### Типы DNS-записей

| Тип | Назначение | Пример |
|-----|-----------|--------|
| A | Домен → IPv4-адрес | `example.com → 93.184.216.34` |
| AAAA | Домен → IPv6-адрес | `example.com → 2606:2800:220:1:...` |
| CNAME | Алиас для другого домена | `www.example.com → example.com` |
| MX | Почтовый сервер | `example.com → mail.example.com` |
| NS | DNS-сервер для домена | `example.com → ns1.example.com` |
| TXT | Текстовая запись (верификация) | SPF, DKIM записи |

**Для QA:**
- DNS-проблемы могут вызывать timeout в тестах (домен не резолвится)
- При переключении между окружениями (staging → production) DNS может кэшировать старый IP
- В файле `/etc/hosts` можно вручную указать IP для домена — полезно для тестирования

## Порты

Порт — числовой идентификатор (0-65535), определяющий конкретное приложение или сервис на машине. IP-адрес определяет машину, порт — конкретный сервис на ней.

### Well-known порты (0-1023)

| Порт | Протокол | Описание |
|------|---------|----------|
| 20, 21 | FTP | Передача файлов |
| 22 | SSH | Безопасный удалённый доступ |
| 25 | SMTP | Отправка email |
| 53 | DNS | Разрешение доменных имён |
| 80 | HTTP | Веб-трафик (нешифрованный) |
| 443 | HTTPS | Веб-трафик (шифрованный) |
| 3306 | MySQL | База данных MySQL |
| 5432 | PostgreSQL | База данных PostgreSQL |
| 5672 | AMQP | RabbitMQ (message broker) |
| 6379 | Redis | Кэш / хранилище |
| 8080 | HTTP (alt) | Альтернативный HTTP (часто для dev-серверов) |
| 8443 | HTTPS (alt) | Альтернативный HTTPS |
| 9092 | Kafka | Apache Kafka (message broker) |
| 27017 | MongoDB | База данных MongoDB |

**Для QA:**
- При запуске тестов локально приложение обычно на `localhost:8080`
- Если порт занят — тесты не запустятся (ошибка `Address already in use`)
- При тестировании микросервисов каждый сервис работает на своём порту

## WebSocket

WebSocket — протокол полнодуплексной связи (full-duplex) поверх TCP. В отличие от HTTP (запрос-ответ), WebSocket позволяет серверу отправлять данные клиенту без запроса.

### Установка WebSocket-соединения

WebSocket начинается как обычный HTTP-запрос с заголовком `Upgrade`:

```
Клиент → Сервер:
GET /chat HTTP/1.1
Host: server.example.com
Upgrade: websocket                    ← Запрос на переход к WebSocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==

Сервер → Клиент:
HTTP/1.1 101 Switching Protocols      ← Подтверждение перехода
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

═══ WebSocket-соединение установлено ═══
```

### HTTP vs WebSocket

| Критерий | HTTP | WebSocket |
|----------|------|-----------|
| Модель | Запрос-ответ | Полнодуплексная |
| Инициатор | Только клиент | Обе стороны |
| Соединение | Создаётся для каждого запроса* | Постоянное |
| Overhead | Заголовки в каждом запросе | Минимальный после handshake |
| Применение | REST API, веб-страницы | Чаты, уведомления, real-time данные |

\* — с HTTP/1.1 Keep-Alive соединение может переиспользоваться, но модель остаётся «запрос-ответ».

**Для QA:**
- WebSocket тестируется иначе, чем REST API — нужны специальные инструменты
- Важно тестировать: установку соединения, отправку/получение сообщений, разрыв соединения, переподключение
- Инструменты: Postman (поддерживает WebSocket), WebSocket-клиент в браузере DevTools, библиотеки для автотестов

## Полный путь HTTP-запроса (от браузера до сервера)

Разберём, что происходит, когда QA вводит `https://api.example.com/users` в браузере:

```
1. DNS Resolution
   Браузер → DNS → IP-адрес сервера (93.184.216.34)

2. TCP Connection (Three-way handshake)
   Клиент ↔ Сервер: SYN → SYN-ACK → ACK

3. TLS Handshake
   Клиент ↔ Сервер: обмен ключами, установка шифрования

4. HTTP Request
   GET /users HTTP/1.1
   Host: api.example.com

5. Сервер обрабатывает запрос
   Маршрутизация → контроллер → бизнес-логика → БД → ответ

6. HTTP Response
   HTTP/1.1 200 OK
   Content-Type: application/json
   [{"id": 1, "name": "John"}, ...]

7. Рендеринг в браузере
   Парсинг JSON → отображение данных
```

Понимание этого пути помогает QA определить, **на каком этапе** возникла проблема: DNS не резолвится? TCP-соединение не устанавливается? TLS-ошибка? Сервер возвращает 500?

## Типичные ошибки

1. **Путаница OSI и TCP/IP** — на интервью важно чётко разделять: OSI — 7 уровней (теоретическая), TCP/IP — 4 уровня (практическая).

2. **Утверждение, что «HTTP всегда на порту 80»** — HTTP **по умолчанию** на порту 80, но может работать на любом порту (8080, 3000 и др.).

3. **Непонимание разницы 401 vs 403** — `401 Unauthorized` означает «не аутентифицирован» (нет токена / невалидный токен); `403 Forbidden` означает «аутентифицирован, но нет прав».

4. **Игнорирование DNS-кэширования** — после смены окружения тесты могут ходить на старый IP. Нужно сбросить DNS-кэш или дождаться истечения TTL.

5. **Отключение проверки сертификатов в production** — допустимо только в тестовом окружении. В production это серьёзная уязвимость.

6. **Непонимание идемпотентности** — повторный POST может создать дубликат ресурса, а PUT — нет. Это важно при тестировании API.

7. **Путаница TCP vs HTTP** — TCP — протокол транспортного уровня, HTTP — прикладного. HTTP работает **поверх** TCP.

8. **Непонимание WebSocket vs polling** — WebSocket поддерживает постоянное соединение, а long polling — это повторные HTTP-запросы. Разные механизмы, разные подходы к тестированию.

## Практические задания

### Задание 1: Анализ HTTP-трафика

1. Откройте браузер DevTools (вкладка Network)
2. Загрузите любой сайт и проанализируйте:
   - Сколько HTTP-запросов было сделано?
   - Какие методы использовались?
   - Какие статус-коды вернулись?
   - Сколько времени заняли DNS lookup, TLS handshake, загрузка ресурсов?

### Задание 2: Работа с DNS

1. Используйте команду `nslookup` или `dig` для разрешения домена
2. Проверьте, какие типы записей (A, CNAME, MX) есть у домена
3. Измените файл `/etc/hosts`, чтобы домен указывал на другой IP
4. Убедитесь, что браузер использует ваш IP вместо реального

### Задание 3: TCP vs UDP в действии

1. Используя `telnet` или `nc` (netcat), подключитесь к порту 80 любого сервера
