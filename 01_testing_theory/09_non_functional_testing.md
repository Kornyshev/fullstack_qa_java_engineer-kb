# Нефункциональное тестирование

## Обзор

Нефункциональное тестирование (Non-functional Testing) проверяет **как** система работает,
в отличие от функционального тестирования, которое проверяет **что** система делает.
Это тестирование характеристик качества: производительности, безопасности, удобства использования,
надёжности, совместимости и других атрибутов, определённых стандартом ISO 25010.

Нефункциональное тестирование часто недооценивается командами, что приводит к проблемам в production:
медленная работа под нагрузкой, уязвимости безопасности, недоступность для людей с ограниченными
возможностями. QA-инженер должен уметь обосновать необходимость нефункционального тестирования
и знать основные подходы и инструменты.

## Performance Testing (Тестирование производительности)

Тестирование производительности оценивает поведение системы при различных уровнях нагрузки.
Цель — убедиться, что система соответствует требованиям по скорости, стабильности и масштабируемости.

### Типы Performance Testing

#### Load Testing (Нагрузочное тестирование)

Проверяет поведение системы при **ожидаемой (нормальной) нагрузке**.

**Цель:** убедиться, что система справляется с типичным количеством пользователей и транзакций.

**Пример:** Интернет-магазин рассчитан на 1 000 одновременных пользователей. Load test проверяет,
что при 1 000 пользователях response time остаётся в пределах SLA (< 2 секунд).

#### Stress Testing (Стресс-тестирование)

Проверяет поведение системы при нагрузке, **превышающей нормальную**, вплоть до точки отказа.

**Цель:** найти предел прочности системы и убедиться, что она корректно деградирует (graceful degradation).

**Пример:** Постепенно увеличиваем нагрузку с 1 000 до 5 000 пользователей и наблюдаем, при каком
количестве система начинает отвечать ошибками или перестаёт отвечать вовсе.

#### Spike Testing (Тестирование всплесков)

Проверяет поведение системы при **резком скачке нагрузки**.

**Цель:** оценить, как система реагирует на внезапный наплыв пользователей и как быстро восстанавливается.

**Пример:** Магазин запустил акцию «Чёрная пятница» — за 5 минут нагрузка выросла с 500 до 10 000 пользователей.

#### Soak / Endurance Testing (Тестирование стабильности)

Проверяет поведение системы при **нормальной нагрузке в течение длительного времени**.

**Цель:** выявить проблемы, проявляющиеся со временем: memory leaks, connection pool exhaustion,
деградация производительности, рост дискового пространства.

**Пример:** Система работает под нагрузкой 500 пользователей непрерывно 48 часов.
Отслеживаем memory consumption, response time, error rate.

#### Scalability Testing (Тестирование масштабируемости)

Проверяет, насколько эффективно система **масштабируется** при увеличении ресурсов.

**Цель:** определить, даёт ли добавление CPU / RAM / инстансов пропорциональный прирост производительности.

**Пример:** При 1 инстансе система обрабатывает 100 RPS. При 2 инстансах — 180 RPS (а не ожидаемые 200).

### Ключевые метрики Performance Testing

| Метрика | Описание | Типичный порог |
|---------|----------|----------------|
| **Response Time** | Время от отправки запроса до получения ответа | < 2 сек для web-страниц |
| **Throughput** | Количество запросов, обработанных за единицу времени (RPS/TPS) | Зависит от SLA |
| **Error Rate** | Процент запросов, завершившихся ошибкой | < 1% при нормальной нагрузке |
| **Concurrent Users** | Количество одновременных пользователей | Определяется бизнес-требованиями |
| **Latency** | Задержка сети от клиента до сервера | < 100ms для внутренней сети |
| **CPU Utilization** | Использование процессора | < 70% при нормальной нагрузке |
| **Memory Utilization** | Использование оперативной памяти | < 80% при нормальной нагрузке |
| **Percentiles (P50, P90, P99)** | Распределение response time | P99 < 5 сек |

### Инструменты Performance Testing

#### Apache JMeter

Самый популярный open-source инструмент для нагрузочного тестирования.

**Особенности:**
- GUI для создания тест-планов + CLI для запуска
- Поддержка HTTP, JDBC, JMS, SOAP, REST, FTP
- Плагины для расширения функциональности
- Распределённое тестирование (master-slave)
- Интеграция с CI/CD

```xml
<!-- Пример структуры JMeter Test Plan -->
<TestPlan>
    <!-- Группа потоков: моделирует 100 виртуальных пользователей -->
    <ThreadGroup>
        <!-- 100 виртуальных пользователей -->
        <num_threads>100</num_threads>
        <!-- Время разгона: 60 секунд -->
        <ramp_time>60</ramp_time>
        <!-- Количество итераций: бесконечно (ограничено по времени) -->
        <loops>-1</loops>
        <!-- Длительность: 300 секунд -->
        <duration>300</duration>

        <HTTPSampler>
            <!-- GET-запрос к главной странице -->
            <method>GET</method>
            <path>/api/products</path>
        </HTTPSampler>
    </ThreadGroup>
</TestPlan>
```

#### Gatling

Высокопроизводительный инструмент на Scala с поддержкой DSL для описания сценариев.

**Особенности:**
- Код-ориентированный подход (Scala DSL)
- Отличные HTML-отчёты из коробки
- Эффективное использование ресурсов (Akka, Netty)
- Хорошо подходит для CI/CD
- Поддержка HTTP/2 и WebSocket

```scala
// Пример сценария Gatling
class BasicSimulation extends Simulation {

  // Настройка HTTP-протокола: базовый URL и заголовки
  val httpProtocol = http
    .baseUrl("https://api.example.com")
    .acceptHeader("application/json")

  // Сценарий: поиск и просмотр товара
  val searchScenario = scenario("Search Products")
    .exec(
      http("Get Products")  // Запрос списка товаров
        .get("/api/products")
        .check(status.is(200))
    )
    .pause(2)  // Пауза 2 секунды (имитация пользователя)
    .exec(
      http("Get Product Details")  // Запрос деталей товара
        .get("/api/products/1")
        .check(status.is(200))
    )

  // Настройка нагрузки: 100 пользователей за 60 секунд, работа 5 минут
  setUp(
    searchScenario.inject(
      rampUsers(100).during(60.seconds)
    )
  ).protocols(httpProtocol)
   .assertions(
     global.responseTime.percentile(99).lt(3000),  // P99 < 3 секунд
     global.successfulRequests.percent.gt(99)       // > 99% успешных запросов
   )
}
```

#### k6 (Grafana k6)

Современный инструмент для нагрузочного тестирования с JavaScript API.

**Особенности:**
- Скрипты на JavaScript (ES6+)
- Низкое потребление ресурсов (Go-runtime)
- Встроенная интеграция с Grafana / InfluxDB
- Cloud-вариант (k6 Cloud)

#### Сравнение инструментов

| Характеристика | JMeter | Gatling | k6 |
|---|---|---|---|
| Язык скриптов | XML / GUI | Scala | JavaScript |
| Потребление ресурсов | Высокое | Среднее | Низкое |
| Кривая обучения | Низкая (GUI) | Средняя | Низкая |
| CI/CD интеграция | Хорошая | Отличная | Отличная |
| Протоколы | Множество | HTTP, WebSocket | HTTP, WebSocket, gRPC |
| Отчёты | Плагины | Встроенные HTML | Grafana / Cloud |
| Распределённая нагрузка | Да | Да (Gatling Enterprise) | Да (k6 Cloud) |

## Security Testing (Тестирование безопасности)

Тестирование безопасности выявляет уязвимости, которые могут привести к несанкционированному доступу,
утечке данных или нарушению работы системы.

### OWASP Top 10 (2021)

OWASP (Open Web Application Security Project) публикует список 10 самых критичных рисков безопасности
для веб-приложений.

| # | Категория | Описание | Пример |
|---|-----------|----------|--------|
| A01 | **Broken Access Control** | Нарушение контроля доступа | Пользователь может видеть данные других пользователей, изменив ID в URL |
| A02 | **Cryptographic Failures** | Проблемы с шифрованием | Пароли хранятся в plaintext; HTTP вместо HTTPS |
| A03 | **Injection** | SQL, NoSQL, OS, LDAP injection | `' OR 1=1 --` в поле логина |
| A04 | **Insecure Design** | Архитектурные проблемы безопасности | Отсутствие rate limiting на форме входа |
| A05 | **Security Misconfiguration** | Неправильная конфигурация | Доступные стандартные учётные записи; verbose error messages |
| A06 | **Vulnerable Components** | Уязвимые зависимости | Использование библиотеки с известной CVE |
| A07 | **Auth Failures** | Проблемы аутентификации и идентификации | Слабая парольная политика; отсутствие brute-force защиты |
| A08 | **Data Integrity Failures** | Нарушение целостности данных | Десериализация ненадёжных данных; незащищённый CI/CD pipeline |
| A09 | **Logging & Monitoring Failures** | Недостаточное логирование | Неудачные попытки входа не логируются |
| A10 | **SSRF** | Server-Side Request Forgery | Сервер выполняет запросы к внутренним ресурсам по URL из пользовательского ввода |

### Основные инструменты Security Testing

| Инструмент | Тип | Описание |
|---|---|---|
| **OWASP ZAP** | DAST (Dynamic) | Бесплатный прокси-сканер, находит уязвимости в работающем приложении |
| **Burp Suite** | DAST | Профессиональный инструмент для penetration testing |
| **SonarQube** | SAST (Static) | Анализ исходного кода на уязвимости |
| **Snyk / Dependabot** | SCA | Проверка зависимостей на известные уязвимости |
| **OWASP Dependency-Check** | SCA | Проверка зависимостей (open-source) |

### Что QA может проверить без специализированных знаний

- **Аутентификация:** слабые пароли, отсутствие блокировки после неудачных попыток
- **Авторизация:** доступ к чужим данным при изменении ID в URL / API
- **Input validation:** SQL injection, XSS через поля ввода
- **HTTPS:** все ли страницы работают через HTTPS
- **Sensitive data:** пароли / токены не отображаются в URL, логах, response
- **Error handling:** сообщения об ошибках не раскрывают внутреннюю информацию (stack trace, пути файлов)

## Usability Testing (Тестирование удобства использования)

Usability Testing оценивает, насколько легко и эффективно пользователи могут взаимодействовать
с продуктом для достижения своих целей.

### Методы

| Метод | Описание |
|-------|----------|
| **Heuristic Evaluation** | Эксперт оценивает интерфейс по набору эвристик (10 эвристик Нильсена) |
| **User Testing** | Реальные пользователи выполняют задачи, наблюдатель фиксирует трудности |
| **A/B Testing** | Сравнение двух вариантов интерфейса по метрикам эффективности |
| **Think Aloud** | Пользователь озвучивает свои мысли при работе с интерфейсом |
| **Card Sorting** | Пользователи группируют элементы для определения оптимальной навигации |

### 10 эвристик Нильсена

1. **Visibility of system status** — система сообщает пользователю, что происходит (loading spinner, progress bar)
2. **Match between system and real world** — язык системы понятен пользователю, а не разработчику
3. **User control and freedom** — возможность отмены действий (Undo, Cancel)
4. **Consistency and standards** — единообразие элементов и поведения
5. **Error prevention** — предотвращение ошибок (confirmation dialogs, validation)
6. **Recognition rather than recall** — подсказки и видимые элементы вместо необходимости запоминать
7. **Flexibility and efficiency** — горячие клавиши для опытных пользователей
8. **Aesthetic and minimalist design** — только необходимая информация на экране
9. **Help users recognize, diagnose, and recover from errors** — понятные сообщения об ошибках
10. **Help and documentation** — доступная документация и подсказки

### Метрики Usability

| Метрика | Описание |
|---------|----------|
| **Task Success Rate** | Процент пользователей, успешно выполнивших задачу |
| **Time on Task** | Время выполнения задачи |
| **Error Rate** | Количество ошибок при выполнении задачи |
| **Learnability** | Улучшение показателей при повторном выполнении |
| **SUS Score** | System Usability Scale — стандартизированный опросник (0–100) |

## Accessibility Testing (Тестирование доступности)

Accessibility Testing проверяет, что продуктом могут пользоваться **люди с ограниченными возможностями**:
нарушениями зрения, слуха, моторики, когнитивными особенностями.

### WCAG (Web Content Accessibility Guidelines)

Стандарт W3C для обеспечения доступности веб-контента. Текущая версия — WCAG 2.2.

**Уровни соответствия:**
- **A** — минимальный уровень (обязательный)
- **AA** — рекомендуемый уровень (требуется по законодательству многих стран)
- **AAA** — максимальный уровень (для специализированных ресурсов)

### Принципы WCAG (POUR)

| Принцип | Описание | Примеры |
|---------|----------|---------|
| **Perceivable** | Информация воспринимаема | Alt-текст для изображений; субтитры для видео |
| **Operable** | Интерфейс управляем | Навигация с клавиатуры; достаточный размер кликабельных элементов |
| **Understandable** | Контент понятен | Ясный язык; предсказуемое поведение; помощь при ошибках ввода |
| **Robust** | Контент совместим с assistive technologies | Правильная HTML-семантика; ARIA-атрибуты |

### Что проверять

- **Клавиатурная навигация** — все функции доступны без мыши (Tab, Enter, Escape)
- **Screen reader** — контент читается корректно (VoiceOver, NVDA, JAWS)
- **Контрастность** — соотношение контраста текста и фона (минимум 4.5:1 для AA)
- **Alt-тексты** — все информативные изображения имеют описание
- **Формы** — label связаны с input, ошибки озвучиваются
- **Focus indicator** — видимый индикатор фокуса при навигации клавиатурой
- **Zoom** — контент читаем при увеличении до 200%

### Инструменты

| Инструмент | Описание |
|---|---|
| **axe DevTools** | Расширение для браузера, автоматическая проверка WCAG |
| **Lighthouse** | Встроен в Chrome DevTools, включает аудит accessibility |
| **WAVE** | Онлайн-инструмент для проверки доступности |
| **Pa11y** | CLI-инструмент для автоматизации проверок в CI/CD |
| **Screen Readers** | VoiceOver (macOS), NVDA (Windows) — ручное тестирование |

## Reliability Testing (Тестирование надёжности)

Reliability Testing оценивает способность системы работать **без сбоев** в течение заданного времени
при определённых условиях.

### Метрики надёжности

| Метрика | Описание | Формула |
|---------|----------|---------|
| **MTBF** (Mean Time Between Failures) | Среднее время между отказами | Общее время работы / Количество отказов |
| **MTTR** (Mean Time To Recovery) | Среднее время восстановления | Общее время простоя / Количество отказов |
| **Availability** | Доступность системы | MTBF / (MTBF + MTTR) × 100% |
| **Failure Rate** | Частота отказов | Количество отказов / Общее время работы |

### Уровни доступности (SLA)

| Уровень | Допустимый простой в год | Допустимый простой в месяц |
|---------|--------------------------|---------------------------|
| 99% (two nines) | 3.65 дня | 7.31 часа |
| 99.9% (three nines) | 8.77 часа | 43.8 минуты |
| 99.99% (four nines) | 52.6 минуты | 4.38 минуты |
| 99.999% (five nines) | 5.26 минуты | 26.3 секунды |

### Chaos Engineering

Современный подход к тестированию надёжности, популяризированный Netflix.

**Принцип:** намеренное внесение сбоев в production / staging для проверки устойчивости системы.

**Инструменты:**
- **Chaos Monkey** (Netflix) — случайно отключает инстансы
- **Gremlin** — платформа для Chaos Engineering
- **Litmus** — Chaos Engineering для Kubernetes

## Compatibility Testing (Тестирование совместимости)

Compatibility Testing проверяет, что приложение корректно работает в различных средах.

### Виды совместимости

| Вид | Что проверяем | Примеры |
|-----|---------------|---------|
| **Browser Compatibility** | Работа в разных браузерах | Chrome, Firefox, Safari, Edge |
| **OS Compatibility** | Работа на разных ОС | Windows, macOS, Linux, Android, iOS |
| **Device Compatibility** | Работа на разных устройствах | Desktop, tablet, mobile различных размеров |
| **Resolution Compatibility** | Корректное отображение при различных разрешениях | 1920×1080, 1366×768, 375×667 (iPhone) |
| **Network Compatibility** | Работа при разных условиях сети | Wi-Fi, 4G, 3G, offline |
| **Backward Compatibility** | Совместимость с предыдущими версиями | Новый API поддерживает запросы от старых клиентов |
| **Forward Compatibility** | Совместимость с будущими версиями | Старый клиент может работать с новым API |

### Инструменты

| Инструмент | Описание |
|---|---|
| **BrowserStack** | Облачная платформа для кроссбраузерного тестирования |
| **Sauce Labs** | Тестирование на реальных устройствах и браузерах |
| **LambdaTest** | Облачное кроссбраузерное тестирование |
| **Chrome DevTools Device Mode** | Эмуляция мобильных устройств в браузере |
| **Responsively** | Одновременный просмотр на нескольких размерах экрана |

## Когда QA должен отстаивать нефункциональное тестирование

### Сигналы, что нефункциональное тестирование необходимо

1. **Рост пользовательской базы** — система, работавшая для 100 пользователей, может не выдержать 10 000
2. **Работа с персональными данными** — требования GDPR, ФЗ-152 делают security testing обязательным
3. **Публичный продукт** — доступность (accessibility) может быть юридическим требованием
4. **Жалобы пользователей** — «сайт тормозит», «неудобно пользоваться» — сигнал для usability и performance testing
5. **Перед крупными релизами** — нагрузочное тестирование обязательно перед Black Friday, запуском рекламы
6. **Изменение инфраструктуры** — миграция на новый сервер / облако требует performance baseline
7. **Регуляторные требования** — финтех, медтех, госсектор имеют строгие требования к безопасности и доступности

### Как обосновать перед менеджментом

- **Финансовые потери:** 1 секунда задержки загрузки = -7% конверсии (по данным различных исследований)
- **Репутационные риски:** утечка данных, недоступность сервиса
- **Юридические риски:** штрафы за несоответствие GDPR, ADA, Section 508
- **Стоимость исправления:** баг производительности в production стоит в 10–100 раз дороже, чем на этапе тестирования
- **Конкурентное преимущество:** быстрый и удобный продукт удерживает пользователей

## Связь с тестированием

- Нефункциональное тестирование **дополняет** функциональное — оба типа необходимы для качественного продукта
- Performance testing должен быть частью **CI/CD pipeline** (хотя бы baseline-тесты)
- Security testing интегрируется через **SAST/DAST** на этапе сборки и деплоя
- Accessibility testing можно **частично автоматизировать** (axe, Pa11y) в CI/CD
- Нефункциональные требования должны быть описаны в **acceptance criteria** user stories
- Результаты нефункционального тестирования влияют на **Go / No Go decision** при релизе
- QA-инженер должен участвовать в определении **SLA** и **NFR** (Non-Functional Requirements)

## Типичные ошибки

1. **Откладывание на потом** — «Сначала сделаем функционал, потом потестируем нагрузку» → нагрузку никогда не тестируют
2. **Тестирование на неправильном окружении** — performance test на dev-сервере бессмысленен, если production в 10 раз мощнее (или слабее)
3. **Нереалистичные сценарии нагрузки** — 100% пользователей одновременно делают один и тот же запрос — так не бывает
4. **Игнорирование security testing** — «Нас никто не взломает, мы маленькие» — атаки автоматизированы и бьют по всем
5. **Путать accessibility и usability** — accessibility — это возможность использования для людей с ограничениями, usability — удобство для всех
6. **Тестирование только одного браузера** — «У меня в Chrome работает» — а 20% пользователей на Safari
7. **Отсутствие baseline** — без замера начальных показателей невозможно оценить, стало ли лучше или хуже
8. **Ручное нагрузочное тестирование** — «Позовём 50 коллег одновременно нажать кнопку» — нерелевантно
9. **Забывают про Soak testing** — memory leak проявляется через 24+ часов работы, короткий тест его не выявит
10. **Нет NFR в требованиях** — «Должно работать быстро» — это не требование, нужны конкретные цифры

## Практические задания

### Задание 1: Составьте план Performance Testing
Для интернет-магазина с 50 000 DAU (daily active users) составьте план нагрузочного тестирования:
- Определите типы тестов (load, stress, spike, soak)
- Укажите целевые метрики и пороговые значения
- Опишите сценарии (какие действия пользователей моделировать)
- Выберите инструмент и обоснуйте выбор

### Задание 2: Security Checklist
Составьте чеклист security testing для веб-приложения с авторизацией, платежами и загрузкой файлов.
Минимум 15 проверок, сгруппированных по категориям OWASP.

### Задание 3: Accessibility Audit
Выберите любой публичный сайт и проведите базовый accessibility audit:
- Проверьте с помощью Lighthouse / axe DevTools
- Попробуйте навигацию только клавиатурой
- Проверьте контрастность текста
- Оцените alt-тексты изображений
- Составьте отчёт с найденными проблемами и рекомендациями

### Задание 4: Сравнение инструментов
Напишите простой нагрузочный тест для публичного API (например, JSONPlaceholder)
на двух инструментах: JMeter и k6 (или Gatling). Сравните:
- Удобство написания
- Читаемость скрипта
- Качество отчётов
- Интеграция с CI/CD

### Задание 5: NFR для стартапа
Стартап запускает мобильное приложение для доставки еды. Сформулируйте Non-Functional Requirements:
- Performance (response time, throughput, concurrent users)
- Security (данные платёжных карт, персональные данные)
- Accessibility (минимальный уровень WCAG)
- Compatibility (устройства, ОС, браузеры)
- Reliability (SLA, допустимый downtime)
