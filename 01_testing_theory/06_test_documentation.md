# Тестовая документация

## Обзор

Тестовая документация -- совокупность артефактов, создаваемых в процессе тестирования:
от стратегии и планов до конкретных тест-кейсов и отчётов. Качество документации напрямую
влияет на эффективность тестирования, передачу знаний в команде и прозрачность процесса.

На интервью вопросы о тестовой документации проверяют понимание процесса тестирования
в целом: кто, когда и зачем создаёт документы, какой уровень детализации уместен,
как организовать тест-кейсы, когда достаточно чек-листа.

В этом разделе разобраны ключевые артефакты тестовой документации: test plan, test strategy,
test case, checklist, test suite -- с шаблонами, примерами и типичными ошибками.

---

## 1. Test Strategy (Стратегия тестирования)

### Что это

**Test strategy** -- высокоуровневый документ, описывающий общий подход к тестированию
на уровне проекта или организации. Создаётся один раз и редко меняется.

### Кто создаёт

- QA Lead / Test Manager
- При участии Project Manager и Tech Lead

### Что включает

| Раздел                        | Описание                                              |
|-------------------------------|-------------------------------------------------------|
| Scope (Область)               | Что тестируем и что не тестируем                      |
| Test levels                   | Unit, Integration, System, Acceptance                 |
| Test types                    | Functional, Performance, Security, Usability          |
| Test approach                 | Risk-based, requirements-based, exploratory           |
| Environment strategy          | Тестовые окружения, управление данными                |
| Tool strategy                 | Инструменты (Jira, Selenium, Allure и т.д.)          |
| Defect management             | Процесс работы с дефектами, severity/priority         |
| Entry/Exit criteria           | Критерии начала и завершения тестирования             |
| Roles and responsibilities    | Кто за что отвечает                                   |
| Risks and mitigations         | Риски и меры по их снижению                           |
| Metrics                       | Метрики качества (defect density, pass rate и т.д.)  |

### Пример Entry/Exit Criteria

**Entry criteria (Критерии входа):**
- Требования утверждены и доступны
- Тестовое окружение развёрнуто и стабильно
- Smoke-тесты пройдены
- Тестовые данные подготовлены

**Exit criteria (Критерии выхода):**
- Все critical и major дефекты исправлены
- Regression suite пройден на 95%+
- Все запланированные тест-кейсы выполнены
- Нет open blockers
- Тест-отчёт подготовлен и согласован

## 2. Test Plan (Тест-план)

### Что это

**Test plan** -- документ, описывающий объём, подход, ресурсы и расписание тестирования
для конкретного релиза, спринта или фичи. Более детальный и конкретный, чем test strategy.

### Кто создаёт

- QA Lead / Senior QA
- Для каждого релиза или крупной фичи

### Структура тест-плана (по IEEE 829)

```
1. Идентификатор тест-плана
2. Введение
3. Объекты тестирования (test items)
4. Тестируемые функции (features to be tested)
5. Функции, не подлежащие тестированию (features not to be tested)
6. Подход к тестированию (approach)
7. Критерии прохождения/непрохождения (pass/fail criteria)
8. Критерии приостановки и возобновления
9. Результаты тестирования (test deliverables)
10. Задачи тестирования (testing tasks)
11. Окружение (environment needs)
12. Ответственные (responsibilities)
13. Расписание (schedule)
14. Риски и непредвиденные обстоятельства
15. Утверждения (approvals)
```

### Пример тест-плана (сокращённый)

```
=== ТЕСТ-ПЛАН: Релиз 2.5.0 (Модуль "Корзина") ===

1. ВВЕДЕНИЕ
   Тестирование нового функционала корзины покупок:
   промокоды, пересчёт стоимости, лимиты.

2. ОБЪЁМ ТЕСТИРОВАНИЯ
   В скоупе:
   - Добавление/удаление товаров
   - Применение промокодов
   - Пересчёт стоимости при изменении количества
   - Лимит 99 единиц одного товара

   Вне скоупа:
   - Оплата (не менялась)
   - Доставка (не менялась)
   - Мобильное приложение (отдельный план)

3. ПОДХОД
   - Functional testing: ручное + автоматизация
   - Regression: автоматизированный suite (Jenkins)
   - Exploratory: 2 сессии по 90 минут
   - Performance: нагрузочное тестирование (до 1000 RPS)

4. ОКРУЖЕНИЕ
   - Staging: staging.example.com
   - БД: PostgreSQL 15 (копия production)
   - Браузеры: Chrome 120+, Firefox 120+, Safari 17+

5. РАСПИСАНИЕ
   - 18-19 марта: Smoke + основное функциональное тестирование
   - 20 марта: Regression suite + exploratory
   - 21 марта: Bugfix verification + финальная регрессия
   - 22 марта: Sign-off и релиз

6. РИСКИ
   - Нестабильность staging-окружения -> запасной сервер
   - Отсутствие QA #2 (отпуск) -> перераспределение задач

7. КРИТЕРИИ ВЫХОДА
   - 0 critical/major open defects
   - Regression pass rate >= 95%
   - Все acceptance criteria проверены
```

### Test Strategy vs Test Plan

| Критерий         | Test Strategy              | Test Plan                    |
|------------------|----------------------------|------------------------------|
| Уровень          | Проект / организация       | Релиз / фича / спринт       |
| Создаётся        | Один раз                   | Для каждого релиза           |
| Детализация      | Высокоуровневая            | Конкретная                   |
| Автор            | QA Lead / Test Manager     | QA Lead / Senior QA          |
| Содержит         | Общий подход, инструменты  | Расписание, ресурсы, скоуп   |
| Меняется         | Редко                      | Каждый релиз                 |
| Аналогия         | Конституция                | Закон                        |

## 3. Test Case (Тест-кейс)

### Что это

**Test case** -- документ, описывающий конкретный тестовый сценарий: входные данные,
шаги выполнения и ожидаемый результат.

### Обязательные поля

| Поле                  | Описание                                     | Пример                      |
|-----------------------|----------------------------------------------|-----------------------------|
| ID                    | Уникальный идентификатор                     | TC-CART-001                 |
| Title                 | Краткое описание (что проверяем)             | Добавление товара в корзину |
| Preconditions         | Предусловия                                  | Пользователь авторизован    |
| Steps                 | Пошаговые действия                           | 1. Открыть каталог...       |
| Expected Result       | Ожидаемый результат                          | Товар отображается в корзине|
| Priority              | Приоритет (High/Medium/Low)                  | High                        |

### Дополнительные (рекомендуемые) поля

| Поле                  | Описание                                     |
|-----------------------|----------------------------------------------|
| Module / Feature      | К какому модулю относится                    |
| Test Data             | Конкретные тестовые данные                   |
| Postconditions        | Состояние системы после выполнения           |
| Author                | Кто написал                                  |
| Automation Status     | Автоматизирован / Не автоматизирован         |
| Tags / Labels         | Метки для фильтрации (smoke, regression)     |

### Уровни детализации

**Высокий уровень (high-level test case):**

```
TC-CART-001: Добавление товара в корзину
Предусловия: Пользователь авторизован
Шаги: Найти товар и добавить в корзину
Ожидаемый результат: Товар в корзине, счётчик увеличился
```

**Низкий уровень (low-level test case):**

```
TC-CART-001: Добавление товара в корзину
Предусловия:
  - Пользователь авторизован (login: testuser, password: Test123!)
  - Корзина пуста
  - Товар "iPhone 15" (SKU: IP15-128-BLK) есть на складе

Шаги:
  1. Открыть главную страницу (https://staging.example.com)
  2. Ввести в поле поиска "iPhone 15"
  3. Нажать кнопку "Найти"
  4. В результатах поиска найти "iPhone 15 128GB Black"
  5. Нажать кнопку "В корзину" рядом с товаром
  6. Дождаться появления уведомления "Товар добавлен"

Ожидаемый результат:
  - Появляется уведомление "Товар добавлен в корзину"
  - Счётчик корзины в шапке сайта показывает "1"
  - При переходе в корзину отображается:
    * Название: iPhone 15 128GB Black
    * Количество: 1
    * Цена: 89 990 руб.
    * Итого: 89 990 руб.

Постусловия: В корзине 1 товар
```

### Когда какой уровень детализации

| Ситуация                         | Рекомендуемый уровень |
|----------------------------------|-----------------------|
| Тесты выполняет автор            | Высокий               |
| Тесты передаются другому QA      | Низкий                |
| Тесты для аутсорс-команды        | Максимально низкий    |
| Regression suite                 | Средний / Низкий      |
| Exploratory testing              | Не нужны (хартия)     |
| Приёмочное тестирование (UAT)    | Низкий                |
| Критичный функционал (оплата)    | Низкий                |

### Пример хорошего vs плохого тест-кейса

**Плохой тест-кейс:**

```
ID: TC-001
Название: Тест логина
Шаги: Залогиниться
Результат: Работает
```

Проблемы: нет предусловий, шаги неконкретные, результат неизмеримый,
нет тестовых данных, невоспроизводимый.

**Хороший тест-кейс:**

```
ID: TC-AUTH-001
Название: Успешная авторизация с корректными учётными данными
Приоритет: High
Предусловия:
  - Пользователь зарегистрирован (email: test@example.com, пароль: ValidPass123!)
  - Пользователь не авторизован (сессия отсутствует)

Шаги:
  1. Открыть страницу авторизации (/login)
  2. Ввести email: test@example.com
  3. Ввести пароль: ValidPass123!
  4. Нажать кнопку "Войти"

Ожидаемый результат:
  - Пользователь перенаправлен на главную страницу (/dashboard)
  - В шапке отображается имя пользователя "Test User"
  - В cookies установлен session token
  - В логах зафиксирована успешная авторизация
```

### Атрибуты качественного тест-кейса

1. **Атомарность** -- один кейс проверяет одну вещь
2. **Независимость** -- не зависит от результата других кейсов
3. **Воспроизводимость** -- любой человек может выполнить и получить тот же результат
4. **Прослеживаемость** -- связан с требованием (traceability)
5. **Однозначность** -- ожидаемый результат чёткий и измеримый
6. **Актуальность** -- соответствует текущему состоянию системы

## 4. Checklist (Чек-лист)

### Что это

**Checklist** -- упрощённый формат тестовой документации: список проверок без
детальных шагов. Каждый пункт -- короткое описание того, что нужно проверить.

### Когда использовать чек-лист вместо тест-кейса

| Ситуация                                  | Чек-лист | Тест-кейс |
|-------------------------------------------|----------|-----------|
| Опытный QA знает продукт                  | +        |           |
| Новый сотрудник выполняет тесты           |          | +         |
| Быстрая проверка нового функционала       | +        |           |
| Регрессионное тестирование                |          | +         |
| Приёмочное тестирование (UAT)             |          | +         |
| Smoke testing                             | +        |           |
| Критичный функционал (оплата, безопасность)|          | +         |
| Agile / быстрые итерации                  | +        |           |
| Передача знаний аутсорс-команде           |          | +         |
| Exploratory testing (как направляющая)    | +        |           |

### Пример чек-листа: форма регистрации

```
МОДУЛЬ: Регистрация пользователя
ДАТА: 20.03.2026
QA: Иванов А.

Positive сценарии:
[  ] Регистрация с валидными данными (все поля заполнены)
[  ] Регистрация с минимально допустимым паролем (8 символов)
[  ] Регистрация с максимально допустимым паролем (50 символов)
[  ] Письмо с подтверждением приходит на указанный email
[  ] Ссылка подтверждения работает

Negative сценарии:
[  ] Регистрация без email -- ошибка валидации
[  ] Регистрация с невалидным email (без @) -- ошибка валидации
[  ] Регистрация с уже существующим email -- ошибка "Email занят"
[  ] Пароль короче 8 символов -- ошибка валидации
[  ] Пароли не совпадают -- ошибка "Пароли не совпадают"
[  ] Пустая форма -- ошибки валидации у всех обязательных полей
[  ] SQL-инъекция в поле email -- корректная обработка

UI/UX:
[  ] Поле пароля маскировано (****)
[  ] Есть кнопка "Показать пароль"
[  ] Сообщения об ошибках на русском языке
[  ] Форма адаптивна на мобильном разрешении (375px)
[  ] Tab-навигация между полями работает корректно
```

### Чек-лист vs Тест-кейс: сравнение

| Критерий              | Чек-лист                     | Тест-кейс                    |
|-----------------------|------------------------------|-------------------------------|
| Детализация           | Низкая (что проверить)       | Высокая (как проверить)       |
| Время создания        | Быстро (минуты)              | Долго (часы)                  |
| Время поддержки       | Минимальное                  | Значительное                  |
| Воспроизводимость     | Зависит от опыта QA          | Высокая                       |
| Передача знаний       | Сложнее                      | Проще                         |
| Гибкость              | Высокая                      | Низкая                        |
| Формальность          | Низкая                       | Высокая                       |

## 5. Test Suite (Набор тестов)

### Что это

**Test suite** -- логически сгруппированная коллекция тест-кейсов,
объединённых по определённому признаку.

### Принципы организации

**По модулям/фичам:**
```
Test Suites/
├── Авторизация/
│   ├── TC-AUTH-001: Успешный логин
│   ├── TC-AUTH-002: Неверный пароль
│   ├── TC-AUTH-003: Блокировка после 5 попыток
│   └── TC-AUTH-004: Восстановление пароля
├── Корзина/
│   ├── TC-CART-001: Добавление товара
│   ├── TC-CART-002: Удаление товара
│   ├── TC-CART-003: Изменение количества
│   └── TC-CART-004: Применение промокода
├── Оплата/
│   ├── TC-PAY-001: Оплата картой Visa
│   ├── TC-PAY-002: Оплата картой Mastercard
│   └── TC-PAY-003: Ошибка оплаты (недостаточно средств)
└── Профиль пользователя/
    ├── TC-PROF-001: Редактирование имени
    ├── TC-PROF-002: Смена пароля
    └── TC-PROF-003: Загрузка аватара
```

**По типу тестирования:**
```
Test Suites/
├── Smoke Suite/            (10-15 ключевых кейсов)
├── Regression Suite/       (все стабильные кейсы)
├── Sanity Suite/           (кейсы для быстрой проверки)
└── Full Suite/             (все тест-кейсы)
```

**По приоритету:**
```
Test Suites/
├── P1 - Critical/          (блокирующие, проверять всегда)
├── P2 - High/              (важные, проверять в каждом релизе)
├── P3 - Medium/            (средний приоритет, по возможности)
└── P4 - Low/               (низкий приоритет, при наличии времени)
```

### Управление test suite в TMS

Популярные инструменты (Test Management Systems):
- **TestRail** -- наиболее распространённый
- **Zephyr** -- интеграция с Jira
- **TestIT** -- российская разработка
- **qase.io** -- облачный, бесплатный для малых команд
- **Allure TestOps** -- интеграция с Allure-отчётами

### Пример иерархии в TestRail

```
Проект: E-commerce Platform
│
├── Milestone: Release 2.5.0
│   ├── Test Run: Smoke (15 кейсов)
│   ├── Test Run: New Features (42 кейса)
│   ├── Test Run: Regression (328 кейсов)
│   └── Test Run: Exploratory Notes
│
├── Test Suites (библиотека):
│   ├── Авторизация (23 кейса)
│   ├── Каталог товаров (45 кейсов)
│   ├── Корзина (34 кейса)
│   ├── Оплата (28 кейсов)
│   ├── Доставка (19 кейсов)
│   ├── Личный кабинет (31 кейс)
│   └── API (56 кейсов)
│
└── Reports:
    ├── Test Run Summary
    ├── Defect Distribution
    └── Coverage Matrix
```

## 6. Другие артефакты тестовой документации

### Bug Report (Баг-репорт)

Не является темой этого раздела, но тесно связан. Обязательные поля:

| Поле                | Пример                                        |
|---------------------|------------------------------------------------|
| Summary             | Цена товара отображается как 0 руб. при скидке 100% |
| Steps to Reproduce  | 1. Открыть товар с промокодом SALE100...       |
| Actual Result       | Цена: 0 руб., кнопка "Купить" активна         |
| Expected Result     | Товар бесплатный, но показано "Бесплатно"      |
| Severity            | Major                                          |
| Priority            | High                                           |
| Environment         | Chrome 120, staging.example.com                |
| Attachments         | screenshot_bug_001.png                         |

### Traceability Matrix (Матрица прослеживаемости)

Связывает требования с тест-кейсами. Позволяет убедиться, что каждое требование покрыто.

| Требование | Описание                          | Тест-кейсы              | Статус |
|------------|-----------------------------------|--------------------------|--------|
| REQ-001    | Регистрация пользователя         | TC-REG-001..005          | Passed |
| REQ-002    | Авторизация                       | TC-AUTH-001..004         | Passed |
| REQ-003    | Добавление в корзину              | TC-CART-001..003         | Failed |
| REQ-004    | Применение промокода              | TC-CART-004..008         | Blocked|
| REQ-005    | Оплата картой                     | TC-PAY-001..003         | Not run|

### Test Report (Отчёт о тестировании)

Итоговый документ, подводящий результаты тестирования. Основные разделы:

```
1. Общая информация (релиз, период, команда)
2. Статистика выполнения тестов:
   - Всего: 328
   - Passed: 310 (94.5%)
   - Failed: 12 (3.7%)
   - Blocked: 4 (1.2%)
   - Not run: 2 (0.6%)
3. Найденные дефекты:
   - Critical: 0
   - Major: 3
   - Minor: 7
   - Trivial: 2
4. Покрытие требований: 98%
5. Рекомендация: GO / NO-GO
6. Открытые риски
```

## Связь с тестированием

Тестовая документация -- не бюрократия, а инструмент:

1. **Планирование:** test plan задаёт рамки, сроки, ресурсы
2. **Выполнение:** тест-кейсы и чек-листы обеспечивают системный подход
3. **Коммуникация:** отчёты и матрицы прослеживаемости показывают прогресс стейкхолдерам
4. **Передача знаний:** новый сотрудник может быстро включиться по документации
5. **Автоматизация:** хорошие тест-кейсы -- основа для автотестов
6. **Аудит:** документация подтверждает, что тестирование проводилось

**Правило:** объём документации должен соответствовать проекту. Стартап из 3 человек
не нуждается в 50-страничном тест-плане. Банковский проект с регуляторными требованиями
не может обойтись одним чек-листом.

## Типичные ошибки

1. **Путать test strategy и test plan.** Strategy -- общий подход (один на проект), plan -- конкретный план (для каждого релиза).
2. **Писать тест-кейсы без ожидаемого результата.** "Проверить логин" -- не тест-кейс. Должно быть: "Пользователь перенаправлен на /dashboard, отображается имя".
3. **Слишком детализированные кейсы для опытной команды.** "Навести курсор на поле Email. Щёлкнуть левой кнопкой мыши..." -- избыточно, если QA знает продукт.
4. **Слишком высокоуровневые кейсы для аутсорс-команды.** "Проверить оплату" -- недостаточно для человека, не знакомого с системой.
5. **Не обновлять документацию.** Устаревшие тест-кейсы хуже, чем их отсутствие -- они вводят в заблуждение.
6. **Не указывать предусловия.** Кейс, требующий авторизованного пользователя, будет падать, если это не указано.
7. **Хранить тест-кейсы в Excel без версионирования.** Используйте TMS (TestRail, Zephyr, qase.io).
8. **Создавать документацию ради документации.** Каждый артефакт должен приносить ценность.

## Вопросы на интервью

### 🟢 Базовый уровень

1. **Что такое тест-кейс? Какие обязательные поля?**
   Документ, описывающий тестовый сценарий. Обязательные поля: ID, название, предусловия, шаги, ожидаемый результат, приоритет.

2. **Чем чек-лист отличается от тест-кейса?**
   Чек-лист -- список проверок без детальных шагов (что проверить). Тест-кейс -- пошаговая инструкция (как проверить). Чек-лист быстрее создать, тест-кейс проще воспроизвести.

3. **Что такое тест-план? Кто его пишет?**
   Документ, описывающий скоуп, подход, ресурсы и расписание тестирования. Пишет QA Lead или Senior QA для конкретного релиза.

4. **Что входит в expected result?**
   Конкретный, измеримый результат: что отображается на экране, какие данные в БД, какой статус-код API. Не "работает корректно", а "отображается сообщение 'Заказ создан', статус 201".

### 🟡 Средний уровень

5. **В чём разница между test strategy и test plan?**
   Strategy -- высокоуровневый документ на весь проект (общий подход, инструменты, роли). Plan -- конкретный план на релиз (скоуп, расписание, ресурсы). Strategy создаётся один раз, plan -- для каждого релиза.

6. **Как организовать test suite для проекта с 500+ тест-кейсами?**
   По модулям (авторизация, корзина, оплата), с вложенной структурой. Отдельные run-наборы: smoke, regression, по приоритету. Использовать TMS с тегами и фильтрами.

7. **Когда достаточно чек-листа, а когда нужны полноценные тест-кейсы?**
   Чек-лист: опытная команда, быстрые итерации, exploratory, smoke. Тест-кейсы: передача знаний, UAT, аутсорс, критичный функционал, регуляторные требования.

8. **Что такое traceability matrix и зачем она нужна?**
   Таблица связей "требование -- тест-кейс". Позволяет убедиться, что каждое требование покрыто тестами, найти непокрытые требования, оценить влияние изменений.

### 🔴 Продвинутый уровень

9. **Вы пришли на проект без тестовой документации. Что создадите в первую очередь и почему?**
   1) Чек-листы для критичного функционала (быстро, покрывают основное). 2) Smoke suite. 3) Тест-кейсы для самых сложных/критичных сценариев. 4) Тест-план на ближайший релиз. Не создавать всё сразу -- итеративный подход.

10. **Как поддерживать актуальность тест-кейсов в Agile-проекте с частыми изменениями?**
    Владелец кейсов (QA) обновляет при изменении требований. Review тест-кейсов в Definition of Done. Автоматизированные тесты обновляются вместе с кодом. Регулярная ревизия (раз в квартал) для удаления устаревших кейсов.

11. **Как оценить качество тестовой документации?**
    Метрики: % требований с тест-кейсами (traceability), % кейсов с результатом прогона, количество "устаревших" кейсов, время на onboarding нового QA, defect leakage (баги, не пойманные тестами).

## Практические задания

### Задание 1: Написать тест-кейс
Напишите 3 тест-кейса (low-level) для функции "Восстановление пароля" любого
знакомого вам сайта. Включите positive и negative сценарии. Используйте все
обязательные поля.

### Задание 2: Чек-лист
Составьте чек-лист из 20 проверок для страницы товара в интернет-магазине.
Разделите на категории: функционал, UI/UX, negative, кроссбраузерность.

### Задание 3: Тест-план
Напишите сокращённый тест-план для тестирования нового функционала
"Избранное" (добавление товаров в избранное, просмотр списка, удаление).
Включите: скоуп, подход, окружение, расписание, риски, критерии выхода.

### Задание 4: Критика тест-кейса
Найдите все проблемы в следующем тест-кейсе и перепишите его:

```
TC-1: Проверка корзины
Шаги: Добавить товары. Проверить корзину. Оформить заказ.
Результат: Всё работает.
```

**Ожидаемые проблемы:** нет предусловий, шаги неконкретные, нет тестовых данных,
результат неизмеримый, нет приоритета, отсутствует ID-стандарт.

### Задание 5: Traceability Matrix
Для следующих требований составьте матрицу прослеживаемости:
- REQ-1: Пользователь может зарегистрироваться по email
- REQ-2: Пользователь может войти по email и паролю
- REQ-3: Пользователь может сбросить пароль
- REQ-4: Система блокирует аккаунт после 5 неверных попыток входа

Придумайте тест-кейсы для каждого требования (минимум 2 на требование).
