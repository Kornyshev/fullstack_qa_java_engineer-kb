# Системная аналитика: артефакты, диаграммы, требования и их тестирование

*Справочный документ для подготовки к собеседованию на позицию QA Engineer в банковском/финтех-проекте*

---

## 1. Артефакты системной аналитики — полный перечень

### 1.1 Документы верхнего уровня

**BRD (Business Requirements Document)** — описывает бизнес-потребность. Отвечает на вопрос «зачем мы это делаем?». Содержит бизнес-цели, метрики успеха, scope проекта, стейкхолдеров. Не содержит технических деталей. Пример в банке: «Необходимо сократить время регистрации бизнеса с 5 дней до 1 дня, увеличив долю онлайн-регистраций до 30% рынка».

**Vision & Scope Document** — высокоуровневое описание продукта: что входит в scope, что не входит, основные фичи, целевая аудитория. Часто заменяет BRD в Agile-командах.

### 1.2 Документы уровня решения

**SRS (Software Requirements Specification)** — основной документ системного аналитика. Описывает ЧТО система должна делать. Классическая структура по IEEE 830:

1. Введение — цель документа, scope, определения и аббревиатуры, ссылки
2. Общее описание — перспектива продукта, функции продукта, характеристики пользователей, ограничения, допущения и зависимости
3. Конкретные требования — функциональные требования, требования к интерфейсам (UI, API, hardware, software), нефункциональные требования (производительность, безопасность, надёжность)
4. Приложения — модели данных, диаграммы, прототипы

В банках SRS часто не существует как единый документ — вместо него набор user stories в Jira + отдельные технические спецификации в Confluence.

**FSD (Functional Specification Document)** — детализация SRS до уровня «КАК это работает». Содержит: описание экранов и полей, логику переходов между состояниями, бизнес-правила и валидации, формулы расчётов, маппинг данных между системами. Пример в банке: «При вводе ИНН из 10 цифр система определяет тип клиента как ЮЛ, из 12 цифр — как ИП. При невалидном ИНН отображается ошибка "Некорректный ИНН" и поле подсвечивается красным».

**SAD (Software Architecture Document)** — архитектурное описание системы. Компоненты, их взаимодействие, технологический стек, принципы развёртывания. Обычно создаётся архитектором, но аналитик вносит вклад в части бизнес-логики.

### 1.3 Форматы описания требований

**Use Cases (сценарии использования)** — формализованное описание взаимодействия актора (пользователя или внешней системы) с системой. Структура:

- Название (например, «UC-001: Регистрация ИП через онлайн-сервис»)
- Актор (кто инициирует — «Клиент-физлицо»)
- Предусловия (что должно быть истинно до начала — «Клиент авторизован в системе»)
- Основной поток (happy path — шаг за шагом)
- Альтернативные потоки (ветвления — «Если ИНН уже зарегистрирован...»)
- Потоки исключений (ошибки — «Если сервис ФНС недоступен...»)
- Постусловия (что истинно после завершения — «Заявка создана в статусе "На проверке"»)
- Бизнес-правила (ссылки на BR-001, BR-002...)

Уровни детализации: краткий (brief) — один абзац; casual — основной + альтернативные потоки текстом; полный (fully dressed) — все поля по шаблону.

**User Stories** — лёгкий формат для Agile. Шаблон: «Как [роль], я хочу [действие], чтобы [ценность]». Пример: «Как предприниматель, я хочу зарегистрировать ИП онлайн, чтобы не посещать ФНС лично».

Критерии приёмки (Acceptance Criteria) — конкретные проверяемые условия. Форматы:

- Список условий: «Форма содержит поля: ФИО, ИНН, паспортные данные, ОКВЭД»
- Given-When-Then (Gherkin): «Given клиент заполнил все обязательные поля, When он нажимает "Отправить", Then заявка создаётся в статусе "Новая" и клиент получает email-подтверждение»

Критерии качества user story — INVEST: Independent (независима), Negotiable (обсуждаема), Valuable (ценна для пользователя), Estimable (оценима), Small (достаточно мала), Testable (тестируема).

**Job Stories** — альтернатива user stories с фокусом на контекст: «Когда [ситуация/контекст], я хочу [мотивация], чтобы [ожидаемый результат]». Пример: «Когда я получил уведомление об отказе в регистрации, я хочу увидеть конкретную причину отказа, чтобы исправить данные и подать повторно».

### 1.4 Вспомогательные артефакты

**Glossary (глоссарий)** — единый словарь терминов проекта. Критически важен в банках, где одно слово может означать разное в разных контекстах. Пример: «Клиент» — в контексте KYC это физлицо, в контексте РКО — юрлицо или ИП.

**Data Dictionary (словарь данных)** — описание каждого поля/атрибута: имя, тип, формат, допустимые значения, обязательность, значение по умолчанию, источник. Пример: «inn — string(12), обязательное, валидация по контрольной сумме, источник: ввод пользователя или ЕГРЮЛ».

**RTM (Requirements Traceability Matrix)** — матрица, связывающая бизнес-требования → функциональные требования → тест-кейсы → дефекты. Позволяет видеть: все ли требования покрыты тестами, какие требования затрагивает изменение, какие дефекты к какому требованию относятся.

**Нефункциональные требования (NFR)** — описывают КАК система должна работать (в отличие от функциональных — ЧТО):

- Производительность: «Время ответа API регистрации не более 3 секунд при 1000 одновременных запросах»
- Доступность: «SLA 99.95%, допустимый простой не более 4.38 часов в год»
- Безопасность: «Все персональные данные шифруются AES-256 при хранении и TLS 1.3 при передаче»
- Масштабируемость: «Система должна поддерживать горизонтальное масштабирование до 10 000 регистраций в сутки»
- Совместимость: «Поддержка браузеров: Chrome 120+, Firefox 115+, Safari 17+»

**API-контракты** — формализованное описание интерфейсов взаимодействия:

- REST API: OpenAPI/Swagger спецификация (YAML/JSON) — endpoints, методы, параметры, тела запросов/ответов, коды ответов, схемы данных
- SOAP API: WSDL (Web Services Description Language) — описание сервиса, операций, типов данных
- Async API: AsyncAPI спецификация — описание event-driven интерфейсов (Kafka-топики, форматы сообщений)

**Модели данных:**

- Концептуальная модель — сущности и связи без деталей (для бизнеса)
- Логическая модель — атрибуты, типы, ключи, нормализация (для аналитиков и разработчиков)
- Физическая модель — таблицы, индексы, партиционирование, конкретная СУБД (для DBA)

## 2. Виды диаграмм в системной аналитике

### 2.1 UML-диаграммы (Unified Modeling Language)

**Use Case Diagram (диаграмма вариантов использования)**

Что показывает: акторов (пользователей и внешние системы) и их взаимодействие с системой на высоком уровне. Ключевые элементы: актор (человечек), вариант использования (овал), связи (include, extend, generalization), границы системы (прямоугольник). Когда применяется: на этапе сбора требований для определения scope — «что система делает и для кого». Банковский пример: актор «Клиент» связан с use cases «Зарегистрировать бизнес», «Открыть расчётный счёт», «Загрузить документы». Актор «Система ФНС» связан с «Проверить данные заявки».

**Sequence Diagram (диаграмма последовательности)**

Что показывает: порядок обмена сообщениями между объектами/компонентами во времени. Ключевые элементы: участники (lifelines — вертикальные линии), сообщения (стрелки: синхронные → сплошная с закрашенным наконечником, асинхронные → сплошная с открытым наконечником, ответы → пунктирная), фрагменты (alt — условие, loop — цикл, opt — опциональный блок, par — параллельное выполнение), активации (прямоугольники на lifeline — период обработки). Когда применяется: при описании интеграций, сложных бизнес-сценариев, взаимодействия микросервисов. Банковский пример: Client → API Gateway → Registration Service → FNS Adapter → ФНС API (запрос проверки) → обратно цепочка ответов. Фрагмент alt: «если ФНС вернула ошибку → уведомить клиента».

**Activity Diagram (диаграмма активности)**

Что показывает: поток выполнения процесса с условиями, параллельными ветками, циклами. По сути — блок-схема, но с нотацией UML. Ключевые элементы: начальное/конечное состояние (кружки), действия (прямоугольники со скруглёнными углами), условия/решения (ромбы), fork/join (толстые линии — параллельные потоки), swimlanes (дорожки — разделение по ролям/системам). Когда применяется: описание бизнес-процессов, workflow, алгоритмов обработки. Банковский пример: процесс обработки заявки на регистрацию — от приёма до одобрения/отказа, с параллельной проверкой KYC и документов.

**State Machine Diagram (диаграмма состояний)**

Что показывает: жизненный цикл объекта — все его состояния и переходы между ними. Ключевые элементы: состояния (прямоугольники со скруглёнными углами), переходы (стрелки с условиями/событиями/действиями), начальное/конечное состояние, вложенные состояния. Когда применяется: описание статусной модели сущности. Банковский пример: заявка на регистрацию бизнеса: Черновик → На проверке → Ожидает документов → На рассмотрении ФНС → Одобрена / Отклонена → Завершена. Каждый переход имеет условие и триггер.

**Class Diagram (диаграмма классов)**

Что показывает: структуру данных — классы/сущности, их атрибуты, методы и связи (наследование, ассоциация, агрегация, композиция). Когда применяется: проектирование доменной модели, описание структуры данных. Банковский пример: Client (ФИО, ИНН, тип) → имеет → Application (номер, дата, статус) → содержит → Document (тип, файл, статус проверки).

**Component Diagram (диаграмма компонентов)**

Что показывает: архитектуру системы — компоненты (модули, сервисы) и интерфейсы между ними. Когда применяется: описание архитектуры на уровне модулей/сервисов. Банковский пример: Registration Service ↔ KYC Service ↔ FNS Adapter ↔ Document Storage ↔ Notification Service. Интерфейсы: REST, Kafka.

**Deployment Diagram (диаграмма развёртывания)**

Что показывает: физическую инфраструктуру — серверы, контейнеры, узлы сети и размещение компонентов на них. Когда применяется: планирование инфраструктуры, описание среды. Банковский пример: K8s Cluster → Pod: Registration Service, Pod: KYC Service; External: ФНС API, СМЭВ; Database Cluster: PostgreSQL Primary + Replica.

### 2.2 BPMN-диаграммы (Business Process Model and Notation)

Что показывает: бизнес-процессы с точки зрения бизнеса и исполнения. BPMN — стандарт ISO 19510.

Ключевые элементы:

- **Events (события):** Start Event (тонкий кружок — начало процесса), End Event (толстый кружок — конец), Intermediate Event (двойной кружок — событие в процессе). Типы: Message, Timer, Signal, Error, Escalation, Conditional.
- **Tasks (задачи):** User Task (выполняет человек), Service Task (выполняет система автоматически), Script Task (скрипт), Send/Receive Task (отправка/получение сообщения), Manual Task (вне системы).
- **Gateways (шлюзы):** Exclusive (XOR, ×) — только один путь; Parallel (AND, +) — все пути одновременно; Inclusive (OR, ○) — один или несколько путей; Event-Based — выбор по событию (что произойдёт первым).
- **Subprocess** — вложенный процесс (свёрнутый или развёрнутый).
- **Boundary Events** — события, привязанные к задаче: Timer (если задача не выполнена за N часов), Error (если произошла ошибка), Message (если пришло сообщение).
- **Pools и Lanes** — пулы (организации/участники) и дорожки (роли/отделы внутри пула).
- **Sequence Flow** (→) — порядок выполнения, **Message Flow** (- - →) — обмен сообщениями между пулами.

Банковский пример: процесс «Регистрация бизнеса»:
Start Event (Клиент подал заявку) → User Task: Заполнить анкету → Service Task: Валидация данных → Exclusive Gateway: данные валидны? → [Да] → Parallel Gateway: одновременно Service Task: Проверка KYC + Service Task: Запрос в ФНС → Parallel Gateway (join) → Exclusive Gateway: все проверки пройдены? → [Да] → Service Task: Создать счёт → End Event (Успех) / [Нет] → User Task: Уведомить клиента об отказе → End Event (Отказ).

### 2.3 Другие типы диаграмм

**ER-диаграмма (Entity-Relationship)**

Что показывает: сущности базы данных, их атрибуты и связи (1:1, 1:N, N:M). Нотации: Chen, Crow's Foot (наиболее распространена), UML. Банковский пример: Client (1) → (N) Application (1) → (N) Document; Client (1) → (N) Account.

**DFD (Data Flow Diagram)**

Что показывает: потоки данных между процессами, хранилищами данных и внешними сущностями. Уровни: контекстная (Level 0) — система как чёрный ящик; Level 1 — основные процессы; Level 2+ — декомпозиция. Банковский пример: Level 0: Клиент → [данные заявки] → Система регистрации → [запрос проверки] → ФНС.

**Context Diagram (контекстная диаграмма)**

Что показывает: систему как единый блок и все внешние системы/акторов, с которыми она взаимодействует. По сути — DFD Level 0 или C4 Level 1. Банковский пример: Система регистрации бизнеса ↔ Клиент, ФНС, КАД (АУЦ), Система KYC, Процессинг РКО, Email/SMS Gateway.

**C4 Model**

Четыре уровня детализации:
1. **Context** — система + внешние акторы/системы (как контекстная диаграмма)
2. **Container** — из каких «контейнеров» состоит система (веб-приложение, API, БД, очередь сообщений)
3. **Component** — из каких компонентов состоит каждый контейнер
4. **Code** — уровень классов/модулей (обычно генерируется из кода)

C4 набирает популярность как альтернатива UML за счёт простоты и наглядности.

**Wireframes и Mockups**

Wireframe — схематичный набросок интерфейса (без дизайна, только структура и расположение элементов). Mockup — детализированный макет с визуальным дизайном. Инструменты: Figma, Axure, Balsamiq. Аналитик создаёт wireframes для согласования с бизнесом, дизайнер превращает их в mockups.

**Mind Maps (интеллект-карты)**

Инструмент для мозгового штурма, структуризации требований, декомпозиции фич. Не является формальным артефактом, но часто используется на этапе сбора требований. Инструменты: Miro, XMind, MindMeister.

## 3. Уровни и типы требований

### 3.1 Иерархия требований

```
Бизнес-требования (Business Requirements)
  └── Требования стейкхолдеров (Stakeholder Requirements)
        └── Требования к решению (Solution Requirements)
              ├── Функциональные (Functional Requirements)
              └── Нефункциональные (Non-Functional Requirements)
                    └── Требования к переходу (Transition Requirements)
```

**Бизнес-требования** — высокоуровневые цели организации. «Увеличить долю онлайн-регистраций до 30%». Источник: бизнес-заказчик, Product Owner.

**Требования стейкхолдеров** — потребности конкретных заинтересованных лиц. «Оператор КЦ должен видеть статус заявки клиента в реальном времени». Источник: интервью с пользователями, воркшопы.

**Функциональные требования** — что система должна делать. «Система должна отправлять заявку на регистрацию в ФНС через СМЭВ в формате XML по схеме v2.3». Формулируются в стиле «система должна...» (shall-style).

**Нефункциональные требования** — как система должна работать: производительность, безопасность, доступность, удобство использования, совместимость, поддерживаемость.

**Требования к переходу** — временные требования для перехода из текущего состояния в целевое. «Все заявки из старой системы должны быть мигрированы в новую с сохранением истории статусов». Перестают быть актуальны после завершения миграции.

### 3.2 Форматы описания требований

**Shall-style (формальный):** «Система ДОЛЖНА позволять клиенту загрузить скан паспорта в форматах JPEG, PNG, PDF размером до 10 МБ».

**Given-When-Then (Gherkin/BDD):**
```
Given клиент находится на странице загрузки документов
  And клиент выбрал файл формата PDF размером 5 МБ
When клиент нажимает кнопку "Загрузить"
Then файл загружается успешно
  And отображается превью документа
  And статус документа меняется на "Загружен"
```

**If-Then (бизнес-правила):**
```
IF тип клиента = "ИП" 
  AND возраст >= 18 
  AND гражданство = "РФ"
THEN разрешить регистрацию через упрощённую анкету
ELSE IF возраст >= 16 AND есть решение суда об эмансипации
THEN разрешить регистрацию с загрузкой решения суда
ELSE отклонить заявку с причиной "Не соответствует возрастным требованиям"
```

**Таблица решений (Decision Table):**

| Тип клиента | Возраст | Гражданство | Результат |
|------------|---------|-------------|-----------|
| ИП | ≥ 18 | РФ | Упрощённая анкета |
| ИП | 16-17 | РФ + эмансипация | Полная анкета + решение суда |
| ИП | < 16 | Любое | Отказ |
| ООО | Любой | РФ | Полная анкета |
| ООО | Любой | Нерезидент | Расширенная анкета + KYC |

## 4. Свойства качественных требований («Требования к требованиям»)

### 4.1 Полнота (Completeness)

Определение: требование описывает всё необходимое для реализации, не ссылается на несуществующие документы, не содержит «TODO» или «TBD». Набор требований покрывает все сценарии.

Плохо: «Система должна валидировать ИНН». (Как именно? Что при ошибке? Какой формат?)

Хорошо: «Система должна валидировать ИНН по контрольной сумме согласно алгоритму ФНС. При невалидном ИНН отображается сообщение "Некорректный ИНН, проверьте введённые данные", поле подсвечивается красным, кнопка "Далее" остаётся недоступной».

Как проверить: есть ли ответы на вопросы «что если?», описаны ли все ветки (happy path, альтернативные, ошибки), нет ли ссылок на «будет определено позже».

### 4.2 Непротиворечивость (Consistency)

Определение: требования не противоречат друг другу. Термины используются единообразно. Нет конфликтов между функциональными и нефункциональными требованиями.

Плохо: REQ-01: «Система должна отправлять заявку в ФНС синхронно и ожидать ответ». REQ-17: «Время отклика любой операции не более 2 секунд». (Синхронный запрос в ФНС может занимать 30+ секунд — противоречие.)

Хорошо: REQ-01: «Система должна отправлять заявку в ФНС асинхронно через Kafka. Статус заявки обновляется при получении ответного сообщения. Если ответ не получен в течение 24 часов — заявка переводится в статус "Требует ручной проверки"».

Как проверить: поиск одинаковых терминов с разными определениями, сравнение временных ограничений между связанными требованиями, проверка общего глоссария.

### 4.3 Однозначность (Unambiguity)

Определение: требование имеет ровно одну интерпретацию. Нет слов-маркеров неоднозначности.

Слова-маркеры проблем: «быстро», «удобно», «интуитивно», «достаточно», «и т.д.», «например», «при необходимости», «как правило», «обычно», «по возможности», «соответствующий», «надлежащий», «адекватный».

Плохо: «Система должна быстро обрабатывать заявки». (Что значит «быстро»?)

Хорошо: «Система должна обрабатывать заявку на регистрацию ИП за время не более 500 мс (без учёта времени ответа внешних систем)».

Как проверить: может ли другой человек, прочитав требование, понять его по-другому? Есть ли количественные метрики вместо качественных оценок?

### 4.4 Тестируемость / Проверяемость (Testability / Verifiability)

Определение: для требования можно написать тест, который однозначно покажет — выполнено оно или нет. Существует конечный, недвусмысленный способ проверки.

Плохо: «Система должна быть надёжной». (Как проверить? Какой тест написать?)

Хорошо: «При отказе одного из трёх узлов кластера система должна продолжить обработку запросов без потери данных. Время переключения на резервный узел — не более 30 секунд. Ни одна заявка, принятая до отказа, не должна быть потеряна».

Как проверить: можно ли сформулировать конкретный тест-кейс? Есть ли числовые критерии? Определены ли граничные условия?

### 4.5 Прослеживаемость (Traceability)

Определение: каждое требование имеет уникальный идентификатор и связано с: источником (бизнес-требование, стейкхолдер), реализацией (задачи в Jira, код), тестами (тест-кейсы), дефектами. Позволяет понять impact при изменении.

Как реализуется: RTM (Requirements Traceability Matrix), связи в Jira (requirement → story → test case → bug), теги и лейблы.

### 4.6 Модифицируемость (Modifiability)

Определение: требования организованы так, что изменения можно внести легко, без каскадных правок. Нет дублирования (DRY-принцип), используются ссылки на общие правила.

Плохо: одно и то же бизнес-правило описано в 5 разных user stories. При изменении нужно найти и поправить все 5.

Хорошо: бизнес-правило описано один раз (BR-042) и user stories ссылаются на него.

### 4.7 Другие свойства

**Необходимость (Necessity):** каждое требование нужно для достижения бизнес-целей. Нет «золотого покрытия» (gold plating). Как проверить: «если убрать это требование, пострадает ли бизнес-цель?»

**Реализуемость (Feasibility):** требование технически реализуемо в рамках бюджета, сроков и технологий. Как проверить: согласование с разработчиками и архитекторами до утверждения.

**Приоритизируемость (Prioritizability):** каждому требованию можно присвоить приоритет. Методы: MoSCoW (Must / Should / Could / Won't), матрица «важность × срочность», метод Кано (обязательные / линейные / восхищающие).

**Атомарность:** одно требование описывает одну вещь. Нет «и»/«или» внутри одного требования, объединяющих разные функции.

## 5. Процесс работы с требованиями

### 5.1 Сбор требований (Elicitation)

**Интервью** — один на один со стейкхолдером. Открытые вопросы: «Расскажите, как вы сейчас обрабатываете заявки?». Закрытые вопросы: «Сколько заявок в день обрабатывается?».

**Воркшопы** — групповая сессия со всеми стейкхолдерами. Позволяет выявить противоречия между разными подразделениями. Техники: Event Storming, Story Mapping, Impact Mapping.

**Наблюдение (Observation / Job Shadowing)** — аналитик наблюдает за реальной работой пользователей. Часто выявляет неформализованные процессы и workaround-ы.

**Анализ документов** — изучение существующей документации, регламентов, нормативных актов (в банках — требований ЦБ).

**Прототипирование** — создание wireframes/mockups для обсуждения с бизнесом. Позволяет «увидеть» требование до реализации.

**Benchmarking** — анализ конкурентов и аналогов. Как это реализовано у других.

### 5.2 Анализ требований

**Приоритизация:**
- MoSCoW: Must have (обязательно), Should have (желательно), Could have (можно), Won't have (не в этот релиз)
- RICE: Reach × Impact × Confidence / Effort
- Kano: базовые (must-be), линейные (one-dimensional), восхищающие (attractive)
- Weighted Scoring: критерии с весами, оценка каждого требования

**Декомпозиция:** разбиение крупных требований/эпиков на user stories → tasks. Техники: story mapping, feature breakdown.

**Моделирование:** создание диаграмм (UML, BPMN, ER) для визуализации и верификации понимания.

### 5.3 Документирование

Выбор формата зависит от контекста: Waterfall → SRS/FSD; Agile → user stories + wiki; SAFe → features → stories → acceptance criteria. Ключевое: независимо от формата, требования должны быть записаны, версионированы и доступны всей команде.

### 5.4 Управление изменениями

**Change Request (CR)** — формальный запрос на изменение требования. Содержит: что меняется, зачем, impact assessment (на какие компоненты влияет), оценка трудозатрат. В банках CR обычно проходит через Change Advisory Board (CAB).

**Requirements Baseline** — утверждённый набор требований на определённый момент. Любое изменение после baseline — через CR. Versioning: v1.0, v1.1, v2.0.

## 6. Интеграционная аналитика

### 6.1 Типы интеграций

**Синхронные (request-response):**
- REST API: HTTP-запрос → ожидание ответа → обработка. Описывается через OpenAPI/Swagger.
- SOAP: XML-конверт (Envelope → Header → Body), описывается через WSDL.
- gRPC: бинарный протокол на базе HTTP/2 + Protocol Buffers. Описывается через .proto файлы.

**Асинхронные (event-driven):**
- Kafka: producer → topic (с партициями) → consumer group. Описание: имя топика, формат сообщения (Avro/JSON/Protobuf), ключ партиционирования, retention, consumer group ID.
- RabbitMQ: publisher → exchange → queue → consumer. Описание: exchange type (direct/topic/fanout), routing key, queue binding.
- IBM MQ: sender → queue manager → queue → receiver. Часто в legacy-банковских системах.

### 6.2 Что описывает аналитик для интеграции

Для каждой интеграции аналитик создаёт:

1. **Sequence Diagram** — визуализация потока взаимодействия
2. **Описание эндпоинта/топика** — URL, метод, формат, авторизация
3. **Формат запроса/сообщения** — JSON-схема или XML-схема с описанием каждого поля
4. **Формат ответа** — включая все возможные коды ответов и их значения
5. **Обработка ошибок** — что делает система при таймауте, 4xx, 5xx, невалидном ответе
6. **Retry-политика** — количество повторов, интервалы (fixed, exponential backoff), максимальное время
7. **Идемпотентность** — как обеспечивается (idempotency key, дедупликация)
8. **Таймауты** — connect timeout, read timeout, общий timeout
9. **Circuit Breaker** — при каких условиях «размыкается», fallback-поведение
10. **SLA** — допустимое время ответа, доступность внешней системы

### 6.3 Пример описания Kafka-интеграции

```
Топик: registration.application.created
Формат: JSON (Avro-схема v2)
Ключ: applicationId (UUID)
Producer: Registration Service
Consumer: KYC Service, Notification Service
Партиционирование: по applicationId (гарантирует порядок для одной заявки)
Retention: 7 дней
Delivery guarantee: at-least-once
Идемпотентность: consumer проверяет applicationId в БД перед обработкой

Тело сообщения:
{
  "applicationId": "uuid",
  "clientInn": "string(12)",
  "clientType": "enum: IP | OOO",
  "createdAt": "ISO-8601 datetime",
  "status": "enum: NEW | VALIDATING | APPROVED | REJECTED"
}

DLQ: registration.application.created.dlq
Условие попадания в DLQ: 3 неудачные попытки обработки
Алерт: при появлении сообщения в DLQ — оповещение в Slack-канал #registration-alerts
```

## 7. Специфика банковской аналитики

### 7.1 Регуляторные требования

Банковский аналитик работает в контексте жёстких регуляторных ограничений:

- **ЦБ РФ** — требования к информационной безопасности (683-П, 719-П), отчётность, лицензирование
- **115-ФЗ (ПОД/ФТ)** — противодействие отмыванию денежных средств и финансированию терроризма. Обязательная проверка клиентов, мониторинг подозрительных операций
- **152-ФЗ** — защита персональных данных. Хранение, обработка, трансграничная передача
- **ФНС** — интеграция через СМЭВ, электронные подписи, форматы обмена

### 7.2 KYC / AML / Compliance

**KYC (Know Your Customer)** — процесс идентификации и верификации клиента. Уровни: упрощённая идентификация, стандартная, углублённая (EDD — Enhanced Due Diligence). Аналитик описывает: какие данные собираются, какие проверки проводятся (ЕГРЮЛ, списки Росфинмониторинга, санкционные списки), критерии принятия решений.

**AML (Anti-Money Laundering)** — мониторинг операций на предмет подозрительной активности. Правила и пороги, автоматические и ручные проверки.

### 7.3 BPMN в банках

В банках BPMN используется не просто для документирования — это **исполняемые процессы**. Camunda / Zeebe / JBPM — BPM-движки, которые реально запускают процесс по BPMN-диаграмме. Service Task → вызов микросервиса, User Task → задача оператору в UI, Timer Event → SLA на выполнение.

Это означает, что BPMN-диаграмма — это одновременно и документация, и код. Ошибка в диаграмме = баг в продакшене.

## 8. Тестирование аналитики и артефактов аналитики

### 8.1 Зачем тестировать аналитику

По различным исследованиям, от 40% до 60% дефектов в ПО зарождаются на этапе требований. Стоимость исправления дефекта, найденного на этапе требований, в 10-100 раз ниже, чем на этапе тестирования готового продукта. Shift-left тестирование — перенос проверок на ранние этапы — начинается именно с ревью требований.

### 8.2 Процесс тестирования требований

**Статическое тестирование (review):**

1. **Inspection (инспекция)** — формальный процесс. Роли: автор, модератор, рецензенты, секретарь. Чек-лист проверки. Протокол с замечаниями. Наиболее эффективный, но дорогой метод.

2. **Walkthrough (прогулка)** — автор «проводит» рецензентов через документ, объясняя логику. Менее формальный, чем инспекция.

3. **Peer Review** — один или несколько коллег читают документ и дают комментарии. Наиболее распространён в Agile.

4. **Ad hoc Review** — неформальный просмотр «на ходу». Наименее эффективный.

### 8.3 Чек-лист для ревью требований

**Проверка каждого отдельного требования:**

- Однозначно ли сформулировано? Нет ли слов-маркеров неоднозначности?
- Тестируемо ли? Можно ли написать тест-кейс?
- Полно ли описано? Все ли ветки (happy path, альтернативные, ошибки)?
- Нет ли противоречий с другими требованиями?
- Указаны ли граничные значения и ограничения?
- Описано ли поведение при ошибках и исключительных ситуациях?
- Есть ли критерии приёмки?
- Реализуемо ли технически?

**Проверка набора требований в целом:**

- Все ли бизнес-требования декомпозированы до функциональных?
- Покрыты ли все пользовательские роли?
- Описаны ли нефункциональные требования?
- Нет ли дублирования?
- Есть ли матрица трассировки?
- Согласован ли глоссарий?

### 8.4 Тестирование диаграмм

**Тестирование Use Case Diagram:**
- Все ли акторы идентифицированы?
- Все ли варианты использования покрыты?
- Правильно ли использованы связи include/extend?
- Нет ли «потерянных» акторов без связей?

**Тестирование Sequence Diagram:**
- Все ли участники указаны?
- Корректен ли порядок сообщений?
- Описаны ли альтернативные потоки (alt-фрагменты)?
- Обработаны ли ошибки и таймауты?
- Соответствует ли диаграмма текстовому описанию API-контракта?

**Тестирование Activity Diagram:**
- Все ли пути от Start к End достижимы?
- Нет ли «мёртвых» ветвей (unreachable paths)?
- Правильно ли работают условия на ромбах (все варианты покрыты)?
- Правильно ли синхронизируются параллельные потоки (fork → join)?

**Тестирование State Machine Diagram:**
- Все ли состояния достижимы из начального?
- Из каждого ли состояния есть путь к конечному?
- Нет ли «тупиковых» состояний (deadlock)?
- Все ли переходы имеют условия/триггеры?
- Что происходит при невалидном переходе?

**Тестирование BPMN-диаграмм:**
- Happy path — процесс проходит от начала до конца?
- Каждый Exclusive Gateway — все ветки покрыты, условия не пересекаются, есть default?
- Parallel Gateway — fork и join совпадают по количеству потоков?
- Boundary Events — что происходит по таймеру? При ошибке?
- User Tasks — правильный assignee? SLA?
- Service Tasks — какой сервис вызывается? Что при недоступности?
- Message Events — корректные топики/очереди? Формат сообщений?

### 8.5 Тестирование API-контрактов

- Все ли эндпоинты описаны (CRUD для каждой сущности)?
- Для каждого эндпоинта: все ли HTTP-коды описаны (200, 201, 400, 401, 403, 404, 409, 500)?
- Формат запроса: все ли поля описаны? Типы? Обязательность? Валидации?
- Формат ответа: структура? Пагинация? Формат ошибок?
- Авторизация: какой механизм? Что при истёкшем токене?
- Идемпотентность: PUT/DELETE идемпотентны? POST — как обрабатывается повторный запрос?
- Версионирование API: как обеспечивается обратная совместимость?

### 8.6 Тестирование модели данных

- Все ли сущности из требований отражены в ER-диаграмме?
- Правильно ли определены связи (1:1, 1:N, N:M)?
- Все ли обязательные поля помечены (NOT NULL)?
- Есть ли индексы для полей, по которым будет поиск/фильтрация?
- Нормализована ли модель (нет дублирования данных)?
- Описаны ли каскадные удаления/обновления?
- Соответствуют ли типы данных бизнес-логике (например, для хранения денежных сумм — DECIMAL, не FLOAT)?

### 8.7 Типичные дефекты в аналитике

1. **Пропущенный сценарий** — не описано, что происходит при ошибке, таймауте, отмене операции
2. **Неоднозначная формулировка** — «система обрабатывает запрос в разумные сроки»
3. **Противоречие между документами** — в SRS одна логика, в API-спецификации другая
4. **Недостающие граничные условия** — максимальная длина поля, допустимые символы, лимиты
5. **Несогласованная терминология** — одно и то же понятие называется по-разному
6. **Отсутствие негативных сценариев** — описан только happy path
7. **Неполная статусная модель** — нет переходов для всех состояний
8. **Рассогласование между диаграммой и текстом** — sequence diagram показывает одно, текст описывает другое
9. **Отсутствие NFR** — нет требований к производительности, безопасности
10. **Нетестируемые критерии приёмки** — «система должна работать стабильно»
