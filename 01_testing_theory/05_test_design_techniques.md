# Техники тест-дизайна

## Обзор

Техники тест-дизайна (test design techniques) -- это формальные методы создания тестовых
сценариев, позволяющие достичь максимального покрытия при минимальном количестве тестов.
Без этих техник тестировщик полагается только на интуицию, что приводит к пробелам
в покрытии и избыточным проверкам.

На интервью знание техник тест-дизайна -- один из ключевых показателей квалификации
QA-инженера. Недостаточно просто знать названия -- нужно уметь применять каждую технику
на конкретном примере и объяснять, как получены тестовые данные.

В этом разделе рассмотрены шесть основных техник с пошаговыми примерами,
результирующими тест-кейсами и ASCII-диаграммами.

## 1. Equivalence Partitioning (Эквивалентное разбиение)

### Суть метода

Входные данные разбиваются на группы (классы эквивалентности), для которых поведение
системы одинаково. Из каждого класса берётся одно представительное значение.

**Принцип:** если система корректно работает с одним значением из класса,
она будет корректно работать со всеми значениями этого класса.

### Пошаговый пример: поле "Возраст" (допустимые значения 1-120)

**Шаг 1.** Определить входные данные и ограничения.
- Поле: возраст
- Тип: целое число
- Допустимый диапазон: 1-120

**Шаг 2.** Выделить классы эквивалентности.

```
Невалидный класс 1:   ..., -2, -1, 0         (значения < 1)
Валидный класс:        1, 2, 3, ..., 119, 120 (значения от 1 до 120)
Невалидный класс 2:   121, 122, 123, ...      (значения > 120)
Невалидный класс 3:   "abc", "@#$", ""        (нечисловые значения)
Невалидный класс 4:   1.5, 99.9              (дробные числа)
```

**Шаг 3.** Выбрать по одному представителю из каждого класса.

| # | Класс                | Значение | Ожидаемый результат      |
|---|----------------------|----------|--------------------------|
| 1 | Значения < 1         | -5       | Ошибка валидации         |
| 2 | Значения 1-120       | 25       | Значение принято         |
| 3 | Значения > 120       | 200      | Ошибка валидации         |
| 4 | Нечисловые значения  | "abc"    | Ошибка валидации         |
| 5 | Дробные числа        | 3.7      | Ошибка валидации         |

**Результат:** 5 тестов вместо бесконечного количества возможных значений.

### Ещё один пример: поле "Email" при регистрации

| # | Класс                               | Пример                | Результат     |
|---|-------------------------------------|-----------------------|---------------|
| 1 | Корректный email                    | user@example.com      | Принято       |
| 2 | Без символа @                       | userexample.com       | Ошибка        |
| 3 | Без домена                          | user@                 | Ошибка        |
| 4 | Без имени пользователя              | @example.com          | Ошибка        |
| 5 | С пробелами                         | us er@example.com     | Ошибка        |
| 6 | Пустая строка                       | (пусто)               | Ошибка        |

## 2. Boundary Value Analysis (Анализ граничных значений)

### Суть метода

Ошибки чаще всего возникают на границах диапазонов. Поэтому тестируются значения
на границе, непосредственно перед границей и сразу после неё.

### Техника двух точек vs трёх точек

**Двухточечная (two-value):** граница и значение за границей.
**Трёхточечная (three-value):** значение до границы, граница, значение после границы.

### Пошаговый пример: поле "Возраст" (допустимые значения 1-120)

**Шаг 1.** Определить границы.
- Нижняя граница: 1
- Верхняя граница: 120

**Шаг 2.** Определить граничные значения (трёхточечный подход).

```
Нижняя граница:                Верхняя граница:

    невалидно | валидно            валидно | невалидно
         0    |  1    2          119  120  |  121
         ^    |  ^    ^           ^    ^   |   ^
       перед  | граница после   перед граница  после
```

**Шаг 3.** Составить тест-кейсы.

| #  | Значение | Описание                     | Ожидаемый результат |
|----|----------|------------------------------|---------------------|
| 1  | 0        | На 1 меньше нижней границы   | Ошибка валидации    |
| 2  | 1        | Нижняя граница               | Значение принято    |
| 3  | 2        | На 1 больше нижней границы   | Значение принято    |
| 4  | 119      | На 1 меньше верхней границы  | Значение принято    |
| 5  | 120      | Верхняя граница              | Значение принято    |
| 6  | 121      | На 1 больше верхней границы  | Ошибка валидации    |

### Комбинация с Equivalence Partitioning

На практике эти две техники применяются вместе:
- Equivalence Partitioning определяет классы
- Boundary Value Analysis уточняет значения на границах классов

**Полный набор для поля "Возраст" (1-120):**

| # | Техника   | Значение | Класс              | Результат |
|---|-----------|----------|---------------------|-----------|
| 1 | BVA       | 0        | Невалидный (< 1)   | Ошибка    |
| 2 | BVA       | 1        | Граница (нижняя)    | Принято   |
| 3 | BVA       | 2        | Валидный            | Принято   |
| 4 | EP        | 60       | Валидный (середина) | Принято   |
| 5 | BVA       | 119      | Валидный            | Принято   |
| 6 | BVA       | 120      | Граница (верхняя)   | Принято   |
| 7 | BVA       | 121      | Невалидный (> 120)  | Ошибка    |
| 8 | EP        | "abc"    | Невалидный (текст)  | Ошибка    |
| 9 | EP        | -100     | Невалидный (< 1)    | Ошибка    |

## 3. Decision Table (Таблица решений)

### Суть метода

Используется, когда результат зависит от комбинации нескольких условий.
Таблица решений систематически перечисляет все комбинации условий и соответствующие действия.

### Пошаговый пример: система скидок интернет-магазина

**Бизнес-правила:**
- Если клиент VIP **И** сумма заказа > 5000 руб. --> скидка 20%
- Если клиент VIP **И** сумма заказа <= 5000 руб. --> скидка 10%
- Если клиент обычный **И** сумма заказа > 5000 руб. --> скидка 5%
- Если клиент обычный **И** сумма заказа <= 5000 руб. --> скидка 0%

**Шаг 1.** Определить условия и действия.
- Условие 1: Статус клиента (VIP / Обычный)
- Условие 2: Сумма заказа (> 5000 / <= 5000)
- Действие: Размер скидки

**Шаг 2.** Построить таблицу решений.

| Правило           | R1    | R2     | R3      | R4      |
|--------------------|-------|--------|---------|---------|
| **Условия**        |       |        |         |         |
| Клиент VIP?        | Да    | Да     | Нет     | Нет     |
| Сумма > 5000?      | Да    | Нет    | Да      | Нет     |
| **Действия**       |       |        |         |         |
| Скидка             | 20%   | 10%    | 5%      | 0%      |

**Шаг 3.** Составить тест-кейсы (один кейс на каждое правило).

| TC  | Статус  | Сумма   | Ожидаемая скидка |
|-----|---------|---------|------------------|
| TC1 | VIP     | 7000    | 20%              |
| TC2 | VIP     | 3000    | 10%              |
| TC3 | Обычный | 7000    | 5%               |
| TC4 | Обычный | 3000    | 0%               |

### Расширенный пример: три условия

**Бизнес-правила для доставки:**
- Условие 1: Город (Москва / Другой)
- Условие 2: Вес (> 10 кг / <= 10 кг)
- Условие 3: Срочная доставка (Да / Нет)

Количество комбинаций: 2 x 2 x 2 = **8 правил**.

| Правило | Москва? | Вес > 10 кг? | Срочная? | Стоимость доставки |
|---------|---------|--------------|----------|--------------------|
| R1      | Да      | Да           | Да       | 1000 руб.          |
| R2      | Да      | Да           | Нет      | 500 руб.           |
| R3      | Да      | Нет          | Да       | 500 руб.           |
| R4      | Да      | Нет          | Нет      | 0 руб. (бесплатно) |
| R5      | Нет     | Да           | Да       | 2000 руб.          |
| R6      | Нет     | Да           | Нет      | 1000 руб.          |
| R7      | Нет     | Нет          | Да       | 1000 руб.          |
| R8      | Нет     | Нет          | Нет      | 500 руб.           |

## 4. State Transition Diagram (Диаграмма переходов состояний)

### Суть метода

Применяется, когда система ведёт себя по-разному в зависимости от текущего состояния
и произошедшего события. Метод моделирует все допустимые состояния и переходы между ними.

### Пошаговый пример: статусы заказа в интернет-магазине

**Шаг 1.** Определить состояния.

```
[Создан] --> [Оплачен] --> [В обработке] --> [Отправлен] --> [Доставлен]
                                                                   |
                                                              [Завершён]
```

С возможностью отмены:

```
                         [Отменён]
                        ^  ^  ^
                       /   |   \
[Создан] --> [Оплачен] --> [В обработке] --> [Отправлен] --> [Доставлен]
   |                                                              |
   v                                                         [Завершён]
[Отменён]
```

**Шаг 2.** Определить события (triggers) и составить таблицу переходов.

| Текущее состояние | Событие           | Новое состояние | Условие              |
|-------------------|--------------------|-----------------|----------------------|
| Создан            | Оплата получена    | Оплачен         | --                   |
| Создан            | Отмена клиентом    | Отменён         | --                   |
| Оплачен           | Начало сборки      | В обработке     | --                   |
| Оплачен           | Отмена клиентом    | Отменён         | Возврат средств      |
| В обработке       | Передан курьеру    | Отправлен       | --                   |
| В обработке       | Отмена клиентом    | Отменён         | Возврат средств      |
| Отправлен         | Курьер доставил    | Доставлен       | --                   |
| Доставлен         | Подтверждение      | Завершён        | Через 14 дней авто   |
| Отменён           | --                 | --              | Конечное состояние   |
| Завершён          | --                 | --              | Конечное состояние   |

**Шаг 3.** Составить тест-кейсы -- по одному на каждый переход.

| TC  | Путь                                                          |
|-----|---------------------------------------------------------------|
| TC1 | Создан -> Оплачен -> В обработке -> Отправлен -> Доставлен -> Завершён |
| TC2 | Создан -> Отменён                                             |
| TC3 | Создан -> Оплачен -> Отменён (проверка возврата)              |
| TC4 | Создан -> Оплачен -> В обработке -> Отменён (проверка возврата)|
| TC5 | Отправлен -> Доставлен (курьер) -> Завершён (авто через 14 дней)|

**Шаг 4.** Проверить невалидные переходы (negative).

| TC  | Попытка перехода                 | Ожидаемый результат          |
|-----|----------------------------------|------------------------------|
| TC6 | Создан -> В обработке (без оплаты)| Переход запрещён            |
| TC7 | Отправлен -> Оплачен             | Переход запрещён             |
| TC8 | Завершён -> Создан               | Переход запрещён             |
| TC9 | Отменён -> Оплачен               | Переход запрещён             |

## 5. Pairwise Testing (Попарное тестирование)

### Суть метода

Большинство дефектов вызываются взаимодействием не более двух параметров.
Pairwise testing гарантирует, что каждая пара значений любых двух параметров
встречается хотя бы в одном тесте. Это резко сокращает количество тестов
по сравнению с полным перебором.

### Пошаговый пример: тестирование веб-приложения

**Параметры и значения:**
- Браузер: Chrome, Firefox, Safari
- ОС: Windows, macOS, Linux
- Язык: RU, EN

**Полный перебор:** 3 x 3 x 2 = **18 комбинаций**

**Pairwise -- покрытие всех пар:**

| TC | Браузер  | ОС       | Язык |
|----|----------|----------|------|
| 1  | Chrome   | Windows  | RU   |
| 2  | Chrome   | macOS    | EN   |
| 3  | Chrome   | Linux    | RU   |
| 4  | Firefox  | Windows  | EN   |
| 5  | Firefox  | macOS    | RU   |
| 6  | Firefox  | Linux    | EN   |
| 7  | Safari   | Windows  | RU   |
| 8  | Safari   | macOS    | EN   |
| 9  | Safari   | Linux    | RU   |

**Результат:** 9 тестов вместо 18 (50% сокращение). При большем количестве параметров
экономия значительно выше.

### Инструмент PICT (Pairwise Independent Combinatorial Testing)

**PICT** -- бесплатный инструмент от Microsoft для генерации pairwise-комбинаций.

**Шаг 1.** Создать файл модели (`model.txt`):

```
Browser:  Chrome, Firefox, Safari, Edge
OS:       Windows, macOS, Linux
Language: RU, EN, DE
Screen:   1920x1080, 1366x768, 375x667
```

**Шаг 2.** Запустить PICT:

```bash
pict model.txt
```

**Шаг 3.** Получить результат (пример):

```
Browser   OS        Language  Screen
Chrome    Windows   RU        1920x1080
Firefox   macOS     EN        1366x768
Safari    Linux     DE        375x667
Edge      Windows   EN        375x667
Chrome    macOS     DE        1366x768
Firefox   Linux     RU        1920x1080
Safari    Windows   DE        1920x1080
Edge      macOS     RU        375x667
Chrome    Linux     EN        1920x1080
Edge      Linux     EN        1366x768
Firefox   Windows   DE        375x667
Safari    macOS     RU        1366x768
```

**Полный перебор:** 4 x 3 x 3 x 3 = **108 комбинаций**
**Pairwise (PICT):** ~**12 комбинаций** (89% сокращение!)

### Когда применять

- Множество параметров с несколькими значениями каждый
- Нет ресурсов на полный перебор
- Конфигурационное тестирование (ОС, браузеры, устройства)
- Тестирование форм с множеством полей

## 6. Error Guessing (Предугадывание ошибок)

### Суть метода

Опытный тестировщик на основе знаний о типичных ошибках предугадывает,
где система может сломаться. Это неформальная техника, дополняющая формальные методы.

### Типичные области для error guessing

**Граничные и специальные значения:**

| Категория           | Примеры значений                                    |
|---------------------|-----------------------------------------------------|
| Пустые значения     | null, "", пробел, табуляция                         |
| Нулевые значения    | 0, 0.0, -0                                          |
| Отрицательные числа | -1, -999999                                         |
| Очень большие числа | 999999999, MAX_INT (2147483647)                     |
| Длинные строки      | 10 000 символов, 1 000 000 символов                 |
| Спецсимволы         | `<>'"&\;/!@#$%^*()`                                 |
| Unicode             | Иероглифы, эмодзи, арабская вязь (RTL)              |
| SQL-инъекции        | `' OR 1=1 --`, `'; DROP TABLE users; --`            |
| XSS                 | `<script>alert('XSS')</script>`                     |
| Форматы дат         | 30.02.2024, 31.04.2024, 29.02.2023 (не високосный)  |

**Типичные сценарии:**

| Сценарий                        | Что проверяем                                |
|---------------------------------|----------------------------------------------|
| Двойной клик на кнопку "Оплатить"| Двойное списание средств                    |
| Нажатие "Назад" после оплаты   | Повторная оплата или ошибка                  |
| Параллельное редактирование     | Конфликт данных (race condition)             |
| Разрыв соединения во время операции | Корректная обработка timeout              |
| Заполнение формы и обновление страницы | Потеря данных                          |
| Копипаст текста с форматированием | Некорректное отображение                   |
| Загрузка файла размером 0 байт  | Обработка пустого файла                     |
| Загрузка .exe вместо .jpg       | Валидация типа файла                        |

### Чек-лист для error guessing

```
[ ] Пустые/null значения во всех полях
[ ] Граничные значения числовых полей
[ ] Максимальная длина текстовых полей
[ ] Спецсимволы и Unicode
[ ] SQL-инъекции в текстовых полях
[ ] XSS в текстовых полях
[ ] Двойная отправка формы
[ ] Навигация кнопкой "Назад"
[ ] Работа с истёкшей сессией
[ ] Параллельные действия в двух вкладках
[ ] Медленное соединение / таймауты
[ ] Загрузка файлов неверного формата/размера
```

## Связь с тестированием

Техники тест-дизайна -- ежедневный инструмент QA-инженера:

1. **Написание тест-кейсов:** EP + BVA -- основа для любого поля ввода
2. **Ревью требований:** Decision Table помогает найти пропущенные бизнес-правила
3. **Тестирование workflow:** State Transition Diagram -- для любых статусных моделей
4. **Конфигурационное тестирование:** Pairwise сокращает количество комбинаций
5. **Exploratory testing:** Error Guessing направляет исследование
6. **Автоматизация:** результаты техник -- готовые данные для параметризованных тестов

**Пример применения в автотесте (JUnit 5, parameterized):**

```java
// Тест-кейсы, полученные техниками EP + BVA для поля "Возраст"
@ParameterizedTest
@CsvSource({
    "0, false",    // BVA: ниже нижней границы
    "1, true",     // BVA: нижняя граница
    "2, true",     // BVA: выше нижней границы
    "60, true",    // EP: середина валидного диапазона
    "119, true",   // BVA: ниже верхней границы
    "120, true",   // BVA: верхняя граница
    "121, false"   // BVA: выше верхней границы
})
void testAgeValidation(int age, boolean expectedValid) {
    assertEquals(expectedValid, validator.isValidAge(age));
}
```

## Типичные ошибки

1. **Применять только одну технику.** EP без BVA пропустит граничные баги. BVA без EP пропустит классы данных.
2. **Забывать невалидные классы в EP.** Невалидных классов обычно больше, чем валидных.
3. **Путать двухточечный и трёхточечный BVA.** На интервью уточняйте, какой вариант используете.
4. **Не учитывать комбинации условий.** Если результат зависит от нескольких факторов -- нужна Decision Table, а не просто EP.
5. **Использовать pairwise, когда нужен полный перебор.** Для критичных комбинаций (например, платёжная система) полный перебор может быть обязателен.
6. **Считать error guessing заменой формальным техникам.** Это дополнение, а не замена.
7. **Забывать про невалидные переходы в State Transition.** Проверка запрещённых переходов не менее важна, чем проверка валидных.

## Вопросы на интервью

### 🟢 Базовый уровень

1. **Что такое Equivalence Partitioning? Приведите пример.**
   Разбиение входных данных на классы с одинаковым поведением. Пример: поле "Возраст" 1-120 -- три класса: < 1, 1-120, > 120.

2. **Что такое Boundary Value Analysis? Какие значения тестируются?**
   Тестирование на границах диапазонов. Для диапазона 1-120: значения 0, 1, 2, 119, 120, 121.

3. **Чем EP отличается от BVA?**
   EP определяет классы данных (любое значение из класса). BVA фокусируется на границах между классами. Применяются вместе.

4. **Что такое error guessing?**
   Неформальная техника, основанная на опыте тестировщика. Предугадывание мест, где система может сломаться (null, спецсимволы, двойные клики и т.д.).

### 🟡 Средний уровень

5. **Составьте Decision Table для системы: скидка зависит от статуса клиента (VIP/обычный) и способа оплаты (карта/наличные).**
   4 правила (2x2). Нужно уточнить бизнес-логику для каждой комбинации и создать по одному тест-кейсу на правило.

6. **Есть форма с полем "Количество товара" (1-99). Примените EP и BVA. Сколько тест-кейсов получится?**
   EP: < 1, 1-99, > 99, нечисловые, дробные = 5 классов. BVA: 0, 1, 2, 98, 99, 100 = 6 значений. Итого ~8-9 уникальных тест-кейсов.

7. **Нарисуйте State Transition Diagram для банкомата (вставка карты, ввод PIN, выдача денег).**
   Состояния: Ожидание -> Карта вставлена -> PIN введён -> Сумма выбрана -> Деньги выданы. Плюс: неверный PIN (3 попытки), отмена, карта заблокирована.

8. **Когда pairwise testing неуместен?**
   Когда важны конкретные комбинации (например, оплата Visa + USD + 3DS в платёжной системе). Pairwise может пропустить именно эту комбинацию.

### 🔴 Продвинутый уровень

9. **У вас форма с 5 полями, каждое имеет 4 варианта. Полный перебор -- 1024 теста. Как сократить?**
   Pairwise: ~16-20 тестов. Дополнительно: risk-based подход для критичных комбинаций, EP + BVA для каждого поля отдельно.

10. **Как формализовать применение error guessing в команде?**
    Создать чек-лист типичных ошибок (по категориям), пополнять его на основе реальных багов, включить в definition of done. Использовать как дополнение к формальным техникам.

11. **Как выбрать технику тест-дизайна для конкретной задачи?**
    Одно поле с диапазоном -> EP + BVA. Несколько взаимосвязанных условий -> Decision Table. Workflow со статусами -> State Transition. Много параметров -> Pairwise. Всегда дополнять Error Guessing.
