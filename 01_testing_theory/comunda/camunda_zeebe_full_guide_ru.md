# Camunda и Zeebe: полное руководство

> Подробный документ для ознакомления с технологией. Объясняет по-простому, с деталями и примерами кода.

---

## Часть I. Основы

### 1. Что такое Camunda — простыми словами

Представь бизнес-процесс: клиент оформляет заявку на кредит. Заявка проходит несколько этапов — проверка данных, скоринг, одобрение менеджером, формирование договора, отправка уведомления. Каждый этап — это либо автоматическое действие (вызов API скоринга), либо действие человека (менеджер нажимает «одобрить»).

**Camunda** — платформа, которая позволяет нарисовать такой процесс в виде диаграммы (BPMN), а потом **исполнять** эту диаграмму. Не красивая картинка для документации, а реально работающая схема: вызывает микросервисы, ждёт ответа от человека, обрабатывает ошибки, делает ветвления.

Ключевое: логика *что за чем идёт* описана не в Java-коде, а в визуальной диаграмме. Код отвечает только за *как выполнить конкретный шаг*. Бизнес-аналитик и разработчик смотрят на одну и ту же диаграмму и понимают друг друга.

### 2. Что такое Zeebe

**Zeebe** — движок внутри Camunda 8. Если Camunda — это автомобиль, то Zeebe — двигатель. Именно Zeebe исполняет BPMN-диаграммы: ведёт каждый экземпляр процесса от старта до конца, создаёт задания для worker'ов, хранит состояние.

Zeebe спроектирован для облака: горизонтально масштабируется, не имеет единой точки отказа, хранит данные распределённо. Написан на Java, но работать с ним можно из любого языка через gRPC и REST API.

### 3. Архитектура Camunda 8 — все компоненты

Camunda 8 — не один сервис, а набор компонентов:

**Ядро (Orchestration Cluster):**

| Компонент | Назначение |
|-----------|------------|
| **Zeebe** | Движок процессов. Исполняет BPMN, хранит состояние, распределяет задания |
| **Operate** | Веб-UI для мониторинга: все запущенные процессы, их статус, ошибки |
| **Tasklist** | Веб-UI для человеческих задач: сотрудники видят и выполняют User Task'и |
| **Identity** | Аутентификация и авторизация |
| **Optimize** | Аналитика: узкие места, время выполнения шагов, KPI |

**Внутри Zeebe** три ключевых элемента. **Broker** — хранит состояние процессов, обрабатывает команды; может работать как кластер из нескольких нод для отказоустойчивости. **Gateway** — точка входа для клиентов (REST + gRPC), маршрутизирует запросы к нужному брокеру. **Exporter** — выгружает события в Elasticsearch, откуда их читают Operate, Tasklist и Optimize.

**Инструменты разработки:** Camunda Modeler (десктопное приложение для рисования BPMN), Web Modeler (в браузере), Connectors (готовые интеграции с внешними системами).

## Часть II. BPMN-нотация в деталях

### 4. Базовые элементы BPMN

BPMN (Business Process Model and Notation) — стандарт ISO для описания бизнес-процессов. Основные категории символов:

**События (Events)** — круглые символы. Обозначают «что-то произошло»:

| Символ | Тип | Пример |
|--------|-----|--------|
| Тонкий круг | Start Event | Клиент подал заявку |
| Двойной круг | Intermediate Event | Прошло 24 часа / пришло сообщение |
| Жирный круг | End Event | Процесс завершён |

Внутри круга — значок, определяющий природу события: конверт (message), часы (timer), молния (error), сигнал (signal). Например, Start Message Event означает «процесс запускается при получении сообщения».

**Задачи (Tasks)** — прямоугольники со скруглёнными углами. Типы:

| Тип | Иконка | Что делает |
|-----|--------|------------|
| Service Task | Шестерёнка | Автоматическое действие — выполняет Job Worker |
| User Task | Человечек | Ждёт действия сотрудника через Tasklist |
| Send Task | Конверт | Отправка сообщения |
| Receive Task | Конверт (входящий) | Ожидание внешнего сообщения |
| Script Task | Свиток | Выполнение встроенного скрипта |
| Business Rule Task | Таблица | Вызов DMN-таблицы решений |
| Manual Task | Рука | Действие за пределами системы (не автоматизируется) |

**Шлюзы (Gateways)** — ромбы. Управляют потоком:

| Тип | Символ | Поведение |
|-----|--------|-----------|
| Exclusive (XOR) | × | Только одна ветка (if/else). Условия на стрелках |
| Parallel (AND) | + | Все ветки одновременно. Ждёт завершения всех |
| Inclusive (OR) | ○ | Одна или несколько веток по условиям |
| Event-Based | Пятиугольник | Ждёт первое из нескольких событий (таймер ИЛИ сообщение) |

**Пулы и дорожки (Pools & Lanes)** — визуальное разделение участников. Пул = организация или система, дорожка = роль внутри (HR, менеджер, бухгалтерия). На исполнение не влияют, но помогают понять кто за что отвечает.

### 5. Boundary Events — обработка исключений прямо на диаграмме

Boundary Event — событие, «прикреплённое» к задаче. Срабатывает пока задача выполняется. Два типа:

**Прерывающий (interrupting):** отменяет задачу и уводит процесс по альтернативному маршруту. Пример: к задаче «Подтверждение менеджером» прикреплён Timer Boundary Event на 48 часов. Если менеджер не ответил за 48 часов — задача отменяется, процесс идёт к «Эскалация руководителю».

**Непрерывающий (non-interrupting):** запускает параллельную ветку, но задача продолжает работать. Пример: к задаче «Ожидание оплаты» прикреплён Timer Non-Interrupting на 24 часа. Через 24 часа отправляется напоминание клиенту, но ожидание оплаты продолжается.

Это мощнейший инструмент BPMN — он позволяет описать таймауты, ошибки и эскалации визуально, без единой строчки if/else в коде.

## Часть III. Как Zeebe связан с кодом

### 6. Паттерн External Task — ключевой принцип

В Camunda 8 используется паттерн **External Task**. Zeebe **не вызывает** твой код (не делает HTTP-запрос к твоему сервису). Наоборот: **твой код сам приходит** к Zeebe и спрашивает «есть работа для меня?»

Почему это важно:

**Нет проблем с доступностью:** если твой сервис упал, Zeebe просто копит задания. Когда сервис поднимется — заберёт всё что накопилось. Нет потерянных вызовов.

**Backpressure:** worker сам контролирует, сколько заданий берёт. Zeebe не может его «завалить» — worker запрашивает ровно столько, сколько способен обработать.

**Масштабирование:** запускаешь 5 экземпляров worker'а — Zeebe автоматически распределяет задания между ними. Каждое задание достаётся только одному worker'у.

**Язык не важен:** worker может быть на Java, Go, Python, C#, Node.js — главное, чтобы умел общаться с Zeebe через gRPC или REST.

### 7. Job Worker на Spring Boot — подробный пример

Типичный Spring Boot сервис, работающий с Zeebe:

```java
@SpringBootApplication
@Deployment(resources = "classpath:order-process.bpmn")
public class OrderServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderServiceApplication.class, args);
    }
}
```

Аннотация `@Deployment` при старте приложения деплоит BPMN-файл в Zeebe. Это одноразовое действие — если диаграмма не изменилась, повторный деплой её не дублирует.

Запуск экземпляра процесса (например, из REST-контроллера):

```java
@RestController
public class OrderController {
    @Autowired
    private ZeebeClient zeebeClient;

    @PostMapping("/api/orders")
    public ResponseEntity<?> createOrder(@RequestBody OrderRequest req) {
        // Создаём экземпляр BPMN-процесса с переменными
        zeebeClient.newCreateInstanceCommand()
            .bpmnProcessId("order-process")
            .latestVersion()
            .variables(Map.of(
                "orderId", req.getOrderId(),
                "amount", req.getAmount(),
                "customerEmail", req.getEmail()
            ))
            .send()
            .join();
        return ResponseEntity.accepted().build();
    }
}
```

Job Worker — обработчик конкретного шага процесса:

```java
@Component
public class PaymentWorker {

    // Zeebe автоматически находит этот метод и подписывает его на задачи типа "process-payment"
    @JobWorker(type = "process-payment")
    public Map<String, Object> processPayment(ActivatedJob job) {
        // Получаем переменные процесса
        String orderId = (String) job.getVariablesAsMap().get("orderId");
        Double amount = (Double) job.getVariablesAsMap().get("amount");

        // Выполняем бизнес-логику — например, вызываем платёжный шлюз
        boolean success = paymentGateway.charge(orderId, amount);

        // Возвращаем результат — он станет переменной процесса
        // На шлюзе Zeebe проверит: paymentSuccess == true?
        return Map.of("paymentSuccess", success);
    }
}
```

Что тут происходит под капотом: Spring Zeebe Starter при старте приложения регистрирует все методы с `@JobWorker`, открывает long-polling соединение с Zeebe Gateway (по gRPC), и когда появляется задание с нужным типом — вызывает твой метод. После успешного возврата автоматически отправляет в Zeebe команду «задание завершено» с результатами.

### 8. Переменные и данные в процессе

Каждый экземпляр процесса имеет набор **переменных** — это Map<String, Object>, хранящийся в Zeebe. Переменные — основной способ передачи данных между шагами.

При создании экземпляра ты задаёшь начальные переменные. Каждый Job Worker может прочитать любые переменные и добавить/изменить их. На шлюзах условия проверяются по переменным.

Важные нюансы:

**Скоупы:** переменные могут быть глобальные (для всего процесса) и локальные (для подпроцесса). Локальные не видны за пределами подпроцесса — это позволяет изолировать данные.

**Input/Output Mappings:** в Camunda Modeler можно настроить, какие переменные передаются в задачу (input) и какие возвращаются (output). Это позволяет не тащить весь набор переменных в каждый worker — отправляешь только нужные.

**FetchVariables:** в `@JobWorker` можно указать, какие переменные тебе нужны. Zeebe отправит только их, а не весь набор. Это оптимизация для процессов с большими данными.

```java
@JobWorker(type = "send-email", fetchVariables = {"customerEmail", "orderId"})
public void sendEmail(ActivatedJob job) {
    String email = (String) job.getVariablesAsMap().get("customerEmail");
    // переменная "amount" сюда не придёт — не запрашивали
}
```

**Сериализация:** переменные хранятся как JSON. Можно передавать примитивы, строки, списки, вложенные объекты. Но не стоит хранить в переменных большие файлы — лучше ссылку (URL или ID в хранилище).

## Часть IV. Обработка ошибок и отказоустойчивость

### 9. Ретраи и incidents

Когда Job Worker падает с ошибкой (exception), Zeebe не паникует. Механизм такой:

1. При создании задания Zeebe назначает **количество ретраев** (по умолчанию 3)
2. Если worker бросил исключение — Zeebe уменьшает счётчик ретраев на 1 и через backoff-интервал отдаёт задание снова (возможно другому worker'у)
3. Если ретраи закончились (счётчик = 0) — создаётся **incident** (инцидент)
4. Инцидент виден в Operate — оператор может исправить данные и возобновить процесс

Ты можешь контролировать ретраи программно:

```java
@JobWorker(type = "call-external-api")
public void callApi(JobClient client, ActivatedJob job) {
    try {
        externalService.call();
        client.newCompleteCommand(job.getKey()).send().join();
    } catch (TemporaryException e) {
        // Временная ошибка — уменьшаем ретраи, увеличиваем backoff
        client.newFailCommand(job.getKey())
            .retries(job.getRetries() - 1)
            .retryBackoff(Duration.ofSeconds(30))
            .errorMessage("API temporarily unavailable: " + e.getMessage())
            .send().join();
    } catch (BusinessException e) {
        // Бизнес-ошибка — бросаем BPMN Error, процесс пойдёт по альтернативному маршруту
        client.newThrowErrorCommand(job.getKey())
            .errorCode("PAYMENT_DECLINED")
            .errorMessage(e.getMessage())
            .send().join();
    }
}
```

### 10. BPMN Error vs техническая ошибка

Это критически важное различие:

**Техническая ошибка** (сеть упала, БД недоступна) — обрабатывается через ретраи. Worker получает задание повторно. Если ретраи исчерпаны — incident. Это **не** отражается на BPMN-диаграмме.

**Бизнес-ошибка** (карта заблокирована, товара нет на складе) — обрабатывается через **BPMN Error Event**. Worker бросает `throwError` с кодом ошибки. На диаграмме к задаче прикреплён Error Boundary Event с этим кодом — процесс уходит по альтернативному маршруту. Это **видно** на диаграмме, бизнес-аналитик понимает что происходит.

Пример на диаграмме:
```
[Списать оплату] ──── Error: PAYMENT_DECLINED ────→ [Уведомить клиента об отказе] → [Конец]
        │
        └──── (успех) ────→ [Собрать заказ] → ...
```

### 11. Saga-паттерн и компенсации

В микросервисной архитектуре нет единой транзакции. Если шаг 3 из 5 упал — нужно «откатить» шаги 1 и 2. Camunda реализует это через **Saga-паттерн**.

Как это выглядит в BPMN: каждый шаг имеет «компенсирующее действие». Если процесс упал — Zeebe запускает компенсации в обратном порядке.

Пример:
```
[Зарезервировать товар] → [Списать оплату] → [Отправить] → [Конец]
         ↓ компенсация          ↓ компенсация
   [Снять резерв]          [Вернуть деньги]
```

Если «Отправить» упал — Zeebe вызывает «Вернуть деньги», потом «Снять резерв». Вся эта логика — на диаграмме, не в коде.

В BPMN для этого используются Compensation Events и Compensation Handlers. Каждый шаг «помнит» свою компенсацию. При ошибке Zeebe проходит по цепочке в обратном порядке.

## Часть V. Пользователи и взаимодействие

### 12. User Task — человеческие задачи

Когда процесс доходит до User Task, Zeebe создаёт задачу в Tasklist. Сотрудник видит её в веб-интерфейсе: описание задачи, форму для заполнения, прикреплённые данные.

Формы можно создавать двумя способами: **Camunda Forms** (визуальный редактор в Modeler, JSON-описание) или **кастомный UI** (твой фронтенд общается с Tasklist API).

Типичный flow:
1. Процесс создаёт User Task с assignee = «manager-role»
2. Все менеджеры видят задачу в своём Tasklist
3. Один менеджер «берёт» задачу (claim) — остальные перестают её видеть
4. Заполняет форму (одобрить/отклонить + комментарий)
5. Нажимает «Завершить» — данные формы становятся переменными процесса
6. Zeebe продолжает процесс

Tasklist API (REST) позволяет встроить всё это в свой интерфейс — не обязательно использовать стандартный UI Camunda.

### 13. Как пользователь взаимодействует через свой фронтенд

Типичная архитектура: пользователь → твой фронтенд (React/Vue) → твой бэкенд (Spring Boot) → Zeebe.

Пользователь **не знает** про Camunda. Он работает с твоим приложением. А внутри:

```
Пользователь нажимает "Оформить заказ"
  → POST /api/orders (твой контроллер)
    → zeebeClient.createInstance("order-process", variables)
      → Zeebe ведёт процесс, раздаёт задания worker'ам
        → Когда нужен человек — User Task появляется в Tasklist
          → Твой фронтенд может запрашивать задачи через Tasklist REST API
            → Или менеджер работает в отдельном UI Tasklist
```

Для «обычных» пользователей (клиентов) ты прячешь Camunda за своим API. Для «внутренних» пользователей (менеджеров, операторов) можно использовать Tasklist напрямую или встроить его в корпоративный портал.

## Часть VI. Продвинутые возможности

### 14. DMN — таблицы решений

**DMN** (Decision Model and Notation) — отдельная нотация для бизнес-правил. Вместо цепочки if/else в коде — визуальная таблица.

Пример таблицы для определения скидки:

| Сумма заказа | Статус клиента | → Скидка |
|-------------|----------------|----------|
| < 1000 | любой | 0% |
| >= 1000 | обычный | 5% |
| >= 1000 | VIP | 10% |
| >= 5000 | VIP | 15% |

Эта таблица — тоже файл (`.dmn`), который деплоится в Zeebe. В BPMN-диаграмме ставишь **Business Rule Task**, указываешь ID таблицы — и Zeebe автоматически прогоняет входные данные через таблицу и возвращает результат.

**Hit Policy** определяет поведение при нескольких совпадениях: Unique (только одно правило), First (первое совпавшее), Collect (все совпавшие, можно суммировать/усреднить), и другие.

Зачем это нужно: бизнес-аналитик может менять правила скидок сам, без привлечения разработчика. Задеплоил новую версию DMN-таблицы — и всё работает.

### 15. Connectors — готовые интеграции

Connectors — это готовые Job Worker'ы для типовых задач. Вместо написания своего worker'а для REST-вызова или отправки в Kafka, ты используешь встроенный Connector прямо из диаграммы.

Доступные Connectors: REST (HTTP-вызовы), Kafka (publish/consume), RabbitMQ, Email (SMTP), Slack, Microsoft Teams, AWS (Lambda, SQS, SNS), Google Cloud, OpenAI, и десятки других.

Два типа: **Outbound** (процесс вызывает внешнюю систему) и **Inbound** (внешняя система запускает/продолжает процесс — webhook, Kafka consumer, подписка на события).

На диаграмме Connector выглядит как обычная задача, но конфигурируется в Properties Panel Modeler'а — указываешь URL, заголовки, тело запроса. Код писать не нужно.

Когда использовать Connector, а когда Job Worker: Connector — для простых интеграций (вызвать REST API, отправить сообщение). Job Worker — когда нужна сложная бизнес-логика, транзакции с БД, обработка ошибок с ретраями.

### 16. Подпроцессы и Call Activity

**Embedded Subprocess** — часть диаграммы, объединённая в блок. Визуально упрощает сложные процессы, позволяет прикрепить общий Error Boundary Event ко всему блоку.

**Call Activity** — вызов отдельного BPMN-процесса как подпроцедуры. Это как вызов функции в коде. Основной процесс ждёт завершения вызванного. Удобно для переиспользования: процесс «Проверка KYC» может вызываться из кредитного процесса, из процесса открытия счёта и т.д.

### 17. Таймеры

Три типа таймеров в BPMN:

**Timer Start Event** — процесс запускается по расписанию. Пример: каждый день в 9:00 запускать процесс «Ежедневная сверка».

**Timer Intermediate Event** — пауза в процессе. Пример: «Подождать 3 дня, потом отправить повторное уведомление».

**Timer Boundary Event** — дедлайн на задачу (описан выше в разделе про Boundary Events).

Таймеры задаются в формате ISO 8601: `PT24H` (24 часа), `P3D` (3 дня), `R3/PT1H` (повторять каждый час, 3 раза), или CRON-выражением.

### 18. Сообщения (Messages) и сигналы

**Message Event** — процесс ждёт конкретного сообщения для конкретного экземпляра. Пример: процесс оформления заказа ждёт webhook от платёжной системы с подтверждением оплаты. Сообщение адресуется через **correlation key** (например, orderId) — Zeebe сопоставляет сообщение с нужным экземпляром процесса.

Как отправить сообщение в Zeebe:

```java
zeebeClient.newPublishMessageCommand()
    .messageName("payment-confirmed")
    .correlationKey(orderId)      // Zeebe найдёт экземпляр, ожидающий сообщение с этим orderId
    .variables(Map.of("paymentId", "PAY-123"))
    .send()
    .join();
```

**Signal Event** — широковещательное: все экземпляры, ожидающие этот сигнал, получат его. Используется редко — например, «рынок закрылся, остановить все торговые процессы».

### 19. Мультиинстансы

Позволяют выполнить задачу или подпроцесс **для каждого элемента коллекции**. Аналог цикла for-each.

Пример: заказ содержит 5 товаров. Для каждого нужно проверить наличие на складе. Вместо 5 одинаковых задач на диаграмме — одна задача с пометкой «Multi-Instance», привязанная к переменной `items`. Zeebe создаст 5 экземпляров задачи (параллельно или последовательно — настраивается).

## Часть VII. Эксплуатация

### 20. Operate — мониторинг в реальном времени

Operate — веб-интерфейс, который показывает:

- Все запущенные экземпляры процессов (где каждый находится прямо на диаграмме — подсвечивается текущий шаг)
- Завершённые и отменённые экземпляры
- **Incidents** (инциденты) — задачи, исчерпавшие ретраи
- Переменные каждого экземпляра (можно редактировать для исправления данных)
- Хронологию выполнения (какие шаги пройдены, когда, сколько заняло)

Оператор может: возобновить инцидент после исправления данных, отменить зависший экземпляр, мигрировать экземпляры на новую версию диаграммы.

### 21. Версионирование процессов

Когда ты деплоишь изменённую BPMN-диаграмму, Zeebe создаёт **новую версию**. Уже запущенные экземпляры продолжают идти по старой версии. Новые экземпляры запускаются на новой.

Что делать, если 1000 экземпляров бегут по старой версии, а ты хочешь их перевести на новую? **Миграция инстансов** — через Operate или API. Zeebe позволяет мигрировать экземпляры, если новая версия совместима (не удалены шаги, на которых стоят экземпляры).

Это важно в продакшене: можно безопасно обновлять процессы без даунтайма.

### 22. Масштабирование и партиции

Zeebe масштабируется через **партиции**. Каждая партиция — независимый поток событий. Экземпляры процессов распределяются по партициям (round-robin по ключу).

Типичная конфигурация: 3 брокера, 3 партиции, replication factor 3. Каждая партиция реплицирована на все 3 брокера. Если один брокер упал — оставшиеся два продолжают работу. Данные не теряются.

Для маленьких проектов (десятки-сотни процессов в день) хватает одного брокера с одной партицией. Для enterprise-нагрузок (миллионы) — десятки брокеров.

## Часть VIII. Тестирование и разработка

### 23. Тестирование BPMN-процессов

Camunda предоставляет **zeebe-process-test** — библиотеку для unit-тестирования процессов. Поднимает in-memory движок Zeebe, деплоит диаграмму, позволяет программно проходить по шагам и проверять результат.

```java
@ZeebeProcessTest
class OrderProcessTest {

    @Autowired
    private ZeebeClient client;

    @Test
    void shouldCompleteHappyPath() {
        // Запускаем процесс
        var instance = client.newCreateInstanceCommand()
            .bpmnProcessId("order-process")
            .latestVersion()
            .variables(Map.of("orderId", "123", "amount", 500.0))
            .send().join();

        // Завершаем задание "process-payment" с результатом
        completeJob("process-payment", Map.of("paymentSuccess", true));

        // Завершаем следующее задание
        completeJob("reserve-items", Map.of("reserved", true));

        // Проверяем что процесс завершился
        assertThat(instance).isCompleted();
    }

    @Test
    void shouldHandlePaymentFailure() {
        var instance = client.newCreateInstanceCommand()
            .bpmnProcessId("order-process")
            .latestVersion()
            .variables(Map.of("orderId", "456", "amount", 999.0))
            .send().join();

        // Оплата не прошла
        completeJob("process-payment", Map.of("paymentSuccess", false));

        // Проверяем что процесс пошёл к уведомлению
        assertThat(instance).hasPassedElement("notify-customer-failure");
        assertThat(instance).isCompleted();
    }
}
```

Это позволяет тестировать именно **логику процесса** (ветвления, таймеры, ошибки) отдельно от бизнес-логики worker'ов. Worker'ы тестируются обычными unit-тестами.

## Часть IX. Деплой и практика

### 24. Как деплоится в реальности

**Локальная разработка:** Docker Compose с Zeebe, Elasticsearch, Operate, Tasklist. Твой Spring Boot запускается отдельно (из IDE) и подключается к Zeebe на localhost:26500.

**Production (Self-Managed):** Kubernetes с Helm-чартом Camunda. Zeebe кластер из 3+ нод, Elasticsearch кластер, Operate, Tasklist — каждый как отдельный Deployment.

**Production (SaaS):** Ты вообще ничего не разворачиваешь — Camunda даёт тебе кластер в облаке. Подключаешь свои worker'ы через OAuth.

Минимальный `application.yml` для подключения к локальному Zeebe:

```yaml
zeebe:
  client:
    broker:
      gateway-address: localhost:26500
    security:
      plaintext: true

camunda:
  operate:
    base-url: http://localhost:8081
  tasklist:
    base-url: http://localhost:8082
```

### 25. Где применяется в реальности

В России Camunda активно используется в финтехе и enterprise: **Т-Банк** (14 000+ уникальных процессов), крупные банки (кредитные конвейеры, платёжные процессы), страховые компании (рассмотрение страховых случаев), ритейл (заказы, возвраты, логистика).

Типичные use-cases: всё, где есть длительный процесс (от секунд до дней/недель) с несколькими участниками (системы и люди), ветвлениями, таймаутами и обработкой ошибок. Если процесс укладывается в один синхронный REST-вызов — Camunda избыточна. Если процесс проходит через 5 сервисов, 2 человеческих согласования и имеет 3 ветки ошибок — Camunda экономит месяцы разработки.

| Что такое Camunda | Платформа для автоматизации бизнес-процессов через исполняемые BPMN-диаграммы |
