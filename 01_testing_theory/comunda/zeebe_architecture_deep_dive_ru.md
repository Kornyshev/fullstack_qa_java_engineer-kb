# Архитектура Zeebe изнутри: как движок исполняет BPMN-процессы

> Документ для QA-инженера, знакомого с микросервисами и Kafka. Объясняет на концептуальном уровне, как Zeebe технически стыкуется с worker'ами, как «токены» двигаются по BPMN-схеме, и чем это отличается от обычной оркестрации через Kafka.

---

## 1. Четыре компонента Zeebe-кластера

Zeebe-кластер состоит из **клиентов**, **шлюзов (Gateway)**, **брокеров** и **экспортёров**. У каждого чётко определённая роль.

### Gateway (шлюз)

Stateless-точка входа. Принимает соединения от клиентов по gRPC (порт 26500) и REST (порт 8080), переводит запросы во внутренний бинарный формат и маршрутизирует к нужному брокеру. Шлюзы узнают топологию кластера через SWIM gossip-протокол. Маршрутизация зависит от типа команды: `CreateProcessInstance` идёт **round-robin** по партициям, а команды для существующего экземпляра маршрутизируются к партиции, которая им владеет (ID партиции закодирован в ключе экземпляра). В продакшене шлюзы запускаются отдельными процессами, для разработки могут быть встроены в брокер.

**Аналогия:** Gateway — это как Nginx перед микросервисами. Он не хранит состояние, только маршрутизирует.

### Broker (брокер)

Ядро движка. Брокеры образуют peer-to-peer сеть без единой точки отказа. Каждый брокер обрабатывает команды, хранит состояние процессов в **RocksDB** (встроенная key-value БД), и назначает задания worker'ам. Один брокер может быть лидером для одних партиций и фолловером для других одновременно. **Никакая бизнес-логика не выполняется внутри брокера** — вся прикладная логика живёт во внешних клиентах и worker'ах.

**Аналогия:** Broker — это как Kafka-брокер, но со встроенным «мозгом», который понимает BPMN и может сам двигать процесс от шага к шагу.

### Exporter (экспортёр)

Стримит каждое изменение состояния из внутреннего лога брокера во внешние системы. Встроенный **Elasticsearch-экспортёр** пакетирует записи (сбрасывает каждые 1 000 записей, 10 МБ или 5 секунд) и пишет их как JSON-документы в индексы по типам (`zeebe-record-process-instance`, `zeebe-record-job` и т.д.). Именно из этих данных Operate и Tasklist строят свои интерфейсы.

**Аналогия:** Exporter — это как Kafka Connect, который читает из внутреннего лога и пишет в Elasticsearch.

### Client (клиент)

Твои микросервисы. Подключаются исключительно через Gateway. Деплоят процессы, создают экземпляры, публикуют сообщения и — самое главное — работают как **Job Worker'ы**: забирают задания, выполняют бизнес-логику, возвращают результат.

## 2. Партиции и Raft: как Zeebe распределяет данные

### Что такое партиция

**Партиция** — это персистентный, append-only поток событий. Концептуально идентична Kafka-партиции. У каждой партиции в каждый момент времени ровно один лидер, остальные — реплики (фолловеры). Количество партиций задаётся при создании кластера и не может быть изменено без переразвёртывания.

### Как работает Raft

Zeebe запускает **независимую Raft-группу на каждую партицию**. Когда лидер получает команду, он:
1. Добавляет запись в свой лог
2. Реплицирует её на фолловеры
3. Ждёт **кворума** (большинство) для подтверждения
4. Только после этого обрабатывает команду

С replication factor = 3 достаточно подтверждения от одного фолловера. С factor = 5 — от двух. Это гарантирует сохранность данных даже при полном выходе из строя одного брокера.

### К какой партиции привязан экземпляр процесса

Когда экземпляр процесса создан на партиции — **он остаётся там навсегда**. Все последующие команды, обновления переменных и создание заданий для этого экземпляра выполняются на той же партиции. Распределение новых экземпляров: прямое создание — round-robin через Gateway; создание через message correlation — хеш от correlation key определяет партицию; запуск по таймеру — всегда партиция 1.

## 3. Как «токен» двигается по BPMN-диаграмме

Это ключевой вопрос: что технически происходит, когда процесс «переходит» от одного шага к другому.

### Event Sourcing — фундамент

Zeebe использует **event sourcing**: каждое изменение состояния — это неизменяемая запись, добавленная в лог партиции. Текущее состояние — это проекция этих записей, кэшированная в RocksDB-снэпшотах (сбрасываются каждые 5 минут). При рестарте брокер восстанавливается из последнего снэпшота и доигрывает незафиксированные записи.

### Команды, события и цепная реакция

Лог содержит три типа записей: **Commands** (запросы на изменение состояния), **Events** (подтверждения изменений), **Rejections** (когда команда невалидна).

Stream-процессор читает записи последовательно и запускает цепную реакцию:

```
1. Приходит команда COMPLETE_JOB (worker завершил задание)
2. Процессор валидирует её по текущему состоянию
3. Если валидна: применяет изменения в RocksDB, 
   пишет событие JOB_COMPLETED, 
   пишет следующую команду COMPLETE_ELEMENT (для Service Task)
4. COMPLETE_ELEMENT → событие ELEMENT_COMPLETED 
   → команда SEQUENCE_FLOW_TAKEN 
   → команда ELEMENT_ACTIVATING (для следующего узла)
```

Эта цепочка **команда → событие + следующая команда → событие + следующая команда** — и есть механизм «движения токена». Никакого физического объекта «токен» нет. Вместо этого каждый BPMN-элемент проходит через состояния: `ACTIVATING → ACTIVATED → COMPLETING → COMPLETED`.

Типичный экземпляр процесса от старта до конца требует примерно **30 циклов коммита**. Обработка намеренно разбита на маленькие шаги, чтобы разные экземпляры могли чередоваться на одной партиции — высокая конкурентность из одного потока обработки.

**Аналогия для QA:** Это как если бы каждое действие в тесте (клик, ввод текста, проверка) записывалось в лог, и «проигрыватель» воспроизводил их одно за другим. Если упал посередине — перезапустил и продолжил с последнего записанного шага.

### Скоупы переменных

Переменные структурированы иерархически. **Корневой скоуп** — это сам экземпляр процесса. Каждый подпроцесс или мультиинстанс создаёт дочерний скоуп. Задачи могут читать переменные из своего скоупа и всех родительских (видимость вверх), но не из «соседних» скоупов.

**Input mappings** создают переменные в локальном скоупе (как параметры функции). **Output mappings** возвращают локальные переменные в родительский скоуп (как return-значения). Без output mappings локальные переменные теряются при выходе из скоупа.

## 4. Жизненный цикл Job Worker — как worker «встраивается» в процесс

Это то, что тебя интересует больше всего: как обычный Spring Boot сервис становится частью BPMN-процесса.

### Что происходит, когда процесс доходит до Service Task

1. Stream-процессор активирует Service Task
2. Создаётся **задание (job)** с типом, указанным в BPMN-модели (например, `"check-credit-history"`)
3. Задание переходит в состояние `CREATED` и становится доступным для активации

### Как worker забирает задание

Worker подключается к Gateway по gRPC и использует один из двух механизмов:

**Long polling** (основной): Worker отправляет запрос `ActivateJobs`, указывая тип задания, максимальное количество, timeout блокировки и список нужных переменных. Gateway веерно опрашивает все партиции. Если заданий нет — запрос висит до 30 секунд, потом возвращается пустым. Это экономит ресурсы по сравнению с постоянным polling'ом.

**Job streaming** (с Camunda 8.3+): Push-модель через persistent gRPC server-streaming. Gateway регистрирует стрим на всех брокерах. Когда задание становится доступным, брокер проталкивает его напрямую через Gateway клиенту. Снижает latency активации на ~50%.

**Ключевое:** worker НЕ «вызывается» — он САМ приходит и спрашивает «есть работа?». Это принципиально отличается от webhook-модели.

### Что приходит в worker вместе с заданием

Каждый `ActivatedJob` несёт: уникальный **ключ задания**, **тип**, **ключ экземпляра процесса**, **ID BPMN-процесса**, **версию**, **ID элемента** (из диаграммы), **custom headers** (статические пары ключ-значение, заданные в модели), **имя worker'а**, оставшиеся **ретраи**, **дедлайн** активации, и запрошенные **переменные** как JSON-строку.

### Четыре варианта ответа worker'а

**CompleteJob** — «готово, вот результат». Переменные из ответа мерджатся в скоуп процесса. Процесс идёт дальше.

**FailJob** — «ошибка, повтори». Указываются оставшиеся ретраи и backoff. Если ретраи > 0 — задание через backoff-интервал снова станет доступным. Если ретраи ≤ 0 — создаётся **incident** (инцидент), видимый в Operate, блокирующий процесс до ручного вмешательства оператора.

**ThrowError** — «бизнес-ошибка». Активирует BPMN Error Boundary Event. Если подходящего обработчика нет — incident.

**Timeout** (worker не ответил до дедлайна) — задание молча возвращается в пул доступных. Другой worker (или тот же после перезапуска) может его подобрать. **Это значит, что два worker'а могут обрабатывать одно задание одновременно** — Zeebe обеспечивает at-least-once семантику, поэтому код worker'а должен быть идемпотентным.

Важный нюанс: при backpressure (когда брокер перегружен и отклоняет запросы) команды `CompleteJob` и `FailJob` **всегда принимаются** — движок никогда не блокирует завершение задания.

## 5. Как Spring Boot автоматически подключает worker'ы

Зависимость `camunda-spring-boot-starter` автоконфигурирует `CamundaClient` из `application.yml` (адрес Gateway, режим аутентификации, TLS).

### Что делает @JobWorker под капотом

`JobWorkerAnnotationProcessor` — Spring `BeanPostProcessor` — при инициализации контекста сканирует все Spring-бины на методы с аннотацией `@JobWorker`. Для каждого аннотированного метода извлекает: тип задания (по умолчанию — **имя метода**), timeout, `maxJobsActive`, `fetchVariables` и другие настройки.

`JobWorkerManager` затем открывает worker'ы через `client.newWorker().jobType(...).handler(...).open()`, оборачивая аннотированный метод в handler с богатой инъекцией параметров:

- `ActivatedJob job` — полный объект задания
- `@Variable String orderId` — одна переменная процесса, автоматический каст
- `@VariablesAsType MyDto dto` — все переменные, десериализованные в POJO
- `@CustomHeaders Map<String, String>` — статические заголовки из BPMN
- `@ProcessInstanceKey long key` — ключ экземпляра процесса

Когда `autoComplete = true` (по умолчанию), фреймворк автоматически вызывает `CompleteJob` после возврата из метода. Если метод возвращает `Map` или POJO — эти значения становятся переменными процесса. Если метод бросает исключение — фреймворк вызывает `FailJob` с `retries - 1` и текстом ошибки.

Worker'ы открываются при завершении инициализации Spring-контекста и закрываются при graceful shutdown.

**Итого: чтобы Spring Boot сервис стал частью BPMN-процесса, нужно:**
1. Добавить зависимость `camunda-spring-boot-starter`
2. Указать адрес Gateway в `application.yml`
3. Написать метод с `@JobWorker(type = "...")` — тип должен совпадать с типом в BPMN-диаграмме
4. Всё. Стартер сам найдёт метод, подключится к Gateway, начнёт забирать задания

## 6. Деплой BPMN: от XML до исполняемого определения

### Как BPMN-файл попадает в Zeebe

Деплой использует RPC `DeployResource` и **атомарен** — все ресурсы в запросе деплоятся вместе или ни один. Gateway пересылает деплой на **партицию 1** (выделенная «партиция деплоев»). Там происходит: валидация XML на корректность, проверка что поддерживаемые Zeebe элементы, проверка атрибута `isExecutable="true"`. При успехе: назначается уникальный `processDefinitionKey`, автоинкрементируется номер версии.

**Дедупликация:** Zeebe вычисляет хеш содержимого и инкрементирует версию только если определение реально изменилось. Повторный деплой того же файла не создаёт новую версию.

После создания на партиции 1 определение **асинхронно распространяется** на все остальные партиции. Клиент получает ответ после обработки на партиции 1 — теоретически возможна ситуация, когда ты пытаешься создать экземпляр на партиции, которая ещё не получила определение.

### Версионирование

Каждый деплой с изменениями создаёт новую версию. Уже запущенные экземпляры продолжают идти по старой версии. Новые экземпляры запускаются на последней. Через Operate можно мигрировать экземпляры на новую версию, если она совместима.

## 7. Operate и Tasklist: откуда берутся данные для мониторинга

### Конвейер данных

Внутренний лог Zeebe не доступен для прямых запросов. Экспортёр читает каждую зафиксированную запись из лога и пишет в Elasticsearch. Operate читает из Elasticsearch, трансформирует и агрегирует данные в свои индексы (`operate-list-view`, `operate-flow-node`, `operate-variable`, `operate-incident`).

**Да, Operate показывает «токены» на BPMN-диаграмме.** React-фронтенд использует библиотеку bpmn-js для рендеринга BPMN XML, затем накладывает: зелёные подсветки активных элементов, пройденные пути, красные маркеры инцидентов. Можно: инспектировать переменные по скоупам, видеть дерево элементов, детали инцидентов с текстами ошибок, резолвить инциденты, отменять экземпляры, и даже **передвигать токены** на другие элементы для дебага.

### Tasklist и User Task'и

Tasklist следует тому же конвейеру данных. Современная Camunda 8 использует **engine-native user tasks**: когда процесс доходит до User Task, Zeebe управляет состоянием задачи внутренне. Tasklist запрашивает Elasticsearch для списков задач, но отправляет команды назначения, завершения и обновления **напрямую в Zeebe** через REST API — без round-trip через Elasticsearch. Завершение задачи мгновенно вступает в силу в движке.

## 8. Чем Zeebe отличается от «просто Kafka между микросервисами»

Самое важное различие: **Kafka — это транспортная труба; Zeebe — это процессный мозг.** Kafka перемещает сообщения между сервисами, не понимая их смысла. Zeebe определяет, отслеживает и обеспечивает порядок и логику всего бизнес-процесса.

### Что ты должен построить сам с голой Kafka

При чистой Kafka-хореографии каждый сервис слушает события и сам решает, что делать дальше. «Процесс» существует только неявно, размазанный по подпискам consumer'ов. Чтобы получить то, что Zeebe даёт из коробки, тебе нужно построить:

- **Кастомный стейт-машину** или сервис-оркестратор для отслеживания, где каждый экземпляр процесса находится
- **Персистентное хранилище** (БД) для длительных процессов (часы, дни)
- **Инфраструктуру таймеров** (сервисы-шедулеры, delayed-message топики) для «подожди 24 часа, потом эскалируй»
- **Retry-логику с dead-letter queue** для каждого consumer'а, с backoff-стратегиями
- **Цепочки компенсаций** для saga-откатов, с отслеживанием завершённых шагов
- **Логику корреляции сообщений** для сопоставления ответов с нужным экземпляром процесса
- **Операционный дашборд** для ответа на вопрос «где сейчас заявка №12345?»

Команды, строящие saga-оркестраторы на Kafka, часто обнаруживают, что **переизобретают workflow engine** — с меньшей надёжностью и без визуальных инструментов.

### Пять конкретных преимуществ Zeebe

**Visibility на уровне процесса** — главный практический выигрыш. Operate позволяет видеть каждый работающий экземпляр на BPMN-диаграмме, инспектировать переменные, резолвить инциденты — невозможно с Kafka без кастомного инструментария.

**Нативные таймеры** (`PT24H`, `R/PT1H`) — это BPMN-элементы, которые переживают рестарты брокеров, заменяя хрупкие сервисы-шедулеры.

**Декларативная обработка ошибок** — ретраи, error boundary events, компенсации — живут в BPMN-модели, не размазаны по коду сервисов.

**Корреляция сообщений** сопоставляет входящие события с правильным ожидающим экземпляром по correlation key с буферизацией и TTL.

**Версионирование процессов** позволяет менять flow обновлением BPMN-модели, а не координированным деплоем нескольких сервисов.

### Прагматичный ответ: используй оба

Zeebe и Kafka **комплементарны**. Рекомендуемый паттерн от сооснователя Camunda Бернда Рюкера: используй Kafka как шину событий **между** bounded contexts (слабая связанность), а Zeebe **внутри** конкретного сервиса, который владеет сложным процессным flow. Kafka Connect Zeebe Connector связывает их: публикуй в Kafka когда workflow достигает определённого шага, consume из Kafka для корреляции сообщений в ожидающие экземпляры.

**Выбирай только Kafka**, когда: простые event-driven реакции, data-pipelines, или 2–3 шаговые flow с базовой обработкой ошибок.

**Выбирай Zeebe**, когда: 5+ шагов с ветвлениями, длительные процессы (дни/недели), человеческие задачи вперемешку с автоматикой, требования аудита по регулятору, или когда ловишь себя на том, что строишь «сервис стейт-машин» поверх Kafka.

## 9. Итого: ментальная модель для тестировщика

| Вопрос | Ответ |
|--------|-------|
| Что такое Zeebe-кластер? | Gateway (маршрутизатор) + Broker'ы (движок + хранилище) + Exporter (стриминг данных в Elasticsearch) |
| Что такое worker? | Обычный Spring Boot сервис с методом `@JobWorker`. Подключается к Gateway по gRPC, сам забирает задания |
| Что worker должен уметь? | Принять входные переменные (JSON) → выполнить логику → вернуть результат (JSON). Быть идемпотентным |
| Где хранится BPMN-схема? | Внутри Zeebe (RocksDB). Деплоится через API или при старте Spring Boot приложения |
| Как «токен» двигается? | Цепная реакция команд и событий в append-only логе. Нет физического токена — есть состояние элементов |
| Где видно токены на схеме? | В Operate — веб-UI, рендерит BPMN через bpmn-js с подсветкой активных элементов |
| Чем отличается от Kafka? | Kafka — труба для сообщений. Zeebe — мозг, который знает весь процесс, двигает его, обрабатывает ошибки, ждёт людей и таймеры |
| Как Zeebe и Kafka работают вместе? | Kafka — между сервисами (шина). Zeebe — внутри одного сервиса для сложного flow. Connector связывает их |
