# Camunda/Zeebe: подготовка к собеседованию — кредитный конвейер

> **Контекст:** Собеседование на позицию QA в проект кредитного конвейера на платформе Camunda. Продукт «Кредит под залог недвижимости» (КНПЗ), также кредиты наличными, кредитные карты. Высоконагруженная система, миллионы заявок, множество интеграций.

---

## Часть I. Вопросы для интервью с ответами

### Блок 1. Общее понимание Camunda и BPMN

**Q1: Что такое Camunda и зачем она нужна в кредитном конвейере?**

Camunda — это платформа для автоматизации бизнес-процессов через исполняемые BPMN-диаграммы. В контексте кредитного конвейера это означает, что весь путь заявки — от подачи клиентом до выдачи кредита — описан как визуальная диаграмма, которую Zeebe (движок Camunda) исполняет шаг за шагом.

Почему не просто код? Потому что кредитный процесс: длительный (от минут до дней), включает десятки шагов (скоринг, проверка документов, андеррайтинг, одобрение, формирование договора), часть шагов автоматические (вызов API), часть — человеческие (решение андеррайтера), логика ветвится (одобрено/отказано/доработка), и эта логика постоянно меняется (регулятор, бизнес-правила). Camunda позволяет менять процесс на уровне диаграммы, а не переписывать код.

**Q2: Чем отличается Camunda 7 от Camunda 8?**

Camunda 7 — встраиваемый движок (embedded), работает внутри Java-приложения, использует базу данных (PostgreSQL/Oracle) для хранения состояния. Хорошо подходит для монолитов.

Camunda 8 (Zeebe) — отдельный распределённый движок, работает как внешний сервис. Использует паттерн External Task — бизнес-логика живёт в отдельных worker'ах. Горизонтально масштабируется. Общение через gRPC/REST.

Кредитный конвейер, скорее всего, использует Camunda 7 (запущен 3 года назад, банковская среда). Но может быть и миграция на 8 — стоит уточнить на собеседовании.

**Q3: Что такое BPMN и какие основные элементы ты знаешь?**

BPMN — стандарт нотации для описания бизнес-процессов. Основные элементы: события (start, end, intermediate — круги), задачи (service task, user task — прямоугольники), шлюзы (exclusive, parallel, inclusive — ромбы), потоки (стрелки между элементами). В контексте кредитного конвейера: Start Event — клиент подал заявку, Service Task — вызов скоринга, User Task — решение андеррайтера, Exclusive Gateway — одобрено или нет, End Event — выдача кредита или отказ.

**Q4: Что такое Service Task и как он связан с кодом?**

Service Task — автоматическая задача на диаграмме. У неё указан тип (например, `scoring-check`). В коде есть Job Worker с аннотацией `@JobWorker(type = "scoring-check")` — Spring-компонент, который подписывается на этот тип задачи. Когда процесс доходит до Service Task, Zeebe создаёт задание, worker его забирает, выполняет логику (вызов API скоринга), возвращает результат. Zeebe идёт дальше.

**Q5: Что такое User Task и как он работает?**

User Task — задача, ожидающая действия человека. Процесс останавливается и ждёт. Задача появляется в Tasklist (веб-интерфейс), конкретный сотрудник (или группа) видит её, выполняет действие (одобряет заявку, запрашивает доп. документы), результат возвращается в процесс. В кредитном конвейере типичные User Task'и: решение андеррайтера, верификация документов, ручная проверка при подозрении на фрод.

**Q6: Какие типы шлюзов (gateways) ты знаешь?**

Exclusive (XOR) — только одна ветка по условию. Пример: «кредитный рейтинг > 700? Да → автоодобрение, Нет → ручная проверка». Parallel (AND) — все ветки одновременно. Пример: параллельно запускаем проверку бюро кредитных историй, проверку залога, проверку работодателя. Inclusive (OR) — одна или несколько веток по условиям. Event-Based — ждём первое из событий (например, ответ от клиента ИЛИ истечение таймера 5 дней).

### Блок 2. Тестирование процессов на Camunda

**Q7: Как тестировать BPMN-процесс?**

Несколько уровней. Unit-тесты процесса: используем zeebe-process-test, поднимаем in-memory Zeebe, деплоим BPMN, программно проходим по шагам, проверяем что процесс пошёл по правильной ветке. Unit-тесты worker'ов: обычные JUnit-тесты, мокаем внешние сервисы (WireMock), проверяем что worker корректно обрабатывает данные. Интеграционные тесты: поднимаем Zeebe в Docker (Testcontainers), деплоим реальную диаграмму, запускаем реальные worker'ы, проверяем полный flow. E2E тесты: запуск всего стека, проверка что заявка проходит от подачи до решения.

**Q8: Какие типичные баги бывают в BPMN-процессах?**

Неправильные условия на шлюзах (забыли ветку default → процесс зависает). Неправильный маппинг переменных (worker ожидает `creditScore`, а в процессе переменная называется `credit_score`). Отсутствие обработки ошибок (Service Task падает, ретраи исчерпаны → incident, никто не заметил). Таймеры с неправильным значением (PT24H вместо P1D — разница при переходе на летнее время). Проблемы с параллельными ветками (race conditions при записи в одну переменную из разных веток). Отсутствие boundary error events — бизнес-ошибки не обработаны.

**Q9: Как ты будешь тестировать кредитный конвейер с множеством интеграций?**

Стратегия по уровням. На уровне unit: мокаю все интеграции, тестирую логику каждого worker'а изолированно. На уровне компонентном: мокаю внешние системы (WireMock), но поднимаю реальный Zeebe и реальных worker'ов — проверяю что процесс проходит корректно. На уровне интеграционном: проверяю конкретные интеграции (скоринг, бюро КИ, ГосУслуги) в тестовой среде. На уровне E2E: прогоняю основные сценарии (happy path, отказ, возврат на доработку) на полном стеке.

Отдельно: контрактное тестирование для интеграций (формат запросов/ответов) и тестирование идемпотентности worker'ов (задание может быть обработано дважды).

**Q10: Что такое incident в Camunda и как его диагностировать?**

Incident — это задание, исчерпавшее все ретраи. Процесс останавливается на этом шаге и ждёт ручного вмешательства. Видно в Operate (веб-UI мониторинга). Диагностика: смотришь incident в Operate → видишь текст ошибки → находишь логи worker'а → определяешь причину (недоступный API, невалидные данные, баг в коде). Решение: исправляешь причину, в Operate нажимаешь «Retry» — процесс продолжается с того же шага.

Для QA это важно: нужно тестировать сценарии, приводящие к incidents, проверять что мониторинг их ловит, и что retry после исправления работает корректно.

### Блок 3. Архитектура и интеграции

**Q11: Как устроен кредитный конвейер на Camunda архитектурно?**

Предполагаемая архитектура кредитного конвейера: Zeebe (или Camunda 7 engine) в центре, вокруг — микросервисы-worker'ы. Каждый worker отвечает за свою область: scoring-worker (вызов бюро КИ, расчёт рейтинга), verification-worker (проверка документов), underwriting-worker (подготовка данных для андеррайтера), document-worker (генерация договоров), notification-worker (уведомления клиенту). Есть и человеческие задачи: андеррайтер работает через Tasklist или кастомный UI.

Интеграции: бюро кредитных историй (НБКИ, Equifax), ГосУслуги/СМЭВ, Росреестр (для залога недвижимости), скоринговые модели (ML), системы документооборота, SMS/email-провайдеры.

**Q12: Как Camunda обеспечивает отказоустойчивость?**

На уровне Zeebe: кластер брокеров с репликацией (Raft consensus), если один брокер упал — остальные продолжают. На уровне worker'ов: паттерн External Task — если worker упал, задание возвращается в очередь и через timeout подбирается другим экземпляром. На уровне процесса: ретраи с backoff, incidents при исчерпании ретраев, мониторинг через Operate.

Для кредитного конвейера это критично: нельзя потерять заявку. Camunda гарантирует: если задание принято — оно будет выполнено (at-least-once semantics). Worker должен быть идемпотентным.

**Q13: Что такое идемпотентность worker'а и почему это важно?**

Идемпотентность — способность обработать одно и то же задание дважды без побочных эффектов. Zeebe гарантирует at-least-once delivery: если worker обработал задание, но не успел сообщить Zeebe (сеть упала) — задание будет отдано повторно. Если worker не идемпотентен — может списать деньги дважды, отправить два SMS.

Как обеспечить: проверять уникальный ID задания перед выполнением, использовать транзакционные гарантии БД (INSERT ... ON CONFLICT DO NOTHING), хранить idempotency key.

### Блок 4. Вопросы специфичные для QA

**Q14: Как автоматизировать тестирование BPMN-процессов?**

Три подхода. Первый — zeebe-process-test (unit): JUnit + in-memory Zeebe, деплой BPMN, программная прогонка. Быстро, изолированно, покрывает логику ветвлений. Второй — Testcontainers (integration): реальный Zeebe в Docker, реальные worker'ы, мок внешних сервисов. Ближе к продакшену. Третий — REST API тестирование (e2e): запуск процесса через API, polling состояния через Operate REST API, проверка что процесс завершился корректно. REST Assured хорошо подходит.

**Q15: Какие метрики важны для мониторинга кредитного конвейера?**

Бизнес-метрики: среднее время обработки заявки (от подачи до решения), процент одобрений, количество заявок в обработке, время на каждом шаге, количество возвратов на доработку. Технические метрики: количество incidents, latency worker'ов, Kafka consumer lag (если используется), доступность интеграций, количество ретраев. В Grafana можно строить дашборды по данным из Prometheus (Zeebe экспортирует метрики) и Elasticsearch (Operate/Optimize).

**Q16: Как тестировать миграцию процессов на новую версию?**

Когда бизнес-логика меняется (новое правило регулятора), деплоится новая версия BPMN. Тестирование: убедиться что новые заявки идут по новой версии, старые — продолжают по старой, миграция инстансов (если нужна) не ломает данные. Нужно: тест на совместимость версий, тест на миграцию (Operate позволяет мигрировать), тест на откат (что будет если новая версия плохая).

**Q17: Расскажи про тестирование Saga-паттерна в кредитном процессе.**

Saga — распределённая транзакция. В кредитном конвейере: зарезервировали лимит → заблокировали залог → сформировали договор. Если на шаге «формирование договора» ошибка — нужно разблокировать залог и освободить лимит (компенсации). Тестирование: намеренно ломаем каждый шаг и проверяем что компенсации выполнились, проверяем порядок компенсаций (обратный), проверяем что после компенсации данные консистентны.

### Блок 5. Продвинутые вопросы

**Q18: Как обеспечить тестовые данные для кредитного конвейера?**

Сложность: нужны реалистичные данные (ФИО, паспорт, ИНН, адрес, данные о недвижимости). Подходы: генерация через Faker/DataFactory с учётом российской специфики (формат паспорта, валидный ИНН), предзаполненные тестовые наборы в БД, мокирование внешних систем (WireMock возвращает контролируемые ответы для конкретных тестовых клиентов).

**Q19: Как тестировать таймеры в процессах?**

Не ждать реальные 48 часов. В unit-тестах: zeebe-process-test позволяет программно «передвинуть» время вперёд. В интеграционных: использовать короткие таймеры в тестовом окружении (PT10S вместо P2D). Проверять: что таймер срабатывает, что процесс идёт по ветке таймера, что эскалация работает.

**Q20: Что такое DMN и зачем он в кредитном конвейере?**

DMN — таблицы бизнес-решений. В кредитном конвейере: определение тарифа (сумма + срок + категория клиента → процентная ставка), скоринг-правила (доход + стаж + КИ → одобрение/отказ/ручная проверка), лимиты и ограничения. Преимущество: бизнес-аналитик может менять правила без разработчика. Тестирование DMN: проверка каждой строки таблицы, пограничные значения, комбинации условий.

## Часть II. Применение Camunda в кредитном конвейере — фантазия по аналогии

### Как может выглядеть BPMN-процесс «Кредит под залог недвижимости»

```
[Заявка от клиента]
    │
    ▼
[Предварительная проверка данных]  ← Service Task: валидация полей, дедупликация
    │
    ▼
[═══ Параллельный шлюз (AND) ═══]
    │              │               │
    ▼              ▼               ▼
[Запрос в      [Проверка        [Запрос в
 НБКИ/БКИ]      залога           ГосУслуги]
    │          (Росреестр)]        │
    │              │               │
    ▼              ▼               ▼
[═══ Ожидание всех результатов (AND join) ═══]
    │
    ▼
[Автоматический скоринг] ← Service Task + DMN: расчёт рейтинга и предварительного решения
    │
    ▼
<Результат скоринга?> ← Exclusive Gateway
    │          │            │
    ▼          ▼            ▼
[Авто-       [Ручной      [Авто-
одобрение]   андеррайтинг] отказ]
    │          │            │
    │          ▼            ▼
    │    [User Task:      [Уведомление
    │     Андеррайтер      об отказе] → [Конец]
    │     проверяет]
    │          │
    │          ▼
    │    <Решение?>
    │     │      │       │
    │     ▼      ▼       ▼
    │  [Одобр.] [Отказ] [Доработка]
    │     │      │       │
    │     │      ▼       ▼
    │     │   [Уведом.  [User Task:
    │     │    отказ]    клиент досылает
    │     │     │        документы]
    │     │     ▼            │
    │     │   [Конец]    [Таймер: 14 дней]
    │     │                  │         │
    │     │              [Получены]  [Не получены]
    │     │                  │         │
    │     │              [↑ Назад    [Отмена
    │     │               к проверке] заявки]
    │     │
    ▼     ▼
[═══ Слияние ═══]
    │
    ▼
[Формирование кредитного договора] ← Service Task
    │
    ▼
[User Task: Клиент подписывает договор (через банк/УКЭП)]
    │
    ▼
[Таймер: 30 дней на подписание]
    │                    │
[Подписано]          [Не подписано → Отмена]
    │
    ▼
[Регистрация залога в Росреестре] ← Service Task (длительная, может идти дни)
    │
    ▼
[Выдача кредита] ← Service Task: перевод денег
    │
    ▼
[Уведомление клиента] ← Service Task
    │
    ▼
[Конец]
```

### Какие Service Task'и (worker'ы/микросервисы) можно выделить

| Worker | Тип задачи | Что делает | Интеграция |
|--------|-----------|------------|------------|
| `validation-worker` | `validate-application` | Проверка полноты и корректности данных заявки | Внутренние справочники |
| `credit-bureau-worker` | `check-credit-history` | Запрос кредитной истории | НБКИ, Equifax, ОКБ |
| `collateral-worker` | `check-collateral` | Проверка залога, оценка недвижимости | Росреестр, оценочные компании |
| `gosuslugi-worker` | `verify-identity` | Верификация личности | ГосУслуги / СМЭВ |
| `scoring-worker` | `calculate-score` | Расчёт кредитного рейтинга | ML-модели, DMN-таблицы |
| `document-worker` | `generate-contract` | Формирование кредитного договора | Шаблонизатор документов |
| `registration-worker` | `register-collateral` | Регистрация залога | Росреестр |
| `disbursement-worker` | `disburse-loan` | Выдача кредита (перевод денег) | АБС банка |
| `notification-worker` | `send-notification` | SMS/email/push уведомления | SMS-провайдер, email |

### Какие User Task'и нужны

| Задача | Исполнитель | Что делает |
|--------|-------------|------------|
| Ручной андеррайтинг | Андеррайтер | Анализ заявки, принятие решения (одобрить/отказать/доработка) |
| Верификация документов | Верификатор | Проверка подлинности документов (паспорт, справка о доходах) |
| Подписание договора | Клиент | Ознакомление и подписание (через ДБО или в отделении) |
| Эскалация | Старший андеррайтер | Рассмотрение сложных случаев (сработал таймер эскалации) |

### Где используются таймеры

| Таймер | Тип | Назначение |
|--------|-----|------------|
| 48 часов на андеррайтинг | Boundary (interrupting) | Если андеррайтер не ответил → эскалация руководителю |
| 14 дней на досылку документов | Boundary (interrupting) | Если клиент не прислал → отмена заявки |
| 30 дней на подписание договора | Boundary (interrupting) | Если не подписан → отмена |
| Ежедневная проверка статуса Росреестра | Timer intermediate (cycle) | Polling статуса регистрации залога |
| 24 часа — напоминание | Boundary (non-interrupting) | Напоминание андеррайтеру, задача продолжает ждать |

### Где применяется DMN

| Таблица решений | Входы | Выход |
|-----------------|-------|-------|
| Определение тарифа | Сумма, срок, LTV, категория клиента | Процентная ставка |
| Предварительный скоринг | Кредитный рейтинг, DTI, стаж | Авто-одобрение / ручная проверка / авто-отказ |
| Требуемые документы | Тип продукта, сумма, категория | Список обязательных документов |
| Лимит полномочий | Сумма кредита, LTV | Уровень андеррайтера (junior / senior / комитет) |

### Где применяется Saga (компенсации)

Если на этапе «Регистрация залога» произошла ошибка после того, как договор уже подписан:
1. Отменить регистрацию залога (если начата)
2. Аннулировать кредитный договор
3. Освободить зарезервированный лимит
4. Уведомить клиента об отмене

Это описывается на BPMN через Compensation Events — к каждому шагу прикрепляется компенсирующий handler.

### Что конкретно тестировать в таком процессе

**Happy path:** заявка → все проверки ОК → автоодобрение → договор → подписание → регистрация → выдача. Проверить что прошло менее чем за X минут.

**Отказ на скоринге:** заявка с низким рейтингом → авто-отказ → уведомление клиенту → процесс завершён. Проверить что не дошло до андеррайтера.

**Ручной андеррайтинг:** средний рейтинг → андеррайтер → одобрение → далее по процессу.

**Возврат на доработку:** андеррайтер запросил доп. документы → клиент получил уведомление → досылает → процесс возвращается к проверке.

**Таймаут клиента:** андеррайтер запросил документы → клиент не прислал за 14 дней → заявка отменена.

**Эскалация:** андеррайтер не ответил за 48 часов → задача ушла к руководителю.

**Параллельные проверки:** все три проверки (БКИ, залог, ГосУслуги) запускаются одновременно → ждём все три → переходим к скорингу. Проверить что одна медленная проверка не блокирует другие.

**Ошибка интеграции:** НБКИ недоступен → ретраи → incident → оператор видит в Operate → после восстановления retry → процесс продолжается.

**Saga/компенсации:** ошибка на позднем шаге → проверить что все предыдущие шаги откатились корректно.

**Версионирование:** задеплоили новую версию процесса → старые заявки идут по старой → новые по новой → мигрировали часть → проверить консистентность.
