# Баг-репорты. Severity vs Priority

## Обзор

Баг-репорт (Bug Report) — это формализованный документ, описывающий обнаруженный дефект в программном обеспечении.
Качественно составленный баг-репорт — один из ключевых артефактов работы QA-инженера. От того,
насколько точно и полно описан дефект, зависит скорость его исправления, приоритизация в backlog
и общая эффективность взаимодействия между тестировщиками и разработчиками.

Умение писать чёткие, воспроизводимые и информативные баг-репорты — **базовый навык**,
который проверяется практически на каждом собеседовании на позицию QA.

## Структура баг-репорта

### Обязательные поля

| Поле | Описание | Пример |
|------|----------|--------|
| **ID** | Уникальный идентификатор (генерируется системой) | BUG-1042 |
| **Summary (Title)** | Краткое, но информативное описание проблемы | «Кнопка "Оплатить" не реагирует на клик после добавления промокода» |
| **Steps to Reproduce** | Пошаговая инструкция для воспроизведения | 1. Открыть корзину 2. Ввести промокод SALE20 3. Нажать "Оплатить" |
| **Expected Result** | Что должно происходить по спецификации | Открывается страница оплаты |
| **Actual Result** | Что происходит на самом деле | Ничего не происходит, кнопка неактивна |
| **Severity** | Степень влияния на систему | Critical / Major / Minor / Trivial |
| **Priority** | Срочность исправления | High / Medium / Low |
| **Environment** | Окружение, где воспроизведён баг | Chrome 120, Windows 11, Production |
| **Attachments** | Скриншоты, видео, логи | screenshot_bug_1042.png |

### Дополнительные поля

| Поле | Описание |
|------|----------|
| **Preconditions** | Условия, которые должны быть выполнены перед воспроизведением |
| **Postconditions** | Состояние системы после воспроизведения бага |
| **Assignee** | Ответственный за исправление |
| **Reporter** | Кто нашёл и зарепортил баг |
| **Version / Build** | Версия продукта, где обнаружен дефект |
| **Component / Module** | Модуль системы, к которому относится баг |
| **Labels / Tags** | Метки для фильтрации и группировки |
| **Linked Issues** | Связанные задачи, user stories, другие баги |
| **Workaround** | Обходной путь, если он существует |

## Summary: правила написания

Summary — это первое, что видят все участники команды. Хороший заголовок должен отвечать на вопрос
**«Что сломалось? Где? При каких условиях?»**

### Формула хорошего Summary

```
[Где] + [Что происходит] + [При каких условиях (опционально)]
```

### Примеры хороших и плохих Summary

| Плохо | Хорошо |
|-------|--------|
| Баг на странице | Страница профиля: аватар не загружается при размере > 5MB |
| Не работает кнопка | Корзина: кнопка «Удалить» не удаляет последний товар |
| Ошибка 500 | API /users/login возвращает 500 при пустом поле email |
| Проблема с регистрацией | Регистрация: повторная отправка формы создаёт дубликат аккаунта |
| Всё упало | Главная страница: белый экран после деплоя версии 2.4.1 |

## Steps to Reproduce: как писать

Шаги воспроизведения — это **рецепт**, следуя которому любой человек сможет увидеть баг.

### Правила

1. **Нумеруйте шаги** — каждое действие на отдельной строке
2. **Начинайте с Preconditions** — что должно быть готово заранее
3. **Один шаг = одно действие** — не объединяйте несколько действий
4. **Используйте точные данные** — конкретные значения полей, URL, учётные данные
5. **Указывайте навигацию** — как добраться до нужного места

### Пример хороших шагов

```
Preconditions:
- Пользователь авторизован с ролью Admin
- В каталоге есть минимум 1 товар

Steps to Reproduce:
1. Перейти на страницу каталога (/catalog)
2. Нажать кнопку "Редактировать" у первого товара
3. Очистить поле "Цена"
4. Ввести отрицательное значение: -100
5. Нажать кнопку "Сохранить"

Expected Result:
Отображается валидационная ошибка: "Цена должна быть положительным числом"

Actual Result:
Товар сохраняется с ценой -100. На витрине отображается "-100 ₽"
```

## Severity (Серьёзность)

Severity определяет **степень влияния дефекта на систему** с технической точки зрения.
Устанавливается тестировщиком.

| Severity | Описание | Примеры |
|----------|----------|---------|
| **Blocker (S1)** | Полная блокировка работы, нет workaround | Приложение не запускается; падение сервера; потеря данных |
| **Critical (S2)** | Ключевая функциональность не работает, workaround сложен или невозможен | Невозможно оплатить заказ; авторизация не работает; crash при основном сценарии |
| **Major (S3)** | Важная функциональность работает некорректно, есть workaround | Фильтры поиска не применяются; экспорт отчёта формирует пустой файл |
| **Minor (S4)** | Незначительное отклонение, не влияет на основной функционал | Неправильный формат даты; некорректная сортировка второстепенного списка |
| **Trivial (S5)** | Косметический дефект | Опечатка в тексте; неровные отступы; разный размер шрифта |

---

## Priority (Приоритет)

Priority определяет **срочность исправления** с точки зрения бизнеса.
Устанавливается менеджером / Product Owner / тимлидом.

| Priority | Описание | Срок исправления |
|----------|----------|-----------------|
| **P1 — Critical** | Исправить немедленно | Hotfix в тот же день |
| **P2 — High** | Исправить в текущем спринте | В течение спринта |
| **P3 — Medium** | Запланировать на ближайшие спринты | 1–3 спринта |
| **P4 — Low** | Исправить, когда будет время | Backlog, без конкретных сроков |

---

## Severity vs Priority: матрица

Severity и Priority — **независимые характеристики**. Баг может быть критическим по серьёзности,
но низким по приоритету, и наоборот.

| | High Priority | Medium Priority | Low Priority |
|---|---|---|---|
| **High Severity** | Crash приложения на странице оплаты (основной поток) | Crash при загрузке файла > 2GB (редкий сценарий) | Crash в админке, которой пользуется 1 человек раз в месяц |
| **Medium Severity** | Неправильный расчёт скидки на главной акции | Неверная сортировка товаров во второстепенной категории | Некорректное отображение графика в редко используемом отчёте |
| **Low Severity** | Опечатка в имени CEO на главной странице | Неправильный цвет иконки в шапке сайта | Опечатка на странице "О компании" в разделе истории |

### Классические примеры для интервью

**High Severity + Low Priority:**
- Приложение падает при вводе 10 000 символов в поле «Имя» — критичный баг, но реальные пользователи так не делают
- Потеря данных при одновременном редактировании одного документа 50 пользователями — серьёзно, но крайне редкий сценарий
- Опечатка в названии компании на странице «О нас» — технически trivial severity, но допустим это не мешает функционалу

**Low Severity + High Priority:**
- Имя CEO написано с ошибкой на главной странице — косметика, но бизнес требует срочного исправления
- Логотип конкурента случайно отображается вместо своего — визуальный баг, но репутационные риски огромны
- Неправильная дата в юридическом соглашении — мелочь визуально, но юридические последствия

## Жизненный цикл бага (Bug Lifecycle)

```
    ┌──────────┐
    │   New    │  Тестировщик создал баг-репорт
    └────┬─────┘
         │
         ▼
    ┌──────────┐
    │   Open   │  Тимлид / менеджер подтвердил и назначил
    └────┬─────┘
         │
         ▼
    ┌──────────────┐
    │  In Progress │  Разработчик взял в работу
    └────┬─────────┘
         │
         ▼
    ┌──────────┐
    │  Fixed   │  Разработчик исправил и передал на проверку
    └────┬─────┘
         │
         ▼
    ┌──────────────┐         ┌────────────┐
    │   Verified   │────────►│   Closed   │  Тестировщик подтвердил исправление
    └────┬─────────┘         └────────────┘
         │
         │  (Баг не исправлен)
         ▼
    ┌──────────┐
    │ Reopened │  Тестировщик вернул баг в работу
    └────┬─────┘
         │
         ▼
    (Возврат к In Progress)
```

### Дополнительные статусы

| Статус | Описание |
|--------|----------|
| **Deferred** | Исправление отложено на будущие версии |
| **Rejected** | Отклонён — «это не баг, а фича» или невоспроизводим |
| **Duplicate** | Дублирует уже существующий баг-репорт |
| **Won't Fix** | Принято решение не исправлять |
| **Need More Info** | Требуется дополнительная информация от репортера |

## Best Practices написания баг-репортов

### 1. Один баг = один репорт

Не смешивайте несколько дефектов в одном репорте. Каждый баг должен быть отдельным тикетом,
чтобы его можно было независимо отслеживать, приоритизировать и закрывать.

### 2. Факты, а не интерпретации

```
Плохо:  "Система работает медленно"
Хорошо: "Страница /catalog загружается 12 секунд при 100 товарах (ожидание: < 3 секунд)"
```

### 3. Воспроизводимость

Перед тем как завести баг, убедитесь, что вы можете его воспроизвести. Укажите частоту:
- **Always** — воспроизводится каждый раз
- **Often** — воспроизводится в ~70% случаев
- **Sometimes** — воспроизводится в ~30% случаев
- **Rarely** — воспроизводится редко, но было зафиксировано
- **Once** — воспроизвелось один раз (приложите доказательства)

### 4. Прикладывайте доказательства

- **Скриншоты** с выделением проблемного места
- **Видео** для сложных сценариев воспроизведения
- **Логи** (browser console, server logs) — часто критичны для разработчика
- **HAR-файлы** для сетевых проблем
- **Дампы** базы данных при проблемах с данными

### 5. Указывайте окружение

```
Browser: Chrome 120.0.6099.129
OS: Windows 11 Pro (22H2)
Device: Desktop, 1920x1080
Backend: v2.4.1 (staging)
Database: PostgreSQL 15.4
```

### 6. Изолируйте баг

Проверьте, воспроизводится ли баг:
- В другом браузере
- В режиме инкогнито
- С другим пользователем
- В другом окружении (staging vs production)

Это поможет сузить область поиска для разработчика.

## Примеры: хороший vs плохой баг-репорт

### Плохой баг-репорт

```
Summary: Не работает оплата
Severity: Critical
Steps: Попытался оплатить — не получилось
Expected: Должно работать
Actual: Не работает
```

**Проблемы:** неинформативный заголовок, нет конкретных шагов, нет данных об окружении,
нет скриншотов, расплывчатые Expected / Actual.

### Хороший баг-репорт

```
Summary: Checkout: ошибка 500 при оплате картой Visa после применения промокода SUMMER2024
Severity: Critical (S2)
Priority: P1 — High
Environment: Chrome 120, macOS Sonoma, Production (v3.2.1)
Preconditions:
  - Пользователь авторизован (test_user@example.com)
  - В корзине минимум 1 товар
  - Промокод SUMMER2024 активен

Steps to Reproduce:
  1. Перейти в корзину (/cart)
  2. Ввести промокод SUMMER2024 в поле "Промокод"
  3. Нажать "Применить" — скидка 20% отображается корректно
  4. Нажать "Перейти к оплате"
  5. Выбрать способ оплаты "Банковская карта"
  6. Ввести данные тестовой карты Visa: 4111 1111 1111 1111, 12/25, CVV 123
  7. Нажать "Оплатить"

Expected Result:
  Заказ создаётся, отображается страница "Спасибо за заказ" с номером заказа.

Actual Result:
  Отображается белая страница с текстом "500 Internal Server Error".
  В Network tab виден ответ POST /api/orders → 500.
  Response body: {"error": "NullPointerException at DiscountService.java:142"}

Reproducibility: Always (5/5 попыток)
Workaround: Если не применять промокод, оплата проходит успешно.
Attachments: screenshot_500.png, network_log.har, console_errors.txt
```

## Связь с тестированием

- Баг-репорт — **основной выходной артефакт** тестирования
- Качество баг-репортов напрямую влияет на **скорость исправления** дефектов
- Severity и Priority используются для **приоритизации** тестирования: сначала проверяем критичные области
- Статистика по багам (количество, severity, reopened rate) — важная **метрика качества** продукта и процесса
- Закрытые баги формируют **базу знаний** — на их основе создают regression test cases
- Анализ частых багов помогает выявить **проблемные модули** и направить усилия на их тестирование

## Типичные ошибки

1. **Неинформативный Summary** — «Баг», «Ошибка», «Не работает» — заставляют разработчика тратить время на понимание сути
2. **Отсутствие шагов воспроизведения** — «Зайдите на сайт, там всё видно» — неприемлемо
3. **Путаница Severity и Priority** — тестировщик ставит Priority, хотя это зона ответственности менеджера
4. **Несколько багов в одном репорте** — «Кнопка не работает + цвет неправильный + текст кривой»
5. **Субъективные формулировки** — «Выглядит некрасиво», «Работает медленно» без конкретных метрик
6. **Отсутствие окружения** — баг может быть специфичен для конкретного браузера или ОС
7. **Нет доказательств** — слова без скриншотов и логов вызывают сомнения
8. **Дубликаты** — не проверяют, существует ли уже аналогичный баг в системе
9. **Эмоциональные комментарии** — «Кто это вообще писал?!» — непрофессионально и бесполезно
10. **Завышение Severity** — все баги помечены как Critical, что обесценивает реальные критичные дефекты

## Вопросы на интервью

### 🟢 Базовый уровень
1. Что такое баг-репорт и из каких полей он состоит?
2. Перечислите обязательные поля баг-репорта.
3. Чем Severity отличается от Priority?
4. Какие уровни Severity вы знаете? Приведите примеры.
5. Кто устанавливает Severity, а кто Priority?
6. Напишите Summary для бага: «пользователь не может войти в систему при использовании пароля с кириллицей».

### 🟡 Средний уровень
7. Приведите пример бага с High Severity и Low Priority. Объясните, почему так.
8. Приведите пример бага с Low Severity и High Priority. Объясните, почему так.
9. Опишите жизненный цикл бага. Какие статусы проходит баг?
10. Что делать, если баг не воспроизводится? Всё равно заводить?
11. В чём разница между статусами Rejected, Won't Fix и Deferred?
12. Как вы определите, что баг — дубликат? Что делать, если похож, но не идентичен?
13. Вы нашли баг в production. Какой процесс вы бы выстроили?

### 🔴 Продвинутый уровень
14. Как метрики по баг-репортам (reopened rate, время жизни бага) могут улучшить процесс разработки?
15. Вы завели баг, разработчик говорит «это не баг, а фича». Ваши действия?
16. Как организовать процесс triage багов в команде? Кто участвует, как часто?
17. Сравните подходы к ведению багов в Jira, YouTrack, Bugzilla. Какие поля критичны?
18. Как автоматизировать создание баг-репортов (например, из результатов автотестов)?

## Практические задания

### Задание 1: Напишите баг-репорт
Вы тестируете интернет-магазин. При добавлении товара в корзину с мобильного устройства (iPhone 15, Safari)
количество товаров в иконке корзины не обновляется, хотя товар добавлен. При переходе в корзину
товар отображается корректно. Составьте полноценный баг-репорт.

### Задание 2: Определите Severity и Priority
Для каждого бага определите Severity и Priority, обоснуйте:
1. Приложение банка падает при переводе суммы > 1 000 000
2. В footer сайта указан 2023 год вместо текущего
3. Логотип компании на странице партнёров отображается в низком разрешении
4. Форма регистрации принимает пустое поле email и создаёт аккаунт
5. Кнопка «Поделиться в Facebook» ведёт на несуществующую страницу

### Задание 3: Улучшите баг-репорт
Перепишите этот баг-репорт, сделав его информативным и профессиональным:
```
Summary: поиск глючит
Steps: ищу товар — показывает не то
Expected: нормально работает
Actual: глючит
Severity: Blocker
```

### Задание 4: Bug Triage
У вас 10 багов и один спринт. Распределите их по приоритетам:
- Crash при оплате (Visa, Chrome)
- Опечатка в email-рассылке (уже отправлена 50 000 пользователям)
- Неправильная сумма в чеке (отклонение 1 копейка)
- Кнопка «Назад» не работает на странице оформления
- Аватарка пользователя не отображается в мобильной версии
Обоснуйте свои решения.
