# Виды тестирования подробно

## Обзор

Классификация видов тестирования -- одна из фундаментальных тем в QA.
Понимание различий между видами тестирования позволяет правильно выстроить стратегию,
оптимально распределить ресурсы и аргументированно отвечать на вопросы на интервью.

В этом разделе разбираются наиболее востребованные виды тестирования с точки зрения
практической работы и собеседований: от smoke-тестов до exploratory testing,
от black-box до grey-box подходов.

---

## 1. Классификация по целям и глубине

### 1.1 Smoke Testing

**Smoke testing** (дымовое тестирование) -- поверхностная проверка самых критичных функций
приложения. Цель -- убедиться, что сборка (build) в принципе работоспособна и пригодна
для дальнейшего тестирования.

**Характеристики:**
- Выполняется первым после получения новой сборки
- Покрывает 10-20% самых важных сценариев
- Занимает минимум времени (минуты, не часы)
- При провале -- сборка отклоняется целиком

**Пример для интернет-магазина:**

| # | Smoke-сценарий                        | Ожидаемый результат         |
|---|---------------------------------------|-----------------------------|
| 1 | Открыть главную страницу              | Страница загружается (200)  |
| 2 | Выполнить поиск товара                | Результаты отображаются     |
| 3 | Добавить товар в корзину              | Товар появляется в корзине  |
| 4 | Перейти к оформлению заказа           | Форма оформления открывается|
| 5 | Авторизоваться под существующим логином| Пользователь входит в систему|

### 1.2 Sanity Testing

**Sanity testing** (тестирование на вменяемость) -- узконаправленная проверка конкретной
функциональности после небольших изменений. Цель -- убедиться, что исправление бага
или небольшая доработка работает корректно.

**Характеристики:**
- Выполняется после конкретного изменения
- Фокус на затронутой области
- Более глубокое, чем smoke, но более узкое
- Обычно без формальной документации

**Пример:** после исправления бага с расчётом скидки тестировщик проверяет различные
сценарии скидок, но не проверяет работу корзины или оплаты целиком.

### 1.3 Сравнение Smoke vs Sanity

| Критерий         | Smoke Testing                  | Sanity Testing                  |
|------------------|--------------------------------|---------------------------------|
| Цель             | Стабильность всей сборки       | Корректность конкретной области  |
| Широта           | Широкое, поверхностное         | Узкое, более глубокое           |
| Когда            | После каждого нового билда     | После багфикса / мелкой доработки|
| Документация     | Обычно есть набор кейсов       | Часто без формальных кейсов     |
| Кто выполняет    | QA или автоматизация (CI/CD)   | QA                              |
| При провале      | Сборка отклоняется             | Возвращается на доработку       |

---

### 1.4 Regression Testing

**Regression testing** (регрессионное тестирование) -- повторное выполнение ранее
пройденных тестов для проверки того, что существующая функциональность не сломалась
после внесённых изменений.

**Характеристики:**
- Покрывает весь функционал или его критичную часть
- Идеальный кандидат для автоматизации
- Выполняется регулярно (перед релизом, после мерджа фич)
- Набор тестов растёт со временем

**Стратегии формирования регрессионного набора:**
1. **Полная регрессия** -- все тест-кейсы (перед мажорным релизом)
2. **Частичная регрессия** -- кейсы, связанные с изменённой областью
3. **Приоритизированная регрессия** -- сначала критичные, потом остальные
4. **Risk-based** -- на основе анализа рисков

### 1.5 Re-testing

**Re-testing** (повторное тестирование) -- проверка конкретного дефекта после его
исправления разработчиком. Цель -- убедиться, что баг действительно исправлен.

**Характеристики:**
- Выполняется строго по шагам из баг-репорта
- Проверяется конкретный дефект
- Обязательно перед закрытием бага

### 1.6 Сравнение Regression vs Re-testing

| Критерий           | Regression Testing              | Re-testing                       |
|--------------------|---------------------------------|----------------------------------|
| Цель               | Проверить, что ничего не сломалось | Проверить, что баг исправлен    |
| Область            | Широкая (весь функционал)       | Узкая (конкретный баг)           |
| Можно автоматизировать | Да, приоритетно              | Можно, но обычно вручную         |
| Когда              | После любых изменений           | После исправления конкретного бага|
| Тест-кейсы         | Существующие из набора          | Шаги воспроизведения из баг-репорта|
| Результат          | Новые баги или "всё ок"         | Баг закрыт или переоткрыт       |

## 2. End-to-End Testing (E2E)

**E2E testing** -- проверка полного пользовательского сценария от начала до конца,
включая все интеграции между компонентами системы.

**Характеристики:**
- Имитирует реальное поведение пользователя
- Затрагивает все слои: UI, backend, БД, внешние сервисы
- Самые дорогие по созданию и поддержке тесты
- Находятся на вершине пирамиды тестирования

**Пример E2E-сценария для интернет-магазина:**

```
1. Пользователь открывает сайт
2. Регистрируется как новый пользователь
3. Подтверждает email
4. Ищет товар по названию
5. Добавляет товар в корзину
6. Применяет промокод
7. Оформляет заказ (вводит адрес, выбирает доставку)
8. Оплачивает картой (интеграция с платёжным шлюзом)
9. Получает подтверждение на email
10. Проверяет статус заказа в личном кабинете
```

**Инструменты для E2E-автоматизации:**
- Web: Selenium, Playwright, Cypress
- Mobile: Appium
- API + UI: RestAssured + Selenium (комбинированный подход)

## 3. Exploratory Testing и Ad-hoc Testing

### 3.1 Exploratory Testing

**Exploratory testing** (исследовательское тестирование) -- одновременное изучение
приложения, проектирование и выполнение тестов. Тестировщик опирается на опыт,
интуицию и знание типичных проблем.

**Структура сессии (session-based approach):**
- **Charter** (хартия) -- цель сессии, например: "Исследовать работу фильтров на странице каталога"
- **Time-box** -- ограничение по времени (обычно 60-90 минут)
- **Notes** -- заметки о найденных проблемах, вопросах, идеях
- **Debrief** -- обсуждение результатов после сессии

**Когда применять:**
- Новый функционал без документации
- Нужно быстро оценить качество
- Дополнение к формальному тестированию
- Поиск "неочевидных" багов

**Пример хартии:**

```
Хартия:   Исследовать процесс восстановления пароля
Область:  Форма "Забыли пароль?", email-уведомления, смена пароля
Время:    60 минут
Фокус:    Граничные случаи, обработка ошибок, безопасность
Риски:    Несанкционированная смена пароля, утечка данных
```

### 3.2 Ad-hoc Testing

**Ad-hoc testing** -- неструктурированное тестирование без плана, хартии и документации.
Тестировщик "ломает" приложение как угодно.

**Отличие от exploratory testing:**

| Критерий         | Exploratory Testing          | Ad-hoc Testing                |
|------------------|------------------------------|-------------------------------|
| Структура        | Есть (хартия, time-box)      | Нет                           |
| Документация     | Заметки во время сессии       | Нет                           |
| Цель             | Определена заранее            | Отсутствует                   |
| Воспроизводимость| Частичная                     | Минимальная                   |
| Навыки           | Требует опыта                 | Любой уровень                 |

## 4. Классификация по доступу к коду

### 4.1 Black-box Testing

**Black-box testing** (тестирование чёрного ящика) -- тестирование без знания
внутренней реализации. Тестировщик работает только с входными данными и результатами.

**Применяемые техники:**
- Equivalence Partitioning
- Boundary Value Analysis
- Decision Tables
- State Transition Diagrams
- Pairwise Testing

**Преимущества:**
- Тестировщику не нужны знания программирования
- Тесты моделируют реальное поведение пользователя
- Независимость от реализации (тесты не ломаются при рефакторинге)

**Недостатки:**
- Невозможно проверить все пути выполнения кода
- Сложно достичь высокого code coverage
- Возможно дублирование проверок

### 4.2 White-box Testing

**White-box testing** (тестирование белого ящика) -- тестирование с полным знанием
внутренней структуры кода. Тестировщик анализирует логику, ветвления, циклы.

**Применяемые техники:**
- Statement coverage (покрытие операторов)
- Branch coverage (покрытие ветвей)
- Path coverage (покрытие путей)
- Condition coverage (покрытие условий)

**Пример:**

```java
// Метод для тестирования
public String classify(int score) {
    if (score >= 90) {           // Ветвь 1
        return "Отлично";
    } else if (score >= 70) {    // Ветвь 2
        return "Хорошо";
    } else if (score >= 50) {    // Ветвь 3
        return "Удовлетворительно";
    } else {                     // Ветвь 4
        return "Неудовлетворительно";
    }
}
```

**Тесты для 100% branch coverage:**

| Тест  | Вход       | Ожидаемый результат     | Покрытая ветвь |
|-------|------------|-------------------------|----------------|
| TC-1  | score = 95 | "Отлично"               | Ветвь 1        |
| TC-2  | score = 80 | "Хорошо"                | Ветвь 2        |
| TC-3  | score = 60 | "Удовлетворительно"     | Ветвь 3        |
| TC-4  | score = 30 | "Неудовлетворительно"   | Ветвь 4        |

### 4.3 Grey-box Testing

**Grey-box testing** -- тестирование с частичным знанием внутренней структуры.
Тестировщик знает архитектуру, схему БД, API-контракты, но не анализирует код построчно.

**Типичные сценарии:**
- Тестирование API: знаешь endpoint-ы, формат запросов, схему БД
- Тестирование интеграций: знаешь, какие сервисы взаимодействуют
- Тестирование миграций: знаешь структуру таблиц до и после

**Сравнение трёх подходов:**

| Критерий            | Black-box       | Grey-box         | White-box        |
|---------------------|-----------------|------------------|------------------|
| Знание кода         | Нет             | Частичное        | Полное           |
| Кто обычно выполняет| QA              | QA / SDET        | Разработчик      |
| Уровень тестирования| System, UAT     | Integration, API | Unit             |
| Фокус               | Функционал      | Архитектура      | Логика кода      |
| Навыки              | Предметная область| Техническая база| Программирование |

## 5. Positive vs Negative Testing

### 5.1 Positive Testing

Проверка системы с корректными (валидными) данными. Цель -- убедиться, что
система работает правильно при нормальном использовании.

**Примеры для формы регистрации:**
- Ввести корректный email: `user@example.com`
- Ввести пароль, удовлетворяющий всем требованиям
- Заполнить все обязательные поля корректными значениями

### 5.2 Negative Testing

Проверка системы с некорректными (невалидными) данными. Цель -- убедиться, что
система корректно обрабатывает ошибки и не падает.

**Примеры для формы регистрации:**
- Ввести email без `@`: `userexample.com`
- Оставить обязательное поле пустым
- Ввести в числовое поле буквы
- Ввести строку длиной 10 000 символов
- Отправить SQL-инъекцию: `' OR 1=1 --`
- Ввести спецсимволы: `<script>alert('XSS')</script>`

**Распределение в тест-наборе:**
Обычно соотношение positive/negative -- примерно 30/70 или 40/60,
потому что невалидных сценариев значительно больше, чем валидных.

## Связь с тестированием

Понимание видов тестирования напрямую влияет на ежедневную работу QA-инженера:

1. **Планирование:** правильный выбор типа тестирования для текущей задачи экономит время
2. **Автоматизация:** regression и smoke -- первые кандидаты на автоматизацию
3. **Коммуникация:** умение объяснить, почему нужен тот или иной вид тестирования
4. **CI/CD:** smoke-тесты в pipeline, regression -- по расписанию, E2E -- перед релизом
5. **Отчётность:** в тест-плане указывается, какие виды тестирования проводятся

**Пример распределения в CI/CD pipeline:**

```
Commit  -->  Unit Tests (разработчики)
        -->  Smoke Tests (автоматизация, 5 мин)
        -->  Sanity Tests (автоматизация, 15 мин)

Nightly -->  Regression Suite (автоматизация, 2-4 часа)

Release -->  Full Regression (авто + ручное)
        -->  E2E Scenarios (автоматизация)
        -->  Exploratory Sessions (ручное)
```

## Типичные ошибки

1. **Путать smoke и sanity.** Smoke -- широкое и поверхностное, sanity -- узкое и глубокое.
2. **Путать regression и re-testing.** Regression -- проверка "не сломалось ли старое", re-testing -- проверка "исправлен ли конкретный баг".
3. **Считать, что exploratory = ad-hoc.** Exploratory testing -- структурированный подход с хартией, ad-hoc -- хаотичный.
4. **Игнорировать negative testing.** Большинство реальных багов находятся именно в негативных сценариях.
5. **Думать, что black-box -- это "неквалифицированное" тестирование.** На самом деле это мощный подход с формальными техниками тест-дизайна.
6. **Автоматизировать exploratory testing.** По определению, его нельзя полностью автоматизировать -- в нём главную роль играет человеческая интуиция.
7. **Запускать полную регрессию на каждый коммит.** Это неоправданно дорого; нужно использовать приоритизацию и risk-based подход.

## Вопросы на интервью

### 🟢 Базовый уровень

1. **Что такое smoke testing и когда его проводят?**
   Поверхностная проверка основного функционала после получения новой сборки. Цель -- решить, стоит ли тестировать дальше.

2. **В чём разница между regression и re-testing?**
   Regression -- проверка, что существующий функционал не сломался после изменений. Re-testing -- проверка конкретного исправленного бага.

3. **Что такое positive и negative testing? Приведите примеры.**
   Positive -- проверка с валидными данными (корректный email). Negative -- проверка с невалидными данными (email без @, пустое поле).

4. **Что такое black-box testing?**
   Тестирование без знания внутренней реализации, на основе входных/выходных данных и требований.

### 🟡 Средний уровень

5. **Чем exploratory testing отличается от ad-hoc?**
   Exploratory -- структурированный подход (хартия, time-box, заметки). Ad-hoc -- полностью неструктурированный.

6. **Как вы формируете regression suite? Какие критерии включения тестов?**
   На основе критичности функционала, частоты использования, истории дефектов, затронутых областей. Приоритизация: critical path > high-risk areas > остальное.

7. **Когда grey-box testing предпочтительнее black-box?**
   Когда знание архитектуры позволяет писать более целевые тесты -- например, при тестировании API, зная схему БД, можно проверять корректность сохранения данных.

8. **Как организовать E2E-тестирование, чтобы оно не тормозило CI/CD?**
   Ограниченный набор критичных сценариев, параллельный запуск, стабильные тестовые данные, отдельный pipeline, запуск по расписанию (не на каждый коммит).

### 🔴 Продвинутый уровень

9. **Вы пришли на проект без тестовой документации. Как построите процесс тестирования? Какие виды тестирования внедрите в первую очередь?**
   Начать с exploratory-сессий для изучения продукта. Параллельно создать smoke-набор. Далее -- critical path regression suite. Потом -- формализация тест-кейсов для ключевых фич.

10. **Как оценить достаточность regression suite? Какие метрики использовать?**
    Метрики: code coverage (для автотестов), requirement coverage, defect detection rate, количество пропущенных в продакшен багов, время выполнения suite. Risk-based подход: все high-risk области должны быть покрыты.

11. **Как сбалансировать scripted testing и exploratory testing в Agile-проекте?**
    Scripted -- для regression и acceptance criteria. Exploratory -- для новых фич и поиска "неочевидных" багов. Соотношение зависит от зрелости продукта: новый продукт -- больше exploratory, зрелый -- больше scripted.

## Практические задания

### Задание 1: Smoke Test Suite
Составьте smoke test suite (5-10 кейсов) для любого знакомого вам веб-приложения
(Gmail, YouTube, Telegram Web). Обоснуйте выбор каждого кейса.

### Задание 2: Классификация
Для каждого из следующих сценариев определите тип тестирования:
- a) Разработчик исправил баг с отображением цены. Вы проверяете, что цена отображается корректно.
- b) Вы прогоняете набор из 200 автотестов перед релизом.
- c) Вы получили новый билд и проверяете, что приложение запускается, логин работает.
- d) Вы 60 минут исследуете новый модуль "Отчёты", делая заметки.
- e) Вы проверяете, что при вводе отрицательной цены система показывает ошибку.

**Ответы:** a) Re-testing, b) Regression, c) Smoke, d) Exploratory, e) Negative testing.

### Задание 3: Regression Suite Prioritization
У вас есть 500 тест-кейсов и 4 часа до релиза. Опишите стратегию приоритизации:
какие тесты запустите первыми, какие можно пропустить, как обоснуете решение.

### Задание 4: Exploratory Session
Проведите 30-минутную exploratory-сессию на любом публичном сайте. Заполните отчёт:
- Хартия (цель)
- Найденные проблемы
- Вопросы к команде
- Области для дальнейшего исследования
