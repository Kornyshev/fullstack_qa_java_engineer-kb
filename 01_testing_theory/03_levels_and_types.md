# Уровни тестирования и пирамида тестирования

## Обзор

Уровни тестирования определяют, НА КАКОМ этапе и с КАКОЙ гранулярностью проверяется
программный продукт. Пирамида тестирования -- это модель, описывающая оптимальное
соотношение тестов разных уровней.

Понимание уровней тестирования критически важно для QA-инженера: оно определяет,
какие тесты вы пишете, какие инструменты используете и с кем взаимодействуете.
На собеседованиях эту тему спрашивают на всех уровнях -- от Junior до Senior.

Знание пирамиды тестирования и её вариаций демонстрирует зрелость инженера
и понимание экономики тестирования. Senior QA-инженер должен уметь обосновать,
почему конкретная стратегия тестирования подходит (или не подходит) для проекта.

---

## Уровни тестирования

### Unit Testing (модульное тестирование)

**Unit Testing** -- тестирование отдельных модулей (функций, методов, классов)
в изоляции от остальной системы.

**Цель:** убедиться, что каждый отдельный компонент работает корректно.

**Кто выполняет:** разработчики (в первую очередь), иногда QA-автоматизаторы.

**Характеристики:**
- Тестируется наименьшая единица кода (метод, функция, класс)
- Зависимости заменяются mock/stub/fake-объектами
- Выполняются быстро (миллисекунды на тест)
- Запускаются часто (при каждом коммите)
- Легко поддерживать и отлаживать

**Инструменты (Java):** JUnit 5, TestNG, Mockito, AssertJ

**Пример:**

```java
// Тестируемый класс -- калькулятор скидок
public class DiscountCalculator {
    public double calculate(double price, int loyaltyYears) {
        if (loyaltyYears >= 5) return price * 0.8;  // скидка 20%
        if (loyaltyYears >= 2) return price * 0.9;  // скидка 10%
        return price;                                 // без скидки
    }
}

// Unit-тест -- проверяем метод calculate в изоляции
@Test
void shouldApply20PercentDiscountForLoyalCustomer() {
    DiscountCalculator calculator = new DiscountCalculator();
    double result = calculator.calculate(100.0, 5);
    assertEquals(80.0, result);
}
```

**Что покрывать unit-тестами:**
- Бизнес-логику (вычисления, правила, валидации)
- Граничные значения
- Обработку ошибок и исключений
- Edge cases

**Что НЕ стоит покрывать:**
- Тривиальные геттеры/сеттеры
- Конфигурационные классы
- Прямые вызовы сторонних библиотек

### Integration Testing (интеграционное тестирование)

**Integration Testing** -- тестирование взаимодействия между компонентами, модулями
или системами.

**Цель:** убедиться, что модули корректно работают вместе.

**Кто выполняет:** разработчики и QA-автоматизаторы.

**Характеристики:**
- Тестируется взаимодействие 2+ компонентов
- Используются реальные (или приближённые к реальным) зависимости
- Выполняются медленнее unit-тестов (секунды)
- Могут требовать внешних ресурсов (БД, очереди, API)
- Выявляют проблемы интерфейсов и контрактов

**Инструменты (Java):** Spring Boot Test, Testcontainers, WireMock, REST Assured

**Подходы к интеграционному тестированию:**

| Подход | Описание | Плюсы | Минусы |
|--------|----------|-------|--------|
| **Big Bang** | Все модули интегрируются и тестируются одновременно | Простота начала | Сложно локализовать ошибку |
| **Top-Down** | От верхнего уровня к нижнему, нижние заменяются stubs | Раннее тестирование UI | Нижние модули тестируются поздно |
| **Bottom-Up** | От нижнего уровня к верхнему, верхние заменяются drivers | Нет нужды в stubs | UI тестируется поздно |
| **Sandwich** | Комбинация Top-Down и Bottom-Up | Гибкость | Сложность организации |

**Пример:**

```java
// Интеграционный тест -- проверяем взаимодействие сервиса с БД
@SpringBootTest
@Testcontainers
class OrderServiceIntegrationTest {

    @Container
    // Реальная PostgreSQL в Docker-контейнере
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:15");

    @Autowired
    private OrderService orderService;

    @Test
    void shouldSaveAndRetrieveOrder() {
        // Создаём заказ через сервис
        Order order = orderService.createOrder("user-1", List.of("item-1", "item-2"));

        // Проверяем, что заказ сохранён в реальной БД и может быть прочитан
        Order retrieved = orderService.getById(order.getId());
        assertEquals(2, retrieved.getItems().size());
        assertEquals("user-1", retrieved.getUserId());
    }
}
```

**Что покрывать интеграционными тестами:**
- Взаимодействие с базой данных (repository layer)
- Вызовы внешних API (с помощью WireMock или реальных sandbox)
- Работу с очередями сообщений (Kafka, RabbitMQ)
- Сериализацию/десериализацию (JSON, XML)
- Конфигурацию Spring-контекста (правильность wiring)

### System Testing (системное тестирование)

**System Testing** -- тестирование полностью интегрированной системы
для проверки соответствия требованиям.

**Цель:** убедиться, что система в целом работает корректно.

**Кто выполняет:** QA-инженеры (ручные и автоматизаторы).

**Характеристики:**
- Тестируется вся система целиком (end-to-end)
- Проводится в среде, максимально приближённой к production
- Включает функциональное и нефункциональное тестирование
- Выполняется медленнее всех предыдущих уровней (минуты, часы)
- Выявляет проблемы, которые невозможно найти на нижних уровнях

**Инструменты:** Selenium, Playwright, Cypress (UI); REST Assured, Postman (API);
JMeter, Gatling (Performance)

**Виды системного тестирования:**
- Functional testing -- проверка функциональных требований
- Performance testing -- нагрузка, стресс, выносливость
- Security testing -- безопасность
- Usability testing -- удобство использования
- Compatibility testing -- совместимость с браузерами, ОС, устройствами
- Reliability testing -- надёжность
- Recovery testing -- восстановление после сбоев

**Пример:**

```java
// End-to-end тест -- полный сценарий покупки
@Test
void shouldCompleteFullPurchaseFlow() {
    // Пользователь открывает каталог
    catalogPage.open();
    catalogPage.searchProduct("Ноутбук");

    // Добавляет товар в корзину
    productPage.addToCart();

    // Переходит к оформлению
    cartPage.proceedToCheckout();

    // Заполняет данные доставки
    checkoutPage.fillDeliveryAddress("ул. Пушкина, д. 10");
    checkoutPage.selectPaymentMethod("Карта");
    checkoutPage.fillCardDetails("4111111111111111", "12/27", "123");

    // Подтверждает заказ
    checkoutPage.confirmOrder();

    // Проверяет подтверждение
    assertThat(confirmationPage.getOrderStatus()).isEqualTo("Оформлен");
    assertThat(confirmationPage.getOrderNumber()).isNotEmpty();
}
```

### Acceptance Testing (приёмочное тестирование)

**Acceptance Testing** -- формальное тестирование, проводимое для определения того,
удовлетворяет ли система критериям приёмки и может ли заказчик/пользователь принять систему.

**Цель:** подтвердить, что система готова к использованию.

**Кто выполняет:** заказчик, конечные пользователи, бизнес-аналитики (при поддержке QA).

**Виды приёмочного тестирования:**

| Вид | Описание | Кто проводит |
|-----|----------|-------------|
| **UAT (User Acceptance Testing)** | Проверка пользователями в реальных сценариях | Конечные пользователи |
| **BAT (Business Acceptance Testing)** | Проверка соответствия бизнес-требованиям | Бизнес-аналитики, Product Owner |
| **Contract Acceptance Testing** | Проверка выполнения условий контракта | Заказчик |
| **Regulation Acceptance Testing** | Проверка соответствия нормативным требованиям | Регуляторы, аудиторы |
| **Alpha Testing** | Тестирование внутри компании-разработчика | Внутренние пользователи |
| **Beta Testing** | Тестирование внешними пользователями до релиза | Ограниченная группа пользователей |

**Роль QA в приёмочном тестировании:**
- Подготовка тестовой среды и данных
- Написание acceptance criteria совместно с бизнесом
- Помощь пользователям в проведении тестов
- Фиксация и отслеживание обнаруженных проблем
- Подготовка отчёта о результатах

## Сравнительная таблица уровней

| Критерий | Unit | Integration | System | Acceptance |
|----------|------|-------------|--------|------------|
| **Объект** | Метод, класс | Модули вместе | Вся система | Бизнес-сценарии |
| **Кто тестирует** | Разработчик | Разработчик / QA | QA | Заказчик / Пользователь |
| **Скорость** | Мс | Секунды | Минуты | Часы/Дни |
| **Изоляция** | Полная (mocks) | Частичная | Нет | Нет |
| **Среда** | Локальная | Тестовая | Staging | Staging / Production-like |
| **Стоимость падения** | Низкая | Средняя | Высокая | Очень высокая |
| **Количество тестов** | Тысячи | Сотни | Десятки-сотни | Десятки |
| **Частота запуска** | Каждый коммит | Каждый PR/build | Daily/Per release | Per release |
| **Основной фокус** | Логика | Взаимодействие | Требования | Потребности бизнеса |

## Пирамида тестирования

### Классическая пирамида (Mike Cohn)

Пирамида тестирования -- это модель, предложенная Mike Cohn в книге "Succeeding with Agile"
(2009). Она описывает оптимальное соотношение тестов разных уровней.

```
         /\
        /  \          UI / E2E Tests
       /    \         (мало, медленные, дорогие)
      /------\
     /        \       Integration Tests
    /          \      (среднее количество)
   /------------\
  /              \    Unit Tests
 /                \   (много, быстрые, дешёвые)
/------------------\
```

**Принцип пирамиды:**
- **Основание (Unit):** максимальное количество быстрых, дешёвых, стабильных тестов.
- **Середина (Integration):** умеренное количество тестов взаимодействия.
- **Вершина (UI/E2E):** минимальное количество медленных, дорогих, хрупких тестов.

**Соотношение (ориентировочно):**
- Unit: 70%
- Integration: 20%
- E2E: 10%

### Почему именно пирамида

| Характеристика | Unit | Integration | E2E |
|---------------|------|-------------|-----|
| **Скорость** | Быстро (мс) | Средне (сек) | Медленно (мин) |
| **Стоимость создания** | Низкая | Средняя | Высокая |
| **Стоимость поддержки** | Низкая | Средняя | Высокая |
| **Стабильность** | Высокая | Средняя | Низкая (flaky tests) |
| **Обратная связь** | Мгновенная | Быстрая | Медленная |
| **Локализация ошибки** | Точная | Приблизительная | Размытая |
| **Покрытие интеграций** | Нет | Частичное | Полное |
| **Confidence** | Частичная | Средняя | Высокая |

### Перевёрнутая пирамида (Ice Cream Cone Anti-Pattern)

Это распространённый антипаттерн, когда большинство тестов -- E2E/UI, а unit-тестов мало.

```
/------------------\
\                  /   Manual Testing
 \                /    (много ручного тестирования)
  \--------------/
   \            /      E2E / UI Tests
    \          /       (много, медленные, нестабильные)
     \--------/
      \      /         Integration Tests
       \    /          (мало)
        \  /
         \/            Unit Tests
                       (почти нет)
```

**Почему это плохо:**
- **Медленная обратная связь** -- цикл тестирования занимает часы/дни.
- **Высокая стоимость поддержки** -- UI-тесты ломаются при любом изменении интерфейса.
- **Flaky tests** -- E2E-тесты нестабильны (сеть, таймауты, состояние среды).
- **Сложная локализация дефектов** -- если E2E-тест упал, непонятно, где именно проблема.
- **Блокировка релизов** -- нестабильные тесты приводят к ложным срабатываниям.
- **Высокая стоимость инфраструктуры** -- нужны реальные среды, браузеры, устройства.

**Как формируется антипаттерн:**
1. Команда начинает без unit-тестов ("потом напишем").
2. Когда появляются баги, QA пишет E2E-тесты ("так надёжнее").
3. E2E-тестов становится всё больше, они становятся медленнее и нестабильнее.
4. Команда тратит больше времени на поддержку тестов, чем на разработку.

### Testing Trophy (Kent C. Dodds)

Модель Testing Trophy, предложенная Kent C. Dodds, адаптирует пирамиду для современных
веб-приложений (особенно с React/frontend).

```
         ___
        |   |        E2E Tests
        |   |        (несколько ключевых сценариев)
       /     \
      /       \
     /         \     Integration Tests
    /           \    (основная масса тестов)
   /             \
  /_______________\
        | |          Unit Tests
        | |          (только сложная логика)
        |_|
  _______________
  |  Static     |    Static Analysis
  |  Analysis   |    (TypeScript, ESLint, Prettier)
  |_____________|
```

**Ключевое отличие от классической пирамиды:**
- **Основание -- Static Analysis** (линтеры, типы, форматтеры) -- ловит ошибки до запуска.
- **Unit-тесты уменьшены** -- только для чистой бизнес-логики и утилит.
- **Integration-тесты в центре** -- основная масса тестов проверяет взаимодействие компонентов.
- **E2E-тесты на вершине** -- несколько критических сценариев.

**Когда применять:**
- Современные веб-приложения (React, Vue, Angular)
- Микросервисная архитектура с API-контрактами
- Проекты с TypeScript и строгой типизацией

### Testing Honeycomb (Spotify)

Модель, предложенная в Spotify для микросервисной архитектуры.

```
      /\
     /  \          E2E / UI Tests
    /    \         (минимум)
   /------\
  /        \
 /          \      Integration Tests
|            |     (максимум -- ядро тестирования)
 \          /
  \        /
   \------/
    \    /         Unit Tests
     \  /          (умеренно)
      \/
```

**Идея:** в микросервисной архитектуре самое важное -- это контракты между сервисами.
Поэтому интеграционные тесты (contract tests, API tests) важнее unit-тестов.

**Когда применять:**
- Микросервисная архитектура
- Event-driven системы
- Проекты с множеством внешних интеграций

## Участие QA на каждом уровне

### Unit Testing

QA-инженер обычно не пишет unit-тесты, но может:
- **Влиять на их качество** -- участвовать в code review тестов.
- **Определять критерии** -- помогать устанавливать coverage thresholds.
- **Анализировать покрытие** -- использовать JaCoCo/SonarQube для оценки.
- **Предлагать test cases** -- подсказывать разработчикам граничные значения.
- **Внедрять mutation testing** -- проверять качество unit-тестов через PIT/Stryker.

### Integration Testing

QA-автоматизатор активно участвует:
- **Пишет API-тесты** -- проверяет контракты между сервисами.
- **Настраивает тестовую инфраструктуру** -- Testcontainers, WireMock, Docker Compose.
- **Тестирует интеграции с внешними системами** -- платёжные системы, почта, SMS.
- **Проводит contract testing** -- Pact, Spring Cloud Contract.

### System Testing

Основная область ответственности QA:
- **Разрабатывает тест-кейсы** на основе требований.
- **Автоматизирует E2E-сценарии** -- Selenium, Playwright, Cypress.
- **Проводит exploratory testing** -- исследование продукта.
- **Выполняет нефункциональное тестирование** -- performance, security, usability.
- **Управляет тестовыми данными и средами.**

### Acceptance Testing

QA выступает как фасилитатор:
- **Подготавливает среду** для пользовательского тестирования.
- **Помогает формулировать acceptance criteria** -- совместно с Product Owner.
- **Координирует UAT** -- организует процесс, собирает feedback.
- **Автоматизирует acceptance tests** -- BDD (Cucumber, JBehave).

## Связь с тестированием

Понимание уровней и пирамиды тестирования напрямую влияет на ежедневную работу QA:

- **Планирование тестирования** -- определяет, какие тесты и на каком уровне писать.
- **Автоматизация** -- помогает выбрать правильный инструмент для каждого уровня.
- **Оценка трудозатрат** -- E2E-тесты стоят в 10 раз дороже unit-тестов.
- **Стратегия тестирования** -- выбор модели (пирамида, trophy, honeycomb) зависит от архитектуры.
- **Коммуникация с командой** -- общий язык для обсуждения тестовой стратегии.
- **Оценка зрелости** -- по форме "пирамиды" можно судить о зрелости тестирования в проекте.

Если в проекте преобладают E2E-тесты -- это сигнал к рефакторингу тестовой стратегии.
Если unit-тестов нет совсем -- нужно работать с командой разработки над их внедрением.

## Типичные ошибки

1. **Путать интеграционные и системные тесты.** Интеграционные тесты проверяют взаимодействие
   конкретных модулей (сервис + БД). Системные тесты проверяют всю систему целиком (E2E).

2. **Автоматизировать всё на уровне UI.** Это самая дорогая и хрупкая автоматизация.
   Если можно проверить логику unit-тестом -- проверяйте unit-тестом.

3. **Считать, что unit-тесты -- ответственность только разработчиков.** QA может и должен
   влиять на качество unit-тестов через code review и определение критериев покрытия.

4. **Слепо следовать пропорциям 70/20/10.** Это ориентир, а не закон. Для микросервисов
   соотношение будет другим (honeycomb). Для frontend-приложений -- другим (trophy).

5. **Игнорировать static analysis.** Линтеры, типизация и форматтеры ловят целый класс
   ошибок бесплатно. Это "нулевой уровень" пирамиды.

6. **Не различать виды интеграционного тестирования.** Component integration test
   (взаимодействие классов) и system integration test (взаимодействие систем) --
   это разные вещи с разными инструментами и подходами.

7. **Отказываться от E2E-тестов.** Несмотря на стоимость, несколько ключевых E2E-сценариев
   (happy path) необходимы для уверенности в работоспособности системы.

## Вопросы на интервью

### 🟢 Базовый уровень (Junior)

1. **Назовите 4 уровня тестирования.**
   Ожидаемый ответ: Unit, Integration, System, Acceptance. Кратко описать каждый.

2. **Что такое пирамида тестирования?**
   Ожидаемый ответ: модель, описывающая оптимальное соотношение тестов.
   Внизу -- много быстрых unit-тестов, вверху -- мало медленных E2E-тестов.

3. **Чем unit-тесты отличаются от интеграционных?**
   Ожидаемый ответ: unit-тесты проверяют отдельный модуль в изоляции (с mocks),
   интеграционные -- взаимодействие нескольких модулей (с реальными зависимостями).

4. **Кто обычно пишет unit-тесты?**
   Ожидаемый ответ: разработчики. QA может влиять на качество через code review
   и определение критериев покрытия.

5. **Что такое UAT?**
   Ожидаемый ответ: User Acceptance Testing -- приёмочное тестирование, которое
   проводят конечные пользователи для подтверждения готовности продукта.

### 🟡 Средний уровень (Middle)

6. **Почему перевёрнутая пирамида -- это антипаттерн? Как это исправить?**
   Ожидаемый ответ: медленная обратная связь, нестабильные тесты, дорогая поддержка.
   Исправление: перенос проверок на нижние уровни, покрытие логики unit-тестами,
   сокращение E2E до ключевых сценариев.

7. **Какие тесты вы бы написали для REST API endpoint?**
   Ожидаемый ответ: unit (бизнес-логика сервиса), integration (сервис + БД, сервис + API),
   E2E (полный сценарий через HTTP). Объяснение, что именно проверяется на каждом уровне.

8. **Как вы решаете, что тестировать unit-тестом, а что -- интеграционным?**
   Ожидаемый ответ: чистая бизнес-логика без зависимостей -- unit. Взаимодействие
   с внешними ресурсами (БД, API, файловая система) -- integration.

9. **Что такое contract testing? На каком уровне пирамиды он находится?**
   Ожидаемый ответ: проверка контрактов между сервисами (consumer-driven contracts).
   Находится на уровне интеграционного тестирования. Инструменты: Pact, Spring Cloud Contract.

10. **Как вы тестируете микросервис? Какую модель используете -- пирамиду или другую?**
    Ожидаемый ответ: для микросервисов подходит honeycomb (Spotify) -- акцент
    на интеграционных тестах и contract tests. Обоснование: в микросервисах
    критичны интеграции, а не изолированная логика.

### 🔴 Продвинутый уровень (Senior)

11. **Как вы строите тестовую стратегию для нового проекта? Как выбираете модель?**
    Ожидаемый ответ: анализ архитектуры (монолит vs микросервисы), стека технологий,
    команды, рисков. Для монолита -- классическая пирамида. Для микросервисов --
    honeycomb. Для frontend -- trophy. Обоснование выбора.

12. **Как вы переводите проект с "перевёрнутой пирамиды" на правильную? Какие шаги предпримете?**
    Ожидаемый ответ: постепенный подход. 1) Анализ существующих E2E -- какие можно
    заменить unit/integration. 2) Внедрение unit-тестов в Definition of Done.
    3) Обучение разработчиков. 4) Настройка coverage gates в CI.
    5) Постепенное удаление дублирующих E2E-тестов. 6) Метрики для отслеживания прогресса.

13. **Сравните Testing Trophy и классическую пирамиду. Когда какую применять?**
    Ожидаемый ответ: Trophy делает акцент на static analysis и integration.
    Подходит для типизированных языков (TypeScript) и современных фреймворков.
    Пирамида -- для backend-систем и монолитов с богатой бизнес-логикой.

14. **Как обеспечить баланс между быстрой обратной связью и confidence?**
    Ожидаемый ответ: многоуровневый CI pipeline. На каждый коммит -- unit + static analysis
    (секунды). На PR -- integration (минуты). На merge в main -- E2E smoke (десятки минут).
    Перед релизом -- full regression (часы). Каждый уровень даёт свой уровень confidence.

## Практические задания

### Задание 1: Определение уровня тестирования
Определите, к какому уровню относится каждый тест:

- Проверка, что метод `calculateTax(amount)` корректно считает НДС 20%.
- Проверка, что при создании заказа запись появляется в базе данных.
- Пользователь может зарегистрироваться, войти и сделать заказ.
- Product Owner проверяет, что новая фича отчётности соответствует бизнес-требованиям.
- Проверка, что сервис заказов корректно вызывает сервис оплаты через REST API.
- Группа пользователей тестирует бета-версию мобильного приложения.

### Задание 2: Анализ тестовой стратегии
Вам дана статистика тестов проекта:
- Unit-тесты: 50, время выполнения 5 секунд
- Integration-тесты: 80, время выполнения 15 минут
- E2E-тесты: 300, время выполнения 4 часа
- Ручные тесты: 200, время выполнения 3 дня

Вопросы:
1. Какой форме соответствует текущая стратегия?
2. Какие проблемы вы видите?
3. Предложите план оптимизации с конкретными шагами.

### Задание 3: Проектирование тестовой пирамиды
Спроектируйте тестовую стратегию для сервиса онлайн-бронирования отелей.
Для каждого уровня пирамиды укажите:
- Количество тестов (приблизительно)
- Что именно тестируется (конкретные примеры)
- Инструменты
- Частоту запуска
- Кто ответственный

### Задание 4: Рефакторинг E2E в unit + integration
Дан E2E-тест: "Пользователь добавляет товар в корзину, применяет промокод,
оформляет заказ, получает подтверждение на email."

Разделите этот E2E-тест на набор тестов нижних уровней:
- Какие unit-тесты нужны?
- Какие integration-тесты нужны?
- Какой минимальный E2E-тест оставить?
