# Метрики и покрытие тестирования

## Обзор

Метрики тестирования (Test Metrics) — это количественные показатели, позволяющие объективно оценить
качество продукта, эффективность процесса тестирования и зрелость QA-практик в команде.
Без метрик тестирование превращается в субъективную оценку «вроде всё работает».

Покрытие тестирования (Test Coverage) — это мера того, насколько полно тесты проверяют
тестируемый объект: требования, код, пользовательские сценарии.

Умение выбирать, собирать и интерпретировать метрики — ключевой навык Senior QA и QA Lead.
На собеседованиях этот блок особенно важен для позиций с элементами управления процессом.

## Типы покрытия тестирования

### Requirements Coverage (Покрытие требований)

Показывает, какая доля требований покрыта хотя бы одним тест-кейсом.

```
Requirements Coverage = (Количество покрытых требований / Общее количество требований) × 100%
```

**Пример:**
- Всего требований: 120
- Покрыто тест-кейсами: 108
- Покрытие: 108/120 × 100% = **90%**

**Особенности:**
- Требование считается покрытым, если существует хотя бы один тест-кейс, проверяющий его
- 100% покрытие требований **не гарантирует** отсутствие дефектов — требования могут быть неполными
- Используется Traceability Matrix для отслеживания связей

### Code Coverage (Покрытие кода)

Показывает, какая доля исходного кода выполняется при запуске тестов.

| Тип покрытия | Что измеряет | Пример |
|---|---|---|
| **Line Coverage** | Процент выполненных строк кода | 85% строк были выполнены |
| **Branch Coverage** | Процент пройденных ветвей (if/else) | 72% ветвей были проверены |
| **Method Coverage** | Процент вызванных методов | 91% методов были вызваны |
| **Condition Coverage** | Процент проверенных логических условий | Каждое условие в if проверено как true и false |
| **Path Coverage** | Процент пройденных путей выполнения | Все комбинации ветвей проверены |

**Инструменты для Java:**
- **JaCoCo** — самый популярный, интегрируется с Maven/Gradle
- **Cobertura** — альтернатива JaCoCo
- **SonarQube** — агрегирует покрытие и добавляет другие метрики качества кода

```xml
<!-- Пример настройки JaCoCo в Maven -->
<plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <version>0.8.11</version>
    <executions>
        <execution>
            <!-- Подготовка агента для сбора покрытия -->
            <goals>
                <goal>prepare-agent</goal>
            </goals>
        </execution>
        <execution>
            <id>report</id>
            <phase>test</phase>
            <!-- Генерация отчёта о покрытии после выполнения тестов -->
            <goals>
                <goal>report</goal>
            </goals>
        </execution>
    </executions>
</plugin>
```

**Важно:** Code Coverage — это метрика **полноты**, а не **качества** тестов. Можно достичь
100% покрытия строк тестами без единого assertion.

### Functional Coverage (Покрытие функциональности)

Процент функций / фич продукта, покрытых тестами. Отличается от Requirements Coverage тем,
что одна фича может включать несколько требований.

### Risk-based Coverage (Покрытие на основе рисков)

Приоритетное покрытие тестами наиболее рискованных областей:
- Критичные бизнес-функции
- Часто изменяемые модули
- Модули с исторически большим количеством багов
- Новая функциональность

## Traceability Matrix (Матрица трассировки)

Traceability Matrix — это таблица, связывающая требования с тест-кейсами, что позволяет
убедиться, что каждое требование протестировано, и отследить влияние изменений.

### Пример матрицы

| ID требования | Описание требования | Тест-кейсы | Статус покрытия |
|---|---|---|---|
| REQ-001 | Регистрация пользователя по email | TC-001, TC-002, TC-003 | Покрыто |
| REQ-002 | Авторизация через Google OAuth | TC-010, TC-011 | Покрыто |
| REQ-003 | Восстановление пароля по SMS | — | **Не покрыто** |
| REQ-004 | Двухфакторная аутентификация | TC-020 | Частично покрыто |

### Виды трассировки

| Вид | Описание |
|-----|----------|
| **Forward Traceability** | От требований к тест-кейсам — «всё ли проверено?» |
| **Backward Traceability** | От тест-кейсов к требованиям — «зачем этот тест?» |
| **Bi-directional Traceability** | В обе стороны — полная картина |

### Инструменты

- **Jira + Xray / Zephyr** — связь tickets и test cases
- **TestRail** — встроенная поддержка traceability
- **Azure DevOps** — связь work items и test plans
- **Ручные таблицы (Excel/Sheets)** — для небольших проектов

## QA-метрики

### Метрики качества продукта

#### Defect Density (Плотность дефектов)

```
Defect Density = Количество дефектов / Размер модуля (KLOC или количество функций)
```

**Пример:** 15 багов найдено в модуле размером 5 KLOC → Defect Density = 3 бага / KLOC

**Применение:** Помогает выявить «проблемные» модули, требующие дополнительного внимания и рефакторинга.

#### Defect Leakage (Утечка дефектов)

```
Defect Leakage = (Дефекты, найденные после релиза / Общее количество дефектов) × 100%
```

**Пример:** Всего найдено 200 багов, из них 8 найдены в production → Defect Leakage = 4%

**Целевые значения:**
- < 5% — хорошо
- 5–10% — приемлемо
- > 10% — процесс тестирования требует улучшения

#### Defect Removal Efficiency (DRE)

```
DRE = (Дефекты, найденные до релиза / Общее количество дефектов) × 100%
```

DRE = 100% — Defect Leakage. Показывает эффективность процесса обнаружения дефектов.

### Метрики процесса тестирования

#### Test Pass Rate (Процент прохождения тестов)

```
Test Pass Rate = (Количество пройденных тестов / Общее количество выполненных тестов) × 100%
```

**Пример:** 450 passed / 500 total = **90%**

**Как интерпретировать:**
- Низкий pass rate перед релизом — продукт не готов
- Pass rate стабильно 100% — возможно, тесты недостаточно глубокие
- Резкое падение pass rate — регрессия после изменений

#### Test Execution Rate (Скорость выполнения тестов)

```
Test Execution Rate = (Выполненные тесты / Запланированные тесты) × 100%
```

Показывает, успевает ли команда выполнить запланированный объём тестирования.

#### Automation Percentage (Процент автоматизации)

```
Automation % = (Автоматизированные тест-кейсы / Общее количество тест-кейсов) × 100%
```

**Целевые значения (зависят от проекта):**
- Unit tests: 80–95%
- API tests: 60–90%
- UI tests: 30–60%
- E2E tests: 20–40%

#### Automation ROI

```
ROI = (Стоимость ручного тестирования − Стоимость автоматизации) / Стоимость автоматизации × 100%
```

Учитывает: время написания автотестов, время поддержки, время ручного прогона, частоту запусков.

#### Reopened Rate (Процент переоткрытых багов)

```
Reopened Rate = (Количество переоткрытых багов / Общее количество закрытых багов) × 100%
```

**Высокий Reopened Rate указывает на:**
- Некачественное исправление дефектов разработчиками
- Недостаточное описание шагов воспроизведения в баг-репорте
- Отсутствие unit-тестов на исправления

### Метрики для отчётности

#### Test Summary Report включает

| Метрика | Значение | Тренд |
|---------|----------|-------|
| Всего тест-кейсов | 500 | +20 vs прошлый спринт |
| Выполнено | 480 (96%) | — |
| Passed | 450 (93.75%) | +2% |
| Failed | 22 (4.58%) | -1.5% |
| Blocked | 8 (1.67%) | -0.5% |
| Найдено багов | 35 | -10 vs прошлый спринт |
| Critical/Blocker | 2 | -3 |
| Reopened | 4 (11.4%) | +2% |
| Automation coverage | 62% | +5% |

## Definition of Done (DoD) для QA

Definition of Done — это согласованный командой набор критериев, при выполнении которых
user story / задача считается **завершённой**.

### Пример DoD

- [ ] Все acceptance criteria проверены вручную
- [ ] Regression тесты выполнены и пройдены
- [ ] Новые тест-кейсы созданы и добавлены в тест-план
- [ ] Автотесты написаны для новой функциональности (unit + API)
- [ ] Code coverage не ниже 80% для нового кода
- [ ] Все баги с Severity Critical/Major исправлены и верифицированы
- [ ] Нет открытых Blocker / Critical багов
- [ ] Проведено exploratory testing
- [ ] Тестирование на целевых окружениях (browsers, devices) выполнено
- [ ] Performance не деградировало (response time в пределах SLA)
- [ ] Документация обновлена (API docs, release notes)

---

## Definition of Ready (DoR) для QA

Definition of Ready — критерии, при выполнении которых user story **готова к тестированию**.

### Пример DoR

- [ ] User story имеет чёткие acceptance criteria
- [ ] Макеты / дизайн доступны и финализированы
- [ ] API-спецификация (Swagger/OpenAPI) актуальна
- [ ] Тестовые данные подготовлены или описан способ их получения
- [ ] Тестовое окружение развёрнуто и доступно
- [ ] Зависимости (внешние сервисы, моки) настроены
- [ ] Feature branch задеплоен на test-окружение
- [ ] Разработчик провёл smoke-тест своих изменений
- [ ] Unit-тесты написаны и проходят

## Как собирать и отчитываться по метрикам

### Принципы

1. **Автоматизируйте сбор** — метрики должны собираться из инструментов (Jira, TestRail, CI/CD), а не вручную
2. **Визуализируйте** — dashboards в Jira, Grafana, Google Data Studio понятнее таблиц
3. **Трендируйте** — одиночное значение малоинформативно, тренд за спринты показывает динамику
4. **Контекстуализируйте** — метрика 90% pass rate бесполезна без контекста (900 smoke-тестов vs 90 глубоких)
5. **Не используйте метрики как карательный инструмент** — иначе команда начнёт «играть» с числами

### Пример дашборда

```
┌─────────────────────────────────────────────────────────┐
│                    QA Dashboard — Sprint 24             │
├──────────────────┬──────────────────────────────────────┤
│  Test Execution  │  ██████████████████░░  90% (450/500) │
│  Pass Rate       │  ████████████████████  95%           │
│  Automation      │  ████████████░░░░░░░░  62%           │
│  Defect Leakage  │  █░░░░░░░░░░░░░░░░░░░  3%           │
│  Reopened Rate   │  ██░░░░░░░░░░░░░░░░░░  8%           │

│  Sprint Velocity: 45 SP (QA capacity: 2 engineers)     │
└─────────────────────────────────────────────────────────┘
```

### Отчёт по качеству (Quality Report)

Структура регулярного отчёта:
1. **Резюме** — готов ли продукт к релизу (Go / No Go)
2. **Покрытие** — какой процент требований / функций протестирован
3. **Результаты** — pass rate, количество и распределение багов
4. **Риски** — что не было протестировано и почему, известные проблемы
5. **Метрики** — ключевые показатели с трендами
6. **Рекомендации** — что улучшить в следующем спринте

## Связь с тестированием

- Метрики помогают **объективно оценить** готовность продукта к релизу (Go / No Go decision)
- Test Coverage определяет **полноту тестирования** — находить слепые зоны до того, как их найдут пользователи
- Traceability Matrix обеспечивает **прослеживаемость** — от бизнес-требований до конкретных тестов
- DoD и DoR создают **единые стандарты** качества в команде
- Defect Leakage показывает **эффективность QA-процесса** в целом
- Automation Percentage помогает планировать **инвестиции в автоматизацию**

## Типичные ошибки

1. **Гонка за числами** — 100% code coverage ценой бессмысленных тестов без assertions
2. **Метрика ради метрики** — собирать 50 показателей, которые никто не анализирует
3. **Путать покрытие кода и покрытие требований** — 90% code coverage не значит, что все фичи проверены
4. **Использовать метрики для оценки людей** — «У тебя мало багов — плохо работаешь» убивает мотивацию
5. **Игнорировать контекст** — 50% pass rate на smoke-тестах при первом деплое — нормально, перед релизом — катастрофа
6. **Не отслеживать тренды** — одиночные значения не показывают, становится ли лучше или хуже
7. **Забывать про DoR** — начинать тестирование без готовых acceptance criteria
8. **Отсутствие Traceability** — невозможно ответить на вопрос «Какие тесты нужно перезапустить при изменении требования X?»
9. **Ручной сбор метрик** — трудоёмко и подвержено ошибкам, метрики быстро устаревают
10. **Низкое качество тест-кейсов** — покрытие высокое, но тесты не находят баги
