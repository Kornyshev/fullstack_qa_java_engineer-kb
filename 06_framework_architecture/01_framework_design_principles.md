# Принципы проектирования тестового фреймворка

## Обзор

Тестовый фреймворк — это не просто набор скриптов, а **инженерный продукт**, который должен быть удобным, масштабируемым
и поддерживаемым. Хорошо спроектированный фреймворк экономит команде сотни часов, плохо спроектированный — превращается
в техдолг, который тормозит всю автоматизацию. В этом разделе мы разберём ключевые принципы проектирования, архитектурные
слои и характеристики хорошего фреймворка с точки зрения QA-инженера.

---

## Зачем нужен фреймворк

Без фреймворка каждый автотест — это изолированный скрипт. Проблемы, которые возникают:

- **Дублирование кода** — одна и та же логика инициализации WebDriver, подключения к БД, авторизации копируется из теста в тест.
- **Хрупкость** — изменение одного локатора или URL ломает десятки тестов.
- **Сложность поддержки** — новому человеку в команде невозможно разобраться.
- **Нет единого стандарта** — каждый автор пишет тесты по-своему.
- **Невозможность масштабирования** — добавить параллельный запуск, отчёты или CI/CD интеграцию становится болезненно.

Фреймворк решает все эти проблемы, предоставляя **единую структуру, набор абстракций и соглашений**.

---

## Framework vs Library

Это фундаментальное различие, которое часто спрашивают на интервью:

| Аспект | Library (Библиотека) | Framework (Фреймворк) |
|--------|---------------------|----------------------|
| Контроль потока | **Вы** вызываете код библиотеки | **Фреймворк** вызывает ваш код (Inversion of Control) |
| Структура | Не навязывает структуру | Диктует структуру проекта |
| Расширяемость | Используете что хотите | Расширяете через точки расширения |
| Пример | Selenide, REST Assured, AssertJ | JUnit 5, TestNG, Spring Test |

**Inversion of Control (IoC)** — ключевое отличие. В библиотеке вы сами управляете потоком выполнения.
Во фреймворке — он управляет вами: вызывает ваши `@Test` методы, `@BeforeEach` хуки, слушатели.

В контексте тестовой автоматизации мы строим **собственный фреймворк**, который использует
**библиотеки** (Selenide, REST Assured) и базируется на **тестовом фреймворке** (JUnit 5).

## Многослойная архитектура

Зрелый тестовый фреймворк строится по принципу слоёв, где каждый слой имеет свою зону ответственности:

```
┌─────────────────────────────────────┐
│         Test Layer (Тесты)          │  ← Что тестируем (сценарии)
├─────────────────────────────────────┤
│   Business Logic Layer (Шаги)       │  ← Как тестируем (бизнес-действия)
├─────────────────────────────────────┤
│   Infrastructure Layer (Базовый)    │  ← Чем тестируем (инструменты)
└─────────────────────────────────────┘
```

### Test Layer (Слой тестов)

Содержит непосредственно тестовые методы. Каждый тест должен читаться как **спецификация поведения**:

```java
@Test
@DisplayName("Пользователь может добавить товар в корзину")
void userCanAddProductToCart() {
    // Подготовка: авторизоваться и открыть каталог
    loginSteps.loginAs(DEFAULT_USER);
    catalogSteps.openCategory("Электроника");

    // Действие: добавить товар в корзину
    catalogSteps.addFirstProductToCart();

    // Проверка: товар отображается в корзине
    cartSteps.verifyCartContainsItems(1);
}
```

Тест **не содержит** низкоуровневых деталей: локаторов, HTTP-запросов, SQL-запросов.

### Business Logic Layer (Слой бизнес-логики)

Содержит Steps-классы, Page Objects, API-клиенты — всё, что описывает действия на языке бизнеса:

```java
public class CatalogSteps {

    private final CatalogPage catalogPage = new CatalogPage();

    @Step("Открыть категорию '{categoryName}'")
    public void openCategory(String categoryName) {
        catalogPage.selectCategory(categoryName);
        catalogPage.waitForProductsLoaded();
    }

    @Step("Добавить первый товар в корзину")
    public void addFirstProductToCart() {
        catalogPage.getFirstProduct().clickAddToCart();
        catalogPage.waitForCartNotification();
    }
}
```

### Infrastructure Layer (Инфраструктурный слой)

Базовые классы, утилиты, конфигурация, драйверы, подключения к БД:

```java
public abstract class BasePage {

    // Ожидание загрузки страницы — общее для всех страниц
    protected void waitForPageLoaded() {
        $(".page-content").shouldBe(Condition.visible);
    }

    // Скриншот текущего состояния — инфраструктурная утилита
    protected byte[] takeScreenshot() {
        return ((TakesScreenshot) WebDriverRunner.getWebDriver())
                .getScreenshotAs(OutputType.BYTES);
    }
}
```

## Ключевые принципы проектирования

### 1. Модульность (Modularity)

Каждый компонент фреймворка должен быть **независимым модулем** с чётким интерфейсом.
Замена Selenide на Playwright не должна требовать переписывания тестов — только инфраструктурного слоя.

```
framework/
├── core/          ← Базовые абстракции и интерфейсы
├── ui/            ← UI-взаимодействие (Selenide/Playwright)
├── api/           ← API-взаимодействие (REST Assured)
├── db/            ← Работа с базами данных
├── reporting/     ← Allure-отчётность
└── config/        ← Управление конфигурацией
```

### 2. Переиспользуемость (Reusability)

Код пишется один раз и используется многократно. Примеры:

- **Page Objects** — одна страница описана один раз, используется во всех тестах.
- **API-клиенты** — один класс `UserApiClient` используется как в API-тестах, так и для подготовки данных в UI-тестах.
- **Утилиты** — генерация тестовых данных, форматирование дат, парсинг ответов.

### 3. Масштабируемость (Scalability)

Фреймворк должен расти вместе с проектом:

- От 10 тестов до 10 000 — без переписывания архитектуры.
- Добавление нового микросервиса — добавление нового модуля, а не переделка существующего.
- Параллельный запуск — архитектура изначально thread-safe.

### 4. Поддерживаемость (Maintainability)

- **Единая точка изменения** — поменять URL среды в одном месте, а не в 200 тестах.
- **Понятные названия** — `loginPage.enterUsername()`, а не `page.input1.type()`.
- **Минимальная связанность** — изменение в одном модуле не ломает другие.

### 5. Читаемость (Readability)

Тесты — это живая документация. Любой член команды (включая ручного тестировщика и менеджера) должен понять,
что делает тест, прочитав его.

### 6. DRY (Don't Repeat Yourself)

Дублирование — враг поддерживаемости. Но применять DRY нужно с умом: иногда небольшое дублирование в тестах
лучше, чем сложная абстракция, которую трудно понять.

### 7. KISS (Keep It Simple, Stupid)

Не переусложняйте. Если для вашего проекта достаточно простого Page Object без Steps-слоя — не добавляйте его
ради «красивой архитектуры». Фреймворк должен решать **реальные** задачи.

## Признаки плохого фреймворка

| Признак | Последствие |
|---------|-------------|
| Копирование кода между тестами | Изменение в одном месте требует правки во многих файлах |
| Тесты зависят друг от друга | Падение одного теста вызывает каскадное падение |
| Хардкод URL, логинов, паролей | Невозможно запустить на другом окружении |
| Огромные тестовые методы (100+ строк) | Трудно понять, что тестируется |
| Нет логирования и отчётности | Невозможно понять причину падения |
| `Thread.sleep()` вместо явных ожиданий | Тесты нестабильны (flaky) |
| Всё в одном пакете | Невозможно навигировать по проекту |
| Нет README и комментариев | Новичку нужны дни на онбординг |

## Характеристики хорошего фреймворка

- **Быстрый старт** — новый QA пишет первый тест за 30 минут.
- **Понятная структура** — по названию пакета ясно, что внутри.
- **Единообразие** — все тесты написаны в одном стиле.
- **Гибкость** — легко добавить новый тип тестирования (mobile, performance).
- **Отчётность** — красивые Allure-отчёты «из коробки».
- **CI/CD-ready** — запускается в Docker без дополнительной настройки.
- **Конфигурируемость** — переключение окружений через переменные.
- **Изолированность тестов** — каждый тест независим и может запускаться отдельно.

## Фреймворк как продукт

Относитесь к фреймворку как к **внутреннему продукту** вашей команды:

1. **Пользователи** — QA-инженеры, которые пишут тесты.
2. **Документация** — README, примеры, JavaDoc.
3. **Версионирование** — стабильные релизы, changelog.
4. **Code Review** — изменения в фреймворке проходят ревью.
5. **Обратная связь** — собирайте фидбек от пользователей.
6. **Метрики** — время написания теста, процент flaky-тестов, покрытие.

## Связь с тестированием

Принципы проектирования фреймворка напрямую влияют на качество тестирования:

- **Плохая архитектура** → медленное написание тестов → низкое покрытие → пропущенные баги.
- **Хорошая архитектура** → быстрое написание тестов → высокое покрытие → ранее обнаружение дефектов.
- **Хрупкие тесты** → команда теряет доверие к автотестам → перестаёт их запускать → автоматизация бесполезна.
- **Стабильный фреймворк** → тесты запускаются на каждом коммите → баги находятся в тот же день.

Фреймворк — это **фундамент**. Если фундамент кривой, никакое количество тестов не спасёт.

---

## Типичные ошибки

1. **Over-engineering** — создание абстракций «на будущее», которые никогда не понадобятся.
2. **Игнорирование SOLID** — God-классы на 2000 строк, которые делают всё.
3. **Завязка на конкретный инструмент** — Selenide-специфичный код в бизнес-логике тестов.
4. **Отсутствие логирования** — при падении теста непонятно, что произошло.
5. **Нет конвенций** — каждый пишет как хочет, проект превращается в хаос.
6. **Копирование чужого фреймворка** — без понимания, зачем в нём те или иные решения.
7. **Попытка автоматизировать всё сразу** — вместо итеративного развития.

## Практические задания

### Задание 1: Анализ архитектуры
Возьмите open-source тестовый фреймворк (например, selenide-examples на GitHub).
Нарисуйте диаграмму его слоёв. Определите, какие принципы соблюдены, а какие нарушены.

### Задание 2: Проектирование с нуля
У вас — интернет-магазин с REST API, веб-интерфейсом и PostgreSQL.
Спроектируйте структуру тестового фреймворка: пакеты, базовые классы, точки расширения.
Нарисуйте диаграмму зависимостей между модулями.

### Задание 3: Рефакторинг
Дан «плохой» тест (копипаста, хардкод, `Thread.sleep()`). Проведите рефакторинг:
выделите Page Object, Steps, конфигурацию, ожидания. Сравните «до» и «после».

### Задание 4: Elevator Pitch
Подготовьте 2-минутный рассказ: «Как бы я спроектировал тестовый фреймворк для проекта X».
Тренируйтесь объяснять решения и trade-offs.
